PRAGMA 语句

PRAGMA 语句是 SQLite 特有的 SQL 扩展，用于修改 SQLite 库的行为或查询 SQLite 库的内部（非表）数据。PRAGMA 语句通过与其他 SQLite 命令（例如 SELECT、INSERT）相同的接口发出，但在以下几个重要方面有所不同：

  • pragma 命令是 SQLite 专用的，不兼容任何其他 SQL 数据库引擎。
  • 特定的 pragma 语句可能在未来的 SQLite 发行版中被移除或新增。没有向后兼容性的保证。
  • 如果发出未知的 pragma，不会产生错误信息。未知的 pragma 会被简单忽略。这意味着如果 pragma 语句中有拼写错误，库不会通知用户这一事实。
  • 某些 pragma 会在 SQL 编译阶段生效，而非执行阶段。这意味着如果使用 C 语言的 sqlite3_prepare()、sqlite3_step()、sqlite3_finalize() API（或包装接口中的类似函数），该 pragma 可能在 sqlite3_prepare() 调用期间执行，而不是像普通 SQL 语句那样在 sqlite3_step() 调用期间执行。或者该 pragma 也可能像普通 SQL 语句一样在 sqlite3_step() 中运行。pragma 是否在 sqlite3_prepare() 期间或 sqlite3_step() 期间运行，取决于该 pragma 以及 SQLite 的具体版本。
  • SQL 语句前缀 EXPLAIN 和 EXPLAIN QUERY PLAN 仅影响该语句在 sqlite3_step() 执行期间的行为。这意味着在 sqlite3_prepare() 阶段生效的 PRAGMA 语句无论是否以 “EXPLAIN” 为前缀，行为都相同。

SQLite 的 C 语言 API 提供了 SQLITE_FCNTL_PRAGMA 文件控制，它为 VFS 实现提供机会添加新的 PRAGMA 语句或覆盖内置 PRAGMA 语句的含义。

PRAGMA 命令语法

pragma-stmt:

pragma-value:

signed-number:

一个 pragma 可以接受零个或一个参数。参数可以用括号括起，也可以用等号与 pragma 名称分隔。这两种语法产生相同的结果。 在许多 pragma 中，参数是布尔值。布尔值可以是：

1 yes true on
0 no false off

关键字参数可以选择性地用引号括起。 （示例：'yes' [FALSE]。）某些 pragma 接受字符串字面量作为参数。当 pragma 接受关键字参数时，通常也接受相应的数字形式。 例如，“0”和“no”表示相同含义，“1”和“yes”也相同。在查询设置的值时，很多 pragma 返回的是数字而不是关键字。

一个 pragma 可以在名称之前带有可选的 schema-name。 schema-name 是 ATTACH-ed 数据库的名称，或主数据库的 “main” 以及 TEMP 数据库的 “temp”。如果省略可选的 schema 名称，则默认使用 “main”。在某些 pragma 中，schema 名称没有意义，仅仅被忽略。在下文的文档中，那些 schema 名称有意义的 pragma 会以 “schema.” 前缀展示。

PRAGMA 函数

返回结果并且无副作用的 PRAGMA 可以通过普通的 SELECT 语句作为 表值函数 访问。 对于每个参与的 PRAGMA，相应的表值函数名称与 PRAGMA 相同，并带有 7 字符的 “pragma_” 前缀。 PRAGMA 的参数和 schema（如有）作为参数传递给表值函数，其中 schema 是可选的最后一个参数。

例如，可以使用 index_info pragma 读取索引列的信息，如下所示：

PRAGMA index_info('idx52');

或者，可以用下面的方式读取相同的内容：

SELECT * FROM pragma_index_info('idx52');

表值函数格式的优点在于查询可以只返回 PRAGMA 列的一部分，可以包含 WHERE 子句，可以使用聚合函数，并且表值函数可以作为连接中的多个数据源之一。 例如，要获取模式中所有索引列的列表，可以查询：

SELECT DISTINCT m.name || '.' || ii.name AS 'indexed-columns'
  FROM sqlite_schema AS m,
       pragma_index_list(m.name) AS il,
       pragma_index_info(il.name) AS ii
 WHERE m.type='table'
 ORDER BY 1;

附加说明：

  • 表值函数仅适用于内置 PRAGMA，而不适用于使用 SQLITE_FCNTL_PRAGMA 文件控制定义的 PRAGMA。

  • 表值函数仅适用于返回结果且无副作用的 PRAGMA。

  • 此功能可用于实现 information schema，方法是首先使用以下命令创建一个单独的模式：

    ATTACH ':memory:' AS 'information_schema';
    然后在该模式中创建 VIEWs，使用表值 PRAGMA 函数实现官方的信息模式表。

  • PRAGMA 表值函数特性在 SQLite 版本 3.16.0（2017-01-02）中添加。早期版本的 SQLite 无法使用此特性。

PRAGMA analysis_limit

PRAGMA analysis_limit;
PRAGMA analysis_limit = N;

查询或更改 approximate ANALYZE 设置的限制。 这是 ANALYZE 命令在每个索引中检查的大致行数。 如果省略了参数 N，那么分析限制保持不变。 如果限制为零，则禁用分析限制，ANALYZE 命令将检查每个索引的所有行。 如果 N 大于零，则分析限制设置为 N，随后的 ANALYZE 命令将在检查大约 N 行后停止分析每个索引。 如果 N 是负数或非整数值，则该 pragma 表现得好像省略了 N 参数。 无论哪种情况，返回的值都是用于随后的 ANALYZE 命令的新分析限制。

此 pragma 可用于帮助 ANALYZE 命令在大型数据库上更快地运行。仅检查每个索引的一部分时，分析结果不是最理想，但通常足够好。将 N 设置为 100 或 1000 可以使 ANALYZE 命令即使在巨大的数据库文件上也能快速完成。

此 pragma 在 SQLite 版本 3.32.0（2020-05-22）中添加。 当前实现仅使用 N 值的低 31 位 —— 高位静默忽略。SQLite 的未来版本可能会开始使用高位。

从 SQLite 版本 3.46.0（2024-05-23）开始，推荐运行 ANALYZE 的方式是使用 PRAGMA optimize 命令。PRAGMA optimize 会自动设置一个合理的、临时的分析限制，以确保 PRAGMA optimize 命令即使在巨大数据库上也能快速完成。使用 PRAGMA optimize 而不是直接运行 ANALYZE 的应用程序无需再设置分析限制。 PRAGMA application_id

PRAGMA schema.application_id;
PRAGMA schema.application_id = integer ;

application_id PRAGMA 用于查询或设置位于 database header 偏移量 68 处的 32 位有符号大端 “Application ID” 整数。将 SQLite 用作 application file-format 的应用程序应将 Application ID 整数设置为唯一值，以便诸如 file(1) 等工具能确定具体文件类型，而不是仅仅报告 “SQLite3 Database”。SQLite 源代码仓库中的 magic.txt 文件可以查看已分配的 application ID 列表。

另请参见 user_version pragma。 PRAGMA auto_vacuum

PRAGMA schema.auto_vacuum;
PRAGMA schema.auto_vacuum = 0 | NONE | 1 | FULL | 2 | INCREMENTAL;

查询或设置数据库中的自动清理状态。

除非使用 SQLITE_DEFAULT_AUTOVACUUM 编译时选项，否则 auto_vacuum 的默认设置为 0 或 “none”。“none” 表示禁用自动清理。 当禁用自动清理并从数据库中删除数据时，数据库文件大小保持不变。未使用的数据库文件页面被加入 “freelist” 并在后续插入时重用。因此不会浪费数据库文件空间。但是，数据库文件不会收缩。在这种模式下，可以使用 VACUUM 命令重建整个数据库文件，从而回收未使用的磁盘空间。

当 auto_vacuum 模式为 1 或 “full” 时，freelist 页面会被移动到数据库文件末尾，并在每次事务提交时截断以移除这些 freelist 页面。 不过请注意，auto-vacuum 仅从文件中截断 freelist 页面。auto-vacuum 不会像 VACUUM 命令那样进行数据库碎片整理或重新打包单个数据库页面。事实上，由于它会在文件内部移动页面，auto-vacuum 实际上可能会使碎片情况更糟。

只有在数据库存储一些额外信息以使每个数据库页面可追溯到其引用页面时，自动清理才成为可能。因此，在创建任何表之前必须先开启自动清理。创建表之后无法再启用或禁用自动清理。

当 auto_vacuum 的值为 2 或 “incremental” 时，执行自动清理所需的额外信息会存储在数据库文件中，但自动清理不会像 auto_vacuum=full 那样在每次提交时自动发生。在增量模式下，必须调用单独的 incremental_vacuum pragma 来触发自动清理。

数据库连接可以在完整自动清理模式和增量模式之间随时切换。但是，从 “none” 更改为 “full” 或 “incremental” 只能在数据库是新的（尚未创建表）时进行，或者通过运行 VACUUM 命令。要更改自动清理模式，首先使用 auto_vacuum pragma 设置新的所需模式，然后调用 VACUUM 命令对整个数据库文件重新组织。要从 “full” 或 “incremental” 更改回 “none”，即使在空数据库上也必须运行 VACUUM。

当调用 auto_vacuum pragma 时不带参数，它会返回当前的 auto_vacuum 模式。

PRAGMA automatic_index

PRAGMA automatic_index;
PRAGMA automatic_index = boolean;

查询、设置或清除 automatic indexing 功能。

Automatic indexing 在 version 3.7.17（2013-05-20）及之后默认启用， 但这可能会在未来的 SQLite 发行版中有所变化。 PRAGMA busy_timeout

PRAGMA busy_timeout;
PRAGMA busy_timeout = milliseconds;

查询或更改 busy timeout 设置。 该 pragma 是 sqlite3_busy_timeout() C 语言接口的替代方案，以便在不提供直接访问 sqlite3_busy_timeout() 的语言绑定中使用。

每个数据库连接只能拥有一个 busy handler。该 PRAGMA 为进程设置 busy handler，可能会覆盖之前设置的 busy handler。

PRAGMA cache_size

PRAGMA schema.cache_size;
PRAGMA schema.cache_size = pages;
PRAGMA schema.cache_size = -kibibytes;

查询或更改 SQLite 每个打开的数据库文件在内存中同时保持的建议最大数据库磁盘页面数。是否遵循该建议由 Application Defined Page Cache 决定。 内置的默认页面缓存会遵循该请求，但替代的应用定义页面缓存实现可能以不同方式解释建议的缓存大小，甚至完全忽略它。 默认建议的缓存大小为 -2000，意味着缓存大小限制为 2048000 字节的内存。 可以通过 SQLITE_DEFAULT_CACHE_SIZE 编译时选项修改默认建议缓存大小。 TEMP 数据库的默认建议缓存大小为 0 页。

如果参数 N 为正，则建议的缓存大小设置为 N。如果 N 为负，则缓存页数调整为在当前页面大小基础上大约使用 abs(N*1024) 字节内存的页数。SQLite 会记住页面缓存中的页数，而不是所用内存量。因此如果使用负数设置缓存大小，然后再更改页面大小（使用 PRAGMA page_size 命令），则最大缓存内存会按页面大小的变化相应增减。

向后兼容性说明：在 version 3.7.10（2012-01-16）之前，cache_size 使用负值时的行为不同。在早期版本中，缓存的页数设置为 N 的绝对值。

通过 cache_size pragma 更改缓存大小后，该更改仅在当前会话中有效。关闭并重新打开数据库后，缓存大小会恢复为默认值。

默认的页面缓存实现不会一次性分配全部缓存内存。缓存内存在按需的基础上以较小的块分配。page_cache 设置是缓存可以使用的（建议）上限，而不是它将始终使用的内存量。这是默认页面缓存实现的行为，但application defined page cache 如果愿意可以表现得不同。 PRAGMA cache_spill

PRAGMA cache_spill;
PRAGMA cache_spill=boolean;
PRAGMA schema.cache_spill=N;

cache_spill pragma 启用或禁用页缓存在事务中将脏页面溢写到数据库文件的能力。cache_spill 默认为启用，大多数应用应保持这一状态，因为缓存溢写通常有利。 但是，缓存溢写有一个副作用：它会对数据库文件获取 EXCLUSIVE lock。因此，一些具有长时间运行事务的应用可能希望禁用缓存溢写，以避免在事务 COMMIT 之前获取排他锁。

“PRAGMA cache_spill=N” 形式的 pragma 设置触发溢写所需的最小缓存大小阈值。缓存中的页面数必须同时超过 cache_spill 阈值以及通过 PRAGMA cache_size 设置的最大缓存大小，才能发生溢写。

“PRAGMA cache_spill=boolean” 形式的该 pragma 适用于连接的所有附加数据库。但该语句的 “PRAGMA cache_spill=N” 形式仅适用于 “main” 模式或语句中指定的其他模式。

PRAGMA case_sensitive_like

PRAGMA case_sensitive_like = boolean;

LIKE 运算符的默认行为是对 ASCII 字符忽略大小写。因此默认情况下 'a' LIKE 'A' 为 true。case_sensitive_like pragma 会根据其值安装一个新的应用程序定义的 LIKE 函数，该函数要么区分大小写，要么不区分大小写。当 case_sensitive_like 被禁用时，应用默认的 LIKE 行为；当 case_sensitive_like 启用时，大小写变得敏感。因此，例如 'a' LIKE 'A' 为 false，但 'a' LIKE 'a' 依然为 true。

该 pragma 使用 sqlite3_create_function() 来重载 LIKE 和 GLOB 函数，这可能会覆盖应用程序之前注册的 LIKE 和 GLOB 的实现。此 pragma 只改变 SQL LIKE 运算符的行为。它不会改变 sqlite3_strlike() C 语言接口的行为，该接口始终不区分大小写。

警告：如果数据库在模式中任何地方都使用 LIKE 运算符，例如在 CHECK 约束 中、在 表达式索引 中，或在 部分索引 的 WHERE 子句中，则使用该 PRAGMA 更改 LIKE 运算符的定义可能导致数据库看起来已损坏。PRAGMA integrity_check 会报告错误。实际上数据库并未真正损坏，只需将 LIKE 的行为恢复到定义模式并填充数据库时的方式即可解决问题。如果 LIKE 仅在索引中使用，则可以通过运行 REINDEX 清除问题。尽管如此，不鼓励使用 case_sensitive_like pragma。

此 pragma 已弃用，仅为向后兼容而存在。新应用应避免使用此 pragma。旧应用应尽早停止使用此 pragma。当 SQLite 使用 SQLITE_OMIT_DEPRECATED 编译时，可从构建中省略此 pragma。

PRAGMA cell_size_check

PRAGMA cell_size_check
PRAGMA cell_size_check = boolean;

cell_size_check pragma 在从磁盘首次读取数据库 b 树页时启用或禁用额外的合理性检查。启用页面大小检查时，可更早检测到数据库损坏，并且损坏不太可能“扩散”。不过，执行这些额外检查会带来一些性能损失，因此默认情况下关闭页面大小检查。

PRAGMA checkpoint_fullfsync

PRAGMA checkpoint_fullfsync
PRAGMA checkpoint_fullfsync = boolean;

查询或更改 checkpoint 操作的 fullfsync 标志。如果设置了此标志，则在支持 F_FULLFSYNC 的系统上执行 checkpoint 操作时使用 F_FULLFSYNC 同步方法。checkpoint_fullfsync 标志的默认值为关闭。只有 Mac OS-X 支持 F_FULLFSYNC。

如果设置了 fullfsync 标志，则对所有同步操作都使用 F_FULLFSYNC 同步方法，此时 checkpoint_fullfsync 设置无关紧要。

PRAGMA count_changes

PRAGMA count_changes;
PRAGMA count_changes = boolean;

查询或更改 count-changes 标志。通常，当 count-changes 标志未设置时，INSERT、UPDATE 和 DELETE 语句不会返回数据。设置了 count-changes 时，这些命令中的每一个都会返回一行数据，该行包含一个整数值 —— 表示命令插入、修改或删除的行数。返回的更改计数不包括触发器执行的插入、修改或删除操作，不包括 外键操作 自动完成的更改，也不包括由 upsert 引起的更新。

获取行更改计数的另一种方法是使用 sqlite3_changes() 或 sqlite3_total_changes() 接口。不过有一个微小差别：当对视图运行 INSERT、UPDATE 或 DELETE 且使用了 INSTEAD OF 触发器 时，count_changes pragma 报告的是触发器触发的视图中行的数目，而 sqlite3_changes() 与 sqlite3_total_changes() 不会这样。

此 pragma 已弃用，仅为向后兼容而存在。新应用应避免使用此 pragma。旧应用应尽早停止使用此 pragma。当 SQLite 使用 SQLITE_OMIT_DEPRECATED 编译时，可从构建中省略此 pragma。

PRAGMA data_store_directory

PRAGMA data_store_directory;
PRAGMA data_store_directory = 'directory-name';

查询或更改 sqlite3_data_directory 全局变量的值，Windows 操作系统接口后端使用该变量来确定使用相对路径名指定的数据库文件的存放位置。

更改 data_store_directory 设置不是线程安全的。不要在应用程序的其他线程同时运行任何 SQLite 接口时更改 data_store_directory 设置。那样会导致未定义行为。更改 data_store_directory 设置会写入 sqlite3_data_directory 全局变量，该全局变量未受互斥锁保护。

此功能是为没有读取或更改当前工作目录机制的 WinRT 提供的。在其他上下文中使用该 pragma 不被鼓励，未来版本可能会禁止。

此 pragma 已弃用，仅为向后兼容而存在。新应用应避免使用此 pragma。旧应用应尽早停止使用此 pragma。当 SQLite 使用 SQLITE_OMIT_DEPRECATED 编译时，可从构建中省略此 pragma。

PRAGMA data_version

PRAGMA schema.data_version;

“PRAGMA data_version” 命令提供了数据库文件已被修改的指示。将数据库内容保留在内存中的交互式程序或在屏幕上显示数据库内容的程序，可以使用 PRAGMA data_version 命令来确定是否需要刷新并重新加载内存或更新屏幕显示。

来自同一连接的两次 “PRAGMA data_version” 调用返回的整数值在任何其他连接在此期间提交了更改时会不同。对于同一数据库连接提交的事务，“PRAGMA data_version” 值保持不变。“PRAGMA data_version”的行为对所有数据库连接都是相同的，包括单独进程中的数据库连接以及共享缓存数据库连接。

“PRAGMA data_version” 值是每个数据库连接的本地属性，因此即使底层数据库相同，两个不同数据库连接的同时调用可能返回不同的值。只有在不同时间点对同一数据库连接调用 “PRAGMA data_version” 时，比较这些值才有意义。

  1. 如果结果上有 AS 子句，则列名为 AS 子句的右侧。

  2. 列在被索引表中的排名。 值为 -1 表示 rowid，值为 -2 表示正在使用 expression。

此 pragma 为给定表关联的每个索引返回一行。

index_list pragma 的输出列如下：

  1. 分配给每个索引的序列号，用于内部跟踪。
  2. 索引名称。
  3. 如果索引是 UNIQUE，则为 "1"，否则为 "0"。

  1. 列在索引中的排名。（0 表示最左边。键列在辅助列之前。）
  2. 列在被索引表中的排名，如果索引列是所索引表的 rowid 则为 -1，如果在 expression 上索引则为 -2。
  3. 正在被索引的列的名称；如果索引列是所索引表的 rowid 或 expression，则为 NULL。
  4. 如果索引列按照逆序（DESC）由索引排序，则为 1，否则为 0。
  5. 用于比较索引列值的 collating sequence 的名称。

PRAGMA schema.integrity_check;

此 pragma 对数据库进行低级格式和一致性检查。integrity_check pragma 会检查：

  • 序列错位的表或索引条目
  • 格式错误的记录
  • 丢失的页
  • 缺失或多余的索引条目
  • UNIQUE、CHECK 和 NOT NULL 约束错误
  • freelist 的完整性
  • 在数据库中被多次使用或根本未使用的部分

如果 integrity_check pragma 发现问题，会返回描述问题的字符串（作为多行，每行一列）。pragma integrity_check 在分析结束前最多返回 N 个错误，默认 N 为 100。如果 pragma integrity_check 未发现错误，则返回一行，值为 'ok'。

通常情况是检查整个数据库文件。但是，如果参数是 TABLENAME，则只检查指定表及其关联的索引。 这称为“部分完整性检查”。由于只检查数据库的子集，因此无法检测未使用的文件部分或两个或更多表使用相同文件部分的错误。 只有当 TABLENAME 是 sqlite_schema 或其别名之一时，才会在部分完整性检查中验证 freelist。部分完整性检查的支持添加于版本 3.33.0（2020-08-14）。

PRAGMA integrity_check 无法发现 FOREIGN KEY 错误。 使用 PRAGMA foreign_key_check 命令查找 FOREIGN KEY 约束中的错误。

另请参阅 PRAGMA quick_check 命令，它执行了大部分 PRAGMA integrity_check 的检查，但运行速度更快。

PRAGMA journal_mode

PRAGMA schema.journal_mode;
PRAGMA schema.journal_mode = DELETE | TRUNCATE | PERSIST | MEMORY | WAL | OFF

此 pragma 查询或设置当前 database connection 关联的数据库的 journal mode。

第一种形式查询 database 的当前日志记录模式。当省略 database 时，会查询 "main" 数据库。

第二种形式更改 "database" 的日志记录模式，或者如果省略 "database"，则更改所有附加的数据库。 返回新的 journal mode。如果无法更改 journal mode，则返回原始 journal mode。

DELETE journaling mode 是默认值。在 DELETE 模式中，每个事务结束后都会删除回滚日志。实际上，删除操作就是让事务提交的操作。 （有关详细信息，请参阅题为 Atomic Commit In SQLite 的文档。）

TRUNCATE journaling mode 通过截断回滚日志到零长度而不是删除它来提交事务。在许多系统中，截断文件比删除文件更快，因为不需要更改包含该文件的目录。

PERSIST journaling mode 阻止在每个事务结束时删除回滚日志。相反，将日志头覆盖为零。这样可以防止其他数据库连接回滚日志。PERSIST journaling mode 在删除或截断文件远比将文件的首块覆盖为零更昂贵的平台上作为一种优化很有用。另见： PRAGMA journal_size_limit 和 SQLITE_DEFAULT_JOURNAL_SIZE_LIMIT。

MEMORY journaling mode 将回滚日志存储在易失性 RAM 中。这节省了磁盘 I/O，但以牺牲数据库安全性和完整性为代价。如果在 MEMORY journaling mode 下应用程序在事务中途崩溃，则数据库文件很可能 go corrupt。

WAL journaling mode 使用 write-ahead log 而不是回滚日志来实现事务。WAL journaling mode 是持久的；设置后，它在多次 database connection 之间以及在关闭并重新打开数据库后仍然有效。处于 WAL journaling mode 的数据库只能由 SQLite version 3.7.0（2010-07-21）或更高版本访问。

OFF journaling mode 完全禁用回滚日志。永远不会创建回滚日志，因此也不会删除回滚日志。OFF journaling mode 禁用 SQLite 的原子提交和回滚功能。在此模式下，ROLLBACK 命令不再有效；其行为未定义。在 journal mode 为 OFF 时，应用程序必须避免使用 ROLLBACK 命令。如果在事务中途应用程序崩溃，数据库文件很可能 go corrupt。没有日志，就无法在约束错误之后撤销部分完成的操作。这可能会使数据库处于损坏状态。例如，如果重复条目导致 CREATE UNIQUE INDEX 语句在中途失败，则会留下一个部分创建的、因此是损坏的索引。由于 OFF journaling mode 允许使用普通 SQL 使数据库文件损坏，当启用 SQLITE_DBCONFIG_DEFENSIVE 时，该模式被禁用。

请注意，in-memory database 的 journal_mode 只能是 MEMORY 或 OFF，不能更改为其他值。尝试将 in-memory database 的 journal_mode 更改为 MEMORY 或 OFF 以外的设置将被忽略。另外请注意，在事务活动期间 journal_mode 无法更改。

PRAGMA journal_size_limit

PRAGMA schema.journal_size_limit
PRAGMA schema.journal_size_limit = N ;

如果数据库连接在 exclusive locking mode 或 persistent journal mode（PRAGMA journal_mode=persist）下运行，则提交事务后，rollback journal 文件可能会保留在文件系统中。这会提高后续事务的性能，因为覆盖现有文件比追加到文件更快，但也会占用文件系统空间。在大型事务之后（例如 VACUUM），回滚日志文件可能占用非常大的空间。

类似地，在 WAL mode 中，完成 checkpoint 后不会截断 write-ahead log 文件。SQLite 会重用现有文件以存放后续的 WAL 条目，因为覆盖比追加更快。

journal_size_limit pragma 可用于限制事务或 checkpoint 之后留在文件系统中的回滚日志和 WAL 文件的大小。 每次事务提交或 WAL 文件重置时，SQLite 将文件系统中剩余的回滚日志或 WAL 文件的大小与此 pragma 设置的大小限制进行比较，如果日志或 WAL 文件大于该限制，则将其截断到限制值。

上面列出的第二种形式用于为指定数据库设置新的字节限制。负数表示无限制。要始终将回滚日志和 WAL 文件截断到最小大小，请将 journal_size_limit 设置为零。 上面列出的第一种和第二种形式都返回一行、一个整数列——以字节为单位的 journal 大小限制值。默认的 journal 大小限制为 -1（无限制）。 SQLITE_DEFAULT_JOURNAL_SIZE_LIMIT 预处理宏可用于在编译时更改默认 journal 大小限制。

此 pragma 仅对在 pragma 名称之前指定的单个数据库（如果未指定数据库，则为 "main" 数据库）起作用。无法使用单个 PRAGMA 语句更改所有附加数据库的 journal 大小限制。必须分别为每个附加数据库设置大小限制。

PRAGMA legacy_alter_table

PRAGMA legacy_alter_table;
PRAGMA legacy_alter_table = boolean

此 pragma 设置或查询 legacy_alter_table 标志的值。当此标志打开时，ALTER TABLE RENAME 命令（用于更改表名称）遵循 SQLite 3.24.0（2018-06-04）及更早版本的行为。更具体地说，当此标志打开时，ALTER TABLE RENAME 命令仅重写其 CREATE TABLE 语句和任何关联的 CREATE INDEX 与 CREATE TRIGGER 语句中的表名的初始出现。对该表的其他引用不做修改，包括：

  • 触发器和视图主体内对该表的引用。
  • 原始 CREATE TABLE 语句中的 CHECK 约束内对该表的引用。
  • partial indexes 的 WHERE 子句中对该表的引用。
此 pragma 的默认设置为 OFF，即 schema 中任何位置对该表的引用都转换为新名称。

此 pragma 提供给包含依赖旧版本 SQLite 中 ALTER TABLE RENAME 未完整行为的旧程序的兼容方法。新应用程序应该保持此标志关闭。

为与旧的 virtual table 实现兼容，在运行 sqlite3_module.xRename 方法期间，此标志会暂时打开。sqlite3_module.xRename 方法完成后，该标志的值会恢复。

legacy alter table 行为也可以使用 SQLITE_DBCONFIG_LEGACY_ALTER_TABLE 选项通过 sqlite3_db_config() 接口打开或关闭。

legacy alter table 行为是每个连接的设置。打开或关闭此功能会影响 database connection 中的所有附加数据库文件。该设置不会持久化。在一个连接中更改此设置不会影响其他连接。

PRAGMA legacy_file_format;

此 pragma 不再起作用，已成为 no-op。 之前由 PRAGMA legacy_file_format 提供的功能现在可以使用 SQLITE_DBCONFIG_LEGACY_FILE_FORMAT 选项通过 sqlite3_db_config() C 语言接口访问。

PRAGMA locking_mode

PRAGMA schema.locking_mode;
PRAGMA schema.locking_mode = NORMAL | EXCLUSIVE

此 pragma 设置或查询数据库连接的 locking-mode。locking-mode 可以是 NORMAL 或 EXCLUSIVE。

在 NORMAL locking-mode（除非在编译时使用 SQLITE_DEFAULT_LOCKING_MODE 覆盖，否则为默认）下，数据库连接在每次读或写事务结束时都会释放数据库文件。当 locking-mode 设置为 EXCLUSIVE 时，数据库连接永不释放文件锁。第一次以 EXCLUSIVE 模式读取数据库时，会获取并保持一个共享锁。第一次写入数据库时，会获取并保持一个独占锁。

在 EXCLUSIVE 模式下获得的数据库锁可以通过关闭数据库连接释放，或者通过使用此 pragma 将 locking-mode 设置回 NORMAL 然后访问数据库文件（用于读或写）来释放。仅将 locking-mode 设置为 NORMAL 并不足以释放锁——直到下次访问数据库文件之前，锁都不会释放。

将 locking-mode 设置为 EXCLUSIVE 有三个理由。

  1. 应用程序希望防止其他进程访问数据库文件。
  2. 文件系统操作的系统调用次数减少，可能会带来轻微的性能提升。
  3. WAL 数据库可以在 EXCLUSIVE 模式下访问而无需使用共享内存。 (Additional information)

当 locking_mode pragma 指定特定数据库时，例如：

PRAGMA main.locking_mode=EXCLUSIVE;

则锁模式仅应用于指定数据库。如果 "locking_mode" 关键字前没有数据库名称限定符，则锁模式应用于所有数据库，包括随后通过 ATTACH 命令添加的任何新数据库。

"temp" 数据库（用于存储 TEMP 表和索引）和 in-memory databases 始终使用 exclusive locking mode。temp 和 in-memory databases 的 locking mode 无法更改。其他所有数据库默认使用 normal locking mode，并受到此 pragma 的影响。

如果首次进入 WAL journal mode 时 locking mode 为 EXCLUSIVE，则必须退出 WAL journal mode 后才能将 locking mode 更改为 NORMAL。 如果首次进入 WAL journal mode 时 locking mode 为 NORMAL，则可以在任何时候在 NORMAL 与 EXCLUSIVE 之间切换，并且无需退出 WAL journal mode。

查询或更改为单个数据库保留用于内存映射 I/O 的最大字节数。第一种形式（没有参数）查询当前限制。第二种形式（带数字参数）为指定数据库设置限制，或者如果省略可选数据库名称，则为所有数据库设置限制。在第二种形式中，如果省略数据库名称，则设置的限制会成为随后的 ATTACH 语句向 database connection 添加的所有数据库的默认限制。

  1. 短期数据库连接的应用程序应在关闭每个数据库连接之前运行一次 "PRAGMA optimize;"。

  2. 使用长期数据库连接的应用程序应在首次打开连接时运行 "PRAGMA optimize=0x10002;"，然后也应定期运行"PRAGMA optimize;"，可能是每天或每小时一次。

此 pragma 执行的优化细节预计会随着时间的推移而变化和改进。 应用程序应该预期该 pragma 在未来版本中会执行新的优化。

可选的 MASK 参数是一个要执行的优化位掩码：

0x00001 调试模式。实际上不执行任何优化 而是对于每个本应执行的优化返回一行文本。默认关闭。
0x00002 对可能受益的表运行 ANALYZE。默认开启。
0x00010 运行 ANALYZE 时，设置一个临时的 PRAGMA analysis_limit，以防止运行时间过长。默认开启。
0x10000 检查所有表的大小，而不仅仅是最近使用过的表，以查看是否有任何表显著增长或缩小，因此可能 受益于重新分析。默认关闭。

默认的 MASK 为 0xfffe。

要查看所有本应执行但实际上不执行的优化，请运行 "PRAGMA optimize(-1)"。

  1. MASK 的 0x02 位被设置。
  2. 该表是普通表，而不是视图或虚拟表。
  3. 表名不以 "sqlite_" 开头。

PRAGMA synchronous

PRAGMA schema.synchronous;
PRAGMA schema.synchronous = 0 | OFF | 1 | NORMAL | 2 | FULL | 3 | EXTRA;

查询或更改 “synchronous” 标志的设置。 第一个（查询）形式将返回 synchronous 设置的整数值。第二种形式更改 synchronous 设置。 各种 synchronous 设置的含义如下：

EXTRA (3)

EXTRA synchronous 类似于 FULL，新增功能是在删除模式下，提交事务时在回滚日志被 unlink 之后同步包含该日志的目录。 如果提交紧随其后发生电源中断，EXTRA 提供了额外的持久性保障。 如果没有 EXTRA，根据底层文件系统，可能会出现单个在电源中断前提交的事务在重启后被回滚的情况。 数据库不会损坏。但如果未设置 EXTRA，可能会丢失最后一个事务，从而违反持久性。

在 WAL mode 中，EXTRA 与 FULL 没有区别。

FULL (2)

当 synchronous 为 FULL (2) 时，SQLite 数据库引擎将调用 VFS 的 xSync 方法，以确保在继续之前所有内容都被安全地写入磁盘表面。 这可确保操作系统崩溃或断电不会损坏数据库。

FULL 是 rollback journal 的默认 synchronous 模式。 FULL 在 WAL mode 中是原子、一致、隔离且持久（ACID）的，在 rollback journal 中则是原子、一致且隔离的。 在某些底层文件系统下，使用 rollback journal 时 FULL 也可能具备持久性。但在 rollback 模式下，FULL 不一定能在电源中断后保持持久性，因此如果需要持久性，最好将 synchronous 模式设置为 EXTRA。

NORMAL (1)

当 synchronous 为 NORMAL (1) 时，SQLite 数据库引擎仍会在最关键的时刻进行同步，但频率低于 FULL 模式。在旧文件系统上，如果在恰当的时刻发生断电，可能会导致在 journal_mode=DELETE 的情况下数据库损坏的极小概率。 WAL mode 在 synchronous=NORMAL 下不会损坏，并且在现代文件系统上 DELETE 模式也可能安全。 WAL 模式在 synchronous=NORMAL 下始终保持一致，但 WAL 模式确实会失去持久性。在 WAL 模式中使用 synchronous=NORMAL 提交的事务可能在断电或系统崩溃后回滚。 无论 synchronous 设置或 journal 模式如何，事务在应用程序崩溃时仍然保持持久性。

synchronous=NORMAL 设置在大多数运行于 WAL mode 的应用中在性能与安全之间提供了最佳平衡。 在 WAL 模式下使用 synchronous NORMAL 会丢失跨断电的持久性，但对于大多数应用来说这并不重要。 事务仍保持原子性、一致性和隔离性，这些是大多数用例中最重要的特性。

OFF (0)

当 synchronous 为 OFF (0) 时，SQLite 在将数据传递给操作系统后立即继续，而不进行同步。 如果运行 SQLite 的应用崩溃，数据仍然安全，但如果操作系统崩溃或计算机在数据写入非易失存储之前断电，数据库 可能会损坏。 另一方面，使用 synchronous OFF 时提交可以快得多。

在从头创建新数据库时，且在途中发生断电后可以重新执行创建过程，同时对性能有较高要求的情况下，将 synchronous 设置为 OFF 是一个不错的选择。

 NORMAL   Maybe not consistent   Maybe not durable 
 OFF   Not consistent   Not consistent 

在 WAL 模式中，当 synchronous 为 NORMAL (1) 时，WAL 文件在每个 checkpoint 之前会被同步，数据库文件在每次完成的 checkpoint 之后会被同步，且 WAL 文件头在 WAL 文件在 checkpoint 后开始被重用时也会被同步，但大多数事务期间不会进行同步操作。 在 WAL 模式下使用 synchronous=FULL 时，每次事务提交后还会额外同步 WAL 文件。 每个事务之后的额外 WAL 同步有助于确保事务在断电后仍保持持久性。无论是否存在 synchronous=FULL 提供的额外同步，事务始终一致。 如果无需持久性，那么在 WAL 模式下一般只需 synchronous=NORMAL。

TEMP 模式始终设置为 synchronous=OFF，因为 TEMP 的内容是短暂的，不期望在断电后存活。 尝试更改 TEMP 的 synchronous 设置会被悄然忽略。

另见 fullfsync 以及 checkpoint_fullfsync pragmas。

PRAGMA temp_store_directory

PRAGMA temp_store_directory;
PRAGMA temp_store_directory = 'directory-name';

查询或更改 sqlite3_temp_directory 全局变量的值，许多操作系统接口后端使用该变量来确定存储 临时表 和索引的位置。

更改 temp_store_directory 设置时，发出 pragma 的数据库连接中的所有现有临时表、索引、触发器和视图会立即被删除。 实际操作中，应在一个进程的第一个数据库连接打开后立即设置 temp_store_directory。 如果在同一进程中其他数据库连接仍打开的情况下更改 temp_store_directory，则其行为未定义，且很可能是不理想的。

PRAGMA wal_checkpoint

PRAGMA schema.wal_checkpoint;
PRAGMA schema.wal_checkpoint(PASSIVE);
PRAGMA schema.wal_checkpoint(FULL);
PRAGMA schema.wal_checkpoint(RESTART);
PRAGMA schema.wal_checkpoint(TRUNCATE);
PRAGMA schema.wal_checkpoint(NOOP);

如果通过 journal_mode pragma 启用了 write-ahead log，此 pragma 会在数据库 database 上运行 checkpoint 操作，或者如果省略 database，则在所有附加数据库上运行。 如果禁用 write-ahead log 模式，则此 pragma 是无害的空操作。

在不带参数的情况下调用此 pragma 等价于调用 sqlite3_wal_checkpoint() C 接口。

带参数调用此 pragma 等价于调用 sqlite3_wal_checkpoint_v2() C 接口，并将 第 3 个参数 设置为对应的参数：
PASSIVE
在不等待任何数据库读者或写者完成的情况下 checkpoint 尽可能多的帧。如果日志中的所有帧都被 checkpoint，则同步数据库文件。此模式与调用 sqlite3_wal_checkpoint() C 接口相同。在此模式下 busy-handler callback 永远不会被调用。
FULL
此模式会阻塞 （调用 busy-handler callback） 直到没有数据库写者，且所有读者都在读取最新的数据库快照。然后 checkpoint 日志文件中的所有帧并同步数据库文件。FULL 在运行时会阻塞并发写者，但读者可以继续。
RESTART
此模式的工作方式与 FULL 相同，新增功能是在 checkpoint 日志文件之后阻塞（调用 busy-handler callback）， 直到所有读者完成对日志文件的使用。这可确保下一个向数据库文件写入内容的客户端将日志文件从头重新开始。RESTART 在运行时会阻塞并发写者，但允许读者继续。
TRUNCATE
此模式的工作方式与 RESTART 相同，新增功能是在成功完成后将 WAL 文件截断为零字节。
NOOP
此模式不会 checkpoint 任何帧。用于仅获取返回值。

wal_checkpoint pragma 返回一行含三个整数列的结果。第一列通常为 0，但如果 RESTART、FULL 或 TRUNCATE checkpoint 被阻止完成（例如因为另一个线程或进程正在主动使用数据库），则为 1。 换言之，如果等价调用 sqlite3_wal_checkpoint_v2() 会返回 SQLITE_OK，则第一列为 0；如果会返回 SQLITE_BUSY，则为 1。 第二列为已写入 write-ahead log 文件的已修改页面数。 第三列为在 checkpoint 结束时已成功移动回数据库文件的 write-ahead log 文件中的页面数。 如果不存在 write-ahead log（例如在不处于 WAL mode 的数据库连接上调用此 pragma），则第二和第三列均为 -1。
