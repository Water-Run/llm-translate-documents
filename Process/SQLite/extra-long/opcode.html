1. 执行摘要

SQLite 的工作方式是将 SQL 语句转换为字节码，然后在虚拟机中运行这些字节码。本文档描述了字节码引擎的运行方式。

本文档介绍了 SQLite 的内部机制。常规的 SQLite 应用开发并不需要阅读本内容。本篇面向希望深入了解 SQLite 内部运行细节的人员。

字节码引擎不是 SQLite 的 API。字节码引擎的细节会随着每个 SQLite 版本的改变而变化。应用程序不应依赖本文件中描述的任何细节。

欲知 SQLite 为什么偏好使用字节码实现 SQL，请参见文档“Why SQLite Uses Bytecode”。

2. 介绍

SQLite 的工作方式是将每个 SQL 语句翻译成字节码并执行该字节码。SQLite 中的prepared statement 实际上就是执行对应 SQL 所需的字节码集合。sqlite3_prepare_v2() 接口是将 SQL 翻译为字节码的编译器。sqlite3_step() 接口则是执行包含在prepared statement 中字节码的虚拟机。

字节码虚拟机是 SQLite 的核心。想要理解 SQLite 内部如何运作的程序员必须熟悉字节码引擎。

从历史上看，SQLite 中的字节码引擎被称为“虚拟数据库引擎”或“VDBE”。本网站中“字节码引擎”、“VDBE”、“虚拟机”与“字节码虚拟机”这些术语可以互换使用，因为它们表示的是同一个概念。

本文档中的“字节码程序”与“prepared statement”也可以互换使用，因为它们在大多数情况下表示的是相同的内容。

2.1. VDBE 源代码

字节码引擎的源代码位于vdbe.c 源文件中。本文档中的操作码定义来自该源文件中的注释。源代码注释是关于字节码引擎信息的权威来源，如有疑问，请查阅源代码。

除了主要的 vdbe.c 源代码文件之外，源码树中还有其他以“vdbe”开头的辅助代码文件——“vdbe”即“Virtual DataBase Engine”的缩写。

请记住，操作码的名称与含义常常随着 SQLite 的版本更迭而发生变化。因此，如果您在分析 SQLite 的EXPLAIN 输出，应参考运行该 EXPLAIN 的 SQLite 版本所对应的本文档（或 vdbe.c 源代码）。否则，操作码的描述可能不准确。本文档基于 SQLite 版本 3.51.2 的提交版本b270f8339eb13（日期为 2026-01-09）。

2.2. 指令格式

SQLite 中的字节码程序由一个或多个指令组成。每条指令都有一个操作码和五个操作数，分别命名为 P1、P2、P3、P4 和 P5。P1、P2 与 P3 操作数是 32 位有符号整数，通常指向寄存器。对于操作 B 树游标的指令，P1 操作数通常是游标编号。对于跳转指令，P2 通常是跳转目的地址。P4 可以是 32 位或 64 位有符号整数、64 位浮点值、字符串字面量、Blob 字面量、指向排序序列比较函数的指针、指向应用程序定义 SQL 函数实现的指针，或其他多种内容。P5 是一个通常用于保存标志的 16 位无符号整数。P5 标志的各个位有时会以微妙的方式影响操作码。例如，如果在Eq 操作码上设置了 P5 操作数的 SQLITE_NULLEQ (0x0080) 位，那么 NULL 值会相互比较为相等；否则 NULL 值会被看作不相等。

某些操作码会使用全部五个操作数。有些操作码只使用一两个。有些操作码根本不使用操作数。

字节码引擎从指令编号 0 开始执行。执行会持续，直到遇到Halt 指令，或者程序计数器变得大于最后一条指令的地址，或者发生错误。字节码引擎停止时，它分配的所有内存都会被释放，并关闭所有可能打开的数据库游标。如果执行因错误而中止，则任何挂起的事务都会被终止，并且对数据库所做的更改会回滚。

ResultRow 操作码会让字节码引擎暂停，并让对应的sqlite3_step() 调用返回SQLITE_ROW。在调用ResultRow 之前，字节码程序会将查询结果某一行加载到一系列寄存器中。像sqlite3_column_int()或sqlite3_column_text()这样的 C 语言 API 会从这些寄存器中提取查询结果。字节码引擎在下一次调用sqlite3_step() 时，从ResultRow 之后的下一条指令继续执行。

2.3. 寄存器

每个字节码程序都有固定（但可能很大）的寄存器数量。单个寄存器可以保存多种对象：

  • NULL 值
  • 有符号 64 位整数
  • IEEE 双精度（64 位）浮点数
  • 任意长度字符串
  • 任意长度 BLOB
  • RowSet 对象（参见RowSetAdd、RowSetRead 和 RowSetTest 操作码）
  • Frame 对象（由子程序使用 —— 参见Program）

寄存器也可以是“未定义”的，表示它根本没有值。未定义不同于 NULL。依据编译时选项，读取未定义寄存器通常会导致运行时错误。如果代码生成器（sqlite3_prepare_v2()）生成了会读取未定义寄存器的prepared statement，那就是代码生成器中的 bug。

寄存器从 0 开始编号。大多数操作码至少引用一个寄存器。

单个 prepared statement 中的寄存器数量在编译时固定。准备语句在被reset或finalize时，会清除所有寄存器的内容。

内部的 Mem 对象负责存储单个寄存器的值。API 中暴露的抽象sqlite3_value对象实际上就是 Mem 对象或寄存器。

2.4. B 树游标

一个 prepared statement 可以有零个或多个打开的游标。每个游标由一个小整数标识，该整数通常是使用该游标的操作码的 P1 参数。同一个索引或表可以打开多个游标。即便是指向同一索引或表的游标也会独立工作。虚拟机与数据库文件交互的唯一方式就是通过游标。虚拟机中的指令可以创建新游标（例如：OpenRead 或 OpenWrite）、从游标读取数据（Column）、将游标移动到表中的下一条记录（例如：Next 或 Prev）等等。prepared statement 在被reset或finalize时，所有游标都会自动关闭。

2.5. 子程序、协程与子程序

字节码引擎没有栈来存储子程序的返回地址，因此返回地址必须保存在寄存器中。因此，字节码子程序不是重入的。

Gosub 操作码将当前程序计数器存入寄存器 P1，然后跳转到地址 P2。Return 操作码跳转到地址 P1+1。因此，每个子程序都关联着两个整数：子程序入口点的地址以及用于保存返回地址的寄存器编号。

Yield 操作码将程序计数器的值与寄存器 P1 中的整数值进行交换。该操作码用于实现协程。协程经常用来实现按需拉取内容的子查询。

触发器需要是可重入的。 由于字节码子程序不可重入，因此必须使用不同的机制来实现触发器。每个触发器通过独立的字节码程序实现，并拥有自己的操作码、程序计数器和寄存器集。Program 操作码调用触发器子程序。Program 指令为每次子程序调用分配并初始化一个新的寄存器集，因此子程序可以是可重入和递归的。Param 操作码用于让子程序访问调用该字节码程序的寄存器内容。

2.6. 自修改代码

某些操作码会自修改。例如，Init 操作码（它总是每个字节码程序的第一条操作码）会递增它的 P1 操作数。后续的Once 操作码会将其 P1 操作数与Init 操作码的 P1 值进行比较，以确定是否跳过随后的仅执行一次的初始化代码。另一个例子是String8 操作码，它将其 P4 操作数从 UTF-8 转换为正确的数据库字符串编码，然后将自身转换为String 操作码。

    如果 P5 中设置了 OPFLAG_LENGTHARG 位，则结果仅保证用于 length() 函数或等效函数。大 blob 的内容不会被加载，从而节省 CPU 周期。如果设置了 OPFLAG_TYPEOFARG 位，则结果仅供 typeof() 函数、IS NULL 或 IS NOT NULL 操作符或其等效者使用。在这种情况下，可以省略所有内容的加载。

    ColumnsUsed 此操作码（仅当 SQLite 使用 SQLITE_ENABLE_COLUMN_USED_MASK 编译时存在）标识游标 P1 所指向的表或索引中使用了哪些列。P4 是一个 64 位整数（P4_INT64），其前 63 位分别对应在游标实际使用的表或索引的前 63 列。如果第 64 列之后的任一列被使用，则设置高位。
    Compare 比较寄存器向量 reg(P1)..reg(P1+P3-1)（称为向量 “A”）与 reg(P2)..reg(P2+P3-1)（“B”）。保存比较结果以供下一条 Jump 指令使用。

    如果 P5 的 OPFLAG_PERMUTE 位被设置，则比较顺序由最近的 Permutation 操作符决定。如果 OPFLAG_PERMUTE 位清除，则按顺序比较寄存器。

    P4 是一个 KeyInfo 结构，定义了用于比较的排序序列和排序顺序。排列仅适用于寄存器。KeyInfo 元素按顺序使用。

    该比较是排序比较，因此 NULL 视为相等，NULL 小于数字，数字小于字符串，字符串小于 blobs。

    此操作码必须紧接在 Jump 操作码之后执行。

    Concat 将寄存器 P1 中的文本追加到寄存器 P2 的文本末尾，并将结果存入寄存器 P3。如果 P1 或 P2 中的文本为 NULL，则在 P3 中存储 NULL。

    P3 = P2 || P1

    P1 和 P3 不能是同一个寄存器。如果 P3 与 P2 相同，某些情况下实现可以避免 memcpy()。

    Copy 将寄存器 P1..P1+P3 的内容复制到寄存器 P2..P2+P3。

    如果 P5 的 0x0002 位设置，则还会清除目标寄存器中的 MEM_Subtype 标志。P5 的 0x0001 位表示该 Copy 操作码不可合并。0x0001 位由查询规划器使用，在查询执行过程中不会起作用。

    此指令会对值进行深拷贝。对任何字符串或 blob 常量都会创建副本。另见 SCopy。

    Count 将游标 P1 打开的表或索引中的条目数（整数值）存入寄存器 P2。

    如果 P3==0，则获取精确计数，需访问表的每个 btree 页面。但如果 P3 非零，则返回基于当前游标位置的估算值。

    CreateBtree 如果 P1==0，则在主数据库文件中分配一个新的 b-tree；如果 P1==1，则在 TEMP 数据库文件中分配；如果 P1>1，则在附加数据库中分配。P3 参数必须为 1（BTREE_INTKEY）用于 rowid 表，必须为 2（BTREE_BLOBKEY）用于索引或 WITHOUT ROWID 表。新 b-tree 的根页号存储在寄存器 P2 中。
    CursorHint 向游标 P1 提供提示，使其只需返回满足 P4 中 Expr 的行。P4 表达式中的 TK_REGISTER 项指的是当前寄存器中的值。P4 表达式中的 TK_COLUMN 项指的是游标 P1 指向的 b-tree 中的列。
    CursorLock 锁定游标 P1 指向的 btree，以防其他游标写入该 btree。
    CursorUnlock 解锁游标 P1 指向的 btree，使其他游标可以写入。
    DecrJumpZero 寄存器 P1 中必须保存整数。将 P1 中的值减 1，如果新值正好为 0，则跳转到 P2。
    DeferredSeek P1 是一个打开的索引游标，P3 是对应表的游标。此操作码对 P3 表游标执行延迟查找，使其定位到与 P1 当前行对应的行。

    这是延迟查找。直到使用游标读取记录之前实际上不会发生任何操作。这样，如果没有读取发生，就不会产生不必要的 I/O。

    P4 可以是一个整数数组（类型 P4_INTARRAY），其中每个条目对应 P3 表的列。如果数组条目 a(i) 非零，则从游标 P3 读取第 a(i)-1 列相当于先执行延迟查找再从 P1 读取第 i 列。该信息存储在 P3 中并用于将对 P3 的读取重定向到 P1，从而可能避免对游标 P3 进行查找和读取。

    Delete 删除游标 P1 当前指向的记录。

    如果 P5 参数的 OPFLAG_SAVEPOSITION 位被设置，则游标将保持指向表中的下一条或上一条记录。如果指向下一条记录，则下一条 Next 指令将是无操作。因此，在这种情况下可以在 Next 循环中删除记录。如果 P5 的 OPFLAG_SAVEPOSITION 位清除，则游标将处于未定义状态。

    如果 P5 设置了 OPFLAG_AUXDELETE 位，则表示此删除是与删除表行及其相关索引条目相关的多个删除之一。确切地说，其中一个删除是“主”删除，其他的要么作用于带有 OPFLAG_FORDELETE 的游标，要么被标记为 AUXDELETE。

    如果 P2（注意是 P2 不是 P5）的 OPFLAG_NCHANGE（0x01）标志被设置，则行更改计数会增加（否则不增加）。

    如果 P2（不是 P5）的 OPFLAG_ISNOOP（0x40）标志被设置，则执行删除的预更新钩子，但 btree 本身不变。这种情况发生在 Delete 紧接着用相同键的 Insert，导致 btree 条目被覆盖。

    P1 不能是伪表。它必须是具有多行的真实表。

    如果 P4 不为 NULL，则它指向 Table 对象。在这种情况下可能会调用更新或预更新钩子或两者。P1 游标必须在调用此操作码之前通过 NotFound 定位。具体来说，如果配置了预更新钩子且 P4 不为 NULL，则调用预更新钩子。如果配置了更新钩子、P4 不为 NULL，且 P2 中设置了 OPFLAG_NCHANGE，则调用更新钩子。

    如果 P2 中设置了 OPFLAG_ISUPDATE 标志，则 P3 包含保存将被更新的行的 rowid 的内存单元的地址。

    Destroy 删除数据库文件中以 P1 为根页的整个表或索引。

    如果 P3==0，则被销毁的表位于主数据库文件。如果 P3==1，则被销毁的表位于用于存储 CREATE TEMPORARY TABLE 创建的表的辅助数据库文件中。

    如果启用了 AUTOVACUUM，则可能会将另一个根页移动到新删除的根页中，以便保持所有根页在数据库开头连续。移动前的根页号（即移动发生前的值）存储在寄存器 P2 中。如果不需要移动（因为被删除的表已经是数据库中的最后一个），则在寄存器 P2 中存储 0。如果禁用了 AUTOVACUUM，则寄存器 P2 中也存储 0。

    如果存在任何活动的读 VM，此操作码会抛出错误。这样做是为了避免在 AUTOVACUUM 数据库中移动根页时更新现有游标所带来的困难。即使数据库不是 AUTOVACUUM 模式，也会抛出此错误，以避免在 autovacuum 与非 autovacuum 模式之间引入不兼容性。

    另见：Clear

    Divide 将寄存器 P1 中的值除以寄存器 P2 中的值，并将结果存入寄存器 P3（P3=P2/P1）。如果寄存器 P1 的值为 0，则结果为 NULL。如果任一输入为 NULL，则结果为 NULL。
    DropIndex 移除描述数据库 P1 中名为 P4 的索引的内部（内存）数据结构。该操作在磁盘上通过 Destroy 操作码删除索引后执行，以使内部架构表示与磁盘上的内容保持一致。
    DropTable 移除描述数据库 P1 中名为 P4 的表的内部（内存）数据结构。该操作在磁盘上通过 Destroy 操作码删除表后执行，以使内部架构表示与磁盘上的内容保持一致。
    DropTrigger 移除描述数据库 P1 中名为 P4 的触发器的内部（内存）数据结构。该操作在磁盘上通过 Destroy 操作码删除触发器后执行，以使内部架构表示与磁盘上的内容保持一致。
    ElseEq 此操作码必须跟在 Lt 或 Gt 比较操作符之后。中间可以有零个或多个 OP_ReleaseReg 操作码，但在此指令与之前的 Lt 或 Gt 之间不允许出现其他操作码。

    如果对与之前 Lt 或 Gt 相同的两个操作数执行 Eq 比较的结果为真，则跳转到 P2。如果结果为假或 NULL，则顺序执行。

    EndCoroutine 寄存器 P1 中地址处的指令是一个 Yield。跳转到该 Yield 的 P2 参数。跳转后，寄存器 P1 中存留的值使得后续的 OP_Yields 会返回到同一个 EndCoroutine 指令。

    另见：InitCoroutine

    Eq 比较寄存器 P1 和 P3 中的值。如果 reg(P3)==reg(P1)，则跳转到地址 P2。

    P5 的 SQLITE_AFF_MASK 部分必须是一个压缩字符——例如 SQLITE_AFF_TEXT、SQLITE_AFF_INTEGER 等。在进行比较前，会尝试根据该压缩将两个输入值强制转换。如果 SQLITE_AFF_MASK 为 0x00，则使用数字压缩。请注意，压缩转换会写回输入寄存器 P1 和 P3，因此此操作码可能会对寄存器 P1 和 P3 造成持久更改。

    完成转换后，如果两个值都不为 NULL，则进行比较。如果两个值都是 blob，则使用 memcmp() 确定结果。如果两个值都是文本，则使用 P4 指定的适当排序函数进行比较。如果未指定 P4，则使用 memcmp() 比较文本字符串。如果两个值都是数字，则使用数字比较。如果两值类型不同，则数字小于字符串、字符串小于 blob。

    如果在 P5 中设置 SQLITE_NULLEQ，则比较结果始终为真或假，不会为 NULL。如果两个操作数都是 NULL，则比较结果为真。如果任一操作数为 NULL，则结果为假。如果两个操作数都不为 NULL，则结果与省略 P5 中 SQLITE_NULLEQ 标志时相同。

    此操作码保存比较结果，以供新的 Jump 操作码使用。

    Expire 使预编译语句过期。当使用 sqlite3_step() 执行已过期的语句时，它要么自动重新准备（如果最初是使用 sqlite3_prepare_v2() 创建的），要么因 SQLITE_SCHEMA 而失败。

    如果 P1 为 0，则全部 SQL 语句过期。如果 P1 非零，则仅当前正在执行的语句过期。

    如果 P2 为 0，则立即使 SQL 语句过期。如果 P2 为 1，则允许正在运行的 SQL 语句继续执行完毕。当执行 CREATE INDEX 或类似的架构更改时，可能会提高语句的运行速度，但不会影响操作的正确性，此时会用到 P2==1 的情况。

    Explain 在正常查询执行期间，此操作码与 Noop 相同。此操作码的目的是保存有关查询计划的信息，以便生成 EXPLAIN QUERY PLAN 输出。

    P4 值是描述查询计划元素的人类可读文本。例如 “SCAN t1” 或 “SEARCH t2 USING INDEX t2x1”。

    P1 值是当前元素的 ID，P2 是用于嵌套查询计划元素时的父元素。如果 P2 为零，则该元素为顶层元素。

    对于循环元素，P3 是每次调用该元素时估计的代码行数。

    如同所有操作码一样，Explain 参数的含义可能随每个发行版发生变化。应用程序不应尝试解释或使用 Explain 操作码中包含的任何信息。该操作码提供的信息仅用于测试和调试。

    Filter 对以 r[P3] 为起点的 P4 寄存器中包含的键计算哈希。检查该哈希是否存在于寄存器 P1 托管的布隆过滤器中。如果不存在，可能跳转到 P2。否则顺序执行。

    误报为假是无害的。即使该值在布隆过滤器中，依然可以安全地顺序执行。误报为假会导致更多的 CPU 周期消耗，但仍应产生正确答案。然而，如果误报为真——即跳转发生而本应顺序执行——可能会得到错误结果。

    FilterAdd 对以 r[P3] 为起点的 P4 寄存器计算哈希，并将该哈希添加到 r[P1] 中包含的布隆过滤器。
    FinishSeek 如果游标 P1 先前通过 DeferredSeek 移动，则立即完成该查找操作。如果查找已完成，则此指令为无操作。
    FkCheck 如果存在任何未解决的外键约束违规，则以 SQLITE_CONSTRAINT 错误终止。如果没有外键约束违规，则此指令为无操作。

    准备语句退出时也会检查外键约束违规。此操作码用于在返回结果（如行更改计数或 RETURNING 子句结果）之前引发外键约束错误。

    FkCounter 将“约束计数器”增加 P2（P2 可为负或正）。如果 P1 非零，则增加数据库约束计数器（延迟外键约束）。否则，如果 P1 为零，则增加语句计数器（立即外键约束）。
    FkIfZero 此操作码测试外键约束计数器是否为零。如果是，则跳转到指令 P2。否则顺序执行下一条指令。

    如果 P1 非零，则当数据库约束计数器（统计延迟约束违规的那一个）为零时执行跳转。如果 P1 为零，则当语句约束计数器为零时执行跳转（立即约束违规）。

    Found 如果 P4==0，则寄存器 P3 包含由 MakeRecord 构造的 blob。如果 P4>0，则寄存器 P3 是构成一个解包记录的 P4 个寄存器中的第一个。

    游标 P1 位于索引 btree 上。如果由 P3 和 P4 标识的记录是 P1 中任一条目 的前缀，则跳转到 P2，并且 P1 保持指向匹配条目。

    该操作使游标保持在可以向前移动的状态。Next 指令仍然可用，但 Prev 指令不可用。

    另见：NotFound、NoConflict、NotExists。SeekGe

    Function 调用用户函数（P4 是指向 sqlite3_context 对象的指针，该对象包含指向要运行的函数的指针），其参数来自寄存器 P2 及其后继寄存器。参数数量由 P4 所指向的 sqlite3_context 对象包含。函数的结果存储在寄存器 P3 中。寄存器 P3 不能是函数输入之一。

    P1 是一个 32 位位掩码，用于指示函数的每个参数在编译时是否被判定为常量。如果第一个参数是常量，则将 P1 的第 0 位置位。此信息用于确定是否可以安全地将使用 sqlite3_set_auxdata() API 与用户函数参数关联的元数据保留到下一次执行此操作码为止。

    See also: AggStep, AggFinal, PureFunc

    Ge 此操作与 Lt 操作码完全相同，唯一区别是当寄存器 P3 的内容大于或等于寄存器 P1 的内容时才会执行跳转。有关更多信息，请参见 Lt 操作码。
    GetSubtype 从寄存器 P1 中提取子类型值，并将该子类型写入寄存器 P2。如果 P1 没有子类型，则 P1 得到 NULL。
    Gosub 将当前地址写入寄存器 P1，然后跳转到地址 P2。
    Goto 无条件跳转到地址 P2。接下来要执行的指令是程序开头起第 P2 个指令。

    P1 参数实际上并未被此操作码使用。不过，有时它会被置为 1 而不是 0，以提示命令行 shell，此 Goto 是一个循环的底部，并且从 P2 到当前行的行应该在 EXPLAIN 输出中缩进。

    Gt 此操作与 Lt 操作码完全相同，唯一区别是当寄存器 P3 的内容大于寄存器 P1 的内容时才会执行跳转。有关更多信息，请参见 Lt 操作码。
    Halt 立即退出。所有打开的游标等都会自动关闭。

    P1 是 sqlite3_exec()、sqlite3_reset() 或 sqlite3_finalize() 返回的结果码。对于正常的 halt，应为 SQLITE_OK (0)。出现错误时，它可以是其他值。如果 P1!=0，则 P2 决定是否回滚当前事务。如果 P2==OE_Fail，则不要回滚。如果 P2==OE_Rollback，则执行回滚。如果 P2==OE_Abort，则撤销此 VDBE 执行期间发生的所有更改，但不要回滚事务。

    如果 P3 不为零且 P4 为 NULL，则 P3 是一个保存错误消息文本的寄存器。

    如果 P3 为零且 P4 不为 NULL，则错误消息字符串保存在 P4 中。

    如果 P5 的取值在 1 到 4（含）之间，则按如下方式修改 P4 错误消息字符串：

    1: NOT NULL constraint failed: P4 2: UNIQUE constraint failed: P4 3: CHECK constraint failed: P4 4: FOREIGN KEY constraint failed: P4

    如果 P3 为零且 P5 不为零且 P4 为 NULL，则省略 “:” 之后的所有内容。

    每个程序的末尾都会隐含插入一个 "Halt 0 0 0" 指令。因此跳过程序的最后一条指令与执行 Halt 相同。

    HaltIfNull 检查寄存器 P3 的值。如果为 NULL，则使用参数 P1、P2 和 P4 按照 Halt 指令的方式执行 Halt。如果寄存器 P3 中的值不是 NULL，则此例程不执行任何操作。P5 参数应为 1。
    IdxDelete 从寄存器 P2 开始的 P3 个寄存器的内容构成一个未打包的索引键。此操作码从由游标 P1 打开索引中删除该条目。

    如果 P5 不为零，则如果未找到匹配的索引条目，则引发 SQLITE_CORRUPT_INDEX 错误。这会在运行 UPDATE 或 DELETE 语句时发生，当要更新或删除的索引条目未找到。对于某些 IdxDelete 的用途（例如：EXCEPT 运算符），未找到匹配条目并没有关系。对于这些情况，P5 为零。此外，如果处于 writable_schema 模式，也不要引发此（自我修正且非关键）错误。

    IdxGT 从 P3 开始的 P4 寄存器值构成一个省略 PRIMARY KEY 的未打包索引键。使用 Compare 将此键与 P1 当前指向的索引进行比较，忽略末尾的 PRIMARY KEY 或 ROWID 字段。

    IdxInsert 寄存器 P2 保存使用 MakeRecord 指令生成的 SQL 索引键。此操作码将该键写入索引 P1。该条目的数据为 nil。

    IdxLE 从 P3 开始的 P4 寄存器值构成一个省略 PRIMARY KEY 或 ROWID 的未打包索引键。使用 Compare 将此键与 P1 当前指向的索引进行比较，忽略 P1 索引上的 PRIMARY KEY 或 ROWID。

    如果 P1 索引条目小于或等于键值，则跳转到 P2。否则继续执行下一条指令。

    IdxLT 从 P3 开始的 P4 寄存器值构成一个省略 PRIMARY KEY 或 ROWID 的未打包索引键。使用 Compare 将此键与 P1 当前指向的索引进行比较，忽略 P1 索引上的 PRIMARY KEY 或 ROWID。

    如果 P1 索引条目小于键值，则跳转到 P2。否则继续执行下一条指令。

    另见：Found、NotFound、SeekGt、SeekGe、SeekLe

    SeekRowid P1是打开在SQL表B树（带整数键）上的游标索引。如果寄存器P3不包含整数或P1中没有rowid为P3的记录，则立即跳转到P2。或者，如果P2为0，则引发SQLITE_CORRUPT错误。如果P1包含rowid为P3的记录，则保持游标指向该记录并继续执行下一条指令。

    NotExists操作码执行相同操作，但在NotExists中，P3寄存器必须保证包含整数。使用本操作码时，寄存器P3可能不包含整数。

    NotFound操作码对索引B树（具有任意多值键）执行相同操作。

    该操作码会使游标处于无法前后移动的状态。换句话说，在该操作码之后，Next和Prev操作码将无法正常工作。

    另见：Found、NotFound、NoConflict、SeekRowid

    SeekScan 此 opcode 是 SeekGE 的前缀 opcode。换句话说，这个 opcode 必须紧接着 SeekGE 执行。这个约束由 assert() 语句检查。

    这个 opcode 使用随后的 SeekGE 的 P1 到 P4 操作数。下文中，随后的 SeekGE opcode 的操作数记作 SeekOP.P1 到 SeekOP.P4。只有这个 opcode 的 P1、P2 和 P5 操作数也会被使用，分别称为 This.P1、This.P2 和 This.P5。

    这个 opcode 有助于通过避免对 btree 的不必要查找，而改为在 b-tree 上移动到下一行，来优化多列索引中 IN 操作符位于索引后续项的情况。即使省略该 opcode 或它作为无操作，也能得到正确的答案。

    SeekGE.P3 和 SeekGE.P4 操作数标识一个未打包的键，是我们期望游标 SeekGE.P1 指向的条目。将该 SeekGE.P3/P4 所在行称为“目标”。

    如果 SeekGE.P1 游标当前未指向任何有效行，那么该 opcode 为无操作，执行流程直接进入 SeekGE。

    如果 SeekGE.P1 游标指向一个有效行，那么该行可能就是目标行，也可能在目标行的前面不远处，或者在目标行之后。如果游标当前位于目标行之前，那么该 opcode 会尝试通过对游标调用 sqlite3BtreeStep() 1 到 This.P1 次的方式，使其定位到目标行或之后的位置。

    This.P5 参数是一个标志，用来指示如果游标最终指向一个在目标行之后的有效行时该怎么做。如果 This.P5 为假（0），则跳转到 SeekGE.P2。如果 This.P5 为真（非零），则跳转到 This.P2。当 P5==0 时表示在 IN 约束右边没有不等式约束。跳转到 SeekGE.P2 会结束循环。P5!=0 的情况出现于 IN 操作符右侧有不等式约束。在这种情况下，This.P2 要么直接指向 IdxGT 或 IdxGE，要么指向检查循环终止的前置代码。

    该 opcode 可能有如下几种结果：

    1. 如果游标最初没有指向任何有效行，则继续执行随后的 SeekGE opcode。

    2. 如果游标即使在调用 sqlite3BtreeNext() 多达 This.P1 次之后仍然指向目标行之前的某一行，那么仍然会落入 SeekGE。

    3. 如果游标最终指向目标行，无论是因为一开始就在目标行，还是因为一个或多个 sqlite3BtreeNext() 调用将游标移动到了目标行，则跳转到 This.P2。

    4. 如果游标一开始位于目标行之前，且一次 sqlite3BtreeNext() 调用将游标移出索引末尾（这表明目标行在 btree 中肯定不存在），则跳转到 SeekGE.P2，以结束循环。

    5. 如果游标最终指向一个在目标行之后的有效行（表明目标行在 btree 中不存在），则在 This.P5==0 时跳转到 SeekOP.P2，否则在 This.P5>0 时跳转到 This.P2。
    Sequence 为游标 P1 查找下一个可用序列号。将该序列号写入寄存器 P2。执行此指令后，游标上的序列号会增加。
    SequenceTest P1 是一个排序游标。如果序列计数器当前为零，则跳转到 P2。无论是否跳转，都对序列值进行递增。
    SetCookie 将整数值 P3 写入数据库 P1 的编号为 P2 的 cookie。P2==1 是 schema version。P2==2 是 database format。P2==3 是推荐的 pager cache size，依此类推。P1==0 表示主数据库文件，P1==1 表示用于临时表的数据库文件。

    在执行此 opcode 之前必须先启动事务。

    如果 P2 是 SCHEMA_VERSION cookie（编号为 1），则内部 schema version 设置为 P3-P5。"PRAGMA schema_version=N" 语句将 P5 设为 1，因此内部 schema version 会与数据库 schema version 不同，从而导致 schema 重置。

    SetSubtype 将寄存器 P2 的 subtype 值设置为寄存器 P1 中的整数。如果 P1 为 NULL，则清除 P2 的 subtype。
    ShiftLeft 将寄存器 P2 中的整数向左移动由寄存器 P1 中的整数指定的位数。将结果存入寄存器 P3。如果任一输入为 NULL，则结果为 NULL。
    ShiftRight 将寄存器 P2 中的整数向右移动由寄存器 P1 中的整数指定的位数。将结果存入寄存器 P3。如果任一输入为 NULL，则结果为 NULL。
    SoftNull 将寄存器 P1 设置为在 MakeRecord 指令看来为 NULL 的值，但不要释放与该寄存器关联的任何字符串或 blob 内存，这样如果该值是先前通过 SCopy 复制的字符串或 blob，副本仍然有效。
    Sort 这个 opcode 与 Rewind 做的完全相同，只不过它会递增一个用于测试的未记录的全局变量。

    排序是通过将记录写入排序索引，然后在该索引上执行 rewind 并从头到尾播放来完成的。我们使用 Sort opcode 而不是 Rewind 来进行 rewind，以便全局变量会被递增，从而让回归测试能够判断优化器是否正确地优化了排序。

    SorterCompare P1 是一个排序游标。此指令将寄存器 P3 中记录 blob 的前缀与排序游标当前指向的条目的前缀进行比较。只比较 r[P3] 和排序记录的前 P4 个字段。

    如果 P3 或排序器在其重要字段（不包括末尾被忽略的 P4 个字段）中有 NULL，则假定比较结果相等。

    如果两条记录比较相等，则落入下一条指令。若不同，则 Jump 到 P2。

    SorterData 将排序游标 P1 当前的排序数据写入寄存器 P2。然后清除游标 P3 上的列头缓存。

    此 opcode 通常用于将记录从排序器中移动到作为伪表游标源的寄存器中，该伪表游标通过 OpenPseudo 创建。该伪表游标由参数 P3 标识。在此 opcode 中清除 P3 的列缓存，免去了发出单独的 NullRow 指令来清理该缓存。

    SorterInsert 寄存器 P2 保存使用 MakeRecord 指令生成的 SQL 索引键。此 opcode 将该键写入排序器 P1。条目的数据为 nil。
    SorterNext 此 opcode 的工作方式与 Next 相同，只不过 P1 必须是已经调用过 SorterSort opcode 的排序对象。此 opcode 将游标前移到下一个排序记录，或者在没有更多排序记录时跳转到 P2。
    SorterOpen 此 opcode 的工作方式类似于 OpenEphemeral，但它打开的是一个专门用于通过外部归并排序算法对大型表进行排序的临时索引。

    如果参数 P3 非零，则表示排序器可能假定只需考虑每个键的前 P3 个字段即可产生所需结果。

    SorterSort 在所有记录都已插入由 P1 标识的排序器对象后，调用此 opcode 实际执行排序。若没有记录可排序，则 Jump 到 P2。

    此 opcode 是用于排序器对象的 Sort 和 Rewind 的别名。

    SqlExec 执行 P4 字符串中指定的 SQL 语句或语句组。

    P1 参数是一个选项位掩码：

    0x0001 在 P4 中的语句运行时禁用 Auth 和 Trace 回调。

    0x0002 在 P4 中的语句运行时，将 db->nAnalysisLimit 设为 P2。

    String 长度为 P1（字节）的字符串值 P4 存入寄存器 P2。

    如果 P3 非零且寄存器 P3 的内容等于 P5，则寄存器 P2 的数据类型会被转换为 BLOB。内容仍然是相同的字节序列，只是像进行了 CAST 一样作为 BLOB 来解释。换句话说：

    if( P3!=0 and reg[P3]==P5 ) reg[P2] := CAST(reg[P2] as BLOB)

    String8 P4 指向一个以空字符结尾的 UTF-8 字符串。此 opcode 在第一次执行前会被转换成一个 String opcode。在转换过程中，P4 字符串的长度会被计算并存入 P1 参数。
    Subtract 从寄存器 P2 中的值减去寄存器 P1 中的值，并将结果存入寄存器 P3。如果任一输入为 NULL，则结果为 NULL。
    TableLock 获取某个表的锁。此指令仅在启用共享缓存功能时使用。

    P1 是 sqlite3.aDb[] 中所获取锁的数据库的索引。如果 P3==0，则获取读锁；如果 P3==1，则获取写锁。

    P2 包含要锁定的表的 root-page。

    P4 包含正在被锁定的表名的指针。只有在无法获取锁时才会用来生成错误信息。

    Trace 如果启用了语句跟踪，则在语句跟踪输出上写入 P4。

    操作数 P1 必须是 0x7fffffff，且 P2 必须为正。

    Transaction 如果数据库 P1 上还没有活动事务，则启动一个事务。如果 P2 非零，则启动写事务，或者如果已有读事务，则将其升级为写事务。如果 P2 为零，则启动读事务。如果 P2 大于等于 2，则启动独占事务。

    P1 是所启动事务的数据库文件的索引。索引 0 是主数据库文件，索引 1 是用于临时表的文件。索引 2 及以上用于附加数据库。

    如果启动了写事务且 Vdbe.usesStmtJournal 标志为真（当 Vdbe 可能修改多行并可能抛出 ABORT 异常时设置此标志），则可能还会打开一个语句事务。更具体地说，只有在数据库连接当前不处于自动提交模式，或者有其他活动语句时才会打开语句事务。语句事务允许在发生错误后只回滚该 VDBE 所做的更改，而不必回滚整个事务。如果没有遇到错误，当 VDBE 终止时语句事务会自动提交。

    如果 P5!=0，则此 opcode 还会将 schema cookie 与 P3 以及 schema generation counter 与 P4 进行比较。每当数据库 schema 发生变化时，cookie 值都会改变。此操作用于检测 cookie 是否已改变以及当前进程是否需要重新读取 schema。如果 P3 中的 schema cookie 与数据库头部中的 schema cookie 不同，或者 P4 中的 schema generation counter 与当前 generation counter 不同，则会引发 SQLITE_SCHEMA 错误并终止执行。sqlite3_step() 封装函数随后可能会重新准备语句并从头重新执行。

    TypeCheck 对以 P1 起始的 P2 个寄存器应用亲和性。亲和性来自 P4 中的 Table 对象。如果某个值无法强制转换为正确类型，则引发错误。

    如果 P3==0，则省略对 VIRTUAL 列的检查。

    如果 P3==1，则省略对所有生成列（包括 VIRTUAL 和 STORED）的检查。

    如果 P3>=2，则仅将表中编号为 P3-2 的列（这是一个 VIRTUAL 列）与 reg[P1] 中的值进行比较。在这种情况下，P2 将是 1。

    此 opcode 类似于 Affinity，但它强制将寄存器类型设置为 Table 列的类型。用于实现“严格亲和性”。

    只有当 P3 为零时才会检查 GENERATED ALWAYS AS ... STATIC 列。P3 非零时，不会对静态生成列执行类型检查。Virtual 列是在查询时计算的，因此永远不会被检查。

    前提条件：

    • P2 应该是 P4 表中非虚拟列的数量，除非 P3>1，在这种情况下 P2 为 1。
    • 表 P4 是 STRICT 表。

    如果任何前提条件不成立，则会发生断言错误。

    Vacuum 对整个数据库 P1 执行 Vacuum。P1 为 0 表示 “main”，2 或更大表示附加数据库。不能对 “temp” 数据库执行 vacuum。

    如果 P2 非零，则它是一个寄存器，保存一个字符串，表示 vacuum 结果应该写入的文件。当 P2 为零时，vacuum 会覆盖原始数据库。

    Variable 将绑定参数 P1 的值传输到寄存器 P2。
    VBegin P4 可能是指向 sqlite3_vtab 结构的指针。如果是，则调用该表的 xBegin 方法。

    此外，无论 P4 是否设置，都需检查该调用是否来自虚拟表 xSync() 方法的回调中。如果是，则错误代码将被设置为 SQLITE_LOCKED。

    VCheck P4 是指向支持 xIntegrity() 方法的 schema P1 中虚拟表的 Table 对象的指针。此 opcode 调用该虚拟表的 xIntegrity() 方法，使用 P3 作为整数参数。如果返回错误，则将表名前置到错误消息中，并将该消息存入 P2。如果未见错误，则将寄存器 P2 设为 NULL。
    VColumn 将游标 P1 的虚拟表当前行中的第 P2 列的值存入寄存器 P3。

    如果在执行 UPDATE 操作时期望通过 VColumn opcode 获取一个不会改变的列的值，则 P5 的值为 OPFLAG_NOCHNG。这会导致 sqlite3_vtab_nochange() 函数在虚拟表实现的 xColumn 方法内部返回 true。P5 列还可能包含其他位（OPFLAG_LENGTHARG 或 OPFLAG_TYPEOFARG），但这些位在 VColumn 中未使用。

    VCreate P2 是一个寄存器，保存数据库 P1 中虚拟表的名称。调用该表的 xCreate 方法。
    VDestroy P4 是数据库 P1 中虚拟表的名称。调用该表的 xDestroy 方法。
    VFilter P1 是使用 VOpen 打开的游标。若筛选后的结果集为空，则跳转到 P2。

    P4 要么为 NULL，要么为模块的 xBestIndex 方法生成的字符串。P4 字符串的解释由模块实现决定。

    此 opcode 调用由 P1 指定的虚拟表的 xFilter 方法。传递给 xFilter 的整数查询计划参数存储在寄存器 P3 中。寄存器 P3+1 保存传递给 xFilter 的 argc 参数。寄存器 P3+2..P3+1+argc 是传递给 xFilter 作为 argv 的 argc 个附加参数。寄存器 P3+2 在传递给 xFilter 时成为 argv[0]。

    如果过滤后的结果集为空，则跳转到 P2。

    VInitIn 将寄存器 P2 设置为指向游标 P1 的 ValueList 对象，缓存寄存器为 P3，输出寄存器为 P3+1。此 ValueList 对象可以作为 sqlite3_vtab_in_first() 和 sqlite3_vtab_in_next() 的第一个参数，用来提取 P1 游标中存储的所有值。寄存器 P3 用于保存 sqlite3_vtab_in_first() 和 sqlite3_vtab_in_next() 返回的值。
    VNext 将虚拟表 P1 移动到其结果集中的下一行，并跳转到指令 P2。或者，如果虚拟表已经达到结果集末尾，则继续执行下一条指令。
    VOpen P4 是指向虚拟表对象 sqlite3_vtab 结构的指针。P1 是游标编号。此 opcode 打开一个指向该虚拟表的游标，并将该游标存入 P1。
    VRename P4 是指向虚拟表对象 sqlite3_vtab 结构的指针。此 opcode 调用相应的 xRename 方法。寄存器 P1 中的值作为 zName 参数传递给 xRename 方法。
    VUpdate P4 是指向虚拟表对象 sqlite3_vtab 结构的指针。此 opcode 调用相应的 xUpdate 方法。P2 个值是从 P3 起连续的内存单元，用于传递给 xUpdate 调用。寄存器 (P3+P2-1) 中的值对应于传递给 xUpdate 的 argv 数组的第 p2 个元素。

    xUpdate 方法将执行 DELETE、INSERT，或两者兼而有之。argv[0] 元素（对应内存单元 P3）是要删除的行的 rowid。如果 argv[0] 为 NULL，则不执行删除。argv[1] 元素是新行的 rowid。它可以为 NULL，让虚拟表自行选择新的 rowid。数组的后续元素是新行中各列的值。

    如果 P2==1，则不执行插入。argv[0] 是要删除的行的 rowid。

    P1 是一个布尔标志。如果它为真且 xUpdate 调用成功，则 sqlite3_last_insert_rowid() 返回的值被设置为刚刚插入行的 rowid。

    P5 是在插入或更新时发生约束失败时应应用的错误处理操作（OE_Replace、OE_Fail、OE_Ignore 等）。

    Yield 将程序计数器与寄存器 P1 中的值互换。此操作的效果是让出给协程。

    如果该指令启动的协程以 Yield 或 Return 结束，则继续执行下一条指令。但如果该协程以 EndCoroutine 结束，则跳转到 P2，而不是继续执行下一条指令。

    另请参见：InitCoroutine

    ZeroOrNull 如果寄存器 P1 和 P3 均非 NULL，则在寄存器 P2 中存入 0。如果寄存器 P1 或 P3 中任意一个为 NULL，则在寄存器 P2 中存入 NULL。
