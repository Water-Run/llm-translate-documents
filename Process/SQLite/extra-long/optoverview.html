1. 引言

本文件概述了 SQLite 的查询规划器和优化器的工作原理。

对于单个 SQL 语句，根据语句自身和底层数据库模式的复杂性，可能存在数十、数百甚至数千种实现方式。 查询规划器的任务是选择能够最小化磁盘 I/O 和 CPU 开销的算法。

有关更多背景信息，请参阅 索引教程 文档。 下一代查询规划器 文档提供了更多关于 如何选择 连接顺序 的细节。

2. WHERE 子句分析

在分析之前，会进行以下转换，将所有连接约束移入 WHERE 子句：

  • 所有 NATURAL 连接都转换为带 USING 子句的连接。
  • 所有 USING 子句（包括前一步中创建的）都转换为等价的 ON 子句。
  • 所有 ON 子句（包括前一步创建的）都作为新的合取项（AND 连接的项）添加到 WHERE 子句中。

SQLite 不区分发生在 WHERE 子句中的连接约束和发生在 INNER JOIN 的 ON 子句中的约束，因为这种区分不会影响结果。 然而，对于 OUTER JOIN 来说，ON 子句约束与 WHERE 子句约束之间存在差别。因此，当 SQLite 将来自 OUTER JOIN 的 ON 子句约束移到 WHERE 子句时， 它会在抽象语法树（AST）中添加特殊标记，以指示该约束来自某个 OUTER JOIN 以及来自哪个 OUTER JOIN。 在纯 SQL 文本中无法添加这些标记。因此，SQL 输入必须在 OUTER JOIN 上使用 ON 子句。 但在内部 AST 中，所有约束都属于 WHERE 子句，因为将所有内容集中在一个位置简化了处理。

在所有约束都移入 WHERE 子句之后，WHERE 子句被分解为合取项（以下简称“项”）。 换言之，WHERE 子句被 AND 运算符分隔开的多个片段分解。 如果 WHERE 子句由 OR 运算符连接的约束（析取）构成，那么整个子句被视为单个“项”，并应用 OR 子句优化。

会分析 WHERE 子句的所有项，看它们是否可以使用索引。 要使某一项可被索引使用，该项通常必须符合下列形式之一： 


  列 = 表达式
  列 IS 表达式
  列 > 表达式
  列 >= 表达式
  列 < 表达式
  列 <= 表达式
  表达式 = 列
  表达式 IS 列
  表达式 > 列
  表达式 >= 列
  表达式 < 列
  表达式 <= 列
  列 IN (表达式列表)
  列 IN (子查询)
  列 IS NULL
  列 LIKE 模式
  列 GLOB 模式

如果通过如下语句创建索引： 

CREATE INDEX idx_ex1 ON ex1(a,b,c,d,e,...,y,z);

如果索引的初始列（列 a、b 等）出现在 WHERE 子句项中，就可能使用该索引。 索引的初始列必须使用 =、IN 或 IS 运算符。 最右侧使用的列可以使用不等式。 对于被使用的索引的最右侧列，最多可以存在两个不等式，用于将列的允许值夹在两个极值之间。

并非所有索引列都必须出现在 WHERE 子句项中，索引仍然可以被使用。 然而，使用的索引列之间不能有间隙。 因此，以上示例索引中，如果没有约束列 c 的 WHERE 子句项，那么约束列 a 与 b 的项可以使用该索引， 但约束列 d 到 z 的项就不能使用。 同样地，如果索引列位于仅由不等式约束的列右侧，通常也不会用于索引目的。 （有关例外情况，请参阅下文的 跳扫优化。）

对于 表达式索引， 上述文本中每当出现“列”一词时，可以替换为“已索引的表达式” （即出现在 CREATE INDEX 语句中的表达式的副本）， 一切仍然适用。

2.1. 索引项使用示例

对于以上索引和如下 WHERE 子句： 

... WHERE a=5 AND b IN (1,2,3) AND c IS NULL AND d='hello'

由于索引的前四列 a、b、c 和 d 构成索引的前缀且都由相等约束绑定，因此这四列都是可用的。

对于以上索引和如下 WHERE 子句： 

... WHERE a=5 AND b IN (1,2,3) AND c>12 AND d='hello'

只有索引的列 a、b 和 c 可用。列 d 不可用，因为它位于 c 的右侧，而 c 仅由不等式约束。

对于以上索引和如下 WHERE 子句： 

... WHERE a=5 AND b IN (1,2,3) AND d='hello'

只有索引的列 a 和 b 可用。列 d 不可用，因为列 c 未被约束，而可用索引列之间不能有间隙。

对于以上索引和如下 WHERE 子句： 

... WHERE b IN (1,2,3) AND c NOT NULL AND d='hello'

索引完全不可用，因为索引的最左列（列 "a"）没有被约束。 在假设没有其它索引的情况下，上述查询将导致全表扫描。

对于以上索引和如下 WHERE 子句： 

... WHERE a=5 OR b IN (1,2,3) OR c NOT NULL OR d='hello'

由于 WHERE 子句项由 OR 连接而不是 AND 连接，此时索引不可用。 该查询将导致全表扫描。 然而，如果再增加三个索引，使得 b、c、d 分别成为各自索引的最左列，则可能适用 OR 子句优化。

3. BETWEEN 优化

如果某个 WHERE 子句项符合以下形式： 


  expr1 BETWEEN expr2 AND expr3

则会添加两个“虚拟”项如下： 


  expr1 >= expr2 AND expr1 <= expr3

虚拟项仅用于分析，不会生成任何字节码。 如果两个虚拟项都最终作为索引约束被使用，则原始的 BETWEEN 项会被省略， 相应的测试也不会在输入行上执行。 因此，如果 BETWEEN 项最终被用作索引约束，则该项永远不会在输入行上进行测试。 另一方面，虚拟项本身也不会导致在输入行上执行测试。 因此，如果 BETWEEN 项未被用作索引约束，而必须用于测试输入行，则表达式 expr1 只会被评估一次。

4. OR 优化

WHERE 子句中由 OR 而非 AND 连接的约束可以通过两种不同的方式处理。

4.1. 将 OR 连接的约束转换为 IN 运算符

如果某一项由多个包含公共列名并由 OR 分隔的子项组成，如下所示： 


  列 = expr1 OR 列 = expr2 OR 列 = expr3 OR ...

则该项会被重写为： 


  列 IN (expr1,expr2,expr3,...)

重写后的项会根据 IN 运算符的常规规则继续对索引造成约束。 注意，在每个 OR 连接的子项中，列 必须是相同的列，尽管该列可以位于 = 运算符的左侧或右侧。

4.2. 分别评估 OR 约束并取结果的 UNION

仅当前述的 OR 到 IN 的转换不奏效时，才会尝试第二种 OR 子句优化。 假设 OR 子句由如下多个子项构成： 


  expr1 OR expr2 OR expr3

单个子项可以是诸如 a=5 或 x>y 的比较表达式，或 LIKE、BETWEEN 表达式， 也可以是一个由 AND 连接的子子项构成的带括号的列表。 每个子项都会被分析，就好像它本身就是整个 WHERE 子句，以验证该子项是否独立可索引。 如果一个 OR 子句的每个子项都能单独使用索引，那么可能会对该 OR 子句进行编码， 使得为 OR 子句的每个项都使用不同的索引。 想象一下 SQLite 如何为 OR 子句的每项分别使用索引，就好像 WHERE 子句被如下重写： 


  rowid IN (SELECT rowid FROM 表 WHERE expr1
            UNION SELECT rowid FROM 表 WHERE expr2
            UNION SELECT rowid FROM 表 WHERE expr3)

上述重写只是概念性的；实际的 WHERE 子句并不会真的这样重写。 OR 子句的实际实现使用的是一种更高效的方法，并且即使针对 WITHOUT ROWID 表 或“rowid”不可访问的表也能奏效。 然而，上述语句很好地概括了其实现要点：对每个 OR 子句项分别使用不同的索引去查找候选结果行， 最终结果就是这些行的并集。

请注意，在大多数情况下，SQLite 只会为查询的 FROM 子句中的每个表使用一个索引。 此处描述的第二种 OR 子句优化是该规则的例外。 在 OR 子句中，每个子项可能使用不同的索引。

对于任意给定的查询，虽然可以尝试使用此处描述的 OR 子句优化，但并不保证最终会使用它。 SQLite 使用基于成本的查询规划器，估算各种竞选查询计划的 CPU 和磁盘 I/O 成本，并选择它认为最快的计划。 如果 WHERE 子句中存在许多 OR 项，或者某些 OR 子句子项上可用的索引不够具备选择性， SQLite 可能会决定使用另一种查询算法，甚至彻底进行全表扫描。 应用程序开发人员可以在语句前加上 EXPLAIN QUERY PLAN 前缀，以获取所选查询策略的高级概览。

5. LIKE 优化

使用 LIKE 或 GLOB 运算符的 WHERE 子句项有时可以借助索引进行范围搜索， 几乎就像 LIKE 或 GLOB 是 BETWEEN 运算符的替代方案。 该优化有许多条件：

  1. LIKE 或 GLOB 的右侧必须是字符串字面量，或 是绑定到不以通配符开头的字符串字面量的 参数。
  2. 必须无法通过在左侧使用数值（而非字符串或 BLOB）使 LIKE 或 GLOB 运算符为真。 这意味着要么：
    1. LIKE 或 GLOB 运算符的左侧是具有 TEXT 亲和性 的已索引列名，或
    2. 右侧模式参数不以减号（"-"）或数字开头。
    该约束来源于数字无法按字典顺序排列。 例如：9<10 但 '9'>'10'。
  3. 用于实现 LIKE 和 GLOB 的内置函数必须未被 sqlite3_create_function() API 重载。
  4. 对于 GLOB 运算符，列必须使用内置的 BINARY 排序序列进行索引。
  5. 对于 LIKE 运算符，如果启用了 case_sensitive_like 模式， 则列必须使用内置的 BINARY 排序序列进行索引； 如果禁用了该模式，则该列必须使用内置的 NOCASE 排序序列进行索引。
  6. 如果使用了 ESCAPE 选项，则 ESCAPE 字符必须是 ASCII 字符或 UTF-8 中的单字节字符。

LIKE 运算符有两种模式，可以通过 pragma 设置。 默认模式下，LIKE 比较对 latin1 字符的大小写差异不敏感。 因此，默认情况下，下述表达式为真： 

'a' LIKE 'A'

如果启用了 case_sensitive_like pragma，如下： 

PRAGMA case_sensitive_like=ON;

则 LIKE 运算符会区分大小写，上述示例将评估为 false。 注意，大小写不敏感仅适用于 latin1 字符——基本上是 ASCII 前 127 个字节码中的英文大小写字母。 除非提供了考虑非 ASCII 字符的应用定义 排序序列 和 like() SQL 函数，否则 SQLite 中的国际字符集是区分大小写的。 如果提供了应用定义的排序序列和/或 like() SQL 函数，则上述 LIKE 优化将永远不会被采用。

LIKE 运算符默认大小写不敏感，因为 SQL 标准要求如此。 你可以在编译时使用编译器的 SQLITE_CASE_SENSITIVE_LIKE 命令行选项 更改默认行为。

如果 LIKE 或 GLOB 运算符左侧的列使用内置的 BINARY 排序序列进行索引，并且 case_sensitive_like 被启用， 则可能会采用 LIKE 优化。 如果该列使用内置的 NOCASE 排序序列进行索引，并且 case_sensitive_like 模式被禁用，也可能采纳此优化。 这两种组合是 LIKE 运算符会被优化的唯一情况。

GLOB 运算符始终区分大小写。 GLOB 运算符左侧的列必须始终使用内置的 BINARY 排序序列，否则不会尝试对该运算符进行索引优化。

只有当 GLOB 或 LIKE 运算符的右侧是字面字符串或已经 绑定 到字符串字面量的参数时， 才会尝试 LIKE 优化。 字符串字面量不得以通配符开头；若右侧以通配符字符开头，则不会尝试此优化。 如果右侧是绑定到字符串的参数， 仅在包含该表达式的预处理语句使用 sqlite3_prepare_v2() 或 sqlite3_prepare16_v2() 编译时才会尝试该优化。 如果右侧是参数，且语句使用 sqlite3_prepare() 或 sqlite3_prepare16() 预编译， 则不会尝试 LIKE 优化。

假设 LIKE 或 GLOB 运算符右侧的非通配符字符的初始序列为 x。 这里使用单个字符表示这个非通配符前缀，但读者应理解前缀可以由多字符组成。 令 y 为与 x 长度相同且比 x 大的最小字符串。 例如，如果 x 是 'hello'， 则 y 是 'hellp'。 LIKE 和 GLOB 优化就是添加如下两个虚拟项： 


  列 >= x AND 列 < y

在大多数情况下，即使虚拟项用于对索引进行约束，原始的 LIKE 或 GLOB 运算符仍会针对每一行输入进行测试。 这是因为我们无法知道在 x 前缀右侧可能施加的额外约束。 然而，如果 x 右侧只有一个全局通配符，则原始的 LIKE 或 GLOB 测试会被禁用。 换句话说，如果模式如下： 


  列 LIKE x%
  列 GLOB x*

若虚拟项约束了索引，则原始的 LIKE 或 GLOB 测试将被禁用， 因为在这种情况下我们知道索引选出的所有行都会通过 LIKE 或 GLOB 测试。

请注意，当 LIKE 或 GLOB 运算符的右侧是 参数，并且语句使用 sqlite3_prepare_v2() 或 sqlite3_prepare16_v2() 预编译时， 如果右侧参数的绑定值与上一次运行不同，则在每次运行的首次 sqlite3_step() 调用时，该语句会自动重新解析并重新编译。 此重新解析与重新编译实质上与模式变更后发生的操作相同。 重新编译是必要的，以便查询规划器能够检查绑定到 LIKE 或 GLOB 运算符右侧的新值，并确定是否采用上述优化。

6. 跳扫优化

通用规则是，只有在 WHERE 子句约束索引的最左列时，索引才有用。 然而，在某些情况下，即使索引的最前几列未出现在 WHERE 子句中，SQLite 仍能使用该索引，只要之后的列被包含在内。

例如考虑如下表： 

CREATE TABLE people(
  name TEXT PRIMARY KEY,
  role TEXT NOT NULL,
  height INT NOT NULL, -- in cm
  CHECK( role IN ('student','teacher') )
);
CREATE INDEX people_idx1 ON people(role, height);

people 表中每个人都有一条记录。 每个人的 “role” 字段指出其是 “student” 还是 “teacher”。 表中还记录了每个人的身高（单位厘米）。 role 和 height 是被索引的列。 注意索引的最左列并不是很具备选择性——它只包含两个可能值。

现在考虑一个查询，用于查找组织中身高至少 180cm 的所有人： 

SELECT name FROM people WHERE height>=180;

由于索引的最左列没有出现在查询的 WHERE 子句中，似乎无法使用该索引。 然而，SQLite 实际上能够使用该索引。 从概念上讲，SQLite 会将查询处理得更像如下形式： 

SELECT name FROM people
 WHERE role IN (SELECT DISTINCT role FROM people)
   AND height>=180;

或者： 

SELECT name FROM people WHERE role='teacher' AND height>=180
UNION ALL
SELECT name FROM people WHERE role='student' AND height>=180;

上述替代查询只是概念性的。 SQLite 并不会真的转换查询。 实际的查询计划如下： SQLite 找到 “role” 的第一个可能值，这可以通过将 "people_idx1" 索引倒回到开头并读取第一条记录实现。 SQLite 将这个第一个 “role” 值存储在一个内部变量中，这里称之为 "$role"。 然后 SQLite 运行如下查询：“SELECT name FROM people WHERE role=$role AND height>=180”。 该查询在索引的最左列上具有相等约束，因此该索引可以用于解析该查询。 一旦该查询完成，SQLite 就会使用 "people_idx1" 索引定位 “role” 列的下一个值， 其代码逻辑类似于 “SELECT role FROM people WHERE role>$role LIMIT 1”。 该新的 “role” 值将覆盖 $role 变量，整个过程不断重复，直到所有可能的 “role” 值都被检查一遍。

我们将这种索引用法称为 “跳扫”，因为数据库引擎基本上在执行索引的全扫描，但它通过偶尔跳到下一个候选值来优化扫描（让扫描变得不再是“全”扫描）。

如果 SQLite 知道索引的最左列包含大量重复值，它可能会在该索引上执行跳扫。 如果索引最左列的重复值太少，那么简单地跳到下一个值（即执行全表扫描）会比在索引上执行二分查找更快，从而不会进行跳扫。

SQLite 只有在数据库上运行了 ANALYZE 命令后，才能知道索引最左列有大量重复值。 没有 ANALYZE 的结果，SQLite 只能对表中的数据“形状”做出猜测，默认猜测是索引最左列的每个值平均有 10 个重复。 约当跳扫只有在重复数量达到约 18 个或更多时才会比全表扫描更快。 因此，未经过 ANALYZE 的数据库永远不会使用跳扫。

7. 连接

SQLite 将连接实现为嵌套循环。 嵌套循环在连接中的默认顺序是让 FROM 子句中最左边的表形成外层循环，最右边的表形成内层循环。 然而，如果以不同顺序嵌套循环能帮助它选择更好的索引，SQLite 会采用不同的顺序。

内连接可以随意重新排序。然而外连接既不满足交换律也不满足结合律，因此不会被重新排序。 位于外连接左右两侧的内连接可能会根据优化器的判断被重新排序，但外连接始终按照它们出现的顺序来执行。

SQLite 将 CROSS JOIN 操作符视为特殊情况。 理论上，CROSS JOIN 操作符满足交换律。然而，SQLite 选择永不重新排序 CROSS JOIN 中的表。 这为程序员提供了一种机制，使得 SQLite 能够选择特定的循环嵌套顺序。

在选择连接中各表的顺序时，SQLite 使用在 Next Generation Query Planner 文档中描述的高效多项式时间图算法。 因此，SQLite 能够在微秒级别内规划 50 路或 60 路连接的查询。

连接重新排序是自动完成的，通常效果足够好，程序员无需操心，特别是在使用 ANALYZE 收集了可用索引的统计信息之后，尽管有时仍可能需要程序员提供一些提示。 例如，考虑如下模式： 

CREATE TABLE node(
   id INTEGER PRIMARY KEY,
   name TEXT
);
CREATE INDEX node_idx ON node(name);
CREATE TABLE edge(
   orig INTEGER REFERENCES node,
   dest INTEGER REFERENCES node,
   PRIMARY KEY(orig, dest)
);
CREATE INDEX edge_idx ON edge(dest,orig);

上述模式定义了一个有向图，并可以在每个节点存储一个名称。现在考虑针对该模式的如下查询： 

SELECT *
  FROM edge AS e,
       node AS n1,
       node AS n2
 WHERE n1.name = 'alice'
   AND n2.name = 'bob'
   AND e.orig = n1.id
   AND e.dest = n2.id;

这个查询请求有关从标记为 "alice" 的节点到标记为 "bob" 的节点的边的所有信息。 SQLite 中的查询优化器基本上有两种选择来实现此查询。（实际上有六种不同的选择，但这里仅考虑其中两种。） 以下伪代码演示了这两种选择。

方案 1： 

foreach n1 where n1.name='alice' do:
  foreach n2 where n2.name='bob' do:
    foreach e where e.orig=n1.id and e.dest=n2.id
      return n1.*, n2.*, e.*
    end
  end
end

方案 2： 

foreach n1 where n1.name='alice' do:
  foreach e where e.orig=n1.id do:
    foreach n2 where n2.id=e.dest and n2.name='bob' do:
      return n1.*, n2.*, e.*
    end
  end
end

两种实现方案中的每个循环都使用相同的索引进行加速。 这两个查询计划之间唯一的区别在于循环嵌套的顺序。

那么哪个查询计划更好？事实证明，这取决于 node 和 edge 表中数据的结构。

设 alice 节点数量为 M，bob 节点数量为 N。 考虑两种场景。第一种场景中，M 和 N 都是 2，但每个节点有数千条边。在这种情况下，方案 1 更可取。 在方案 1 中，内层循环检查一对节点之间是否存在边，如果存在则输出结果。 由于只有两个 alice 和两个 bob 节点，内层循环只需运行四次，查询非常快。而方案 2 在这里会花费更长时间。 方案 2 的外层循环仅执行两次，但由于每个 alice 节点有大量出边，中间循环必须迭代数千次。因此会慢得多。 所以在第一种场景中，我们更愿意使用方案 1。

现在考虑 M 和 N 都是 3500 的情况。alice 节点非常多。 这次假设每个节点仅连接一到两条边。现在方案 2 更可取。 在方案 2 中，外层循环仍需运行 3500 次，但中间循环对每个外层循环仅运行一到两次，内层循环也仅在极少数情况下运行一次。 于是内层循环的总迭代次数约为 7000。 而方案 1 的外层和中间循环都会运行 3500 次，使得中间循环的迭代次数达到 1200 万。 因此在第二种场景中，方案 2 比方案 1 快近 2000 倍。

所以可以看出，根据表中数据的结构，查询计划 1 或查询计划 2 中可能有一个更好。 SQLite 默认选择哪个计划？截至版本 3.6.18，在未运行 ANALYZE 的情况下， SQLite 会选择方案 2。 如果运行了 ANALYZE 命令以收集统计信息， 那么如果统计信息表明另一个计划可能更快，就可能会做出不同的选择。

7.1. 手动控制连接顺序

SQLite 几乎总是自动选择最佳的连接顺序。 开发者很少需要干预来给查询规划器提供最佳连接顺序的提示。 最佳策略是使用 PRAGMA optimize，确保查询规划器能够获得数据库中数据形状的最新统计信息。

本节介绍开发人员可以控制 SQLite 中连接顺序的技术，以解决可能出现的性能问题。 但是，除非万不得已，否则不推荐使用这些技术。

如果在运行 PRAGMA optimize 后仍遇到 SQLite 选择了次优的连接顺序的情况， 请将情况报告到 SQLite Community Forum， 以便 SQLite 维护人员对查询规划器作出新的改进，从而不再需要手动干预。

7.1.1. 使用 SQLITE_STAT 表手动控制查询计划

SQLite 为高级程序员提供了控制优化器所选查询计划的能力。 实现这一目标的一种方法是篡改 ANALYZE 结果中 sqlite_stat1 表的数据。

7.1.2. 使用 CROSS JOIN 手动控制查询计划

程序员可以通过在连接中使用 CROSS JOIN 操作符（而不是 JOIN、INNER JOIN、NATURAL JOIN 或 “,” 连接）来强制 SQLite 使用特定的循环嵌套顺序。 虽然理论上 CROSS JOIN 满足交换律，SQLite 却选择永不重新排序 CROSS JOIN 中的表。 因此，CROSS JOIN 的左表总是在相对于右表的外层循环中。

在下面的查询中，优化器可以按照它认为合适的方式重新排序 FROM 子句中的表： 

SELECT *
  FROM node AS n1,
       edge AS e,
       node AS n2
 WHERE n1.name = 'alice'
   AND n2.name = 'bob'
   AND e.orig = n1.id
   AND e.dest = n2.id;

在下面这一逻辑上等价的查询中，将 “,” 替换为 “CROSS JOIN” 表示表的顺序必须是 N1、E、N2。 

SELECT *
  FROM node AS n1 CROSS JOIN
       edge AS e CROSS JOIN
       node AS n2
 WHERE n1.name = 'alice'
   AND n2.name = 'bob'
   AND e.orig = n1.id
   AND e.dest = n2.id;

在后一种查询中，查询计划必须是 方案 2。 请注意，必须使用关键字 “CROSS” 来禁用表重新排序优化；INNER JOIN、NATURAL JOIN、JOIN 和其它类似组合的行为与逗号连接相同，优化器可以按自己的判断重新排序表。（在外连接上也同样禁用了表重排，因为外连接不满足结合律也不满足交换律。重新排序 OUTER JOIN 中的表会改变结果。）

参见同一文档中稍后部分的 "The Fossil NGQP Upgrade Case Study"，了解另一个在实际中使用 CROSS JOIN 手动控制连接嵌套顺序的例子。 同一文档中稍后部分的 查询规划器检查清单 提供了有关手动控制查询规划器的进一步指导。

8. 在多个索引之间做选择

查询的 FROM 子句中的每个表最多只能使用一个索引（OR 子句优化 除外）， 并且 SQLite 力求让每个表至少使用一个索引。 有时，一个表可能有两个或多个索引可以选用。 例如： 

CREATE TABLE ex2(x,y,z);
CREATE INDEX ex2i1 ON ex2(x);
CREATE INDEX ex2i2 ON ex2(y);
SELECT z FROM ex2 WHERE x=5 AND y=6;

对于上面的 SELECT 语句，优化器可以使用 ex2i1 索引查找包含 x=5 的 ex2 行，然后再针对 y=6 条件测试每一行。 或者它可以使用 ex2i2 索引查找包含 y=6 的 ex2 行，然后再针对 x=5 条件测试这些行。

面对两个或多个索引的选择时，SQLite 会尝试估算使用每种选项执行查询所需的总工作量。 然后选择预计工作量最少的选项。

为了帮助优化器更准确地估算使用各种索引所需的工作量，用户可以选择运行 ANALYZE 命令。 ANALYZE 命令扫描数据库中所有可能涉及多个索引选择的索引，并收集这些索引的选择性统计信息。 这些扫描所得的统计信息被存储在以 "sqlite_stat" 为前缀的特殊数据库表中。 这些表的内容不会随着数据库的变化而自动更新，因此在做出重大更改后，重新运行 ANALYZE 可能是明智的。 ANALYZE 命令的结果只对在 ANALYZE 命令完成后打开的数据库连接可用。

各种 sqlite_statN 表包含有关各索引选择性的相关信息。 例如，sqlite_stat1 表可能表示对列 x 的等值约束平均将搜索空间缩小到 10 行，而对列 y 的等值约束则将搜索空间缩小到 3 行。 在这种情况下，SQLite 会偏好使用 ex2i2 索引，因为该索引更加选择性。

8.1. 使用一元 “+” 排除 WHERE 子句项

注意：以这种方式排除 WHERE 子句项并不推荐。 这是一种权宜之计。 只有在绝对必要以获得所需性能时才使用此方法。 如果你发现必须使用此种方法，请在 SQLite Community Forum 报告该情况， 以便 SQLite 维护人员尝试改进查询规划器，使得在你的场景中不再需要此种权宜之计。

可以通过在列名前加一元 + 运算符来手动排除 WHERE 子句项在索引中的使用。 一元 + 是无操作的，并不会在准备好的语句中生成任何字节码。 但是，一元 + 运算符会阻止该项约束索引。 因此，在上述示例中，如果查询被重写为： 

SELECT z FROM ex2 WHERE +x=5 AND y=6;

列 x 上的一元 + 运算符会阻止该项对索引施加约束。 这将迫使使用 ex2i2 索引。

请注意，一元 + 运算符还会从表达式中移除 类型亲和性，在某些情况下这可能会导致表达式含义的细微变化。 在上述示例中， 如果列 x 具有 TEXT 亲和性， 那么比较 “x=5” 将作为文本进行。 + 运算符会移除亲和性。 因此比较 "+x=5" 将把列 x 中的文本与数值 5 进行比较，并且将始终为假。

8.2. 范围查询

再考虑一种略有不同的情况： 

CREATE TABLE ex2(x,y,z);
CREATE INDEX ex2i1 ON ex2(x);
CREATE INDEX ex2i2 ON ex2(y);
SELECT z FROM ex2 WHERE x BETWEEN 1 AND 100 AND y BETWEEN 1 AND 100;

再进一步假设列 x 包含分布在 0 到 1,000,000 之间的值，而列 y 包含分布在 0 到 1,000 之间的值。 在该场景中，x 上的范围约束应该将搜索空间缩小 10,000 倍，而 y 上的范围约束则仅缩小 10 倍。 所以 ex2i1 索引应当被优先使用。

SQLite 会做出这个判断，但前提是它已使用 SQLITE_ENABLE_STAT3 或 SQLITE_ENABLE_STAT4 编译。 SQLITE_ENABLE_STAT3 和 SQLITE_ENABLE_STAT4 选项使 ANALYZE 命令在 sqlite_stat3 或 sqlite_stat4 表中收集列内容的直方图，并利用这些直方图对如上述范围约束所需的最佳查询做出更好的猜测。 STAT3 和 STAT4 之间的主要区别在于：STAT3 仅为索引的最左列记录直方图数据，而 STAT4 会为索引的所有列记录直方图数据。 对于单列索引，STAT3 与 STAT4 的工作方式相同。

直方图数据只有在约束的右侧是简单的编译时常量或 参数 而非表达式时才有用。

直方图数据的另一个限制是，它只适用于索引的最左列。 例如： 

CREATE TABLE ex3(w,x,y,z);
CREATE INDEX ex3i1 ON ex2(w, x);
CREATE INDEX ex3i2 ON ex2(w, y);
SELECT z FROM ex3 WHERE w=5 AND x BETWEEN 1 AND 100 AND y BETWEEN 1 AND 100;

这里的不等式条件位于 x 和 y 列，而它们并不是索引中的最左列。 因此，直方图数据在帮助选择 x 和 y 的范围约束时毫无用处。

9. 覆盖索引

在进行索引查找某一行时，通常的过程是先在索引上执行二分查找以找到索引项，然后从索引中提取 rowid，再利用该 rowid 在原始表上执行二分查找。 因此，一个典型的索引查找涉及两次二分查找。 然而，如果要从表中获取的所有列都已包含在索引本身中，SQLite 会直接使用索引中包含的值，而永远不去查找原始表行。 这样每行就节省了一次二分查找，从而可能使许多查询运行速度提升一倍。

当一个索引包含了查询所需的所有数据，并且永远不需要访问原始表时，我们称该索引为“覆盖索引”。

10. ORDER BY 优化

SQLite 在可能的情况下会尝试使用索引来满足查询的 ORDER BY 子句。 在面临使用索引满足 WHERE 子句约束或满足 ORDER BY 子句的选择时，SQLite 会进行前面所述的相同成本分析，并选择其认为能得到最快响应的索引。

SQLite 也会尝试使用索引协助满足 GROUP BY 子句和 DISTINCT 关键字。 如果连接的嵌套循环可以按照某个 GROUP BY 或 DISTINCT 等价的顺序来排列， 那么 GROUP BY 或 DISTINCT 逻辑只需将当前行与前一行进行比较即可确定当前行是否属于同一组或是否是不同值。 这通常比将每一行与之前所有行比较要快得多。

10.1. 通过索引实现部分 ORDER BY

如果查询包含一个带有多个项的 ORDER BY 子句，SQLite 可能会使用索引使行按照 ORDER BY 的某个前缀项的顺序输出，但 ORDER BY 中的后续项并不能得到满足。 在这种情况下，SQLite 会进行块排序。 假设 ORDER BY 子句中有四项，而查询的自然顺序会使结果按照前两项排序。 每当查询引擎输出一行并将其输入排序器时， 与 ORDER BY 前两项对应的输出会与前一行进行比较。 如果发生变化，则当前排序结束并输出，随后开始新的排序。 这会带来略快的排序性能。更大的优势在于：需要驻留在内存中的行数大大减少，从而降低了内存需求，并且在核心查询运行完成之前就可以开始输出结果。

11. 子查询扁平化

当一个子查询出现在 SELECT 的 FROM 子句中时，最简单的行为是将子查询求值为临时表，然后对该临时表执行外部 SELECT。 这种计划可能是次优的，因为临时表没有任何索引，并且外部查询（可能是一个连接）将被迫要么对临时表执行全表扫描，要么对临时表构造一个 查询时索引，这两种方法都不是特别快。

为了解决这一问题，SQLite 会尝试扁平化 SELECT 的 FROM 子句中的子查询。 这涉及将子查询的 FROM 子句插入外部查询的 FROM 子句，并重写外部查询中引用子查询结果集的表达式。 例如： 

SELECT t1.a, t2.b FROM t2, (SELECT x+y AS a FROM t1 WHERE z<100) WHERE a>5

扁平化查询后会被重写为： 

SELECT t1.x+t1.y AS a, t2.b FROM t2, t1 WHERE z<100 AND a>5

必须满足许多条件才能执行查询扁平化。其中一些约束被斜体标注为过时。 这些额外约束保留在文档中是为了保持其他约束编号的一致性。

普通读者不需要理解所有这些规则。 关键点在于，扁平化规则非常微妙且复杂。 多年来曾有多个 Bug 是由于过度激进的查询扁平化引起的。 另一方面，如果查询扁平化过于谨慎，复杂查询和/或涉及视图的查询的性能往往会受到影响。

12. 子查询协程

SQLite 以三种方式之一实现 FROM-clause 的子查询：

  1. Flatten 子查询到外层查询中
  2. 将子查询评估为一个在正在执行的 SQL 语句期间存在的临时表，然后针对该临时表运行外层查询。
  3. 在与外层查询并行运行的协程中评估子查询，根据需要向外层查询提供行。

本节描述第三种技术：将子查询实现为协程。

协程类似于子程序，因为它在与调用者相同的线程中运行，并最终将控制权返回给调用者。 区别在于协程还可以在未完成时返回，然后在下次调用时从中断处继续。

当子查询以协程的方式实现时，会生成字节码来将子查询实现得像一个独立查询， 只是它不是将结果行返回给应用程序，而是在计算出每一行后将控制权让给调用者。 调用者然后可以将该行用作其计算的一部分，并在准备好获取下一行时再次调用协程。

协程优于将子查询的完整结果集存储在临时表中，因为协程使用更少的内存。 使用协程时只需要记住结果的单行，而临时表必须存储全部行。 此外，由于协程不需要在外层查询开始工作之前运行完毕，因此输出的前几行可以更早出现， 如果整体查询在完成之前被放弃，所做的工作也更少。

另一方面，如果必须多次扫描子查询的结果（例如，它只是连接中的某个表）， 那么最好使用临时表来记住整个子查询的结果，以避免重复计算子查询。

12.1. 在排序之后延迟工作的协程

从 SQLite 版本 3.21.0 (2017-10-24) 起，当外层查询的结果集“复杂”时， 查询优化器总是更倾向于使用协程来实现包含 ORDER BY 子句且不属于 join 的 FROM-clause 子查询。 此功能允许应用程序将昂贵的计算从排序器之前移动到排序器之后，从而可能获得更快的运行速度。 例如，考虑以下查询： 

SELECT expensive_function(a) FROM tab ORDER BY date DESC LIMIT 5;

该查询的目标是在表中为最近的五条记录计算某些值。 在上述查询中，"expensive_function()" 在排序之前被调用，因此在表的每一行上都会调用它，即使某些行最终由于 LIMIT 而被省略。 协程可以用来规避这一点： 

SELECT expensive_function(a) FROM (
  SELECT a FROM tab ORDER BY date DESC LIMIT 5
);

在修改后的查询中，由协程实现的子查询计算了最近的五个 "a" 值。 这五个值传递给外层查询，在那里仅在应用程序关心的特定行上调用 "expensive_function()"。

未来版本的 SQLite 的查询优化器可能会变得足够智能，从而在两个方向上自动完成上述转换。 也就是说，未来版本可能将第一种形式的查询转换为第二种形式，或者将第二种形式写出的查询转换为第一种形式。 从 SQLite 版本 3.22.0 (2018-01-22) 开始，如果外层查询的结果集中没有使用任何用户定义函数或子查询， 查询优化器将会 Flatten 子查询。然而，对于上面展示的示例，SQLite 仍然按原样实现每个查询。

13. MIN/MAX 优化

包含单个 MIN() 或 MAX() 聚合函数且其参数是某个索引的最左侧列的查询 可以通过进行单次索引查找而非扫描整个表来满足。示例： 

SELECT MIN(x) FROM table;
SELECT MAX(x)+1 FROM table;

14. 自动查询时索引

当没有可用索引来帮助评估查询时，SQLite 可能会创建一个仅在单个 SQL 语句期间存在的自动索引。 自动索引有时也称为“查询时索引”。 由于构造自动或查询时索引的成本为 O(NlogN)（其中 N 是表中条目的数量），而进行全表扫描的成本仅为 O(N)， 因此只有当 SQLite 预计在 SQL 语句执行期间查找将被运行超过 logN 次时，才会创建自动索引。 例如： 

CREATE TABLE t1(a,b);
CREATE TABLE t2(c,d);
-- Insert many rows into both t1 and t2
SELECT * FROM t1, t2 WHERE a=c;

在上面的查询中，如果 t1 和 t2 都大约有 N 行，那么在没有任何索引的情况下，查询需要 O(N*N) 时间。 另一方面，在表 t2 上创建索引需要 O(NlogN) 时间，使用该索引评估查询又需要额外的 O(NlogN) 时间。 在缺乏 ANALYZE 信息的情况下，SQLite 猜测 N 为一百万，因此认为构造自动索引是成本更低的方法。

自动查询时索引也可能用于子查询： 

CREATE TABLE t1(a,b);
CREATE TABLE t2(c,d);
-- Insert many rows into both t1 and t2
SELECT a, (SELECT d FROM t2 WHERE c=b) FROM t1;

在这个示例中，t2 表在子查询中用于转换 t1.b 列的值。 如果每个表都有 N 行，SQLite 预计子查询将运行 N 次，因此它会认为先在 t2 上构造一个自动的临时索引，然后再使用该索引来满足子查询的 N 次实例更快。

自动索引功能可以在运行时通过 automatic_index pragma 禁用。 自动索引默认开启，但可以通过 SQLITE_DEFAULT_AUTOMATIC_INDEX 编译时选项将自动索引默认设置为关闭。 通过使用 SQLITE_OMIT_AUTOMATIC_INDEX 编译时选项可以完全禁用创建自动索引的能力。

在 SQLite 版本 3.8.0 (2013-08-26) 及之后， 每次准备使用自动索引的语句时都会向 错误日志 发送一个 SQLITE_WARNING_AUTOINDEX 消息。 应用程序开发者可以并且应该使用这些警告来识别架构中需要新的持久索引的情况。

不要将自动索引与有时用于实现 PRIMARY KEY 约束 或 UNIQUE 约束 的内部索引（名称类似 "sqlite_autoindex_table_N"）混淆。 这里描述的自动索引仅存在于单个查询期间，永远不会持久化到磁盘，并且只对单个数据库连接可见。 内部索引是 PRIMARY KEY 和 UNIQUE 约束实现的一部分，具有持久性并保存到磁盘，对所有数据库连接都可见。 “autoindex” 这个术语出现在 内部索引 的名称中是出于历史原因，并不表示内部索引与自动索引存在关联。

14.1. 哈希连接

自动索引几乎就是一个 hash join。 唯一不同的是使用的是 B-Tree 而不是哈希表。 如果你愿意认为为自动索引构造的临时 B-Tree 真正只是一个炫酷的哈希表， 那么使用自动索引的查询实际上就是一个哈希连接。

SQLite 在这种情况下构造临时索引而不是哈希表，因为它已经有一个稳健且高性能的 B-Tree 实现，而哈希表需要额外添加。 为这个单一情况添加一个独立的哈希表实现会增加库的体积（该库设计用于低内存嵌入式设备），但性能提升不大。 SQLite 也许将来会增强哈希表实现，但目前看来，在客户端/服务器数据库引擎可能使用哈希连接的情况下继续使用自动索引更为合适。

15. 谓词下推优化

如果子查询无法被 Flatten 到外层查询中，仍可能通过将外层查询的 WHERE 子句项向下推送到子查询来提高性能。 例如： 

CREATE TABLE t1(a INT, b INT);
CREATE TABLE t2(x INT, y INT);
CREATE VIEW v1(a,b) AS SELECT DISTINCT a, b FROM t1;

SELECT x, y, b
  FROM t2 JOIN v1 ON (x=a)
 WHERE b BETWEEN 10 AND 20;

由于 v1 是 DISTINCT，不能被 Flatten。 它必须作为子查询运行，其结果存储在临时表中，然后在 t2 与该临时表之间执行连接。 谓词下推优化会将 "b BETWEEN 10 AND 20" 这一项下推到视图中。 这使得临时表更小，并有助于如果 t1.b 上有索引，则子查询运行更快。 最终计算如下所示： 

SELECT x, y, b
  FROM t2
  JOIN (SELECT DISTINCT a, b FROM t1 WHERE b BETWEEN 10 AND 20)
 WHERE b BETWEEN 10 AND 20;

WHERE 子句下推优化并非总是可用。例如，如果子查询包含 LIMIT，则从外层查询下推 WHERE 子句的任何部分都可能改变内层查询的结果。 在实现该优化的 pushDownWhereTerms() 例程的源代码注释中解释了其他限制。

不要将此优化与 MySQL 中名称相似的优化混淆。 MySQL 的下推优化改变 WHERE 子句约束的评估顺序，使得那些只能使用索引并且无需找到相应表行即可评估的约束优先评估， 从而在约束失败时避免不必要的表行查找。 为了区分，SQLite 将该优化称为 “MySQL 下推优化”。 除了 WHERE 子句下推优化之外，SQLite 也确实执行 MySQL 下推优化。 但是本节的重点是 WHERE 子句下推优化。

16. 外部连接强度降低优化

OUTER JOIN（LEFT JOIN、RIGHT JOIN 或 FULL JOIN）有时可以简化。 LEFT 或 RIGHT JOIN 可能被转换为普通（INNER）JOIN，或者 FULL JOIN 可能被转换为 LEFT JOIN 或 RIGHT JOIN。 如果 WHERE 子句中存在确保简化后结果相同的项，就可以发生此情况。 例如，如果 LEFT JOIN 的右表中的任何列在 WHERE 子句为真时必须为非 NULL，那么 LEFT JOIN 就可以降级为普通 JOIN。

判定连接是否可简化的定理证明器并不完美。 它有时会返回错误的否定结果。 换句话说，它有时无法证明将 OUTER JOIN 的强度降低是安全的——即使实际上是安全的。 例如，证明器不知道 datetime() SQL 函数在其第一个参数为 NULL 时总是返回 NULL， 因此它不会识别以下查询中可以进行强度降低的 LEFT JOIN： 

SELECT urls.url
  FROM urls
  LEFT JOIN
    (SELECT *
      FROM (SELECT url_id AS uid, max(retrieval_time) AS rtime
              FROM lookups GROUP BY 1 ORDER BY 1)
      WHERE uid IN (358341,358341,358341)
    ) recent
    ON u.source_seed_id = recent.xyz OR u.url_id = recent.xyz
 WHERE
     DATETIME(recent.rtime) > DATETIME('now', '-5 days');

未来对证明器的增强可能会使其认识到某些内建函数在输入为 NULL 时总是返回 NULL。 然而，并非所有内建函数都具有这种属性（例如 coalesce()），当然证明器永远无法推理 应用程序定义的 SQL 函数。

17. 省略 OUTER JOIN 优化

有时 LEFT JOIN 或 RIGHT JOIN 可以在不改变结果的情况下完全从查询中省略。 只有在以下所有条件都满足时才可能发生这种情况：

  1. 查询不是聚合查询
  2. 查询是 DISTINCT，或者 OUTER JOIN 的 ON 或 USING 子句将连接约束为仅匹配一行
  3. LEFT JOIN 的右表或者 RIGHT JOIN 的左表在查询中除了其自身的 USING 或 ON 子句之外未被使用。

当 OUTER JOIN 在视图内部使用，然后该视图以不引用 LEFT JOIN 右表或 RIGHT JOIN 左表任何列的方式被使用时，OUTER JOIN 消除经常出现。

这里是一个省略 LEFT JOIN 的简单示例： 

CREATE TABLE t1(ipk INTEGER PRIMARY KEY, v1);
CREATE TABLE t2(ipk INTEGER PRIMARY KEY, v2);
CREATE TABLE t3(ipk INTEGER PRIMARY KEY, v3);

SELECT v1, v3 FROM t1 
  LEFT JOIN t2 ON (t1.ipk=t2.ipk)
  LEFT JOIN t3 ON (t1.ipk=t3.ipk)

在上述查询中，t2 表完全未被使用，因此查询优化器可以像下面这样实现查询： 

SELECT v1, v3 FROM t1 
  LEFT JOIN t3 ON (t1.ipk=t3.ipk)

截至本文撰写时，仅 LEFT JOIN 会被消除。 该优化尚未推广到 RIGHT JOIN，因为 RIGHT JOIN 是 SQLite 的一个相对较新的添加项。 这种不对称可能会在未来的版本中得到纠正。

18. 常量传播优化

当 WHERE 子句包含两个或更多由 AND 运算符连接的等式约束，并且这些约束的各个 affinities 相同时， SQLite 可能会利用等式的传递性构造新的“虚拟”约束，用于简化表达式和/或提升性能。 这称为“常量传播优化”。

例如，考虑以下模式和查询： 

CREATE TABLE t1(a INTEGER PRIMARY KEY, b INT, c INT);
SELECT * FROM t1 WHERE a=b AND b=5;

SQLite 查看 "a=b" 和 "b=5" 约束并推断，如果这两个约束为真，那么 "a=5" 也必须为真。 这意味着可以使用 INTEGER PRIMARY KEY 的值 5 快速查找所需的行。
