1. 引言

SQLite 库被设计得非常容易从Tcl 或 Tcl/Tk脚本中使用。SQLite 最初是作为Tcl 扩展开始的， 而 SQLite 的主要测试套件是用 Tcl 编写的，并使用本文档所描述的 Tcl SQLite 扩展。SQLite 可以与任何编程语言一起使用，但它与 Tcl 的联系非常深。

本文档概述了 SQLite 的 Tcl 编程接口。

2. 为 Tcl 构建并安装 SQLite 扩展

从 SQLite 3.48.0 (2025-01-14) 起，构建 SQLite Tcl 扩展的首选方法是使用规范的 SQLite 源树并运行带有“tclextension-install”目标的 makefile。如果你安装了多个 Tcl 版本，可以在 unix 下使用 --with-tclsh=PATH 命令行选项指定要安装 SQLite 扩展的 Tcl 安装。例如，如果你在 $HOME/tcl 有一个自定义的 Tcl 9.0 安装，可以通过运行以下命令为该安装安装 SQLite：

./configure --with-tclsh=$HOME/tcl/bin/tclsh9.0
make tclextension-install

在 Windows 上，使用 TCLDIR 环境变量指定 Tcl 安装的顶层目录。同样，如果你有一个自定义的 Tcl 安装在 %HOME%/tcl，你的命令如下：

nmake /f Makefile.msc TCLDIR=%HOME%\tcl tclextension-install

在这两种情况下，还有一些其他的 makefile 目标可能有用：

  • tclextension → 构建 SQLite 扩展，但不将文件移动到安装目录。
  • tclextension-list → 显示通过 ./configure 指定或通过 TCLDIR 指定的 Tcl 解释器可用的全部 SQLite 扩展。
  • tclextension-uninstall → 卸载 SQLite 扩展。
  • tclextension-verify → 验证已安装的 SQLite 扩展是否使用与运行该命令的源树相同的 SQLite 版本。

使用上述技术将 SQLite 扩展安装到你的 Tcl 解释器后，可以从任何 tclsh 或 wish 运行以下命令启动它：

package require sqlite3

2.1. 构建过程如何运作

SQLite 的 Tcl 适配器由一个名为src/tclsqlite.c的源代码文件实现。该文件被附加到实现 SQLite 的合并源（"sqlite3.c"）上，创建出一个新的源文件 "tclsqlite3.c"。将 tclsqlite3.c 编译为共享库，以便构建可在运行时加载的 Tcl SQLite 扩展。“tclextension-install” make 目标会这样做，然后还会构建一个合适的 pkgIndex.tcl 脚本，并将共享库和 pkgIndex.tcl 脚本移动到相应的安装目录。

有一个 makefile 目标用于构建 tclsqlite3.c：

make tclsqlite3.c

tclsqlite3.c 文件不必编译为共享库。tclsqlite3.c 库可以静态链接到一个也静态链接 Tcl 库的应用程序中，以构建一个独立的可执行文件。用于测试 SQLite 的 "testfixture" 可执行文件就是这样工作的。

2.2. 额外的构建文档

有关如何测试 SQLite 的 Tcl 扩展，请参阅 SQLite 源码中的src/tcl-extension-testing.md文档，该文档提供了逐步说明。如果遇到问题，可能会提供有关如何构建 SQLite Tcl 扩展的更多提示。

3. SQLite 库的 Tcl 接口

SQLite 的 Tcl 扩展由一个名为 sqlite3 的单一新 Tcl 命令组成。因为只有这一个命令，所以接口没有放在单独的命名空间中。

sqlite3 命令通常如下所示用于打开或创建数据库：

sqlite3  dbcmd  ?database-name?  ?options?

如果只需要获取信息，可以只给 sqlite3 命令一个参数，"-version"、"-sourceid" 或 "-has-codec" 中的任意一个，它将仅返回指定数据而不产生其他效果。

使用其他参数时，sqlite3 命令会打开第二个非选项参数中命名的数据库，如果没有则为空字符串 ""。如果打开成功，会创建一个由第一个参数命名的新 Tcl 命令并返回 ""。（这种方式类似于在 Tk 中创建小部件。）如果打开失败，则不创建 Tcl 命令，而是引发错误并返回错误消息字符串。

如果数据库尚不存在，默认行为是自动创建它（虽然可以通过使用 "-create false" 选项更改此行为）。

数据库名称通常只是存储数据库的磁盘文件名。如果数据库的名称是特殊名称 ":memory:"，则会在内存中创建一个新数据库。如果数据库名称为空字符串，则在自动删除的空文件中创建数据库，当数据库连接关闭时该文件会被删除。如果在 sqlite3 命令上提供 "-uri yes" 选项，则可以使用URI 文件名。

sqlite3 命令支持以下选项：

-create BOOLEAN

如果为 true，则在不存在数据库时创建一个新数据库。 如果为 false，则尝试打开一个先前不存在的数据库文件时会引发错误。默认行为为 "true"。

-nomutex BOOLEAN

如果为 true，则禁用数据库连接的所有互斥锁。这在单线程应用中可带来小幅性能提升。

-readonly BOOLEAN

如果为 true，则以只读方式打开数据库文件。如果为 false，则在文件系统权限允许的情况下以读写方式打开，或者在操作系统拒绝写权限时仅以只读方式打开。默认设置为 "false"。请注意，如果先前拥有该数据库的进程未正常退出并留下了热日志，则需要写权限以便在打开后恢复数据库，因此不能以只读方式打开。

-uri BOOLEAN

如果为 true，则将文件名参数解释为URI 文件名。如果为 false，则该参数为字面文件名。默认值为 "false"。

-vfs VFSNAME

使用由参数命名的替代VFS。

-fullmutex BOOLEAN

如果为 true，则多个线程可以安全地尝试使用数据库。 如果为 false，则此类尝试不安全。默认值取决于扩展的构建方式。

-nofollow BOOLEAN

如果为 true，且数据库名称指向符号链接，则不会跟随该链接打开真实的数据库文件。 如果为 false，则会跟随符号链接。默认值为 "false"。

3.1. eval 方法

最有用的 dbcmd 方法是 "eval"。eval 方法用于在数据库上执行 SQL。eval 方法的语法如下：

dbcmd  eval  ?-withoutnulls? ?-asdict?  ?  sql   var-name?  ?script?

eval 方法的任务是执行第二个参数中给出的 SQL 语句。例如，要在数据库中创建一个新表，可以这样做：

sqlite3 db1 ./testdb
db1 eval {CREATE TABLE t1(a int, b text)}

上述代码创建了一个名为 t1 的新表，包含列 a 和 b。还能更简单吗？

查询结果以列值列表的形式返回。如果查询请求了 2 列且有 3 行匹配结果，则返回的列表将包含 6 个元素。例如：

db1 eval {INSERT INTO t1 VALUES(1,'hello')}
db1 eval {INSERT INTO t1 VALUES(2,'goodbye')}
db1 eval {INSERT INTO t1 VALUES(3,'howdy!')}
set x [db1 eval {SELECT * FROM t1 ORDER BY a}]

上述代码将变量 $x 设置为：

1 hello 2 goodbye 3 howdy!

你也可以通过指定数组名称或在使用 -asdict 标志时指定 dict 变量和 SQL 代码之后的脚本，逐行处理查询结果。对于查询结果的每一行，所有列的值都会插入到数组或 dict 变量中，并执行脚本。例如：

db1 eval {SELECT * FROM t1 ORDER BY a} values {
    parray values
    puts ""
}

以下代码将输出：

values(*) = a b
values(a) = 1
values(b) = hello

values(*) = a b
values(a) = 2
values(b) = goodbye

values(*) = a b
values(a) = 3
values(b) = howdy!

对于结果中每行的每一列，列名都用作数组的索引，并将该列的值存储在相应的数组项中。（注意：如果查询结果集中有两个或更多列具有相同名称，则最后一个具有该名称的列将覆盖之前的值，并且具有相同名称的较早列将无法访问。）特殊的数组索引 * 用于按出现顺序存储列名称列表。

通常，NULL SQL 结果使用 nullvalue 设置存储在数组中。但是，如果使用了 -withoutnulls 选项，则 NULL SQL 值将导致相应的数组元素被取消设置。

如果省略数组变量名或变量名为空字符串，则每个列值会存储在与该列相同名称的变量中。例如：

db1 eval {SELECT * FROM t1 ORDER BY a} {
    puts "a=$a b=$b"
}

由此得到以下输出：

a=1 b=hello
a=2 b=goodbye
a=3 b=howdy!

Tcl 变量名可以出现在第二个参数的 SQL 语句中的任何合法放置字符串或数字字面值的位置。变量的值会替换变量名。如果变量不存在，则使用 NULL。例如：

db1 eval {INSERT INTO t1 VALUES(5,$bigstring)}

注意，不需要对 $bigstring 值加引号。那会自动完成。如果 $bigstring 是一个大字符串或二进制对象，这种技术不仅更容易书写，而且更高效，因为它避免了复制 $bigstring 的内容。

如果 $bigstring 变量具有字符串和 “bytearray” 表示形式，则 TCL 会以字符串插入该值。如果它只有 “bytearray” 表示形式，则以 BLOB 插入。即使值也有文本表示，也想强制以 BLOB 插入，可以在 "$" 前使用 "@"。如下所示：

db1 eval {INSERT INTO t1 VALUES(5,@bigstring)}

如果变量没有 bytearray 表示，则 "@" 的作用就像 "$"。注意 ":" 在所有情况下都像 "$" 那样工作，因此以下语句是另一种表达方式：

db1 eval {INSERT INTO t1 VALUES(5,:bigstring)}

在变量名前使用 ":" 而不是 "$" 有时非常有用，尤其是当 SQL 文本用双引号 "..." 而不是大括号 {...} 包裹时。当 SQL 包含在双引号 "..." 中时，TCL 会执行 $ 变量替换，这可能导致 SQL 注入（如果不格外小心）。但无论是使用双引号 "..." 还是使用大括号 {...} 包裹 SQL，TCL 都不会替换 :-变量，因此使用 :-变量可以为防御 SQL 注入增加一层保护。

3.2. close 方法

顾名思义，SQLite 数据库的 "close" 方法只是关闭数据库。这会附带删除 dbcmd Tcl 命令的副作用。下面是打开然后立即关闭数据库的示例：

sqlite3 db1 ./testdb
db1 close

如果直接删除 dbcmd，其效果与调用 "close" 方法相同。因此以下代码等价于上面的代码：

sqlite3 db1 ./testdb
rename db1 {}

3.3. transaction 方法

"transaction" 方法用于在 SQLite 数据库事务内部执行 Tcl 脚本。脚本完成后事务提交；如果脚本失败，则回滚。如果该事务在另一个事务内（即使是使用 BEGIN 手动启动的），也不会产生任何效果。

transaction 命令可用于安全地将多个 SQLite 命令组合在一起。当然，你也可以始终使用 BEGIN 手动启动事务。但如果发生错误以致 COMMIT 或 ROLLBACK 从未运行，则数据库将无限期保持锁定。此外，BEGIN 不支持嵌套，因此在开始新事务之前必须确保没有其他事务处于活动状态。"transaction" 方法会自动处理所有这些细节。

语法如下：

dbcmd  transaction  ?transaction-type?   script

transaction-type 可以是 deferred、exclusive 或 immediate 之一。默认值为 deferred。

3.4. cache 方法

上面描述的 "eval" 方法为最近执行的 SQL 命令维护了一个预处理语句的缓存。"cache" 方法用于控制此缓存。该命令的第一种形式如下：

dbcmd  cache size  N

这会设置缓存中最多可以存储的语句数量。上限为 100。默认值为 10。如果将缓存大小设为 0，则不进行缓存。

命令的第二种形式如下：

dbcmd  cache flush

cache flush 方法结束当前缓存中的所有预处理语句。

3.5. complete 方法

"complete" 方法将一段假定为 SQL 的字符串作为唯一参数。如果该字符串是完整的 SQL 语句，则返回 TRUE；如果需要输入更多内容，则返回 FALSE。

"complete" 方法在构建交互式应用程序时非常有用，可以判断用户何时完成输入一行 SQL 代码。这实际上只是对sqlite3_complete() C 函数的接口。

3.6. config 方法

"config" 方法使用 sqlite3_db_config() 接口查询或更改数据库连接的某些配置设置。无参数运行此方法会返回一个包含可用配置设置及其当前值的 Tcl 列表：

dbcmd  config

上述命令将返回类似以下内容：

defensive 0 dqs_ddl 1 dqs_dml 1 enable_fkey 0 enable_qpsg 0 enable_trigger 1 enable_view 1 fts3_tokenizer 1 legacy_alter_table 0 legacy_file_format 0 load_extension 0 no_ckpt_on_close 0 reset_database 0 trigger_eqp 0 trusted_schema 1 writable_schema 0

添加单个配置设置的名称可查询该设置的当前值。可选地添加布尔值来更改设置。

为了最大程度提升应用程序安全性，建议进行以下四项配置更改。关闭 trust_schema 设置可防止在触发器、视图、CHECK 约束、生成列和表达式索引中使用虚拟表和可疑 SQL 函数。关闭 dqs_dml 和 dqs_ddl 设置可防止使用双引号字符串。开启 defensive 可防止直接写入影子表。

db config trusted_schema 0
db config defensive 1
db config dqs_dml 0
db config dqs_ddl 0

3.7. copy 方法

"copy" 方法将文件中的数据复制到表中。它返回成功处理的行数。copy 方法的语法如下：

dbcmd  copy  conflict-algorithm   table-name   file-name      ?column-separator?   ?null-indicator?

conflict-algorithm 必须是 INSERT 语句的 SQLite 冲突算法之一：rollback、abort、fail、ignore 或 replace。有关更多信息，请参阅 SQLite 语言章节中的ON CONFLICT。conflict-algorithm 必须以小写指定。

table-name 必须已存在。file-name 必须存在，且每行必须包含与表中定义的列数相同。如果文件中的某行包含的列数多于或少于定义的列数，则 copy 方法会回滚任何插入，并返回错误。

column-separator 是可选的列分隔符字符串。默认是 ASCII 制表符 \t。

null-indicator 是一个可选字符串，用于指示某列值为 null。默认是空字符串。请注意 column-separator 和 null-indicator 是可选的位置参数；如果指定了 null-indicator，则必须先指定 column-separator 参数，并且 column-separator 在 null-indicator 之前。

copy 方法实现的功能与 SQLite shell 命令 .import 类似。

3.8. The timeout 方法

“timeout” 方法用来控制 SQLite 库在放弃数据库事务之前会等待锁释放多长时间。默认超时为 0 毫秒。（换句话说，默认行为是不等待。）

SQLite 数据库允许多个读者同时访问或者单个写者访问，但不能两者同时存在。如果任何进程正在写入数据库，则不允许其他进程读取或写入。如果任何进程正在读取数据库，则其他进程可以读取但不能写入。整个数据库共享一个锁。

当 SQLite 尝试打开数据库并发现它被锁定时，可以选择稍作延迟然后再次尝试打开文件。这个过程会重复，直到查询超时并且 SQLite 返回失败。超时时间是可调的。默认设置为 0，因此如果数据库被锁定，SQL 语句会立即失败。但你可以使用 “timeout” 方法将超时值更改为正数。例如：

db1 timeout 2000

timeout 方法的参数是等待锁释放的最大毫秒数。所以在上面的例子中，最大延迟为 2 秒。

3.9. The busy 方法

“busy” 方法与 “timeout” 类似，仅在数据库被锁定时才起作用。但是 “busy” 方法为程序员提供了更多控制可以采取何种措施。“busy” 方法指定一个回调 Tcl 过程，每当 SQLite 尝试打开一个被锁定的数据库时就会调用它。回调在调用之前会附加一个整数参数。该参数表示当前锁定事件之前对 busy 回调的调用次数。此回调的目的是在短时间内执行一些其他有用的工作（例如处理 GUI 事件），然后返回，使得锁可以再次尝试。回调过程如果希望 SQLite 再次尝试打开数据库，应返回 “0”；如果希望放弃当前操作，则应返回 “1”。

如果在调用 busy 方法时没有提供参数，则返回最后一次通过 busy 方法设置的回调过程名称。如果未设置任何回调过程，则返回空字符串。

3.10. The enable_load_extension 方法

SQLite 的扩展加载机制（通过 load_extension() SQL 函数访问）默认是关闭的。这是一个安全预防措施。如果应用程序想使用 load_extension() 函数，必须先使用此方法开启该能力。

此方法接受一个布尔参数，用于开启或关闭扩展加载功能。

为了最佳安全，除非确有需要，否则不要使用此方法，并在调用此方法之前运行 PRAGMA trusted_schema=OFF 或者使用 “db config trusted_schema 0” 方法。

此方法映射到 sqlite3_enable_load_extension() C/C++ 接口。

3.11. The exists 方法

“exists” 方法类似于 “onecolumn” 和 “eval”，因为它执行 SQL 语句。不同之处在于 “exists” 方法始终返回一个布尔值：如果它执行的 SQL 语句中的查询返回一个或多个行，则返回 TRUE，如果返回空集则返回 FALSE。

“exists” 方法通常用于测试表中行的存在性。例如：

if {[db exists {SELECT 1 FROM table1 WHERE user=$user}]} {
   # 当 $user 存在时的处理
} else {
   # 当 $user 不存在时的处理
}

3.12. The last_insert_rowid 方法

“last_insert_rowid” 方法返回一个整数，表示最近一次插入的数据库行的 ROWID。

3.13. The function 方法

“function” 方法向 SQLite 引擎注册新的 SQL 函数。参数是新 SQL 函数的名称以及实现该函数的 TCL 命令。函数的参数会附加到 TCL 命令后再调用。

出于安全原因，建议应用程序在使用此方法之前先设置 PRAGMA trusted_schema=OFF 或运行 “db config trusted_schema 0” 方法。

语法如下：

dbcmd  function   sql-name   ?options?   script

以下示例创建一个名为 “hex” 的 SQL 函数，将其数字参数转换为十六进制编码字符串：

db function hex {format 0x%X}

“function” 方法接受以下选项：

-argcount INTEGER

指定 SQL 函数接受的参数数量。默认值 -1 表示任意数量的参数。

-deterministic

此选项表示函数在相同参数值下总是返回相同结果。SQLite 查询优化器利用此信息缓存常量输入的函数调用结果并重用，而不是重复调用该函数。

-directonly

此选项限制函数仅可由直接的顶层 SQL 语句使用。该函数将无法被触发器、视图、CHECK 约束、生成的列或索引表达式访问。此选项推荐用于所有应用程序定义的 SQL 函数，并且对于具有副作用或揭示应用程序内部状态的任何 SQL 函数 强烈推荐 使用。

安全警告： 如果不使用此开关，攻击者可能能够通过更改数据库文件的模式在触发器、视图或 CHECK 约束中包含新函数，从而诱使应用程序以攻击者选择的参数运行该函数。因此，如果新函数具有副作用或泄露应用程序的内部状态，且未使用 -directonly 选项，则可能存在安全漏洞。

-innocuous

此选项表示函数没有副作用，并且不会泄露任何不能直接从输入参数计算得出的信息。当指定此选项时，即使 PRAGMA trusted_schema=OFF，该函数也可用于触发器、视图、CHECK 约束、生成的列或索引表达式。除非确实需要，否则不建议使用此选项。

-returntype integer|real|text|blob|any

此选项用于配置函数返回值的类型。如果此选项设置为 “any”（默认值），SQLite 会根据 Tcl 值的内部类型尝试确定函数实现返回的每个值的类型。或者，如果设置为 “text” 或 “blob”，则返回值始终为文本或 blob 值。如果设置为 “integer”，SQLite 会尝试强制将函数返回值转换为整数。如果无法在不丢失数据的情况下完成，它会尝试转换为 real 值，最后再回退到 text。如果设置为 “real”，则尝试返回 real 值，如果不可能则回退到 text。
从版本 3.51 开始，从 SQL 函数中传播的 break 将被解释为 SQL NULL，而不是空字符串。

3.14. The nullvalue 方法

“nullvalue” 方法更改 “eval” 方法返回的 NULL 的表示。

db1 nullvalue NULL

“nullvalue” 方法有助于区分 Tcl 不存在的 NULL 与空列值。NULL 值的默认表示是空字符串。

3.15. The onecolumn 方法

“onecolumn” 方法类似于 “eval”，因为它评估作为参数提供的 SQL 查询语句。不同之处在于 “onecolumn” 返回单个元素，即查询结果第一行的第一列。

这是一个便捷方法。它免去了用户需要在 “eval” 的结果上执行 “[lindex ... 0]” 来提取单列结果的麻烦。

由于方法名可以缩短为任何唯一前缀，此方法通常写作 “one” 而不是全名 “onecolumn”。

3.16. The changes 方法

“changes” 方法返回一个整数，表示最近一次 “eval” 方法在数据库中插入、删除和/或修改的行数。

3.17. The total_changes 方法

“total_changes” 方法返回一个整数，表示自当前数据库连接首次打开以来在数据库中插入、删除和/或修改的总行数。

3.18. The authorizer 方法

“authorizer” 方法提供了对 sqlite3_set_authorizer C/C++ 接口的访问。authorizer 的参数是一个过程的名称，该过程在 SQL 语句编译时被调用以授权某些操作。回调过程接受 5 个参数，用于描述正在编码的操作。如果回调返回文本字符串 “SQLITE_OK”，则允许该操作。如果返回 “SQLITE_IGNORE”，则该操作被静默禁用。如果返回 “SQLITE_DENY”，则编译失败并报错。

如果参数为空字符串，则禁用 authorizer。如果省略参数，则返回当前的 authorizer。

3.19. The bind_fallback 方法

“bind_fallback” 方法让应用程序控制在没有与参数名匹配的 TCL 变量时如何处理参数绑定。

当 eval 方法 在 SQL 语句中看到命名 SQL 参数（如 "$abc"、":def" 或 "@ghi"）时，会尝试查找同名的 TCL 变量，并将该 TCL 变量的值绑定到 SQL 参数。如果不存在这样的 TCL 变量，默认行为是将 SQL NULL 绑定到参数。但是，如果指定了 bind_fallback 过程，则会使用 SQL 参数名称调用该过程，并将过程返回值绑定到 SQL 参数。或者如果过程返回一个错误，SQL 语句则以该错误中止。如果过程返回的代码既不是 TCL_OK 也不是 TCL_ERROR，则 SQL 参数绑定为 NULL，就像默认行为一样。

“bind_fallback” 方法有一个可选参数。如果参数为空字符串，则取消 bind_fallback 并恢复默认行为。如果参数是非空字符串，则该参数是一个 TCL 命令（通常是某个过程的名称），当遇到不匹配任何 TCL 变量的 SQL 参数时调用。如果 “bind_fallback” 方法没有参数，则返回当前的 bind_fallback 命令。

举例来说，下面的设置会导致在 SQL 语句包含不匹配任何全局 TCL 变量的参数时，TCL 抛出错误：


proc bind_error {nm} {
  error "no such variable: $nm"
}
db bind_fallback bind_error

3.20. The progress 方法

此方法注册一个回调，该回调在查询处理期间按周期调用。共有两个参数：调用之间间隔的 SQLite 虚拟机操作码数，以及要调用的 TCL 命令。将 progress 回调设置为空字符串可禁用它。

progress 回调可用于在长时间查询期间显示状态或处理 GUI 事件。

3.21. The collate 方法

此方法注册新的文本排序序列。共有两个参数：排序序列的名称以及实现排序比较函数的 TCL 过程的名称。

例如，下面的代码实现了一个名为 “NOCASE” 的排序序列，该序列按文本顺序排序而不区分大小写：


proc nocase_compare {a b} {
  return [string compare [string tolower $a] [string tolower $b]]
}
db collate NOCASE nocase_compare

3.22. The collation_needed 方法

此方法注册一个回调例程，当 SQLite 引擎需要某个排序序列但尚未注册该排序序列时会调用该回调。回调可以注册所需的排序序列。回调在调用时会带一个参数，该参数是所需排序序列的名称。

3.23. The commit_hook 方法

此方法注册一个回调例程，在 SQLite 尝试提交数据库更改之前调用。如果回调抛出异常或返回非零结果，则事务会回滚而不是提交。

3.24. The rollback_hook 方法

此方法注册一个回调例程，在 SQLite 尝试回滚之前调用。脚本参数按原样运行。

3.25. The status 方法

此方法返回最近评估的 SQL 语句的状态信息。status 方法接受一个参数，该参数应为 “steps” 或 “sorts”。如果参数为 “steps”，则该方法返回上一个 SQL 语句执行的完整表扫描步骤数。如果参数为 “sorts”，则该方法返回排序操作的数量。这些信息可用于检测未使用索引来加速搜索或排序的查询。

status 方法基本上是对 sqlite3_stmt_status() C 语言接口的封装。

3.26. The update_hook 方法

此方法注册一个回调例程，在每次通过 UPDATE、INSERT 或 DELETE 语句修改行之后调用。回调在调用之前会附加四个参数：

  • 适当情况下为 “INSERT”、“UPDATE” 或 “DELETE” 关键字
  • 正在更改的数据库名称
  • 正在更改的表
  • 要更改的表中行的 rowid

回调不得执行会修改调用 update hook 的数据库连接的操作，例如运行查询。

3.27. The preupdate 方法

此方法要么注册一个回调例程，在每次通过 UPDATE、INSERT 或 DELETE 语句修改行之前调用，要么执行与即将进行的更新相关的某些操作。

要注册或移除 preupdate 回调，请使用以下语法：

dbcmd  preupdate  hook  ?SCRIPT?
当注册 preupdate 回调后，每次在修改行之前，回调将使用以下参数运行：
  • 适当情况下为 “INSERT”、“UPDATE” 或 “DELETE” 关键字
  • 正在更改的数据库名称
  • 正在更改的表
  • 要更改的表中行的原始 rowid
  • 要更改的表中行的新 rowid（如果有）
回调不得执行会修改调用 preupdate hook 的数据库连接的操作，例如运行查询。

仅当回调正在执行时，可以使用以下语法执行这些 preupdate 操作：

dbcmd  preupdate  count
dbcmd  preupdate  depth
dbcmd  preupdate  new  INDEX
dbcmd  preupdate  old  INDEX

count 子方法返回正在插入、更新或删除的行中的列数。

depth 子方法返回更新的嵌套深度。直接插入、更新或删除操作为 0；由顶层触发器调用的插入、更新或删除操作为 1；由触发器调用的触发器导致的更改为更高值。

old 和 new 子方法分别返回正在更新的行中选定的原始或更改后的列值。

请注意，Tcl 接口（以及底层 SQLite 库）必须在预处理符号 SQLITE_ENABLE_PREUPDATE_HOOK 定义的情况下构建，preupdate 方法才可用。

3.28. The wal_hook 方法

此方法注册一个回调例程，当数据库处于 WAL 模式 且事务提交之后调用。回调在调用之前会附加两个参数：

  • 事务提交所在的数据库名称
  • 该数据库的预写日志（WAL）文件中的条目数

此方法可能决定自行执行 checkpoint 或作为随后空闲回调执行。请注意 SQLite 仅允许一个 WAL hook。默认情况下，此唯一的 WAL hook 被用于自动 checkpoint。如果你设置了显式 WAL hook，则该 WAL hook 必须确保 checkpoint 正在发生，因为自动 checkpoint 机制将被禁用。

此方法应返回一个等价于 SQLite 错误码的整数值（通常成功时为 0 对应 SQLITE_OK，出错时为 1 对应 SQLITE_ERROR）。与 sqlite3_wal_hook() 一样，返回不对应任何 SQLite 错误码的整数的后果未定义。如果脚本返回的值无法解释为整数，或脚本抛出 Tcl 异常，则不会向 SQLite 返回错误，但会引发 Tcl 后台错误。

3.29. The incrblob 方法

此方法打开一个 TCL 通道，可用于读取或写入数据库中预先存在的 BLOB。语法如下：

dbcmd  incrblob  ?-readonly?   ?DB?  TABLE  COLUMN  ROWID

该命令返回一个用于读写 BLOB 的新的 TCL 通道。该通道通过底层的 sqlite3_blob_open() C 语言接口打开。请使用 TCL 的 close 命令关闭该通道。

3.30. The errorcode 方法

此方法返回最近一次 SQLite 操作产生的数字错误码。

3.31. The trace 方法

“trace” 方法注册一个回调，在每条 SQL 语句编译时调用。SQL 的文本在调用前作为单个字符串附加到命令中。此功能可以用于（例如）记录应用程序执行的所有 SQL 操作。

3.32. The trace_v2 方法

“trace_v2” 方法注册一个回调，在每条 SQL 语句编译时调用。语法如下：

dbcmd  trace_v2  ?callback?  ?mask?

该命令使 “callback” 脚本在某些条件发生时被调用。条件由 mask 参数决定，该参数应为包含以下关键字之一或多个的 TCL 列表：

  • statement
  • profile
  • row
  • close

statement 的跟踪在每次运行新 SQL 语句时调用回调并传入两个参数。第一个参数是一个整数，表示指向底层 sqlite3_stmt 对象的指针值。该整数可用于将 SQL 语句文本与 profile 或 row 回调的结果相关联。第二个参数是正在执行的 SQL 语句的未展开文本。所谓“未展开”，是指文本中的变量替换未被替换为变量值。这与 “trace” 方法不同，后者会展开变量替换。

profile 的跟踪在每条 SQL 语句结束时调用回调并传入两个参数。第一个参数是一个整数，表示底层 sqlite3_stmt 对象的值。第二个参数是该语句的近似运行时间，单位为纳秒。运行时间是基于应用程序运行平台能力的最佳估计。

row 的跟踪在 SQL 语句返回新结果行时调用回调并传入一个参数。该参数是一个整数，表示底层 sqlite3_stmt 对象指针的值。

close 的跟踪在数据库连接关闭时调用回调并传入一个参数。该参数是一个整数，表示指向底层 sqlite3 对象的指针值。

每个数据库连接只能注册一个 trace 回调。每次使用 “trace” 或 “trace_v2” 都会取消之前的所有 trace 回调。

3.33. The backup 方法

“backup” 方法可创建一个运行中数据库的备份副本。命令语法如下：

dbcmd  backup  ?source-database?  backup-filename

可选的 source-database 参数指出当前连接中的哪个数据库应被备份。默认值为 main（即主数据库文件）。备份 TEMP 表使用 temp。要备份通过 ATTACH 命令添加的辅助数据库，请使用在 ATTACH 命令中指定的数据库名称。

backup-filename 是备份写入的文件名。Backup-filename 不必事先存在，但如果存在，必须是结构良好的 SQLite 数据库。

3.34. The restore 方法

“restore” 方法将内容从独立的数据库文件复制到当前数据库连接，覆盖任何预先存在的内容。命令语法如下：

dbcmd  restore  ?target-database?  source-filename

可选的 target-database 参数指定当前连接中的哪个数据库应被新内容覆盖。默认值为 main（即主数据库文件）。要重新填充 TEMP 表，请使用 temp。要覆盖通过 ATTACH 命令添加的辅助数据库，请使用在 ATTACH 命令中指定的数据库名称。

source-filename 是一个现有的结构良好的 SQLite 数据库文件的名称，内容将从中提取。

3.35. serialize 方法

serialize 方法创建一个 BLOB，它是底层数据库的完整副本。语法如下：

dbcmd  serialize  ?database?

可选参数是要进行序列化的架构或数据库的名称。默认值为 "main"。

该例程返回一个 TCL byte-array，它是所标识数据库的完整内容。这个 byte-array 可以被写入文件并作为普通的 SQLite 数据库使用，或者通过 TCP/IP 连接发送给其他应用程序，或传递给另一个数据库连接的 deserialize 方法。

3.36. deserialize 方法

deserialize 方法接受一个包含 SQLite 数据库文件的 TCL byte-array，并将其添加到数据库连接中。语法为：

dbcmd  deserialize  ?database?  value

database 选项参数用于标识应该接收反序列化的附加数据库。默认值是 "main"。

该命令使 SQLite 与先前的数据库断开连接，并重新附加到包含 value 内容的内存数据库。如果 value 不是包含定义良好的 SQLite 数据库的 byte-array，那么随后的命令很可能返回 SQLITE_CORRUPT 错误。

3.37. interrupt 方法

interrupt 方法调用 sqlite3_interrupt() 接口，从而终止任何挂起的查询。

3.38. version 方法

返回当前库版本。例如："3.23.0"。

3.39. profile 方法

该方法用于分析应用程序运行的 SQL 语句的执行情况。语法如下：

dbcmd  profile  ?script?

除非 script 是空字符串，否则该方法会安排在每条 SQL 语句执行后评估 script。在调用之前，会将两个参数附加到 script：所执行的 SQL 语句文本以及执行该语句所消耗的时间（以纳秒为单位）。

一个数据库句柄在任意时间只能注册一个 profile 脚本。如果在调用 profile 方法时已经注册了脚本，之前的 profile 脚本会被新脚本替换。如果 script 参数为空字符串，则取消之前注册的 profile 回调，但不注册新的 profile 脚本。

3.40. unlock_notify 方法

unlock_notify 方法用于访问SQLite核心库的 sqlite3_unlock_notify() 接口，用于测试目的。应用程序不鼓励使用此方法。
