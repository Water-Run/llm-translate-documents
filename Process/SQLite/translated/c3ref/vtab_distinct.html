SQLite C 接口

确定虚拟表查询是否为 DISTINCT

int sqlite3_vtab_distinct(sqlite3_index_info*);

此 API 只能在虚拟表实现的 xBestIndex 方法 中使用。 在 xBestIndex() 之外调用此接口的结果是未定义的，可能会造成损害。

sqlite3_vtab_distinct() 接口返回一个介于 0 和 3 的整数。 sqlite3_vtab_distinct() 返回的整数为虚拟表提供额外信息， 说明查询规划器希望如何排序输出。只要虚拟表能够满足查询规划器的排序要求， 即可设置 “orderByConsumed” 标志。

  1. 如果 sqlite3_vtab_distinct() 接口返回 0，说明查询规划器需要虚拟表按照 sqlite3_index_info 对象中的 “nOrderBy” 和 “aOrderBy” 字段 所定义的排序顺序返回所有行。这是默认预期。如果虚拟表以排序后的顺序输出所有行， 那么无论 sqlite3_vtab_distinct() 返回何值，xBestIndex 方法都可以安全地设置 “orderByConsumed” 标志。

  2. 如果 sqlite3_vtab_distinct() 接口返回 1，说明查询规划器并不要求按照排序顺序返回行， 只要所有在 “aOrderBy” 字段标识的列上具有相同值的行相邻即可。当查询规划器执行 GROUP BY 时会使用此模式。

  3. 如果 sqlite3_vtab_distinct() 接口返回 2，说明查询规划器不要求按特定顺序返回行， 只要在 “aOrderBy” 中标识的所有列值相同的行相邻即可。此外，当两行或多行在 “colUsed” 标识的所有列上值都相同时，结果中可以选择性地省略其中除一行以外的其他行。 虚拟表并不要求在 “colUsed” 列上值重复时一定省略重复行，但如果虚拟表能够在不额外消耗太多精力的情况下省略它们， 可能会让查询运行得更快。此模式用于 DISTINCT 查询。

  4. 如果 sqlite3_vtab_distinct() 接口返回 3，说明虚拟表必须像 sqlite3_vtab_distinct() 返回 0 时那样 按照 “aOrderBy” 所定义的顺序返回行。但是，如果结果中有两行或多行在所有由 “colUsed” 标识的列上都有相同值， 那么可以选择性地省略其中除一行以外的其他行。与返回 2 时一样，虚拟表并不要求在 “colUsed” 列上值重复时一定省略重复行， 但如果虚拟表能够在不额外消耗太多精力的情况下省略它们， 可能会让查询运行得更快。此模式用于同时包含 DISTINCT 和 ORDER BY 子句的查询。

下表总结了 sqlite3_vtab_distinct() 返回值所对应的虚拟表是否可以设置 “orderByConsumed” 标志的条件。 此表是前面四段文字的重述：

sqlite3_vtab_distinct() 返回值 按 aOrderBy 排序返回 在所有 aOrderBy 列上值相同的行相邻 在所有 colUsed 列上值重复的行可以省略
0是是否
1否是否
2否是是
3是是是

为了比较虚拟表输出值是否相同（用于排序），两个 NULL 值被视为相同。 换句话说，比较操作符是 “IS”（或 “IS NOT DISTINCT FROM”），而不是 “==”。

如果虚拟表实现无法满足上述要求，那么它就不得在 sqlite3_index_info 对象中设置 “orderByConsumed” 标志，否则可能导致错误结果。

只要未设置 “orderByConsumed” 标志，虚拟表实现始终可以按照任意顺序返回行。 当 “orderByConsumed” 标志未设置时，查询规划器会添加额外的 字节码，以确保 SQL 查询最终返回的结果顺序正确。 使用 “orderByConsumed” 标志和 sqlite3_vtab_distinct() 接口只是一个优化手段。 谨慎使用 sqlite3_vtab_distinct() 接口和 “orderByConsumed” 标志，可能会让针对虚拟表的查询运行得更快。 另一方面，如果在无效的情况下过于激进地设置 “orderByConsumed” 标志，可能会导致 SQLite 返回错误结果。
