SQLite C 接口

解锁通知

int sqlite3_unlock_notify(
  sqlite3 *pBlocked,                          /* 等待连接 */
  void (*xNotify)(void **apArg, int nArg),    /* 要调用的回调函数 */
  void *pNotifyArg                            /* 传递给 xNotify 的参数 */
);

当运行在共享缓存模式时，如果无法获取共享缓存或共享缓存中某个表的所需锁，数据库操作可能会因 SQLITE_LOCKED 错误而失败。 有关共享缓存锁定的描述，请参见 SQLite 共享缓存模式。 此 API 可用于注册一个回调，当当前持有所需锁的连接释放锁时，SQLite 会调用该回调。 仅当库在编译时定义了 SQLITE_ENABLE_UNLOCK_NOTIFY C 预处理器符号时，此 API 才可用。

另请参阅：使用 SQLite 解锁通知功能。

当数据库连接完成当前事务（无论是提交还是回滚）时，都会释放共享缓存锁。

当一个连接（称为被阻塞连接）未能获得共享缓存锁并向调用者返回 SQLITE_LOCKED 时，内部会存储锁定所需资源的数据库连接（阻塞连接）的身份。 当应用收到 SQLITE_LOCKED 错误后，它可使用被阻塞连接句柄作为第一个参数调用 sqlite3_unlock_notify() 方法，以注册一个将在阻塞连接当前事务结束时调用的回调。 该回调从结束阻塞连接事务的 sqlite3_step 或 sqlite3_close 调用中被调用。

如果在多线程应用中调用 sqlite3_unlock_notify()，则可能在调用 sqlite3_unlock_notify() 之前，阻塞连接已经结束其事务。 如果发生这种情况，则在 sqlite3_unlock_notify() 的调用内部立即调用指定的回调。

如果被阻塞连接试图在某个共享缓存表上获取写锁，并且有多个其他连接当前在同一表上拥有读锁，则 SQLite 会任意选择其中一个连接作为阻塞连接。

每个被阻塞连接最多只能注册一个解锁通知回调。如果在被阻塞连接已经注册了解锁通知回调时再次调用 sqlite3_unlock_notify()，则新回调将替换旧回调。 如果 sqlite3_unlock_notify() 的第二个参数为 NULL，则会取消现有的解锁通知回调。也可以通过使用 sqlite3_close() 关闭被阻塞连接来取消其解锁通知回调。

解锁通知回调不是可重入的。如果应用在解锁通知回调中调用任何 sqlite3_xxx API 函数，可能会导致崩溃或死锁。

除非检测到死锁（见下文），否则 sqlite3_unlock_notify() 始终返回 SQLITE_OK。

回调调用细节

当注册解锁通知回调时，应用提供一个 void* 指针，该指针在回调被调用时传递给它。 不过，回调函数的签名允许 SQLite 传递一个 void* 上下文指针数组给它。 传递给解锁通知回调的第一个参数是一个 void* 指针数组，第二个参数是数组中的条目数。

当阻塞连接的事务结束时，可能有多个被阻塞连接已注册了解锁通知回调。 如果两个或更多此类被阻塞连接指定了相同的回调函数，则不会多次调用该回调函数，而是会将被阻塞连接指定的 void* 上下文指针集合打包到一个数组中，并一次性调用回调。 这让应用有机会对这些已解除阻塞的数据库连接集相关的操作进行优先排序。

死锁检测

假定在注册了解锁通知回调后，数据库等待回调发出后才采取进一步操作（这是合理的假设）， 那么使用此 API 可能会导致应用死锁。例如，如果连接 X 正在等待连接 Y 的事务完成，而连接 Y 同样在等待连接 X 的事务完成，则两者都无法继续，系统可能会无限期死锁。

为避免这种情况，sqlite3_unlock_notify() 会执行死锁检测。 如果某次 sqlite3_unlock_notify() 调用会使系统陷入死锁状态，则会返回 SQLITE_LOCKED，且不会注册任何解锁通知回调。 当连接 A 为连接 B 的事务完成注册了解锁通知回调，而连接 B 也为连接 A 的事务完成注册了解锁通知回调时，系统即被视为处于死锁状态。 间接死锁也会被检测到，因此如果连接 B 为连接 C 的事务完成注册了解锁通知回调，而连接 C 正在等待连接 A，则系统也被视为死锁。 允许任意多个层级的间接死锁。

“DROP TABLE” 异常

当某次 sqlite3_step() 调用返回 SQLITE_LOCKED 时，几乎总是应该调用 sqlite3_unlock_notify()。 不过有一个例外。当执行 “DROP TABLE” 或 “DROP INDEX” 语句时，SQLite 会检查同一连接是否存在正在执行的 SELECT 语句。 如果存在，则返回 SQLITE_LOCKED。在这种情况下不存在“阻塞连接”，因此调用 sqlite3_unlock_notify() 会导致解锁通知回调立即被调用。 如果应用随后重新尝试执行 “DROP TABLE” 或 “DROP INDEX” 查询，可能会导致无限循环。

解决此问题的一种方法是检查 sqlite3_step() 调用返回的扩展错误码。如果存在阻塞连接，则扩展错误码设置为 SQLITE_LOCKED_SHAREDCACHE。 否则，在特殊的 “DROP TABLE/INDEX” 情况下，扩展错误码仅为 SQLITE_LOCKED。
