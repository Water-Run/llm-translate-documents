SQLite C 接口

操作系统接口文件虚拟方法对象

typedef struct sqlite3_io_methods sqlite3_io_methods;
struct sqlite3_io_methods {
  int iVersion;
  int (*xClose)(sqlite3_file*);
  int (*xRead)(sqlite3_file*, void*, int iAmt, sqlite3_int64 iOfst);
  int (*xWrite)(sqlite3_file*, const void*, int iAmt, sqlite3_int64 iOfst);
  int (*xTruncate)(sqlite3_file*, sqlite3_int64 size);
  int (*xSync)(sqlite3_file*, int flags);
  int (*xFileSize)(sqlite3_file*, sqlite3_int64 *pSize);
  int (*xLock)(sqlite3_file*, int);
  int (*xUnlock)(sqlite3_file*, int);
  int (*xCheckReservedLock)(sqlite3_file*, int *pResOut);
  int (*xFileControl)(sqlite3_file*, int op, void *pArg);
  int (*xSectorSize)(sqlite3_file*);
  int (*xDeviceCharacteristics)(sqlite3_file*);
  /* 以上方法适用于版本 1 */
  int (*xShmMap)(sqlite3_file*, int iPg, int pgsz, int, void volatile**);
  int (*xShmLock)(sqlite3_file*, int offset, int n, int flags);
  void (*xShmBarrier)(sqlite3_file*);
  int (*xShmUnmap)(sqlite3_file*, int deleteFlag);
  /* 以上方法适用于版本 2 */
  int (*xFetch)(sqlite3_file*, sqlite3_int64 iOfst, int iAmt, void **pp);
  int (*xUnfetch)(sqlite3_file*, sqlite3_int64 iOfst, void *p);
  /* 以上方法适用于版本 3 */
  /* 未来版本可能会添加额外方法 */
};

每个通过 sqlite3_vfs.xOpen 方法打开的文件都会填充一个 sqlite3_file 对象（或更常见的 sqlite3_file 对象的子类），并将该对象的 sqlite3_io_methods 实例指针存入其中。 该对象定义了用于对由 sqlite3_file 表示的打开文件执行各种操作的方法。

如果 sqlite3_vfs.xOpen 方法将 sqlite3_file.pMethods 元素设置为非 NULL 指针， 那么即便 sqlite3_vfs.xOpen 报告失败，也可能调用 sqlite3_io_methods.xClose 方法。 防止在 sqlite3_vfs.xOpen 失败后调用 xClose 的唯一方法是将 sqlite3_file.pMethods 元素设置为 NULL。

xSync 的 flags 参数可以是 SQLITE_SYNC_NORMAL 或 SQLITE_SYNC_FULL。第一个选项是普通的 fsync()。 第二个选项是 macOS 风格的 fullsync。可以 OR 入 SQLITE_SYNC_DATAONLY 标志， 以表明只需要同步文件的数据而不是 inode。

xLock() 和 xUnlock() 的整数值为以下之一：

  • SQLITE_LOCK_NONE,
  • SQLITE_LOCK_SHARED,
  • SQLITE_LOCK_RESERVED,
  • SQLITE_LOCK_PENDING, 或
  • SQLITE_LOCK_EXCLUSIVE.
xLock() 会升级数据库文件锁。换句话说，xLock() 会将数据库文件锁从 NONE 向 EXCLUSIVE 方向移动。 xLock() 的参数总是 SHARED、RESERVED、PENDING 或 EXCLUSIVE 之一，永远不会是 SQLITE_LOCK_NONE。 如果数据库文件锁已经处于请求的锁级别或更高级别，那么对 xLock() 的调用就是一个空操作。 xUnlock() 会将数据库文件锁降级为 SHARED 或 NONE。 如果锁已经处于请求的锁状态或更低，那么对 xUnlock() 的调用也是空操作。 xCheckReservedLock() 方法检查当前进程或其他进程是否持有文件的 RESERVED、PENDING 或 EXCLUSIVE 锁。 该方法会通过输出指针参数返回 true 表示存在此类锁，若不存在则返回 false。

xFileControl() 方法是一个通用接口，允许自定义 VFS 实现通过 sqlite3_file_control() 接口直接控制一个打开的文件。 第二个 “op” 参数是一个整数操作码。第三个参数是一个通用指针，指向可能包含参数或用于写入返回值的空间的结构。 xFileControl() 的潜在用途可能包括：启用具有超时的阻塞锁、改变锁策略（例如使用点文件锁）、查询锁的状态或解除陈旧锁。 SQLite 内核保留所有小于 100 的操作码供自身使用。 可以参考 小于 100 的操作码列表。 定义自定义 xFileControl 方法的应用应使用大于 100 的操作码以避免冲突。 VFS 实现应该对其不识别的文件控制操作码返回 SQLITE_NOTFOUND。

xSectorSize() 方法返回承载该文件的设备的扇区大小。扇区大小是可以在不影响文件中其他字节的情况下执行的最小写入量。 xDeviceCharacteristics() 方法返回一个位向量，描述了底层设备的行为：

  • SQLITE_IOCAP_ATOMIC
  • SQLITE_IOCAP_ATOMIC512
  • SQLITE_IOCAP_ATOMIC1K
  • SQLITE_IOCAP_ATOMIC2K
  • SQLITE_IOCAP_ATOMIC4K
  • SQLITE_IOCAP_ATOMIC8K
  • SQLITE_IOCAP_ATOMIC16K
  • SQLITE_IOCAP_ATOMIC32K
  • SQLITE_IOCAP_ATOMIC64K
  • SQLITE_IOCAP_SAFE_APPEND
  • SQLITE_IOCAP_SEQUENTIAL
  • SQLITE_IOCAP_UNDELETABLE_WHEN_OPEN
  • SQLITE_IOCAP_POWERSAFE_OVERWRITE
  • SQLITE_IOCAP_IMMUTABLE
  • SQLITE_IOCAP_BATCH_ATOMIC
  • SQLITE_IOCAP_SUBPAGE_READ

SQLITE_IOCAP_ATOMIC 属性表示任意大小的写入都是原子的。 SQLITE_IOCAP_ATOMICnnn 值表示大小为 nnn 字节且对齐到 nnn 整数倍地址的块写入是原子的。 SQLITE_IOCAP_SAFE_APPEND 表示向文件追加数据时会先追加数据然后再扩展文件大小，绝不会反向操作。 SQLITE_IOCAP_SEQUENTIAL 属性表示数据以与调用 xWrite() 相同的顺序写入磁盘。

如果 xRead() 返回 SQLITE_IOERR_SHORT_READ，它还必须用零填充缓冲区中未读的部分。 未对短读进行零填充的 VFS 似乎也能工作。不过，未对短读进行零填充最终会导致数据库损坏。
