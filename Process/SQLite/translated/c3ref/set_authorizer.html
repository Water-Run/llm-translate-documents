SQLite C 接口

编译时授权回调

int sqlite3_set_authorizer(
  sqlite3*,
  int (*xAuth)(void*,int,const char*,const char*,const char*,const char*),
  void *pUserData
);

该例程在第一个参数中提供的特定数据库连接上注册一个授权器回调。 当通过 sqlite3_prepare() 或其变体 sqlite3_prepare_v2()、 sqlite3_prepare_v3()、sqlite3_prepare16()、sqlite3_prepare16_v2() 和 sqlite3_prepare16_v3() 编译 SQL 语句时，会调用授权器回调。在编译过程中，在创建执行各种操作的逻辑时，会调用授权器回调来判断这些操作是否被允许。授权器回调应返回 SQLITE_OK 以允许该操作，返回 SQLITE_IGNORE 以拒绝特定操作但允许 SQL 语句继续编译，或返回 SQLITE_DENY 以使整个 SQL 语句因错误被拒绝。如果授权器回调返回的值既不是 SQLITE_IGNORE、SQLITE_OK 也不是 SQLITE_DENY，那么触发该授权器的 sqlite3_prepare_v2() 或等效调用将以错误信息失败。

当回调返回 SQLITE_OK 时，意味着所请求的操作是允许的。当回调返回 SQLITE_DENY 时，触发授权器的 sqlite3_prepare_v2() 或等效调用将失败，并给出访问被拒绝的错误信息。

授权器回调的第一个参数是向 sqlite3_set_authorizer() 接口的第三个参数的副本。回调的第二个参数是一个整数动作代码，用于指定需要授权的具体动作。回调的第三到第六个参数要么是 NULL 指针，要么是包含有关要授权动作的额外细节的以空字符结尾的字符串。应用程序必须始终准备在授权回调的第三至第六个参数中遇到 NULL 指针。

如果动作代码是 SQLITE_READ 且回调返回 SQLITE_IGNORE，那么准备好的语句会构造出用 NULL 值替代原本要读取的表列（如果返回的是 SQLITE_OK 则会读取该列）。SQLITE_IGNORE 返回可用于拒绝不可信用户访问表中的单个列。当表被 SELECT 引用但并未从该表中提取任何列值（例如查询 "SELECT count(*) FROM tab"）时，SQLITE_READ 授权回调会针对该表调用一次，而列名为空字符串。如果动作代码是 SQLITE_DELETE 且回调返回 SQLITE_IGNORE，那么 DELETE 操作仍会继续，但截断优化会被禁用，且所有行将逐行删除。

当从不可信来源 准备 SQL 语句时会使用一个授权器，以确保这些 SQL 语句不会尝试访问它们无权查看的数据，也不会尝试执行损坏数据库的恶意语句。例如，某个应用程序可能允许用户输入任意 SQL 查询供数据库评估。但该应用程序并不希望用户对数据库进行任意更改。在用户输入的 SQL 被 准备 时，可以设置授权器，只允许 SELECT 语句。

需要处理来自不可信来源的 SQL 的应用程序还可以考虑在使用授权器的同时，通过 sqlite3_limit() 降低资源限制，并使用 max_page_count PRAGMA 限制数据库大小。

同一时间只能在一个数据库连接上设置一个授权器。每次调用 sqlite3_set_authorizer 都会覆盖先前的调用。通过安装一个 NULL 回调可以禁用授权器。授权器默认被禁用。

授权器回调不得执行会修改调用授权器回调的数据库连接的任何操作。注意在本段所述的“修改”含义中，sqlite3_prepare_v2() 和 sqlite3_step() 都会修改它们的数据库连接。

当使用 sqlite3_prepare_v2() 准备语句时，由于模式更改，该语句可能会在 sqlite3_step() 期间被重新准备。因此，应用程序应确保在 sqlite3_step() 期间正确的授权器回调仍保持有效。

请注意，授权器回调仅在 sqlite3_prepare() 或其变体期间被调用。除非如前一段所述，sqlite3_step() 在模式更改后调用 sqlite3_prepare_v2() 重新准备语句，否则在 sqlite3_step() 执行语句评估时不会执行授权。
