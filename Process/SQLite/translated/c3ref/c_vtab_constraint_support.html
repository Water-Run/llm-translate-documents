SQLite C 接口

虚拟表配置选项

#define SQLITE_VTAB_CONSTRAINT_SUPPORT 1
#define SQLITE_VTAB_INNOCUOUS          2
#define SQLITE_VTAB_DIRECTONLY         3
#define SQLITE_VTAB_USES_ALL_SCHEMAS   4

这些宏定义了sqlite3_vtab_config()接口的各项选项，虚拟表实现可以利用这些选项来自定义并优化它们的行为。

SQLITE_VTAB_CONSTRAINT_SUPPORT
支持形如 sqlite3_vtab_config(db,SQLITE_VTAB_CONSTRAINT_SUPPORT,X) 的调用， 其中 X 是一个整数。如果 X 为零，那么其 xCreate 或 xConnect 方法在调用 sqlite3_vtab_config() 时的虚拟表不支持约束。在这种配置下（这是默认配置），如果 xUpdate 方法返回 SQLITE_CONSTRAINT，则整个语句会像用户 SQL 语句中显式指定了 OR ABORT 一样回滚，不管实际指定了哪种 ON CONFLICT 模式。

如果 X 非零，那么虚拟表实现保证只要 xUpdate 返回 SQLITE_CONSTRAINT，它就会在对内部或持久数据结构做出任何修改之前返回结果。 如果当前的 ON CONFLICT 模式是 ABORT、FAIL、IGNORE 或 ROLLBACK，SQLite 就可以回滚语句或数据库事务，并根据需要放弃或继续执行当前 SQL 语句。 如果 ON CONFLICT 模式是 REPLACE，且 xUpdate 方法返回了 SQLITE_CONSTRAINT，SQLite 就会像该冲突模式是 ABORT 一样处理。

必须处理 OR REPLACE 的虚拟表实现需要在 xUpdate 方法中完成此工作。如果调用 sqlite3_vtab_on_conflict() 函数表明当前的 ON CONFLICT 策略是 REPLACE，虚拟表实现应当在 xUpdate 回调中悄悄替换合适的行并返回 SQLITE_OK。或者，如果无法实现，也可以返回 SQLITE_CONSTRAINT，这种情况下 SQLite 会回退到 OR ABORT 的约束处理方式。

SQLITE_VTAB_DIRECTONLY
在xConnect或xCreate方法内调用 sqlite3_vtab_config(db,SQLITE_VTAB_DIRECTONLY) 会禁止该虚拟表在触发器和视图中使用。

SQLITE_VTAB_INNOCUOUS
在xConnect或xCreate方法内调用 sqlite3_vtab_config(db,SQLITE_VTAB_INNOCUOUS) 会将该虚拟表标记为可安全在触发器和视图中使用。 从概念上讲，SQLITE_VTAB_INNOCUOUS 标签意味着即便受恶意黑客控制，该虚拟表也不会造成严重危害。开发者除非万不得已，不应设置 SQLITE_VTAB_INNOCUOUS 标志。

SQLITE_VTAB_USES_ALL_SCHEMAS
在xConnect或xCreate方法内调用 sqlite3_vtab_config(db,SQLITE_VTAB_USES_ALL_SCHEMA) 会通知查询规划器在使用该虚拟表时至少在所有 schema（"main"、"temp" 以及任何 ATTACH 的数据库）上开始一个读取事务。
