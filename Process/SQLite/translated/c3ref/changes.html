SQLite C 接口

计算修改的行数

int sqlite3_changes(sqlite3*);
sqlite3_int64 sqlite3_changes64(sqlite3*);

这些函数返回最近完成的 INSERT、UPDATE 或 DELETE 语句在指定数据库连接上修改、插入或删除的行数。两个函数在返回值类型上不同，如果最近的 INSERT、UPDATE 或 DELETE 修改的行数超过 int 类型支持的最大值，则 sqlite3_changes() 的返回值未定义。执行其他类型的 SQL 语句不会改变这些函数的返回值。对于此接口，CREATE TABLE AS SELECT 语句不视为 INSERT、UPDATE 或 DELETE，因此该语句添加到新表中的行不计入。

只有由 INSERT、UPDATE 或 DELETE 语句直接执行的更改才被考虑——由触发器、外键操作或REPLACE约束解析引起的辅助更改不计入。

被INSTEAD OF 触发器拦截的视图更改不计入。在视图上运行 INSERT、UPDATE 或 DELETE 语句后立即调用 sqlite3_changes() 返回的值始终为零。仅对真实表所做的更改才会计入。

如果在触发器程序运行时执行 sqlite3_changes() 函数，情况会更复杂。这可能发生在程序使用changes() SQL 函数，或其他回调函数直接调用 sqlite3_changes() 时。主要情况如下：

  • 在进入触发器程序之前，会保存 sqlite3_changes() 函数返回的值。触发器程序完成后，原始值会恢复。

  • 在触发器程序内部，每个 INSERT、UPDATE 和 DELETE 语句在完成时会照常设置 sqlite3_changes() 返回的值。当然，该值不会包含子触发器执行的任何更改，因为 sqlite3_changes() 的值会在每个子触发器运行后被保存和恢复。

这意味着如果在触发器内部的第一个 INSERT、UPDATE 或 DELETE 语句中使用 changes() SQL 函数（或类似函数），它会返回调用语句开始执行时设置的值。如果在同一触发器程序中的第二个或随后的此类语句中使用，它返回的值反映的是前一个 INSERT、UPDATE 或 DELETE 语句所修改的行数。

如果在 sqlite3_changes() 运行期间有其他线程在同一数据库连接上进行更改，那么返回值将不可预测且没有意义。
