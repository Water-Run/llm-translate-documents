SQLite C 接口

编译 SQL 语句

int sqlite3_prepare(
  sqlite3 *db,            /* Database handle */
  const char *zSql,       /* SQL statement, UTF-8 encoded */
  int nByte,              /* Maximum length of zSql in bytes. */
  sqlite3_stmt **ppStmt,  /* OUT: Statement handle */
  const char **pzTail     /* OUT: Pointer to unused portion of zSql */
);

要执行 SQL 语句，必须首先使用这些例程之一将其编译为字节码程序。换句话说，这些例程是 预处理语句 对象的构造函数。

推荐使用的例程是 sqlite3_prepare_v2()。sqlite3_prepare() 接口是旧版本的，应该避免使用。sqlite3_prepare_v3() 提供了额外的 “prepFlags” 选项，用于特殊用途。

优先使用 UTF-8 接口，因为 SQLite 当前使用 UTF-8 执行所有解析。UTF-16 接口仅作为方便提供。UTF-16 接口通过将输入文本转换为 UTF-8，然后调用相应的 UTF-8 接口来工作。

第一个参数 “db” 是从先前成功调用 sqlite3_open()、sqlite3_open_v2() 或 sqlite3_open16() 获得的 数据库连接。该数据库连接不得已经关闭。

第二个参数 “zSql” 是要编译的语句，以 UTF-8 或 UTF-16 编码。sqlite3_prepare()、sqlite3_prepare_v2() 和 sqlite3_prepare_v3() 接口使用 UTF-8，而 sqlite3_prepare16()、sqlite3_prepare16_v2() 和 sqlite3_prepare16_v3() 使用 UTF-16。

如果 nByte 参数为负，则 zSql 一直读到第一个空终止符。如果 nByte 为正，则它是从 zSql 中读取的最大字节数。当 nByte 为正时，zSql 一直读到第一个空终止符或读取到 nByte 字节为止，以先到者为准。如果 nByte 为零，则不会生成预处理语句。如果调用者知道提供的字符串以 null 结尾，那么传入包含 包括 null 终止符的字节数的 nByte 参数可以略微提高性能。请注意，nByte 测量的是输入的字节长度，而不是字符长度，即使是 UTF-16 接口也是如此。

如果 pzTail 不为 NULL，则 *pzTail 会指向 zSql 中第一条 SQL 语句末尾之后的第一个字节。这些例程只编译 zSql 中的第一条语句，因此 *pzTail 仍然指向未编译的剩余部分。

*ppStmt 会指向一个可使用 sqlite3_step() 执行的已编译 预处理语句。如果发生错误，*ppStmt 会被设置为 NULL。如果输入文本不包含 SQL（例如输入为空串或仅为注释），则 *ppStmt 会被设置为 NULL。调用过程在使用完毕后负责调用 sqlite3_finalize() 删除编译后的 SQL 语句。ppStmt 不得为 NULL。

成功时，sqlite3_prepare() 系列例程返回 SQLITE_OK；否则将返回一个 错误代码。

推荐所有新程序使用 sqlite3_prepare_v2()、sqlite3_prepare_v3()、sqlite3_prepare16_v2() 和 sqlite3_prepare16_v3() 接口。旧接口（sqlite3_prepare() 和 sqlite3_prepare16()）仅为向后兼容而保留，但不建议使用。在 “vX” 接口中，返回的预处理语句（sqlite3_stmt 对象）包含原始 SQL 文本的副本。这会导致 sqlite3_step() 接口在三个方面表现不同：

  1. 如果数据库模式发生变化，sqlite3_step() 不再像以前一样仅返回 SQLITE_SCHEMA，而是会自动重新编译 SQL 语句并再次尝试执行。在 sqlite3_step() 放弃并返回错误之前，最多会发生 SQLITE_MAX_SCHEMA_RETRY 次重试。

  2. 当发生错误时，sqlite3_step() 会返回更详细的 错误代码 或 扩展错误代码。旧行为是在发生错误时仅返回通用的 SQLITE_ERROR 结果代码，应用程序必须再次调用 sqlite3_reset() 才能找出问题的根本原因。使用 “v2” prepare 接口，错误的底层原因会立即返回。

  3. 如果 WHERE 子句中绑定到 宿主参数 的具体值可能影响语句的查询计划选择，那么在修改该参数的 绑定 后第一次调用 sqlite3_step() 时，该语句会像发生模式更改一样自动重新编译。WHERE 子句参数的具体值可能影响查询计划，如果该参数是 LIKE 或 GLOB 操作符的左侧，或者该参数与索引列比较，并且启用了 SQLITE_ENABLE_STAT4 编译时选项。

sqlite3_prepare_v3() 与 sqlite3_prepare_v2() 的区别仅在于增加了 prepFlags 参数，该参数是由零个或多个 SQLITE_PREPARE_* 标志组成的位数组。sqlite3_prepare_v2() 接口在 prepFlags 参数为零时与 sqlite3_prepare_v3() 的行为完全相同。
