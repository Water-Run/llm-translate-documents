SQLite C 接口

在 xBestIndex 中识别和处理 IN 约束

int sqlite3_vtab_in(sqlite3_index_info*, int iCons, int bHandle);

此接口只能在虚拟表实现的 xBestIndex()方法内部使用。从其他上下文调用该接口的结果 是未定义的，并且可能有害。

形式为 "列 IN (...)" 的虚拟表约束 会作为SQLITE_INDEX_CONSTRAINT_EQ约束传递给 xBestIndex 方法。 如果 xBestIndex 想要使用该约束，它必须将相应的 aConstraintUsage[].argvIndex 设置为正整数。 随后，在处理 IN 操作符的常规模式下，SQLite 会生成字节码， 对 IN 操作符右侧的每个值调用一次xFilter()方法。因此虚拟表一次只能看到 IN 操作符右侧的一个值。

然而，在某些情况下，虚拟表如果能够一次性看到 IN 操作符右侧的所有值反而更有利。 sqlite3_vtab_in() 接口通过两种方式来实现这一点：

  1. 对 sqlite3_vtab_in(P,N,-1) 的调用当且仅当 P->aConstraint[N] 约束 是一个可以一次性处理的IN 操作符时返回 true（非零）。 换句话说，当第三个参数为 -1 时，sqlite3_vtab_in() 提供了一种机制， 虚拟表可以借此询问 SQLite 是否支持对该 IN 操作符的一次性处理。

  2. 对 sqlite3_vtab_in(P,N,F) 的调用，如果 F==1 或 F==0，分别表示虚拟表希望或不希望一次性处理该 IN 操作符。 因此当第三个参数（F）为非负值时，该接口就是虚拟表告诉 SQLite 它希望如何处理 IN 操作符的机制。

在同一次 xBestIndex 方法调用中可以多次调用 sqlite3_vtab_in(P,N,F)。 对于任意给定的 P,N 对，在相同的 xBestIndex 调用中 sqlite3_vtab_in(P,N,F) 的返回值始终相同。 如果该接口返回 true（非零），说明该约束是一个可以一次性处理的 IN 操作符。 如果该约束不是 IN 操作符或者不能一次性处理，则该接口返回 false。

当同时满足下列两个条件时，才采用 IN 操作符的一次性处理：

  1. P->aConstraintUsage[N].argvIndex 的值设置为正整数。这是虚拟表告诉 SQLite 它希望使用第 N 个约束的方式。

  2. 最近一次对 sqlite3_vtab_in(P,N,F) 的调用（且 F 为非负值）必须满足 F>=1。

如果上述任一条件或两个都不满足，那么 SQLite 会使用传统的逐个处理 IN 约束策略。 如果两个条件都为真，则传给 xFilter 方法的第 argvIndex 个参数将是一个看起来为 NULL 的 sqlite3_value，但可以将其传递给 sqlite3_vtab_in_first() 和 sqlite3_vtab_in_next() 来查找 IN 约束右侧的所有值。
