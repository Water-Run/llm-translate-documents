SQLite C 接口

评估 SQL 语句

int sqlite3_step(sqlite3_stmt*);

在使用任何 sqlite3_prepare_v2()、sqlite3_prepare_v3()、sqlite3_prepare16_v2()、sqlite3_prepare16_v3() 或传统接口 sqlite3_prepare()、sqlite3_prepare16() 之一准备好一个 预处理语句 之后，必须调用此函数一次或多次来评估该语句。

sqlite3_step() 接口的行为细节取决于语句是使用较新的 “vX” 接口 sqlite3_prepare_v3()、sqlite3_prepare_v2()、sqlite3_prepare16_v3()、sqlite3_prepare16_v2() 还是使用旧的传统接口 sqlite3_prepare() 与 sqlite3_prepare16()。建议新应用使用新的 “vX” 接口，但传统接口仍将继续受支持。

在传统接口中，返回值将是 SQLITE_BUSY、SQLITE_DONE、SQLITE_ROW、SQLITE_ERROR 或 SQLITE_MISUSE 中的一个。使用 “v2” 接口时，也可能返回其他 结果代码 或 扩展结果代码。

SQLITE_BUSY 表示数据库引擎无法获取完成工作所需的数据库锁。如果语句是一个 COMMIT，或发生在显式事务之外，那么可以重试该语句。如果语句不是 COMMIT 并且发生在显式事务内部，则应在继续之前回滚该事务。

SQLITE_DONE 表示语句已成功执行完毕。在没有先调用 sqlite3_reset() 将虚拟机重置到初始状态之前，不应再次调用 sqlite3_step()。

如果正在执行的 SQL 语句返回任何数据，那么每当有新行可供调用者处理时就会返回 SQLITE_ROW。这些值可以使用列访问函数 获取。然后再次调用 sqlite3_step() 以检索下一行数据。

SQLITE_ERROR 表示运行时错误（例如约束违规）已经发生。不应再次在该虚拟机上调用 sqlite3_step()。可以通过调用 sqlite3_errmsg() 获取更多信息。使用传统接口时，可以通过在 预处理语句 上调用 sqlite3_reset() 来获取更具体的错误代码（例如 SQLITE_INTERRUPT、SQLITE_SCHEMA、SQLITE_CORRUPT 等）。在 “v2” 接口中，更具体的错误代码由 sqlite3_step() 直接返回。

SQLITE_MISUSE 表示此例程被不当调用。可能是对已经 终结 的 预处理语句 调用，或者是对此前已返回 SQLITE_ERROR 或 SQLITE_DONE 的语句调用。也可能是同一个数据库连接在同一时间被两个或多个线程使用。

在 SQLite 3.6.23.1 及以前的所有版本中，当 sqlite3_step() 返回除 SQLITE_ROW 外的值时，在再次调用 sqlite3_step() 之前必须调用 sqlite3_reset()。如果未使用 sqlite3_reset() 重置预处理语句，则 sqlite3_step() 会返回 SQLITE_MISUSE。但在 3.6.23.1 版本（2010-03-26）之后，sqlite3_step() 在这种情况下会自动调用 sqlite3_reset()，而不再返回 SQLITE_MISUSE。这不被视为兼容性破坏，因为凡是收到 SQLITE_MISUSE 错误的应用本身就已“损坏”。可以通过编译时选项 SQLITE_OMIT_AUTORESET 来恢复旧行为。

奇怪的接口警告：在传统接口中，sqlite3_step() 在除 SQLITE_BUSY 和 SQLITE_MISUSE 外的任何错误之后总是返回通用错误码 SQLITE_ERROR。必须调用 sqlite3_reset() 或 sqlite3_finalize() 才能找到更具体的 错误代码 以更好地描述该错误。我们承认这是一个奇怪的设计。该问题已在 “v2” 接口中修复。如果你使用 sqlite3_prepare_v3()、sqlite3_prepare_v2()、sqlite3_prepare16_v2() 或 sqlite3_prepare16_v3() 来准备所有 SQL 语句，而非传统的 sqlite3_prepare() 与 sqlite3_prepare16() 接口，那么 sqlite3_step() 会直接返回更具体的 错误代码。建议使用 “vX” 接口。
