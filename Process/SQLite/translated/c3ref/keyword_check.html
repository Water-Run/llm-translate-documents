SQLite C 接口

SQL关键字检查

int sqlite3_keyword_count(void);
int sqlite3_keyword_name(int,const char**,int*);
int sqlite3_keyword_check(const char*,int);

这些例程提供对SQLite所识别的SQL语言关键字集合的访问。应用程序可以使用这些例程来判断某个特定标识符是否需要进行转义（例如用双引号包裹），以免让解析器混淆。

sqlite3_keyword_count() 接口返回SQLite所理解的不同关键字数量。

sqlite3_keyword_name(N,Z,L) 接口查找基于0的第N个关键字，并使 *Z 指向以UTF8表示的该关键字，同时将关键字的字节数写入 *L。*Z 指向的字符串不是以空字符结尾。如果N在有效范围内，sqlite3_keyword_name(N,Z,L) 例程返回 SQLITE_OK，否则返回 SQLITE_ERROR。如果 Z 或 L 为 NULL 或无效指针，则调用 sqlite3_keyword_name(N,Z,L) 会导致未定义行为。

sqlite3_keyword_check(Z,L) 接口检查 Z 所指向的 L 字节 UTF8 标识符是否为关键字，如果是则返回非零，否则返回零。

SQLite 使用的解析器比较宽容。通常可以将关键字用作标识符，只要这种用法不会导致解析歧义。例如，语句 "CREATE TABLE BEGIN(REPLACE,PRAGMA,END);" 会被 SQLite 接受，并创建一个名为 "BEGIN" 的新表，其中有三个列分别命名为 "REPLACE"、"PRAGMA" 和 "END"。尽管如此，最佳实践仍是避免将关键字用作标识符。避免关键字名称冲突的常见技巧包括：

  • 将所有标识符名称放在双引号内。这是官方的 SQL 转义标识符名称的方式。
  • 将标识符名称放在 [...] 内。这不是标准 SQL，但 SQL Server 如此做，很多程序员也使用这种技巧。
  • 让每个标识符都以字母 “Z” 开头，因为没有 SQL 关键字以 “Z” 开头。
  • 在每个标识符名称中包含一个数字。

请注意，SQLite 所理解的关键字数量可能取决于编译时选项。例如，如果 SQLite 使用 -DSQLITE_OMIT_VACUUM 选项编译，则 "VACUUM" 并不是关键字。此外，未来版本的 SQLite 可能会新增关键字。
