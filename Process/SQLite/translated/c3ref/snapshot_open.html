SQLite C 接口

在历史快照上启动只读事务

int sqlite3_snapshot_open(
  sqlite3 *db,
  const char *zSchema,
  sqlite3_snapshot *pSnapshot
);

sqlite3_snapshot_open(D,S,P) 接口会为数据库连接 D 的模式 S 启动一个新的只读事务或升级现有的只读事务，使该只读事务关联到历史的 快照 P，而不是对数据库的最新更改。sqlite3_snapshot_open() 接口在成功时返回 SQLITE_OK，失败时返回合适的 错误代码。

为了成功，当调用 sqlite3_snapshot_open(D,S,P) 时，数据库连接不能处于 自动提交模式。如果模式 S 上已经存在打开的只读事务，那么数据库句柄不能有活动的语句（即已经传递给 sqlite3_step() 但尚未 sqlite3_reset() 或 sqlite3_finalize() 的 SELECT 语句）。如果违反了上述任一条件、模式 S 不存在或快照对象无效，将返回 SQLITE_ERROR。

如果指定的快照已被 检查点 覆盖，则调用 sqlite3_snapshot_open() 会无法打开该快照。在这种情况下会返回 SQLITE_ERROR_SNAPSHOT。

如果调用此函数时已经存在打开的只读事务，那么当返回 SQLITE_ERROR、SQLITE_BUSY 或 SQLITE_ERROR_SNAPSHOT 时，原来的只读事务仍然保持打开状态（在同一数据库快照上）。如果返回的是其他错误代码，例如 SQLITE_PROTOCOL 或 SQLITE_IOERR 错误代码，则只读事务的最终状态未定义。如果返回 SQLITE_OK，则只读事务现在已在数据库快照 P 上打开。

如果数据库连接 D 不知道模式 S 的数据库文件处于 WAL 模式，则调用 sqlite3_snapshot_open(D,S,P) 会失败。如果该数据库连接以前没有对该数据库执行过 I/O，或者数据库在该连接最近一次 I/O 之后才进入 WAL 模式，那么它可能不知道该文件处于 WAL 模式。（提示：针对新打开的数据库连接运行 "PRAGMA application_id"，以便让它准备好使用快照。）

sqlite3_snapshot_open() 接口仅在使用了 SQLITE_ENABLE_SNAPSHOT 编译时选项时可用。
