SQLite C 接口

应用程序定义的页面缓存。

typedef struct sqlite3_pcache_methods2 sqlite3_pcache_methods2;
struct sqlite3_pcache_methods2 {
  int iVersion;
  void *pArg;
  int (*xInit)(void*);
  void (*xShutdown)(void*);
  sqlite3_pcache *(*xCreate)(int szPage, int szExtra, int bPurgeable);
  void (*xCachesize)(sqlite3_pcache*, int nCachesize);
  int (*xPagecount)(sqlite3_pcache*);
  sqlite3_pcache_page *(*xFetch)(sqlite3_pcache*, unsigned key, int createFlag);
  void (*xUnpin)(sqlite3_pcache*, sqlite3_pcache_page*, int discard);
  void (*xRekey)(sqlite3_pcache*, sqlite3_pcache_page*,
      unsigned oldKey, unsigned newKey);
  void (*xTruncate)(sqlite3_pcache*, unsigned iLimit);
  void (*xDestroy)(sqlite3_pcache*);
  void (*xShrink)(sqlite3_pcache*);
};

通过 sqlite3_config(SQLITE_CONFIG_PCACHE2, ...) 接口可以通过传入 sqlite3_pcache_methods2 结构的实例来注册替代的页面缓存实现。 在许多应用中，SQLite 分配的大部分堆内存都用于页面缓存。 通过使用此 API 实现自定义页面缓存，应用程序可以更好地控制 SQLite 所消耗的内存总量、内存的分配与释放方式，以及用来决定数据库文件的哪些部分被缓存以及缓存多长期限的策略。

替代的页面缓存机制属于极端手段，仅在最苛刻的应用中才需要。对于大多数用途，推荐使用内置的页面缓存。

在 sqlite3_config 调用期间，SQLite 会将 sqlite3_pcache_methods2 结构的内容复制到内部缓冲区。因此，应用程序可以在 sqlite3_config() 返回后丢弃该参数。

xInit() 方法在每次有效调用 sqlite3_initialize() 时只会被调用一次（通常在进程整个生命周期内仅调用一次）。xInit() 方法会接收到 sqlite3_pcache_methods2.pArg 值的副本。xInit() 方法的目的是设置自定义页面缓存实现所需的全局数据结构。 如果 xInit() 方法为 NULL，那么内置的默认页面缓存将替代应用程序定义的页面缓存。

xShutdown() 方法由 sqlite3_shutdown() 调用。如有需要，可以用来在进程关闭前清理任何未释放的资源。 xShutdown() 方法可以为 NULL。

SQLite 会自动对 xInit 方法的调用进行序列化，因此 xInit 方法无需线程安全。xShutdown 方法仅由 sqlite3_shutdown() 调用，也无需线程安全。所有其他方法在多线程应用中必须是线程安全的。

SQLite 在未调用 xShutdown() 的情况下永远不会对 xInit() 进行多于一次的调用。

SQLite 调用 xCreate() 方法来构建新的缓存实例。通常 SQLite 会为每个打开的数据库文件创建一个缓存实例，但这并不保证。 第一个参数 szPage 是缓存必须分配的页面的字节大小。szPage 总是 2 的幂。 第二个参数 szExtra 是与每个页面缓存条目关联的额外存储字节数。szExtra 的值小于 250。SQLite 会在每个页面中使用这额外的 szExtra 字节来存储磁盘上对应数据库页面的元数据。传入 szExtra 的值取决于 SQLite 版本、目标平台以及 SQLite 的编译方式。 xCreate() 的第三个参数 bPurgeable 为 true 时表示创建的缓存将用于缓存磁盘文件的数据库页面；如果用于内存数据库则为 false。缓存实现无需根据 bPurgeable 的值做特殊处理；它仅作为建议。 对于 bPurgeable 为 false 的缓存，SQLite 除了刻意删除页面外永远不会调用 xUnpin()。 换句话说，在 bPurgeable 为 false 的缓存上调用 xUnpin() 时，“discard” 标志总会被设置为 true。 因此，bPurgeable 为 false 的缓存永远不会包含任何未固定的页面。

SQLite 可以随时调用 xCachesize() 方法来设置传入缓存实例的建议最大缓存大小（存储页面数）。该值通过 SQLite 的 "PRAGMA cache_size" 命令配置。 与 bPurgeable 参数一样，实现不需要对此值做出响应；它仅作为建议。

xPagecount() 方法必须返回当前存储在缓存中的页面总数，包括固定和未固定的页面。

xFetch() 方法会在缓存中定位页面并返回与该页面关联的 sqlite3_pcache_page 对象的指针，或者返回 NULL。所返回的 sqlite3_pcache_page 对象的 pBuf 元素将指向一个 szPage 字节大小的缓冲区，用来存储单个数据库页面的内容。sqlite3_pcache_page 的 pExtra 元素将指向 SQLite 为每个页面缓存条目请求的 szExtra 字节额外存储空间。

要获取的页面由 key 决定。最小的 key 值是 1。在通过 xFetch 获取之后，该页面视为“固定”状态。

如果请求的页面已经存在于页面缓存中，那么页面缓存实现必须返回一个指向内容完整页面缓冲区的指针。如果请求的页面尚未缓存，则缓存实现应根据 createFlag 参数的值来决定采取何种行动：

createFlag 页面尚未缓存时的行为
0 不分配新页面。返回 NULL。
1 如果容易且方便，分配新页面。否则返回 NULL。
2 尽一切努力分配新页面。只有在几乎不可能分配新页面时才返回 NULL。

SQLite 通常会以 createFlag 为 0 或 1 调用 xFetch()。只有在先前使用 createFlag 为 1 的调用失败之后，SQLite 才会使用 createFlag 为 2。在 xFetch() 调用之间，SQLite 可能会尝试通过将固定页面的内容溢写到磁盘并同步操作系统磁盘缓存来取消固定一个或多个缓存页面。

xUnpin() 由 SQLite 调用，第二个参数为当前固定的页面指针。如果第三个参数 discard 非零，则必须将该页面从缓存中逐出。如果 discard 参数为零，则页面可以由页面缓存实现自行决定抛弃还是保留。页面缓存实现可以在任何时候选择逐出未固定页面。

缓存不能进行任何引用计数。无论之前调用了多少次 xFetch()，只要调用一次 xUnpin() 就能取消该页面的固定状态。

xRekey() 方法用于更改与作为第二个参数传入页面关联的 key 值。如果缓存中之前存在与 newKey 相关的条目，则必须将其丢弃。与 newKey 相关的任何先前缓存条目都保证未被固定。

当 SQLite 调用 xTruncate() 方法时，缓存必须丢弃所有页面编号（key）大于或等于传入的 iLimit 参数值的现有缓存条目。如果这些页面中有固定页面，它们会被隐式取消固定，也就是说可以安全地丢弃。

xDestroy() 方法用于删除由 xCreate() 分配的缓存。所有与指定缓存相关的资源都应被释放。在调用 xDestroy() 方法之后，SQLite 会将 sqlite3_pcache* 句柄视为无效，并不会将其与任何其他 sqlite3_pcache_methods2 函数一起使用。

当 SQLite 希望页面缓存尽可能释放堆内存时，会调用 xShrink() 方法。页面缓存实现没有义务释放任何内存，但表现良好的实现应该尽最大努力。
