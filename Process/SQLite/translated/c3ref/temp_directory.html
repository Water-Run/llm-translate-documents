SQLite C 接口

临时文件夹名称

SQLITE_EXTERN char *sqlite3_temp_directory;

如果这个全局变量被设置为指向某个字符串，该字符串表示一个文件夹（即目录）的名称，那么 SQLite 在使用内建的 虚拟文件系统 时创建的所有临时文件都会放在该目录下。 如果这个变量为 NULL 指针，那么 SQLite 会搜索合适的临时文件目录。

强烈不建议应用程序使用这个全局变量。 在 Windows Runtime（WinRT）上需要设置临时文件夹。 但在其他所有平台上，强烈建议应用程序不要读取或写入这个变量。 这个全局变量是为兼容旧版应用程序而保留的遗留项，应在新项目中避免使用。

在多于一个线程中同时读取或修改这个变量是不安全的。 如果在单独的线程中正在使用 数据库连接，同时读取或修改这个变量也是不安全的。 这个变量预计会在进程初始化期间设置一次，并在调用任何 SQLite 接口例程之前完成设置，并且之后保持不变。

temp_store_directory pragma 可能会修改这个变量，使其指向由 sqlite3_malloc 获取的内存。 此外，temp_store_directory pragma 总是假设这个变量指向的字符串位于由 sqlite3_malloc 获取的内存中，并且该 pragma 可能会尝试使用 sqlite3_free 释放这块内存。因此，如果直接修改这个变量，要么将其设为 NULL，要么让它指向由 sqlite3_malloc 获取的内存，否则就应该避免使用 temp_store_directory pragma。除非 temp_store_directory pragma 显式要求，否则 SQLite 不会释放 sqlite3_temp_directory 指向的内存。 如果应用程序希望释放这块内存，它必须自行负责，注意仅在销毁所有 数据库连接 对象之后才进行释放。

给 Windows Runtime 用户的提示：必须在调用 sqlite3_open 或 sqlite3_open_v2 之前设置临时目录。 否则，依赖临时文件的各种功能可能会失败。 下面是在 Windows Runtime 中使用 C++ 的示例：

LPCWSTR zPath = Windows::Storage::ApplicationData::Current->
      TemporaryFolder->Path->Data();
char zPathBuf[MAX_PATH + 1];
memset(zPathBuf, 0, sizeof(zPathBuf));
WideCharToMultiByte(CP_UTF8, 0, zPath, -1, zPathBuf, sizeof(zPathBuf),
      NULL, NULL);
sqlite3_temp_directory = sqlite3_mprintf("%s", zPathBuf);

另请参阅 对象、 常量 和 函数 的列表。
