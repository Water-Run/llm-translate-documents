SQLite C 接口

互斥方法对象

typedef struct sqlite3_mutex_methods sqlite3_mutex_methods;
struct sqlite3_mutex_methods {
  int (*xMutexInit)(void);
  int (*xMutexEnd)(void);
  sqlite3_mutex *(*xMutexAlloc)(int);
  void (*xMutexFree)(sqlite3_mutex *);
  void (*xMutexEnter)(sqlite3_mutex *);
  int (*xMutexTry)(sqlite3_mutex *);
  void (*xMutexLeave)(sqlite3_mutex *);
  int (*xMutexHeld)(sqlite3_mutex *);
  int (*xMutexNotheld)(sqlite3_mutex *);
};

此结构体的实例定义了分配和使用互斥锁所需的底层例程。

通常，SQLite 提供的默认互斥锁实现已经足够，然而应用程序可以针对特定部署或 SQLite 未提供合适实现的系统替换自定义实现。在这种情况下，应用程序会创建并填充此结构体的实例，并将其与 SQLITE_CONFIG_MUTEX 选项一起传递给 sqlite3_config()。另外，此结构体的实例还可以在使用 SQLITE_CONFIG_GETMUTEX 选项查询当前互斥锁实现时，作为输出变量使用。

此结构体定义的 xMutexInit 方法会在 sqlite3_initialize() 函数初始化系统时调用。对于每一次有效调用 sqlite3_initialize()，SQLite 都会恰好调用一次 xMutexInit 例程。

此结构体定义的 xMutexEnd 方法会在 sqlite3_shutdown() 函数关闭系统时调用。此方法的实现应当释放互斥锁方法实现获取的所有未完成资源，尤其是由 xMutexInit 方法获取的资源。对于每一次调用 sqlite3_shutdown()，xMutexEnd() 接口都会被恰好调用一次。

此结构体剩余的七个方法（xMutexAlloc、xMutexFree、xMutexEnter、xMutexTry、xMutexLeave、xMutexHeld 和 xMutexNotheld）分别实现以下接口：

  • sqlite3_mutex_alloc()
  • sqlite3_mutex_free()
  • sqlite3_mutex_enter()
  • sqlite3_mutex_try()
  • sqlite3_mutex_leave()
  • sqlite3_mutex_held()
  • sqlite3_mutex_notheld()

唯一的区别在于，上述公共 sqlite3_XXX 函数在传入 NULL 指针而不是有效互斥锁句柄时会静默忽略该调用。而该结构体定义的方法的实现并不需要处理这种情况。将 NULL 指针传入而不是有效互斥锁句柄的结果是未定义的（即可以接受实现会在传入 NULL 指针时发生段错误）。

xMutexInit() 方法必须是线程安全的。在同一进程内多次调用 xMutexInit()，且在调用期间未插入 xMutexEnd()，都不应造成任何影响。第二次及以后的 xMutexInit() 调用必须是无操作的。

xMutexInit() 不得使用 SQLite 的内存分配（sqlite3_malloc() 及其相关函数）。类似地，xMutexAlloc() 在为静态互斥锁分配时不得使用 SQLite 内存分配。然而 xMutexAlloc() 在为快速或递归互斥锁分配时可以使用 SQLite 内存分配。

当调用 sqlite3_shutdown() 时，SQLite 会调用 xMutexEnd() 方法，但前提是之前对 xMutexInit 的调用返回了 SQLITE_OK。如果 xMutexInit 在任何方面失败，期望其在返回前自行清理资源。
