SQLite C 接口

操作系统接口对象

typedef struct sqlite3_vfs sqlite3_vfs;
typedef void (*sqlite3_syscall_ptr)(void);
struct sqlite3_vfs {
  int iVersion;            /* Structure version number (currently 3) */
  int szOsFile;            /* Size of subclassed sqlite3_file */
  int mxPathname;          /* Maximum file pathname length */
  sqlite3_vfs *pNext;      /* Next registered VFS */
  const char *zName;       /* Name of this virtual file system */
  void *pAppData;          /* Pointer to application-specific data */
  int (*xOpen)(sqlite3_vfs*, sqlite3_filename zName, sqlite3_file*,
               int flags, int *pOutFlags);
  int (*xDelete)(sqlite3_vfs*, const char *zName, int syncDir);
  int (*xAccess)(sqlite3_vfs*, const char *zName, int flags, int *pResOut);
  int (*xFullPathname)(sqlite3_vfs*, const char *zName, int nOut, char *zOut);
  void *(*xDlOpen)(sqlite3_vfs*, const char *zFilename);
  void (*xDlError)(sqlite3_vfs*, int nByte, char *zErrMsg);
  void (*(*xDlSym)(sqlite3_vfs*,void*, const char *zSymbol))(void);
  void (*xDlClose)(sqlite3_vfs*, void*);
  int (*xRandomness)(sqlite3_vfs*, int nByte, char *zOut);
  int (*xSleep)(sqlite3_vfs*, int microseconds);
  int (*xCurrentTime)(sqlite3_vfs*, double*);
  int (*xGetLastError)(sqlite3_vfs*, int, char *);
  /*
  ** The methods above are in version 1 of the sqlite_vfs object
  ** definition.  Those that follow are added in version 2 or later
  */
  int (*xCurrentTimeInt64)(sqlite3_vfs*, sqlite3_int64*);
  /*
  ** The methods above are in versions 1 and 2 of the sqlite_vfs object.
  ** Those below are for version 3 and greater.
  */
  int (*xSetSystemCall)(sqlite3_vfs*, const char *zName, sqlite3_syscall_ptr);
  sqlite3_syscall_ptr (*xGetSystemCall)(sqlite3_vfs*, const char *zName);
  const char *(*xNextSystemCall)(sqlite3_vfs*, const char *zName);
  /*
  ** The methods above are in versions 1 through 3 of the sqlite_vfs object.
  ** New fields may be appended in future versions.  The iVersion
  ** value will increment whenever this happens.
  */
};

sqlite3_vfs 对象的实例定义了 SQLite 核心与下层操作系统之间的接口。“vfs”代表“虚拟文件系统”。有关详细信息，请参见 VFS 文档。

VFS 接口有时通过在末尾添加新方法来扩展。每次扩展发生时，iVersion 字段都会增加。iVersion 的初始值在 2007-09-04 的 SQLite 3.5.0 版本中是 1，随后在 2010-07-21 的 SQLite 3.7.0 版本中增加到 2，再在 2011-04-12 的 SQLite 3.7.6 版本中增加到 3。未来版本可能会向 sqlite3_vfs 对象附加额外字段并再次提高 iVersion 值。请注意，由于一个疏忽，sqlite3_vfs 对象的结构在 2008-07-16 从 SQLite 3.5.9 版过渡到 3.6.0 版时发生了变化，但 iVersion 字段并未随之增加。

szOsFile 字段是此 VFS 使用的子类化 sqlite3_file 结构的大小。mxPathname 是该 VFS 中路径名的最大长度。

已注册的 sqlite3_vfs 对象通过 pNext 指针组成一个链表。sqlite3_vfs_register() 和 sqlite3_vfs_unregister() 接口以线程安全的方式管理该链表。sqlite3_vfs_find() 接口用于搜索链表。无论是应用程序代码还是 VFS 实现都不应使用 pNext 指针。

pNext 字段是 sqlite3_vfs 结构中唯一会被 SQLite 修改的字段。SQLite 仅在持有特定静态互斥锁时访问或修改该字段。对象注册后，应用程序绝不应修改 sqlite3_vfs 对象中的任何内容。

zName 字段保存 VFS 模块的名称，该名称在所有 VFS 模块中必须唯一。

SQLite 保证传递给 xOpen 的 zFilename 参数要么是 NULL 指针，要么是在 xFullPathname() 返回的字符串基础上再加了可选后缀。如果 zFilename 参数附加了后缀，则后缀由一个 "-" 字符以及最多 11 个字母数字和/或 "-" 组成。SQLite 进一步保证在调用 xClose() 之前该字符串是有效且不被修改的。鉴于前述内容，sqlite3_file 可以安全地保存该文件名的指针，如果需要的话可以记住文件名。如果传递给 xOpen 的 zFilename 参数是 NULL 指针，则 xOpen 必须为该文件自行生成临时名称。每当 zFilename 参数为 NULL 时，flags 参数也会包含 SQLITE_OPEN_DELETEONCLOSE。

传递给 xOpen() 的 flags 参数包含传递给 sqlite3_open_v2() 的 flags 参数中的所有位。或者如果使用 sqlite3_open() 或 sqlite3_open16()，那么 flags 至少包含 SQLITE_OPEN_READWRITE | SQLITE_OPEN_CREATE。如果 xOpen() 以只读方式打开文件，则会在 *pOutFlags 中设置 SQLITE_OPEN_READONLY。*pOutFlags 中的其他位也可能被设置。

SQLite 还会根据正在打开的对象在 xOpen() 调用中添加以下标志之一：

  • SQLITE_OPEN_MAIN_DB
  • SQLITE_OPEN_MAIN_JOURNAL
  • SQLITE_OPEN_TEMP_DB
  • SQLITE_OPEN_TEMP_JOURNAL
  • SQLITE_OPEN_TRANSIENT_DB
  • SQLITE_OPEN_SUBJOURNAL
  • SQLITE_OPEN_SUPER_JOURNAL
  • SQLITE_OPEN_WAL

文件 I/O 实现可以使用对象类型标志来改变处理文件的方式。例如，一个不关心崩溃恢复或回滚的应用程序可能会使日志文件的打开操作成为空操作。对该日志的写入也将是空操作，任何试图读取日志的行为都会返回 SQLITE_IOERR。或者实现可能会识别出数据库文件会以页面对齐的扇区执行随机顺序的读写，并据此设置其 I/O 子系统。

SQLite 还可能在 xOpen 方法中添加以下标志之一：

  • SQLITE_OPEN_DELETEONCLOSE
  • SQLITE_OPEN_EXCLUSIVE

SQLITE_OPEN_DELETEONCLOSE 标志意味着文件在关闭时应被删除。TEMP 数据库及其日志、临时数据库和子日志会设置 SQLITE_OPEN_DELETEONCLOSE。

SQLITE_OPEN_EXCLUSIVE 标志总是与 SQLITE_OPEN_CREATE 标志一起使用，这两者都直接类似于 POSIX open() API 中的 O_EXCL 和 O_CREAT 标志。SQLITE_OPEN_EXCLUSIVE 与 SQLITE_OPEN_CREATE 配合使用时用于指示该文件必须被创建，如果文件已存在则会报错。它并非用于表示应以独占访问方式打开文件。

SQLite 至少会为传入 xOpen 的第三个参数所指向的 sqlite3_file 结构分配 szOsFile 字节的内存。xOpen 方法无需自己分配该结构；只需填写它即可。注意 xOpen 方法必须将 sqlite3_file.pMethods 设置为有效的 sqlite3_io_methods 对象或为 NULL。即使打开失败，xOpen 也必须这样做。无论 xOpen 调用是否成功，SQLite 都期望 sqlite3_file.pMethods 在 xOpen 返回后都是有效的。

传递给 xAccess() 的 flags 参数可以是 SQLITE_ACCESS_EXISTS，用于测试文件是否存在；也可以是 SQLITE_ACCESS_READWRITE，用于测试文件是否可读可写；也可以是 SQLITE_ACCESS_READ，用于测试文件是否至少可读。SQLITE_ACCESS_READ 标志实际上从未被使用，也未在 SQLite 内置的 VFS 中实现。文件名由第二个参数指定，可以是目录。xAccess 方法在成功时返回 SQLITE_OK，如果发生 I/O 错误或第二个参数提供的文件名非法，则返回某个非零错误代码。如果返回 SQLITE_OK，则在 *pResOut 中写入非零或零值，以指示文件是否可访问。

SQLite 始终为 xFullPathname 的输出缓冲区分配至少 mxPathname+1 字节。输出缓冲区的确切大小也作为参数传入该方法。如果输出缓冲区不够大，就应返回 SQLITE_CANTOPEN。由于 SQLite 将此视为致命错误，VFS 实现应尝试通过将 mxPathname 设置为足够大的值来避免这种情况。

xRandomness()、xSleep()、xCurrentTime() 和 xCurrentTimeInt64() 接口严格来说并不是文件系统的一部分，但为了完整性而包含在 VFS 结构中。xRandomness() 函数尝试在 zOut 中返回 nByte 字节的高质量随机数。返回值是实际获得的随机字节数。xSleep() 方法会使调用线程至少休眠给定的微秒数。xCurrentTime() 方法作为浮点值返回当前日期和时间的儒略日数。xCurrentTimeInt64() 方法则以整数形式返回儒略日数乘以 86400000（24 小时中的毫秒数）。如果 xCurrentTimeInt64() 可用（iVersion 为 2 或更高且函数指针非 NULL），SQLite 会使用该方法来获取当前日期和时间；如果不可用，则退回到 xCurrentTime()。

xSetSystemCall()、xGetSystemCall() 和 xNestSystemCall() 接口不会被 SQLite 核心使用。这些可选接口由某些 VFS 提供以便测试 VFS 代码。通过用自己控制的函数覆盖系统调用，测试程序可以模拟其他方式难以或不可能触发的故障和错误条件。可被覆盖的系统调用集合因 VFS 而异，即使是相同 VFS 的不同版本也可能会不同。使用这些接口的应用程序必须准备好面对这些接口中的任意一个或全部为 NULL，或者它们的行为在每个发行版之间发生变化。当 VFS 的 iVersion 小于 3 时，应用程序不得尝试访问这些方法。
