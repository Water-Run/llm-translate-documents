SQLite C 接口

确定 SQL 语句是否写入数据库

int sqlite3_stmt_readonly(sqlite3_stmt *pStmt);

sqlite3_stmt_readonly(X) 接口仅在 预处理语句 X 未对数据库文件内容做出任何直接更改时才返回 true（非零）。

请注意，应用程序自定义的 SQL 函数 或 虚拟表 可能作为副作用间接修改数据库。例如，如果应用程序定义了一个调用 sqlite3_exec() 的函数 “eval()”，那么以下 SQL 语句将通过副作用修改数据库文件：

SELECT eval('DELETE FROM t1') FROM t2;

但因为 SELECT 语句 本身没有直接更改数据库文件，sqlite3_stmt_readonly() 仍然会返回 true。

诸如 BEGIN 语句、COMMIT 语句、ROLLBACK 语句、 SAVEPOINT 语句 和 RELEASE 语句 之类的事务控制语句会导致 sqlite3_stmt_readonly() 返回 true， 因为它们本身并不真正修改数据库，而是控制其他语句何时修改数据库。ATTACH 语句 和 DETACH 语句 也会导致 sqlite3_stmt_readonly() 返回 true， 因为这些语句虽然更改了数据库连接的配置，却不会更改磁盘上数据库文件的内容。 sqlite3_stmt_readonly() 接口对 BEGIN 语句 返回 true，因为 BEGIN 语句 只设置内部标志， 但 BEGIN IMMEDIATE 命令 与 BEGIN EXCLUSIVE 命令 会触及数据库，因此 sqlite3_stmt_readonly() 对这些命令返回 false。

如果语句可能更改数据库文件，即便只是存在这种可能性，该例程也会返回 false。false 的返回值并不保证语句一定修改数据库文件。 例如，一个 UPDATE 语句可能带有使其变成无操作的 WHERE 子句，但 sqlite3_stmt_readonly() 的结果仍然为 false。同样，如果表已经存在，CREATE TABLE IF NOT EXISTS 语句虽然在事实上是只读的无操作，但 sqlite3_stmt_readonly() 对此类语句仍然返回 false。

如果预处理语句 X 是一个 EXPLAIN 或 EXPLAIN QUERY PLAN 语句，那么 sqlite3_stmt_readonly(X) 会返回与省略 EXPLAIN 或 EXPLAIN QUERY PLAN 前缀时相同的值。
