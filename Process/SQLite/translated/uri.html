统一资源标识符

1. SQLite 中的 URI 文件名

自 3.7.7 版（2011-06-23）开始，传递给 sqlite3_open()、sqlite3_open16() 与 sqlite3_open_v2() 接口 以及 ATTACH 命令的 SQLite 数据库文件参数可以指定为普通文件名或统一资源标识符（URI）。 使用 URI 文件名的优势在于可以通过 URI 上的查询参数控制新创建的数据库连接的细节。 例如，可以使用 "vfs=" 查询参数指定备用的 VFS。 或者，也可以通过将 "mode=ro" 作为查询参数以只读方式打开数据库。

2. 向后兼容性

为了保持对旧版应用的完全向后兼容性，URI 文件名功能默认处于禁用状态。 URI 文件名可以通过编译时选项 SQLITE_USE_URI=1 或 SQLITE_USE_URI=0 启用或禁用。 URI 文件名的编译时设置可以在程序启动时通过 sqlite3_config(SQLITE_CONFIG_URI,1) 或 sqlite3_config(SQLITE_CONFIG_URI,0) 配置调用进行更改。 无论编译时或启动时的设置如何，都可以通过在传递给 sqlite3_open_v2(N,P,F,V) 的 F 参数位集中包含 SQLITE_OPEN_URI 位来为单个数据库连接启用 URI 文件名。

如果在最初打开数据库连接时认可了 URI 文件名，则 ATTACH 语句也会认可 URI 文件名。 同样地，如果在首次打开数据库连接时没有识别 URI 文件名，则 ATTACH 也不会识别它们。

由于 SQLite 无论 URI 设置为何，总是将任何不以 "file:" 开头的文件名解释为普通文件名， 并且实际文件名以 "file:" 开头的情况极为罕见，因此即便当前尚未使用 URI 文件名，大多数应用也可以安全地启用 URI 处理。

3. URI 格式

根据 RFC 3986，一个 URI 由 scheme、authority、路径、查询字符串和片段组成。 scheme 始终是必需的。authority 或路径中的一个也必须存在。 查询字符串和片段是可选的。

SQLite 使用 "file:" URI 语法来标识数据库文件。 SQLite 力求以与流行浏览器如 Firefox、 Chrome、 Safari、 Internet Explorer 与 Opera 相同的方式来解析 file: URI， 以及与命令行程序如 Windows 的 "cmd start" 或 "powershell start"、 或 macOS 的 "open" 及 Linux 的 "xdg-open" 命令相同。 以下是 URI 解析规则的简要总结：

  • URI 的 scheme 必须是 "file:"。任何其他 scheme 都会导致输入被视为普通文件名。
  • authority 可以省略、为空或为 "localhost"。任何其他 authority 都会导致错误。 例外：如果 SQLite 使用 SQLITE_ALLOW_URI_AUTHORITY 编译，则除 "localhost" 外的 authority 值会被作为 UNC 文件名传递给底层操作系统。
  • 如果 authority 存在，则路径是可选的。如果省略 authority，则路径是必需的。
  • 查询字符串是可选的。如果存在查询字符串，则所有查询参数都会传递到底层 VFS 的 xOpen 方法。
  • 片段是可选的。如果存在，则会被忽略。

路径、查询字符串或片段中可以出现零个或多个 "%HH"（其中 H 表示任意十六进制数字）形式的转义序列。

不是格式良好的 URI 的文件名会被解释为普通文件名。

URI 以 UTF8 文本进行处理。 sqlite3_open16() 的文件名参数会在处理前从 UTF16 本地字节序转换为 UTF8。

3.1. URI 路径

URI 的路径部分指定要打开的 SQLite 数据库所在的磁盘文件。 如果省略路径部分，则数据库存储在一个临时文件中，该文件会在数据库连接关闭时被自动删除。 如果 authority 部分存在，则路径始终是绝对路径。 如果省略 authority，则路径在以 "/" 字符（ASCII 码 0x2f）开头时为绝对路径，否则为相对路径。 在 Windows 上，如果绝对路径以 "/X:/" 开头，其中 X 是任意单个 ASCII 字母（"a" 到 "z" 或 "A" 到 "Z"），则 "X:" 被理解为包含该文件的卷的驱动器字母，而不是顶层目录。

普通文件名通常可以按照下列步骤转换为等价的 URI。唯一的例外是带有驱动器字母的相对 Windows 路径无法直接转换为 URI，必须先改为绝对路径。

  1. 将所有 "?" 字符转换为 "%3f"。
  2. 将所有 "#" 字符转换为 "%23"。
  3. 仅在 Windows 上，将所有 "\" 字符转换为 "/"。
  4. 将所有两个或多个连续的 "/" 字符序列转换为单个 "/" 字符。
  5. 仅在 Windows 上，如果文件名以驱动器字母开头，则在前面加一个 "/" 字符。
  6. 在前面添加 "file:" 方案。

3.2. 查询字符串

URI 文件名可以选择性地后跟一个查询字符串。 查询字符串由第一个 "?" 字符之后但不包括以 "#" 开头的可选片段的文本组成。 查询字符串被划分为键/值对，我们通常将它们称为“查询参数”。 键/值对由单个 "&" 字符分隔。 键位于前面，并通过单个 "=" 字符与值分隔。 键和值都可以包含 %HH 转义序列。

查询参数的文本附加到 VFS 的 xOpen 方法的文件名参数后面。 查询参数中的任何 %HH 转义序列都会在附加到 xOpen 文件名之前被解析。 单个零字节将 xOpen 文件名参数与第一个查询参数的键、每个键与其值，以及后续每个键与前一个值分隔开。 附加到 xOpen 文件名的查询参数列表以一个长度为零的键作为结尾。 注意：查询参数的值可以是空字符串。

3.3. 已识别的查询参数

某些查询参数会被 SQLite 核心解释并用于修改新连接的特性。 即便这些查询参数已被 SQLite 核心读取和解释，它们仍会一并传递到 VFS 的 xOpen 方法。

截至 3.15.0 版（2016-10-14），SQLite 已识别如下查询参数。 未来可能会增加新的查询参数。

cache=shared
cache=private

cache 查询参数决定新数据库是以共享缓存模式还是私有缓存打开。

immutable=1

immutable 查询参数是一个布尔值，用于告知 SQLite 底层数据库文件存储在只读介质上且不可修改，即使是具有更高权限的其他进程也不能修改。 SQLite 始终以只读方式打开不可变数据库文件，并在这些文件上跳过所有文件锁定和更改检测。 如果该查询参数（或 xDeviceCharacteristics 中的 SQLITE_IOCAP_IMMUTABLE 位）声明数据库文件不可变，但该文件仍然发生更改，则 SQLite 可能返回错误的查询结果和/或 SQLITE_CORRUPT 错误。

mode=ro
mode=rw
mode=rwc
mode=memory

mode 查询参数分别决定新数据库是以只读、读写、读写并在不存在时创建，还是作为一个永远不与磁盘交互的纯内存数据库打开。

modeof=filename

在 unix 系统上调用 sqlite3_open_v2() 创建新数据库文件时，SQLite 会尝试让新数据库文件的权限与现有文件 "filename" 相匹配。

nolock=1

nolock 查询参数是一个布尔值，当其为 true 时会禁用 VFS 的 xLock、xUnlock 与 xCheckReservedLock 方法的所有调用。 例如，在尝试访问一个不支持文件锁定的文件系统上的文件时，可能会使用 nolock 查询参数。 注意：如果两个或更多 数据库连接 尝试与同一个 SQLite 数据库交互，并且其中一个或多个连接启用了 "nolock"，则可能导致数据库损坏。 只有当应用可以保证对数据库的写入是串行化的情况下，才应使用 "nolock" 查询参数。

psow=0
psow=1

psow 查询参数会覆盖正被打开的数据库文件的 安全覆盖 属性。 psow 查询参数在默认的 Windows 与 unix VFS 上有效，但对其他专有或非标准的 VFS 可能无效。

vfs=NAME

vfs 查询参数使数据库连接通过名为 NAME 的 VFS 打开。 如果 NAME 不是内置于 SQLite 的 VFS 名称，也不是通过 sqlite3_vfs_register() 预先注册的名称，则打开尝试会失败。
