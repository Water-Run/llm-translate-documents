1. 概述

SQLite 支持以下七个标量日期和时间函数：

  1. date(time-value, modifier, modifier, ...)
  2. time(time-value, modifier, modifier, ...)
  3. datetime(time-value, modifier, modifier, ...)
  4. julianday(time-value, modifier, modifier, ...)
  5. unixepoch(time-value, modifier, modifier, ...)
  6. strftime(format, time-value, modifier, modifier, ...)
  7. timediff(time-value, time-value)

前六个日期和时间函数可接受一个可选的 时间值 作为参数，后面可以接零个或多个 修饰符。 strftime() 函数还将格式字符串作为第一个参数。 timediff() 函数严格接受两个参数，并且这两个参数都是 时间值。

SQLite 没有专用的日期/时间数据类型。相反， 日期和时间值可以以以下任意形式存储：

ISO-8601 一个文本字符串，必须是 下面第 1 至 10 项 中所示的 ISO 8601 日期/时间格式之一。 示例：'2025-05-29 14:16:00'
儒略日数 自 -4713-11-24 12:00:00 起的天数，包括小数部分 示例：2460825.09444444
Unix 时间戳 自 1970-01-01 00:00:00 起的秒数，包括小数部分 示例：1748528160

这三种格式统称为 时间值。 所有日期时间函数都接受 ISO-8601 文本或儒略日数格式的时间值。 也可以通过添加可选的修饰符 'auto' 或 'unixepoch' 来使它们接收 unix 时间戳。 由于 timediff() 函数不接受修饰符，因此它只能使用 ISO-8601 和儒略日数格式的时间值。

date() 函数以文本形式返回日期，格式为 YYYY-MM-DD。

time() 函数以文本形式返回时间，格式为 HH:MM:SS；如果 使用了 subsec 修饰符，则格式为 HH:MM:SS.SSS。

datetime() 函数以 YYYY-MM-DD HH:MM:SS 的格式返回日期和时间； 如果使用了 subsec 修饰符，则格式为 YYYY-MM-DD HH:MM:SS.SSS。

julianday() 函数返回儒略日—— 自格林威治 4714 年前 11 月 24 日中午起的累积天数 （参照 推历格里历）。

unixepoch() 函数返回 unix 时间戳——自 1970-01-01 00:00:00 UTC 起的秒数。 unixepoch() 通常返回整数秒，但使用可选的 subsec 修饰符 时 会返回带小数部分的浮点数，表示秒数的小数部分。

strftime() 函数根据作为第一个参数指定的格式字符串 返回格式化的日期。 该格式字符串支持标准 C 库中 strftime() 函数 的常见替换符， 并新增了两个替换符 %f 和 %J。 以下是在 3.46.0 版（2024-05-23）及以后的 SQLite 中有效的 strftime() 替换符的完整列表。 早期版本可能不支持全部替换符。如果出现未定义或不受支持的替换符，结果将为 NULL。

%d 月中的第几天：01-31
%e 无前导零的月份中的第几天：1-31
%f 秒的小数部分：SS.SSS
%F ISO 8601 日期：YYYY-MM-DD
%G 与 %V 对应的 ISO 8601 年
%g 与 %V 对应的两位 ISO 8601 年
%H 小时：00-24
%I 12 小时制的小时：01-12
%j 年中的第几天：001-366
%J 儒略日数（带小数）
%k 无前导零的小时：0-24
%l 无前导零的 %I：1-12
%m 月份：01-12
%M 分钟：00-59
%p 根据小时返回 "AM" 或 "PM"
%P 根据小时返回 "am" 或 "pm"
%R ISO 8601 时间：HH:MM
%s 自 1970-01-01 起的秒数
%S 秒：00-59
%T ISO 8601 时间：HH:MM:SS
%U 年中的第几周（00-53）——第 1 周从第一个星期日开始
%u 周几：1-7，星期一为 1
%V ISO 8601 年中的第几周
%w 周几：0-6，星期日为 0
%W 年中的第几周（00-53）——第 1 周从第一个星期一开始
%Y 年份：0000-9999
%% %

其他日期和时间函数可以用 strftime() 来表示：

函数对应的 strftime()
date(...) strftime('%F', ...)
time(...) strftime('%T', ...)
datetime(...) strftime('%F %T', ...)
julianday(...) CAST(strftime('%J', ...) as REAL)
unixepoch(...) CAST(strftime('%s', ...) as INT)

date()、time() 和 datetime() 函数都返回文本，因此它们的 strftime() 等价形式是精确的。然而，julianday() 和 unixepoch() 函数返回数值。它们的 strftime() 等价形式 则返回相应数字的文本表示。

提供除 strftime() 以外函数的主要原因 是为了方便性和效率。julianday() 和 unixepoch() 分别返回实数和整数值，因此不会产生使用 strftime() 中 '%J' 或 '%s' 格式说明符时所带来的格式转换开销或精度损失。

timediff(A,B) 函数返回一个字符串，描述必须加到 B 上 才能得到时间 A 所需的时间量。timediff() 的输出格式是：

(+|-)YYYY-MM-DD HH:MM:SS.SSS

这个时间差字符串也可以作为其他 日期/时间函数的修饰符。以下不变量对任意时间值 A 和 B 成立：

datetime(A) = datetime(B, timediff(A,B))

月份和年份的长度不同。二月比三月短。 闰年比平年长。timediff() 的输出考虑了这些差异。 timediff() 函数旨在提供人类友好的时间跨度描述。 如果想知道 A 和 B 之间相差多少天或秒，可以始终选择以下方式之一：

SELECT julianday(B) - julianday(A);
SELECT unixepoch(B) - unixepoch(A);

即使 A 和 B 跨越的天数不同，timediff(A,B) 也可能 返回相同的结果——这取决于起始日期。比如，下面两个 timediff() 调用都返回 相同的结果（"-0000-01-00 00:00:00.000"），尽管第一个跨度为 28 天，第二个为 31 天：

SELECT timediff('2023-02-15','2023-03-15');
SELECT timediff('2023-03-15','2023-04-15');

总结：如果希望得到人类友好的时间跨度，请使用 timediff()。 如果需要精确的时间差（以天或秒为单位），请使用两个 julianday() 或 unixepoch() 调用之间的差值。

2. 时间值

时间值可以采用下列任意一种格式。这个值通常是字符串，但在第 12 种格式中也可以是整数或浮点数。

  1. YYYY-MM-DD
  2. YYYY-MM-DD HH:MM
  3. YYYY-MM-DD HH:MM:SS
  4. YYYY-MM-DD HH:MM:SS.SSS
  5. YYYY-MM-DDTHH:MM
  6. YYYY-MM-DDTHH:MM:SS
  7. YYYY-MM-DDTHH:MM:SS.SSS
  8. HH:MM
  9. HH:MM:SS
  10. HH:MM:SS.SSS
  11. now
  12. DDDDDDDDDD

在第 5 至 7 种格式中，“T”是分隔日期和时间的字面字符，这是 ISO-8601 所要求的。 仅指定时间的第 8 至 10 种格式假设日期为 2000-01-01。第 11 种格式，即字符串 'now'，会被转换为当前的日期和时间，该值来自当前所用 sqlite3_vfs 对象的 xCurrentTime 方法。 针对日期和时间函数的 'now' 参数，在同一次 sqlite3_step() 调用中，多次调用返回相同的值。 使用的是 协调世界时 (UTC)。 第 12 种格式是以整数或浮点值表示的 儒略日数。 如果第 12 种格式后紧跟 'auto' 或 'unixepoch' 修饰符， 则该格式可能被解释为 unix 时间戳。

第 2 至 10 种格式之后可选择性地加上时区指示符，形如 "[+-]HH:MM" 或简单的 "Z"。日期和时间函数在内部使用 UTC（又称“祖鲁”时间）， 因此 "Z" 后缀没有效果。任何非零的 "HH:MM" 后缀都会从所指示的日期时间中减去相应的时间，以计算祖鲁时间。 例如，下列所有时间值是等价的：

2013-10-07 08:23:19.120
2013-10-07T08:23:19.120Z
2013-10-07 04:23:19.120-04:00
2456572.84952685

在第 4、7 和 10 种格式中，秒的小数部分 SS.SSS 可以在小数点后有一位或多位数字。 示例中仅展示了三位，因为只有前三位对结果有效，但输入字符串可以少于或多于三位， 日期/时间函数仍然能够正确处理。同样，第 12 种格式显示了 10 位有效数字，但日期/时间函数实际上在所需的任何位数上都能正确接受儒略日数。

ISO-8601 支持多种替代的日期/时间格式， 但 SQLite 只支持上文明确列出的几种。

在除 timediff() 之外的所有函数中， 时间值（以及所有修饰符）都可以省略，这种情况下会默认使用 'now'。

3. 修饰符

对于除 timediff() 之外的所有日期/时间函数， 时间值参数后面可以接零个或多个修饰符，以便修改日期和/或时间。每个修饰符 会将它左侧的时间值转换成新的时间值。 修饰符按从左到右的顺序应用；顺序很重要。 可用的修饰符如下。

  1. NNN days
  2. NNN hours
  3. NNN minutes
  4. NNN seconds
  5. NNN months
  6. NNN years
  7. ±HH:MM
  8. ±HH:MM:SS
  9. ±HH:MM:SS.SSS
  10. ±YYYY-MM-DD
  11. ±YYYY-MM-DD HH:MM
  12. ±YYYY-MM-DD HH:MM:SS
  13. ±YYYY-MM-DD HH:MM:SS.SSS
  14. ceiling
  15. floor
  16. start of month
  17. start of year
  18. start of day
  19. weekday N
  20. unixepoch
  21. julianday
  22. auto
  23. localtime
  24. utc
  25. subsec
  26. subsecond

前十三个修饰符（1 到 13）会将指定的时间量加到它左侧的日期和时间上。 修饰符名称末尾的 's' 字符在 1 到 6 中是可选的。 NNN 的值可以是任意浮点数，并可带可选的 '+' 或 '-' 前缀。

时间偏移修饰符（7 到 13）会按指定的年、月、日、小时、分钟和/或秒来移动时间值。 对于格式 10 到 13，必须提供初始的 "+" 或 "-"；而格式 7、8、9 的 "+" 或 "-" 是可选的。 变化按从左到右的顺序进行。首先按 YYYY 调整年，然后按 MM 调整月，再按 DD 调整日，依此类推。 timediff(A,B) 返回的时间偏移就是格式 13，它会将时间值 B 调整到 A。

由于每个月或每年的天数不同，按月或按年移动日期时可能会产生歧义。 例如，2024-02-29 的下一年是哪一天？是 2025-02-28 还是 2025-03-01？ 再比如，2023-12-31 之后两个月是哪一天？是 2024-02-29 还是 2024-03-02？ 对于这种歧义没有统一的解决方案，因此提供了 "ceiling" 和 "floor" 修饰符（14 和 15）， 让程序员自行选择。如果时间偏移后紧跟 "ceiling"，则会选择较晚的日期作为歧义的解决方式。 "floor" 修饰符则会回退到前一个月的最后一天。默认行为是 "ceiling"。

"start of" 修饰符（16 至 18）会将日期向前移动到其所属月、年或日的起始位置。

"weekday" 修饰符会将日期向前推进（如有必要），直到达到指定的 weekday N。 星期日为 0，星期一为 1，依此类推。 如果日期已经处于目标星期，"weekday" 修饰符不会更改日期。

"unixepoch" 修饰符（20）只在紧跟在 DDDDDDDDDD 格式的时间值后面时有效。 此修饰符会让该 DDDDDDDDDD 不再按儒略日数解释，而是按 Unix 时间——自 1970 年起的秒数。 如果 "unixepoch" 修饰符后面没有紧跟表达自 1970 年起秒数的 DDDDDDDDDD 格式的时间值， 或者其他修饰符将 "unixepoch" 与前面的 DDDDDDDDDD 分隔开来，则行为未定义。

"julianday" 修饰符必须紧跟初始时间值，该时间值必须采用 DDDDDDDDD 格式。 其他任何用法都会报错并导致函数返回 NULL。 'julianday' 修饰符会强制将该数值按照儒略日数解析。 由于这是默认行为，因此 'julianday' 修饰符几乎等同于无操作。唯一的区别是， 添加 'julianday' 可以强制使用 DDDDDDDDD 时间值格式，如果以其他格式表示时间值则会返回 NULL。

"auto" 修饰符必须紧跟初始时间值。 如果时间值是数字（DDDDDDDDDD 格式），那么 'auto' 修饰符会根据其大小将时间值解释为儒略日数或 unix 时间戳。 如果数值介于 0.0 与 5373484.499999 之间，则解释为儒略日数（对应的日期范围是 -4713-11-24 12:00:00 到 9999-12-31 23:59:59，含端点）。 对于在有效儒略日数范围之外但在 -210866760000 到 253402300799 之间的数值，'auto' 修饰符会将其解释为 unix 时间戳。 其他数值则超出范围，导致返回 NULL。在 ISO 8601 文本时间值中，'auto' 修饰符等同于无操作。 "auto" 修饰符旨在处理数据库中无法确定所存储时间值格式的情况，以及同一列在不同行中存储了不同格式时间值的情况。 'auto' 修饰符会自动选择合适的格式。然而，也存在一些歧义。1970 年前 63 天的 unix 时间戳会被解释为儒略日数。 当数据集中可以确定不存在该范围内的日期时，'auto' 修饰符非常有用，但对于可能使用 1970 年头几个月日期的应用不宜使用。

"localtime" 修饰符假定其左侧的时间值是协调世界时 (UTC)，并将其转换为本地时间。 如果 "localtime" 紧跟的时间不是 UTC，则行为未定义。 "utc" 修饰符则与 "localtime" 相反， 它假定其左侧的时间值处于本地时区，并将其转换为 UTC。 如果左侧时间不是本地时间，则 "utc" 的结果未定义。

"subsecond" 修饰符（也可简写为 "subsec"）可以提高 datetime()、time() 和 unixepoch() 的输出精度， 以及 strftime() 的 "%s" 格式字符串。"subsecond" 修饰符对其他日期/时间函数无效。 当前实现将输出精度从秒提升到毫秒，但未来版本可能会支持更高精度。 当 "subsec" 与 datetime() 或 time() 一同使用时， 结尾的秒字段后会加一个小数点和一位或多位数字以表示秒的小数部分。 当 "subsec" 用于 unixepoch() 时， 返回值是自 1970-01-01 起的秒数（包括小数部分）。 "subsecond" 与 "subsec" 修饰符有一个特殊属性： 它们可以作为日期/时间函数的第一个参数（或 strftime() 中格式字符串之后的第一个参数）出现。 这种情况下，原本应处于第一个参数位置的时间值会被视为 "now"。 例如，获取当前 unix 时间戳（带毫秒精度）的简写方式是：

SELECT unixepoch('subsec');

4. 示例

计算当前日期。

SELECT date();

计算当前月的最后一天。

SELECT date('now','start of month','+1 month','-1 day');

给定 unix 时间戳 1092941466，计算对应的日期和时间。

SELECT datetime(1092941466, 'unixepoch');
SELECT datetime(1092941466, 'auto'); -- 1970 年初无效！

给定 unix 时间戳 1092941466，并考虑本地时区，计算对应的日期和时间。

SELECT datetime(1092941466, 'unixepoch', 'localtime');

计算当前的 unix 时间戳。

SELECT unixepoch();
SELECT strftime('%s');

计算自美国独立宣言签署以来的天数。

SELECT julianday('now') - julianday('1776-07-04');

计算自 2004 年某一时刻以来的秒数：

SELECT unixepoch() - unixepoch('2004-01-01 02:34:56');

计算本年度十月第一个星期二的日期。

SELECT date('now','start of year','+9 months','weekday 2');

以毫秒精度计算自 unix 纪元以来的秒数：

SELECT (julianday('now') - 2440587.5)*86400.0;
SELECT unixepoch('now','subsec');

如果亚伯拉罕·林肯还活着，他现在会多大年龄：

SELECT timediff('now','1809-02-12');

5. 注意事项与已知问题

本地时间的计算很大程度上取决于政治决策，因此对所有地区都正确实现非常困难。 在本实现中，使用标准 C 库函数 localtime_r() 来辅助本地时间的计算。 localtime_r() 通常仅对 1970 至 2037 年之间的年份有效。 对于该范围之外的日期，SQLite 会尝试将年份映射到该范围内的等价年份， 执行计算后再映射回原年份。

这些函数仅对 0000-01-01 00:00:00 至 9999-12-31 23:59:59 之间的日期有效 （儒略日数范围为 1721059.5 到 5373484.5）。 对于该范围之外的日期，这些函数的结果未定义。

非 Vista 的 Windows 平台只支持一套夏令时规则，Vista 只支持两套。 因此在这些平台上，对历史夏令时的计算会不正确。 例如，在美国，2007 年的夏令时规则发生了变化。 非 Vista Windows 平台会将新的 2007 年夏令时规则应用到所有过去的年份； 而 Vista 则在一定程度上能够正确处理回溯到 1986 年（当时规则也发生过变化）的情况。

所有内部计算假定使用的是 格里历 系统。 同时也假定每一天正好包含 86400 秒；不考虑闰秒。
