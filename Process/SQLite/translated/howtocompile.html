如何编译 SQLite
目录

2. 合并源与独立源文件

1. 概述

请按如下方式编译 SQLite：

  • Unix 和 Mac → ./configure && make
  • Windows → nmake /f Makefile.msc

上述命令将构建合并源码和命令行界面（又名：CLI）。没有（必需的）依赖项，因此只要安装了编译器，以上命令就应该可以运行。无需第三方软件。详情请参见：

  • https://sqlite.org/src/doc/trunk/doc/compile-for-unix.md
  • https://sqlite.org/src/doc/trunk/doc/compile-for-windows.md

但本文并不是提供一步一步的 SQLite 编译配方。相反，本文介绍 SQLite 编译背后的原理。典型的编译命令作为示例提供，期望应用程序开发者可以将这些示例作为制定自己自定义编译流程的参考。换句话说，本文提供的是思路和洞见，而不是交钥匙式的解决方案。因此，如果上面概述中的命令没有告诉你想知道的内容，请继续阅读。

2. 合并源与独立源文件

SQLite 由超过一百个 C 代码文件和脚本构成，分布在多个目录中。SQLite 的实现是纯 ANSI-C 的，但许多 C 语言源代码文件在被合并到最终的 SQLite 库之前，都会由辅助的 C 程序以及 AWK、SED 和 TCL 脚本生成或转换。构建这些必要的 C 程序并转换和/或创建 SQLite 的 C 源代码是一个复杂的过程。

为简化处理，SQLite 还提供了预打包的合并源码文件：sqlite3.c。该合并源码是一个实现整个 SQLite 库的单一 ANSI-C 文件。使用合并源码要容易得多。所有内容都包含在一个代码文件中，因此很容易将其直接放入更大的 C 或 C++ 项目的源代码树中。所有的代码生成和转换步骤都已经完成，因此不需要配置和编译辅助的 C 程序，也无需运行脚本。而且，由于整个库包含在一个翻译单元中，编译器可以进行更高级的优化，从而带来 5% 到 10% 的性能提升。出于这些原因，推荐所有应用程序都使用合并源码文件（“sqlite3.c”）。

建议所有应用程序都使用合并源码，无论是从网站下载的预构建版本，还是从规范源自行构建。

可以使用各个源代码文件将 SQLite 库链接到你的应用程序。 但是，更容易的方法是下载合并源码，或者使用“sqlite3.c” makefile 目标自行构建它，然后在后续的构建步骤中使用该合并源码。对于一些特殊应用，可能需要使用非标准的编译时选项对合并源码进行自定义构建，而不是直接使用网站上预构建的合并源码。换句话说，即使一个项目需要从各个源文件开始构建 SQLite，仍然建议以合并源码文件作为中间步骤。

3. 编译命令行界面

构建命令行界面需要三个源文件：

  • sqlite3.c：SQLite 合并源码文件
  • sqlite3.h：与 sqlite3.c 配套的头文件，定义了 SQLite 的 C 语言接口。
  • shell.c：命令行界面程序本身。该 C 源代码文件包含了main()例程的定义以及提示用户输入并将其传递给 SQLite 数据库引擎处理的循环。

以上三个源文件都包含在下载页面提供的合并源码压缩包中。或者可以使用“sqlite3.c” makefile 目标从规范源构建它们。

要构建 CLI，只需将这三个文件放在同一目录下并一起编译。使用 MSVC：

cl shell.c sqlite3.c -Fesqlite3.exe

在 Unix 系统上，命令通常如下所示：

gcc shell.c sqlite3.c -lpthread -ldl -lm -o sqlite3

为了使 SQLite 线程安全，需要 pthreads 库。但是由于 CLI 是单线程的，我们可以指示 SQLite 以非线程安全模式构建，从而省略 pthreads 库：

gcc -DSQLITE_THREADSAFE=0 shell.c sqlite3.c -ldl -lm -o sqlite3

-ldl 库用于支持动态加载、sqlite3_load_extension() 接口以及load_extension() SQL 函数。如果不需要这些功能，可以使用SQLITE_OMIT_LOAD_EXTENSION 编译时选项将其省略：

gcc -DSQLITE_THREADSAFE=0 -DSQLITE_OMIT_LOAD_EXTENSION shell.c sqlite3.c -lm -o sqlite3

还可以提供其他编译时选项，例如：

  • -DSQLITE_ENABLE_FTS4 或 -DSQLITE_ENABLE_FTS5 以启用全文搜索，
  • -DSQLITE_ENABLE_RTREE 以启用 R*Tree 搜索引擎扩展，
  • -DSQLITE_ENABLE_DBSTAT_VTAB 以启用dbstat 虚拟表，或
  • -DSQLITE_ENABLE_MATH_FUNCTIONS 以启用扩展的数学函数。

为了在EXPLAIN 列表中看到额外的注释，请添加-DSQLITE_ENABLE_EXPLAIN_COMMENTS 选项。添加 -DHAVE_READLINE 以及 -lreadline 和 -lncurses 库以获得命令行编辑支持。你可能还想指定一些编译器优化开关。（SQLite 网站提供的预编译 CLI 使用的是“-Os”。）此处可能有无数的变种。一个用于编译功能完整 shell 的命令可能如下所示：

gcc -Os -I. -DSQLITE_THREADSAFE=0 -DSQLITE_ENABLE_FTS4 \
   -DSQLITE_ENABLE_FTS5 -DSQLITE_ENABLE_JSON1 \
   -DSQLITE_ENABLE_RTREE -DSQLITE_ENABLE_EXPLAIN_COMMENTS \
   -DHAVE_READLINE \
   shell.c sqlite3.c -ldl -lm -lreadline -lncurses -o sqlite3

关键在于：构建 CLI 就是将两个 C 文件一起编译。shell.c 文件包含了入口点和用户输入循环的定义，而 SQLite 合并源码 sqlite3.c 则包含了 SQLite 库的完整实现。

4. 编译 TCL 接口

SQLite 的 TCL 接口是一个小模块，它被添加到常规合并源码中。结果是一个新的合并源码文件，名为“tclsqlite3.c”。这个单一源文件足以生成一个共享库，该库可以通过tclsh或wish使用TCL load 命令加载，或者生成一个内置 SQLite 的独立 tclsh。

要在 Linux 上生成可被 TCL 加载的 SQLite 库，可使用如下命令：

gcc -o libtclsqlite3.so -shared tclsqlite3.c -lpthread -ldl -ltcl

遗憾的是，为 Mac OS X 和 Windows 构建共享库并不那么简单。

要生成一个静态链接了 SQLite 的独立 tclsh，请使用以下编译命令：

gcc -DTCLSH=1 tclsqlite3.c -ltcl -lpthread -ldl -lz -lm

关键在于 -DTCLSH=1 选项。当使用 -DTCLSH=1 编译时，SQLite 的 TCL 接口模块包含了一个初始化 TCL 解释器并进入命令行循环的main()过程。上述命令在 Linux 和 Mac OS X 上都有效，但你可能需要根据平台以及所链接的 TCL 版本调整库选项。

5. 构建合并源码

在下载页面提供的 SQLite 合并源码版本通常已经足够大多数用户使用。然而，一些项目可能希望或需要自己构建合并源码。构建自定义合并源码的一个常见原因是为了使用某些编译时选项来自定义 SQLite 库。请记住，SQLite 合并源码包含了大量由辅助程序和脚本生成的 C 代码。许多编译时选项会影响这些生成代码，因此必须在组装合并源码之前将它们提供给代码生成器。必须传递给代码生成器的编译时选项集可能会因 SQLite 的不同版本而变化，但在本文撰写时（大约 SQLite 3.6.20，2009-11-04），必须由代码生成器知晓的选项包括：

  • SQLITE_ENABLE_UPDATE_DELETE_LIMIT
  • SQLITE_OMIT_ALTERTABLE
  • SQLITE_OMIT_ANALYZE
  • SQLITE_OMIT_ATTACH
  • SQLITE_OMIT_AUTOINCREMENT
  • SQLITE_OMIT_CAST
  • SQLITE_OMIT_COMPOUND_SELECT
  • SQLITE_OMIT_EXPLAIN
  • SQLITE_OMIT_FOREIGN_KEY
  • SQLITE_OMIT_PRAGMA
  • SQLITE_OMIT_REINDEX
  • SQLITE_OMIT_SUBQUERY
  • SQLITE_OMIT_TEMPDB
  • SQLITE_OMIT_TRIGGER
  • SQLITE_OMIT_VACUUM
  • SQLITE_OMIT_VIEW
  • SQLITE_OMIT_VIRTUALTABLE

要构建自定义合并源码，首先在 Unix 或类 Unix 的开发平台上下载原始的各个源文件。务必获取原始源文件，而不是“预处理过的源文件”。你可以从下载页面或直接从配置管理系统获取完整的原始源文件集。

假设 SQLite 源代码树存储在名为“sqlite”的目录中。计划在一个并行目录（例如“bld”）中构建合并源码。首先，通过运行 SQLite 源代码树顶层的 configure 脚本，或复制源代码树顶层的模板 Makefile 之一，来构造一个合适的 Makefile。然后手工编辑该 Makefile，以包含所需的编译时选项。最后运行：

make sqlite3.c

或者在使用 MSVC 的 Windows 上：

nmake /f Makefile.msc sqlite3.c

“sqlite3.c” make 目标会自动构造常规的“sqlite3.c”合并源码文件、其头文件“sqlite3.h”以及包含 TCL 接口的“tclsqlite3.c”合并源码文件。之后，所需的文件可以复制到项目目录并按上述流程编译。

6. 构建 Windows DLL

要为 Windows 构建可用的 SQLite DLL，首先获取相应的合并源码文件 sqlite3.c 和 sqlite3.h。可以从SQLite 网站下载这些文件，或者如上文所示从源代码自定义生成。

在工作目录中放入源文件后，可以使用 MSVC 通过以下命令生成 DLL：

cl sqlite3.c -link -dll -out:sqlite3.dll

上述命令应在 MSVC Native Tools 命令提示符下运行。如果你的机器上安装了 MSVC，通常会有多个这样的命令提示符，用于 x86 和 x64 的本机构建，也可能用于交叉编译到 ARM。请选择适合目标 DLL 的命令提示符。

如果使用 MinGW 编译器，命令行如下：

gcc -shared sqlite3.c -o sqlite3.dll

请注意，MinGW 仅生成 32 位 DLL。MinGW64 项目可用于生成 64 位 DLL。推测命令行语法类似。还需注意，较新的 MSVC 生成的 DLL 无法在 WinXP 及更早版本的 Windows 上运行。因此，为了使生成的 DLL 兼容性最佳，推荐使用 MinGW。一个良好的经验法则是：使用 MinGW 生成 32 位 DLL，使用 MSVC 生成 64 位 DLL。

在大多数情况下，你还需要为你的应用补充适当的编译时选项。常用的编译时选项包括：

  • -Os - 优化体积，使 DLL 尽量小。

  • -O2 - 优化速度。通过展开循环和内联函数会使 DLL 变大。

  • -DSQLITE_ENABLE_FTS4 - 在 SQLite 中包含全文搜索引擎代码。

  • -DSQLITE_ENABLE_RTREE - 包含R-Tree 扩展。

  • -DSQLITE_ENABLE_COLUMN_METADATA - 启用一些某些常见系统（包括 Ruby-on-Rails）所需的额外 API。
