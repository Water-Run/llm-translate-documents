SQLite 中的限制

在本文中，“限制”是指不能超出的大小或数量。我们关注诸如 BLOB 中的最大字节数或表中最大列数之类的内容。

SQLite 最初的设计策略是避免任意限制。当然，运行在有限内存和磁盘空间的机器上的任何程序都有某种限制。但是在 SQLite 中，这些限制并没有明确定义。政策是只要它能装入内存并且可以用 32 位整数计数，它就应该能工作。

不幸的是，无限制的策略已经被证明会造成问题。由于上限没有明确定义，它们就没有经过测试，在将 SQLite 推到极限时经常会发现错误。因此，从大约 2008-04-16 发布的 3.5.8 版本起，SQLite 的各项限制都有明确的定义，并且这些限制会作为测试套件的一部分进行测试。

本文定义了 SQLite 的限制，以及如何为特定应用自定义这些限制。默认设置通常非常大，足以满足几乎所有应用。有些应用可能希望在某些地方增加限制，但我们预计这种需求很少。更常见的情况是，某个应用可能希望用更低的限制重新编译 SQLite，以避免在上层 SQL 语句生成器出现 bug 时过度消耗资源，或帮助阻止注入恶意 SQL 语句的攻击者。

某些限制可以使用sqlite3_limit()接口在运行时按连接逐个调整，并使用该接口定义的限制类别之一。运行时限制适用于具有多个数据库的应用，其中有些数据库仅供内部门使用，而其他数据库可能受潜在敌对的外部 agent 影响或控制。例如，一个 Web 浏览器应用可能会使用内部数据库来跟踪历史页面浏览情况，但同时还会有一个或多个由从互联网下载的 JavaScript 应用创建和控制的独立数据库。sqlite3_limit() 接口允许受信任代码管理的内部数据库不受限制，同时对不受信任的外部代码创建或控制的数据库施加严格限制，以帮助防止拒绝服务攻击。

  1. 字符串或 BLOB 的最大长度

    SQLite 中字符串或 BLOB 的最大字节数由预处理宏 SQLITE_MAX_LENGTH 定义。该宏的默认值为十亿（即一千百万或 1,000,000,000）。你可以在编译时使用如下命令行选项提高或降低这个值：

    -DSQLITE_MAX_LENGTH=123456789

    当前实现仅支持的字符串或 BLOB 长度最多为 231-3，即 2147483645。并且某些内置函数如 hex() 在达到该值之前就可能失败。在安全敏感的应用中，最好不要尝试增加最大字符串和 BLOB 长度。实际上，如果可能的话，最好将最大字符串和 BLOB 长度降低到几百万的范围内。

    在 SQLite 的 INSERT 和 SELECT 处理的某个阶段，数据库中每一行的完整内容都会被编码为单个 BLOB。因此 SQLITE_MAX_LENGTH 参数也决定了每行的最大字节数。

    可以使用 sqlite3_limit(db,SQLITE_LIMIT_LENGTH,size) 接口在运行时降低最大字符串或 BLOB 长度。

  2. 最大列数

    编译时参数 SQLITE_MAX_COLUMN 用于设置以下内容的上限：

    • 表中的列数
    • 索引中的列数
    • 视图中的列数
    • UPDATE 语句 SET 子句中的项数
    • SELECT 语句结果集中的列数
    • GROUP BY 或 ORDER BY 子句中的项数
    • INSERT 语句中的值数

    SQLITE_MAX_COLUMN 的默认设置为 2000。你可以在编译时将其更改为最大 32767。另一方面，许多经验丰富的数据库设计师会认为一个合理规范化的数据库永远不需要表中超过 100 列。

    在大多数应用中，列数很少——几十列。SQLite 代码生成器中的某些地方使用 O(N²) 算法，其中 N 是列数。所以如果你把 SQLITE_MAX_COLUMN 重新定义为非常大的数字，并且生成使用大量列的 SQL，你可能会发现 sqlite3_prepare_v2() 运行得很慢。

    可以使用 sqlite3_limit(db,SQLITE_LIMIT_COLUMN,size) 接口在运行时降低最大列数。

  3. SQL 语句的最大长度

    SQL 语句文本的最大字节数受限于 SQLITE_MAX_SQL_LENGTH，默认值为 1,000,000,000。

    如果 SQL 语句被限制为一百万字节，那么显然你不能在 INSERT 语句中通过将几十万字节的字符串作为字面量来插入数据。但你本就不应该那样做。请为数据使用主机参数。准备像下面这样的短 SQL 语句：

    INSERT INTO tab1 VALUES(?,?,?);

    然后使用 sqlite3_bind_XXXX() 函数将你的大字符串值绑定到 SQL 语句。绑定的使用避免了对字符串中引号字符进行转义，从而降低 SQL 注入攻击的风险。同时，由于大字符串无需被频繁解析或复制，执行速度也会更快。

    可以使用 sqlite3_limit(db,SQLITE_LIMIT_SQL_LENGTH,size) 接口在运行时降低 SQL 语句的最大长度。

  4. 联接中允许的最大表数

    SQLite 不支持包含超过 64 个表的联接。这个限制源于 SQLite 代码生成器在查询优化器中使用每个联接表一个位的位图。

    SQLite 使用高效的查询规划算法，因此即便是大的联接也可以迅速执行准备操作。因此，没有机制可以提高或降低联接中表数的限制。

  5. 表达式树的最大深度

    SQLite 将表达式解析为供处理的树。在代码生成过程中，SQLite 递归地遍历这棵树。因此，为避免占用过多栈空间，表达式树的深度受到限制。

    SQLITE_MAX_EXPR_DEPTH 参数决定了表达式树的最大深度。如果值为 0，则不强制任何限制。当前实现的默认值为 1000。

    如果 SQLITE_MAX_EXPR_DEPTH 最初为正数，则可以使用 sqlite3_limit(db,SQLITE_LIMIT_EXPR_DEPTH,size) 接口在运行时降低表达式树的最大深度。换句话说，如果编译时已经对表达式深度设置了限制，那么该最大深度就可以在运行时降低。如果在编译时将 SQLITE_MAX_EXPR_DEPTH 设置为 0（表达式深度无限制），那么 sqlite3_limit(db,SQLITE_LIMIT_EXPR_DEPTH,size) 就不会起作用。

  6. 函数参数的最大数量

    SQLITE_MAX_FUNCTION_ARG 参数决定了可以传递给 SQL 函数的参数最大数量。多年来默认值大约是 100，但从 SQLite 3.48.0（2025-01-14）版本开始，默认值提高到 1000。

    函数的参数数量有时会存储在带符号的 16 位整数中。因此 SQLITE_MAX_FUNCTION_ARG 的硬上限是 32767。

    可以使用 sqlite3_limit(db,SQLITE_LIMIT_FUNCTION_ARG,size) 接口在运行时降低函数参数的最大数量。

  7. 复合 SELECT 语句中的项数上限

    复合SELECT语句是由 UNION、UNION ALL、EXCEPT 或 INTERSECT 运算符连接的两个或多个 SELECT 语句。我们将复合 SELECT 中的每个单独 SELECT 称为一个“项”。

    SQLite 中的代码生成器使用递归算法处理复合 SELECT 语句。为了限制栈的大小，我们对复合 SELECT 中的项数进行限制。最大项数由 SQLITE_MAX_COMPOUND_SELECT 决定，默认值为 500。我们认为这是一个慷慨的额度，因为在实际中几乎从未见过复合 SELECT 的项数超过个位数。

    可以使用 sqlite3_limit(db,SQLITE_LIMIT_COMPOUND_SELECT,size) 接口在运行时降低复合 SELECT 项数的最大值。

  8. LIKE 或 GLOB 模式的最大长度

    默认LIKE和GLOB实现中使用的模式匹配算法在某些异常情况下表现出 O(N²) 的性能（其中 N 是模式中的字符数）。为了防止能够指定任意 LIKE 或 GLOB 模式的不良分子发起拒绝服务攻击，LIKE 或 GLOB 模式的长度被限制为 SQLITE_MAX_LIKE_PATTERN_LENGTH 字节。该限制的默认值为 50000。现代工作站即便是评估长度为 50000 字节的异常 LIKE 或 GLOB 模式也比较迅速。拒绝服务问题只有在模式长度达到数百万字节时才会出现。然而，由于大多数有用的 LIKE 或 GLOB 模式最多只有几十个字节，如果应用开发者知道外部用户能够生成任意模式，那么出于多虑的考虑，他们可能希望将该参数减少到几百字节左右。

    可以使用 sqlite3_limit(db,SQLITE_LIMIT_LIKE_PATTERN_LENGTH,size) 接口在运行时降低 LIKE 或 GLOB 模式的最大长度。

  9. 单个 SQL 语句中的主机参数数量上限

    主机参数是 SQL 语句中的占位符，使用sqlite3_bind_XXXX() 接口来填充。许多 SQL 程序员熟悉使用问号（“?”）作为主机参数。SQLite 还支持前缀为 “:”、 “$” 或 “@” 的命名主机参数，以及形式为 “?123” 的编号主机参数。

    SQLite 为 SQL 语句中的每个主机参数分配一个编号。编号通常从 1 开始并且每个新参数递增 1。然而，当使用 “?123” 形式时，主机参数的编号就是问号后面的数字。

    SQLite 为所有编号从 1 到使用的最大主机参数编号之间的参数分配空间。因此，如果在 SQL 语句中使用类似 ?1000000000 这样的主机参数编号，就需要数千兆字节的存储空间。这很容易让宿主机器的资源耗尽。为了防止过度内存分配，主机参数编号的最大值为 SQLITE_MAX_VARIABLE_NUMBER，对于 3.32.0（2020-05-22）之前的 SQLite 版本默认为 999，对于 3.32.0 之后的版本默认为 32766。

    可以使用 sqlite3_limit(db,SQLITE_LIMIT_VARIABLE_NUMBER,size) 接口在运行时降低主机参数编号的最大值。

  10. 触发器递归的最大深度

    SQLite 限制触发器递归的深度，以防止涉及递归触发器的语句使用无限制的内存。

    在 SQLite 3.6.18 版本（2009-09-11）之前，触发器不支持递归，因此该限制没有意义。从 3.6.18 版本开始，支持递归触发器，但必须使用PRAGMA recursive_triggers 语句显式启用。从 3.7.0 版本（2009-09-11）起，递归触发器默认启用，但可以通过PRAGMA recursive_triggers 手动禁用。只有在启用了递归触发器的情况下，SQLITE_MAX_TRIGGER_DEPTH 才有意义。

    默认的触发器递归最大深度是 1000。

  11. 附加数据库的最大数量

    ATTACH 语句是 SQLite 的扩展，允许将两个或多个数据库关联到同一个数据库连接，并使它们像单个数据库一样运行。可同时附加的数据库数量受 SQLITE_MAX_ATTACHED 限制，默认值为 10。附加数据库的最大数量无法增加到 125 以上。

    可以使用 sqlite3_limit(db,SQLITE_LIMIT_ATTACHED,size) 接口在运行时降低附加数据库的最大数量。

  12. 数据库文件中允许的最大页数

    SQLite 可以限制数据库文件的大小，以防止文件变得过大而占用过多磁盘空间。SQLITE_MAX_PAGE_COUNT 参数是单个数据库文件允许的最大页数。尝试插入新数据而使数据库文件变得比该值更大时将返回 SQLITE_FULL。

    SQLITE_MAX_PAGE_COUNT 的最大可能设置为 4294967294（232-2）。从 3.45.0 版本（2024-01-15）起，4294967294 也是 SQLITE_MAX_PAGE_COUNT 的默认值。当使用默认页大小 4096 字节时，这意味着最大数据库大小约为 17.5 TB。如果页大小增加到最大 65536 字节，那么数据库文件最多可增长到约 281 TB。

    max_page_count PRAGMA 可用于在运行时提高或降低该限制。

  13. 表中的最大行数

    表的理论最大行数是 264（18446744073709551616，约为 1.8×10¹⁹）。由于最大数据库大小为 281 TB，因此这一限制无法达到。一个 281 TB 的数据库最多可容纳约 2×10¹³ 行，而且前提是没有索引且每行数据非常少。

  14. 数据库的最大大小

    每个数据库都由一个或多个“页”组成。在单个数据库中，每页大小相同，但不同数据库的页大小可以是 512 到 65536 之间的 2 的幂。数据库文件的最大大小为 4294967294 页。在最大页大小 65536 字节时，这相当于大约 2.8×10¹⁴ 字节（281 TB，或 256 の tebibytes，或 281474 GB，或 262143 gibibytes）。

    这个特定的上限没有经过测试，因为开发者没有能够达到该限制的硬件。不过，测试确实验证了当数据库达到底层文件系统允许的最大文件大小（通常远小于理论最大值）时以及当数据库由于磁盘空间耗尽而无法增长时，SQLite 的行为仍然正确并且合理。

  15. 模式中表的最大数量

    每个表和索引在数据库文件中至少需要一页。因此，数据库文件的最大页数也就是模式中表和索引数量的上限。此前所说的“索引”是指使用CREATE INDEX 语句显式创建的索引，或由 UNIQUE 和 PRIMARY KEY 约束隐式创建的索引。

    每次打开数据库时，整个模式都会被扫描和解析，并且解析树会保存在内存中。这意味着数据库连接的启动时间和初始内存使用量与模式的大小成正比。
