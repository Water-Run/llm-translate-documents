SQLite 中的约束冲突解决

在大多数 SQL 数据库中，如果在表上有 UNIQUE、NOT NULL 或 CHECK 约束，并且你尝试执行会违反该约束的 UPDATE 或 INSERT， 数据库会中止正在进行的操作，撤销与同一 UPDATE 或 INSERT 语句相关的任何先前更改，并返回错误。 这是 SQLite 的默认行为，不过 SQLite 也允许定义处理约束违规的替代方式。 本文介绍这些替代方法以及如何使用它们。

冲突解决算法

SQLite 定义了以下五种约束冲突解决算法：

ROLLBACK

当发生约束冲突时，会立即执行 ROLLBACK，从而结束当前事务，并以 SQLITE_CONSTRAINT 返回码中止该命令。如果没有活动事务（除了每个命令隐式创建的事务），则该算法的行为与 ABORT 相同。

ABORT

当发生约束冲突时，该命令会撤销它可能所做的任何先前更改，并以 SQLITE_CONSTRAINT 返回码中止。但不会执行 ROLLBACK，因此同一事务内先前命令的更改会被保留。这是 SQLite 的默认行为。

FAIL

当发生约束冲突时，该命令会以 SQLITE_CONSTRAINT 返回码中止。但命令在遇到约束冲突之前对数据库所做的任何更改都会被保留，不会被撤销。例如，如果 UPDATE 语句在尝试更新的第 100 行遇到约束冲突，那么前 99 行的更改会被保留，而第 100 行及之后的行则不会发生更改。

IGNORE

当发生约束冲突时，包含该冲突的那一行不会被插入或更改。但命令继续正常执行。冲突行之前和之后的其他行会照常被插入或更新。不会返回错误。

REPLACE

当发生 UNIQUE 约束冲突时，导致冲突的预先存在的行会在插入或更新当前行之前被删除。因此插入或更新总是会发生。命令继续正常执行。不会返回错误。
