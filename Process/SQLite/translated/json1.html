1. 概览

默认情况下，SQLite 支持三十个用于处理 JSON 值的函数和两个操作符。还有四个表值函数可用于分解 JSON 字符串。下方列出的所有函数都带有SQLITE_INNOCUOUS与SQLITE_DETERMINISTIC标志。

共有二十六个标量函数和操作符：

  1. json(json)
  2. jsonb(json)
  3. json_array(value1,value2,...)
  4. jsonb_array(value1,value2,...)
  5. json_array_length(json)
    json_array_length(json,path)
  6. json_error_position(json)
  7. json_extract(json,path,...)
  8. jsonb_extract(json,path,...)
  9. json -> path
  10. json ->> path
  11. json_insert(json,path,value,...)
  12. jsonb_insert(json,path,value,...)
  13. json_object(label1,value1,...)
  14. jsonb_object(label1,value1,...)
  15. json_patch(json1,json2)
  16. jsonb_patch(json1,json2)
  17. json_pretty(json)
  18. json_remove(json,path,...)
  19. jsonb_remove(json,path,...)
  20. json_replace(json,path,value,...)
  21. jsonb_replace(json,path,value,...)
  22. json_set(json,path,value,...)
  23. jsonb_set(json,path,value,...)
  24. json_type(json)
    json_type(json,path)
  25. json_valid(json)
    json_valid(json,flags)
  26. json_quote(value)

还有四个聚合 SQL 函数：

  1. json_group_array(value)
  2. jsonb_group_array(value)
  3. json_group_object(label,value)
  4. jsonb_group_object(name,value)

这四个表值函数是：

  1. json_each(json)
    json_each(json,path)
  2. json_tree(json)
    json_tree(json,path)
  3. jsonb_each(json)
    jsonb_each(json,path)
  4. jsonb_tree(json)
    jsonb_tree(json,path)

2. 编译 JSON 支持

JSON 函数与操作符从 SQLite 3.38.0（2022-02-22）起默认内置。 如果添加 -DSQLITE_OMIT_JSON 编译选项，它们可以被省略。 在 3.38.0 版本之前，JSON 函数是扩展，只有在包含 -DSQLITE_ENABLE_JSON1 编译选项时才会被包含。 换句话说，JSON 函数从 SQLite 3.37.2（及以前）版本的“显式启用”模式，变成 SQLite 3.38.0（及以后）版本的“显式禁用”模式。

3. 接口概览

SQLite 将 JSON 作为普通文本存储。 向后兼容性限制意味着 SQLite 只能存储 NULL、整数、浮点数、文本与 BLOB 值。无法新增“JSON”类型。

3.1. JSON 参数

对于接受 JSON 作为第一个参数的函数，该参数可以是 JSON 对象、数组、数字、字符串或 null。 SQLite 的数值与 NULL 被分别解释为 JSON 数字与 null。 SQLite 文本值可以理解为 JSON 对象、数组或字符串。 如果将一个不是合法 JSON 对象、数组或字符串的 SQLite 文本值传入 JSON 函数，该函数通常会抛出错误。（此规则的例外为 json_valid()、json_quote() 与 json_error_position()。）

这些例程支持所有RFC-8259 JSON 语法以及JSON5 扩展。 这些例程生成的 JSON 文本始终严格遵循规范 JSON 定义，不包含任何 JSON5 或其他扩展。 在 3.42.0（2023-05-16）版本中新增了读取与理解 JSON5 的能力。 此前的 SQLite 版本只能读取规范 JSON。

3.2. JSONB

从 3.45.0（2024-01-15）版本开始，SQLite 允许将内部 JSON “解析树”表示以 BLOB 形式存储在磁盘上，称为“JSONB”格式。 通过直接在数据库中存储 SQLite 的内部二进制 JSON 表示，应用可以在读取与更新 JSON 值时绕过解析与渲染 JSON 的开销。 内部 JSONB 格式在磁盘上通常也比文本 JSON 更节省空间。

任何接受文本 JSON 输入的 SQL 函数参数也同时接受 JSONB 格式的 BLOB。 函数在两种情况下行为相同，只是在输入为 JSONB 时运行更快，因为无需执行 JSON 解析。

大多数返回文本 JSON 的 SQL 函数都有对应的返回 JSONB 的函数。 返回文本 JSON 格式的函数名以 "json_" 开头，返回二进制 JSONB 格式的函数名以 "jsonb_" 开头。

3.2.1. JSONB 格式

JSONB 是 SQLite 使用的 JSON 二进制表示，仅供 SQLite 内部使用。 应用不应在 SQLite 之外使用 JSONB，也不要试图逆向工程 JSONB 格式。

“JSONB”名称源自PostgreSQL，但 SQLite 的 JSONB 磁盘格式并不与 PostgreSQL 相同。 两个格式名称相同，但不兼容二进制。 PostgreSQL 的 JSONB 格式宣称可以在对象与数组中实现 O(1) 元素查找。 SQLite 的 JSONB 格式并不做此类宣称。SQLite 的 JSONB 对多数操作的时间复杂度为 O(N)，如同文本 JSON。 SQLite 的 JSONB 优势在于其比文本 JSON 更小、更快——可能快数倍。 JSONB 格式中留有空间可用于未来增强，未来版本的 SQLite 可能包含可实现 JSONB O(1) 查找的选项，但目前尚无此类功能。

3.2.2. 错误 JSONB 的处理

SQLite 生成的 JSONB 始终是合法格式。 如果遵循建议，将 JSONB 视为不透明的 BLOB，则不会出现问题。 但 JSONB 只是 BLOB，一位恶意的程序员可能设计出与 JSONB 相似但技术上格式错误的 BLOB。 当将格式错误的 JSONB 输入 JSON 函数时，可能发生以下任意情况：

  • SQL 语句可能以“malformed JSON”（格式错误的 JSON）错误中止。

  • 如果 JSONB BLOB 中格式错误的部分不影响答案，可能会返回正确结果。

  • 可能返回荒谬或不合逻辑的答案。

SQLite 如何处理无效 JSONB 可能会在不同版本之间改变。系统遵循“垃圾进垃圾出”规则：如果提供无效 JSONB，结果也会无效。 如果怀疑 JSONB 是否有效，可以使用 json_valid() 函数验证。

我们做出如下承诺： 格式错误的 JSONB 永远不会导致内存错误或其他可能引发漏洞的问题。 无效 JSONB 可能导致奇怪的回答或查询中止，但不会导致崩溃。

3.3. PATH 参数

对于接受 PATH 参数的函数，该 PATH 必须是合法格式，否则函数会抛出错误。 合法 PATH 是一个文本值，以一个 '$' 字符开头，后面跟零个或多个 ".objectlabel" 或 "[arrayindex]"。

arrayindex 通常是非负整数 N，此时所选元素为数组中从左起第 N 项，索引从零开始。 arrayindex 也可以是形如 "#-N"，此时所选元素是从右往左数第 N 项。 数组最后一项为 "#-1"。可将 "#" 视作“数组元素数”。 表达式 "#-1" 计算出的整数即数组最后一项索引。 在追加值到现有 JSON 数组时，有时索引仅为字符 # 是有用的：

  • json_set('[0,1,2]','$[#]','new') → '[0,1,2,"new"]'

3.4. VALUE 参数

对于接受“value”参数（或“value1”、“value2”）的函数，这些参数通常被理解为字面字符串，会被加引号并在结果中作为 JSON 字符串值。 即便输入字符串看起来像合法 JSON，在结果中仍视作字面字符串。

然而，如果 value 参数直接来自另一个 JSON 函数的结果或-> 操作符（但不是 ->> 操作符），则会被理解为实际 JSON，完整 JSON 会被插入而非作为带引号的字符串。

例如，在下面这个 json_object() 调用中，value 参数看起来像合法 JSON 数组。但因为只是普通 SQL 文本，它会被当作字面字符串并作为带引号字符串插入结果：

  • json_object('ex','[52,3.14159]') → '{"ex":"[52,3.14159]"}'
  • json_object('ex',('[52,3.14159]'->>'$')) → '{"ex":"[52,3.14159]"}'

但如果外层 json_object() 调用中的 value 参数来自另一个 JSON 函数，比如 json() 或 json_array()，则会被理解为实际 JSON 并如实插入：

  • json_object('ex',json('[52,3.14159]')) → '{"ex":[52,3.14159]}'
  • json_object('ex',json_array(52,3.14159)) → '{"ex":[52,3.14159]}'
  • json_object('ex','[52,3.14159]'->'$') → '{"ex":[52,3.14159]}'

说明一下：“json”参数无论其来源如何，都始终被解释为 JSON。但只有当“value”参数直接来自另一个 JSON 函数或-> 操作符时，才会被解释为 JSON。

在被解释为 JSON 字符串的 VALUE 参数中，Unicode 转义序列不会被视作该 Unicode 代码点对应的字符或转义控制字符。它们不会被翻译或特殊处理；SQLite 的 JSON 函数会将它们作为普通文本处理。

3.5. 兼容性

当前的 JSON 库实现使用递归下降解析器。为了避免过多的栈空间使用，任何嵌套级别超过 1000 的 JSON 输入都被视为无效。 对嵌套深度的限制在RFC-8259 第 9 节中被允许用于兼容实现。

3.6. JSON5 扩展

从 3.42.0（2023-05-16）版本开始，这些例程将读取并解释包含JSON5扩展的输入 JSON 文本。 然而，这些例程产生的 JSON 文本始终严格符合规范 JSON 定义。

以下是 JSON5 扩展的概要（改编自JSON5 规范）：

  • 对象键可以是不带引号的标识符。
  • 对象可以有一个尾随逗号。
  • 数组可以有一个尾随逗号。
  • 字符串可以使用单引号。
  • 字符串可以通过转义换行符跨多行。
  • 字符串可以包含新的字符转义。
  • 数字可以是十六进制。
  • 数字可以有前导或尾随小数点。
  • 数字可以是 “Infinity”、“-Infinity” 与 “NaN”。
  • 数字可以以显式的加号开头。
  • 允许使用单行（//...）与多行（/*...*/）注释。
  • 允许更多的空白字符。

要将 JSON5 字符串 X 转换为规范 JSON，请调用 "json(X)"。 "json()" 函数的输出始终是规范 JSON，无论输入中包含何种 JSON5 扩展。 为了向后兼容，不带“flags”参数的 json_valid(X) 函数仍然会对非规范 JSON 输入返回 false，即使该输入是该函数能够理解的 JSON5。 要判断输入字符串是否是有效 JSON5，请在 json_valid 的“flags”参数中包含 0x02 位：“json_valid(X,2)”。

这些例程理解完整的 JSON5，并额外扩展了两点语法：

  1. 严格的 JSON5 要求非引号对象键必须是 ECMAScript 5.1 IdentifierName。 但需要庞大的 unicode 表与大量代码才能判定一个键是否为 ECMAScript 5.1 IdentifierName。 因此，SQLite 允许对象键包含任何大于 U+007f 的非空白 unicode 字符。 这个宽松的“标识符”定义大大简化了实现，使得 JSON 解析器更小且运行更快。

  2. JSON5 允许浮点无穷在严格大小写下表示为 "Infinity"、"-Infinity" 或 "+Infinity"。 SQLite 也允许使用 "Inf" 作为 "Infinity" 的简写，并允许上述关键词以任意大小写组合出现。 同样，JSON5 允许使用 "NaN" 表示非数。SQLite 将其扩展为允许任意大小写组合的 "QNaN" 与 "SNaN"。 请注意，SQLite 将 NaN、QNaN 与 SNaN 统一解释为“null”。 之所以添加这一扩展，是因为（据说）现实中存在大量采用上述非标准表示方式来描述无穷与非数的 JSON 数据。

3.7. 性能注意事项

多数 JSON 函数内部使用 JSONB 进行处理。因此，如果输入为文本，首先必须将其转换为 JSONB。 如果输入已经是 JSONB 格式，就无需转换，可跳过这一步，性能更快。

因此，当一个 JSON 函数的参数来自另一个 JSON 函数时，通常更高效的做法是将作为参数的函数替换为 "jsonb_" 版本。

  • ... json_insert(A,'$.b',json(C)) ...   ← 效率较低。
  • ... json_insert(A,'$.b',jsonb(C)) ...   → 效率更高。

聚合 JSON SQL 函数 是例外。这些函数的处理均使用文本而非 JSONB。因此，对于聚合 JSON SQL 函数而言，使用 "json_" 而非 "jsonb_" 函数传入参数更高效。

  • ... json_group_array(json(A))) ...   ← 更高效。
  • ... json_group_array(jsonb(A))) ...   → 效率较低。

3.8. JSON BLOB 输入错误

如果 JSON 输入是一个非 JSONB 的 BLOB，并且将其转换为文本时看起来像文本 JSON，那么该输入会被接受为文本 JSON。 这实际上是原始实现中一个长期存在的错误，SQLite 开发者当时未曾察觉。文档表明 JSON 函数应对 BLOB 输入报错， 但实际实现只要 BLOB 内容在数据库的文本编码下是合法 JSON 字符串，就会接受该输入。

这个 JSON BLOB 输入错误在为 3.45.0（2024-01-15）版本重新实现 JSON 例程时意外修复。 这导致依赖旧行为的应用出现中断。（为这些应用辩护：它们常被 CLI 中的readfile() SQL 函数引诱使用 BLOB 作为 JSON。readfile() 用于从磁盘文件读取 JSON，但返回的是 BLOB。既然能用，为何不继续使用呢？）

出于向后兼容的 bug 兼容性考虑， 如果不进行其他解释，而仅在 BLOB 与文本 JSON 之间自动解释，则该“旧”行为在 3.45.1（2024-01-30）及之后版本中被正式记录并支持。 不过请警惕，仍然存在同时是有效 JSONB 且在转换为文本后也是有效 JSON 的 BLOB。例如，BLOB x'33343535' 是合法 JSONB（即整数 456）， 而转为文本后则是合法 JSON 整数字面量 3456。因此，如果你有旧数据库将 JSON 文本存为 BLOB，并希望在 SQLite 3.45.0 及以后继续使用，建议运行一次 UPDATE， 将旧 JSON 文本实际存储为 TEXT，而非 BLOB。

4. 函数细节

以下章节提供各个 JSON 函数与操作符的进一步说明：

4.1. json() 函数

json(X) 函数验证其参数 X 是否为合法 JSON 字符串或 JSONB BLOB，并返回去除所有多余空白后的压缩 JSON 字符串。 如果 X 并非合法 JSON 字符串或 JSONB BLOB，该例程会抛出错误。

如果输入是 JSON5 文本，则在返回前先转换为规范的 RFC-8259 文本。

如果传入 json(X) 的参数 X 包含具有重复标签的 JSON 对象，则是否保留重复项未定义。目前实现会保留重复项。 不过，未来的增强可能会选择静默去除重复。

示例：

  • json(' { "this" : "is", "a": [ "test" ] } ') → '{"this":"is","a":["test"]}'

4.2. jsonb() 函数

jsonb(X) 函数返回传入参数 X 的二进制 JSONB 表示。 如果 X 是 TEXT 且不具有合法 JSON 语法，则会抛出错误。

如果 X 是 BLOB 且看起来像 JSONB，则该例程直接返回 X 的副本。 但仅检查 JSONB 输入的最外层元素，不会验证深层结构。

4.3. json_array() 函数

json_array() SQL 函数接受零个或多个参数，并返回一个由这些参数组成的合法 JSON 数组。 如果传入的任何参数是 BLOB，会抛出错误。

SQL 类型为 TEXT 的参数通常被转换为带引号的 JSON 字符串。 但如果该参数是另一个 json1 函数的输出，则会作为 JSON 存储。 这允许嵌套调用 json_array() 与 json_object()。 json() 函数也可以用来强制让字符串被识别为 JSON。

示例：

  • json_array(1,2,'3',4) → '[1,2,"3",4]'
  • json_array('[1,2]') → '["[1,2]"]'
  • json_array(json_array(1,2)) → '[[1,2]]'
  • json_array(1,null,'3','[4,5]','{"six":7.7}') → '[1,null,"3","[4,5]","{\"six\":7.7}"]'
  • json_array(1,null,'3',json('[4,5]'),json('{"six":7.7}')) → '[1,null,"3",[4,5],{"six":7.7}]'

4.4. jsonb_array() 函数

jsonb_array() SQL 函数与json_array() 函数完全相同，唯一不同之处在于它返回的是 SQLite 私有的 JSONB 格式，而非标准的 RFC 8259 文本格式。

4.5. json_array_length() 函数

json_array_length(X) 函数返回 JSON 数组 X 中的元素个数，如果 X 不是数组类型的 JSON 值则返回 0。 json_array_length(X,P) 会在 X 中定位路径 P 所指的数组，并返回该数组的长度；如果路径 P 定位到的元素不是数组，则返回 0；如果路径 P 没有匹配 X 中的任何元素，则返回 NULL。 如果 X 不是合法 JSON，或 P 不是合法路径，会抛出错误。

示例：

  • json_array_length('[1,2,3,4]') → 4
  • json_array_length('[1,2,3,4]', '$') → 4
  • json_array_length('[1,2,3,4]', '$[2]') → 0
  • json_array_length('{"one":[1,2,3]}') → 0
  • json_array_length('{"one":[1,2,3]}', '$.one') → 3
  • json_array_length('{"one":[1,2,3]}', '$.two') → NULL

4.6. json_error_position() 函数

json_error_position(X) 函数如果输入 X 是合法 JSON 或 JSON5 字符串，则返回 0。 如果输入 X 包含一个或多个语法错误，则返回第一个语法错误的字符位置。最左侧字符的位置为 1。

如果输入 X 是 BLOB，则当 X 是格式良好的 JSONB BLOB 时返回 0。 如果返回值为正数，则表示在 BLOB 中检测到第一个错误的大致 1 基准位置。

json_error_position() 函数是在 SQLite 3.42.0（2023-05-16）版本中添加的。

4.7. json_extract() 函数

json_extract(X,P1,P2,...) 从合法 JSON X 中提取并返回一个或多个值。 如果只提供单个路径 P1，则返回结果的数据类型：如果是 JSON null 则为 NULL，JSON 数值返回 INTEGER 或 REAL，JSON false 返回 INTEGER 0，JSON true 返回 INTEGER 1，JSON 字符串返回去引号后的文本，而 JSON 对象与数组返回文本表示。 如果提供多个路径参数（P1、P2 等），则该例程返回一个合法的 JSON 数组文本，包含各个值。

示例：

  • json_extract('{"a":2,"c":[4,5,{"f":7}]}', '$') → '{"a":2,"c":[4,5,{"f":7}]}'
  • json_extract('{"a":2,"c":[4,5,{"f":7}]}', '$.c') → '[4,5,{"f":7}]'
  • json_extract('{"a":2,"c":[4,5,{"f":7}]}', '$.c[2]') → '{"f":7}'
  • json_extract('{"a":2,"c":[4,5,{"f":7}]}', '$.c[2].f') → 7
  • json_extract('{"a":2,"c":[4,5],"f":7}','$.c','$.a') → '[[4,5],2]'
  • json_extract('{"a":2,"c":[4,5],"f":7}','$.c[#-1]') → 5
  • json_extract('{"a":2,"c":[4,5,{"f":7}]}', '$.x') → NULL
  • json_extract('{"a":2,"c":[4,5,{"f":7}]}', '$.x', '$.a') → '[null,2]'
  • json_extract('{"a":"xyz"}', '$.a') → 'xyz'
  • json_extract('{"a":null}', '$.a') → NULL

SQLite 中的 json_extract() 与 MySQL 中的 json_extract() 有一个微妙的不兼容之处。 MySQL 版本的 json_extract() 始终返回 JSON。 SQLite 版本的 json_extract() 仅在有两个或以上 PATH 参数（此时结果为 JSON 数组）或单个 PATH 参数引用数组或对象时返回 JSON。 在 SQLite 中，如果 json_extract() 只有一个 PATH 参数，且该 PATH 指向 JSON null、字符串或数值，那么 json_extract() 会返回对应的 SQL NULL、TEXT、INTEGER 或 REAL 值。

当访问 JSON 中的字符串或 NULL 等单个值时，MySQL 与 SQLite json_extract() 的差异才明显。 下表展示了差异：

操作SQLite 结果MySQL 结果
json_extract('{"a":null,"b":"xyz"}','$.a')NULL'null'
json_extract('{"a":null,"b":"xyz"}','$.b')'xyz''"xyz"'

4.8. jsonb_extract() 函数

jsonb_extract() 函数与json_extract() 完全一致，唯一的区别是：当 json_extract() 通常会返回文本 JSON 数组对象时，该例程返回 JSONB 格式的数组或对象。 在返回文本、数值、null 或布尔类型等常见情况时，jsonb_extract() 与 json_extract() 表现完全一致。

4.9. -> 与 ->> 操作符

从 SQLite 3.38.0（2022-02-22）版本起，-> 与 ->> 操作符可用于提取 JSON 的子组件。 SQLite 版本的 -> 与 ->> 尽力兼容 MySQL 与 PostgreSQL。 这两个操作符的左操作数接受 JSON 字符串或 JSONB BLOB，右操作数接受 PATH 表达式、对象字段标签或数组索引。 -> 操作符返回所选子组件的文本 JSON 表示，若子组件不存在则返回 NULL。 ->> 操作符返回代表所选组件的 SQL TEXT、INTEGER、REAL 或 NULL 值，若子组件不存在则返回 NULL。

-> 与 ->> 操作符选择相同的 JSON 子组件。差异在于 -> 始终返回该子组件的 JSON 表示，而 ->> 始终返回该子组件的 SQL 表示。 因此，这些操作符与具有两个参数的json_extract() 函数调用略有不同。 两参数 json_extract() 仅当子组件为 JSON 数组或对象时才返回 JSON 表示；当子组件是 JSON null、字符串或数值时，则返回 SQL 表示。

当 -> 返回 JSON 时，始终返回该 JSON 的 RFC 8565 文本形式，而非 JSONB。 如果需要 JSONB 格式的子组件，请使用jsonb_extract() 函数。

-> 与 ->> 的右操作数可以是合法 JSON 路径表达式，这是 MySQL 使用的形式。 为兼容 PostgreSQL，它们也接受文本对象标签或整数数组索引作为右操作数。 如果右操作数为文本标签 X，则解释为 JSON 路径 '$.X'。 若右操作数为非负整数 N，则解释为 JSON 路径 '$[N]'； 若 N 为值为 -K 的负整数，则解释为 JSON 路径 '$[#-K]'，即从数组末尾向前索引。 负值索引仅在 SQLite 3.47.0（2024-10-21）及之后版本中支持。

示例：

  • '{"a":2,"c":[4,5,{"f":7}]}' -> '$' → '{"a":2,"c":[4,5,{"f":7}]}'
  • '{"a":2,"c":[4,5,{"f":7}]}' -> '$.c' → '[4,5,{"f":7}]'
  • '{"a":2,"c":[4,5,{"f":7}]}' -> 'c' → '[4,5,{"f":7}]'
  • '{"a":2,"c":[4,5,{"f":7}]}' -> '$.c[2]' → '{"f":7}'
  • '{"a":2,"c":[4,5,{"f":7}]}' -> '$.c[2].f' → '7'
  • '{"a":2,"c":[4,5,{"f":7}]}' ->> '$.c[2].f' → 7
  • '{"a":2,"c":[4,5,{"f":7}]}' -> 'c' -> 2 ->> 'f' → 7
  • '{"a":2,"c":[4,5],"f":7}' -> '$.c[#-1]' → '5'
  • '{"a":2,"c":[4,5,{"f":7}]}' -> '$.x' → NULL
  • '[11,22,33,44]' -> 3 → '44'
  • '[11,22,33,44]' ->> 3 → 44
  • '{"a":"xyz"}' -> '$.a' → '"xyz"'
  • '{"a":"xyz"}' ->> '$.a' → 'xyz'
  • '{"a":null}' -> '$.a' → 'null'
  • '{"a":null}' ->> '$.a' → NULL

4.10. json_insert()、json_replace() 与 json_set() 函数

json_insert()、json_replace() 与 json_set() 函数都接受一个 JSON 值作为第一个参数，后面跟零个或多个路径和值的成对参数，并通过这些路径/值对更新输入 JSON，并返回新的 JSON 字符串。 这三个函数在创建新值与覆盖现有值方面各有不同。

函数已存在时是否覆盖？不存在时是否创建？
json_insert()否是
json_replace()是否
json_set()是是

json_insert()、json_replace() 与 json_set() 函数始终接受奇数个参数。 第一个参数总是待编辑的原始 JSON。 后续参数以路径/值成对出现，每对的第一个元素是路径，第二个元素是要在该路径插入、替换或设置的值。

编辑按从左到右顺序依次进行。前面的编辑可能会影响后续编辑的路径搜索。

如果路径/值对的值是 SQLite TEXT，则通常作为带引号的 JSON 字符串插入，即便该字符串看起来像合法 JSON。 但如果该值是另一个 json 函数（如 json()、json_array() 或 json_object()）的结果，或是-> 操作符的结果，则视为 JSON 并插入其完整子结构。 由->> 操作符得到的值始终被解释为 TEXT，即便看起来像合法 JSON，也会作为 JSON 字符串插入。

如果第一个 JSON 参数不是合法 JSON，或任何 PATH 参数不是合法路径，或任一参数是 BLOB，会抛出错误。

要在数组末尾追加元素，可对 json_insert() 使用数组索引 "#”。示例：

  • json_insert('[1,2,3,4]','$[#]',99) → '[1,2,3,4,99]'
  • json_insert('[1,[2,3],4]','$[1][#]',99) → '[1,[2,3,99],4]'

其他示例：

  • json_insert('{"a":2,"c":4}', '$.a', 99) → '{"a":2,"c":4}'
  • json_insert('{"a":2,"c":4}', '$.e', 99) → '{"a":2,"c":4,"e":99}'
  • json_replace('{"a":2,"c":4}', '$.a', 99) → '{"a":99,"c":4}'
  • json_replace('{"a":2,"c":4}', '$.e', 99) → '{"a":2,"c":4}'
  • json_set('{"a":2,"c":4}', '$.a', 99) → '{"a":99,"c":4}'
  • json_set('{"a":2,"c":4}', '$.e', 99) → '{"a":2,"c":4,"e":99}'
  • json_set('{"a":2,"c":4}', '$.c', '[97,96]') → '{"a":2,"c":"[97,96]"}'
  • json_set('{"a":2,"c":4}', '$.c', json('[97,96]')) → '{"a":2,"c":[97,96]}'
  • json_set('{"a":2,"c":4}', '$.c', json_array(97,96)) → '{"a":2,"c":[97,96]}'

4.11. jsonb_insert()、jsonb_replace() 与 jsonb_set() 函数

jsonb_insert()、jsonb_replace() 与 jsonb_set() 函数的行为与json_insert()、json_replace()、json_set() 相同，不同之处在于 "jsonb_" 版本的结果以二进制 JSONB 格式返回。

4.12. json_object() 函数

json_object() SQL 函数接受零个或多个参数对，返回一个由这些参数组成的合法 JSON 对象。 每对参数的第一个元素是标签，第二个元素是值。 如果传入的任一参数是 BLOB，会抛出错误。

json_object() 当前允许标签重复而不报错，但未来可能更改。

SQL 类型为 TEXT 的参数通常被转换为带引号的 JSON 字符串，即便输入文本是合法 JSON。 但如果参数直接来自另一个 JSON 函数或-> 操作符（但不是 ->> 操作符），则会被视为 JSON，并保留其 JSON 类型信息与子结构。这使得可以嵌套调用 json_object() 与 json_array()。 json() 函数也可以用来强制让字符串被识别为 JSON。

示例：

  • json_object('a',2,'c',4) → '{"a":2,"c":4}'
  • json_object('a',2,'c','{e:5}') → '{"a":2,"c":"{e:5}"}'
  • json_object('a',2,'c',json_object('e',5)) → '{"a":2,"c":{"e":5}}'

4.13. jsonb_object() 函数

jsonb_object() 函数与json_object() 功能相同，只不过返回结果为二进制 JSONB 格式。

4.14. json_patch() 函数

json_patch(T,P) SQL 函数运行RFC-7396 MergePatch 算法，将补丁 P 应用到输入 T，并返回补丁完成后的 T 副本。

MergePatch 可以添加、修改或删除 JSON 对象的元素，因此对于 JSON 对象而言，json_patch() 是json_set() 与 json_remove() 的广义替代。 不过 MergePatch 将 JSON 数组视为原子。MergePatch 无法追加数组或修改数组中的单个元素，只能整体插入、替换或删除整个数组。 因此，当处理包含数组（尤其是包含大量子结构的数组）的 JSON 时，json_patch() 就不太实用了。

示例：

  • json_patch('{"a":1,"b":2}','{"c":3,"d":4}') → '{"a":1,"b":2,"c":3,"d":4}'
  • json_patch('{"a":[1,2],"b":2}','{"a":9}') → '{"a":9,"b":2}'
  • json_patch('{"a":[1,2],"b":2}','{"a":null}') → '{"b":2}'
  • json_patch('{"a":1,"b":2}','{"a":9,"b":null,"c":8}') → '{"a":9,"c":8}'
  • json_patch('{"a":{"x":1,"y":2},"b":3}','{"a":{"y":9},"c":8}') → '{"a":{"x":1,"y":9},"b":3,"c":8}'

4.15. jsonb_patch() 函数

jsonb_patch() 函数与json_patch() 相同，只不过补丁后的 JSON 以 JSONB 格式返回。

4.16. json_pretty() 函数

json_pretty() 函数的工作方式类似于 json()，但会添加额外空白以便人类阅读。 第一个参数是要美化的 JSON 或 JSONB。 可选的第二个参数是用于缩进的文本字符串。 如果省略第二个参数或其为 NULL，则默认每层缩进四个空格。

json_pretty() 在 SQLite 3.46.0（2024-05-23）版本中被加入。

4.17. json_remove() 函数

json_remove(X,P,...) 函数以单个 JSON 值 X 作为第一个参数，后面跟零个或多个路径参数。 json_remove(X,P,...) 返回 X 的副本，并移除所有由路径参数标识的元素。 未找到的路径会被静默忽略。

移除操作从左到右依次执行。前一次移除可能影响后续路径搜索。

如果 json_remove(X) 没有路径参数，则返回重新格式化的输入 X，去除多余空白。

如果第一个参数不是合法 JSON 或任何之后的参数不是合法路径，json_remove() 会抛出错误。

示例：

  • json_remove('[0,1,2,3,4]','$[2]') → '[0,1,3,4]'
  • json_remove('[0,1,2,3,4]','$[2]','$[0]') → '[1,3,4]'
  • json_remove('[0,1,2,3,4]','$[0]','$[2]') → '[1,2,4]'
  • json_remove('[0,1,2,3,4]','$[#-1]','$[0]') → '[1,2,3]'
  • json_remove('{"x":25,"y":42}') → '{"x":25,"y":42}'
  • json_remove('{"x":25,"y":42}','$.z') → '{"x":25,"y":42}'
  • json_remove('{"x":25,"y":42}','$.y') → '{"x":25}'
  • json_remove('{"x":25,"y":42}','$') → NULL

4.18. jsonb_remove() 函数

jsonb_remove() 函数的行为与json_remove() 完全一致，只不过编辑后的 JSON 以 JSONB 格式返回。

4.19. json_type() 函数

json_type(X) 函数返回 X 最外层元素的“类型”。 json_type(X,P) 返回 X 中由路径 P 所选择元素的“类型”。 json_type() 返回的“类型”为以下 SQL 文本之一： 'null'、'true'、'false'、'integer'、'real'、'text'、'array' 或 'object'。 如果 json_type(X,P) 的路径 P 所指元素在 X 中不存在，则该函数返回 NULL。

如果第一个参数不是合法 JSON 或 JSONB，或第二个参数不是合法 JSON 路径，json_type() 会抛出错误。

示例：

  • json_type('{"a":[2,3.5,true,false,null,"x"]}') → 'object'
  • json_type('{"a":[2,3.5,true,false,null,"x"]}','$') → 'object'
  • json_type('{"a":[2,3.5,true,false,null,"x"]}','$.a') → 'array'
  • json_type('{"a":[2,3.5,true,false,null,"x"]}','$.a[0]') → 'integer'
  • json_type('{"a":[2,3.5,true,false,null,"x"]}','$.a[1]') → 'real'
  • json_type('{"a":[2,3.5,true,false,null,"x"]}','$.a[2]') → 'true'
  • json_type('{"a":[2,3.5,true,false,null,"x"]}','$.a[3]') → 'false'
  • json_type('{"a":[2,3.5,true,false,null,"x"]}','$.a[4]') → 'null'
  • json_type('{"a":[2,3.5,true,false,null,"x"]}','$.a[5]') → 'text'
  • json_type('{"a":[2,3.5,true,false,null,"x"]}','$.a[6]') → NULL

4.20. json_valid() 函数

json_valid(X,Y) 函数如果参数 X 是合法 JSON，则返回 1；否则返回 0。 参数 Y 是一个整数位掩码，用于定义“合法”的含义。目前已定义 Y 的如下位：

  • 0x01 → 输入是严格符合 RFC-8259 规范 JSON 的文本，不含任何扩展。
  • 0x02 → 输入是包含上述JSON5 扩展的文本 JSON。
  • 0x04 → 输入是表面看起来像JSONB 的 BLOB。
  • 0x08 → 输入是严格符合内部JSONB 格式的 BLOB。

通过组合各个位，可以得到以下常用的 Y 值：

  • 1 → X 是 RFC-8259 JSON 文本
  • 2 → X 是JSON5 文本
  • 4 → X 可能是JSONB
  • 5 → X 是 RFC-8259 JSON 文本或JSONB
  • 6 → X 是JSON5 文本或JSONB ← 这可能是你想要的值
  • 8 → X 是严格规范的JSONB
  • 9 → X 是 RFC-8259 或严格规范的JSONB
  • 10 → X 是 JSON5 或严格规范的JSONB

Y 参数是可选的。若省略，默认为 1，即默认行为只在输入 X 是严格符合 RFC-8259 的 JSON 文本（无扩展）时返回 true。 这使得带一个参数的 json_valid() 可与在添加对JSON5与JSONB 支持之前的旧版本 SQLite 兼容。

Y 参数中的 0x04 与 0x08 区别在于，0x04 仅检查 BLOB 的最外层包装，判断其是否表面上像JSONB。 这足以应对大多数场景且速度很快。 0x08 位则对 BLOB 内部所有细节进行彻底检查，所需时间与输入 X 的大小线性相关，速度较慢。 因此建议在大多数情况下使用 0x04 位。

如果只是想知道某个值是否可能成为其他 JSON 函数的有效输入，Y 值为 6 也许是最合适的选择。

对于当前版本的 json_valid()，任何小于 1 或大于 15 的 Y 值都会引发错误。 不过，将来版本的 json_valid() 可能会扩展为接收该范围之外的标志，赋予新的含义。

如果 json_valid() 的 X 或 Y 输入为 NULL，该函数返回 NULL。

示例：

  • json_valid('{"x":35}') → 1
  • json_valid('{x:35}') → 0
  • json_valid('{x:35}',6) → 1
  • json_valid('{"x":35') → 0
  • json_valid(NULL) → NULL

4.21. json_quote() 函数

json_quote(X) 函数将 SQL 值 X（数字或字符串）转换为相应的 JSON 表示。 如果 X 是另一个 JSON 函数返回的 JSON 值，该函数相当于无操作。

示例：

  • json_quote(3.14159) → 3.14159
  • json_quote('verdant') → '"verdant"'
  • json_quote('[1]') → '"[1]"'
  • json_quote(json('[1]')) → '[1]'
  • json_quote('[1,') → '"[1,"'

4.22. 数组与对象聚合函数

json_group_array(X) 是一个聚合 SQL 函数，它返回一个由聚合中所有 X 值组成的 JSON 数组。 同样，json_group_object(NAME,VALUE) 返回一个由聚合中所有 NAME/VALUE 对组成的 JSON 对象。 “jsonb_” 变体与其完全相同，只不过返回的是二进制JSONB 格式。

4.23. 用于解析 JSON 的表值函数：json_each()、jsonb_each()、json_tree() 与 jsonb_tree()

json_each(X)、jsonb_each(X)、json_tree(X) 与 jsonb_tree(X) 表值函数都会遍历其第一个参数中的 JSON 值，并为每个元素返回一行。 json_each(X) 与 jsonb_each(X) 只遍历顶层数组或对象的直接子元素；如果顶层元素是原子值，则只遍历该元素本身。 json_tree(X) 与 jsonb_tree(X) 则递归遍历整个 JSON 子结构，从顶层元素开始。

json_each(X,P)、jsonb_each(X,P)、json_tree(X,P) 与 jsonb_tree(X,P) 函数的行为与它们的一参版本相同，只不过它们将路径 P 所标识的元素视为顶层元素。

jsonb_each() 与 jsonb_tree() 变体从 SQLite 3.51.0（2025-11-04）版本开始提供。 这两个变体与不带“b”的对应函数完全一样，只不过当“type”列为 'object' 或 'array' 时，“value”列返回的是JSONB，而非文本 JSON。

所有 JSON 表值函数返回的表结构如下： 

CREATE TABLE json_tree(
    key ANY,             -- 当前元素相对于父项的键
    value ANY,           -- 当前元素的值
    type TEXT,           -- 'object','array','string','integer' 等
    atom ANY,            -- 原子类型的值，数组与对象为 NULL
    id INTEGER,          -- 当前元素的整数 ID
    parent INTEGER,      -- 当前元素的父元素 ID
    fullkey TEXT,        -- 描述当前元素的完整路径
    path TEXT,           -- 当前行容器的路径
    json JSON HIDDEN,    -- 第一个输入参数：原始 JSON
    root TEXT HIDDEN     -- 第二个输入参数：起始 PATH
);

"key" 列表示 JSON 数组元素的整数索引或 JSON 对象元素的文本标签。 在其它情况下键列为 NULL。

"value" 列是由“key”或“fullkey”指定的 JSON 元素的 SQL 值。 如果元素的“type”为 'null'、'true'、'false'、'integer'、'real' 或 'text'，则返回该元素对应的 SQL 值作为"value"。 当“type”为 'array' 或 'object' 时，在 json_each() 与 json_tree() 中，"value" 列返回数组或对象的文本 JSON； 在 jsonb_each() 或 jsonb_tree() 中则返回 JSONB。 "value" 列的格式是 json_each() 与 jsonb_each()、json_tree() 与 jsonb_tree() 行为之间的唯一差别。

"type" 列是一个取自 ('null', 'true', 'false', 'integer', 'real', 'text', 'array', 'object') 的 SQL 文本值，对应当前 JSON 元素的类型。

"atom" 列对应原子元素（即数组与对象之外的元素）的 SQL 值。 对于 JSON 数组或对象，"atom" 列为 NULL。

"id" 列是一个整数，用于标识完整 JSON 字符串中的特定元素。 "id" 是一个内部管理编号，未来版本可能会改变，但唯一保证是，每一行的 "id" 都不同。

"parent" 列在 json_each() 中始终为 NULL。 对于 json_tree()，"parent" 列是当前元素父元素的 "id" 整数；对于顶层 JSON 元素或由第二个参数指定的根路径元素，则为 NULL。

"fullkey" 列是一个文本路径，唯一标识原始 JSON 字符串中的当前行元素。 即便提供了“root”参数作为不同起点，也会返回指向真实顶层元素的完整键。

"path" 列是当前行所在容器（数组或对象）的路径；或者当迭代从原子类型开始并仅输出单行时，则表示当前行自身的路径。

4.23.1. 使用 json_each() 与 json_tree() 的示例

假设表 "CREATE TABLE user(name,phone)" 将用户的电话号码以 JSON 数组对象存储在 user.phone 字段。 要查找拥有任何 704 区号电话号码的用户： 

SELECT DISTINCT user.name
  FROM user, json_each(user.phone)
 WHERE json_each.value LIKE '704-%';

现在假设 user.phone 字段在用户只有一个电话号码时存储普通文本，而在有多个电话号码时存储 JSON 数组。 同样的问题是：“哪些用户的电话号码包含 704 区号？”。 不过由于 json_each() 的第一个参数必须是合法 JSON，因此只能对拥有两个或以上电话号码的用户调用 json_each()： 

SELECT name FROM user WHERE phone LIKE '704-%'
UNION
SELECT user.name
  FROM user, json_each(user.phone)
 WHERE json_valid(user.phone)
   AND json_each.value LIKE '704-%';

再看一个不同的数据库，表定义为 "CREATE TABLE big(json JSON)"。 要逐行分解数据，可以这样： 

SELECT big.rowid, fullkey, value
  FROM big, json_tree(big.json)
 WHERE json_tree.type NOT IN ('object','array');

上述查询中的 WHERE 子句“type NOT IN ('object','array')”会压制容器，仅保留叶子元素。 也可以通过下面的方式达到同样效果： 

SELECT big.rowid, fullkey, atom
  FROM big, json_tree(big.json)
 WHERE atom IS NOT NULL;

假设 BIG 表的每条记录是一个 JSON 对象，包含唯一标识符字段 '$.id'，以及可能包含深度嵌套对象的 '$.partlist' 字段。 你想找出每一条记录的 id，只要其 '$.partlist' 中任意位置包含 uuid '6fa5181e-5721-11e5-a04e-57f3d7b32808'。 

SELECT DISTINCT json_extract(big.json,'$.id')
  FROM big, json_tree(big.json, '$.partlist')
 WHERE json_tree.key='uuid'
   AND json_tree.value='6fa5181e-5721-11e5-a04e-57f3d7b32808';
