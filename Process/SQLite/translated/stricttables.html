STRICT 表

1. 介绍

SQLite 力求在其存储内容的数据类型方面保持灵活。例如，如果某个表的列类型为“INTEGER”，则 SQLite 会尝试将插入该列的任何内容转换为整数。因此尝试插入字符串 '123' 会插入整数 123。但如果内容无法无损地转换为整数，例如输入是 'xyz'，则会插入原始字符串。有关更多信息，请参见 SQLite 中的数据类型 文档。

一些开发者 欣赏 SQLite 灵活的类型规则 并利用这份自由。但其他开发者对 SQLite 明目张胆地违反规则感到震惊，并更倾向于其他所有 SQL 数据库引擎和 SQL 标准中存在的传统严格类型系统。对于后者的这部分开发者，SQLite 从 3.37.0（2021-11-27）版本起支持一种针对每张表单独启用的严格类型模式。

2. STRICT 表

在 CREATE TABLE 语句中，如果在最后闭合的 “)” 之后添加了 “STRICT” 表选项关键字，则该表会应用严格的类型规则。 STRICT 关键字会导致以下情况发生差异：

  1. 每个列定义都必须为该列指定数据类型。不能省略列的数据类型。

  2. 数据类型必须是下列之一：

    • INT
    • INTEGER
    • REAL
    • TEXT
    • BLOB
    • ANY

    不允许使用其他数据类型名称，未来 SQLite 版本可能会添加新类型。

  3. 插入到非 ANY 类型列的内容必须是 NULL（前提是该列没有 NOT NULL 约束）或指定的类型。SQLite 会按照通常的亲和性规则尝试将数据强制转换为相应类型，正如 PostgreSQL、MySQL、SQL Server 和 Oracle 所做的那样。如果值无法在指定的数据类型中无损地转换，则会引发 SQLITE_CONSTRAINT_DATATYPE 错误。

  4. 数据类型为 ANY 的列可以接受任何类型的数据（当然如果有 NOT NULL 约束则会拒绝 NULL 值）。STRICT 表中类型为 ANY 的列不会发生类型强制转换。

  5. 属于 PRIMARY KEY 的列隐式具有 NOT NULL 约束。然而，即使 PRIMARY KEY 隐式有 NOT NULL 约束，当向 INTEGER PRIMARY KEY 列插入 NULL 值时，该 NULL 会自动按普通非严格表的 INTEGER PRIMARY KEY 规则转换为唯一整数。

  6. PRAGMA integrity_check 和 PRAGMA quick_check 命令会检查 STRICT 表中所有列内容的类型，如果有任何异常会报告错误。

STRICT 表的其他方面与普通非严格表相同：

  • CHECK 约束 的行为相同。
  • NOT NULL 约束 的行为相同。
  • FOREIGN KEY 约束 的行为相同。
  • UNIQUE 约束 的行为相同。
  • DEFAULT 子句 的行为相同。
  • COLLATE 子句 的行为相同。
  • 生成列 的行为相同。
  • ON CONFLICT 子句 的行为相同。
  • 索引 的行为相同。
  • AUTOINCREMENT 的行为相同。
  • INTEGER PRIMARY KEY 列是 rowid 的别名，但 INT PRIMARY KEY 列不是。
  • 表数据的磁盘格式 与 普通表 相同。

3. ANY 数据类型

在单列中承载任意类型数据的能力多年来被证明非常有用。为了在 STRICT 表中继续支持此能力，引入了新的 ANY 数据类型名称。当某列的数据类型为 “ANY” 时，表示整张表可以插入任意类型的数据——整数、浮点数、字符串或二进制 BLOB，其值和数据类型会如实保留。就我们所知，SQLite 是唯一支持此高级能力的 SQL 数据库引擎。

STRICT 表中 ANY 的行为与普通非严格表略有不同。在 STRICT 表中，类型为 ANY 的列总是按接收时的原样保存数据。对于普通非严格表中类型为 ANY 的列，它会尝试将看起来像数字的字符串转换为数值，如果成功则存储数值而不是原始字符串。例如：

STRICT普通非严格
CREATE TABLE t1(a ANY) STRICT;
INSERT INTO t1 VALUES('000123');
SELECT typeof(a), quote(a) FROM t1;
-- 结果: 文本 '000123'
CREATE TABLE t1(a ANY);
INSERT INTO t1 VALUES('000123');
SELECT typeof(a), quote(a) FROM t1;
-- 结果: 整数 123

4. 向后兼容性

CREATE TABLE 语句末尾的 STRICT 关键字仅在 SQLite 3.37.0（2021-11-27）及以后版本中被识别。如果尝试使用早期版本的 SQLite 打开包含 STRICT 关键字的数据库，它将无法识别该关键字并报告错误（除非另有说明）。不过，除了额外的 STRICT 关键字之外，数据库的底层 文件格式 是相同的。

因此，一般而言，包含一个或多个 STRICT 表的数据库文件只能由 SQLite 3.37.0 或更高版本读取和写入。然而，由 SQLite 3.37.0 或更高版本创建的数据库仍然可以被早期版本的 SQLite 读取和写入，只要数据库不包含任何 STRICT 表或其他在较旧版本中引入的功能，并且版本可追溯至 3.0.0（2004-06-18）。

STRICT 关键字仍可作为标识符使用。（它仅在语法的某个部分被视为关键字，而 sqlite3_keyword_check(..) 并不将其识别为常规关键字。）

4.1. 在旧版本 SQLite 中访问 STRICT 表

由于 SQL 语言解析器的一个特性，早于 3.37.0 的 SQLite 版本如果在打开数据库文件后、在执行任何需要了解模式的操作之前立即设置 “PRAGMA writable_schema=ON”，仍然可以读取和写入 STRICT 表。PRAGMA writable_schema=ON 的一个特性是它会禁用模式解析器中的错误。这是有意为之，因为设置 PRAGMA writable_schema=ON 的一个重要作用是方便恢复模式损坏的数据库文件。因此，在 writable_schema=ON 的情况下，当模式解析器遇到 STRICT 关键字时，它会对自己说“我不知道该怎么处理这个，但到目前为止的内容看起来像是一个有效的表定义，所以我就用我已有的东西。”因此 STRICT 关键字实际上被忽略。由于 STRICT 表的文件格式没有其他变化，其余部分会正常工作。当然，由于旧版本的 SQLite 不知道如何执行严格的类型检查，因此不会发生严格类型强制。

.dump 命令在 CLI 中会设置 PRAGMA writable_schema=ON，因为 .dump 的设计目标是即使从损坏的数据库文件中也尽可能多地提取内容。因此，如果您在旧版本的 SQLite 中打开一个包含 STRICT 表的数据库，并在进行其他操作之前在 CLI 中执行 ".dump" 命令，就可以在不强制类型检查的情况下访问那些 STRICT 表。这可能会通过允许不正确的类型进入 STRICT 表而导致数据库损坏。使用较新版本的 SQLite 重新打开数据库并运行 “PRAGMA quick_check” 将检测并报告所有此类损坏。

5. 其他表选项

SQLite 解析器在 CREATE TABLE 语句最终闭合的右括号之后接受一个逗号分隔的表选项列表。截至本文撰写之时（2021-08-23），仅识别两个选项：

  • STRICT
  • WITHOUT ROWID

如果存在多个选项，则可以按任意顺序指定。为了简化，当前的解析器会接受重复选项而不报错，但未来版本可能会改变，因此应用程序不应依赖这一行为。
