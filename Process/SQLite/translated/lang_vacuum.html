VACUUM

1. 语法

vacuum-stmt:

2. 描述

VACUUM 命令重新构建数据库文件，将其压缩到最小的磁盘空间。应用程序可能出于以下几个原因执行此操作：

  • 除非 SQLite 以 “auto_vacuum=FULL” 模式运行，否则当大量数据从数据库文件中被删除后，会留下空闲空间，或称“空闲”数据库页。这意味着数据库文件可能比实际所需更大。运行 VACUUM 重新构建数据库可以回收这些空间并减小数据库文件的大小。

  • 频繁的插入、更新和删除可能导致数据库文件碎片化——单个表或索引的数据散布在数据库文件的各处。运行 VACUUM 可确保每个表和索引在数据库文件中大致连续存储。在某些情况下，VACUUM 还会减少数据库中部分填充页的数量，进一步减小文件大小。

  • 当内容从 SQLite 数据库中删除时，内容通常不会被真正抹去，而是将用于保存内容的空间标记为可重用。这可能允许黑客或法医分析恢复已删除内容。运行 VACUUM 会清除数据库中的所有删除痕迹，从而防止对手恢复已删除内容。以这种方式使用 VACUUM 是设置 PRAGMA secure_delete=ON 的替代方案。

  • 通常，数据库的 page_size 以及是否支持 auto_vacuum 必须在数据库文件实际创建之前配置。然而，在非 write-ahead log 模式下，可以使用 page_size 和/或 pragma auto_vacuum 这两个 pragma 修改现有数据库的 page_size 和/或 auto_vacuum 属性，然后立即对数据库运行 VACUUM。处于 write-ahead log 模式时，只有 auto_vacuum 支持属性可以通过 VACUUM 修改。

默认情况下，VACUUM 在主数据库上运行。附加数据库 可以通过在 VACUUM 语句后追加适当的 schema-name 来进行清理。

兼容性警告： 在 版本 3.15.0（2016-10-14）中增加了清理附加数据库的能力。此前，在 VACUUM 语句中添加的 schema-name 会被静默忽略，并且只会清理“main” 模式。

2.1. 带 INTO 子句的 VACUUM

如果包含 INTO 子句，则原始数据库文件保持不变，并且会在 INTO 子句参数指定的文件中创建一个新数据库。该参数是一个标量的 表达式（例如文本字面值）。新数据库将包含与原始数据库相同的逻辑内容，并且已经完全清理。

带 INTO 子句的 VACUUM 命令是生成活动数据库备份的 备份 API 的替代方案。使用 VACUUM INTO 的优点是生成的备份数据库体积最小，从而减少了文件系统 I/O 的量。此外，所有已删除的内容都会从备份中清除，不留下任何取证痕迹。另一方面，备份 API 消耗的 CPU 周期更少，并且可以增量执行。

INTO 子句中的文件名可以是任何求值为字符串的 SQL 表达式。INTO 子句指定的文件在执行 VACUUM INTO 命令之前必须不存在，或者必须是一个空文件，否则 VACUUM INTO 命令将因错误而失败。

如果启用了 URI 文件名，那么 INTO 的参数可以是 URI 文件名。满足下列任一条件即可启用 URI 文件名：

  • SQLite 库在编译时使用了 -DSQLITE_USE_URI=1。
  • 启动时调用了 sqlite3_config(SQLITE_CONFIG_URI,1) 接口。
  • 正在运行 VACUUM INTO 语句的 数据库连接 最初是使用 SQLITE_OPEN_URI 标志打开的。

VACUUM INTO 命令具有事务性，意味着生成的输出数据库是原始数据库的一致快照。然而，如果 VACUUM INTO 命令在计划外的关闭或断电时被中断，那么生成的输出数据库可能会不完整并损坏。

但是，如果原始数据库的 PRAGMA synchronous 设置为 NORMAL 或 FULL，那么 SQLite 会在写入完成后调用 fsync() 或 FileFlushBuffers() 将输出数据库同步到磁盘。这意味着在这些情况下，如果 VACUUM INTO 命令完成后发生断电或计划外关闭（假设操作系统、文件系统和硬件均正常运行），则应该不会损坏数据库。

3. VACUUM 的工作原理

VACUUM 命令通过将数据库内容复制到临时数据库文件中，然后将原始数据库覆盖为临时文件的内容来工作。在覆盖原始文件时，会使用回滚日志或 write-ahead log WAL 文件，就像执行任何其他数据库事务一样。这意味着在对数据库运行 VACUUM 时，最多需要原始数据库文件大小两倍的空闲磁盘空间。

VACUUM INTO 命令的工作方式相同，只是它将 INTO 子句中指定的文件用作临时数据库，并省略了将清理后的数据库复制回原始数据库这一过程。

VACUUM 命令可能会更改任何没有显式 INTEGER PRIMARY KEY 的表中条目的 ROWIDs。

如果正在运行 VACUUM 的数据库连接上存在未完成的事务，则 VACUUM 会失败。未完成的 SQL 语句通常会保持读取事务打开，因此如果同一连接上有未完成的 SQL 语句，VACUUM 可能会失败。VACUUM（但不包括 VACUUM INTO）属于写操作，因此如果另一个数据库连接持有阻止写入的锁，则 VACUUM 会失败。

在删除数据后，VACUUM 命令之外的回收空间替代方案是通过 auto_vacuum pragma 启用自动清理模式。当数据库启用了 auto_vacuum 时，删除数据后可以回收空闲页，从而使文件缩小，而无需使用 VACUUM 重建整个数据库。然而，启用 auto_vacuum 可能会导致额外的数据库文件碎片化。并且 auto_vacuum 不会像 VACUUM 那样压缩数据库中部分填充的页。
