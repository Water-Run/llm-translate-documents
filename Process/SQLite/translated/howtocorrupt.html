概述

SQLite 数据库对于损坏具有很高的抗性。 如果在事务中发生应用崩溃、操作系统崩溃甚至断电， 在下一次访问数据库文件时，部分写入的事务应该会自动回滚。 恢复过程是完全自动的，不需要用户或应用的任何操作。

虽然 SQLite 对数据库损坏具有抗性，但并非万无一失。 本文档描述了 SQLite 数据库可能损坏的各种方式。

1. 恶意线程或进程覆盖文件

SQLite 数据库文件是普通磁盘文件。 这意味着任何进程都可以打开该文件，并用垃圾内容覆盖它。 SQLite 库无法对此提供防护。

1.1. 在文件描述符关闭后继续使用它

我们看到多个案例中，一个文件的文件描述符被打开， 随后该文件描述符被关闭并在 SQLite 数据库上重新打开。 之后，某个其他线程继续向旧的文件描述符写入数据， 没有意识到原始文件已经关闭。 但是因为 SQLite 已重新打开该文件描述符， 原计划写入原始文件的信息最终覆盖了 SQLite 数据库的一部分， 导致数据库损坏。

一个示例发生在 2013-08-30 左右，出现在 Fossil DVCS 的规范仓库中。 在该事件中，文件描述符 2（标准错误）在 stunnel（我们推测）在 sqlite3_open_v2() 之前错误地被关闭， 因此用于仓库数据库文件的文件描述符是 2。 后来，一个应用程序错误导致 assert() 语句通过调用 write(2,...) 输出错误信息。 但是因为文件描述符 2 此时已连接到数据库文件， 错误信息覆盖了数据库的一部分。为了防止此类问题， SQLite 从 版本 3.8.1（2013-10-17）起 拒绝将低编号的文件描述符用于数据库文件。 （更多信息见 SQLITE_MINIMUM_FILE_DESCRIPTOR。）

另一个因使用已关闭文件描述符而导致损坏的示例 在 2014-08-12 由 Facebook 工程师报告。

另一个错误示例在 2019-07-11 针对 Fossil 报告。一个文件描述符被打开用于调试输出， 但随后被关闭并由 SQLite 重新打开。 然而调试逻辑继续向原始文件描述符写入。 参见论坛讨论了解该漏洞报告及修复链接。

1.2. 在事务活动期间备份或恢复

在后台运行自动备份的系统可能会试图在事务进行中 复制 SQLite 数据库文件。 备份副本可能包含旧内容和新内容，从而导致损坏。

有多种安全的方法来复制 SQLite 数据库——安全是指生成正确且 未损坏的备份。以下不分先后：

  1. sqlite3_rsync 工具程序（从 SQLite 3.47.0（2024-10-21）起提供） 将通过带宽高效的协议在 SSH 上复制正在运行的 SQLite 数据库。

  2. VACUUM INTO filename 命令将 当前 SQLite 数据库的状态复制到单独的文件中。

  3. 备份 API 是一个 C 语言接口，可以创建一致的 SQLite 数据库副本。

上述任一方法即使在数据库仍在运行时也适用。 只要复制期间没有事务在进行，就可以安全地复制 SQLite 数据库文件。 如果前一个写事务失败，那么必须确保任何回滚日志（*-journal 文件） 或写前日志（*-wal 文件）与数据库文件一起被复制。

1.3. 删除活动日志

SQLite 通常将所有内容存储在单个磁盘文件中。 然而，在执行事务期间，用于在崩溃或断电后恢复数据库的信息 会存储在辅助日志文件中。这样的日志文件被描述为 “活动”。 日志文件的名称与原数据库文件相同，仅加上 -journal 或 -wal 后缀。

SQLite 必须看到这些日志文件才能在崩溃或断电后恢复。 如果这些 活动日志文件 在崩溃或断电后被移动、删除或重命名， 则自动恢复无法进行，数据库可能损坏。

此问题的另一种表现为 因 8+3 文件名使用不一致导致的数据库损坏。

1.4. 数据库文件与活动日志不匹配

前面的示例是一个更普遍问题的特例： SQLite 数据库的状态由数据库文件和日志文件共同控制。 在静止状态下，日志文件不存在，只有数据库文件起作用。 但如果日志文件存在，它必须与数据库一起保留以避免损坏。 以下操作都可能导致损坏：

  • 在两个不同数据库间交换日志文件。
  • 用不同日志文件覆盖原有日志文件。
  • 将日志文件从一个数据库移动到另一个。
  • 复制数据库文件时未同时复制其日志。
  • 用另一个数据库文件覆盖原有文件却不删除原数据库的活动日志。

2. 文件锁定问题

SQLite 在数据库文件与 写前日志 或 WAL 文件上使用文件锁， 以协调并发进程之间的访问。 若没有协调，两个线程或进程可能会同时对数据库文件进行不兼容的修改，从而导致损坏。

2.1. 锁实现有缺陷或缺失的文件系统

SQLite 依赖底层文件系统按照文档所述执行锁定。 但一些文件系统在锁定逻辑上存在缺陷，使得锁并不总是按预期工作。 尤其是网络文件系统，尤其是 NFS。 如果在有缺陷的锁原语的文件系统上使用 SQLite，并且两个或更多线程或进程同时尝试访问同一数据库， 则可能导致数据库损坏。

2.2. 另一个线程调用 close() 取消 POSIX 建议锁

SQLite 在 unix 平台默认使用 POSIX 建议锁。 不幸的是，POSIX 建议锁设计上有一些奇怪之处，容易被误用或失效。 特别的是，任何在同一进程中持有文件描述符的线程，都可以使用另一个文件描述符覆盖该锁。 一个特别恶性的情况是 close() 系统调用会取消该进程中所有线程和所有文件描述符对同一文件的 POSIX 建议锁。

例如，假设一个多线程进程有两个或更多线程分别通过独立 SQLite 数据库连接访问同一数据库文件。 然后第三个线程出现，希望在不使用 SQLite 库的情况下从同一数据库文件中读取内容。 也许第三个线程想备份数据库，也许第三个线程只是试图识别文件类型，因此尝试读取前 16 个字节以确定它是否为 SQLite 数据库。 无论原因如何，第三个线程调用 open()、read() 然后 close()。 看起来这似乎无害。 但 close() 系统调用会导致该数据库上所有其他线程持有的锁被释放。 那些线程无法得知它们的锁刚被破坏（POSIX 不提供相关机制）， 因此它们继续假设锁仍然有效。 这可能导致两个或更多线程或进程同时尝试写入数据库，从而导致数据库损坏。

注意：多个线程使用 SQLite 库访问同一数据库文件是完全安全的。 SQLite 的 unix 驱动了解 POSIX 建议锁的问题并进行了规避。 该问题仅在线程试图绕过 SQLite 库直接读取数据库文件时发生。

从 SQLite 版本 3.51.0（2025-11-04）起， SQLite 实现了额外防御措施，以尽量避免因 close() 破坏锁而导致的问题。 这些新防御在数据库处于 WAL 模式 且被多个进程访问时有所帮助。 但它们不是万能良方。 为了避免损坏，开发者务必小心，切勿在一个或多个数据库连接仍打开时在任一线程中对 SQLite 数据库文件调用 close()。

2.3. 同一应用链接多份SQLite

如前一节所述，SQLite 会采取措施规避 POSIX 建议锁的怪癖。 其中一部分措施是维护一个全局列表（由互斥锁保护），列出所有已打开的 SQLite 数据库文件。 但如果同一应用链接了多个 SQLite 副本，就会存在多个这样的全局列表。 使用一个副本打开的数据库连接不会意识到使用另一副本打开的连接， 也无法规避 POSIX 建议锁的问题。 在一个连接上执行 close() 可能无意中清除了另一个连接上的锁，从而导致数据库损坏。

上述场景听起来有些牵强。 但 SQLite 开发者确实知道至少有一个商业产品正是因为该 bug 而发布。 该厂商向 SQLite 开发者寻求帮助，追查在 Linux 和 Mac 上出现的偶发性数据库损坏问题。 最终发现问题源于应用同时链接了两份 SQLite。 解决方案是更改构建流程，只链接一份 SQLite 而不是两份。

2.4. 两个进程使用不同锁协议

SQLite 在 unix 平台默认使用 POSIX 建议锁，但还可以选择其他方式。 通过使用 sqlite3_open_v2() 接口选择替代的 sqlite3_vfs，应用可以使用可能更适合某些文件系统的锁协议。 例如，在不能使用 POSIX 建议锁的 NFS 文件系统上运行的应用可能会选择点文件锁。

同一数据库文件的所有连接必须使用相同的锁协议。 如果一个应用使用 POSIX 建议锁而另一个使用点文件锁， 那么两个应用无法看到彼此的锁，也无法协调数据库访问， 可能导致数据库损坏。

2.5. 使用时取消链接或重命名数据库文件

如果两个进程同时打开同一数据库文件， 其中一个进程关闭连接、取消链接该文件，然后用相同名称创建新文件并重新打开它， 那么这两个进程实际上访问的是不同的数据库文件但名称相同。 （注意：这只可能发生在 Posix 或类似系统上，这些系统允许在文件仍被读写时取消链接。 Windows 不允许这种情况。） 由于回滚日志和 WAL 文件基于数据库文件名称， 这两个不同数据库文件将共享相同的回滚日志或 WAL 文件。 其中一个数据库的回滚或恢复可能使用来自另一个数据库的内容， 导致损坏。 如果在打开数据库文件时重命名它，并用旧名称创建一个新文件，会发生类似的问题。

换句话说，对打开的数据库文件执行取消链接或重命名会导致未定义且可能不受欢迎的行为。

从 SQLite 版本 3.7.17（2013-05-20）起， unix OS 接口如果在数据库文件仍在使用时取消链接它，会向 错误日志发送 SQLITE_WARNING 消息。

2.6. 同一文件的多个链接

如果单个数据库文件有多个链接（无论是硬链接还是软链接）， 那么这就意味着该文件有多个名称。 如果两个或更多进程使用不同名称打开数据库， 那么它们将使用不同的回滚日志和 WAL 文件。 这意味着如果一个进程崩溃，另一个进程将无法恢复正在进行的事务， 因为它将寻找错误的日志。

换句话说，打开并使用具有两个或更多名称的数据库文件将导致未定义且可能不受欢迎的行为。

从 SQLite 版本 3.7.17（2013-05-20）起， unix OS 接口如果数据库文件有多个硬链接，会向 错误日志发送 SQLITE_WARNING 消息。

从 SQLite 版本 3.10.0（2016-01-06）起， unix OS 接口会尝试解析符号链接，并通过其规范名称打开数据库文件。 在 3.10.0 之前，通过符号链接打开数据库文件类似于通过多个硬链接打开文件，行为未定义。

2.7. 在 fork() 过程中携带打开的数据库连接

不要打开 SQLite 数据库连接，然后 fork()，再在子进程中使用该连接。 这会导致各种锁定问题，极易造成数据库损坏。 SQLite 无法设计用于这种行为。 任何在子进程中使用的数据库连接都必须在子进程中打开，而不是从父进程继承。

即便如此，如果连接在父进程中打开，也不要在子进程中调用 sqlite3_close()。 关闭底层文件描述符是安全的，但 sqlite3_close() 可能会触发清理操作， 从而删除父进程正在使用的内容，导致错误甚至数据库损坏。

3. 同步失败

为了保证数据库文件始终一致，SQLite 会偶尔请求操作系统将所有待写内容刷新到持久存储， 然后等待刷新完成。 在 unix 上使用 fsync() 系统调用，在 Windows 上使用 FlushFileBuffers()。 我们把这种待写刷新称为“同步”。

实际上，如果只关心原子性与一致性，并且愿意放弃耐久性， 同步操作无需等待内容完全写入持久介质。 同步操作可以被视为 I/O 障碍。 只要同步之前的所有写入在同步之后的任何写入发生之前完成，就不会发生数据库损坏。 如果同步作为 I/O 障碍而非真正同步运行， 那么断电或系统崩溃可能会使一个或多个先前提交的事务回滚 （违反“ACID”中的“持久性”），但数据库至少仍将保持一致， 而这正是多数人关心的。

3.1. 不遵守同步请求的磁盘

不幸的是，大多数消费级大容量存储设备都在同步方面撒谎。 磁盘驱动器会在内容到达磁道缓冲区后而未写入氧化层之前，报告其已经安全存储在持久介质中。 这使磁盘看似运行更快（对制造商而言非常重要，以便在行业期刊上展示优良基准）。 公平地说，这种谎言通常不会造成伤害，只要在磁道缓冲区实际写入氧化层之前没有发生断电或硬复位。 但如果断电或硬复位确实发生，且导致同步之后写入的内容已写入氧化层， 而同步之前写入的内容仍在磁道缓冲区中，就可能导致数据库损坏。

USB 闪存盘在同步请求方面尤其容易撒谎。 可以通过向 USB 闪存盘上的 SQLite 数据库提交一个大事务来观察到这一点。 COMMIT 命令返回较快，说明闪存告诉操作系统，操作系统又告诉 SQLite 所有内容已安全写入持久存储， 但闪存盘尾部的 LED 灯会继续闪烁几秒钟。 在 LED 仍闪烁时拔出闪存盘通常会导致数据库损坏。

请注意，SQLite 必须相信操作系统和硬件关于同步请求状态的信息。 SQLite 无法检测它们是否在撒谎以及写入是否可能乱序。 然而，在 WAL 模式 中的 SQLite 对乱序写入要宽容得多。 在 WAL 模式中，唯一次同步失败可能导致数据库损坏的情况是在 检查点 操作期间。 COMMIT 中的同步失败可能导致耐久性丢失，但不会导致数据库文件损坏。 因此，应对同步失败导致数据库损坏的一条防线是使用 WAL 模式，并尽可能少做检查点。

3.2. 通过 PRAGMA 禁用同步

SQLite 为确保完整性所执行的同步操作可以通过 同步 PRAGMA 在运行时禁用。 将 PRAGMA synchronous=OFF 设置后，所有同步操作都会被省略。 这使得 SQLite 运行得更快，但也允许操作系统随意重新排序写入， 如果在所有内容到达持久存储之前发生断电或硬复位，可能会导致数据库损坏。

为了获得最大可靠性和对数据库损坏的弹性， SQLite 应始终以默认的 FULL 同步设置运行。

4. 磁盘和闪存故障

如果文件内容因磁盘或闪存故障而发生变化，SQLite 数据库可能会损坏。 这很罕见，但磁盘有时会在扇区中间翻转某一位。

4.1. 非安全断电的闪存控制器

据说在某些闪存控制器中，如果写入过程中断电，磨损均衡逻辑可能会导致随机的文件系统损坏。 这可能表现为文件中随机位置的变化，该文件当时甚至都没有被打开。 例如，设备正在向闪存中的 MP3 文件写入内容时断电， 则即便 SQLite 数据库未被使用，也可能被损坏。

4.2. 虚假容量 USB 盘

市场上流通着许多声称具有高容量（如 8GB）但实际上只能存储更小容量（如 1GB）的诈欺 USB 盘。 在这些设备上写入通常会覆盖其他文件。 使用此类虚假闪存设备极易导致数据库损坏。 搜索“fake capacity usb”可以找到大量关于该问题的令人不安的信息。

5. 内存损坏

SQLite 是一个 C 库，运行于它所服务的应用的相同地址空间。 这意味着应用中的野指针、缓冲区溢出、堆损坏或其他故障可能破坏 SQLite 的内部数据结构，最终产生损坏的数据库文件。 通常这类问题会先表现为段错误，然后才导致数据库损坏，但也曾有应用代码错误使 SQLite 微妙地异常，导致数据库文件损坏而非崩溃。

当使用 内存映射 I/O 时，内存损坏问题更为严重。 当数据库文件的全部或部分映射到应用地址空间后，任何野指针覆盖映射区域的任意部分都会立即损坏数据库文件，而无需应用再做写入系统调用。

6. 其他操作系统问题

有时操作系统会表现出非标准行为，导致问题。 有时这种非标准行为是故意的，有时是实现中的错误。 但无论如何，如果操作系统的表现不同于 SQLite 期望的行为，就存在数据库损坏的可能性。

6.1. Linux 线程

一些较旧版本的 Linux 使用 LinuxThreads 库支持线程。 LinuxThreads 类似于 Pthreads，但在处理 POSIX 建议锁方面稍有不同。 SQLite 2.2.3 到 3.6.23 版本会在运行时检测到使用 LinuxThreads，并采取相应措施以应对其非标准行为。 但大多数现代 Linux 实现使用更新的、正确的 NPTL 实现 Pthreads。 从 SQLite 版本 3.7.0（2010-07-21）起，默认使用 NPTL，不再检查。 因此，较新版本的 SQLite 在运行于使用 LinuxThreads 的旧 Linux 系统的多线程应用中可能会表现异常并损坏数据库文件。

6.2. QNX 上 mmap() 的失败

在 QNX 上，mmap() 存在一个微妙的问题： 对同一个文件描述符进行第二次 mmap() 调用可能会导致第一次 mmap() 得到的内存被清零。 Unix 下的 SQLite 使用 mmap() 为 WAL 模式 中的事务协调创建共享内存区域， 并在处理大事务时多次调用 mmap()。 已证实，在这种场景中 QNX 的 mmap() 会损坏数据库文件。 QNX 工程师已意识到该问题并正在解决；在你阅读本文时问题可能已修复。

在 QNX 上运行时，建议不要使用 内存映射 I/O。 此外，如需使用 WAL 模式，建议应用采用 独占锁模式，以便使用 无共享内存的 WAL。

6.3. 文件系统损坏

由于 SQLite 数据库是普通磁盘文件，任何文件系统故障都可能损坏数据库。 现代操作系统中的文件系统非常可靠，但仍然会出错。 例如，2013-10-01，保存 Tcl/Tk 维基 的 SQLite 数据库在托管计算机迁移到一个有文件系统层问题的（Linux）内核后几天就损坏了。 当时文件系统最终损坏严重到无法使用，但最早的症状就是 SQLite 数据库损坏。

7. SQLite 配置错误

SQLite 拥有许多内建保护来防止数据库损坏。 但许多保护机制可以通过配置选项禁用。 如果保护被禁用，则可能发生数据库损坏。

以下是禁用 SQLite 内建保护机制的一些示例：

  • 将 PRAGMA synchronous=OFF 设置为 OFF， 在操作系统崩溃或断电时可能导致数据库损坏，尽管该设置对于应用崩溃是安全的。

  • 在其他数据库连接仍打开时更改 PRAGMA schema_version。

  • 使用 PRAGMA journal_mode=OFF 或 PRAGMA journal_mode=MEMORY， 并在写事务进行中发生应用崩溃。

  • 设置 PRAGMA writable_schema=ON，然后通过 DML 语句更改数据库模式， 如果操作不慎可能导致数据库完全不可读。

8. SQLite 中的缺陷

SQLite 经过 非常严格的测试，以确保尽可能无缺陷。 每个 SQLite 版本都有大量测试，用以模拟断电、I/O 错误和内存不足（OOM）错误， 以验证在这些情况下不会发生数据库损坏。 SQLite 在约二十亿个活跃部署中也得到了实地验证，没有出现严重问题。

尽管如此，没有软件是百分之百完美的。 SQLite 曾有一些历史缺陷（现已修复），可能导致数据库损坏。 也许仍有未发现的问题。 由于 SQLite 广泛测试与广泛使用，导致数据库损坏的缺陷往往非常隐蔽。 应用遇到 SQLite 缺陷的概率很小。 为了说明这一点，下面给出了一个帐目， 列出了在 2009-04-01 至 2013-04-15 四年期间发现的所有数据库损坏缺陷。 这份帐目应该能让读者直观理解那些成功躲过测试并进入发行版的 SQLite 缺陷类型。

8.1. 数据库收缩导致的错误损坏报告

如果一个数据库先由 SQLite 3.7.0 或更高版本写入， 然后以某种方式被 SQLite 3.6.23 或更早版本再次写入，使数据库文件大小减小， 那么下次 SQLite 3.7.0 访问该文件时，可能会报告数据库文件已损坏。 然而数据库文件实际上并未损坏。 只是 3.7.0 版本在损坏检测上过于激进。

该问题于 2011-02-20 解决。 修复首次出现在 SQLite 版本 3.7.6（2011-04-12）。

8.2. 在回滚与 WAL 模式之间切换导致的损坏

在单个线程或进程中反复在 WAL 模式 和普通模式之间切换， 并在切换间运行 VACUUM 命令， 可能会导致另一个进程或线程在其打开数据库的同时未察觉数据库已更改。 该进程或线程随后可能尝试使用过时缓存修改数据库， 从而导致损坏。

该问题在内部测试中发现，未曾在实际环境中观察到。 问题于 2011-01-27 被发现，并在版本 3.7.5 中修复。

8.3. 获取锁时的 I/O 错误导致损坏

如果操作系统在尝试获取 WAL 模式 中共享内存的某个锁时返回 I/O 错误， 则 SQLite 可能无法重置其缓存， 如果随后尝试写入，可能导致数据库损坏。

请注意，该问题仅在获取锁的尝试返回 I/O 错误时发生。 如果锁只是未被授予（因为另一个线程或进程已经持有冲突锁），则永远不会发生损坏。 我们不知道有哪个操作系统在尝试获取共享内存文件锁时会以 I/O 错误失败。 因此这是一个理论问题，而非现实问题。 可以肯定的是，该问题从未在实际环境中观察到。 该问题是在使用模拟 I/O 错误的测试平台进行 SQLite 压力测试时发现的。

该问题在 2010-09-20 被修复，针对 SQLite 版本 3.7.3。

8.4. 数据库页从空闲页列表泄露

当从 SQLite 数据库中删除内容时，不再使用的页面会加入空闲列表， 并被后续插入重用。 存在于 SQLite 3.6.16 到 3.7.2 版本中的一个错误可能导致在使用 incremental_vacuum 时， 空闲列表中的页面丢失。 这不会导致数据丢失，但会导致数据库文件比必要的要大。 并且会导致 integrity_check pragma 报告空闲列表中缺失页面。

该问题在 2010-08-23 被修复，针对 SQLite 版本 3.7.2。

8.5. 3.6 与 3.7 交替写入导致的损坏

SQLite 3.7.0 版本在 SQLite 数据库文件格式上引入了许多新特性（例如但不限于 WAL）。 3.7.0 是这些新特性的试运行版本。 我们预期会发现问题，也确实发现了。

如果数据库最初由 SQLite 3.7.0 创建， 然后由 SQLite 3.6.23.1 写入，使数据库文件大小增加， 再由 SQLite 3.7.0 写入， 则数据库文件可能会损坏。

该问题在 2010-08-04 被修复，针对 SQLite 版本 3.7.1。

8.6. Windows 系统恢复中的竞争条件

SQLite 3.7.16.2 修复了 Windows 系统锁定逻辑中的一个微妙竞争条件。 当数据库文件需要恢复（前一个写入该文件的进程在事务中途崩溃）， 且两个或更多进程同时尝试打开该数据库时，竞争条件可能导致其中一个进程错误地认为恢复已完成，从而继续使用该数据库而未先运行恢复。 如果该进程写入文件，则可能导致文件损坏。 此竞争条件从 2004 年起在所有旧版本 Windows 的 SQLite 中存在。 但竞争窗口非常狭窄。 实际上，你需要在一台快速多核机器上同时在两个核心上启动两个进程运行恢复。 此缺陷仅影响 Windows 系统，不影响 posix OS 接口。

8.7. 嵌套事务次级日志的边界值错误

当使用 SAVEPOINT 启动嵌套事务时， SQLite 使用次级回滚日志来追踪嵌套事务的变更，以便在需要回滚内部事务时使用。 次级日志并不用于在程序崩溃或断电时保护数据库。 它们仅在回滚嵌套事务的内部事务时发挥作用。

这些次级日志可以保存在内存中，也可以作为磁盘临时文件。 默认行为是在磁盘上存储。 但可以通过编译时选项 -DSQLITE_TEMP_STORE 或运行时的 PRAGMA temp_store 更改。 该缺陷仅在次级日志保存在内存中时出现。

在 SQLite 版本 3.35.0（2021-03-12）中， 添加了一个新优化：当 SQLite 在内存中保存次级日志时，会使用更少的内存。 不幸的是，新逻辑中的边界检查写错了。 本应是 “<” 的地方写成了 “<=”。 该错误可能导致次级日志在回滚时进入不一致状态。 如果之后有更多改变并且外层事务最终提交，数据库可能会处于不一致状态。

该问题由一位 独立研究人员发现， 他使用模糊测试器搜索 SQLite 缺陷。 模糊测试器触发了一个用于验证次级日志内部状态的 assert() 语句失败。 该缺陷属于极其隐蔽的边界情况，如果不是因为 SQLite 中广泛使用的 assert() 语句、 安全研究人员坚持不懈，以及他们定制的尖端模糊测试器，可能多年都不会被发现。

该问题在 修复提交 在 版本 3.37.2（2022-01-06）中被修复。
