冒充表

1. 简介

冒充表是附属于某个索引所在的相同B 树的表。 冒充表允许按照普通表的方式查询索引的内容。

冒充表仅用于分析和调试。 这并不是大多数应用开发者需要理解甚至知道的功能。

冒充表默认是只读的。只要它们只读，它们就是无害的。 但当PRAGMA writable_schema=ON启用时， 冒充表可以被写入。这可能导致索引损坏，不过 通过执行REINDEX可以修复由此造成的任意损坏。

2. 详细信息

SQLite 中的每个表和每个索引都保存在数据库文件中的独立 B 树中。 每棵 B 树都由它的根页号标识。可以通过查询sqlite_schema 表的 “rootpage” 列 来找到任何索引或表的根页号。 有关该设计的更多背景，请参阅索引教程和文件格式文档。

通常表和索引的 B 树略有不同。 表 B 树包含一个 64 位整数键和任意数据。 这个 64 位整数键是ROWID。 索引 B 树包含任意的二进制键且没有数据。 因此表 B 树和索引 B 树并不直接兼容。

然而，WITHOUT ROWID 表的 B 树与索引 B 树格式相同。 因此，可以像访问 WITHOUT ROWID 表一样访问索引 B 树。

2.1. 手动创建的冒充表

创建冒充表的一种方法是直接编辑 sqlite_schema 表以插入描述该表的新行。 例如，假设架构如下： 

CREATE TABLE t1(a INTEGER PRIMARY KEY,b TEXT,c INT, d INT);
CREATE INDEX t1bc ON t1(b,c);

与 t1bc 索引结构相同的 WITHOUT ROWID 表如下所示： 

CREATE TABLE t2(b TEXT,c INT,a INT, PRIMARY KEY(b,c,a)) WITHOUT ROWID;

要为索引 “t1bc” 创建永久冒充表 “t2”，首先应通过运行 "PRAGMA writable_schema=ON"来启用对 sqlite_schema 表的编辑。 （务必遵循该 PRAGMA 所附带的警告。一个错误可能导致严重的数据库损坏。） 然后像下面这样向 sqlite_schema 表插入新条目： 

INSERT INTO sqlite_schema(type,name,tbl_name,rootpage,sql)
 SELECT 'table','t2','t2',rootpage,
   'CREATE TABLE t2(b,c,a,PRIMARY KEY(b,c,a))WITHOUT ROWID'
   FROM sqlite_schema
  WHERE name='t1bc';

上述 INSERT 语句向 sqlite_schema 表添加新行，定义名为 “t2” 的表， 它与索引 “t1bc” 具有相同的磁盘格式，并指向相同的 B 树。 添加此 sqlite_schema 表条目后，需要关闭并重新打开数据库以使 SQLite 重新读取架构。 然后可以查询 “t2” 表，以查看 “t1bc” 索引的内容。

2.1.1. 数据库损坏

上述手动创建冒充表的方法的一个严重问题是， 在向 “sqlite_schema” 表中添加新的 “t2” 条目后， 数据库文件在技术上会被标记为损坏。 “t1bc” 索引和 “t2” 表都会指向同一棵 B 树。 虽然这不会立即造成问题，但应避免运行VACUUM。

如果PRAGMA writable_schema 被启用， 那么可以写入 “t2” 表，从而改变索引的内容。 这样会使 “t1bc” 索引与其父表 “t1” 不再同步。 不同步的索引可能导致错误的查询结果。

由于 “t2” 冒充表是一种数据库损坏的形式，手动创建冒充表的方法并不推荐。 事实上，除非是资深专家，任何使用冒充表的行为都应受到劝阻，但手动创建的冒充表尤其不推荐， 因为它们是永久性的。

2.2. 瞬态冒充表

另一种（更安全的）创建冒充表的方法是将冒充表的条目添加到 SQLite 的内部符号表， 而不更新磁盘上的 “sqlite_schema” 表。 这样，冒充表只存在于单个数据库连接中，并在架构重新加载时自动删除。

创建瞬态冒充表需要调用特殊的sqlite3_test_control()。 不同于其他 SQLite API， sqlite3_test_control() 接口可能在不同版本之间存在不兼容的变化， 因此下面描述的机制不能保证在 SQLite 的未来版本中继续有效。 SQLite 开发者并不认为这是问题，因为冒充表不应在应用程序中使用。 冒充表仅用于分析和测试。

创建瞬态冒充表时，首先按如下方式调用 sqlite3_test_control()： 

sqlite3_test_control(SQLITE_TESTCTRL_IMPOSTER, db, "main", 1, tnum);

参数 “db” 是指向数据库连接 的指针。 “main” 参数是要创建冒充表的架构名称。 参数 “1” 用于启用冒充表机制。如果改为 “2”，则创建只读的冒充表。 “tnum” 是要镜像的索引的根页。

在上述 sqlite3_test_control() 调用之后，执行定义冒充表的CREATE TABLE语句。 在启用冒充机制的情况下，该 CREATE TABLE 语句不会创建真正的表， 而只是向 SQLite 的内部符号表中添加一个条目。 注意，CREATE TABLE 语句必须符合索引的格式。 如果冒充表的列数错误、不是WITHOUT ROWID 表或以其他方式与索引 B 树不兼容， 则在使用冒充表时会导致SQLITE_CORRUPT 错误。

执行 CREATE TABLE 语句后，按如下方式禁用冒充机制： 

sqlite3_test_control(SQLITE_TESTCTRL_IMPOSTER, db, "main", 0, 0);

换言之，调用与之前相同的 sqlite3_test_control()，但将最后两个参数设置为零。

经过上述步骤将冒充表加载到 SQLite 的内部架构后，该冒充表就可以像其他表一样使用。 但该冒充表仅对创建它的数据库连接可见。 磁盘上的数据库文件不会被修改。 下次加载架构时，冒充表就会消失。

2.3. .imposter 外壳命令

命令行外壳 包含一个点命令 ".imposter"，可以完成设置瞬态冒充表的所有工作。 无需多次调用 sqlite3_test_control() 并手动构造兼容的 CREATE TABLE 语句， 可以使用如下命令创建瞬态冒充表： 

.imposter t1bc t2

当然，请将示例中的 “t1bc” 和 “t2” 替换成所需的索引和冒充表名称。 “.imposter” 命令会读取 “t1bc” 索引的架构，使用这些信息构造兼容的 CREATE TABLE 语句， 然后自动完成创建瞬态冒充表所需的所有调用。

默认情况下，".imposter" 命令在 SQLite 版本 3.51.0（2025-11-04）或更高版本中可用，不过创建的冒充表是只读的。 在较旧版本的 CLI 中使用--unsafe-testing 命令行选项 可激活 ".imposter" 命令，或者创建可读写的冒充表。

3. 总结与最终警告

冒充表机制是 SQLite 的强大分析和调试工具。 但与所有锋利工具一样，如果误用也可能危险并导致数据库文件损坏。 切勿在应用程序中尝试使用冒充表。冒充表应仅由实验室中的专家使用。
