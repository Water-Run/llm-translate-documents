1. 介绍

SQLite 不使用 Git 版本控制系统。SQLite 使用 Fossil，它是一个专门为支持 SQLite 而设计与编写的版本控制系统。

人们常常想知道为什么 SQLite 不像其他人那样使用 Git 版本控制系统。本文尝试回答这个问题。同时，在 第 3 节 中，本文也为 Git 用户提供提示，说明他们如何轻松访问 SQLite 源代码。

本文并不是 Fossil 与 Git 之间的比较。请参阅 https://fossil-scm.org/fossil/doc/trunk/www/fossil-v-git.wiki 了解两者的比较。其他第三方比较也可以找到——使用搜索引擎查找即可。

本文并不鼓励你将项目从 Git 迁移。你可以使用任何你想要的版本控制系统。如果你对 Git 完全满意，那就继续使用 Git。但是，如果 Git 对你而言不顺手，或者你在思考它是否可以改进，或者是否存在更好的选择，那么也许尝试理解下文所阐述的观点。利用由此获得的洞见，寻找或撰写一种不同且更优秀的版本控制系统，或者仅仅对 Git 自身进行改进。

2. SQLite 不使用 Git 的几个原因

2.1. Git 无法提供良好的情境感知

当我想了解 SQLite 最近发生了什么时，我会访问 时间线，在一个屏幕中就能看到所有分支的最新变化摘要。再经过几次点击，我就能深入到想要的细节。即使在手机上也能做到这一点。

GitHub 和 GitLab 没有类似的东西。我目前找到的最接近的是 网络图，但它加载缓慢（除非已经缓存），提供的细节远不如上面那个，也几乎无法在移动设备上正常使用。GitHub 的 提交 页面虽然提供了更多细节、渲染较快并且能在移动设备上使用，但它一次只显示单个分支，因此我无法轻松判断自己是否已经看到了所有近期的变化。而即便 GitHub/GitLab 提供了更好的界面，它们仍然是第三方服务，并非 Git 的核心部分。因此使用它们会给项目再增加一个依赖。

我听说 Git 用户通常会安装第三方图形工具来查看 Git，很多工具在展示项目近期活动方面做得更好。这很好，但它们仍然是额外的第三方应用，需要单独安装并管理。很多工具针对特定平台。（表现较好的工具之一 GitUp 例如只适用于 Mac。）所有这些工具都要求你先同步本地仓库，再在桌面上打开其图形界面。即便如此，没有多次点击，我仍看不到自己通常想看到的内容。在外出的情况下用手机查看项目状态是不可能的。

2.2. Git 使得查找某个提交的后继（子孙）变得困难

Git 让你可以回溯历史，但无法顺藤摸瓜。给定某个历史提交，你可以看到之前发生了什么，但要找出之后发生了什么就比较难。

相比之下，Fossil 提供了向前推进的有用显示。以下是几个示例：

  • 所有源自最近一次主版本发布的 SQLite 提交

  • 所有源自最近一次主发行并最终合入 trunk 分支的提交

  • 所有既是版本 3.48.0 的子孙又是版本 3.49.0 的祖先的提交

  • 从 2025-02-03T13:20Z 提交的 bug 修复开始向后一直到下一个主版本发布的直接路径。

  • 上一条路径的简略形式。

在 Git 中查找某个提交的子孙并非不可能，只是比较困难。例如，Stack Overflow 页面 展示了在 unix 上查找提交子孙的命令序列： 

git rev-list --all --parents | grep ".\{40\}.*.*" | awk '{print $1}'

但这并不是同一回事。上述命令仅仅列出子孙，却没有显示分支结构，而分支结构对于理解发生了什么至关重要。而且该命令只有在你有本地克隆库时才能使用；使用 GitHub 或 GitLab 等网页界面无法查找某个提交的子孙。

这不仅仅是偶尔要查找某个提交的子孙。Fossil 提供子孙信息意味着该信息渗透在 Fossil 的网页之中。举例而言：每个 Fossil 提交信息页面（示例）都会展示一个小型“上下文”图，显示该提交的直接前驱和后继。这有助于用户保持更好的情境感知，并提供了诸如在序列中向前跳转到下一个提交等实用功能。另一个示例：Fossil 轻松显示某个特定提交周围的上下文（示例），同样有助于提升情境感知和对代码动向的更深入理解。在 Fossil 文档 中有一整页的附加示例（完整页面）。

以上所有内容理论上在 Git 中都有可能实现，只要使用合适的扩展、工具和命令。但实际操作并不容易，因此很少有人去做。结果，开发者对代码进展的认知就减少了。

2.3. Git 的心智模型过于复杂

Git 的复杂性会分散对正在开发的软件的注意力。Git 用户需要记住以下所有内容：

  1. 工作目录
  2. “索引”或暂存区
  3. 本地 head
  4. 远程 head 的本地副本
  5. 实际的远程 head

Git 提供了在这些位置之间移动和比较内容的命令（或命令选项）。

相比之下，Fossil 用户只需考虑他们的工作目录和正在工作的提交。这样可减少 60% 的认知负担。每个开发者的大脑资源有限。Fossil 需要更少的脑力，从而释放出更多智力资源来专注于正在开发的软件。

一位同时使用 Git 和 Fossil 的用户在 HN 上写道：

Fossil 让我放心：我只需一个命令就能把所有东西同步到服务器…… 我从 Git 中从未获得这种安心感。

2.4. Git 不会跟踪历史分支名称

Git 维护着提交序列的完整 DAG。但分支标签是本地信息，不会同步，一旦分支关闭就不会被保留。这使得审查历史分支很乏味。

举个例子，假设客户问你：“两年前的那个 ‘prefer-coroutine-sort-subquery’ 分支后来怎样了？”你可能通过查看版本控制系统的历史来回答，方式如下：

  • GitHub： https://github.com/sqlite/sqlite/commits/prefer-coroutine-sort-subquery
  • Fossil： https://sqlite.org/src/timeline?r=prefer-coroutine-sort-subquery

Fossil 的视图清楚显示该分支最终被合并回 trunk，显示了分支的起点以及 trunk 多次合并入分支的情况。GitHub 则没有这些内容。事实上，在试图弄清楚发生了什么时，GitHub 的展示基本无用。

很多读者推荐各种第三方 Git GUI，它们在展示历史开发活动方面可能做得比原生 Git 或 GitHub 更好。也许某些工具确实优于原生 Git 和/或 GitHub，但它们仍然受到 Git 不在同步中保留历史分支名称这一事实的限制。即便那些工具更优秀，要获取所需信息仍然需要借助第三方工具，这也进一步说明核心系统本身并不出色。

2.5. Git 需要更多的管理支持

Git 是复杂的软件。要在开发者工作站上安装 Git，或升级到更新版本，都需要某种安装程序。搭建一套 Git 服务器并不容易，因此大多数开发者依赖 GitHub 或 GitLab 等第三方服务，从而引入了额外的依赖。

相比之下，Fossil 是一个独立的单个可执行文件，只需将其放到 $PATH 中即可安装。这个单个二进制文件包含了 Git 核心及 GitHub/GitLab 的所有功能。它可以管理包含维基、错误跟踪、聊天和论坛的社区服务器，提供面向用户的打包下载、登录管理等等，无需额外软件。搭建一个 Fossil 社区服务器只需几分钟。而且 Fossil 效率很高。一个 Fossil 服务器可以在每月 5 美元的 VPS 或树莓派上顺利运行，而 GitLab 等则需要更强的硬件。

更少的管理意味着程序员能把更多时间用于处理软件（本例中是 SQLite），而不是为版本控制系统操心。

2.6. Git 提供糟糕的用户体验

以下这幅 https://xkcd.com/1597/ 漫画虽略有夸张，却说中了要害：

我们要实事求是。很少有人会否认 Git 的用户体验不够理想。很多底层实现细节都暴露在用户界面中。界面糟糕到了让人开设一个 假 Git 手册页生成器 的地步。

设计软件很难，需要大量专注力。一个优秀的版本控制系统应该为开发者提供帮助，而不是挫败感。过去十年里 Git 在这方面有所进步，但仍然任重道远。

3. Git 用户访问 SQLite 源代码指南

如果你是忠实的 Git 用户，仍然可以轻松访问 SQLite。本节提供了一些提示，指导你如何做到这一点。

3.1. 官方 GitHub 镜像

从 2019-03-20 起，SQLite 源代码在 GitHub 上已有 官方 Git 镜像。

这个镜像是对 SQLite 官方 Fossil 仓库 的增量导出。一个 cron 作业每小时更新一次 GitHub 仓库。这是一个单向的只读代码镜像。GitHub 不接受拉取请求或直接改动。GitHub 仓库仅仅复制 Fossil 仓库中的内容，所有变更仍然由 Fossil 输入。

在 Git 镜像中识别提交和文件的哈希与 Fossil 中不同。原因有很多，其中主要是 Fossil 使用 SHA3-256 哈希而 Git 使用 SHA1。在导出过程中，每个提交原始的 Fossil 哈希会作为脚注添加到提交注释中。为了避免混淆，在引用 SQLite 提交时，请始终使用原始 Fossil 哈希，而不是 Git 哈希。

3.2. Web 访问

SQLite Fossil 仓库 提供了任何历史版本的 Tarball、ZIP 归档或 SQLite Archive 的下载链接。这些下载的 URL 简洁明了，可以很容易地嵌入自动化工具。格式如下：

https://sqlite.org/src/tarball/VERSION/sqlite.tar.gz

只需将 VERSION 替换为要下载版本的描述。VERSION 可以是特定提交的哈希前缀，也可以是某个分支的名称（此时会获取该分支的最新版本），或者是指定提交的标签名称，例如 “version-3.23.1”：

https://sqlite.org/src/tarball/version-3.23.1/sqlite.tar.gz

要获取最新发布版，请将 VERSION 设置为 “release”，如下所示：

https://sqlite.org/src/tarball/release/sqlite.tar.gz

要获取最新的 trunk 提交，请将 VERSION 设置为 “trunk”：

https://sqlite.org/src/tarball/trunk/sqlite.tar.gz

依此类推。对于 ZIP 归档或 SQLite Archive，只需将 “/tarball/” 替换为 “/zip/” 或 “/sqlar/”，并根据需要将下载文件名的后缀改为 “.zip” 或 “.sqlar”。

3.3. Fossil 访问

Fossil 安装与使用都很容易。以下是在 Unix 上的步骤。（Windows 也类似。）

  1. 从 https://fossil-scm.org/fossil/uv/download.html 下载自包含的 Fossil 可执行文件，并将该可执行文件放到 $PATH 中的某个位置。
  2. mkdir ~/fossils
  3. fossil clone https://sqlite.org/src ~/fossils/sqlite.fossil
  4. mkdir ~/sqlite; cd ~/sqlite
  5. fossil open ~/fossils/sqlite.fossil

此时你就可以输入 “./configure; make” 了（或者在 Windows + MSVC 环境下输入 “nmake /f Makefile.msc”）。

要将检出切换到另一个 Fossil 版本，请使用 “update” 命令：

fossil update VERSION

要获取最新 trunk 版本，请将 VERSION 设置为 “trunk”。也可以使用某个哈希前缀，或某个分支或标签名。有关可以用于 VERSION 的名称，请参阅 https://fossil-scm.org/fossil/doc/trunk/www/checkin_names.wiki。

在 ~/sqlite 检出中使用 “fossil ui” 命令，可以启动本地版本的网站。

有关 Fossil 的更多文档，可见 https://fossil-scm.org/fossil/doc/trunk/www/permutedindex.html

不要害怕探索与尝试。没有登录的话你就无法推送任何修改，因此也不可能破坏项目。

3.4. 验证源代码完整性

如果你需要验证所持有的 SQLite 源代码是正宗且未被任何方式（可能是对手）篡改，可以借助几个简单的命令行工具来实现。在 SQLite 源代码树的根目录有一个名为 “manifest” 的文件。manifest 文件记录了源树中每个其他文件的名称及其对应的 SHA1 或 SHA3-256 哈希。（旧文件使用 SHA1，新文件使用 SHA3-256。）你可以编写脚本来提取这些哈希值，并与源代码文件进行核对。提交的哈希名称就是 “manifest” 文件本身的 SHA3-256 哈希，若最后一行以 “# Remove this line...” 开头，则可以省略最后一行。
