SQLite 的虚拟数据库引擎

过时文档警告： 本文档描述的是 SQLite 版本 2.8.0 中使用的虚拟机。SQLite 3.0 和 3.1 中的虚拟机在概念上类似，但现在是基于寄存器而非基于栈，且每条操作码有五个操作数而不是三个，并且操作码集合与下文所示的不同。有关当前 VDBE 操作码集合以及 VDBE 如何运行的简要概览，请参见虚拟机指令文档。本文件保留作为历史参考。

如果您想了解 SQLite 库的内部工作方式，就需要从对虚拟数据库引擎（VDBE）有一个扎实的理解开始。VDBE 位于处理流的正中央（参见架构图），因此它似乎涉及库的大部分区域。即使是不直接与 VDBE 交互的代码部分，通常也是在扮演辅助角色。VDBE 真正是 SQLite 的核心。

本文简要介绍了 VDBE 的工作原理，特别是各种 VDBE 指令（可在此处查阅）如何协同工作以对数据库执行有用的操作。风格上采用教程形式，从简单任务入手，逐步解决更复杂的问题。过程中我们将涉及 SQLite 库中的大多数子模块。完成本教程后，您应该对 SQLite 的工作原理有一个相当不错的理解，并可以开始研究实际的源代码。

前言

VDBE 实现了一个虚拟计算机，它在其虚拟机器语言中运行程序。每个程序的目标是查询或修改数据库。为此，VDBE 实现的机器语言是专门为搜索、读取和修改数据库而设计的。

VDBE 语言的每条指令都包含一个操作码和三个标记为 P1、P2 和 P3 的操作数。操作数 P1 是任意整数。P2 是一个非负整数。P3 是指向数据结构或以零结尾的字符串的指针，也可能为空。只有少数 VDBE 指令使用所有三个操作数。许多指令只使用一个或两个操作数。相当数量的指令根本不使用操作数，而是从执行栈中获取数据并将结果存储在那里。每条指令的具体行为及其使用的操作数在单独的操作码描述文档中有所说明。

VDBE 程序从指令 0 开始执行，并按顺序连续执行，直到它遇到 (1) 致命错误、(2) 执行 Halt 指令、或 (3) 将程序计数器推进到程序的最后一条指令之后。VDBE 执行完毕后，所有打开的数据库游标都会关闭，所有内存都会释放，栈上的所有内容都会被弹出。因此永远不用担心内存泄漏或资源未释放。

如果您曾经进行过任何汇编语言编程或使用过任何抽象机器，那么所有这些细节对您来说应该很熟悉。那么让我们直奔主题，开始查看一些代码吧。

向数据库插入记录

我们先从一个可以通过仅由几条指令组成的 VDBE 程序解决的问题开始。假设我们有一个 SQL 表按照以下方式创建：

CREATE TABLE examp(one text, two int);

换句话说，我们有一个名为 “examp” 的数据库表，该表有两个名为 “one” 和 “two” 的数据列。现在假设我们想向该表插入一条记录，如下所示：

INSERT INTO examp VALUES('Hello, World!',99);

我们可以使用 sqlite 命令行实用程序查看 SQLite 用于实现此 INSERT 的 VDBE 程序。首先在一个新的空数据库上启动 sqlite，然后创建该表。接下来，将 sqlite 的输出格式更改为专为 VDBE 程序转储设计的形式，方法是输入 “.explain” 命令。最后，输入上面显示的 [INSERT] 语句，并在 [INSERT] 前加上特殊关键字 [EXPLAIN]。[EXPLAIN] 关键字将使 sqlite 打印 VDBE 程序而不是执行它。我们得到：

$ sqlite test_database_1
sqlite> CREATE TABLE examp(one text, two int);
sqlite> .explain
sqlite> EXPLAIN INSERT INTO examp VALUES('Hello, World!',99);
addr  opcode        p1     p2     p3                                      
----  ------------  -----  -----  -----------------------------------
0     Transaction   0      0                                         
1     VerifyCookie  0      81                                        
2     Transaction   1      0                                         
3     Integer       0      0                                         
4     OpenWrite     0      3      examp                                    
5     NewRecno      0      0                                 
6     String        0      0      Hello, World!                      
7     Integer       99     0      99                                 
8     MakeRecord    2      0                                     
9     PutIntKey     0      1                                     
10    Close         0      0                                       
11    Commit        0      0                                      
12    Halt          0      0

如上所示，我们的简单插入语句是由 12 条指令实现的。前 3 条和后 2 条是标准的前奏和尾声，因此真正的工作是在中间的 7 条指令中完成的。程序没有跳转，因此从上到下只执行一遍。现在我们逐条详细查看每个指令。

0     Transaction   0      0                                         
1     VerifyCookie  0      81                                        
2     Transaction   1      0

Transaction 指令启动一个事务。当遇到 Commit 或 Rollback 操作码时事务结束。P1 是事务所启动的数据库文件的索引。索引 0 是主数据库文件。启动事务时会对数据库文件加写锁。在事务进行期间，其他进程不能读取或写入该文件。启动事务还会创建回滚日志。必须先启动事务才能对数据库进行任何更改。

VerifyCookie 指令检查 cookie 0（即数据库模式版本），以确保它等于 P2（数据库模式上次读取时获得的值）。P1 是数据库编号（主数据库为 0）。这样做是为了确保数据库模式没有被其他线程更改，否则必须重新读取。

第二个 Transaction 指令开始一个事务并为数据库 1（用于临时表的数据库）启动回滚日志。

3     Integer       0      0                                    
4     OpenWrite     0      3      examp

Integer 指令将整数值 P1（0）压入栈中。此处 0 是随后 OpenWrite 指令中要使用的数据库编号。如果 P3 非 NULL，则它是相同整数的字符串表示。之后栈如下所示：

(integer) 0

OpenWrite 指令在表 “examp” 上以句柄 P1（此处为 0）打开一个新的读写游标，其根页为 P2（在当前数据库文件中为 3）。游标句柄可以是任意非负整数。但 VDBE 在一个数组中分配游标，数组大小等于最大游标值加一。因此为了节省内存，最好从 0 开始依次使用句柄。此处 P3（“examp”）是要打开的表名，但并未使用，仅用于使代码更易读。此指令会从栈顶弹出尚未使用的数据库编号（0，即主数据库），因此之后栈再次为空。

5     NewRecno      0      0

NewRecno 指令为由游标 P1 指向的表创建一个新的整数记录号。该记录号是表中当前未被用作键的数字。新记录号被压入栈中。之后栈如下所示：

(integer) new record key
6     String        0      0      Hello, World!

String 指令将其 P3 操作数压入栈。之后栈如下所示：

(string) "Hello, World!"
(integer) new record key
7     Integer       99     0      99

Integer 指令将其 P1 操作数（99）压入栈。之后栈如下所示：

(integer) 99
(string) "Hello, World!"
(integer) new record key
8     MakeRecord    2      0

MakeRecord 指令弹出栈顶的 P1 个元素（此处为 2 个），并将它们转换为用于在数据库文件中存储记录的二进制格式。（详见文件格式说明。）MakeRecord 指令生成的新记录被重新压入栈。之后栈如下所示：

(record) "Hello, World!", 99
(integer) new record key
9     PutIntKey     0      1

PutIntKey 指令使用栈顶的两个条目向由游标 P1 指向的表写入一条记录。如果该记录尚不存在则创建新条目，否则覆盖现有条目的数据。记录数据是栈顶条目，键是下一个条目。此指令会连续弹出两个条目。因为操作数 P2 为 1，所以会增加行更改计数，并将 rowid 存储以供 sqlite_last_insert_rowid() 函数随后返回。如果 P2 为 0，则行更改计数保持不变。该指令就是实际执行插入的地方。

10    Close         0      0

Close 指令关闭之前以 P1（0，即唯一打开的游标）打开的游标。如果 P1 当前未打开，则该指令是空操作。

11    Commit        0      0

Commit 指令使自上次 Transaction 以来所做的所有修改真正生效。在另一个事务开始之前不能再进行额外修改。Commit 指令会删除日志文件并释放数据库上的写锁。如果仍有游标打开，则会继续保持读锁。

12    Halt          0      0

Halt 指令导致 VDBE 引擎立即退出。所有打开的游标、列表、排序等都会自动关闭。P1 是 sqlite_exec() 返回的结果代码。对于正常的 halt，应为 SQLITE_OK (0)。对于错误，它可能是其他值。操作数 P2 仅在出现错误时使用。每个程序末尾都有一个隐含的 “Halt 0 0 0” 指令，这是 VDBE 在准备运行程序时附加的。

跟踪 VDBE 程序执行

如果 SQLite 库在编译时未定义 NDEBUG 预处理宏，则 PRAGMA vdbe_trace 会使 VDBE 跟踪程序的执行。尽管该功能最初用于测试和调试，但在了解 VDBE 操作方式时也很有用。使用 “PRAGMA vdbe_trace=ON;” 打开跟踪，使用 “PRAGMA vdbe_trace=OFF” 关闭跟踪。示例如下：

sqlite> PRAGMA vdbe_trace=ON;

sqlite> INSERT INTO examp VALUES('Hello, World!',99);

   1 VerifyCookie    0   81
   2 Transaction     1    0
   3 Integer         0    0
Stack: i:0
   4 OpenWrite       0    3 examp
   5 NewRecno        0    0
Stack: i:2
   6 String          0    0 Hello, World!
Stack: t[Hello,.World!] i:2
   7 Integer        99    0 99
Stack: si:99 t[Hello,.World!] i:2
   8 MakeRecord      2    0
Stack: s[...Hello,.World!.99] i:2
   9 PutIntKey       0    1
  10 Close           0    0
  11 Commit          0    0

打开跟踪模式后，VDBE 会在执行指令前打印每条指令。指令执行完成后，会显示栈顶的几项内容。如果栈为空，则不显示栈。

在栈显示中，大多数条目都带有前缀，用以指示该栈条目的数据类型。整数以 “i:” 开头。浮点值以 “r:” 开头。（“r” 代表 “real-number”。）字符串以 “s:”、“t:”、“e:” 或 “z:” 开头。字符串前缀之间的差异取决于它们的内存分配方式。z: 字符串存储在从 malloc() 获取的内存中。t: 字符串是静态分配的。e: 字符串是短暂的。其他所有字符串均带有 s: 前缀。对您观察者而言这没有任何区别，但对 VDBE 来说却至关重要，因为 z: 字符串在弹出时需要传递给 free() 以避免内存泄漏。请注意，字符串值只显示前 10 个字符，且二进制值（例如 MakeRecord 指令的结果）也作为字符串处理。VDBE 栈中唯一可以存储的其他数据类型是 NULL，它显示为单独的 “NULL”，没有前缀。如果一个整数既以整数形式又以字符串形式被压入栈，其前缀为 “si:”。

简单查询

至此，您应该已经了解了 VDBE 如何向数据库写入数据的基础。现在让我们看看它如何执行查询。我们将以下简单 SELECT 语句作为示例：

SELECT * FROM examp;

此 SQL 语句生成的 VDBE 程序如下所示：

sqlite> EXPLAIN SELECT * FROM examp;
addr  opcode        p1     p2     p3                                 
----  ------------  -----  -----  -----------------------------------
0     ColumnName    0      0      one                                
1     ColumnName    1      0      two                                
2     Integer       0      0                                         
3     OpenRead      0      3      examp                              
4     VerifyCookie  0      81                                        
5     Rewind        0      10                                        
6     Column        0      0                                         
7     Column        0      1                                         
8     Callback      2      0                                         
9     Next          0      6                                         
10    Close         0      0                                         
11    Halt          0      0

在我们开始分析这个问题之前，让我们简要回顾一下 SQLite 中的查询是如何工作的，以便我们知道要完成什么任务。对于查询结果中的每一行，SQLite 将调用一个原型如下的回调函数：

int Callback(void *pUserData, int nColumn, char *azData[], char *azColumnName[]);

SQLite 库向 VDBE 提供了指向回调函数的指针以及 pUserData 指针。（回调和用户数据最初作为参数传递给 sqlite_exec() API 函数。）VDBE 的任务是为 nColumn、azData[] 和 azColumnName[] 生成值。nColumn 当然是结果中的列数。azColumnName[] 是一个字符串数组，其中每个字符串是结果列之一的名称。azData[] 是一个字符串数组，保存实际数据。

0     ColumnName    0      0      one                                
1     ColumnName    1      0      two

查询的 VDBE 程序中的前两条指令负责设置 azColumn 的值。ColumnName 指令告诉 VDBE 应在 azColumnName[] 数组的每个元素中填入何种值。每个查询都以一条 ColumnName 指令开始，针对结果中的每一列各有一条，并且稍后查询中会有与之匹配的 Column 指令。

2     Integer       0      0                                         
3     OpenRead      0      3      examp                              
4     VerifyCookie  0      81

指令 2 和 3 在要查询的数据库表上打开了一个读游标。这与 INSERT 示例中的 OpenWrite 指令的工作方式相同，只不过此处游标是以读取方式打开的。指令 4 与 INSERT 示例一样用于验证数据库模式。

5     Rewind        0      10

Rewind 指令初始化一个循环，用于遍历 “examp” 表。它将游标 P1 倒回到表中的第一个条目。这对于 Column 和 Next 指令是必要的，它们使用该游标迭代表。若表为空，则跳转到 P2（10），即循环之后的指令。如果表非空，则继续执行下一条指令 6，即循环体的起始处。

6     Column        0      0                                         
7     Column        0      1                                         
8     Callback      2      0

指令 6 至 8 构成循环体，每条指令针对数据库文件中的每条记录执行一次。

Column 指令在地址 6 与 7 处分别从游标 P1 中取出 P2 所指定的列并将其压入栈。在本示例中，第一条 Column 指令将列 “one” 的值压入栈，第二条 Column 指令将列 “two” 的值压入栈。

Callback 指令在地址 8 调用回调函数。Callback 的 P1 操作数成为 nColumn 的值。Callback 指令从栈中弹出 P1 个值，并用它们填充 azData[] 数组。

9     Next          0      6

地址 9 处的指令实现了循环的分支逻辑。它与地址 5 处的 Rewind 一起构成了循环结构。这是一个关键概念，值得您密切关注。Next 指令将游标 P1 推进到下一条记录。如果游标成功前进，则立即跳转到 P2（6，即循环体的起始处）。如果游标已经到达末尾，则继续执行下面的指令，从而结束循环。

10    Close      &Thinking... (1s elapsed)Thinking... (2s elapsed)Thinking... (3s elapsed)Thinking... (4s elapsed)Thinking... (5s elapsed)Thinking... (6s elapsed)

一个稍微复杂一些的查询

前一个示例的关键点是使用 Callback instruction 调用回调函数，以及使用 Next instruction 实现对数据库文件所有记录的循环。这个示例通过演示一个稍微复杂一些的查询来强化这些想法，该查询涉及更多输出列，其中有些是计算值，并且有一个 WHERE 子句限制哪些记录会真正传递给回调函数。考虑下面这个查询：

SELECT one, two, one || two AS 'both'
FROM examp
WHERE one LIKE 'H%'

这个查询也许有些牵强，但它确实有助于说明我们的重点。结果会有三个列，名称分别为 “one”、 “two” 和 “both”。前两个列是表中对应列的直接复制，第三个结果列则是通过连接表中的第一列和第二列生成的字符串。最后，WHERE 子句说明我们只会为“one”列以 “H” 开头的行选择结果。以下是这个查询对应的 VDBE 程序：

addr  opcode        p1     p2     p3                                      
----  ------------  -----  -----  -----------------------------------
0     ColumnName    0      0      one
1     ColumnName    1      0      two
2     ColumnName    2      0      both
3     Integer       0      0
4     OpenRead      0      3      examp
5     VerifyCookie  0      81
6     Rewind        0      18
7     String        0      0      H%                                      
8     Column        0      0
9     Function      2      0      ptr(0x7f1ac0)
10    IfNot         1      17
11    Column        0      0
12    Column        0      1
13    Column        0      0
14    Column        0      1
15    Concat        2      0
16    Callback      3      0
17    Next          0      7
18    Close         0      0
19    Halt          0      0

除了 WHERE 子句外，这个示例的程序结构与前面的示例非常相似，只是多了一个输出列。现在有 3 列，而不是之前的 2 列，也有三条 ColumnName 指令。就像前面的示例一样，用 OpenRead 指令打开了一个游标。地址 6 处的 Rewind 和地址 17 处的 Next 构成了对表中所有记录的循环。结尾处的 Close 指令是为了在 Rewind 完成时有一个跳转目标。所有这些都与第一个查询示例完全一样。

这个示例中的 Callback instruction 需要为三个结果列生成数据，而不是两个，但其他方面与第一个查询一样。当 Callback instruction 被调用时，结果的最左侧列应该在栈的底部，最右侧的结果列则应在栈顶。我们可以在地址 11 到 15 看到栈是这样设置的。地址 11 和 12 的 Column 指令压入结果中前两列的值。地址 13 和 14 的两条 Column 指令则拉取用于计算第三个结果列所需的值，地址 15 的 Concat 指令将它们组合成栈上的一个条目。

当前示例中唯一真正新颖的部分是由地址 7 到 10 的指令实现的 WHERE 子句。地址 7 和 8 的指令将表中 “one” 列的值和字面字符串 “H%” 压入栈中。地址 9 的 Function 指令从栈中弹出这两个值，并将 LIKE() 函数的结果推回栈。地址 10 的 IfNot 指令弹出栈顶值，如果该值为 false（即不匹配字面字符串 “H%”），则立即跳转到 Next 指令。跳过回调的目的正是 WHERE 子句的作用。如果比较结果为 true，就不会跳转，控制流会继续执行下方的 Callback 指令。

注意 LIKE 操作符如何实现。它是 SQLite 中的一个用户定义函数，所以函数定义的地址在 P3 中指定。操作数 P1 表示函数需要从栈上取多少个参数。在本例中，LIKE() 函数需要 2 个参数。参数是按逆序（从右到左）从栈上取出的，因此要匹配的模式位于栈顶，下一个元素是要比较的数据。返回值被压回栈。

SELECT 程序的一个模板

前两个查询示例展示了每个 SELECT 程序都会遵循的一种模板。基本流程如下：

  1. 为回调初始化 azColumnName[] 数组。
  2. 打开要查询的表的游标。
  3. 对表中的每一条记录执行：
    1. 如果 WHERE 子句的结果为 FALSE，则跳过下面的步骤，继续处理下一条记录。
    2. 计算当前行结果的所有列。
    3. 为当前行结果调用回调函数。
  4. 关闭游标。

当我们考虑连接、复合 SELECT、使用索引加速搜索、排序以及带或不带 GROUP BY 和 HAVING 子句的聚合函数等更多复杂情况时，这个模板会大幅扩展。但相同的基本思想仍然适用。

UPDATE 和 DELETE 语句

UPDATE 和 DELETE 语句的代码模板与 SELECT 语句非常相似。主要区别当然在于最终操作是修改数据库，而不是调用回调函数。由于它修改了数据库，所以还会使用事务。我们先来看一个 DELETE 语句：

DELETE FROM examp WHERE two<50;

该 DELETE 语句会删除 “examp” 表中 “two” 列小于 50 的所有记录。为此生成的代码如下：

这个循环完成实际删除工作。它与 UPDATE 示例中的循环组织方式不同。ListRead 在这个循环中承担了 INSERT 循环中 Next 的角色，但因为 ListRead 在失败时跳转至 P2，而 Next 在成功时跳转，我们将它放在循环的开头而不是末尾。这意味着我们必须在循环末尾添加 Goto，跳回循环开始的测试位置。因此这个循环的形式类似 C 的 while(){...}，而 INSERT 示例的循环则类似 do{...}while()。Delete 指令在这个循环中执行的工作相当于前面示例中的回调函数。

ListRead 指令从临时存储列表中读取一个元素并将其推入栈顶。如果成功，就继续执行下一条指令；如果因为列表为空而失败，就跳转到 P2，即紧挨着循环之后的指令。执行后栈变为：

(integer) key for current record

注意 ListRead 与 Next 指令之间的相似性。它们都遵循如下规则：

Push the next "thing" onto the stack and fall through OR jump to P2, depending on whether or not there is a next "thing" to push.

Next 和 ListRead 的区别之一在于它们所指的 “thing” 是什么。Next 的 “thing” 是数据库文件中的记录；ListRead 的 “thing” 则是列表中的整数键。另一个区别在于当没有下一个 “thing” 时是选择跳转还是继续执行——在这个例子中，Next 继续执行，而 ListRead 则跳转。稍后我们还会看到其他遵循相同原则的循环指令（如 NextIdx 和 SortNext）。

NotExists 指令弹出栈顶元素，并将其作为整数键。如果该键对应的记录在表 P1 中不存在，则跳转到 P2；如果存在，则继续执行下一条指令。在本例中，P2 将程序导向循环末尾的 Goto，从而跳回循环开头的 ListRead。这本可以把 P2 设置为 16（循环开始处的 ListRead），但生成此代码的 SQLite 解析器并未进行此类优化。

Delete 指令完成循环的核心工作：它弹出栈顶的整数键（由前面的 ListRead 放入），并删除游标 P1 中具有该键的记录。由于 P2 为 true，行更改计数器也会相应递增。

Goto 指令跳回循环的起点。至此循环结束。

22    Commit        0      0

UPDATE 语句的工作方式与 DELETE 非常相似，不同之处在于它用新记录替换旧记录而不是删除它。考虑如下示例：

CREATE and DROP

从 VDBE 的角度来看，使用 CREATE 或 DROP 来创建或销毁表或索引，与对特殊的 "sqlite_master" 表执行 INSERT 或 DELETE 是一样的。sqlite_master 表是每个 SQLite 数据库自动创建的特殊表。它看起来像这样：

CREATE TABLE sqlite_master (
  type      TEXT,    -- either "table" or "index"
  name      TEXT,    -- name of this table or index
  tbl_name  TEXT,    -- for indices: name of associated table
  sql       TEXT     -- SQL text of the original CREATE statement
)

每个表（除了 sqlite_master 本身）以及每个命名的索引在 SQLite 数据库中都会在 sqlite_master 表中有一个条目。你可以像查询其他表一样使用 SELECT 语句查询这个表。但你不能直接通过 UPDATE、INSERT 或 DELETE 修改它。必须通过 CREATE 和 DROP 命令来更改 sqlite_master，因为在添加或删除表和索引时，SQLite 还需要更新一些内部数据结构。

但从 VDBE 的角度来看，CREATE 的工作方式很像 INSERT，DROP 则像 DELETE。当 SQLite 库打开一个已有的数据库时，首先执行 SELECT，从 sqlite_master 表的所有条目中读取 "sql" 列。 "sql" 列包含最初生成该索引或表的 CREATE 语句的完整 SQL 文本。这段文本会被重新送入 SQLite 解析器，用来重建描述该索引或表的内部数据结构。

Using Indexes To Speed Searching

在上面的示例查询中，被查询表的每一行都必须从磁盘加载并检查，即使结果中只有很小一部分行。这在大表上可能非常耗时。为了加快速度，SQLite 可以使用索引。

SQLite 文件将 key 与一些数据关联起来。对于 SQLite 表，数据库文件按照 key 为整数，而数据为表中一行的信息来设置。SQLite 中的索引则相反：索引 key 是所存储的部分信息，而索引数据是一个整数。要访问包含特定内容的表行，我们首先在索引表中查找该内容以获得它的整数索引，然后使用该整数到表中查找完整记录。

请注意，SQLite 使用 b-tree（一种有序数据结构），所以当 SELECT 语句的 WHERE 子句包含等值或不等值测试时可以使用索引。下面这样的查询如果有合适的索引就能使用它：

SELECT * FROM examp WHERE two==50;
SELECT * FROM examp WHERE two<50;
SELECT * FROM examp WHERE two IN (50, 100);

如果存在一个将 examp 表的 "two" 列映射到整数的索引，那么 SQLite 就会使用该索引来查找 examp 中所有 column two 值为 50 的行的整数键，或所有小于 50 的行等。但下面这些查询就不能使用索引：

SELECT * FROM examp WHERE two%50 == 10;
SELECT * FROM examp WHERE two&127 == 3;

注意，即便解析器可以使用索引，它也不总是生成对应的代码。以下查询目前不会使用索引：

SELECT * FROM examp WHERE two+10 == 50;
SELECT * FROM examp WHERE two==50 OR two==100;

连接

在一个 join 中，两个或更多表被组合以生成单个结果。结果表由被 join 的各表中的每一个可能的行组合组成。实现这一点最简单、最自然的方法是使用嵌套循环。

回想上面讨论的查询模板，其中有一个循环遍历表的每一条记录。在 join 中我们基本上做的是同样的事情，只不过有嵌套循环。例如，为了 join 两个表，查询模板可能看起来像这样：

  1. 为回调初始化 azColumnName[] 数组。
  2. 打开两个游标，分别对应被查询的两个表。
  3. 对于第一个表的每条记录，执行：
    1. 对于第二个表的每条记录，执行：
      1. 如果 WHERE clause 的求值结果为 FALSE，那么跳过后续步骤并继续到下一条记录。
      2. 计算当前结果行的所有列。
      3. 为当前结果行调用回调函数。
  4. 关闭两个游标。

这个模板可以工作，但可能会很慢，因为我们现在处理的是 O(N2) 循环。但通常可以将 WHERE clause 分解成多个项，而其中一个或多个项只涉及第一个表的列。发生这种情况时，我们可以将 WHERE clause 条件的一部分从内层循环中提取出来，从而获得大量效率提升。因此，更好的模板可能类似如下：

  1. 为回调初始化 azColumnName[] 数组。
  2. 打开两个游标，分别对应被查询的两个表。
  3. 对于第一个表的每条记录，执行：
    1. 求值只涉及第一个表列的 WHERE clause 项。如果任意一项为 false（意味着整个 WHERE clause 必须为 false），则跳过本次循环的其余部分并继续到下一条记录。
    2. 对于第二个表的每条记录，执行：
      1. 如果 WHERE clause 的求值结果为 FALSE，那么跳过后续步骤并继续到下一条记录。
      2. 计算当前结果行的所有列。
      3. 为当前结果行调用回调函数。
  4. 关闭两个游标。

如果可以使用索引加快任一循环的搜索，还可以进一步提高速度。

SQLite 总是按照 SELECT 语句中 FROM 子句中表出现的顺序构造循环。最左边的表成为外层循环，而最右边的表成为内层循环。理论上，在某些情况下有可能重新排序循环以加快 join 的求值，但 SQLite 并不尝试这种优化。

The very name "Structured Query Language" tells us that SQL should support nested queries. And, in fact, two different kinds of nesting are supported. Any SELECT statement that returns a single-row, single-column result can be used as a term in an expression of another SELECT statement. And, a SELECT statement that returns a single-column, multi-row result can be used as the right-hand operand of the IN and NOT IN operators. 我们将以一个第一类嵌套的例子开始本节，在该例中单行单列的 SELECT 作为另一个 SELECT 表达式中的项。 以下是我们的示例：

外部 SELECT 由指令 11 至 25 实现。 特别地，包含 IN 操作的 WHERE 子句由指令 16、17 和 20 实现。 指令 16 将当前行 "two" 列的值推入栈，指令 17 检查其是否非 NULL。 如果成功，执行跳转到 20，在那里测试栈顶是否与临时表中的任意键匹配。 剩下的代码与前面示例所示相同。

Compound SELECT Statements

SQLite 还允许使用 UNION、UNION ALL、INTERSECT 和 EXCEPT 操作符将两个或多个 SELECT 语句作为同级连接。 这些复合 select 语句通过瞬态表实现。 每个操作的实现略有不同，但基本思想相同。 我们以 EXCEPT 操作为例。

SELECT two FROM examp
EXCEPT
SELECT four FROM examp2;

此最后示例的结果应该是 examp 表中 "two" 列的每个唯一值，但会排除任何在 examp2 的 "four" 列中出现的值。 实现该查询的代码如下：

构建结果的瞬态表由第 0 条指令创建。接下来依次出现三个循环。 指令 5 到 10 的循环实现第一个 SELECT 语句。 第二个 SELECT 语句由指令 14 到 19 的循环实现。 最后，指令 22 到 25 的循环读取瞬态表，并在结果的每一行调用一次回调。

指令 1 在本例中特别重要。通常 Column 指令是从 SQLite 文件记录的数据中提取某列的值。 而指令 1 在瞬态表上设置一个标记，使得 Column 改为将 SQLite 文件记录的键当作数据，并从键中提取列信息。

以下将要发生：第一个 SELECT 语句生成结果行，并将每一行作为瞬态表中条目的键保存。 瞬态表中每个条目的数据从未使用，因此填入空字符串。 第二个 SELECT 也构造行，但这些行会从瞬态表中移除。 这正是我们希望将行存储在 SQLite 文件键中的原因，而不是数据部分——这样就可以轻松定位并删除它们。

让我们更仔细地看看这里发生了什么。 第一个 SELECT 由指令 5 到 10 的循环实现。 指令 5 通过倒带游标初始化循环。 指令 6 从 "examp" 中提取 "two" 列的值，指令 7 将其转换为行。 指令 8 将一个空字符串压入堆栈。 最后，指令 9 将行写入临时表。 但请记住，PutStrKey opcode 使用栈顶作为记录数据，栈上一个作为键。 对 INSERT 语句而言， MakeRecord opcode 生成的行作为记录数据，NewRecno opcode 创建的整数作为记录键。 但在此处角色颠倒，MakeRecord 生成的行作为记录键，而记录数据则是空字符串。

第二个 SELECT 由指令 14 到 19 实现。 指令 14 通过倒带游标初始化循环。 指令 15 和 16 从表 "examp2" 的 "four" 列创建一个新的结果行。 但我们没有使用 PutStrKey 将此新行写入临时表， 而是调用 Delete，如果存在则将其从临时表中删除。

复合 select 的结果由指令 22 到 25 的循环传递给回调例程。 这个循环并没有什么新奇之处，唯一值得注意的是 指令 23 的 Column 将从记录键中提取列而不是从记录数据中提取。

总结

本文回顾了SQLite的VDBE用于实现SQL语句的所有主要技术。尚未展示的是，这些技术中的大多数可以结合使用，以生成适用于复杂查询语句的代码。例如，我们已经展示了如何在简单查询上完成排序，也展示了如何实现复合查询。但我们并未给出在复合查询中排序的例子。这是因为对复合查询进行排序并不会引入任何新概念：它只是将之前的两个想法（排序与复合）在同一个VDBE程序中组合起来。
