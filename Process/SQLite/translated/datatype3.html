1. SQLite 中的数据类型

大多数 SQL 数据库引擎（据我们所知除 SQLite 以外的所有 SQL 数据库引擎）都使用静态、刚性类型。使用静态类型时，值的数据类型由其容器决定——值所存储的特定列。

SQLite 使用更通用的动态类型系统。在 SQLite 中，值的数据类型与值本身相关，而不是它的容器。SQLite 的动态类型系统与其他数据库引擎更常见的静态类型系统向后兼容，这意味着在静态类型数据库中有效的 SQL 语句在 SQLite 中也能以相同方式工作。然而，SQLite 中的动态类型使其能够完成传统刚性类型数据库无法实现的操作。灵活类型是一个特性，不是缺陷。

更新：截至 3.37.0 版本（2021-11-27），SQLite 提供了STRICT 表，为偏好刚性类型行为的开发者提供了严格的类型检查。

2. 存储类和数据类型

存储在 SQLite 数据库中（或由数据库引擎处理）的每个值都属于下列存储类之一：

  • NULL。该值是 NULL 值。

  • INTEGER。该值是有符号整数，根据值的大小，可使用 0、1、2、3、4、6 或 8 个字节存储。

  • REAL。该值是浮点数，以 8 字节 IEEE 浮点数表示。

  • TEXT。该值是文本字符串，使用数据库编码（UTF-8、UTF-16BE 或 UTF-16LE）存储。

  • BLOB。该值是二进制大对象，按输入时完全原样存储。

存储类比数据类型更通用。例如 INTEGER 存储类包含 7 种不同长度的整数数据类型。这在磁盘上有所区别。但只要 INTEGER 值从磁盘读取并加载到内存中进行处理，它们就会转换为最通用的数据类型（8 字节有符号整数）。因此在大多数情况下，“存储类”与“数据类型”几乎无法区分，这两个术语可互换使用。

除INTEGER PRIMARY KEY 列之外，SQLite 3 数据库中的任意列都可用于存储任何存储类的值。

SQL 语句中的所有值，无论是嵌入在 SQL 语句文本中的字面量，还是绑定到参数的预编译 SQL 语句，都具有隐含的存储类。在下文描述的某些情况下，数据库引擎可能会在查询执行过程中在数值存储类（INTEGER 和 REAL）与 TEXT 之间转换值。

2.1. 布尔数据类型

SQLite 没有单独的布尔存储类。相反，布尔值作为整数 0（false）和 1（true）存储。

从 3.23.0 版本（2018-04-02）起，SQLite 识别关键字“TRUE”和“FALSE”，但这些关键字实际上只是整数字面量 1 和 0 的另一种写法。

2.2. 日期和时间数据类型

SQLite 没有专门用于存储日期和/或时间的存储类。相反，SQLite 内置的日期和时间函数能够将日期和时间以 TEXT、REAL 或 INTEGER 值存储：

  • TEXT 以 ISO8601 字符串形式表示（"YYYY-MM-DD HH:MM:SS.SSS"）。
  • REAL 以儒略日数表示，从公元前 4714 年 11 月 24 日格林威治中午起的天数，按推测的公历计算。
  • INTEGER 以 Unix 时间表示，从 1970-01-01 00:00:00 UTC 起的秒数。

应用程序可以选择以上任何格式存储日期和时间，并使用内置日期和时间函数在格式之间自由转换。

3. 类型亲和性

使用刚性类型的 SQL 数据库引擎通常会尝试自动将值转换为合适的数据类型。考虑如下示例：

CREATE TABLE t1(a INT, b VARCHAR(10));
INSERT INTO t1(a,b) VALUES('123',456);

刚性类型数据库会在插入前将字符串 '123' 转换为整数 123，并将整数 456 转换为字符串 '456'。

为了最大限度地提高 SQLite 与其他数据库引擎之间的兼容性，并使上述示例在 SQLite 中与其他 SQL 数据库引擎一样工作，SQLite 支持列的“类型亲和性”概念。一列的类型亲和性是推荐用于存储在该列中的数据的类型。这里的重要思想是类型是建议而非强制的。任何列仍然可以存储任何类型的数据。只是某些列在可选情况下会更倾向于使用某种存储类。某列的首选存储类称为其“亲和性”。

SQLite 3 数据库中的每一列都被分配以下类型亲和性之一：

  • TEXT
  • NUMERIC
  • INTEGER
  • REAL
  • BLOB

（历史备注：BLOB 类型亲和性以前称为 “NONE”。但该术语容易与“无亲和性”混淆，因此改名。）

具有 TEXT 亲和性的列使用存储类 NULL、TEXT 或 BLOB 存储所有数据。如果将数值数据插入到 TEXT 亲和性列中，它会在保存前转换为文本形式。

具有 NUMERIC 亲和性的列可以包含所有五种存储类的值。当文本数据插入到 NUMERIC 列时，如果文本是格式良好的整数或实数文字，并且分别按照优先顺序，则会将文本的存储类转换为 INTEGER 或 REAL。如果 TEXT 值是格式良好的整数文字，但过大无法适配 64 位有符号整数，则会转换为 REAL。对于 TEXT 和 REAL 存储类之间的转换，仅保留数字的前 15 位有效十进制数字。如果 TEXT 值不是格式良好的整数或实数文字，则以 TEXT 形式存储。出于本段目的，十六进制整数文字不被视为格式良好，并将被存储为 TEXT。（这是为了与在 2014-08-15 的3.8.6 版本之前的 SQLite 兼容，当时才引入十六进制整数文字。）如果将可精确表示为整数的浮点值插入 NUMERIC 亲和性列，该值会被转换为整数。不会尝试转换 NULL 或 BLOB 值。

一个字符串可能看起来像带小数点和/或指数的浮点文字，但只要该值可以表示为整数，NUMERIC 亲和性就会将其转换为整数。因此字符串 '3.0e+5' 在 NUMERIC 亲和性列中存储为整数 300000，而不是浮点 300000.0。

使用 INTEGER 亲和性的列的行为与 NUMERIC 亲和性的列相同。INTEGER 与 NUMERIC 亲和性的差异仅在CAST 表达式中可见：表达式 "CAST(4.0 AS INT)" 返回整数 4，而 "CAST(4.0 AS NUMERIC)" 则保持浮点 4.0。

具有 REAL 亲和性的列类似于 NUMERIC 亲和性列，但其会将整数强制转换为浮点表示。（作为内部优化，在 REAL 亲和性列中存储的小浮点值且无小数部分会以整数形式写入磁盘以节省空间，并在读取时自动转换回浮点。这种优化在 SQL 层完全不可见，只能通过检查数据库文件的原始位来检测。）

具有 BLOB 亲和性的列不会偏向任何存储类，也不会尝试将数据从一种存储类强制转换为另一种。

3.1. 列亲和性的确定

对于未声明为STRICT 的表，列的亲和性由该列声明的类型决定，按以下顺序的规则：

  1. 如果声明类型包含字符串 "INT"，则分配 INTEGER 亲和性。

  2. 如果列的声明类型包含字符串 "CHAR"、"CLOB" 或 "TEXT"，则该列具有 TEXT 亲和性。注意，类型 VARCHAR 包含字符串 "CHAR"，因此分配 TEXT 亲和性。

  3. 如果列的声明类型包含字符串 "BLOB" 或未指定类型，则该列具有 BLOB 亲和性。

  4. 如果列的声明类型包含字符串 "REAL"、"FLOA" 或 "DOUB"，则该列具有 REAL 亲和性。

  5. 否则，亲和性为 NUMERIC。

请注意，确定列亲和性的规则顺序很重要。声明类型为 "CHARINT" 的列将同时匹配规则 1 和 2，但由于第一个规则具有优先级，该列亲和性为 INTEGER。

3.2. 表达式的亲和性

每个表列都有类型亲和性（BLOB、TEXT、INTEGER、REAL 或 NUMERIC 之一），但表达式不一定具有亲和性。

表达式亲和性由以下规则决定：

  • 如果 IN 或 NOT IN 操作符的右操作数是列表，则无亲和性；如果是 SELECT，则具有与结果集表达式相同的亲和性。

  • 如果表达式仅是对实际表（非VIEW 或子查询）某列的简单引用，则表达式具有与该表列相同的亲和性。

    • 对列名加括号会被忽略。因此如果 X 和 Y.Z 是列名，则 (X) 和 (Y.Z) 也被视为列名，并具有相应列的亲和性。

    • 应用于列名的任何操作符，包括无操作的单目 "+" 操作符，都会将列名转换为始终无亲和性的表达式。因此即使 X 和 Y.Z 是列名，表达式 +X 和 +Y.Z 也不是列名，并且没有亲和性。

  • 表达式形式为 "CAST(expr AS type)" 时，其亲和性与声明类型为 "type" 的列相同。

  • COLLATE 操作符具有与其左操作数相同的亲和性。

  • 否则，表达式没有亲和性。

3.3. 视图和子查询的列亲和性

VIEW 或 FROM 子句的子查询中的“列”实际上是实现 VIEW 或子查询的SELECT 语句结果集中的表达式。因此，VIEW 或子查询列的亲和性由上述表达式亲和性规则决定。考虑示例：

CREATE TABLE t1(a INT, b TEXT, c REAL);
CREATE VIEW v1(x,y,z) AS SELECT b, a+c, 42 FROM t1 WHERE b!=11;

v1.x 列的亲和性与 t1.b 相同（TEXT），因为 v1.x 直接映射到 t1.b。但 v1.y 和 v1.z 两列均无亲和性，因为它们分别映射到表达式 a+c 和 42，而表达式总是无亲和性。

3.3.1. 复合视图的列亲和性

当实现 VIEW 或 FROM 子句子查询的SELECT 语句是复合 SELECT 时，该 VIEW 或子查询中每一列的亲和性将取决于组成复合语句的某个单独 SELECT 语句中相应结果列的亲和性。然而，无法确定究竟使用哪个 SELECT 来确定亲和性。不同的组成 SELECT 语句可能在查询评估的不同时间用来确定亲和性。不同版本的 SQLite 可能使用不同的选择。同一版本中连续两个查询之间也可能不同。同一查询内部的不同时候也可能不同。因此，对于在组成子查询中具有不同亲和性的复合 SELECT 的列，永远无法确定将使用哪种亲和性。

最佳做法是，如果你在意结果的数据类型，就避免在复合 SELECT 中混合不同亲和性。混合亲和性可能导致令人惊讶且不直观的结果。例如，请查阅论坛帖子 02d7be94d7。

3.4. 列亲和性行为示例

下面的 SQL 演示了 SQLite 插入表时如何使用列亲和性进行类型转换。

CREATE TABLE t1(
    t  TEXT,     -- text affinity by rule 2
    nu NUMERIC,  -- numeric affinity by rule 5
    i  INTEGER,  -- integer affinity by rule 1
    r  REAL,     -- real affinity by rule 4
    no BLOB      -- no affinity by rule 3
);

-- Values stored as TEXT, INTEGER, INTEGER, REAL, TEXT.
INSERT INTO t1 VALUES('500.0', '500.0', '500.0', '500.0', '500.0');
SELECT typeof(t), typeof(nu), typeof(i), typeof(r), typeof(no) FROM t1;
text|integer|integer|real|text

-- Values stored as TEXT, INTEGER, INTEGER, REAL, REAL.
DELETE FROM t1;
INSERT INTO t1 VALUES(500.0, 500.0, 500.0, 500.0, 500.0);
SELECT typeof(t), typeof(nu), typeof(i), typeof(r), typeof(no) FROM t1;
text|integer|integer|real|real

-- Values stored as TEXT, INTEGER, INTEGER, REAL, INTEGER.
DELETE FROM t1;
INSERT INTO t1 VALUES(500, 500, 500, 500, 500);
SELECT typeof(t), typeof(nu), typeof(i), typeof(r), typeof(no) FROM t1;
text|integer|integer|real|integer

-- BLOBs are always stored as BLOBs regardless of column affinity.
DELETE FROM t1;
INSERT INTO t1 VALUES(x'0500', x'0500', x'0500', x'0500', x'0500');
SELECT typeof(t), typeof(nu), typeof(i), typeof(r), typeof(no) FROM t1;
blob|blob|blob|blob|blob

-- NULLs are also unaffected by affinity
DELETE FROM t1;
INSERT INTO t1 VALUES(NULL,NULL,NULL,NULL,NULL);
SELECT typeof(t), typeof(nu), typeof(i), typeof(r), typeof(no) FROM t1;
null|null|null|null|null

4. 比较表达式

SQLite 3 版本具有常见的 SQL 比较运算符，包括 “=”, “==”, “<”, “<=”, “>”, “>=”, “!=”, “”, “IN”, “NOT IN”, “BETWEEN”, “IS”, 和 “IS NOT”。

4.1. 排序顺序

比较结果取决于操作数的存储类，按以下规则判断：

  • 存储类为 NULL 的值被视为小于任何其他值（包括另一个 NULL 值）。

  • INTEGER 或 REAL 值小于任何 TEXT 或 BLOB 值。当 INTEGER 或 REAL 与另一个 INTEGER 或 REAL 比较时，执行数值比较。

  • TEXT 值小于 BLOB 值。当比较两个 TEXT 值时，使用适当的排序序列确定结果。

  • 当比较两个 BLOB 值时，使用 memcmp() 确定结果。

4.2. 比较前的类型转换

SQLite 可能在执行比较之前尝试在 INTEGER、REAL 和/或 TEXT 存储类之间转换值。是否在比较之前尝试转换取决于操作数的类型亲和性。

“应用亲和性”意味着仅在不会丢失关键信息的前提下，将操作数转换为特定存储类。数值总是可以转换为 TEXT。TEXT 值可以转换为数值，但前提是文本内容是格式良好的整数或实数文字，而不是十六进制整数文字。BLOB 值通过将二进制内容解释为当前数据库编码的文本字符串而转换为 TEXT。

在比较之前，按以下规则顺序对比较操作符的操作数应用亲和性：

  • 如果一个操作数具有 INTEGER、REAL 或 NUMERIC 亲和性，而另一个操作数具有 TEXT、BLOB 或无亲和性，则对另一个操作数应用 NUMERIC 亲和性。

  • 如果一个操作数具有 TEXT 亲和性而另一个无亲和性，则对另一个操作数应用 TEXT 亲和性。

  • 否则，不应用亲和性，两个操作数按原样比较。

表达式 "a BETWEEN b AND c" 被视为两个独立的二元比较 "a >= b AND a <= c"，即使这意味着在各个比较中对 'a' 应用不同的亲和性。形式为 "x IN (SELECT y ...)" 的比较中的数据类型转换处理方式如同比较实际上为 "x=y"。表达式 "a IN (x, y, z, ...)" 等价于 "a = +x OR a = +y OR a = +z OR ..."。换句话说，IN 操作符右侧的值（示例中的“x”“y”“z”）被视为无亲和性，即使它们恰好是列值或 CAST 表达式。

4.3. 比较示例

CREATE TABLE t1(
    a TEXT,      -- text affinity
    b NUMERIC,   -- numeric affinity
    c BLOB,      -- no affinity
    d            -- no affinity
);

-- Values will be stored as TEXT, INTEGER, TEXT, and INTEGER respectively
INSERT INTO t1 VALUES('500', '500', '500', 500);
SELECT typeof(a), typeof(b), typeof(c), typeof(d) FROM t1;
text|integer|text|integer

-- Because column "a" has text affinity, numeric values on the
-- right-hand side of the comparisons are converted to text before
-- the comparison occurs.
SELECT a < 40,   a < 60,   a < 600 FROM t1;
0|1|1

-- Text affinity is applied to the right-hand operands but since
-- they are already TEXT this is a no-op; no conversions occur.
SELECT a < '40', a < '60', a < '600' FROM t1;
0|1|1

-- Column "b" has numeric affinity and so numeric affinity is applied
-- to the operands on the right.  Since the operands are already numeric,
-- the application of affinity is a no-op; no conversions occur.  All
-- values are compared numerically.
SELECT b < 40,   b < 60,   b < 600 FROM t1;
0|0|1

-- Numeric affinity is applied to operands on the right, converting them
-- from text to integers.  Then a numeric comparison occurs.
SELECT b < '40', b < '60', b < '600' FROM t1;
0|0|1

-- No affinity conversions occur.  Right-hand side values all have
-- storage class INTEGER which are always less than the TEXT values
-- on the left.
SELECT c < 40,   c < 60,   c < 600 FROM t1;
0|0|0

-- No affinity conversions occur.  Values are compared as TEXT.
SELECT c < '40', c < '60', c < '600' FROM t1;
0|1|1

-- No affinity conversions occur.  Right-hand side values all have
-- storage class INTEGER which compare numerically with the INTEGER
-- values on the left.
SELECT d < 40,   d < 60,   d < 600 FROM t1;
0|0|1

-- No affinity conversions occur.  INTEGER values on the left are
-- always less than TEXT values on the right.
SELECT d < '40', d < '60', d < '600' FROM t1;
1|1|1

如果将示例中的比较交换顺序——将形式 "a<40" 重写为 "40>a"——所有结果仍相同。

5. 运算符

算术运算符（+、-、*、/、%、<<、>>、& 和 |）会将两个操作数都解释为数字。STRING 或 BLOB 操作数会自动转换为 REAL 或 INTEGER 值。如果 STRING 或 BLOB 看起来像实数（具有小数点或指数），或者其值超出 64 位有符号整数的表示范围，则会转换为 REAL。否则，操作数转换为 INTEGER。算术操作数的隐式类型转换与CAST 到 NUMERIC略有不同：看起来像实数但没有小数部分的字符串和 BLOB 值会保留为 REAL，而不是像 CAST 到 NUMERIC 那样转换为 INTEGER。即使转换是有损且不可逆的，也会将 STRING 或 BLOB 转换为 REAL 或 INTEGER。一些算术运算符（%、<<、>>、& 和 |）期望 INTEGER 操作数。对于这些运算符，REAL 操作数会像CAST 到 INTEGER一样转换为 INTEGER。<<、>>、& 和 | 运算符始终返回 INTEGER（或 NULL）结果，但 % 运算符根据操作数类型返回 INTEGER 或 REAL（或 NULL）。算术运算符的 NULL 操作数会产生 NULL 结果。对于既不是 NULL 也看起来绝非数值的算术操作数，会将其转换为 0 或 0.0。除零会返回 NULL。

6. 排序、分组和复合 SELECT

当查询结果通过 ORDER BY 子句排序时，存储类为 NULL 的值排在最前，其后以数值顺序穿插的 INTEGER 和 REAL 值，再之后是按照排序序列顺序排列的 TEXT 值，最后是按 memcmp() 顺序排列的 BLOB 值。排序之前不会发生存储类转换。

当使用 GROUP BY 子句对值进行分组时，不同存储类的值被视为不同，除非 INTEGER 和 REAL 值在数值上相等。在 GROUP BY 结果中不对任何值应用亲和性。

复合 SELECT 运算符 UNION、INTERSECT 和 EXCEPT 在值之间执行隐式比较。与 UNION、INTERSECT 或 EXCEPT 相关的隐式比较的操作数不会应用亲和性——值按原样比较。

7. 排序序列

当 SQLite 比较两个字符串时，会使用排序序列或排序函数（两个术语指同一事物）来确定哪个字符串更大或两字符串是否相等。SQLite 提供三种内置排序函数：BINARY、NOCASE 和 RTRIM。

  • BINARY - 使用 memcmp() 比较字符串数据，无论文本编码为何。
  • NOCASE - 类似于 BINARY，但比较时使用sqlite3_strnicmp()。因此，ASCII 的 26 个大写字符在比较前会折叠为其小写形式。注意只有 ASCII 字符会被折叠。由于所需表格庞大，SQLite 不尝试进行完整的 UTF 大小写折叠。此外，字符串中的任何 U+0000 字符在比较中被视为字符串终止符。
  • RTRIM - 与 BINARY 相同，但忽略尾随空格字符。

应用程序可以通过sqlite3_create_collation()接口注册额外的排序函数。

排序函数仅在比较字符串值时有意义。数值始终按数值比较，BLOB 始终按字节逐个比较（memcmp()）。

7.1. 通过 SQL 分配排序序列

每张表的每一列都有一个关联的排序函数。如果未显式定义排序函数，则默认为 BINARY。列定义中的 COLLATE 子句用于为列定义可选的排序函数。

确定用于二元比较操作符（=、<、>、<=、>=、!=、IS 和 IS NOT）的排序函数的规则如下：

  1. 如果任一操作数通过后缀COLLATE 操作符显式指定了排序函数，则使用该排序函数，优先级归属于左操作数的排序函数。

  2. 如果任一操作数是列，则使用该列的排序函数，左操作数优先。在此前提下，即使列名前面带有一个或多个单目 "+" 操作符和/或 CAST 操作符，也仍然被视为列名。

  3. 否则，使用 BINARY 排序函数进行比较。

如果比较操作的任一子表达式使用了COLLATE 操作符，则该操作被视为对该操作数显式指定了排序函数（规则 1）。因此，如果比较表达式中任一处使用了 COLLATE 操作符，则无论表达式中包含哪些表列，都将使用该操作符定义的排序函数进行字符串比较。如果比较中出现两个或多个COLLATE 操作符子表达式，将使用最左侧的显式排序函数，而不考虑这些 COLLATE 操作符在表达式中的嵌套深度或表达式的括号结构。

表达式 "x BETWEEN y and z" 逻辑上等价于两个比较 "x >= y AND x <= z"，在排序函数方面也如同两个独立比较。表达式 "x IN (SELECT y ...)" 处理方式与表达式 "x = y" 相同，以确定排序序列。形式为 "x IN (y, z, ...)" 表达式使用 x 的排序序列。如果需要在 IN 操作符上显式指定排序序列，应将 COLLATE 应用于左操作数，例如 "x COLLATE nocase IN (y,z, ...)"。

作为SELECT 语句一部分的 ORDER BY 子句中的项可以使用COLLATE 操作符指定排序序列，在这种情况下，将使用指定的排序函数进行排序。否则，如果 ORDER BY 排序的是某列，则使用该列的排序序列确定排序顺序。如果排序表达式不是列且未使用 COLLATE 子句，则使用 BINARY 排序序列。

7.2. 排序序列示例

下面的示例说明了在执行各种 SQL 语句时用于确定字符串比较结果的排序序列。请注意，对于数值、BLOB 或 NULL 值，可能不需要字符串比较，也不会使用排序序列。

CREATE TABLE t1(
    x INTEGER PRIMARY KEY,
    a,                 /* collating sequence BINARY */
    b COLLATE BINARY,  /* collating sequence BINARY */
    c COLLATE RTRIM,   /* collating sequence RTRIM  */
    d COLLATE NOCASE   /* collating sequence NOCASE */
);
                   /* x   a     b     c       d */
INSERT INTO t1 VALUES(1,'abc','abc', 'abc  ','abc');
INSERT INTO t1 VALUES(2,'abc','abc', 'abc',  'ABC');
INSERT INTO t1 VALUES(3,'abc','abc', 'abc ', 'Abc');
INSERT INTO t1 VALUES(4,'abc','abc ','ABC',  'abc');
 
/* Text comparison a=b is performed using the BINARY collating sequence. */
SELECT x FROM t1 WHERE a = b ORDER BY x;
--result 1 2 3

/* Text comparison a=b is performed using the RTRIM collating sequence. */
SELECT x FROM t1 WHERE a = b COLLATE RTRIM ORDER BY x;
--result 1 2 3 4

/* Text comparison d=a is performed using the NOCASE collating sequence. */
SELECT x FROM t1 WHERE d = a ORDER BY x;
--result 1 2 3 4

/* Text comparison a=d is performed using the BINARY collating sequence. */
SELECT x FROM t1 WHERE a = d ORDER BY x;
--result 1 4

/* Text comparison 'abc'=c is performed using the RTRIM collating sequence. */
SELECT x FROM t1 WHERE 'abc' = c ORDER BY x;
--result 1 2 3

/* Text comparison c='abc' is performed using the RTRIM collating sequence. */
SELECT x FROM t1 WHERE c = 'abc' ORDER BY x;
--result 1 2 3

/* Grouping is performed using the NOCASE collating sequence (Values
** 'abc', 'ABC', and 'Abc' are placed in the same group). */
SELECT count(*) FROM t1 GROUP BY d ORDER BY 1;
--result 4

/* Grouping is performed using the BINARY collating sequence.  'abc' and
** 'ABC' and 'Abc' form different groups */
SELECT count(*) FROM t1 GROUP BY (d || '') ORDER BY 1;
--result 1 1 2

/* Sorting or column c is performed using the RTRIM collating sequence. */
SELECT x FROM t1 ORDER BY c, x;
--result 4 1 2 3

/* Sorting of (c||'') is performed using the BINARY collating sequence. */
SELECT x FROM t1 ORDER BY (c||''), x;
--result 4 2 3 1

/* Sorting of column c is performed using the NOCASE collating sequence. */
SELECT x FROM t1 ORDER BY c COLLATE NOCASE, x;
--result 2 4 3 1
