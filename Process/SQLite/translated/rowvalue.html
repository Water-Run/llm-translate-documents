1. 定义

“值”是单个数字、字符串、BLOB 或 NULL。有时会使用限定名称“标量值”以强调只涉及单个量。

“行值”是两个或两个以上标量值的有序列表。换句话说，“行值”是一个向量或元组。

行值的“大小”是该行值包含的标量值数量。行值的大小始终至少为 2。只有单列的行值就是标量值。没有列的行值是语法错误。

2. 语法

SQLite 允许以两种方式表示行值：

  1. 用括号括起并用逗号分隔的标量值列表。
  2. 带有两个或多个结果列的子查询表达式。

SQLite 可以在两种上下文中使用行值：

  1. 两个大小相同的行值可以使用运算符 <, <=, >, >=, =, <>, IS, IS NOT, IN, NOT IN, BETWEEN 或 CASE 进行比较。
  2. 在 UPDATE 语句中，一列名称列表可以设置为与之大小相同的行值。

行值的语法以及可以使用行值的情况将在下面的示例中说明。

2.1. 行值比较

比较两个行值时，会从左到右依次查看其组成的标量值。NULL 表示“未知”。如果通过将组成中的 NULL 替换为其它值可以使得比较结果为真或者为假，则整体比较结果为 NULL。下面的查询演示了一些行值比较： 

SELECT
  (1,2,3) = (1,2,3),          -- 1
  (1,2,3) = (1,NULL,3),       -- NULL
  (1,2,3) = (1,NULL,4),       -- 0
  (1,2,3) < (2,3,4),          -- 1
  (1,2,3) < (1,2,4),          -- 1
  (1,2,3) < (1,3,NULL),       -- 1
  (1,2,3) < (1,2,NULL),       -- NULL
  (1,3,5) < (1,2,NULL),       -- 0
  (1,2,NULL) IS (1,2,NULL);   -- 1

“(1,2,3)=(1,NULL,3)” 的结果是 NULL，因为如果我们将 NULL 替换为 2 结果可能为真，替换为 9 结果可能为假。“(1,2,3)=(1,NULL,4)” 的结果不是 NULL，因为不存在将组成的 NULL 替换为某个值后使表达式为真（第三列中 3 永远不会等于 4）。

前面示例中的任何行值都可以用返回三列的子查询替换，结果相同。例如： 

CREATE TABLE t1(a,b,c);
INSERT INTO t1(a,b,c) VALUES(1,2,3);
SELECT (1,2,3)=(SELECT * FROM t1); -- 1

2.2. 行值 IN 运算符

对于行值 IN 运算符，左侧（下文简称为 “LHS”）可以是括号括起的值列表或具有多列的子查询。但右侧（下文简称为 “RHS”）必须是子查询表达式。 

CREATE TABLE t2(x,y,z);
INSERT INTO t2(x,y,z) VALUES(1,2,3),(2,3,4),(1,NULL,5);
SELECT
   (1,2,3) IN (SELECT * FROM t2),  -- 1
   (7,8,9) IN (SELECT * FROM t2),  -- 0
   (1,3,5) IN (SELECT * FROM t2);  -- NULL

2.3. UPDATE 语句中的行值

行值还可以用于 UPDATE 语句的 SET 子句中。LHS 必须是列名列表。RHS 可以是任何行值。例如： 

UPDATE tab3 
   SET (a,b,c) = (SELECT x,y,z
                    FROM tab4
                   WHERE tab4.w=tab3.d)
 WHERE tab3.e BETWEEN 55 AND 66;

3. 行值示例用法

3.1. 滚动窗口查询

假设一个应用程序希望按姓、名的字母顺序显示联系人列表，在一个一次仅能展示 7 条记录的滚动窗口中。初始化窗口以显示前 7 条记录很容易： 

SELECT * FROM contacts
 ORDER BY lastname, firstname
 LIMIT 7;

当用户向下滚动时，应用需要找到第二组 7 条记录。一种做法是使用 OFFSET 子句： 

SELECT * FROM contacts
 ORDER BY lastname, firstname
 LIMIT 7 OFFSET 7;

OFFSET 会给出正确的答案。然而，OFFSET 所需时间与偏移值成正比。实际上 “LIMIT x OFFSET y” 会被 SQLite 处理为 “LIMIT x+y”，并丢弃前 y 个值而不返回给应用。因此，随着窗口向长列表底部滚动，y 值越来越大，连续的偏移计算会越来越耗时。

一种更高效的方法是记住当前屏幕底部显示的最后一条记录，然后在 WHERE 子句中使用行值比较： 

SELECT * FROM contacts
 WHERE (lastname,firstname) > (?1,?2)
 ORDER BY lastname, firstname
 LIMIT 7;

如果前一屏幕底部的 lastname 和 firstname 分别绑定到 ?1 和 ?2，则上述查询会计算出下一组 7 条记录。并且，只要存在合适的索引，这样做非常高效——远比 OFFSET 更高效。

3.2. 将日期拆分字段存储后的比较

在数据库表中存储日期的常见方式是一个字段，内容可以是 unix 时间戳、儒略日编号或 ISO-8601 日期字符串。但有些应用将日期拆分为年、月、日三列分别存储。 

CREATE TABLE info(
  year INT,          -- 4 digit year
  month INT,         -- 1 through 12
  day INT,           -- 1 through 31
  other_stuff BLOB   -- blah blah blah
);

当日期以这种方式存储时，行值比较提供了一种方便的方式来比较日期： 

SELECT * FROM info
 WHERE (year,month,day) BETWEEN (2015,9,12) AND (2016,9,12);

3.3. 针对多列键的搜索

假设我们想知道在订单号 365 中任意商品的产品号和数量所对应的订单号、产品号与数量： 

SELECT ordid, prodid, qty
  FROM item
 WHERE (prodid, qty) IN (SELECT prodid, qty
                           FROM item
                          WHERE ordid = 365);

上述查询可以重写为一个连接表达式，而不使用行值： 

SELECT t1.ordid, t1.prodid, t1.qty
  FROM item AS t1, item AS t2
 WHERE t1.prodid=t2.prodid
   AND t1.qty=t2.qty
   AND t2.ordid=365;

因为相同的查询可以在不使用行值的情况下编写，所以行值并不提供新的能力。不过，许多开发者认为行值语法更容易阅读、编写和调试。

即使在 JOIN 的形式中，通过使用行值也可以让查询更清晰： 

SELECT t1.ordid, t1.prodid, t1.qty
  FROM item AS t1, item AS t2
 WHERE (t1.prodid,t1.qty) = (t2.prodid,t2.qty)
   AND t2.ordid=365;

后一个查询生成的 字节码与前一个标量表述完全相同，但语法更简洁、更易于阅读。

3.4. 基于查询更新多个列

行值表示法对于用单个查询结果更新表中的两个或多个列很有用。一个例子存在于 Fossil 版本控制系统的全文搜索功能中。

在 Fossil 的全文搜索系统中，参与全文搜索的文档（维基页面、问题单、提交、文档文件等）都由名为 “ftsdocs”（全称 full text search documents 的表）跟踪。随着新文档被添加到仓库，它们不会立即被编入索引。索引操作会延迟到有搜索请求时再执行。ftsdocs 表包含一个名为 idxed 的字段，如果文档已经被编入索引则为 true，否则为 false。

当发生搜索请求并且待处理文档第一次被索引时，需要通过设置 idxed 列为 true 并填充与搜索相关的其他若干列来更新 ftsdocs 表。其他信息来自一个连接。查询如下： 

UPDATE ftsdocs SET
  idxed=1,
  name=NULL,
  (label,url,mtime) = 
      (SELECT printf('Check-in [%%.16s] on %%s',blob.uuid,
                     datetime(event.mtime)),
              printf('/timeline?y=ci&c=%%.20s',blob.uuid),
              event.mtime
         FROM event, blob
        WHERE event.objid=ftsdocs.rid
          AND blob.rid=ftsdocs.rid)
WHERE ftsdocs.type='c' AND NOT ftsdocs.idxed

（参见 源代码了解更多细节。其他示例见 此处和 此处。）

ftsdocs 表的九列中有五列会被更新。两个可独立于查询更新的列是 “idxed” 和 “name”。但另外三列 “label”、“url” 和 “mtime” 全部需要针对 “event” 和 “blob” 表的连接查询。没有行值的话，相当的 UPDATE 必须为每个待更新的列重复一次连接。

3.5. 表达更加清晰

有时候使用行值可以让 SQL 更易读、易写。考虑下面两个 UPDATE 语句： 

UPDATE tab1 SET (a,b)=(b,a);
UPDATE tab1 SET a=b, b=a;

这两个 UPDATE 语句实现的效果完全相同。（它们生成相同的 字节码。）但第一种形式，即行值形式，更清楚地表明语句的意图是交换 A 和 B 两列的值。

或者再考虑下面两个等价的查询： 

SELECT * FROM tab1 WHERE a=?1 AND b=?2;
SELECT * FROM tab1 WHERE (a,b)=(?1,?2);

再次强调，这些 SQL 语句生成的字节码一致，因此以完全相同的方式执行。但第二种形式通过将查询参数按行值组合在一起，而不是散落在 WHERE 子句中，使得人类更容易阅读。

4. 向后兼容性

行值在 SQLite 3.15.0 版本（2016-10-14）中被引入。尝试在更早的 SQLite 版本中使用行值将产生语法错误。
