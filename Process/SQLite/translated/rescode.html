概览

许多 SQLite C-language Interface 中的例程会返回 数值型结果代码，用以表示成功还是失败，如果发生失败，也能 提供一些有关失败原因的线索。本文档力求解释每一个 这些数值结果代码的含义。

1. 结果代码与错误代码

“错误代码”是“结果代码”的一个子集，用来表明 发生了一些问题。只有少数几个非错误结果 代码：SQLITE_OK、SQLITE_ROW、SQLITE_DONE。术语 “错误代码”指除这三个之外的任何结果代码。

2. 主结果代码与扩展结果代码

结果代码是有符号的 32 位整数。 结果代码的最低 8 位定义了一个大类， 称为“主结果代码”。更高位提供 更详细的错误信息，称为 “扩展结果代码”。

注意主结果代码始终包含在扩展 结果代码中。给定完整的 32 位扩展结果代码，应用程序 总能仅通过提取扩展结果代码的最低 8 位来找到相应的主结果代码。

所有扩展结果代码也是错误代码。因此 “扩展结果代码”和“扩展错误代码”可以互换使用。

为保持历史兼容性，C 语言接口默认返回 主结果代码。 最近一次错误的扩展结果代码可以通过 sqlite3_extended_errcode() 接口获取。 sqlite3_extended_result_codes() 接口可用于将 数据库连接置于一个模式，使其返回 扩展结果代码而非主结果代码。

3. 定义

所有结果代码都是整数。 sqlite3.h 头文件中使用“#define”宏 为所有结果代码创建符号名称。 sqlite3.h 中分别为 结果代码定义和 扩展结果代码定义设置了不同的章节。

主结果代码的符号名称形式为“SQLITE_XXXXXX”，其中 XXXXXX 是一串大写字母。扩展 结果代码名称的形式为“SQLITE_XXXXXX_YYYYYYY”，其中 XXXXXX 部分对应主结果代码，YYYYYYY 是对结果代码 进一步分类的扩展。

现有结果代码的名称和数值是固定的 且不会变化。然而，新的结果代码，特别是新的扩展 结果代码，可能会在 SQLite 的未来版本中出现。

6. 结果代码含义

下列内容按数值顺序列出了所有 105 个 结果代码值的含义。

(0) SQLITE_OK

SQLITE_OK 结果代码表示操作成功且 无任何错误。大多数其他结果代码表示错误。

(1) SQLITE_ERROR

SQLITE_ERROR 结果代码是一种通用错误代码，当 没有更具体的错误代码可用时使用。

(2) SQLITE_INTERNAL

SQLITE_INTERNAL 结果代码表示内部故障。 在正常工作的 SQLite 中，应用程序不应当看到此 结果代码。如果应用程序遇到此结果代码，则表明 数据库引擎中存在缺陷。

此结果代码可能由 SQLite 中的一个 Bug 引起。 但是，应用程序定义的 SQL 函数或 虚拟表、VFSes或其他扩展也可能导致返回此 结果代码，因此问题未必由 SQLite 核心引起。

(3) SQLITE_PERM

SQLITE_PERM 结果代码表示无法为新建数据库 提供所请求的访问模式。

(4) SQLITE_ABORT

SQLITE_ABORT 结果代码表示某个操作在完成前被中止， 通常是由应用程序请求引起。 另见：SQLITE_INTERRUPT。

如果 sqlite3_exec() 的回调函数返回非零值，则 sqlite3_exec() 将返回 SQLITE_ABORT。

若在与待处理的读或写操作位于同一个 数据库连接上的 ROLLBACK 操作发生，则该待处理的读或写可能会 返回 SQLITE_ABORT 或 SQLITE_ABORT_ROLLBACK 错误。

除了用作结果代码外， SQLITE_ABORT 值还用作从 sqlite3_vtab_on_conflict() 接口返回的 冲突解决模式。

(5) SQLITE_BUSY

SQLITE_BUSY 结果代码表示无法写入（在某些情况下也无法读取）数据库文件， 原因是某个其他 数据库连接，通常是位于不同进程中的连接，正在并发访问。

例如，如果进程 A 正在执行大规模写事务， 同时进程 B 试图启动新的写事务，则进程 B 会收到 SQLITE_BUSY，因为 SQLite 一次仅支持一个写者。进程 B 必须等待进程 A 完成 其事务后才能启动新事务。sqlite3_busy_timeout()、sqlite3_busy_handler() 接口以及 busy_timeout pragma 可供进程 B 使用，以帮助其处理 SQLITE_BUSY 错误。

在事务中的任何阶段都可能发生 SQLITE_BUSY 错误：当 事务刚开始、执行任何写或更新操作期间，或在提交时。 为避免在事务中途遇到 SQLITE_BUSY 错误， 应用程序可以使用 BEGIN IMMEDIATE，而不仅仅是 BEGIN 来 启动事务。BEGIN IMMEDIATE 命令本身可能返回 SQLITE_BUSY，但如果成功，则 SQLite 保证在接下来的 COMMIT 之前通过同一个数据库进行的后续操作不会返回 SQLITE_BUSY。

另见：SQLITE_BUSY_RECOVERY 和 SQLITE_BUSY_SNAPSHOT。

SQLITE_BUSY 结果代码与 SQLITE_LOCKED 不同，SQLITE_BUSY 表示与 另一个 数据库连接（可能在不同进程中）之间的冲突， 而 SQLITE_LOCKED 表示在同一个 数据库连接（或有 共享缓存 的连接）内部的冲突。

(6) SQLITE_LOCKED

SQLITE_LOCKED 结果代码表示写操作因 同一 数据库连接 内部的冲突或 与使用 共享缓存 的不同数据库连接发生冲突而无法继续。

例如，当另一个线程在同一个 数据库连接上从该表读取时， DROP TABLE 语句无法运行， 因为删除表会将表从并发读取者下方删除。

SQLITE_LOCKED 结果代码与 SQLITE_BUSY 不同， SQLITE_LOCKED 表示在同一个 数据库连接 （或有 共享缓存 的连接）上的冲突，而 SQLITE_BUSY 表示与另一个数据库连接（可能在不同进程中）之间的冲突。

(7) SQLITE_NOMEM

SQLITE_NOMEM 结果代码表示 SQLite 无法分配 完成操作所需的全部内存。换言之， 内部对 sqlite3_malloc() 或 sqlite3_realloc() 的调用失败， 而被分配的内存对于继续操作是必要的。

(8) SQLITE_READONLY

当尝试修改当前数据库连接 没有写权限的数据时，会返回 SQLITE_READONLY 结果代码。

(9) SQLITE_INTERRUPT

SQLITE_INTERRUPT 结果代码表示某个操作被 sqlite3_interrupt() 接口中断。 另见：SQLITE_ABORT

(10) SQLITE_IOERR

SQLITE_IOERR 结果代码表示操作无法完成， 因为操作系统报出了 I/O 错误。

当磁盘已满时，通常会返回 SQLITE_FULL 错误，而不是 SQLITE_IOERR。

有许多不同的扩展结果代码用于 I/O 错误， 以标识具体失败的 I/O 操作。

(11) SQLITE_CORRUPT

SQLITE_CORRUPT 结果代码表示数据库文件已 损坏。有关损坏可能如何发生的更多讨论，请参阅 如何破坏数据库文件。

(12) SQLITE_NOTFOUND

SQLITE_NOTFOUND 结果代码通过三种方式暴露：

  1. sqlite3_file_control() 接口可能返回 SQLITE_NOTFOUND，以指示作为第三个参数传入的 文件控制操作码 未被底层的 VFS 识别。

  2. sqlite3_vfs 对象的 xSetSystemCall() 方法也可能返回 SQLITE_NOTFOUND。

  3. sqlite3_vtab_rhs_value() 可能返回 SQLITE_NOTFOUND， 以指示约束的右操作数对于发出调用的 xBestIndex 方法 不可用。

SQLITE_NOTFOUND 结果代码也在 SQLite 实现的内部使用， 但这些内部用途不会向应用程序暴露。

(13) SQLITE_FULL

SQLITE_FULL 结果代码表示写操作无法完成， 因为磁盘已满。注意，此错误可能发生在尝试 向主数据库文件写入信息时，也可能 发生在向 临时磁盘文件 写入时。

有时即便主磁盘空间很充足，应用程序仍会遇到此错误， 因为该错误发生在线程向 临时磁盘文件 写入，而系统将临时文件存放在 空间远小于主磁盘的独立分区上。

(14) SQLITE_CANTOPEN

SQLITE_CANTOPEN 结果代码表示 SQLite 无法 打开某个文件。该文件可能是主数据库文件 或若干 (临时磁盘文件) 中的某一个。

(15) SQLITE_PROTOCOL

SQLITE_PROTOCOL 结果代码表示 SQLite 使用的文件锁协议 出现了问题。目前只有在使用 WAL 模式 并试图启动新事务时 会返回 SQLITE_PROTOCOL 错误。 当两个独立的 数据库连接 在 WAL 模式 中几乎同时尝试启动事务时，可能出现竞争条件。 竞争失败者会退避并稍后重试。如果同一个连接在数秒内多次在锁竞争中失败， 它最终会放弃并返回 SQLITE_PROTOCOL。 SQLITE_PROTOCOL 错误在实践中极少出现，仅在多个进程 激烈竞争写入同一数据库时才可能出现。

(16) SQLITE_EMPTY

SQLITE_EMPTY 结果代码当前未使用。

(17) SQLITE_SCHEMA

SQLITE_SCHEMA 结果代码表示数据库模式 已更改。对于通过 sqlite3_prepare() 或 sqlite3_prepare16() 生成的 prepared statement，sqlite3_step() 可能返回此错误。 如果在 statement 准备完成与执行之间，数据库模式被其他进程更改，则可能发生该错误。

如果通过 sqlite3_prepare_v2() 生成 prepared statement，则如果模式更改， 该语句会自动重新准备，最多重试 SQLITE_MAX_SCHEMA_RETRY 次（默认值：50）。 只有当重试次数耗尽后，sqlite3_step() 接口才会将 SQLITE_SCHEMA 返回给应用程序。

(18) SQLITE_TOOBIG

SQLITE_TOOBIG 错误码表示字符串或 BLOB 太大。SQLite 中字符串或 BLOB 的默认最大长度为 1,000,000,000 字节。此最大长度可以通过 SQLITE_MAX_LENGTH 编译时选项进行更改，或通过 sqlite3_limit(db,SQLITE_LIMIT_LENGTH,...) 接口在运行时更改。当 SQLite 遇到超过编译时或运行时限制的字符串或 BLOB 时，就会返回 SQLITE_TOOBIG 错误。

当将过大的 SQL 语句传入某个 sqlite3_prepare_v2() 接口时，也会返回 SQLITE_TOOBIG 错误码。SQL 语句的最大长度默认也是 1,000,000,000 字节。可以通过 SQLITE_MAX_SQL_LENGTH 在编译时设置最大 SQL 语句长度，或通过 sqlite3_limit(db,SQLITE_LIMIT_SQL_LENGTH,...) 在运行时设置。

(19) SQLITE_CONSTRAINT

SQLITE_CONSTRAINT 错误码表示在处理 SQL 语句时发生了 SQL 约束违反。可以通过查看随附的错误消息（通过 sqlite3_errmsg() 或 sqlite3_errmsg16() 返回）或查看 扩展错误码 来获取有关失败约束的其他信息。

SQLITE_CONSTRAINT 代码也可以用作 xBestIndex() 方法的返回值，该方法属于 虚拟表 的实现。当 xBestIndex() 返回 SQLITE_CONSTRAINT 时，表示提交给 xBestIndex() 的特定输入组合无法生成可用的查询计划，不应继续考虑。

(20) SQLITE_MISMATCH

SQLITE_MISMATCH 错误码表示数据类型不匹配。

SQLite 通常对值的类型与其要存储的容器声明类型之间的不匹配非常宽容。例如，SQLite 允许应用在声明类型为 BOOLEAN 的列中存储大型 BLOB。但在少数情况下，SQLite 会对类型严格要求。当类型不匹配时，就会在这些情况下返回 SQLITE_MISMATCH 错误。

表的 rowid 必须是整数。尝试将 rowid 设置为非整数（或会自动转换为下一个可用整数 rowid 的 NULL）会导致 SQLITE_MISMATCH 错误。

(21) SQLITE_MISUSE

如果应用以未定义或不受支持的方式使用任何 SQLite 接口，可能会返回 SQLITE_MISUSE 返回码。例如，在 prepared statement 被 finalized 之后继续使用该 prepared statement，可能会导致 SQLITE_MISUSE 错误。

SQLite 会尝试检测滥用并通过该结果码报告。然而，并不能保证滥用检测一定成功。滥用检测具有概率性。应用绝不应依赖 SQLITE_MISUSE 的返回值。

如果 SQLite 在任何接口中返回 SQLITE_MISUSE，说明应用代码存在错误，需要修复。不要发布一个在标准 SQLite 接口中偶尔返回 SQLITE_MISUSE 的应用，因为该应用可能包含严重缺陷。

(22) SQLITE_NOLFS

当数据库增长到超过文件系统支持的大小且系统不支持大文件时，可能会返回 SQLITE_NOLFS 错误。“NOLFS” 表示 “NO Large File Support（不支持大文件）”。

(23) SQLITE_AUTH

当 authorizer callback 表示正在准备的 SQL 语句未经授权时，会返回 SQLITE_AUTH 错误。

(24) SQLITE_FORMAT

SQLite 当前并不使用 SQLITE_FORMAT 错误码。

(25) SQLITE_RANGE

SQLITE_RANGE 错误表示传递给某个 sqlite3_bind 例程的参数编号或某个 sqlite3_column 例程的列编号超出了范围。

(26) SQLITE_NOTADB

尝试打开文件时，如果 SQLITE_NOTADB 错误出现，表示所打开的文件看起来并非 SQLite 数据库文件。

(27) SQLITE_NOTICE

任何 C/C++ 接口都不会返回 SQLITE_NOTICE 结果码。然而，当通过 sqlite3_log() 回调报告某个不寻常的操作时，有时会使用 SQLITE_NOTICE（或其 扩展错误码 之一）作为第一个参数。

(28) SQLITE_WARNING

任何 C/C++ 接口都不会返回 SQLITE_WARNING 结果码。然而，当通过 sqlite3_log() 回调报告可能不明智且不同寻常的操作时，有时会使用 SQLITE_WARNING（或其 扩展错误码 之一）作为第一个参数。

(100) SQLITE_ROW

由 sqlite3_step() 返回的 SQLITE_ROW 结果码表示又有可用的输出行。

(101) SQLITE_DONE

SQLITE_DONE 结果码表示某个操作已完成。最常见的情况是从 sqlite3_step() 返回 SQLITE_DONE，表示 SQL 语句已经执行完毕。但 SQLITE_DONE 也可能由其他多步接口返回，例如 sqlite3_backup_step()。

(256) SQLITE_OK_LOAD_PERMANENTLY

sqlite3_load_extension() 接口将一个 extension 加载到单个数据库连接中。默认行为是在数据库连接关闭时自动卸载该扩展。然而，如果扩展入口点返回 SQLITE_OK_LOAD_PERMANENTLY 而不是 SQLITE_OK，那么在数据库连接关闭后，该扩展仍会保留在进程地址空间中。换句话说，在数据库连接关闭时，不会对该扩展调用 sqlite3_vfs 对象的 xDlClose 方法。

对于注册新 VFSes 的 loadable extensions 来说，SQLITE_OK_LOAD_PERMANENTLY 返回码非常有用。

(257) SQLITE_ERROR_MISSING_COLLSEQ

SQLITE_ERROR_MISSING_COLLSEQ 结果码表示因 SQL 语句中引用的某个排序序列无法找到而无法准备该 SQL 语句。

有时遇到此错误码时，sqlite3_prepare_v2() 例程会将该错误转换为 SQLITE_ERROR_RETRY，并尝试使用不需要未知排序序列的其他查询计划再次准备 SQL 语句。

(261) SQLITE_BUSY_RECOVERY

SQLITE_BUSY_RECOVERY 错误码是 扩展错误码，用于表示 SQLITE_BUSY，提示某个操作无法继续，因为另一个进程正忙于在崩溃后恢复 WAL mode 数据库文件。SQLITE_BUSY_RECOVERY 错误只会在 WAL mode 数据库上出现。

(262) SQLITE_LOCKED_SHAREDCACHE

SQLITE_LOCKED_SHAREDCACHE 结果码表示对 SQLite 数据记录的访问被另一个正在以 shared cache mode 使用同一记录的数据库连接阻塞。当两个或更多数据库连接共享同一缓存且其中一个连接正在该缓存中修改记录时，其它连接在修改进行中无法访问该数据，以防读者看到已损坏或未完成的修改。

(264) SQLITE_READONLY_RECOVERY

SQLITE_READONLY_RECOVERY 错误码是 扩展错误码，用于表示 SQLITE_READONLY。该错误码表示由于数据库文件需要恢复且恢复需要写访问权限，但当前只有读访问权限，因此 WAL mode 数据库无法打开。

(266) SQLITE_IOERR_READ

SQLITE_IOERR_READ 错误码是 扩展错误码，用于表示 SQLITE_IOERR，指示在试图从磁盘文件读取时 VFS 层发生 I/O 错误。该错误可能由硬件故障引起，或由于在文件打开期间文件系统被卸载。

(267) SQLITE_CORRUPT_VTAB

SQLITE_CORRUPT_VTAB 错误码是 扩展错误码，用于表示 SQLITE_CORRUPT，供 virtual tables 使用。虚拟表可能返回 SQLITE_CORRUPT_VTAB，以表明虚拟表中的内容已损坏。

(270) SQLITE_CANTOPEN_NOTEMPDIR

SQLITE_CANTOPEN_NOTEMPDIR 错误码已不再使用。

(275) SQLITE_CONSTRAINT_CHECK

SQLITE_CONSTRAINT_CHECK 错误码是 扩展错误码，用于表示 SQLITE_CONSTRAINT，提示某个 CHECK constraint 失败。

(279) SQLITE_AUTH_USER

SQLITE_AUTH_USER 错误码是 扩展错误码，用于表示 SQLITE_AUTH，说明尝试在当前登录用户权限不足的数据库上进行某个操作。

(283) SQLITE_NOTICE_RECOVER_WAL

SQLITE_NOTICE_RECOVER_WAL 结果码在恢复 WAL mode 数据库文件时通过 sqlite3_log() 的回调传递。

(284) SQLITE_WARNING_AUTOINDEX

每当使用 automatic indexing 时，SQLITE_WARNING_AUTOINDEX 结果码会作为第一个参数传递给 sqlite3_log() 的回调。这可提醒应用设计者数据库可能需要额外的索引。

(513) SQLITE_ERROR_RETRY

SQLITE_ERROR_RETRY 在内部用于促使 sqlite3_prepare_v2()（或其用于创建 prepared statements 的兄弟例程）在先前尝试失败后再次尝试准备语句。

(516) SQLITE_ABORT_ROLLBACK

SQLITE_ABORT_ROLLBACK 错误码是 扩展错误码，用于表示 SQLITE_ABORT，说明某个 SQL 语句在启动时的事务被回滚，因此终止。当发生回滚时，所有未完成的写操作都会失败并返回此错误。只有在回滚的事务中更改了模式时，未完成的读操作才会因为回滚而失败。

(517) SQLITE_BUSY_SNAPSHOT

SQLITE_BUSY_SNAPSHOT 错误码是 扩展错误码，用于表示 SQLITE_BUSY，当数据库连接尝试将读取事务升级为写入事务但发现另一 数据库连接 已经写入数据库并使先前的读取失效时，在 WAL mode 数据库上会出现该错误。

以下场景说明 SQLITE_BUSY_SNAPSHOT 错误可能如何产生：

  1. 进程 A 在数据库上启动读取事务并执行一个或多个 SELECT 语句。进程 A 保持事务未提交。
  2. 进程 B 更新数据库，更改了进程 A 先前读取的值。
  3. 此时进程 A 尝试写入数据库。但由于进程 B 在进程 A 读取后修改了数据库文件，进程 A 当前所见的数据库内容已过时。因此进程 A 会收到 SQLITE_BUSY_SNAPSHOT 错误。

(518) SQLITE_LOCKED_VTAB

SQLITE_LOCKED_VTAB 结果码不是 SQLite 核心使用的，但可供扩展使用。虚拟表实现可以返回此结果码，以表明由于其他线程或进程持有的锁，当前操作无法完成。

R-Tree extension 在尝试更新 R-Tree 时，如果另一条 prepared statement 仍在读取 R-Tree，则会返回该结果码。更新无法继续，因为对 R-Tree 的任何更改可能涉及重新排序和重新平衡节点，这会干扰读取游标，使某些行重复而其他行被省略。

(520) SQLITE_READONLY_CANTLOCK

SQLITE_READONLY_CANTLOCK 错误码是 扩展错误码，用于表示 SQLITE_READONLY。该错误表示 SQLite 无法在 WAL mode 数据库上获取读取锁，因为与该数据库关联的共享内存文件是只读的。

(522) SQLITE_IOERR_SHORT_READ

SQLITE_IOERR_SHORT_READ 错误码是 扩展错误码，用于表示 SQLITE_IOERR，说明在 VFS 层的读取尝试中未能获取请求的字节数，可能是由于文件被截断所致。

(523) SQLITE_CORRUPT_SEQUENCE

SQLITE_CORRUPT_SEQUENCE 结果码表示 sqlite_sequence 表的模式已损坏。sqlite_sequence 表用于辅助实现 AUTOINCREMENT 功能。sqlite_sequence 表应具有如下格式： 

  CREATE TABLE sqlite_sequence(name,seq);

如果 SQLite 发现在 sqlite_sequence 表中存在其他格式，就会返回 SQLITE_CORRUPT_SEQUENCE 错误。

(526) SQLITE_CANTOPEN_ISDIR

SQLITE_CANTOPEN_ISDIR 错误码是 扩展错误码，用于表示 SQLITE_CANTOPEN，说明文件打开操作失败，因为该“文件”实际上是一个目录。

(531) SQLITE_CONSTRAINT_COMMITHOOK

SQLITE_CONSTRAINT_COMMITHOOK 错误码是 扩展错误码，用于表示 SQLITE_CONSTRAINT，说明某个 commit hook callback 返回非零值，因而导致该 SQL 语句被回滚。

(539) SQLITE_NOTICE_RECOVER_ROLLBACK

SQLITE_NOTICE_RECOVER_ROLLBACK 结果码在回滚 hot journal 时通过 sqlite3_log() 的回调传递。

(769) SQLITE_ERROR_SNAPSHOT

当使用 sqlite3_snapshot_open() 接口尝试在数据库的历史版本上启动读取事务时，可能会返回 SQLITE_ERROR_SNAPSHOT 结果码。如果该历史快照不再可用，读取事务就会因 SQLITE_ERROR_SNAPSHOT 而失败。只有在 SQLite 使用 -DSQLITE_ENABLE_SNAPSHOT 编译时，才可能出现此错误码。

(773) SQLITE_BUSY_TIMEOUT

SQLITE_BUSY_TIMEOUT 错误码表示 VFS 层中的阻塞 Posix Advisory 文件锁请求因超时而失败。阻塞的 Posix Advisory 锁仅作为 SQLite 的专有扩展提供，且仅在使用 SQLITE_ENABLE_SETLK_TIMEOUT 编译选项编译时才受支持。

(776) SQLITE_READONLY_ROLLBACK

SQLITE_READONLY_ROLLBACK 错误码是 扩展错误码，用于表示 SQLITE_READONLY。该错误表示由于数据库具有需要回滚但无法回滚的 hot journal，且数据库是只读的，因此无法打开数据库。

(778) SQLITE_IOERR_WRITE

SQLITE_IOERR_WRITE 错误码是 扩展错误码，用于表示 SQLITE_IOERR，说明在 VFS 层尝试向磁盘文件写入时发生 I/O 错误。该错误可能由硬件故障引起，或在文件打开期间文件系统被卸载所致。如果文件系统满了，本不应出现此错误，因为存在另一个专门用于该情况的错误码（SQLITE_FULL）。

(779) SQLITE_CORRUPT_INDEX

SQLITE_CORRUPT_INDEX 结果码表示 SQLite 检测到索引中缺少或曾缺少某个条目。这是 SQLITE_CORRUPT 错误码的一个特例，暗示问题可能通过执行 REINDEX 命令来解决，前提是数据库文件的其他部分没有额外问题。

(782) SQLITE_CANTOPEN_FULLPATH

SQLITE_CANTOPEN_FULLPATH 错误码是 扩展错误码，用于表示 SQLITE_CANTOPEN，说明文件打开操作失败，因为操作系统无法将文件名转换为完整路径名。

(787) SQLITE_CONSTRAINT_FOREIGNKEY

SQLITE_CONSTRAINT_FOREIGNKEY 错误码是 扩展错误码，用于表示 SQLITE_CONSTRAINT，说明某个 foreign key constraint 失败。

(1032) SQLITE_READONLY_DBMOVED

SQLITE_READONLY_DBMOVED 错误码是 扩展错误码，用于表示 SQLITE_READONLY。该错误表示由于数据库文件自打开以来已被移动，因此无法修改数据库；若在此情况下尝试修改数据库，一旦进程崩溃可能导致数据库损坏，因为 rollback journal 无法正确命名。

(1034) SQLITE_IOERR_FSYNC

SQLITE_IOERR_FSYNC 错误码是 扩展错误码，用于表示 SQLITE_IOERR，指出在 VFS 层尝试将已写入的内容从操作系统和/或磁盘控制缓冲区刷新到持久存储时发生了 I/O 错误。换句话说，该代码表示 unix 平台上的 fsync() 系统调用或 windows 平台上的 FlushFileBuffers() 系统调用出现问题。

(1043) SQLITE_CONSTRAINT_FUNCTION

SQLITE_CONSTRAINT_FUNCTION 错误码目前未被 SQLite 核心使用。 不过，该错误码可供扩展函数使用。

(1288) SQLITE_READONLY_CANTINIT

SQLITE_READONLY_CANTINIT 结果码源自 VFS 的 xShmMap 方法，用于表示 WAL mode 所使用的共享内存区域存在， 但其内容对当前进程而言不可靠且不可用，因为当前进程对该共享内存区域没有写权限。 （WAL 模式的共享内存区域通常是一个以 "-wal" 为后缀的文件，它会被映射进进程空间。 如果当前进程对该文件没有写权限，那么就无法写入共享内存。）

SQLite 的更高层逻辑通常会拦截该错误码并创建一个临时的内存共享内存区域， 这样当前进程至少可以读取数据库的内容。这个结果码不应传递到应用接口层。

(1290) SQLITE_IOERR_DIR_FSYNC

SQLITE_IOERR_DIR_FSYNC 错误码是 SQLITE_IOERR 的 extended error code， 表示在尝试对目录调用 fsync() 时的 VFS 层出现 I/O 错误。 Unix VFS 在创建或删除某些文件后会尝试对目录调用 fsync()， 以确保在断电或系统崩溃后这些文件仍然出现在文件系统中。 该错误码表示在尝试执行该 fsync() 时出现了问题。
