SQLite的版本号

1. SQLite版本号

从3.9.0版本（2015-10-14）开始，SQLite 使用 语义化版本控制。 在此之前，SQLite 使用包含两个到四个数字的版本标识。

1.1. 新的版本号系统（2015-10-14之后）

从3.9.0开始的所有SQLite发布都使用形如 X.Y.Z 的三位数 “语义版本”。 第一个数字 X 仅在出现破坏向后兼容性的更改时才会增加。 当前 X 的值是3，SQLite开发者计划支持 当前的SQLite数据库文件格式、SQL语法和C接口 直到至少2050年。 因此，人们可以预期未来几十年内的所有SQLite版本都将以“3.”开头。

第二个数字 Y 在添加新特性导致破坏向前兼容性时会递增。 预计大多数未来的SQLite版本都会 将第二个数字 Y 增加。每当 Y 增加时， Z 都会重置为零。

第三个数字 Z 用于仅包含小改动、实现性能优化和/或修复漏洞的版本发布。

过去五年（2010-2015年）的SQLite增强速度大约是每年6次 Y 递增。 用于SQLITE_VERSION_NUMBER和 sqlite3_libversion_number()的编号格式支持到3.999.999， 这已足够覆盖SQLite计划在2050年结束支持的日期。 不过，当前的压缩包命名约定只为 Y 留出两位， 因此下载命名格式需要在2030年前后进行调整。

1.2. 历史版本号系统（2015-10-14之前）

这个历史版本号系统使用两位、三位或四位版本号：W.X、W.X.Y 或 W.X.Y.Z。 W 是文件格式：1、2 或 3。 X 是主版本号。 Y 是次版本号。 Z 仅用于修复漏洞的补丁发布。

SQLite 共有三种历史文件格式。 SQLite 1.0 到 1.0.32 使用 gdbm 库作为存储引擎。 SQLite 2.0.0 到 2.8.17 使用自定义的B树存储引擎，仅支持文本键和值。 所有现代的SQLite版本（3.0.0至今）都使用支持二进制数据和Unicode的B树存储引擎。

这个主版本号 X 在历史上仅在代码发生重大且重要变更时才会递增。 对“重大且重要”的判断具有主观性。 从3.6.23到3.7.0的变化是由于添加了对WAL模式的支持。 从3.7.17到3.8.0的变化是代号为下一代查询规划器的重写。

历史上，次版本号 Y 在引入不显著改变代码结构的新特性和/或新接口时递增。 添加公共表表达式、部分索引和 表达式索引都是“次要”更改的例子。 再次强调，“主要”和“次要”的划分具有主观性。

历史上，补丁级别 Z 仅用于修复少量代码行的错误修复发布。
