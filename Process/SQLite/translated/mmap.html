内存映射I/O

SQLite访问和更新数据库磁盘文件的默认机制是 sqlite3_io_methods VFS 对象的 xRead() 和 xWrite() 方法。 这些方法通常实现为“read()”和“write()”系统调用，使操作系统能够在内核缓冲区缓存和用户空间之间复制磁盘内容。

从 版本 3.7.17（2013-05-20）开始，SQLite 提供了使用内存映射 I/O 直接访问磁盘内容的可选方案，以及 在 sqlite3_io_methods 上的新 xFetch() 和 xUnfetch() 方法。

使用内存映射 I/O 有其优点和缺点。优点包括：

  1. 许多操作，尤其是 I/O 密集型操作，可以更快，因为不需要在内核空间和用户空间之间复制内容。

  2. SQLite 库可能需要更少的 RAM，因为它与操作系统页面缓存共享页，并不总是需要自己维护工作页的副本。

但也有缺点：

  1. 内存映射文件上的 I/O 错误无法由 SQLite 捕获和处理。相反，该 I/O 错误会导致信号，如果应用未捕获该信号，则会导致程序崩溃。

  2. 操作系统必须具有统一缓冲区缓存，内存映射 I/O 扩展才能正确工作，特别是在两个进程访问同一个数据库文件的情况下，其中一个进程使用内存映射 I/O，而另一个没有。并非所有操作系统都有统一缓冲区缓存。在某些声称拥有统一缓冲区缓存的操作系统中，其实现存在缺陷，可能导致数据库损坏。

  3. 性能并不总是随着内存映射 I/O 而提升。事实上，可以构造出使用内存映射 I/O 后性能下降的测试用例。

  4. Windows 无法截断内存映射文件。因此，在 Windows 上，如果像 VACUUM 或 auto_vacuum 这样的操作尝试缩小内存映射数据库文件的大小，则缩小尝试会静默失败，留下文件末尾的未使用空间。此问题不会导致数据丢失，且当数据库再次增长时，未使用空间会被重新利用。但是，如果 3.7.0 之前的 SQLite 版本对这样的数据库运行 PRAGMA integrity_check，则会因为文件末尾的未使用空间而（错误地）报告数据库损坏。或者如果 3.7.0 之前的 SQLite 版本在数据库末尾仍有未使用空间的情况下写入数据，则可能使这部分未使用空间变得不可访问，并在下一次 VACUUM 之前无法复用。

由于存在潜在的缺点，内存映射 I/O 默认被禁用。要激活内存映射 I/O，可以使用 mmap_size pragma 并将 mmap_size 设置为某个较大的数值，通常至少为 256MB，具体取决于应用可以腾出的地址空间。剩下的全部由系统自动完成。 在不支持内存映射 I/O 的系统上，PRAGMA mmap_size 语句将默默地不执行任何操作。

内存映射I/O如何工作

使用传统的 xRead() 方法读取数据库页面时，SQLite 首先分配一个页面大小的堆内存块，然后调用 xRead() 方法将数据库页面内容复制到新分配的堆内存中。这至少涉及整个页面的复制。

但如果 SQLite 想访问数据库文件的某一页面，同时启用了内存映射 I/O，它会先调用 xFetch() 方法。 xFetch() 方法请求操作系统返回所需页面的指针（如果可能的话）。如果所需页面已经被映射到应用地址空间，或者可以被映射，那么 xFetch() 会返回该页面的指针，供 SQLite 使用，而无需进行任何复制。 正是省略复制步骤，使内存映射 I/O 更快。

SQLite 不假定 xFetch() 方法一定会成功。如果调用 xFetch() 返回空指针（表示所需页面当前未映射到应用地址空间），则 SQLite 会自动回退到使用 xRead()。只有在 xRead() 也失败时才会报告错误。

更新数据库文件时，SQLite 始终先将页面内容复制到堆内存，然后再修改该页面。这是必要的，原因有两点。首先，在事务提交之前，对数据库所做的更改不应对其他进程可见，因此更改必须发生在私有内存中。其次，SQLite 使用只读内存映射，以防止应用中的游离指针覆盖并损坏数据库文件。

完成所有必要更改后，会使用 xWrite() 将内容写回数据库文件。因此，使用内存映射 I/O 并不会显著改变数据库更改的性能。内存映射 I/O 对查询才主要有利。

配置内存映射I/O

“mmap_size”是 SQLite 在一次操作中尝试映射到进程地址空间的数据库文件的最大字节数。 mmap_size 对每个数据库文件分别应用，因此潜在使用的进程地址空间总量等于 mmap_size 乘以打开的数据库文件数。

要启用内存映射 I/O，应用可以将 mmap_size 设置为某个较大的值。例如：

PRAGMA mmap_size=268435456;

要禁用内存映射 I/O，只需将 mmap_size 设置为零：

PRAGMA mmap_size=0;

如果将 mmap_size 设置为 N，那么目前的所有实现都会映射数据库文件的前 N 个字节，并对 N 字节之外的内容使用传统的 xRead() 调用。如果数据库文件小于 N 字节，则整个文件都会被映射。未来新的操作系统接口理论上可以映射除前 N 个字节之外的文件区域，但目前还没有这样的实现。

mmap_size 是通过“PRAGMA mmap_size”语句分别为每个数据库文件设置的。通常默认的 mmap_size 为零，这意味着内存映射 I/O 默认被禁用。 但是，默认 mmap_size 可以在编译时通过 SQLITE_DEFAULT_MMAP_SIZE 宏 或在运行时通过 sqlite3_config(SQLITE_CONFIG_MMAP_SIZE,...) 接口增加。

SQLite 还对 mmap_size 维持一个硬性上限。尝试将 mmap_size 增加到该硬性上限以上（通过 PRAGMA mmap_size）会自动将 mmap_size 限制在硬性上限内。如果硬性上限为零，则无法实现内存映射 I/O。 硬性上限可以在编译时通过 SQLITE_MAX_MMAP_SIZE 宏设置。 如果将 SQLITE_MAX_MMAP_SIZE 设置为零，则实现内存映射 I/O 的代码将不会被编译进构建版本。在某些平台（例如 OpenBSD）上，由于缺乏统一缓冲区缓存，内存映射 I/O 无法工作，因此硬性上限会自动设置为零。

如果在编译时 mmap_size 的硬性上限非零，仍然可以在运行时通过 sqlite3_config(SQLITE_CONFIG_MMAP_SIZE,X,Y) 接口将其减少或置零。X 和 Y 参数必须都是 64 位有符号整数。X 参数是进程的默认 mmap_size，而 Y 是新的硬性上限。 硬性上限不能通过 SQLITE_CONFIG_MMAP_SIZE 增大到其编译时设置以上，但可以减少或设置为零。
