概述

SQL（在其所有实现中，而不仅仅是 SQLite）的最佳特性是它是一种声明式语言，而不是过程式语言。当用 SQL 编程时，你告诉系统要计算什么，而不是如何计算。确定如何的任务被委托给 SQL 数据库引擎内的查询规划器子系统。

对于任何给定的 SQL 语句，可能存在数百、数千甚至数百万种执行该操作的不同算法。所有这些算法都会得出正确答案，尽管有些比其他的更快。查询规划器是一个尝试为每个 SQL 语句选择最快、最有效算法的AI。

大多数情况下，SQLite 中的查询规划器表现良好。然而，查询规划器需要索引才能发挥作用。这些索引通常需要程序员来添加。偶尔，查询规划器 AI 会做出次优的算法选择。在这些情况下，程序员可能希望提供额外的提示，以帮助查询规划器做得更好。

本文档提供有关 SQLite 查询规划器和查询引擎工作方式的背景信息。程序员可以利用这些信息来帮助创建更好的索引，并在需要时为查询规划器提供提示。

更多信息请参阅SQLite 查询规划器和下一代查询规划器文档。

1. 搜索

1.1. 无索引表

大多数 SQLite 表由零条或多条带有唯一整数键（rowid 或 INTEGER PRIMARY KEY）的行组成，后面跟着内容。（例外是 WITHOUT ROWID 表。）这些行在逻辑上按 rowid 增量顺序存储。作为示例，本文使用一个名为 "FruitsForSale" 的表，它将各种水果与其生长的州以及市场上单位价格相关联。其模式如下： 

CREATE TABLE FruitsForSale(
  Fruit TEXT,
  State TEXT,
  Price REAL
);

如果填入一些（任意的）数据，这样的表在磁盘上的逻辑存储可能如图 1 所示：


图 1：表“FruitsForSale”的逻辑布局

在这个示例中，rowid 并不连续，但它们是有序的。SQLite 通常从 1 开始为 rowid 分配，并随着每次插入递增。但如果删除了行，序列中可能会出现间隙。而且应用程序可以在需要时控制分配的 rowid，使得行不一定插入在末尾。但无论发生什么，rowid 始终是唯一且严格递增的。

假设你想查找桃子的价格。查询如下： 

SELECT price FROM fruitsforsale WHERE fruit='Peach';

为了满足此查询，SQLite 会读取表中的每一行，检查 "fruit" 列是否为 "Peach"，如果是，就输出该行的 "price" 列。该过程如下方的 图 2 所示。这个算法被称为全表扫描，因为必须读取并检查整个表的内容才能找到那一行感兴趣的记录。对于只有 7 行的表来说，全表扫描是可以接受的，但如果表包含 700 万行，全表扫描可能需要读取数兆字节的数据才能找到一个 8 字节的数字。因此，通常尽量避免全表扫描。


图 2：全表扫描

1.2. 通过 rowid 查找

避免全表扫描的一种技巧是按 rowid（或等价的 INTEGER PRIMARY KEY）进行查找。要查找桃子的价格，可以查询 rowid 为 4 的条目： 

SELECT price FROM fruitsforsale WHERE rowid=4;

由于信息按 rowid 顺序存储在表中，SQLite 可以使用二分查找来找到正确的行。如果表中包含 N 个元素，查找所需时间与 logN 成正比，而不是像全表扫描那样与 N 成正比。如果表包含 1000 万个元素，那么该查询大约会快 N/logN，也就是快约 100 万倍。


图 3：按 rowid 查找

1.3. 通过索引查找

按 rowid 查找的问题是，你可能并不关心“第 4 项”的价格——你是想知道桃子的价格。因此，rowid 查找并没有帮助。

为了让原始查询更高效，我们可以在 "fruitsforsale" 表的 "fruit" 列上添加一个索引，如下所示： 

CREATE INDEX Idx1 ON fruitsforsale(fruit);

索引是一个与原始 "fruitsforsale" 表类似的表，但内容（本例中是 fruit 列）被存储在 rowid 之前，并且所有行按内容顺序排列。图 4 展示了 Idx1 索引的逻辑视图。 "fruit" 列是用于排序元素的主键，而 "rowid" 是在两个或多个行具有相同 "fruit" 时用于打破平局的次键。在示例中，对于 "Orange" 行，必须使用 rowid 来打破平局。由于 rowid 在原始表的所有元素中始终唯一，因此由 "fruit" 加上 "rowid" 构成的组合键在索引中也会对所有元素唯一。


图 4：fruit 列上的索引

这个新索引可以用来实现原始“桃子价格”查询的更快算法。 

SELECT price FROM fruitsforsale WHERE fruit='Peach';

查询首先在 Idx1 索引上进行二分查找，寻找 fruit='Peach' 的条目。SQLite 能在 Idx1 上进行二分查找，而无法在原始 FruitsForSale 表中做到这一点，因为 Idx1 中的行是按 "fruit" 列排序的。找到 Idx1 中 fruit='Peach' 的行后，数据库引擎可以提取该行的 rowid。然后数据库引擎会在原始 FruitsForSale 表上执行第二次二分查找，以找到包含 fruit='Peach' 的原始行。最后，SQLite 从 FruitsForSale 表中的那一行提取 price 列的值。该过程如 图 5 所示。


图 5：使用索引查找桃子价格

使用上述方法查找桃子价格时，SQLite 必须进行两次二分查找。但对于包含大量行的表而言，仍远比执行全表扫描要快。

1.4. 多个结果行

在前面的查询中，fruit='Peach' 的限制将结果缩小到了单行。但相同的技术即使在获取多个行时也适用。假设我们查询的是橙子的价格： 

SELECT price FROM fruitsforsale WHERE fruit='Orange'


图 6：使用索引查找橙子价格

在这种情况下，SQLite 仍只需进行一次二分查找即可找到索引中第一个 fruit='Orange' 的条目。然后，它从索引中提取 rowid，并使用该 rowid 通过二分查找在原始表中查找该行并输出价格。但它并不会就此结束，而是继续前进到索引的下一行，对下一个 fruit='Orange' 项重复该过程。前进到索引（或表）的下一行所需的成本远低于执行二分查找，因为下一行通常位于与当前行相同的数据库页中。事实上，与二分查找相比，前进到下一行的成本如此之低，以至于我们通常可以忽略不计。因此，我们对该查询的总成本估算为 3 次二分查找。如果输出行数为 K，表中行数为 N，则该查询的成本通常与 \( (K+1)\log N \) 成正比。

1.5. WHERE 子句中多个由 AND 连接的项

接下来，假设你想查找的不仅仅是任何橙子的价格，而是具体在加州生长的橙子的价格。相应的查询如下： 

SELECT price FROM fruitsforsale WHERE fruit='Orange' AND state='CA'


图 7：加州橙子的索引查找

一种处理此查询的方法是先利用 WHERE 子句中的 fruit='Orange' 项找到所有与橙子相关的行，然后再过滤掉来自加州以外州的行。该过程如上面的 图 7 所示。在大多数情况下，这是一个完全合理的方法。是的，数据库引擎确实需要对后来被拒绝的佛罗里达橙子行多做一次二分查找，所以效率没有我们预期的那么高，尽管对于很多应用来说已经足够高效。

假设除了 "fruit" 上的索引之外，还有一个 "state" 上的索引。 

CREATE INDEX Idx2 ON fruitsforsale(state);


图 8：state 列上的索引

"state" 索引的工作方式与 "fruit" 索引类似：它是在 rowid 前添加一列的新表，并以该额外列作为主键排序。唯一的不同在于，在 Idx2 中第一列是 "state"，而 Idx1 中是 "fruit"。在我们的示例数据集中，"state" 列中有更多冗余，因此重复条目也更多。平局仍然由 rowid 解决。

使用 Idx2 索引而非 Idx1，SQLite 有了另一种查找加州橙子价格的选项：它可以查找所有来自加州的行，然后过滤掉那些不是橙子的。


图 9：加州橙子的索引查找

使用 Idx2 替代 Idx1 会让 SQLite 检查不同的行集，但最终得到的答案相同（这非常重要——请记住，索引不应改变答案，仅帮助 SQLite 更快地得到答案），它执行的工作量也相同。因此在这个例子中，Idx2 索引并未提升性能。

在我们的示例中，最后两个查询所需时间相同。那么 SQLite 会选择 Idx1 还是 Idx2？如果已经对数据库运行了 ANALYZE 命令，从而让 SQLite 有机会收集可用索引的统计信息，那么 SQLite 就会知道 Idx1 索引通常会将搜索缩小到单个项目（我们关于 fruit='Orange' 的示例是该规则的例外），而 Idx2 通常只会将搜索缩小到两行左右。因此，如果其他条件相同，SQLite 会选择 Idx1，以期将搜索的行数尽可能缩小。这种选择只有在提供了 ANALYZE 的统计信息后才有可能。如果未执行 ANALYZE，则选择使用哪个索引就是任意的。

1.6. 多列索引

为了从 WHERE 子句中多个由 AND 连接的项中获得最大性能，确实需要为每个 AND 项创建多列索引。在这种情况下，我们在 FruitsForSale 的 "fruit" 和 "state" 列上创建一个新的索引： 

CREATE INDEX Idx3 ON FruitsForSale(fruit, state);


图 1：两列索引

多列索引遵循与单列索引相同的模式；被索引的列被添加在 rowid 前面。唯一的区别在于现在添加了多列。最左侧的列是用于为索引中的行排序的主键。第二列用于打破第一列的平局。如果存在第三列，它就用于打破前两列的平局。依此类推，直到索引中的所有列。因为 rowid 保证唯一，即使两行的所有内容列都相同，索引中的每一行仍然是唯一的。我们的示例数据中没有这种情况，但确实存在一个案例（fruit='Orange'）在第一列上有平局，必须由第二列来打破。

有了新的多列 Idx3 索引，SQLite 现在可以仅用两次二分查找就找到加州橙子的价格： 

SELECT price FROM fruitsforsale WHERE fruit='Orange' AND state='CA'


图 11：使用两列索引的查找

由于 Idx3 索引包含 WHERE 子句约束的两个列，SQLite 可以对 Idx3 进行一次二分查找以找到加州橙子的 rowid，然后再进行一次二分查找以在原始表中找到该项的价格。没有死胡同，也没有浪费的二分查找。这是一个更高效的查询。

请注意，Idx3 包含与原始 Idx1 相同的信息。因此如果我们拥有 Idx3，就不再真的需要 Idx1。只需忽略 Idx3 中的 "state" 列，就可以使用 Idx3 满足“桃子价格”查询： 

SELECT price FROM fruitsforsale WHERE fruit='Peach'


图 12：在多列索引上执行单列查找

因此，一个经验法则是，数据库模式中不应存在一个索引是另一个索引前缀的情况。删除列数更少的索引。SQLite 仍然可以使用更长的索引执行高效查找。

1.7. 覆盖索引

通过使用两列索引，使“加州橙子价格”查询更高效。但 SQLite 可以借助一个同时包含 "price" 列的三列索引做得更好： 

CREATE INDEX Idx4 ON FruitsForSale(fruit, state, price);


图 13：覆盖索引

这个新索引包含了原始 FruitsForSale 表中查询所使用的所有列——既包括搜索项，也包括输出项。我们称之为“覆盖索引”。由于所有需要的信息都在覆盖索引中，SQLite 永远不需要查阅原始表来找到价格。 

SELECT price FROM fruitsforsale WHERE fruit='Orange' AND state='CA';


图 14：使用覆盖索引的查询

因此，通过在索引末尾添加额外的“输出”列，可以避免引用原始表，从而将查询所需的二分查找次数减半。这在性能上是一个常数因子的改进（大致相当于速度翻倍）。但另一方面，这也只是一个细微改进；相对于表首次建立索引时所见的百万倍的提升，翻倍的性能提升并不那么戏剧化。而对于大多数查询而言，1 微秒和 2 微秒之间的差异也不太可能被察觉。

1.8. WHERE 子句中由 OR 连接的项

多列索引只有在查询的 WHERE 子句中的约束项通过 AND 连接时才有效。因此，Idx3 和 Idx4 对于同时满足“是橙子”并且“在加州生长”的项目有帮助，但如果我们想要的是所有橙子或在加州生长的项目，那么这两个索引就不那么有用了。 

SELECT price FROM FruitsForSale WHERE fruit='Orange' OR state='CA';

当遇到 WHERE 子句中的 OR 连接项时，SQLite 会分别检查每个 OR 项，并尝试使用索引找出与每个项相关的 rowid。然后，它将所得 rowid 集合取并集以得到最终结果。下图展示了该过程：


图 15：包含 OR 约束的查询

上图暗示 SQLite 会先计算所有 rowid，然后再用并集操作将它们合并，最后才开始对原始表进行 rowid 查找。实际上，rowid 查找会与 rowid 计算交错进行。SQLite 每次只使用一个索引来查找 rowid，同时记住先前已看到的 rowid 以避免重复。这只是实现细节。尽管该图并不完全精确，但它很好地概括了正在发生的事情。

为了让上述 OR-按并集 技术有用，必须为 WHERE 子句中的每个 OR 连接项都提供可帮助解析的索引。如果哪怕一个 OR 项没有索引，就必须执行全表扫描来查找该项生成的 rowid；而如果 SQLite 必须执行全表扫描，不如直接在原始表上一次性获取所有结果，省去并集操作和后续的二分查找。

你可以看到，OR-按并集 技术也可以用于 WHERE 子句中由 AND 连接的项，通过使用交集操作代替并集。许多 SQL 数据库引擎会这样做。不过，与仅使用单个索引相比，其性能提升微乎其微，因此 SQLite 目前尚未实现这项技术。然而，未来版本的 SQLite 可能会增强以支持 AND-按交集。

2. 排序

SQLite（如所有其他 SQL 数据库引擎一样）也可以使用索引来满足查询中的 ORDER BY 子句，除了加速查找之外。换言之，索引可以同时用于加速排序与搜索。

当没有合适的索引可用时，带 ORDER BY 子句的查询必须作为一个单独的步骤来排序。考虑以下查询： 

SELECT * FROM fruitsforsale ORDER BY fruit;

SQLite 会收集该查询的所有输出，然后将其送入排序器处理。


图 16：无索引排序

如果输出行数为 K，那么排序所需时间与 \( K\log K \) 成正比。如果 K 较小，排序时间通常不会成为瓶颈；但在上述这种 K==N 的查询中，排序所需时间可能远大于执行全表扫描所需的时间。此外，所有输出都要累积在临时存储中（根据不同的编译时与运行时设置，这些临时存储可能位于主存或磁盘上），这意味着完成查询可能需要大量临时存储。

2.1. 按 rowid 排序

由于排序可能成本较高，SQLite 会尽力将 ORDER BY 子句变成无操作。如果 SQLite 判断输出自然会按照指定顺序出现，那么就不会执行排序。因此，例如，如果你请求按照 rowid 顺序输出，那么就不会进行排序： 

SELECT * FROM fruitsforsale ORDER BY rowid;


图 17：按 rowid 排序

你也可以像下面这样请求倒序排序： 

SELECT * FROM fruitsforsale ORDER BY rowid DESC;

SQLite 仍会省略排序步骤。但为了让输出按正确顺序出现，SQLite 会从表尾开始扫描并向前推进，而不是像 图 17 那样从前向后扫描。

2.2. 按索引排序

当然，仅按 rowid 排序很少有用。通常人们希望按照其他某一列排序输出。

如果 ORDER BY 所使用的列上存在索引，则可以用该索引来排序。考虑按 "fruit" 排序全部项目的请求： 

SELECT * FROM fruitsforsale ORDER BY fruit;


图 18：使用索引排序

Idx1 索引从上到下（如果使用 "ORDER BY fruit DESC" 则从下到上）扫描，以按 fruit 顺序找到每个项目的 rowid。然后对于每个 rowid，都执行一次二分查找来查找并输出该行。通过这种方式，输出以请求的顺序出现，而不需要先收集全部输出再用单独的步骤排序。

但这真的节省时间吗？原始无索引排序 中的步骤数量与 NlogN 成正比，因为这就是排序 N 行所需的时间。而当我们像这里那样使用 Idx1 时，我们必须做 N 次 rowid 查找，每次花费 logN，因此总时间仍然是 NlogN，两者相同！

SQLite 使用基于成本的查询规划器。当存在两种或更多种方式解决同一查询时，SQLite 会尝试估算每种执行方案所需的总时间，然后选择估算成本最低的方案。成本主要由估算时间组成，因此该情形可能会根据表大小、WHERE 子句的约束条件等而有所不同。但一般而言，索引排序可能更常被选中，至少是因为它无需先将整个结果集累积到临时存储再排序，因此使用的临时存储要少得多。

2.3. 按覆盖索引排序

如果查询可以使用覆盖索引，那么就可以避免多次 rowid 查找，从而显著降低查询成本。


图 19：使用覆盖索引排序

有了覆盖索引，SQLite 只需从索引的一端走到另一端，就能按与 N 成正比的时间返回输出，而无需分配大的缓冲区来保存结果集。

3. 同时搜索和排序

前面的讨论把搜索和排序当作两个独立的话题。但在实践中，常常需要同时进行搜索与排序。幸运的是，可以使用单个索引来实现这一点。

3.1. 使用多列索引同时搜索和排序

假设我们想找出各种橙子的价格，并按它们生长的州来排序。查询如下： 

SELECT price FROM fruitforsale WHERE fruit='Orange' ORDER BY state

该查询在 WHERE 子句中包含了搜索限制，在 ORDER BY 子句中包含了排序顺序。可以使用两列索引 Idx3 同时完成搜索与排序。


图 20：使用多列索引搜索并排序

查询在索引上做二分查找，找出 fruit='Orange' 的那部分行。（由于 fruit 列是索引中最左边的列，且索引行按该列排序，因此所有此类行都相邻。）然后，它从上到下扫描匹配的索引行以获取原始表的 rowid，并对每个 rowid 在原始表中执行二分查找以找到价格。

你会注意到上图中没有任何“排序”模块。查询的 ORDER BY 子句已经变成了无操作。这里不需要进行排序，因为输出顺序就是按 state 列，而 state 列恰好是索引中 fruit 列右侧的第一列。因此，如果从上到下扫描具有相同 fruit 值的索引条目，那么这些索引条目就一定是按 state 排序的。

3.2. 使用覆盖索引同时搜索和排序

覆盖索引同样可以用于同时搜索和排序。考虑以下查询： 

SELECT * FROM fruitforsale WHERE fruit='Orange' ORDER BY state


图 21：使用覆盖索引搜索并排序

与之前一样，SQLite 会对 满足 WHERE 子句 的覆盖索引中的行范围进行一次二分查找，然后从上到下扫描该范围以获取所需结果。由于 WHERE 子句在索引的最左列上是等值约束，满足条件的行一定相邻。通过从上到下扫描这些匹配的索引行，输出就保证按照 state 排列，因为 state 列紧随 fruit 列。因此，该查询非常高效。

SQLite 也可以对降序的 ORDER BY 使用类似技巧： 

SELECT * FROM fruitforsale WHERE fruit='Orange' ORDER BY state DESC

基本算法相同，只是这次匹配的索引行是从下到上扫描，而不是从上到下，以便使州按降序出现。

3.3. 使用索引进行部分排序（又名块排序）

有时候 ORDER BY 子句只有部分可以通过索引满足。例如，考虑以下查询： 

SELECT * FROM fruitforsale ORDER BY fruit, price

如果使用覆盖索引做扫描，"fruit" 列会自然地按正确顺序出现，但当多个行具有相同 fruit 时，price 可能会变得无序。发生这种情况时，SQLite 会进行许多小排序——每个不同的 fruit 值分别排序，而不是一次对整个集合排序。下图 22 展示了这个概念。


图 22：使用索引进行部分排序

在该示例中，并不是对 7 个元素做一次排序，而是对 5 个元素分别做一次一元素排序，另外对 fruit=='Orange' 的 2 个元素做一次排序。

进行多个较小的排序而非一次大型排序的好处包括：

  1. 多个小排序总体上使用的 CPU 周期比单个大排序少。
  2. 每个小排序独立运行，这意味着在任意时刻需要保留的临时存储信息要少得多。
  3. 由于索引已经使 ORDER BY 的某些列保持正确顺序，因此可以将它们从排序键中省略，从而进一步减少存储需求和 CPU 时间。
  4. 每个小排序完成后就能将输出行返回给应用程序，而不必等待整个表扫描结束。
  5. 如果存在 LIMIT 子句，可能就可以避免扫描整个表。

正因为这些优势，即使无法通过索引完全排序，SQLite 也总是尝试进行部分排序。

4. WITHOUT ROWID 表

上述基本原理适用于普通 rowid 表和WITHOUT ROWID 表。唯一的区别是，作为表键并出现在索引最右侧项的 rowid 列被 PRIMARY KEY 所替代。
