使用 sqlite3_unlock_notify() API

/* This example uses the pthreads API */
#include <pthread.h>

/*
** A pointer to an instance of this structure is passed as the user-context
** pointer when registering for an unlock-notify callback.
*/
typedef struct UnlockNotification UnlockNotification;
struct UnlockNotification {
  int fired;                         /* True after unlock event has occurred */
  pthread_cond_t cond;               /* Condition variable to wait on */
  pthread_mutex_t mutex;             /* Mutex to protect structure */
};

/*
** This function is an unlock-notify callback registered with SQLite.
*/
static void unlock_notify_cb(void **apArg, int nArg){
  int i;
  for(i=0; i<nArg; i++){
    UnlockNotification *p = (UnlockNotification *)apArg[i];
    pthread_mutex_lock(&p->mutex);
    p->fired = 1;
    pthread_cond_signal(&p->cond);
    pthread_mutex_unlock(&p->mutex);
  }
}

/*
** This function assumes that an SQLite API call (either sqlite3_prepare_v2() 
** or sqlite3_step()) has just returned SQLITE_LOCKED. The argument is the
** associated database connection.
**
** This function calls sqlite3_unlock_notify() to register for an 
** unlock-notify callback, then blocks until that callback is delivered 
** and returns SQLITE_OK. The caller should then retry the failed operation.
**
** Or, if sqlite3_unlock_notify() indicates that to block would deadlock 
** the system, then this function returns SQLITE_LOCKED immediately. In 
** this case the caller should not retry the operation and should roll 
** back the current transaction (if any).
*/
static int wait_for_unlock_notify(sqlite3 *db){
  int rc;
  UnlockNotification un;

  /* Initialize the UnlockNotification structure. */
  un.fired = 0;
  pthread_mutex_init(&un.mutex, 0);
  pthread_cond_init(&un.cond, 0);

  /* Register for an unlock-notify callback. */
  rc = sqlite3_unlock_notify(db, unlock_notify_cb, (void *)&un);
  assert( rc==SQLITE_LOCKED || rc==SQLITE_OK );

  /* The call to sqlite3_unlock_notify() always returns either SQLITE_LOCKED 
  ** or SQLITE_OK. 
  **
  ** If SQLITE_LOCKED was returned, then the system is deadlocked. In this
  ** case this function needs to return SQLITE_LOCKED to the caller so 
  ** that the current transaction can be rolled back. Otherwise, block
  ** until the unlock-notify callback is invoked, then return SQLITE_OK.
  */
  if( rc==SQLITE_OK ){
    pthread_mutex_lock(&un.mutex);
    if( !un.fired ){
      pthread_cond_wait(&un.cond, &un.mutex);
    }
    pthread_mutex_unlock(&un.mutex);
  }

  /* Destroy the mutex and condition variables. */
  pthread_cond_destroy(&un.cond);
  pthread_mutex_destroy(&un.mutex);

  return rc;
}

/*
** This function is a wrapper around the SQLite function sqlite3_step().
** It functions in the same way as step(), except that if a required
** shared-cache lock cannot be obtained, this function may block waiting for
** the lock to become available. In this scenario the normal API step()
** function always returns SQLITE_LOCKED.
**
** If this function returns SQLITE_LOCKED, the caller should rollback
** the current transaction (if any) and try again later. Otherwise, the
** system may become deadlocked.
*/
int sqlite3_blocking_step(sqlite3_stmt *pStmt){
  int rc;
  while( SQLITE_LOCKED==(rc = sqlite3_step(pStmt)) ){
    rc = wait_for_unlock_notify(sqlite3_db_handle(pStmt));
    if( rc!=SQLITE_OK ) break;
    sqlite3_reset(pStmt);
  }
  return rc;
}

/*
** This function is a wrapper around the SQLite function sqlite3_prepare_v2().
** It functions in the same way as prepare_v2(), except that if a required
** shared-cache lock cannot be obtained, this function may block waiting for
** the lock to become available. In this scenario the normal API prepare_v2()
** function always returns SQLITE_LOCKED.
**
** If this function returns SQLITE_LOCKED, the caller should rollback
** the current transaction (if any) and try again later. Otherwise, the
** system may become deadlocked.
*/
int sqlite3_blocking_prepare_v2(
  sqlite3 *db,              /* Database handle. */
  const char *zSql,         /* UTF-8 encoded SQL statement. */
  int nSql,                 /* Length of zSql in bytes. */
  sqlite3_stmt **ppStmt,    /* OUT: A pointer to the prepared statement */
  const char **pz           /* OUT: End of parsed string */
){
  int rc;
  while( SQLITE_LOCKED==(rc = sqlite3_prepare_v2(db, zSql, nSql, ppStmt, pz)) ){
    rc = wait_for_unlock_notify(db);
    if( rc!=SQLITE_OK ) break;
  }
  return rc;
}

当两个或更多连接在共享缓存模式下访问同一个数据库时，会在各个表上使用读写（共享与独占）锁，以确保并发执行的事务保持隔离。在写入表之前必须先获取写（独占）锁；在读取之前则必须先获取读（共享）锁。连接在事务结束时会释放其持有的所有表锁。如果某个连接无法获取所需的锁，那么对 sqlite3_step() 的调用会返回 SQLITE_LOCKED。

虽然不太常见，但对 sqlite3_prepare() 或 sqlite3_prepare_v2() 的调用如果无法在每个附加数据库的 sqlite_schema 表 上获取读锁，同样也可能返回 SQLITE_LOCKED。这些 API 需要读取 sqlite_schema 表中包含的架构数据，以便将 SQL 语句编译成 sqlite3_stmt* 对象。

本文介绍了一种利用 SQLite sqlite3_unlock_notify() 接口的技术，使得对 sqlite3_step() 和 sqlite3_prepare_v2() 的调用在所需锁可用之前会阻塞，而不是立即返回 SQLITE_LOCKED。如果右侧所示的 sqlite3_blocking_step() 或 sqlite3_blocking_prepare_v2() 函数返回 SQLITE_LOCKED，则说明阻塞将使系统死锁。

sqlite3_unlock_notify() API 仅在以预处理器符号 SQLITE_ENABLE_UNLOCK_NOTIFY 编译库时可用，具体文档可在 此处 查阅。本文不能替代完整的 API 文档阅读！

sqlite3_unlock_notify() 接口是为每个 数据库连接 分配独立线程的系统设计的。实现中并没有阻止单个线程运行多个数据库连接。然而，sqlite3_unlock_notify() 接口一次只能对一个连接起作用，因此这里所描述的锁解决逻辑仅对每个线程一个数据库连接有效。

sqlite3_unlock_notify() API

在调用 sqlite3_step() 或 sqlite3_prepare_v2() 返回 SQLITE_LOCKED 之后，可以调用 sqlite3_unlock_notify() 来注册 unlock-notify 回调。unlock-notify 回调是在阻塞此次 sqlite3_step() 或 sqlite3_prepare_v2() 的连接在完成事务并释放所有锁之后由 SQLite 触发的。例如，如果一次 sqlite3_step() 尝试从表 X 中读取数据，而另一个连接 Y 对表 X 持有写锁，则 sqlite3_step() 会返回 SQLITE_LOCKED。此时如果调用 sqlite3_unlock_notify()，当连接 Y 的事务结束后，unlock-notify 回调就会被触发。此时回调正在等待的连接（在该例中为连接 Y）即被称为“阻塞连接”。

如果尝试写入数据库表的 sqlite3_step() 返回 SQLITE_LOCKED，说明可能有多个其他连接对该表持有读锁。在这种情况下，SQLite 会任意选择其中一个连接，并在该连接的事务完成时发出 unlock-notify 回调。无论是被一个还是多个连接阻塞，在对应的 unlock-notify 回调发出时，并不能保证所需锁已可用，只能确定它有可能可用。

unlock-notify 回调是在阻塞连接正在执行的 sqlite3_step()（或 sqlite3_close()）调用内部发出的。在 unlock-notify 回调内调用任何 sqlite3_XXX() API 函数都是非法的。典型的使用方式是，unlock-notify 回调会通知其他正在等待的线程或安排后续操作。

sqlite3_blocking_step() 函数采用以下算法：

  1. 对提供的语句句柄调用 sqlite3_step()。如果返回值不是 SQLITE_LOCKED，则直接返回该值给调用者；否则继续下一步。

  2. 对与语句句柄关联的数据库连接调用 sqlite3_unlock_notify() 并注册 unlock-notify 回调。如果 unlock_notify() 返回 SQLITE_LOCKED，那么立即将其返回给调用者。

  3. 阻塞，直到其他线程触发 unlock-notify 回调。

  4. 对语句句柄调用 sqlite3_reset()。由于 SQLITE_LOCKED 错误只能在第一次调用 sqlite3_step() 时发生（无法出现先返回 SQLITE_ROW 然后再返回 SQLITE_LOCKED 的情形），因此此时可以重新设置语句句柄，而不会影响调用者视角下查询的结果。如果此时不调用 sqlite3_reset()，下一次调用 sqlite3_step() 将返回 SQLITE_MISUSE。

  5. 返回第 1 步。

sqlite3_blocking_prepare_v2() 函数使用的算法与此类似，区别在于省略了第 4 步（重置语句句柄）。

写入者饥饿

多个连接可以同时持有读锁。如果多个线程不断获取重叠的读锁，可能始终至少有一个线程持有读锁。这会导致等待写锁的表永远无法获得写锁。这种情况称为“写入者饥饿”。

SQLite 帮助应用避免写入者饥饿。在任何尝试获取某个表的写锁失败（因为一个或多个其他连接持有读锁）之后，在下述任一条件成立之前，所有尝试在共享缓存上打开新事务的行为都会失败：

  • 当前写入者完成其事务，或者
  • 共享缓存上的打开读事务数量降为零。

无法打开新读事务的尝试会向调用者返回 SQLITE_LOCKED。如果调用者随后调用 sqlite3_unlock_notify() 来注册 unlock-notify 回调，则阻塞连接就是当前在共享缓存上持有写事务的连接。这就防止了写入者饥饿，因为在不允许打开新读事务且所有现存读事务最终完成的前提下，写入者最终会获得所需的写锁。

pthreads API

当 wait_for_unlock_notify() 调用 sqlite3_unlock_notify() 时，被阻塞的 sqlite3_step() 或 sqlite3_prepare_v2() 所依赖的阻塞连接可能早已完成其事务。在这种情况下，unlock-notify 回调会被立即触发，甚至在 sqlite3_unlock_notify() 返回之前。也可能是在尚未开始等待异步信号前，另一个线程在 sqlite3_unlock_notify() 调用之后触发了回调。

如何处理这种潜在的竞态条件，取决于应用所使用的线程与同步原语接口。本文的示例使用 pthreads，即现代类 UNIX 系统（包括 Linux）提供的接口。

pthreads 接口提供了 pthread_cond_wait() 函数。该函数可让调用者同时释放互斥锁并开始等待异步信号。结合该函数、一个“fired”标志以及互斥锁，可以按如下方式消除上述竞态条件：

当 unlock-notify 回调被触发时（可能发生在调用 sqlite3_unlock_notify() 的线程开始等待异步信号之前），它会：

  1. 获取互斥锁。
  2. 将“fired”标志设为真。
  3. 尝试通知正在等待的线程。
  4. 释放互斥锁。

当 wait_for_unlock_notify() 线程准备开始等待 unlock-notify 回调时，它会：

  1. 获取互斥锁。
  2. 检查“fired”标志是否已被设置。如果是，则说明 unlock-notify 回调已经执行。释放互斥锁并继续执行。
  3. 原子地释放互斥锁并开始等待异步信号。信号到来后继续执行。

这样一来，无论 unlock-notify 回调是在 wait_for_unlock_notify() 线程开始阻塞之前、之后还是正在执行，都无关紧要。

可能的改进

本文中的代码至少可以在两个方面进行改进：

  • 它可以管理线程优先级。
  • 它可以处理在删除表或索引时可能出现的 SQLITE_LOCKED 的特殊情况。

虽然 sqlite3_unlock_notify() 函数一次只能让调用者指定一个用户上下文指针，但 unlock-notify 回调会得到这些上下文指针组成的数组。这是因为当一个阻塞连接完成事务时，如果多个 unlock-notify 已注册用来调用同一个 C 函数，这些上下文指针就会收集成数组，并触发一次回调。如果每个线程都有优先级，那么可以在不像本文实现那样任意唤醒线程的情况下，先唤醒高优先级线程。

如果执行一次“DROP TABLE”或“DROP INDEX”SQL 命令，而当前连接正在执行一个或多个 SELECT 语句，那么会返回 SQLITE_LOCKED。在这种情况下，如果调用 sqlite3_unlock_notify()，则指定的回调会立即执行。再次尝试“DROP TABLE”或“DROP INDEX”语句会再次返回 SQLITE_LOCKED。在右侧所示的 sqlite3_blocking_step() 实现中，这可能导致无限循环。

调用者可以通过使用 扩展错误代码 来区分这种特殊的“DROP TABLE|INDEX”情况和其他情况。当适合调用 sqlite3_unlock_notify() 时，扩展错误代码为 SQLITE_LOCKED_SHAREDCACHE；在“DROP TABLE|INDEX”情况下，则仅为 SQLITE_LOCKED。另一种解决方案是限制任何单个查询的重试次数（比如上限为 100 次）。虽然效率可能不如理想，但这种情况不太可能频繁发生。
