错误和警告日志

概述

SQLite 可以配置为在出现异常时调用包含错误代码和简短错误信息的回调函数。该机制对于追踪罕见且在现场才会出现的隐蔽问题非常有帮助。我们鼓励应用开发者在产品中利用 SQLite 的错误记录功能，因为它占用极少的 CPU 和内存，但对于调试却有极大帮助。

1. 设置错误日志回调

每个进程只能有一个错误日志回调。该回调在启动时通过下面类似的 C 代码进行注册：

sqlite3_config(SQLITE_CONFIG_LOG, errorLogCallback, pData);

错误记录器回调函数可能如下所示：

void errorLogCallback(void *pArg, int iErrCode, const char *zMsg){
  fprintf(stderr, "(%d) %s\n", iErrCode, zMsg);
}

上例展示了错误记录器回调的签名。然而，在嵌入式应用中通常不会在 stderr 上打印消息。取而代之的是，可能将消息存储在预分配的循环缓冲区中，在调试时需要诊断信息时再访问。或者也可以将消息发送到 Syslog。无论如何，消息需要存储在开发者能够访问的地方，而不是展示给最终用户。

不要误解：将错误记录器消息展示给最终用户在技术上并无问题。消息不包含需要保护免受未经授权查看的敏感或私密信息。只是这些消息技术性较强，对普通最终用户来说没有用处也不具意义。错误记录器输出的信息是面向数据库专家的，请据此进行显示。

2. 接口细节

sqlite3_config(SQLITE_CONFIG_LOG,...) 接口的第三个参数（上例中的“pData”参数）是一个指向任意数据的指针。SQLite 会将此指针传递给错误记录器回调的第一个参数。如果需要，该指针可以用于传递应用特定的设置或状态信息。也可以仅仅传递一个 NULL 指针，在回调中忽略。

错误记录器回调的第二个参数是一个整数的扩展错误代码。第三个参数是错误消息的文本。该错误消息文本存储在调用函数的固定长度栈缓冲区中，因此只在错误记录器回调函数的持续期间有效。如果需要保留该消息，错误记录器应该将其复制到持久存储中。

错误记录器回调应该像信号处理程序一样对待。应用应该保存或处理错误，然后尽快返回。错误记录器中不应直接或间接调用其他 SQLite API。通过错误记录器回调进行的操作并非 SQLite 可重入的。特别地，错误记录器在内存分配失败时会被调用，因此在错误记录器内部尝试分配内存通常不是好主意。甚至不应考虑尝试将错误消息存储到另一个 SQLite 数据库中。

如果需要，可以使用 sqlite3_log(E,F,..) API 向日志发送新消息，但这种做法不推荐。sqlite3_log() 接口是为扩展设计的，不是为应用程序设计的。

3. 各种错误消息

可能发送到错误记录器的错误消息及其具体格式会在各个版本之间发生变化。因此，应用程序不应依赖任何特定的错误消息文本格式或错误代码。虽然这些变化不会任意发生，但有时确实会改变。

下面是可能出现在错误记录器回调中的消息类型的部分列表。

  • 只要在编译 SQL 语句（使用 sqlite3_prepare_v2() 或其兄弟接口）或运行 SQL 语句（使用 sqlite3_step()）时出现错误，该错误都会被记录。

  • 当发生需要重新解析并重新准备准备好的语句的模式更改时，该事件会以 SQLITE_SCHEMA 错误代码记录。重新解析和重新准备通常是自动完成的（前提是最初使用 sqlite3_prepare_v2() 来准备语句，这是推荐做法），因此这些日志事件通常是了解重新准备正在进行的唯一方式。

  • 当需要恢复数据库，因为上一个写操作在未完成事务时崩溃，SQLITE_NOTICE 消息会被记录。在恢复回滚日志时，错误代码为 SQLITE_NOTICE_RECOVER_ROLLBACK；在恢复预写日志时，错误代码为 SQLITE_NOTICE_RECOVER_WAL。

  • 当数据库文件以可能导致数据库损坏的方式重命名或别名化时，会记录 SQLITE_WARNING 消息。（有关更多信息，请参见 1 和 2。）

  • 内存不足（OOM）错误会生成带有 SQLITE_NOMEM 错误代码的日志事件，并附带一个消息，说明失败的分配请求了多少字节的内存。

  • 操作系统接口中的 I/O 错误会生成错误日志事件。这些事件的消息会给出源代码中错误起始的行号以及在存在相应文件时的文件名。

  • 当检测到数据库损坏时，会调用错误记录器回调并生成 SQLITE_CORRUPT 错误。与 I/O 错误类似，错误消息文本包含最初检测到错误的源代码中的行号。

  • 遇到 SQLITE_MISUSE 错误时也会调用错误记录器回调。这有助于发现应用程序设计问题，尤其是在应用代码中未始终一致检查返回码的情况下。

SQLite 力求保持错误记录器的流量较低，仅在确实存在问题时才向错误记录器发送消息。应用可以进一步筛选这些错误消息流量，通过故意忽略他们不关心的特定错误类型。例如，频繁进行数据库模式更改的应用可能希望忽略所有 SQLITE_SCHEMA 错误。

4. 总结

强烈建议使用错误记录器回调。错误记录器提供的调试信息在追踪应用在现场发生的隐蔽问题方面极为有用。错误记录器回调对于捕捉偶发错误也很有帮助，这些错误可能由于 API 返回码检查不一致而被应用遗漏。鼓励开发者在开发周期早期实现错误记录器回调，以便快速发现意外行为，并在部署过程中保持错误记录器回调的启用状态。如果错误记录器从未发现问题，也不会带来任何坏处。但未能设置合适的错误记录器可能会在后期影响诊断能力。
