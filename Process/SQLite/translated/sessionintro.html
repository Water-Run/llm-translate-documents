1. 引言

会话扩展提供了一种方便的机制，用于记录对 SQLite 数据库中表的更改，将这些更改打包成“changeset”或“patchset”文件，并稍后将相同的更改应用到具有相同模式且起始数据兼容的另一个数据库。一个“changeset”也可以被反向处理，用来“撤销”一次会话。

本文档是会话扩展的介绍。接口细节请参阅单独的 会话扩展 C 语言接口 文档。

1.1. 典型使用场景

假设 SQLite 被用作某个设计应用的 应用文件格式。两个用户 Alice 与 Bob 各自从一个大约一 GB 的基线设计开始。他们整天并行工作，各自对设计做出定制和调整。到一天结束时，他们希望将各自的更改合并成一个统一的设计。

会话扩展通过记录 Alice 与 Bob 数据库的所有更改并将这些更改写入 changeset 或 patchset 文件来促进这一过程。一天结束时，Alice 可以将她的 changeset 发送给 Bob，而 Bob 可以将其“应用”到自己的数据库。最终结果（假定没有冲突）是 Bob 的数据库同时包含了他和 Alice 的更改。同样，Bob 也可以将他的 changeset 发给 Alice，让她将更改应用到她的数据库。

换句话说，会话扩展为 SQLite 数据库文件提供了一种类似于 unix patch 实用程序的功能，或类似于版本控制系统（如 Fossil、Git、或 Mercurial）中的“合并”能力。

1.2. 获取会话扩展

自 3.13.0 版本（2016-05-18）以来，会话扩展已包含在 SQLite 的 amalgamation 源代码发布包中。默认情况下，会话扩展是禁用的。要启用它，请使用如下编译器开关构建： 

-DSQLITE_ENABLE_SESSION -DSQLITE_ENABLE_PREUPDATE_HOOK

或者，如果使用标准构建系统，则向 configure 脚本传递 --enable-session 选项。

1.3. 限制

  • 在 SQLite 3.17.0 版本之前，会话扩展仅能与rowid 表一起使用，而不能与WITHOUT ROWID 表搭配。从 3.17.0 起，rowid 与 WITHOUT ROWID 表都受支持。但对于 WITHOUT ROWID 表更改，需要额外步骤来记录主键。

  • 不支持虚拟表。对虚拟表的更改不会被捕获。

  • 会话扩展仅对具有声明主键的表有效。表的主键可以是 INTEGER PRIMARY KEY（rowid 别名）或外部主键。

  • SQLite 允许在主键列中存储NULL 值。然而，会话扩展会忽略所有此类行。对包含一个或多个 NULL 主键列的行所做的更改不会被 sessions 模块记录。

2. 概念

2.1. Changeset 与 Patchset

sessions 模块围绕创建和操作 changeset 展开。changeset 是一个编码了对数据库一系列更改的数据块。changeset 中的每个更改均为以下之一：

  • 一个 INSERT。INSERT 更改包含一行要添加到数据库表中。INSERT 更改的有效载荷由新行每个字段的值组成。

  • 一个 DELETE。DELETE 更改表示要从数据库表中删除的一行，通过其主键值进行识别。DELETE 更改的有效载荷由被删除行的所有字段值组成。

  • 一个 UPDATE。UPDATE 更改表示在数据库表中某行的一项或多项非主键字段被修改，该行通过其主键字段进行标识。UPDATE 更改的有效载荷包括：

    • 标识被修改行的主键值，
    • 行中每个被修改字段的新值，和
    • 行中每个被修改字段的原始值。

    UPDATE 更改不会包含未被修改的非主键字段的任何信息。UPDATE 更改无法指定对主键字段的修改。

单个 changeset 可以包含应用于多个数据库表的更改。对于 changeset 涉及至少一个更改的每个表，它还会编码以下数据：

  • 表的名称，
  • 表拥有的列数，以及
  • 哪些列是主键列。

changeset 只能应用于包含与 changeset 中存储的上述三项一致的表的数据库。

patchset 与 changeset 类似。它比 changeset 略微紧凑，但提供的冲突检测和解决选项更有限（详情见下一节）。patchset 与 changeset 的区别在于：

  • 对于 DELETE 更改，有效载荷仅包含主键字段。其他字段的原始值不会作为 patchset 的一部分存储。

  • 对于 UPDATE 更改，有效载荷仅包含主键字段和被修改字段的新值。被修改字段的原始值不会作为 patchset 的一部分存储。

2.2. 冲突

当 changeset 或 patchset 被应用到数据库时，会尝试为每个 INSERT 更改插入新行，为每个 DELETE 更改删除行，并为每个 UPDATE 更改修改行。如果目标数据库与记录 changeset 时的原始数据库处于相同状态，则这一操作很简单。但如果目标数据库的内容并非完全处于该状态，在应用 changeset 或 patchset 时就可能发生冲突。

当处理 INSERT 更改时，可能发生以下冲突：

  • 目标数据库中可能已存在主键与 INSERT 更改指定的值相同的行。
  • 插入新行时可能违反了其他数据库约束，例如 UNIQUE 或 CHECK 约束。

当处理 DELETE 更改时，可能检测到以下冲突：

  • 目标数据库中可能没有具有指定主键值的行可供删除。
  • 目标数据库中可能存在具有指定主键值的行，但其他字段的值可能与 changeset 中存储的值不匹配。使用 patchset 时不会检测到此类冲突。

当处理 UPDATE 更改时，可能检测到以下冲突：

  • 目标数据库中可能没有具有指定主键值的行可供修改。
  • 目标数据库中可能存在具有指定主键值的行，但将被修改字段的当前值可能与 changeset 中存储的原始值不一致。使用 patchset 时不会检测到此类冲突。
  • 更新行时可能违反了其他数据库约束，例如 UNIQUE 或 CHECK 约束。

根据冲突类型，sessions 应用程序可通过各种可配置选项来处理冲突，从忽略冲突更改、中止整个 changeset 应用，到在冲突情况下仍然应用更改。详情请参阅 sqlite3changeset_apply() API 的文档。

2.3. Changeset 构造

在配置会话对象后，它开始监视其配置表的更改。然而，它并不会在数据库中的每次行修改时记录整个更改。相反，它仅记录插入行的主键字段，以及更新或删除行的主键和所有原始行值。如果单个会话在同一行上执行多次修改，则不会记录新的信息。

生成 changeset 或 patchset 所需的其他信息在调用 sqlite3session_changeset() 或 sqlite3session_patchset() 时从数据库文件中读取。具体来说，

  • 对于 INSERT 操作记录的每个主键，sessions 模块会检查表中是否仍存在具有匹配主键的行。如果存在，就将一个 INSERT 更改添加到 changeset。

  • 对于 UPDATE 或 DELETE 操作记录的每个主键，sessions 模块也会检查表中是否存在具有匹配主键的行。如果能找到，但一个或多个非主键字段与原始记录值不匹配，则将一个 UPDATE 添加到 changeset。或者，如果根本不存在具有指定主键的行，则将一个 DELETE 添加到 changeset。如果该行存在但所有非主键字段均未被修改，则不会向 changeset 添加任何更改。

上述机制的一项含义是，如果在单个会话中对某行进行了更改又撤销该更改（如插入后又删除），sessions 模块不会报告任何更改。或者如果在同一会话中对某行执行多次更新，那么所有更新会在任何 changeset 或 patchset 数据块中合并为一次更新。

3. 使用会话扩展

本节提供示例，演示如何使用会话扩展。

3.1. 捕获 Changeset

下面的示例代码展示了在执行 SQL 命令时捕获 changeset 的步骤。概述如下：

  1. 通过调用 sqlite3session_create() API 函数创建一个会话对象（类型为 sqlite3_session*）。

    单个会话对象通过单个 sqlite3* 数据库句柄监视单个数据库（即 “main”、“temp” 或附加数据库）的更改。

  2. 配置会话对象以监视一组表的更改。

    默认情况下，会话对象不会监视任何数据库表的更改。在开始监视之前必须配置它。有三种方式可以配置要监视更改的表集合：

    • 通过对每个表分别调用 sqlite3session_attach() 显式指定表，或
    • 通过使用 NULL 参数调用 sqlite3session_attach()，指定应监视数据库中的所有表，或
    • 通过配置回调，在每个表第一次被写入时调用该回调，以告知 sessions 模块是否应监视该表的更改。

    下面的示例代码使用上述第二种方法——它监视数据库中所有表的更改。

  3. 通过执行 SQL 语句对数据库进行更改。会话对象记录这些更改。

  4. 使用 sqlite3session_changeset()（如果使用 patchset，则调用 sqlite3session_patchset()）从会话对象中提取 changeset 数据块。

  5. 使用 sqlite3session_delete() API 调用删除会话对象。

    在从会话对象中提取 changeset 或 patchset 之后并不需要删除会话对象。它可以继续附着在数据库句柄上，并继续监视配置表的更改。然而，如果对某个会话对象第二次调用 sqlite3session_changeset() 或 sqlite3session_patchset()，那么 changeset 或 patchset 将包含自会话创建以来该连接发生的所有更改。换句话说，调用 sqlite3session_changeset() 或 sqlite3session_patchset() 并不会重置或清零会话对象。 

/*
** 参数 zSql 指向一个包含要对第一个参数传入的数据库句柄执行的 SQL 脚本的缓冲区。除了执行 SQL 脚本之外，
** 该函数还会收集记录对 “main” 数据库文件所做所有更改的 changeset。假设没有发生错误，
** 输出变量 (*ppChangeset) 与 (*pnChangeset) 会在返回 SQLITE_OK 之前被设置为指向包含 changeset 的缓冲区及其字节数。
** 在这种情况下，调用者有责任最终通过 sqlite3_free 函数释放 changeset 数据块。
**
** 如果发生错误，则返回 SQLite 错误码。在这种情况下，(*pChangeset) 与 (*pnChangeset) 的最终值未定义。
*/
int sql_exec_changeset(
  sqlite3 *db,                  /* 数据库句柄 */
  const char *zSql,             /* 要执行的 SQL 脚本 */
  int *pnChangeset,             /* 输出：changeset 数据块的大小（字节） */
  void **ppChangeset            /* 输出：changeset 数据块的指针 */
){
  sqlite3_session *pSession = 0;
  int rc;

  /* 创建新的会话对象 */
  rc = sqlite3session_create(db, "main", &pSession);

  /* 配置会话对象以记录所有表的更改 */
  if( rc==SQLITE_OK ) rc = sqlite3session_attach(pSession, NULL);

  /* 执行 SQL 脚本 */
  if( rc==SQLITE_OK ) rc = sqlite3_exec(db, zSql, 0, 0, 0);

  /* 收集 changeset */
  if( rc==SQLITE_OK ){
    rc = sqlite3session_changeset(pSession, pnChangeset, ppChangeset);
  }

  /* 删除会话对象 */
  sqlite3session_delete(pSession);

  return rc;
}

  void *pChangeset              /* changeset 数据块指针 */
){
  return sqlite3changeset_apply(
      db,
      nChangeset, pChangeset,
      0, xConflict,
      (void*)bIgnoreConflicts
  );
}

3.3. 检查 Changeset 内容

下面的示例代码演示了遍历并提取 changeset 中所有更改相关数据的技术。总结如下：

  1. 调用 sqlite3changeset_start() API 创建并初始化迭代器，以遍历 changeset 的内容。最初，迭代器并未指向任何元素。

  2. 第一次调用 sqlite3changeset_next() 会将迭代器移动到 changeset 中的第一条更改（如果 changeset 完全为空则移动到 EOF）。sqlite3changeset_next() 返回 SQLITE_ROW 表示移动到了有效条目，返回 SQLITE_DONE 表示移动到了 EOF，或在发生错误时返回 SQLite 错误码。

  3. 如果迭代器指向有效条目，可以使用 sqlite3changeset_op() API 来确定迭代器所指更改的类型（INSERT、UPDATE 或 DELETE）。此外，同一 API 还能用于获取更改所作用的表名及其预期列数和主键列数。

  4. 如果迭代器指向有效的 INSERT 或 UPDATE 条目，可使用 sqlite3changeset_new() API 获取更改有效载荷中的 new.* 值。

  5. 如果迭代器指向有效的 DELETE 或 UPDATE 条目，可使用 sqlite3changeset_old() API 获取有效载荷中的 old.* 值。

  6. 使用 sqlite3changeset_finalize() API 删除迭代器。如果遍历过程中出现错误，则返回 SQLite 错误码（即便 sqlite3changeset_next() 已返回相同的错误码）。如果未发生错误，则返回 SQLITE_OK。

4. 扩展功能

大多数应用仅会使用上一节描述的 sessions 模块功能。然而，还提供了以下额外功能，用于使用和操作 changeset 与 patchset 数据块：

  • 可以使用 sqlite3changeset_concat() 或 sqlite3_changegroup 接口将两个或多个 changeset/patchset 合并。

  • 可以使用 sqlite3changeset_invert() API 将 changeset “反向”。反向的 changeset 会撤销原始 changeset 所做的更改。如果 changeset C+ 是 changeset C 的逆，那么先应用 C 再应用 C+ 应该会使数据库保持不变。
