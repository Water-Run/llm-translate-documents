1. C 是最佳

注意：本文章的 2.0 和 3.0 节是为回应 Hacker News 和 Reddit 上的评论而添加的。

自 2000-05-29 诞生以来，SQLite 就以通用 C 语言实现。C 曾经也是现在继续是实现像 SQLite 这样的软件库的最佳语言。目前没有将 SQLite 以其他编程语言重写的计划。

C 是实现 SQLite 的最佳语言的原因包括：

  • 性能
  • 兼容性
  • 低依赖
  • 稳定性

1.1. 性能

像 SQLite 这样使用频繁的底层库必须要快。（SQLite 确实很快，参见 内部与外部 BLOBs 和 比文件系统快 35% 以查看示例。）

C 是编写高性能代码的绝佳语言。C 有时被描述为“可移植的汇编语言”。它让开发者在尽可能靠近底层硬件的同时仍然可以跨平台可移植。

其他编程语言有时声称“和 C 一样快”。但没有哪种语言宣称在通用编程方面比 C 更快，因为的确没有。

1.2. 兼容性

几乎所有系统都有调用用 C 编写的库的能力。其他实现语言就不一定有这种能力。

举个例子，用 Java 编写的 Android 应用可以通过适配层调用 SQLite。也许如果 SQLite 用 Java 编写，Android 会更方便，因为接口会更简单。然而，iPhone 上的应用是用 Objective-C 或 Swift 编写的，二者都无法调用用 Java 编写的库。因此如果 SQLite 用 Java 编写，就无法在 iPhone 上使用。

1.3. 低依赖

用 C 编写的库没有巨大的运行时依赖。在其最小配置下，SQLite 只需要标准 C 库中的如下几个例程：

  • memcmp()
  • memcpy()
  • memmove()
  • memset()
   
  • strcmp()
  • strlen()
  • strncmp()

在更完整的构建中，SQLite 还会使用 malloc() 和 free() 之类的库例程以及用于打开、读取、写入和关闭文件的操作系统接口。但即便如此，依赖项仍然非常少。相比之下，其他“现代”语言往往需要带有成千上万接口的多兆字节运行时。

1.4. 稳定性

C 语言古老而乏味。它是一个众所周知且被充分理解的语言。正是开发像 SQLite 这样的模块时所需要的。编写一个小巧、快速且可靠的数据库引擎已经够难了，更不希望实现语言的规范在每次更新时都发生变化。

2. 为什么 SQLite 不用面向对象语言编写？

一些程序员无法想象在非“面向对象”的语言中开发像 SQLite 这样复杂的系统。那么为什么 SQLite 不用 C++ 或 Java 编写呢？

  1. 用 C++ 或 Java 编写的库通常只能被用相同语言编写的应用调用。让用 Haskell 或 Java 编写的应用调用一个用 C++ 编写的库是困难的。另一方面，用 C 编写的库可以被任何编程语言调用。

  2. 面向对象是一种设计模式，而不是一种编程语言。你可以在任何语言中进行面向对象编程，包括汇编语言。有些语言（例如 C++ 或 Java）让面向对象变得更容易。但你仍然可以在像 C 这样的语言中进行面向对象编程。

  3. 面向对象并不是唯一合理的设计模式。许多程序员被教导只用对象来思考。公平地说，对象常常是分解问题的好方法。但对象并不是唯一的方式，也并非总是分解问题的最佳方式。有时候老式的过程式代码更容易编写、更容易维护和理解，并且比面向对象代码更快。

  4. 当 SQLite 最初开发时，Java 还是一门年轻而不成熟的语言。C++ 虽然更老，但因经历成长阵痛而难以找到两款行为一致的 C++ 编译器。因此当时 C 显然是更好的选择。现在情况有所改善，但在目前阶段重新用其他语言重写 SQLite 几乎没有好处。

3. 为什么 SQLite 不用“安全”语言编写？

近来人们对像 Rust 或 Go 这样的“安全”语言产生了浓厚兴趣，这些语言中几乎不可能或至少很难犯诸如内存泄漏或数组越界之类的常见编程错误。因此人们常常会问，为什么 SQLite 不用“安全”语言编写。

  1. 在 SQLite 出现的最初十年中，这些安全编程语言都不存在。虽然 SQLite 可以用 Go 或 Rust 重写，但这么做可能引入远比修复更多的 Bug，并且可能导致代码变慢。

  2. 安全语言会插入额外的机器分支来完成诸如验证数组访问是否越界之类的工作。在正确的代码中，这些分支永远不会被执行。那就意味着机器代码无法做到 100% 的分支覆盖测试，而这是 SQLite 质量策略中的重要组成部分。

  3. 安全语言通常在遇到内存不足（OOM）情况时会选择中止。SQLite 的设计是可以从 OOM 中优雅恢复。目前这些安全语言还不清楚如何实现这一点。

  4. 所有现有的安全语言都是新的。SQLite 的开发者赞赏计算机语言研究人员为开发更容易安全编程的语言所做的努力。我们鼓励这些努力继续。但在实现 SQLite 时，我们更倾向于使用古老而乏味的语言。

尽管如此，SQLite 将来某天可能会用 Rust 重新编写。用 Go 重写 SQLite 的可能性较小，因为 Go 讨厌 assert()。但 Rust 是有可能的。让 SQLite 在重写为 Rust 之前必须满足的一些前提条件包括：

  1. Rust 需要再成熟一点，停止快速变化，并进一步向“古老而乏味”靠拢。
  2. Rust 需要证明自己能够用来创建可被所有其他编程语言调用的通用库。
  3. Rust 需要证明它能够生成在偏僻的嵌入式设备上工作的目标代码，包括那些没有操作系统的设备。
  4. Rust 需要补齐必要的工具，以便对编译后的二进制执行 100% 的分支覆盖测试。
  5. Rust 需要一个机制来优雅地从 OOM 错误中恢复。
  6. Rust 需要证明它可以在不带来显著性能损失的情况下完成 C 在 SQLite 中所做的工作。

如果你是“rustacean”，并且认为 Rust 已经满足了上述的前提条件，并且 SQLite 应该用 Rust 重写，欢迎你私下联系 SQLite 的开发者并陈述你的理由。
