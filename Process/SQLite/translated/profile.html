1. 概述

SQLite 内置支持 SQL 查询分析，但默认未启用。要启用查询分析支持，必须使用 以下选项 编译 SQLite： 

-DSQLITE_ENABLE_STMT_SCANSTATUS

使用此选项构建 SQLite 会启用 sqlite3_stmt_scanstatus_v2() API，它提供访问各种分析指标的能力。页面剩余部分讨论 SQLite 命令行 shell 基于这些指标生成的分析报告，而非直接使用该 API。

shell 生成的分析报告与 EXPLAIN QUERY PLAN 命令生成的查询计划报告非常相似。本文假定读者熟悉此格式。

在使用上述选项构建的命令行 shell 中，可用“.scanstats on”命令启用查询分析： 

sqlite> .scanstats on

启用后，shell 会在每次执行 SQL 查询后自动输出查询分析。可以使用“.scanstats off”禁用。例如： 

sqlite> .scanstats on
sqlite> SELECT a FROM t1, t2 WHERE a IN (1,2,3) AND a=d+e;
QUERY PLAN (cycles=255831538 [100%])
|--SEARCH t1 USING INTEGER PRIMARY KEY (rowid=?)     (cycles=60048488 [23%] loops=1 rows=3)
`--SCAN t2                                           (cycles=133558052 [52%] loops=3 rows=150000)

2. 简单情况 - 行、循环与周期

考虑下列模式的数据库： 

CREATE VIRTUAL TABLE ft USING fts5(text);
CREATE TABLE t1(a, b);
CREATE TABLE t2(c INTEGER PRIMARY KEY, d);

然后，在首次执行“.scanstats on”之后： 

sqlite3> SELECT * FROM t1, t2 WHERE t2.c=t1.a;
<...query results...>
QUERY PLAN (cycles=1140768 [100%])
|--SCAN t1                                           (cycles=455974 [40%] loops=1 rows=500)
`--SEARCH t2 USING INTEGER PRIMARY KEY (rowid=?)     (cycles=619820 [54%] loops=500 rows=250)

上述示例中“<...query results...>”之后的文本就是刚执行的连接查询的分析报告。与 EXPLAIN QUERY PLAN 输出相似的部分表明，该查询是通过对表 “t1” 做全表扫描，并针对每个访问的行在表 “t2” 上执行整数主键查找来实现的。

“SCAN t1” 行上的 “loops=1” 表示该循环 —— 即对表 “t1” 的全表扫描 —— 只运行了一次。“rows=500” 表示该单次扫描访问了 500 行。

“SEARCH t2 USING ...” 行包含注释 “loops=500”，表示该 “循环” （实际上是整数主键查找）执行了 500 次。这是合理的 —— 每访问表 “t1” 的一行就执行一次查找。“rows=250” 表示这 500 次循环总共访问了 250 行。换言之，对表 t2 的整数主键查找只在一半的情况下成功，另半数查找未找到对应行。

SEARCH 或 SCAN 条目的循环计数不一定等于外层循环输出的行数。例如，如果将上述查询修改为： 

sqlite3> SELECT * FROM t1, t2 WHERE t1.b<=100 AND t2.c=t1.a;
<...query results...>
QUERY PLAN (cycles=561002 [100%])
|--SCAN t1                                           (cycles=345950 [62%] loops=1 rows=500)
`--SEARCH t2 USING INTEGER PRIMARY KEY (rowid=?)     (cycles=128690 [23%] loops=100 rows=50)

这次虽然 “SCAN t1” 仍然访问 500 行，但 “SEARCH t2” 仅执行 100 次。这是因为 SQLite 能够丢弃不满足 “t1.b<=100” 约束的 t1 行。

“cycles” 测量基于 CPU 时间戳计数器，因此是墙钟时间的良好近似。上述查询的总周期数为 561002。对于两个循环（“SCAN t1...” 和 “SEARCH t2...”），周期计数表示可直接归因于该循环的时间消耗。具体来说，这是在该循环中导航和从数据库 B 树中抽取数据所耗费的时间。这些值的总和通常略低于查询的总周期，因为 SQLite 还执行了其他无法直接归因于任一循环的内部操作。

“SCAN t1” 循环的周期计数为 345950 —— 占查询总量的 62%。“SEARCH t2” 循环的 100 次查找消耗了 128690 个周期，占总量的 23%。

使用虚拟表时，“rows” 和 “loops” 指标的含义与常规 SQLite 表中的循环相同。“cycles” 测量表示与该循环相关联的虚拟表方法所消耗的总周期数。例如： 

sqlite3> SELECT * FROM ft('sqlite'), t2 WHERE t2.c=ft.rowid;
<...query results...>
QUERY PLAN (cycles=836434 [100%]
|--SCAN ft VIRTUAL TABLE INDEX 0:M1                  (cycles=739602 [88%] loops=1 rows=48)
`--SEARCH t2 USING INTEGER PRIMARY KEY (rowid=?)     (cycles=62866 [8%] loops=48 rows=25)

在此示例中，对 fts5 表 “ft” 的单个查询（loops=1）返回 48 行（rows=48），消耗了 739602 个周期（cycles=739602），约占查询总时间的 88%。

3. 复杂情况 - 行、循环与周期

使用上一节中的相同模式，考虑下列更复杂的示例： 

sqlite3> WITH cnt(i) AS (
  SELECT 1 UNION SELECT i+1 FROM cnt WHERE i<100
)
SELECT
  *, (SELECT d FROM t2 WHERE c=ft.rowid)
FROM
  (SELECT count(*), a FROM t1 GROUP BY a) AS v1 CROSS JOIN
  ft('sqlite'),
  cnt
WHERE cnt.i=ft.rowid AND v1.a=ft.rowid;
<...query results...>
QUERY PLAN (cycles=177665334 [100%])
|--CO-ROUTINE v1                                        (cycles=4500444 [3%])
|  |--SCAN t1                                           (cycles=397052 [0%] loops=1 rows=500)
|  `--USE TEMP B-TREE FOR GROUP BY
|--MATERIALIZE cnt                                      (cycles=1275068 [1%])
|  |--SETUP
|  |  `--SCAN CONSTANT ROW
|  `--RECURSIVE STEP
|     `--SCAN cnt                                       (cycles=129166 [0%] loops=100 rows=100)
|--SCAN v1                                              (loops=1 rows=500)
|--SCAN ft VIRTUAL TABLE INDEX 0:M1=                    (cycles=161874340 [91%] loops=500 rows=271)
|--SCAN cnt                                             (cycles=7336350 [4%] loops=95 rows=9500)
`--CORRELATED SCALAR SUBQUERY 3                         (cycles=168538 [0%] loops=37)
   `--SEARCH t2 USING INTEGER PRIMARY KEY (rowid=?)     (cycles=94724 [0%] loops=37 rows=21)

上述示例中最复杂的部分是理解查询计划 —— 也就是该报告中与 EXPLAIN QUERY PLAN 命令所生成的内容一致的部分。其他值得关注的点包括：

  • 子查询 “v1” 作为协程实现。在此情况下，子查询会单独报告，并为整个子查询提供一个 “cycles” 计数。还有一行 “SCAN v1” —— 它表示主查询调用子查询协程。此项不关联任何周期计数，因为子查询的全部开销都归因于协程自身。但它依然有 “loops” 与 “rows” 值 —— 子查询被扫描一次并返回 500 行。

  • 递归子查询 “cnt” 在主查询运行前被物化（缓存到临时表）。物化的全部开销归因于 “MATERIALIZE cnt” 元素。此外还有一个 “SCAN cnt” 项，表示对物化子查询的扫描。该项关联的周期数代表扫描包含物化子查询的临时表所需的时间，独立于用来填充临时表的周期。

  • 标量子查询也有周期与循环计数。这些值分别表示执行该子查询所消耗的总周期数与执行次数。

  • 当某个条目是另一个条目的父项时（例如 “CORRELATED SCALAR SUBQUERY 3” 与 “SEARCH t2 USING...”），与父项关联的周期值包括所有子元素所消耗的周期。在所有情况下，百分比值是相对于整个查询的总周期，而非父项所对应的周期。

以下查询使用了自动索引以及外部排序： 

sqlite> SELECT * FROM
  t2,
  (SELECT count(*) AS cnt, d FROM t2 GROUP BY d) AS v2
WHERE v2.d=t2.d AND t2.d>100
ORDER BY v2.cnt;
<...query results...>
QUERY PLAN (cycles=6234376 [100%])
|--MATERIALIZE v2                                     (cycles=2351916 [38%])
|  |--SCAN t2                                         (cycles=188428 [3%] loops=1 rows=250)
|  `--USE TEMP B-TREE FOR GROUP BY
|--SCAN t2                                            (cycles=455736 [7%] loops=1 rows=250)
|--CREATE AUTOMATIC INDEX ON v2(d, cnt)               (cycles=1668380 [27%] loops=1 rows=250)
|--SEARCH v2 USING AUTOMATIC COVERING INDEX (d=?)     (cycles=932824 [15%] loops=200 rows=200)
`--USE TEMP B-TREE FOR ORDER BY                       (cycles=662456 [11%] loops=1 rows=200)

值得关注的点包括：

  • 该查询先将子查询物化到临时表，然后在其上创建自动（即临时）索引，接着使用该索引优化连接。这三步 —— “MATERIALIZE v2”、“CREATE AUTOMATIC INDEX” 与 “SEARCH ... USING AUTOMATIC INDEX” —— 都有独立的周期计数。“CREATE AUTOMATIC INDEX” 行的 “rows” 表示索引包含的总行数。“SEARCH ... USING AUTOMATIC INDEX” 行的 “loops” 与 “rows” 表示索引被用于多少次查找以及这些查找返回的总行数。

  • 外部排序 “USE TEMP B-TREE FOR ORDER BY” 也单独列出。该项的周期计数表示排序所消耗的额外周期，相对于若返回行顺序随意时所需的周期。“rows” 表示被排序的行数。

4. 规划器估算

除了用“.scanstats on”启用分析与“.scanstats off”禁用之外，shell 还有一个“.scanstats est”命令： 

sqlite> .scanstats est

此命令启用一种特殊的分析报告，报告中每个 “SCAN...” 与 “SEARCH...” 元素会附带两个额外值：

  • rpl（每循环行数）：该值等于“rows”指标除以“loops”指标。
  • est（估算值）：这是查询规划器估算当前查询元素每个循环将返回的行数。如果该值与实际每循环行数差别很大且查询性能不佳，可通过运行 ANALYZE 来改进估算质量。 
sqlite> SELECT a FROM t1, t2 WHERE a IN (1,2,3) AND a=d+e ORDER BY a;
<query results...>
QUERY PLAN (cycles=264725190 [100%]
|--SEARCH t1 USING INTEGER PRIMARY KEY (rowid=?)     (cycles=60511568 [23%] loops=1 rows=3 rpl=3.0 est=3.0)
`--SCAN t2                                           (cycles=139461608 [53%] loops=3 rows=150000 rpl=50000.0 est=1048576.0)
