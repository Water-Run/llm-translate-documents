1. 背景

SQLite 是一个软件库，它将应用程序生成的高级磁盘 I/O 请求转换为操作系统可以执行的低级 I/O 操作。应用程序使用SQL 语言构建这些高级 I/O 请求。SQLite 将每条高级 SQL 语句翻译为一系列低级 I/O 请求（打开文件、从文件读取几个字节、向文件写入几个字节等），以执行 SQL 所请求的工作。

应用程序可以通过直接调用操作系统 I/O 例程完成所有磁盘 I/O，或者使用像Berkeley DB或RocksDB这样的键/值存储引擎（仅举几例）。但使用基于 SQL 语言的更高级接口仍然具有优势。

  1. SQL 是一种非常高级的语言。几行 SQL 代码就可以替代数百或数千行过程式代码。SQL 因此减少了开发和维护应用程序所需的工作量，并有助于降低应用程序中的 bug 数量。

  2. SQL 和 SQLite 都是事务性的。使用事务性存储系统可以让人更容易推理应用程序的行为，并编写在软件 bug、硬件故障或断电时也能保持稳健的应用程序。

  3. SQLite 通常比直接使用低级 I/O 更快。这听起来有悖常理。人们本来会认为像 SQLite 这样的高级接口会带来运行时开销。从理论上说确实如此。但实际上，基于 SQL 的系统（如 SQLite）在幕后进行了许多优化，这些优化是应用程序开发者根本没有时间去构建和维护的，因此基于 SQL 的系统最终能提供净性能提升。

1.1. SQLite 与大多数其他 SQL 数据库不同

除了 SQLite 之外，还有许多基于 SQL 的数据库管理系统可用。常见的选项包括 MySQL、PostgreSQL 和 SQL Server。所有这些系统都像 SQLite 一样使用 SQL 语言与应用程序通信。但在若干重要方面，这些其他系统与 SQLite 不同。

  1. SQLite 是一个无服务器的软件库，而其他系统则基于客户端-服务器模型。在 MySQL、PostgreSQL、SQL Server 等系统中，应用程序将包含一些 SQL 的消息发送给独立的服务器线程或进程。那个独立的线程或进程执行所请求的 I/O，然后将结果返回给应用程序。但 SQLite 没有独立的线程或进程。SQLite 与应用程序运行在同一地址空间，使用相同的程序计数器和堆存储。SQLite 不进行进程间通信（IPC）。当应用程序通过调用相应的 SQLite 库子例程向 SQLite 发送 SQL 语句时，SQLite 在与调用者相同的线程中解释该 SQL。当 SQLite API 例程返回时，它不会留下任何独立于应用程序运行的后台任务。

  2. SQLite 数据库只是磁盘上的一个普通文件（具有明确定义的文件格式）。而其他系统中的“数据库”通常隐藏在文件系统晦涩目录下的许多独立文件中，甚至散布在多台机器上。但在 SQLite 中，一个完整的数据库只是一个普通的磁盘文件。

2. SQL 是一种编程语言

理解 SQL 数据库引擎工作原理的最佳方式是将 SQL 视为一种编程语言，而不是“查询语言”。每条 SQL 语句都是一个独立的程序。应用程序构建 SQL 程序源文件并将其发送给数据库引擎。数据库引擎将 SQL 源代码编译成可执行形式，运行该可执行代码，然后将结果返回给应用程序。

虽然 SQL 是一种编程语言，但它与 C、Javascript、Python 或 Go 等其他编程语言不同，因为 SQL 是一种声明式语言，而其他语言则是命令式语言。这是一个重要的区别，影响到将程序源文本翻译为可执行格式的编译器设计。不过，这些细节不应掩盖这样一个事实：SQL 其实只是另一种编程语言。

2.1. 编程语言处理步骤

所有编程语言的处理都分为两步：

  1. 将程序源文本翻译为可执行格式。

  2. 运行前一步生成的可执行文件以执行所需操作。

所有编程语言都使用这两个基本步骤。主要的区别在于可执行格式。

像 C++ 和 Rust 这样的“编译型”语言将源文本翻译为可以由底层硬件直接执行的机器码。确实存在一些 SQL 数据库系统会直接将 SQL 语句翻译成机器码，但这类做法并不常见，而且不是 SQLite 所采用的方式。

其他语言如 Java、Perl、Python 和 TCL 通常将程序源文本翻译为字节码。然后通过解释器运行该字节码，解释器读取字节码并执行所需操作。SQLite 使用这种字节码方式。如果在 SQLite 中在任何 SQL 语句前加上 "EXPLAIN" 关键字，它将显示生成的字节码，而不是运行字节码。

另一种方法是将程序源文本翻译为内存中的对象树。该树就是“可执行文件”。解释器通过遍历这棵树来运行可执行文件。这正是 MySQL、PostgreSQL 和 SQL Server 采用的技术。

当然，并非每种语言都严格属于上述某一类。这对 SQL 数据库引擎和更常见的命令式编程语言都适用。Javascript 就以使用混合执行模型著称——代码最初被编译成对象树，但可能会进一步被翻译（通过即时编译）成更高效的字节码或机器码，以提高性能。

可执行格式最终只是一个实现细节。关键点是所有语言都有一个编译步骤，将程序翻译成某种可执行格式，以及一个运行步骤，用于执行编译后的程序。

2.2. 编译 SQLite 程序

当 SQL 程序提交给 SQLite 时，第一步是将源文本拆分成“记号”。一个记号可能是：

  • 像 “SELECT” 或 “UPDATE” 这样的语言关键字。
  • 表、列或变量的标识符。
  • 像 “,”、“==” 或 “;” 这样的标点符号。
  • 字面值：数字或字符串常量。
  • 空白或注释。

空白和注释记号会被丢弃。所有其他记号则被送入LALR(1) 解析器，该解析器分析输入程序的结构并生成输入程序的抽象语法树 (AST)。

解析器将 AST 传递给代码生成器。代码生成器是 SQLite 的核心，也是大多数魔法发生的地方。代码生成器在 AST 中解析符号名称——将输入 SQL 中的列名和表名匹配到数据库的实际列和表。代码生成器还对 AST 进行各种转换以“优化”它。最后，代码生成器选择适当的算法来实现 AST 请求的操作，并构建执行这些操作的字节码。

由代码生成器生成的字节码称为“预处理语句”。将 SQL 源文本翻译为预处理语句类似于通过调用 gcc 或 clang 将 C++ 程序转换为机器码。人类可读的源文本（SQL 或 C++）作为输入，机器可读的可执行文件（字节码或机器码）作为输出。
