1. 引言

“查询规划器”的任务是找出完成一个 SQL 语句的最佳算法或“查询计划”。从 SQLite version 3.8.0 (2013-08-26) 开始，查询规划器组件已经重写，使其运行更快并生成更好的计划。这次重写被称为“下一代查询规划器”或“NGQP”。

本文概述了查询规划的重要性，描述了查询规划固有的一些问题，并概述了 NGQP 如何解决这些问题。

NGQP 几乎总是优于旧版查询规划器。然而，可能存在某些旧应用无意中依赖于旧查询规划器中的未定义和/或次优行为，并且在这些旧应用上升级到 NGQP 可能会导致性能回退。本文考虑了这种风险，并提供了一套清单以减少风险并修复实际出现的问题。

本文件重点介绍 NGQP。如需更全面地了解涵盖整个 SQLite 历史的 SQLite 查询规划器，请参见“SQLite 查询优化器概述”和“索引的工作原理”文档。

2. 背景

对于具有少量索引的单表简单查询，通常可以明显地选择最佳算法。但对于更大、更复杂的查询，例如具有许多索引和子查询的多路联接，可能存在数百、数千甚至数百万种合理的计算结果的算法。查询规划器的工作就是从这众多可能性中选择唯一“最佳”的查询计划。

查询规划器使 SQL 数据库引擎如此强大和实用。（这对所有 SQL 数据库引擎都成立，而不仅限于 SQLite。）查询规划器免除了程序员选择特定查询计划的劳动，从而让程序员可以将更多的精力集中在更高层次的应用问题上，并为最终用户提供更多的价值。对于查询计划选择显而易见的简单查询来说，这是方便但并非至关重要。但随着应用、模式和查询变得更复杂，聪明的查询规划器可以极大地加快和简化应用开发的工作。能够告诉数据库引擎所需内容，然后让数据库引擎找出检索这些内容的最佳方式，其潜力令人惊叹。

编写好的查询规划器更偏向于艺术而非科学。查询规划器必须处理不完整的信息。它无法在不真正执行该计划的情况下确定任何特定计划将花费多长时间。因此，在比较两个或更多计划以确定哪个“最佳”时，查询规划器必须做出一些猜测和假设，而这些猜测和假设有时会出错。一个好的查询规划器就是能够足够经常找到正确解决方案的那种，这样应用程序员就很少需要介入。

2.1. SQLite 中的查询规划

SQLite 使用嵌套循环计算联接，每个联接中的每个表对应一个循环。（对于 WHERE 子句中的 IN 和 OR 操作符，可能会插入额外的循环。SQLite 也会考虑这些情况，但为了简洁起见，这篇文章将忽略它们。）每个循环可能会使用一个或多个索引来加速搜索，或者该循环可能是“全表扫描”，读取表中的每一行。因此查询规划分解为两个子任务：

  1. 选择各个循环的嵌套顺序
  2. 为每个循环选择合适的索引

选择嵌套顺序通常是更具挑战性的问题。一旦联接的嵌套顺序确立，选择每个循环的索引通常是显而易见的。

2.2. SQLite 查询规划器稳定性保证

当启用查询规划器稳定性保证 (QPSG) 时，只要满足以下条件，SQLite 就会始终为给定的 SQL 语句选择相同的查询计划：

  1. 数据库模式没有重大变化，例如添加或删除索引。
  2. 没有重新运行 ANALYZE 命令。
  3. 使用的是相同版本的 SQLite。

QPSG 默认是禁用的。可以通过编译时使用 SQLITE_ENABLE_QPSG 选项来启用，或者在运行时调用 sqlite3_db_config(db,SQLITE_DBCONFIG_ENABLE_QPSG,1,0)。

QPSG 的意思是，如果所有查询在测试阶段运行良好，并且应用未更改模式，那么 SQLite 不会突然开始使用不同的查询计划，从而可能在应用发布后导致性能问题。如果应用在实验室中运行正常，那么部署后也会继续以相同的方式运行。

客户端/服务器型 SQL 数据库引擎通常不会作出这种保证。在这类引擎中，服务器会跟踪表大小和索引质量等统计信息，而查询规划器会利用这些统计信息选择最佳计划。随着数据库中不断添加、删除或更改内容，统计信息也会随之演化，可能导致查询规划器开始对某些特定查询使用不同的查询计划。通常情况下，新的计划会更适合数据结构的演化，但有时也会引起性能下降。对于客户端/服务器数据库引擎，通常会有数据库管理员 (DBA) 处理这些罕见问题。但像 SQLite 这样的嵌入式数据库无法依赖 DBA，因此 SQLite 非常小心，确保部署后计划不会出人意料地改变。

需要注意的是，SQLite 版本的变化可能会导致查询计划的变化。相同版本的 SQLite 始终会选择相同的查询计划，但如果您将应用重新链接到不同版本的 SQLite，则查询计划可能会发生变化。在少数情况下，SQLite 版本的更换可能会导致性能回退。这也是您应该考虑让应用静态链接 SQLite，而不是使用系统范围的 SQLite 共享库，因为后者可能在您不知情或无法控制的情况下发生变化。

3. 一个棘手的案例

“TPC-H Q8” 是来自事务处理性能委员会的测试查询。SQLite 3.7.17 及更早版本的查询规划器无法为 TPC-H Q8 选择良好计划。并且已经确定，旧查询规划器的任何调整都无法解决这个问题。为了找到 TPC-H Q8 查询的良好解决方案，并继续提高 SQLite 查询规划器的质量，有必要重新设计查询规划器。本节试图解释为何需要进行此重新设计，以及 NGQP 有何不同并如何解决 TPC-H Q8 问题。

3.1. 查询细节

TPC-H Q8 是一个八路联接。如前所述，查询规划器的主要任务是找出八个循环的最佳嵌套顺序，以最小化完成联接所需的工作量。针对 TPC-H Q8 的此问题的简化模型由下图显示：

在图中，查询的 FROM 子句中的 8 个表分别由标有 FROM 子句项的圆圈表示：N2、S、L、P、O、C、N1 和 R。图中的弧表示在假设该弧的起点位于外层循环时，计算每个项的估计成本。例如，将 S 循环作为 L 的内层循环运行的成本为 2.30，而将 S 循环作为 L 的外层循环运行的成本为 9.17。

这里的“成本”是对数的。通过嵌套循环，工作量是相乘的，而不是相加。但通常习惯于将图表示为加权图，因此图显示的是各种成本的对数。图中显示 S 位于 L 内部的成本优势约为 6.87，但这意味着当 S 循环位于 L 循环内部而不是外部时，查询运行速度大约快 963 倍。

来自带有“*”标记的小圆圈的箭头表示在没有依赖的情况下运行每个循环的成本。最外层循环必须使用此 *-成本。内层循环可以选择使用 *-成本，或者假设其他某一项位于外层循环的成本，以取最佳结果。可以将 *-成本视为一种简化表示，表示多个弧，其中每一个弧都来自图中的其他节点。因此，图是“完全的”，意味着图中每对节点之间都有弧（有的显式，有的隐含），并且具有双向。

找到最佳查询计划的问题相当于找到一条以最低成本访问图中每个节点一次的路径。

（旁注：上述 TPC-H Q8 图中的成本估算是由 SQLite 3.7.16 中的查询规划器计算并使用自然对数转换得到的。）

3.2. 复杂性

上述对查询规划器问题的描述是简化的。成本是估算值。我们无法在实际运行循环之前知道它的真实成本。SQLite 会根据索引的可用性以及在 WHERE 子句中发现的约束来猜测循环的成本。这些猜测通常相当准确，但有时候也会偏差。使用 ANALYZE 命令收集更多数据库统计信息，有时能让 SQLite 做出更好的成本估算。

成本由多个数字构成，而不像图中所示的那样是单一数字。SQLite 为每个循环计算几种不同的估计成本，适用于不同的阶段。例如，有一个“设置”成本，只在查询开始时发生一次。设置成本是为尚未存在索引的表计算查询时索引的成本。然后，还有运行循环每一步的成本。最后，还有循环生成的行数估计，这是估算内层循环成本所需的信息。如果查询包含 ORDER BY 子句，则排序成本也可能涉及其中。

在一般查询中，依赖关系不一定局限于单个循环，因此依赖关系矩阵可能无法表示为图。例如，WHERE 子句中的某个约束可能是 S.a=L.b+P.c，意味着 S 循环必须同时作为 L 和 P 的内层循环。这种依赖关系无法绘制为图，因为没有办法让一条弧同时起始于两个或更多节点。

如果查询包含 ORDER BY 子句或 GROUP BY 子句，或者查询使用 DISTINCT 关键字，那么选择通过图的一条路径，使行自然以排序顺序出现，从而无需进行额外的排序步骤，会更合适。自动消除 ORDER BY 子句可以显著提升性能，因此这是完整实现中需要考虑的另一个因素。

在 TPC-H Q8 查询中，设置成本都可以忽略，所有依赖关系都发生在独立节点之间，并且没有 ORDER BY、GROUP BY 或 DISTINCT 子句。因此，对于 TPC-H Q8，上述图是需要计算内容的合理表示。通常情况下会涉及大量额外的复杂性，为了清晰起见，本文剩余部分忽略了这些复杂性。

3.3. 寻找最佳查询计划

在 version 3.8.0 (2013-08-26) 之前，SQLite 搜索最佳查询计划时始终使用“最近邻”或 “NN” 启发式。NN 启发式对图进行一次遍历，每一步总是选择下一步成本最低的弧。NN 启发式在大多数情况下表现出乎意料地好，而且 NN 很快，因此 SQLite 能够快速为即便是大型 64 路联接也找到良好计划。相比之下，其他进行更广泛搜索的 SQL 数据库引擎在联接表数量超过 10 或 15 时往往会陷入困境。

不幸的是，NN 计算出的 TPC-H Q8 查询计划并不最优。NN 所计算的计划是 R-N1-N2-S-C-O-L-P，成本为 36.92。前一句中的符号表示 R 表在外层循环，N1 在下一个内层循环，以此类推，到最内层循环的 P。通过穷尽搜索找到的图中最短路径是 P-L-O-C-N1-R-S-N2，成本为 27.38。差距可能看起来不大，但请记住成本是对数的，因此最短路径比 NN 启发式找到的路径快近 750 倍。

一种解决方案是让 SQLite 为最佳路径执行穷尽搜索。但穷尽搜索所需时间与 K! 成正比（K 为联接中的表数），因此当表数超过 10 时，运行 sqlite3_prepare() 所需时间就会非常长。

3.4. N 最近邻或 “N3” 算法

NGQP 使用一种用于寻找图中最佳路径的新算法：“N 最近邻”（下文称为 “N3”）。在 N3 中，不再每次只选择一个最近邻，而是在每一步跟踪 N 条最佳路径，其中 N 是某个小整数。

假设 N=4。那么对于 TPC-H Q8 图，第一步会找到访问图中任意单个节点的四条最短路径：

R (成本: 3.56)
N1 (成本: 5.52)
N2 (成本: 5.52)
P (成本: 7.71)

第二步从上一步的四条路径之一出发，找到访问两个节点的四条最短路径。在两条或多条路径等价（即访问的节点集合相同，尽管顺序可能不同）的情况下，只保留第一条且成本最低的路径。我们得到：

R-N1 (成本: 7.03)
R-N2 (成本: 9.08)
N2-N1 (成本: 11.04)
R-P {成本: 11.27}

第三步从四条最短的两节点路径开始，找到四条最短的三节点路径：

R-N1-N2 (成本: 12.55)
R-N1-C (成本: 13.43)
R-N1-P (成本: 14.74)
R-N2-S (成本: 15.08)

如此继续。TPC-H Q8 查询有 8 个节点，因此这个过程总共重复 8 次。在一般的 K 路联接中，存储需求为 O(N)，计算时间为 O(K*N)，这明显快于 O(2K) 的精确解法。

那么该选择多大的 N 呢？可以尝试 N=K，这使算法成为 O(K2)，实际上仍相当高效，因为 K 的最大值是 64 且通常不超过 10。但这对 TPC-H Q8 问题而言仍不够。对于 TPC-H Q8，当 N=8 时，N3 算法找到的解是 R-N1-C-O-L-S-N2-P，成本为 29.78。虽然比 NN 有很大改善，但仍非最优。当 N ≥ 10 时，N3 能找到 TPC-H Q8 的最优解。

NGQP 的初始实现对简单查询选择 N=1，对二路联接选择 N=5，对三个或更多表的联接选择 N=10。后来，对三个或更多表的联接的 N 值提高到 12，若联接属于星型模式则提高到 18。未来可能会有进一步的改进。

4. 星型查询启发式

N3 算法 对大多数查询都表现得很好。但在某些情况下会生成不佳计划。具体来说，对于某些针对 星型模式 的查询——那里有一张大型事实表和六张或更多较小的维度表，并且索引存在双向连接——除非将 N 参数设得非常大（随维度表数量呈指数增长），否则 N3 算法会错过最优查询计划。我们称这类查询为“星型查询”。较新的 SQLite 版本（大约 2024 年或 2025 年及之后）实现了一种启发式来解决这一问题，从而无需程序员提供提示就能为星型查询选择高效的查询计划。

4.1. 背景：什么是星型查询

星型模式是一种常见的关系数据库设计，其中有一张大型“事实表”被较小的“维度表”包围。维度表通常以整数 ID 值作为主键，并包含通过该 ID 访问的各种其他属性。事实表包含引用各个维度表的 ID 值。

例如，假设您有以下维度表，它们都使用整数 ID 作为 PRIMARY KEY：

  • PRODUCT 表保存每个产品的信息。
  • SELLER 表保存每位销售人员的信息。
  • CUSTOMER 表保存每位客户的信息。
  • CARRIER 表保存每个运输提供商的信息。
  • WAREHOUSE 表保存所有仓库的信息。

每个维度表相对较小。您的企业运营多少个仓库呢？也许 PRODUCT 和 CUSTOMER 略大一些，但条目数量可能仍以千计，而非百万。

维度表由一张记录每次销售的 SALES 表连接在一起。SALES 表包含如下内容：

  • 销售时间和金额。
  • PRODUCT-id、SELLER-id、CUSTOMER-id、CARRIER-id 和 WAREHOUSE-id。

SALES 表就是我们的事实表。SALES 中的条目数比任一维度表都要多一个数量级。而每条 SALES 表记录都引用各个维度表中的单个条目。反过来看，一个维度表的行可能引用成千上万条 SALES 表记录。

这就是星型模式的典型示例。这类数据表示在各个领域经常出现。我们称对该模式的查询为“星型查询”。例如，查询 2024-01-04 12:00 到 14:00 之间所有销售的完整报告就是一个星型查询。首先会查找 SALES 表中指定时间段内的所有条目，然后再查询各个维度表以获取每次销售的详细信息。

4.2. 高维星型查询的查询规划问题

上述N3 查询规划算法在处理维度表数量少于五或六的星型查询时效果良好且能生成最优解。然而，如果维度表数量进一步增加，那么取决于索引定义方式以及事实表与维度表的相对大小，N3 有时会找到次优计划。

这是因为 N3 仍然是贪心算法。尽管它在每一步保留最多 N 个不同的解，但只会继续扩展这 N 个最佳解。其问题如下：

找到事实表中所有相关条目的步骤可能很慢，因为事实表很大。找到事实表的相关值可能比对相对较小的维度表进行全表扫描还慢。最佳计划通常是先花时间找到事实表中的相关行，然后对相应的维度表行执行主键查找。然而由于 N3 是贪心算法，它倾向于在搜索早期在其 N 个最优计划集内积累一堆维度表的全表扫描。如果维度表数量足够多，它们可能占满全部 N 个可用槽，从而将首先对事实表进行较昂贵搜索的最佳计划挤出。

只要 N 足够大，N3 最终依然会找到最优计划。问题是，所需的 N 可能随维度表数量呈指数增长。这意味着整体查询规划算法将需要指数时间。N3 在三个或四个维度表时运行良好。但当维度表数量较多时，它开始生成包含较小维度表的全表扫描的查询计划，从而严重降低性能。

4.3. 一个大型测试案例

SQLite 源码树中的 test/starschema1.test 测试脚本展示了一个具有一张事实表和 63 张维度表的星型查询。该测试案例是人为构造的，但它确实展示了 N3 算法如何在未进行调整时失败，正如下一节所描述的那样。

4.4. 为星型查询调整 N3

为了帮助 SQLite 查询规划器为星型查询制定更好的计划，使用启发式方法来调整搜索各表时的成本估算。

截至 3.49.0 版本 (2025-02-06)，当前的启发式是人为地提高维度表的全表扫描估算成本，使其略高于查找事实表中相关行的成本。这样就能防止在算法早期，维度表的一堆无意义全表扫描将事实表搜索从 N 个最优解集中挤出，从而确保事实表搜索最终胜出。只有维度表的全表扫描成本会被调整。所有其他访问方式的估算成本保持不变，也不会对不是明显的维度表的查询表或已被 OUTER JOIN 约束的评估顺序进行任何成本调整。因此对算法的干扰最小，但足以在维度数量较多的情况下始终为星型查询找到良好解决方案。

在 SQLite 3.47.0 和 3.48.0 版本中使用的是不同的启发式。较早的启发式是降低所有事实表访问方式的成本，使它们能够击败维度表的全表扫描，从而避免在算法早期被淘汰。然而，这种人为降低事实表成本的做法会导致其他复杂联接出现性能回退。

5. 升级到 NGQP 的风险

→ 本节已过时，仅保留作历史参考。 ← 本节在 NGQP 刚推出时曾经很重要。但十年已经过去，NGQP 已成功部署到数十亿台设备上，而且每个人都完成了升级，也没有向 SQLite 开发人员报告任何性能回归。升级风险已不复存在。跳至 查询计划检查表。

对于大多数应用来说，从旧版查询计划器升级到 NGQP 几乎不需要什么思考或努力。只需将旧版 SQLite 替换为新版并重新编译，应用程序就会运行得更快。这期间没有 API 改动，也无需更改编译流程。

但和任何查询计划器的变更一样，升级到 NGQP 确实带来少量引入性能回归的风险。问题并不在于 NGQP 错误、存在缺陷或逊色于旧版查询计划器。只要有关于索引选择性的可靠信息，NGQP 应该总能挑选出至少和以前一样好、甚至更好的计划。问题在于某些应用可能在没有执行 ANALYZE 的情况下使用质量低、选择性差的索引。旧的查询计划器在每个查询上考虑的实现方案要少得多，因此它们可能凭借偶然的运气就碰巧选到了一个不错的计划。而 NGQP 则会考虑更多的查询计划可能性，它可能选出一个在理论上优于旧计划、假设索引良好时也确实工作更好，但由于数据形态的原因在实践中却导致性能回归的不同查询计划。

要点：

  • 只要能在 SQLITE_STAT1 文件中读到准确的 ANALYZE 数据，NGQP 总能比早期的查询计划器找到同样好或更好的查询计划。

  • 只要模式中不包含在索引最左侧列中具有多于约 10 或 20 个相同值的行，NGQP 将总能找到一个良好的查询计划。

并非所有应用都满足这些条件。幸运的是，即便没有这些条件，NGQP 通常仍然会找到良好的计划。不过，确实偶尔会出现性能回归的情况。

5.1. 案例研究：将 Fossil 升级到 NGQP

Fossil DVCS 是用来追踪所有 SQLite 源代码的版本控制系统。Fossil 仓库就是一个 SQLite 数据库文件。（读者可将此递归现象作为独立练习来思考。）Fossil 既是 SQLite 的版本控制系统，也是 SQLite 的测试平台。只要 SQLite 有任何增强，Fossil 通常是最先测试和评估这些增强的应用之一。所以 Fossil 是 NGQP 的早期采用者。

不幸的是，NGQP 在 Fossil 中引入了一个性能回归。

Fossil 提供的众多报告之一是显示单个分支的变更时间线，包含所有进入和离开该分支的合并。See https://sqlite.org/src/timeline?nd&n=200&r=trunk 获取此类报告的典型示例。生成这类报告通常只需几毫秒。但升级到 NGQP 后，我们发现对于主干来说，这个报告耗时接近 10 秒。

生成分支时间线所用的核心查询如下所示。（读者无需理解查询的细节。接下来会做说明。）

SELECT
     blob.rid AS blobRid,
     uuid AS uuid,
     datetime(event.mtime,'localtime') AS timestamp,
     coalesce(ecomment, comment) AS comment,
     coalesce(euser, user) AS user,
     blob.rid IN leaf AS leaf,
     bgcolor AS bgColor,
     event.type AS eventType,
     (SELECT group_concat(substr(tagname,5), ', ')
        FROM tag, tagxref
       WHERE tagname GLOB 'sym-*'
         AND tag.tagid=tagxref.tagid
         AND tagxref.rid=blob.rid
         AND tagxref.tagtype>0) AS tags,
     tagid AS tagid,
     brief AS brief,
     event.mtime AS mtime
  FROM event CROSS JOIN blob
 WHERE blob.rid=event.objid
   AND (EXISTS(SELECT 1 FROM tagxref
                WHERE tagid=11 AND tagtype>0 AND rid=blob.rid)
        OR EXISTS(SELECT 1 FROM plink JOIN tagxref ON rid=cid
                   WHERE tagid=11 AND tagtype>0 AND pid=blob.rid)
        OR EXISTS(SELECT 1 FROM plink JOIN tagxref ON rid=pid
                   WHERE tagid=11 AND tagtype>0 AND cid=blob.rid))
 ORDER BY event.mtime DESC
 LIMIT 200;

这条查询并不特别复杂，但它替代了数百甚至数千行过程式代码。查询的要点是：扫描 EVENT 表以查找满足以下三种条件之一的最近 200 个 check-in：

  1. 该 check-in 拥有 “trunk” 标签。
  2. 该 check-in 有一个子 check-in 拥有 “trunk” 标签。
  3. 该 check-in 有一个父 check-in 拥有 “trunk” 标签。

第一种条件会将所有 trunk check-in 显示出来，第二和第三种条件则会将合并到 trunk 或从 trunk 分叉出去的 check-in 都包括进来。这三个条件由查询 WHERE 子句中的三个通过 OR 连接的 EXISTS 语句实现。NGQP 出现的慢下来是由第二和第三种条件引起的。这两个条件的问题是相同的，所以我们只分析第二个。第二个条件的子查询可以重写为（做出一些微小且无关紧要的简化）：

SELECT 1
  FROM plink JOIN tagxref ON tagxref.rid=plink.cid
 WHERE tagxref.tagid=$trunk
   AND plink.pid=$ckid;

PLINK 表记录 check-in 之间的父子关系。TAGXREF 表将标签映射到 check-in。相关表结构的片段如下：

CREATE TABLE plink(
  pid INTEGER REFERENCES blob,
  cid INTEGER REFERENCES blob
);
CREATE UNIQUE INDEX plink_i1 ON plink(pid,cid);

CREATE TABLE tagxref(
  tagid INTEGER REFERENCES tag,
  mtime TIMESTAMP,
  rid INTEGER REFERENCE blob,
  UNIQUE(rid, tagid)
);
CREATE INDEX tagxref_i1 ON tagxref(tagid, mtime);

实现此查询只有两种合理方式。（其他算法虽然也存在，但没有任何一个能称得上是“最佳”的候选者。）

  1. 找出 check-in $ckid 的所有子节点，并逐个检查它们是否具有 $trunk 标签。

  2. 找出所有带有 $trunk 标签的 check-in，并逐个检查它们是否是 $ckid 的子节点。

直观上我们人类都理解算法-1 更佳。每个 check-in 通常只有很少的子节点（一个子节点最常见），每个子节点检查 $trunk 标签的时间是对数级的。事实上，算法-1 在实践中更快。但 NGQP 没有直觉，它必须依赖严谨的数学，且算法-2 在数学上略占优势。这是因为在缺乏其他信息的情况下，NGQP 必须假设 PLINK_I1 和 TAGXREF_I1 索引的质量相当且选择性相同。算法-2 使用了 TAGXREF_I1 索引的一个字段以及 PLINK_I1 索引的两个字段，而算法-1 只使用了这两个索引的第一个字段。由于算法-2 使用了更多的索引信息，NGQP 判断它是更好的算法也没有错。两者得分接近，算法-2 仅仅领先于算法-1。但在这里算法-2 真的是正确的选择。

不幸的是，在这个应用中算法-2 比算法-1 慢。

问题在于，这些索引的质量并不相同。一个 check-in 通常只有一个子节点。因此 PLINK_I1 的第一个字段通常就能将搜索缩小到一行。但标记为 “trunk” 的 check-in 多达数千甚至数万条，所以 TAGXREF_I1 的第一个字段几乎不能帮助缩小范围。

除非已经在数据库上运行过 ANALYZE，否则 NGQP 无法知道 TAGXREF_I1 在此查询中几乎毫无作用。ANALYZE 命令会收集各索引质量的统计信息，并将其存入 SQLITE_STAT1 表。掌握了这些统计信息后，NGQP 会轻松地将算法-1 视为最优算法，优势非常明显。

为什么旧版查询计划器没有选择算法-2？很简单：NN 算法根本没有考虑算法-2。规划问题的图如下所示：

在左边的“没有 ANALYZE”情况下，NN 算法将循环 P（PLINK）选为外层循环，因为 4.9 小于 5.2，于是生成了路径 P-T，即算法-1。NN 每一步只看最好的选项，因此完全忽略了 5.2+4.4 事实上比 4.9+4.8 稍微便宜一点。但 N3 算法会跟踪二表连接的前五条路径，所以最终因为整体成本略低而选择路径 T-P。路径 T-P 就是算法-2。

5.2. 解决该问题

在仓库数据库上运行 ANALYZE 立即解决了性能问题。不过我们希望 Fossil 足够健壮，无论仓库是否已被分析都能快速运行。因此我们将查询修改为使用 CROSS JOIN 运算符，而不是普通的 JOIN 运算符。SQLite 不会重新排序 CROSS JOIN 的表。这个特性在 SQLite 中由来已久，专门用于让熟悉细节的程序员强制指定某个循环嵌套顺序。一旦连接改为 CROSS JOIN（只需加一个关键字），NGQP 就会被迫始终选择更快的算法-1，无论是否已通过 ANALYZE 收集了统计信息。

我们说算法-1 “更快”，但这并不绝对。算法-1 在常见仓库中速度更快，但可以构造这样的仓库：每个 check-in 都在不同的、唯一命名的分支上，所有 check-in 都是根 check-in 的子节点。在这种情况下，TAGXREF_I1 的选择性会超过 PLINK_I1，算法-2 才是真正的更快选择。然而此类仓库在实际中极不常见，所以在这个场景下通过 CROSS JOIN 语法硬编码循环嵌套顺序是合理的解决方案。

5.3. 2017 年更新：更好的修复方案

前文写于 2013 年初，即 SQLite 3.8.0 发行之前。本段在 2021 年中旬新增。虽然前面讨论的内容仍然成立，但查询计划器已有许多改进，使得本节整体已无太大意义。

2017 年，Fossil 增加了对新的 PRAGMA optimize 语句的使用。每当 Fossil 准备关闭与仓库的数据库连接时，都会先执行 “PRAGMA optimize”，这将根据需要触发 ANALYZE。通常不需要 ANALYZE，因此这样做不会带来可测的性能损失。但偶尔也会针对仓库数据库中的部分表运行 ANALYZE。因此像本节所述的查询规划问题在 Fossil 中不再出现。由于 ANALYZE 会定期运行以保持 sqlite_stat1 表最新，因此无需手动微调查询。我们多年都不用再调整 Fossil 中的任何查询。

因此，当前避免此类问题的建议是：在关闭每个数据库连接之前运行一次 “PRAGMA optimize”。或者，如果你的应用是长期运行并从不关闭数据库连接，那么每天左右运行一次 “PRAGMA optimize”。在任何模式变更之后也请考虑运行一次 “PRAGMA optimize”。

6. 避免或修复查询计划器问题的检查清单

  1. 别惊慌！ 查询计划器挑选出较差计划的情况实际上很少见。你很可能在自己的应用中都不会遇到相关问题。如果目前没有性能问题，就无需担心这些事情。

  2. 创建合适的索引。 大多数 SQL 性能问题并非由查询计划器导致，而是因为缺少合适的索引。确保所有大型查询都有索引可用。大多数性能问题都可以通过一两个 CREATE INDEX 命令解决，而无需更改应用程序代码。

  3. 避免创建质量低的索引。（此检查清单中的“质量低”是指在索引最左侧列中存在超过 10 或 20 行具有相同值的情况。）特别要避免将布尔值或“枚举”列用作索引最左侧列。

    本文件上一节描述的 Fossil 性能问题就是因为 TAGXREF_I1 索引的最左侧列（TAGID）在 TAGXREF 表中有超过一万条相同的值。

  4. 如果必须使用质量低的索引，请务必运行 ANALYZE。 只要查询计划器知道某个索引质量不佳，它就不会被误导。而查询计划器如何知道？靠的是 SQLITE_STAT1 表的内容，而该内容由 ANALYZE 命令计算得出。

    当然，ANALYZE 只有在数据库中已有相当数量内容时才有效。若刚创建一个预计会积累大量数据的新数据库，可以先运行 “ANALYZE sqlite_schema” 来创建 SQLITE_STAT1 表，然后用普通 INSERT 语句预先填充 sqlite_stat1 表，使其包含描述应用典型数据库的内容——也许是从实验室中对一个数据充分的模板数据库运行 ANALYZE 后导出的内容。或者，在关闭数据库连接前先运行 “PRAGMA optimize”，这样 ANALYZE 就会自动在需要时运行，以保持 sqlite_stat1 表的最新状态。

  5. 归档你的代码。 添加逻辑，让你可以快速、轻松地知道哪些查询耗时过长。然后就针对这些具体查询进行优化。

    2024 年更新：多年来查询计划器已经提升得非常厉害，你基本无需使用下面列出的任何技巧。这些能力仍保留以兼容旧版本，但你不应该再使用它们。如果你确实发现了某个查询结果欠佳的情况，请在 SQLite 论坛 上向 SQLite 开发人员报告，以便他们努力修复该问题。换句话说：

    此处停止阅读！

    为了鼓励你停止阅读，本清单的剩余部分现在已灰显。

  6. 使用 unlikely() 和 likelihood() SQL 函数。SQLite 通常假设 WHERE 子句中无法被索引利用的条件很可能为真。如果这个假设不正确，可能会导致次优的查询计划。unlikely() 和 likelihood() 函数可用来向查询计划器提示那些很可能不成立的 WHERE 条件，从而帮助选择最优计划。

  7. 对可能在未分析的数据库中使用低质量索引的查询，使用 CROSS JOIN 语法来强制指定特定的循环嵌套顺序。SQLite 对 CROSS JOIN 运算符有特殊处理，强制使左边的表作为右边表相对于的外层循环。

  8. 使用一元 “+” 运算符来使 WHERE 子句某些条件失效。如果查询计划器执意为某个查询选取质量低的索引，而可用的更高质量索引却未被采用，那么在 WHERE 子句中谨慎使用一元 “+” 运算符可以将查询计划器引导远离该低质量索引。尽可能避免使用这一技巧，尤其不要在应用开发初期就使用。注意，如果涉及 类型亲和性，在等式表达式中添加一元 “+” 可能会改变该表达式的结果。

  9. 使用 INDEXED BY 语法来强制在问题查询上选择特定索引。和前面两点一样，如无必要请避免这样做，尤其不要在开发早期阶段就进行这样的明显过早优化。

7. 总结

SQLite 中的查询计划器通常非常善于为你的 SQL 语句挑选快速的算法。无论是老旧的查询计划器还是新的 NGQP 都是如此。有时因为信息不全，查询计划器可能仍会选出次优计划。NGQP 出现这种情况的概率比旧版计划器更低，但仍可能发生。只有在那些极少数情况下，应用开发者才需要介入并帮助查询计划器做出正确的决定。在常见情形下，NGQP 只是为 SQLite 带来的一项新增强，使应用运行更快，同时无需开发者额外思考或操作。
