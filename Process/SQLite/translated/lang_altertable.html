SQLite 支持 ALTER TABLE 的有限子集。 ALTER TABLE 命令允许对现有表进行以下修改： 可以重命名； 可以重命名列； 可以添加列； 或者 可以删除列。

2. ALTER TABLE 重命名

RENAME TO 语法会将 table-name 的名称更改为 new-table-name。 该命令不能用于将在附加数据库之间移动表，只能在同一数据库中重命名表。 如果被重命名的表已经存在触发器或索引，它们在重命名后仍会附属于该表。

兼容性说明： 在版本 3.25.0（2018-09-15）和 3.26.0（2018-12-01）中增强了 ALTER TABLE 在重命名表时的行为， 以便将重命名操作传播到引用该表的触发器和视图中。这被视为一种改进。 依赖早期（可争议的错误）行为的应用可以通过 PRAGMA legacy_alter_table=ON 语句或 SQLITE_DBCONFIG_LEGACY_ALTER_TABLE 配置参数在 sqlite3_db_config() 接口上启用 ALTER TABLE RENAME， 使其在版本 3.25.0 之前保持原始行为。

从 3.25.0（2018-09-15）起，触发器主体和视图定义中对该表的引用也会被重命名。

在 3.26.0（2018-12-01）之前，对被重命名表的外键引用只有在 PRAGMA foreign_keys=ON 即启用外键约束时才会更新。 换句话说，如果 PRAGMA foreign_keys=OFF，则外键约束不会在“父表”重命名时更新。 自 3.26.0 起，只要没有启用 PRAGMA legacy_alter_table=ON，重命名表时始终会转换外键约束。 下表汇总了差异：

3. ALTER TABLE 重命名列

RENAME COLUMN TO 语法会将表 table-name 的 column-name 更改为 new-column-name。 列名在表定义及引用该列的所有索引、触发器和视图中同时更改。 如果这种更名会在触发器或视图中导致语义歧义，则 RENAME COLUMN 会报错并且不进行任何更改。

4. ALTER TABLE 添加列

ADD COLUMN 语法用于向现有表追加新列。 新列总是附加在现有列列表的末尾。 column-def 规则定义了新列的特性。 新列可以采用 CREATE TABLE 语句中允许的任意形式，但有以下限制：

  • 该列不能具有 PRIMARY KEY 或 UNIQUE 约束。
  • 该列的默认值不得是 CURRENT_TIME、CURRENT_DATE、 CURRENT_TIMESTAMP 或括号内的表达式。
  • 如果指定了 NOT NULL 约束，则该列必须具备非 NULL 的默认值。
  • 如果启用了外键约束并且添加带有REFERENCES 子句的列，则该列必须具有 NULL 默认值。
  • 该列不能是GENERATED ALWAYS ... STORED，但允许 VIRTUAL 列。

当添加带有CHECK 约束的列，或在生成列上添加 NOT NULL 约束时，新增约束会针对已有行进行验证，如果验证未通过则 ADD COLUMN 会失败。 这一点在 SQLite 3.37.0（2021-11-27）中引入的新增强功能。

ALTER TABLE 通过修改存储在sqlite_schema 表中的 SQL 文本来工作。 对于重命名或不带约束的列添加，不会更改表内容。 因此，此类 ALTER TABLE 命令的执行时间与表中的数据量无关，即使是拥有千万行的表也能像仅有一行的表一样快速完成。 当添加带 CHECK 约束的新列、添加带 NOT NULL 的生成列或删除列时，必须读取（用于验证）或写入（用于删除）全部现有数据。 在那些情况下，ALTER TABLE 命令的耗时与被修改表的内容量成正比。

执行 ADD COLUMN 后，数据库将无法被 SQLite 3.1.3（2005-02-20）及更早版本读取。

5. ALTER TABLE 删除列

DROP COLUMN 语法用于从表中移除已有列。 DROP COLUMN 命令会删除指定列并重写其内容，清除与该列相关的数据。 该命令只在列未被架构其他部分引用、不是 PRIMARY KEY、且没有 UNIQUE 约束时有效。 DROP COLUMN 可能失败的原因包括：

  • 该列是 PRIMARY KEY 或其部分。
  • 该列具有 UNIQUE 约束。
  • 该列已经建立索引。
  • 该列在部分索引的 WHERE 子句中被引用。
  • 该列出现在与其无关的表或列的CHECK 约束中。
  • 该列被外键约束使用。
  • 该列出现在生成列的表达式中。
  • 该列在触发器或视图中出现。

5.1. 其原理

SQLite 将架构以文本形式存储在sqlite_schema 表中。 所有 ALTER TABLE 命令都会修改该文本并重析整个架构。只有在修改后架构仍然有效时命令才会成功。 对于 DROP COLUMN，只修改 CREATE TABLE 语句中的列定义，删除后如果架构其他部分仍存在该列的引用，则会导致 DROP COLUMN 失败。

6. 使用 PRAGMA writable_schema=ON 关闭错误检查

正常情况下一旦 ALTER TABLE 遇到无法解析的sqlite_schema 表条目就会失败并回滚。 例如，如果与名为 "tbl1" 的表相关联的视图或触发器存在语法错误，尝试将 "tbl1" 改名为 "tbl1neo" 会因解析失败而全部失败。

从 SQLite 3.38.0（2022-02-22）开始，可以通过设置 "PRAGMA writable_schema=ON;" 关闭此错误检查。 在可写架构模式下，ALTER TABLE 会自动忽略任何无法解析的 sqlite_schema 行。

7. 进行其它类型的表结构更改

SQLite 直接支持的唯一模式改变命令是上述的 "重命名表"、"重命名列"、 "添加列"、"删除列"。 但应用仍可以通过简单的操作序列对某张表 X 的格式进行任意修改。 修改表 X 模式的步骤如下：

  1. 如果启用了外键约束，请使用 PRAGMA foreign_keys=OFF 关闭它们。

  2. 开启事务。

  3. 记录与表 X 关联的所有索引、触发器和视图的定义。这些信息将在第 8 步中使用。 一种方式是执行如下查询： SELECT type, sql FROM sqlite_schema WHERE tbl_name='X'.

  4. 使用 CREATE TABLE 构造一个新的表 "new_X"，其格式为修改后的设计，确保名称不与现有表冲突。

  5. 使用类似 INSERT INTO new_X SELECT ... FROM X 的语句将 X 中的数据转移到 new_X。

  6. 删除旧表 X：DROP TABLE X。

  7. 使用 ALTER TABLE new_X RENAME TO X 将 new_X 重命名回 X。

  8. 使用 CREATE INDEX、CREATE TRIGGER 和 CREATE VIEW 重建与表 X 关联的索引、触发器和视图。可参考第 3 步保存的旧定义，按需做出调整。

  9. 若视图引用表 X 并受到模式更改影响，请使用 DROP VIEW 删除，然后用 CREATE VIEW 创建更新后的视图。

  10. 若原始启用了外键约束，请运行 PRAGMA foreign_key_check 验证更改未破坏外键。

  11. 提交第 2 步打开的事务。

  12. 若原先启用了外键约束，现在重新启用它们。

注意： 务必严格按照上述流程执行。下图总结了两种修改表定义的步骤。乍看之下两者似乎等效，但右侧流程并非总能成功，尤其在 3.25.0 和 3.26.0 增强的重命名表能力下。右侧流程中先把旧表重命名为临时名可能会破坏触发器、视图以及外键中对该表的引用。左侧安全流程先通过新临时表构造新结构，再重命名到最终名称，避免破坏链接。

上述 12 步的通用 ALTER TABLE 流程可处理即使会改变表中信息的架构修改。因此，此流程适用于删除列、改变列顺序、添加或移除 UNIQUE/PRIMARY KEY、添加 CHECK/FOREIGN KEY/NOT NULL 约束，或更改列的数据类型等场景。然而，对于不会影响实际磁盘内容的更改，可以选择更简单更快的流程。下列流程适用于移除 CHECK/FOREIGN KEY/NOT NULL 约束，或添加/移除/更改列默认值等调整。

  1. 开始事务。

  2. 运行 PRAGMA schema_version 获取当前模式版本号，后续第 6 步需要使用该值。

  3. 使用 PRAGMA writable_schema=ON 激活可写架构。

  4. 执行 UPDATE 语句修改表 X 在 sqlite_schema 表 中的定义： UPDATE sqlite_schema SET sql=... WHERE type='table' AND name='X';

    注意： 直接编辑 sqlite_schema 可能因语法错误导致数据库损坏不可读。建议先在单独测试库中仔细检验 UPDATE 语句，之后再在重要数据上执行。

  5. 若更改影响到其它表、索引、触发器或视图，请也通过 UPDATE 语句同步更新这些对象。例如修改列名时，所有引用该列的外键、触发器、索引和视图都必须更新。

    注意： 再次提醒，直接修改 sqlite_schema 存在潜在风险。务必先在测试数据库中演练整个流程，或在操作前备份重要数据库。

  6. 使用 PRAGMA schema_version=X 将模式版本加 1，其中 X 是第 2 步记录的旧版本加 1。

  7. 使用 PRAGMA writable_schema=OFF 关闭可写架构。

  8. （可选）运行 PRAGMA integrity_check 验证架构更改未破坏数据库。

  9. 提交第 1 步开始的事务。

若未来版本的 SQLite 引入新的 ALTER TABLE 功能，很可能采用上述两种流程之一。

8. 为何 ALTER TABLE 对 SQLite 而言如此棘手

大多数 SQL 引擎将解析后的架构存储在系统表中，因此 ALTER TABLE 只需修改相应系统表即可。

而 SQLite 的不同之处在于它将架构原始的 CREATE 语句文本存储在 sqlite_schema 表中。因此 ALTER TABLE 需要修改文本，这对某些“创造性”架构设计来说比较棘手。

将架构作为文本存储对嵌入式数据库有优势：架构占据的空间更少。一个常见的 SQLite 使用模式是使用多个小型独立的数据库文件，而非一个全局的大文件，这是客户端/服务器数据库常见的做法。因为每个数据库文件都会复制架构，所以保持架构表示紧凑非常重要。

以文本形式存储架构也让实现更灵活。由于每次打开数据库时都会重新解析架构，其内部表示可以随版本迭代而改变。这点很重要，因为某些新特性需要增强内部架构表示。如果该表示暴露在数据库文件中，那么更改它将更加困难。换句话说，存储为文本有助于向后兼容，并确保新版本可以读写旧数据库文件。

将架构存为文本还使SQLite 数据库文件格式更容易定义、记录和理解。这也有助于 SQLite 文件成为长期归档的推荐存储格式。

存储为文本的缺点在于让修改架构变得棘手。因此 SQLite 中 ALTER TABLE 的支持通常落后于其它使用解析后系统表的数据库引擎。
