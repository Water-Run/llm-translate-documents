内存数据库

一般情况下，SQLite 数据库存储在一个普通的磁盘文件中。然而，在某些情况下，数据库也可能存储在内存中。

强制 SQLite 数据库完全存在于内存中的最常见方式是使用特殊文件名":memory:"打开数据库。换句话说，与其将真实磁盘文件名传递给 sqlite3_open()、sqlite3_open16() 或 sqlite3_open_v2() 函数之一，不如传入字符串 ":memory:"。例如：

rc = sqlite3_open(":memory:", &db);

这样做时不会打开任何磁盘文件，而是创建一个完全位于内存中的新数据库。只要数据库连接保持打开状态，该数据库就存在。每个 :memory: 数据库都与其他数据库互不相同。因此，打开两个数据库连接并分别使用文件名 ":memory:" 会创建两个独立的内存数据库。

只要允许指定数据库文件名，就可以在任何地方使用特殊文件名 ":memory:"。例如，它可以用作 ATTACH 命令中的文件名：

ATTACH DATABASE ':memory:' AS aux1;

请注意，为了使特殊名称 ":memory:" 生效并创建一个纯内存数据库，文件名中不能包含其他文本。因此，通过在前面附加路径名可以在磁盘文件中创建数据库，例如这样："./:memory:"。

在使用 URI 文件名 时，特殊文件名 ":memory:" 也同样适用。例如：

rc = sqlite3_open("file::memory:", &db);

或者：

ATTACH DATABASE 'file::memory:' AS aux1;

内存数据库与共享缓存

如果使用 URI 文件名 打开，就允许内存数据库使用 共享缓存。如果使用未修饰的 ":memory:" 名称来指定内存数据库，那么该数据库始终拥有私有缓存，并且只能对最初打开它的数据库连接可见。然而，可以通过以下方式由两个或多个数据库连接打开同一个内存数据库：

rc = sqlite3_open("file::memory:?cache=shared", &db);

或者：

ATTACH DATABASE 'file::memory:?cache=shared' AS aux1;

这允许不同的数据库连接共享相同的内存数据库。当然，所有共享该内存数据库的数据库连接都必须位于同一进程中。当最后一个连接关闭时，数据库会自动被删除并回收内存。

如果在单个进程中需要两个或多个既互不相同又可共享的内存数据库，可以在 URI 文件名 中使用 mode=memory 查询参数来创建一个具名内存数据库：

rc = sqlite3_open("file:memdb1?mode=memory&cache=shared", &db);

或者：

ATTACH DATABASE 'file:memdb1?mode=memory&cache=shared' AS aux1;

当以这种方式为内存数据库命名后，只有使用完全相同名称的另一个连接才会与其共享缓存。

临时数据库

当传递给 sqlite3_open() 或 ATTACH 的数据库文件名为空字符串时，就会创建一个新的临时文件来保存数据库。

rc = sqlite3_open("", &db);

ATTACH DATABASE '' AS aux2;

每次都会创建不同的临时文件，所以就像使用特殊字符串 ":memory:" 一样，两个连接到临时数据库的数据库连接也各自拥有自己的私有数据库。临时数据库在创建它们的连接关闭时会被自动删除。

尽管每个临时数据库都分配了磁盘文件，但实际上临时数据库通常驻留在内存分页缓存中，因此由 ":memory:" 创建的纯内存数据库与通过空文件名创建的临时数据库之间几乎没有差别。唯一的区别在于，":memory:" 数据库必须始终保留在内存中，而临时数据库的某些部分可能在数据库变大或 SQLite 面临内存压力时被刷新到磁盘。

前面的段落描述了默认 SQLite 配置下临时数据库的行为。如果需要，也可以使用 temp_store pragma 和编译时参数 SQLITE_TEMP_STORE 来强制临时数据库表现得像纯内存数据库。
