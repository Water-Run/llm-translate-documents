1. 入门

SQLite 项目提供了一个名为 sqlite3（在 Windows 上为 sqlite3.exe） 的简单命令行程序，允许用户手动输入并执行 SQL 语句，用于操作 SQLite 数据库或 ZIP 归档。本文档简要介绍如何使用 sqlite3 程序。

1.1. SQLite 命令行程序与 SQLite 库的区别

SQLite 库是实现 SQL 数据库引擎的代码。 “sqlite3”命令行程序或“CLI”则是接受用户输入并将其传递到底层 SQLite 库进行 求值的应用程序。请理解这两者是不同的。当有人提到“SQLite”或“sqlite3”时， 他们可能说的是 SQLite 库本身，也可能指向提供人机接口的 CLI。 你常常需要根据上下文判断发言者究竟在指哪一个。

本文档讲述的是 CLI，而不是底层的 SQLite 库。

1.2. 命令行的 GUI 替代方案

sqlite3 程序由核心 SQLite 开发者编写，其正式支持的方式是通过该程序 交互地访问 SQLite 数据库文件。 不过，有些用户可能更喜欢图形用户界面 (GUI)， 第三方提供了几款此类程序。 其中之一是 Visual DB， 它也是 SQLite 项目的赞助商：

感谢 Visual DB 帮助我们让 SQLite 对所有人更友好！

1.3. 启动命令行界面

在命令提示符下键入“sqlite3”即可启动 sqlite3 程序， 也可以在其后加上保存 SQLite 数据库的文件名 （或 ZIP 归档）。如果指定的文件不存在，将自动创建一个具有该名称的新数据库文件。 如果命令行没有指定数据库文件，则使用临时内存数据库。该内存数据库会在程序退出时删除。

启动时，sqlite3 程序会显示简短的横幅 信息，然后提示你输入 SQL。键入 SQL 语句（以分号结尾）， 按“回车”即可执行 SQL。

例如，要创建一个名为 “ex1” 的新 SQLite 数据库并包含一个名为 “tbl1” 的表，可执行： 

$ sqlite3 ex1
SQLite version 3.36.0 2021-06-18 18:36:39
Enter ".help" for usage hints.
sqlite> create table tbl1(one text, two int);
sqlite> insert into tbl1 values('hello!',10);
sqlite> insert into tbl1 values('goodbye', 20);
sqlite> select * from tbl1;
hello!|10
goodbye|20
sqlite>

通过键入系统的文件结束符（通常是 Control-D）即可终止 sqlite3 程序。若要停止长时间运行的 SQL 语句，请使用中断字符（通常是 Control-C）。

请务必在每个 SQL 命令末尾加上分号！ sqlite3 程序通过查找分号来判断 SQL 命令何时结束。 如果省略了分号，sqlite3 会提示继续输入，并等待你输入更多内容以完成 SQL 命令。 这一特性让你可以输入跨多行的 SQL 命令。例如：

sqlite> CREATE TABLE tbl2 (
   ...>   f1 varchar(30) primary key,
   ...>   f2 text,
   ...>   f3 real
   ...> );
sqlite>

1.4. Windows 上双击启动

Windows 用户可以双击 sqlite3.exe 图标，使 命令行外壳弹出终端窗口并运行 SQLite。 然而，由于双击会在没有命令行参数的情况下启动 sqlite3.exe，因此不会指定数据库文件， SQLite 会使用临时的内存数据库，程序退出时自动删除。 要使用持久磁盘文件作为数据库，请在终端窗口启动后立即输入 ".open" 命令： 

SQLite version 3.36.0 2021-06-18 18:36:39
Enter ".help" for usage hints.
Connected to a transient in-memory database.
Use ".open FILENAME" to reopen on a persistent database.
sqlite> .open ex1.db
sqlite>

上述示例会打开并使用名为 “ex1.db” 的数据库文件。 如果该文件不存在，会自动创建。建议使用完整路径以确保文件位于你认为的目录中。 请使用正斜杠作为目录分隔符。 换句话说，应使用 “c:/work/ex1.db”，而不是 “c:\work\ex1.db”。

另外，你也可以使用默认的临时存储创建一个新的数据库， 然后使用 “.save” 命令将该数据库保存到磁盘文件中： 

SQLite version 3.36.0 2021-06-18 18:36:39
Enter ".help" for usage hints.
Connected to a transient in-memory database.
Use ".open FILENAME" to reopen on a persistent database.
sqlite> ... many SQL commands omitted ...
sqlite> .save ex1.db
sqlite>

使用 “.save” 命令时务必小心，因为它会在不提示确认的情况下覆盖同名的现有数据库文件。 与 “.open” 一样，建议使用带正斜杠的完整路径以避免歧义。

3. “点命令”、SQL 及其他的规则

3.1. 行结构

CLI 的输入会被解析为如下序列：

  • SQL 语句；
  • 点命令；或
  • CLI 注释

SQL 语句是自由格式的，可以跨多行书写， 并可在任意位置嵌入空白或 SQL 注释。 它们以输入行末尾的 ';' 字符终结， 或单独一行的 '/' 字符或“go” 单词终结。 当 ';' 不在输入行末尾时，它充当 SQL 语句之间的分隔符。 结尾处的空白会被忽略以判断命令结束。

点命令的结构更为严格：

  • 它必须以 "." 开头，且位于左边距， 前面不能有任何空白。
  • 它必须完全在一行内。
  • 它不能出现在普通 SQL 语句的中间。换句话说，它 不能出现在续行提示符下。
  • 点命令没有注释语法。

CLI 还接受整行注释，以 '#' 字符开头并延伸到该行末尾。 前面不能有空白。

3.2. 点命令参数

点命令的参数按照以下规则从命令尾部解析：

  1. 丢弃末尾的换行符和任何其他尾部空白；
  2. 点命令名称之后或任何参数输入结束位置之后紧接的空白会被丢弃；
  3. 参数输入以任意非空白字符开头；
  4. 参数输入以其起始字符决定的字符作为结束符：
    • 若以单引号（'）开头，则以单引号作为结束定界符；
    • 若以双引号（"）开头，则以未转义的双引号作为结束定界符；
    • 其他起始字符，以任意空白作为结束定界符；
    • 命令尾部也可作为任何参数的结束定界符；
  5. 在双引号参数中，反斜杠转义的双引号属于参数本身；
  6. 在双引号参数内，按照传统 C 字符串的反斜杠转义序列进行翻译；
  7. 参数输入的定界符（括起的引号或空白）会被丢弃，从而得到传入的参数。

3.3. 点命令执行

点命令由 sqlite3.exe 命令行程序解释，而不是由 SQLite 自身解释。因此这些点命令不能作为参数传给 例如 sqlite3_prepare() 或 sqlite3_exec() 等 SQLite 接口。

4. 更改输出格式

sqlite3 程序能够以 14 种不同的输出格式显示查询结果：

  • ascii
  • box
  • csv
  • column
  • html
  • insert
  • json
  • line
  • list
  • markdown
  • quote
  • table
  • tabs
  • tcl

可以使用 ".mode" 点命令在这些输出格式之间切换。 默认输出模式是 “list”。在 list 模式下，每一行输出代表查询的一行结果， 每列之间由特定的分隔符字符串分隔。 默认分隔符是竖线符号 (“|”)。 当你要把查询结果传给其他程序（比如 AWK）进一步处理时，list 模式特别有用。

sqlite> .mode list
sqlite> select * from tbl1;
hello!|10
goodbye|20
sqlite>

输入不带参数的 ".mode" 可查看当前模式： 

sqlite> .mode
current output mode: list
sqlite>

使用 ".separator" 点命令可更改分隔符。例如，要将分隔符改成逗号加空格，可执行：

sqlite> .separator ", "
sqlite> select * from tbl1;
hello!, 10
goodbye, 20
sqlite>

下一次 ".mode" 命令可能会根据其参数重置 ".separator" 到某个默认值。 因此，每次切换模式后如果想继续使用非标准分隔符， 可能都需要重新执行 ".separator" 命令。

在 “quote” 模式下，输出格式为 SQL 字面量。字符串用单引号括起，内部单引号通过成对出现转义。 Blob 会以十六进制的 blob 字面量形式显示（例如：x'abcd'）。 数字以 ASCII 文本显示，NULL 显示为 “NULL”。 所有列之间由逗号隔开（或者被 ".separator" 选择的其他字符）。 

sqlite> .mode quote
sqlite> select * from tbl1;
'hello!',10
'goodbye',20
sqlite>

在 “line” 模式下，数据库的一行数据每列占据独立一行。 每行由列名、等号以及列值组成。连续记录之间由空行分隔。 下面是 line 模式的输出示例：

sqlite> .mode line
sqlite> select * from tbl1;
one = hello!
two = 10

one = goodbye
two = 20
sqlite>

在 column 模式下，每条记录以单独的一行显示， 数据以列对齐的方式排列。例如：

sqlite> .mode column
sqlite> select * from tbl1;
one       two
--------  ---
hello!    10
goodbye   20
sqlite>

在 “column” 模式（以及 “box”、“table” 和 “markdown” 模式）中， 列宽会自动调整。不过你可以使用 ".width" 命令强行指定各列宽度。 ".width" 的参数是整型，表示给每列分配的字符数。 负值表示右对齐。例如：

sqlite> .width 12 -6
sqlite> select * from tbl1;
one              two
------------  ------
hello!            10
goodbye           20
sqlite>

列宽为 0 表示自动选择宽度。 未指定的列宽默认为 0。因此，在没有参数的情况下执行 “.width” 会将所有列宽重置为 0，从而让系统自动确定所有列宽。

“column” 模式是一种表格输出格式。其他表格格式还包括 “box”、“markdown” 和 “table”： 

sqlite> .width
sqlite> .mode markdown
sqlite> select * from tbl1;
|   one   | two |
|---------|-----|
| hello!  | 10  |
| goodbye | 20  |
sqlite> .mode table
sqlite> select * from tbl1;
+---------+-----+
|   one   | two |
+---------+-----+
| hello!  | 10  |
| goodbye | 20  |
+---------+-----+
sqlite> .mode box
sqlite> select * from tbl1;
┌─────────┬─────┐
│   one   │ two │
├─────────┼─────┤
│ hello!  │ 10  │
│ goodbye │ 20  │
└─────────┴─────┘
sqlite>

列格式模式可以接受额外选项来控制格式。 “--wrap N” 选项（其中 N 是整数）会在文本长度超过 N 字符时换行。 若 N 为 0，则禁用换行。 

sqlite> insert into tbl1 values('The quick fox jumps over a lazy brown dog.',90);
sqlite> .mode box --wrap 30
sqlite> select * from tbl1 where two>50;
┌────────────────────────────────┬─────┐
│              one               │ two │
├────────────────────────────────┼─────┤
│ The quick fox jumps over a laz │ 90  │
│ y brown dog.                   │     │
└────────────────────────────────┴─────┘
sqlite>

换行会在第 N 个字符后发生， 可能会发生在单词中间。 若希望在单词边界换行，请添加 “--wordwrap on” 选项 （或缩写为 “-ww”）： 

sqlite> .mode box --wrap 30 -ww
sqlite> select * from tbl1 where two>50;
┌─────────────────────────────┬─────┐
│             one             │ two │
├─────────────────────────────┼─────┤
│ The quick fox jumps over a  │ 90  │
│ lazy brown dog.             │     │
└─────────────────────────────┴─────┘
sqlite>

“--quote” 选项会将每列的结果按 SQL 字面量方式引用，就像 “quote” 模式那样。更多选项请查阅在线帮助。

“.mode box --wrap 60 --quote” 这一命令对于通用数据库查询非常有用，因此提供了别名。 不必输入整条 27 个字符的命令，只需输入 “.mode qbox”。

另一个实用的输出模式是 “insert”。在 insert 模式下，输出格式为 SQL INSERT 语句。 使用 insert 模式可以生成稍后用于插入数据到其他数据库的文本。

指定 insert 模式时需要额外提供一个参数，即要插入的表名。例如：

sqlite> .mode insert new_table
sqlite> select * from tbl1 where two<50;
INSERT INTO "new_table" VALUES('hello',10);
INSERT INTO "new_table" VALUES('goodbye',20);
sqlite>

如果 .headers on 已启用，INSERT 输出会包含列名列表：

sqlite> .mode insert mytable
sqlite> .headers on
sqlite> select 1 a, 2 b, 3 c;
INSERT INTO mytable(a,b,c) VALUES(1,2,3);
sqlite>

其他输出模式还包括 “csv”、“json” 和 “tcl”，可以自行尝试查看效果。

4.1. 控制行结束符

Windows 上默认的行结束符可能是 "\r\n"（CRLF） 或 "\n"（NL）。行结束符由 ".crlf" 点命令控制。 使用 ".crlf on" 设为 CRLF，使用 ".crlf off" 设为 NL。 按照 Windows 传统，CRLF 是默认值。然而这会让部分输出与非 Windows 平台不同， 因为多了 "\r" 字符。若希望在 Windows 上获得与其他系统一致的输出，请运行 ".crlf off"。

在非 Windows 平台上，".crlf" 命令无效且始终为 “off”。 对于 CSV 输出，行结束符始终是 "\r\n"，这是由 RFC-4180 的要求决定的。

4.2. 输出中的控制字符

默认情况下（自 3.50.0 版本起），CLI 会避免显示位于 U+0001 到 U+001f 范围内的大多数控制字符， 即便这些字符位于 CLI 要输出的内容字符串中。 这是为了避免在输出被显示在支持 ANSI 转义码的设备上时出现问题。 CLI 会忽略大多数控制字符，但对以下例外：制表符（0x09）、换行（0x0a）与 CRLF（0x0d,0x0a）保持原样。

默认行为是将控制字符像 Linux 的 “cat -A” 命令一样显示： 对控制字符 X，显示两个字符 “^Y”，其中 Y 为 X+0x40。 例如，“ESC” 字符（0x1b）显示为 “^[”， 退格字符（0x08）显示为 “^H”。

控制字符显示由 CLI 的 “--escape T” 命令行选项或 “.mode” 点命令控制，其中 T 表示控制字符处理算法。 默认的 T 是 “ascii”。若将其改为 “--escape symbol”，控制字符会映射为 Unicode 可打印字符，范围在 U+2401 到 U+241f。例如，ESC 显示为 “␛”，退格显示为 “␈”。 设置 “--escape off” 会关闭所有控制字符映射，使 CLI 输出与数据库中所存字符完全一致。

5. 查询数据库模式

sqlite3 程序提供了几个方便的命令，用于查看数据库模式。 这些命令并没有做任何其他方式不能实现的事，纯粹是快捷方式。

例如，要查看数据库中的表列表，可以输入 ".tables"。

sqlite> .tables
tbl1 tbl2
sqlite>

“.tables” 命令类似于先设置 list 模式然后执行以下查询：

SELECT name FROM sqlite_schema
WHERE type IN ('table','view') AND name NOT LIKE 'sqlite_%'
ORDER BY 1

但 “.tables” 还更多。它会查询 sqlite_schema 表，列出所有附加的数据库，而不只是主数据库。并且它会将输出格式整理成整齐的列。

“.indexes” 命令也以类似方式列出所有索引。如果给定了参数（一个表名），则仅显示该表上的索引。

“.schema” 命令显示完整的数据库模式， 如提供可选的表名参数，则仅显示该表： 

sqlite> .schema
create table tbl1(one varchar(10), two smallint)
CREATE TABLE tbl2 (
  f1 varchar(30) primary key,
  f2 text,
  f3 real
);
sqlite> .schema tbl2
CREATE TABLE tbl2 (
  f1 varchar(30) primary key,
  f2 text,
  f3 real
);
sqlite>

“.schema” 命令与先设置 list 模式再执行以下查询大致相同：

SELECT sql FROM sqlite_schema
ORDER BY tbl_name, type DESC, name

与 “.tables” 一样，“.schema” 会显示所有附加的数据库的模式。 如果你只想查看某个特定数据库（例如 “main”）的模式，可以向 “.schema” 添加参数限制输出： 

sqlite> .schema main.*

可以为 “.schema” 命令添加 “--indent” 选项，此时它会尝试重新格式化模式中的各种 CREATE 语句，使其更易读。

“.databases” 命令显示当前连接中打开的所有数据库。输出至少包含两个数据库。 第一个是 “main”，即最初打开的数据库。第二个是 “temp”，用于临时表。 如使用 ATTACH 语句附加了其他数据库，也会列出。 输出的第一列是该数据库的附加名，第二列是对应的外部文件名。 可能还有第三列，显示 "'r/o'" 或 "'r/w'"，指示数据库文件是只读还是可读写。 还可能有第四列，显示该数据库文件的 sqlite3_txn_state()。 

sqlite> .databases

“.fullschema” 点命令类似于 “.schema”，会显示整个数据库模式。 但它还包括统计表 “sqlite_stat1”、“sqlite_stat3” 和 “sqlite_stat4”（若存在）的转储。 “.fullschema” 通常提供了准确重现特定查询计划所需的全部信息。 当向 SQLite 开发团队报告查询规划器问题时，请附带完整的 “.fullschema” 输出。 注意 sqlite_stat3 与 sqlite_stat4 表包含索引项样本，因此可能包含敏感数据，因此不要通过公开渠道发送专有数据库的 “.fullschema” 输出。

6. 打开数据库文件

“.open” 命令在关闭先前打开的数据库连接后打开一个新连接。 最简单的形式是将 sqlite3_open() 应用于命令参数指定的文件。 使用名称 ":memory:" 可打开一个在 CLI 退出或再次运行 “.open” 时消失的内存数据库。 若不提供名称，则会打开一个私有的、临时的磁盘数据库，且在退出或调用 “.open” 后消失。

如果在 “.open” 中包含 --new 选项，则在打开前会清空数据库。 任何之前的数据都会被销毁。这是一次破坏性的覆盖操作，不会提示确认，请谨慎使用。

若包含 --ifexists 选项，则只有当数据库文件已存在时 “.open” 才会执行。 也就是说，--ifexists 阻止创建新的空数据库。

--readonly 选项以只读模式打开数据库，禁止写操作。

--deserialize 选项会将磁盘文件中的全部内容读入内存，然后通过 sqlite3_deserialize() 接口将其作为内存数据库打开。 对于大型数据库这会消耗大量内存。此外，对数据库的任何修改除非通过 “.save” 或 “.backup” 显式保存，否则不会写回磁盘。

--append 选项会把 SQLite 数据库追加到已存在文件末尾，而不是作为独立文件运行。 详情请参阅 appendvfs 扩展。

--zip 选项会把指定的输入文件视作 ZIP 归档，而不是 SQLite 数据库文件。

--hexdb 选项会从后续的输入行以十六进制格式读取数据库内容，而不是从磁盘上的单独文件读取。 “.dbtotxt” 点命令或 dbtotxt 命令行工具可用于生成相应的文本。 --hexdb 选项主要供 SQLite 开发者用于测试。我们尚未发现此选项在内部测试和开发之外的实际用途。

7. 重定向 I/O

7.1. 将结果写入文件

默认情况下，sqlite3 会将查询结果发送到标准输出。 可以使用 “.output” 与 “.once” 命令更改这一行为。 只需为 .output 提供输出文件名，之后所有查询结果就会写入该文件。 也可以使用 .once 命令，在执行下一条命令时将输出重定向到文件， 命令执行完后输出恢复到控制台。 使用没有参数的 .output 可恢复向标准输出发送结果。例如：

sqlite> .mode list
sqlite> .separator |
sqlite> .output test_file_1.txt
sqlite> select * from tbl1;
sqlite> .exit
$ cat test_file_1.txt
hello|10
goodbye|20
$

若 “.output” 或 “.once” 的文件名以管道符 (“|”) 开头，则后面的字符会被视作命令， 并将输出发送给该命令。这样可以轻松将查询结果管道传递给其他进程。例如， Mac 上的 “open -f” 命令会打开文本编辑器以显示其从标准输入读取的内容。 要在文本编辑器中查看查询结果，可以输入：

sqlite> .once | open -f
sqlite> SELECT * FROM bigTable;

若 “.output” 或 “.once” 命令包含 “-e” 参数，则会先将输出收集到临时文件， 再用系统默认文本编辑器打开该文件。因此，“.once -e” 与 “.once '|open -f'” 效果相同， 但更加跨平台。

若参数为 “-x”，则会将输出以 CSV 格式累积到临时文件， 然后在默认系统程序（通常是电子表格）中打开。 这是一种快速将查询结果发送到电子表格的方式： 

sqlite> .once -x
sqlite> SELECT * FROM bigTable;

“.excel” 命令是 “.once -x” 的别名，作用完全相同。

“.output” 或 “.once” 的 “-w” 选项会在浏览器中显示输出。 “.www” 命令是 “.once -w” 的别名。通常浏览器中显示的是 HTML 表格， 通过添加 “--plain” 参数可以改为纯文本显示。 

sqlite> .www
sqlite> SELECT * FROM users WHERE email LIKE '%@aol.com';

7.2. 从文件读取 SQL

在交互模式下，sqlite3 会从键盘读取输入文本（SQL 语句或 点命令）。 当然你也可以在启动 sqlite3 时重定向输入到文件，但那样就无法与程序交互。 有时需要从文件运行 SQL 脚本的同时从命令行输入其他命令。 为此提供了 ".read" 点命令。

“.read” 命令接收一个参数，通常是要读取输入文本的文件名。 

sqlite> .read myscript.sql

“.read” 命令会暂时停止从键盘读取，改为从指定文件读取输入。 读到文件末尾后会恢复从键盘读取。脚本文件可以包含点命令， 就像普通的交互输入一样。

若参数以 “|” 字符开头，则不会作为文件打开，而是执行该命令（去掉前导“|”）， 并使用该命令的输出作为输入。因此，如果你有一个产生日志 SQL 的脚本， 可以像下面这样直接执行其中的 SQL： 

sqlite> .read |myscript.bat

7.3. 文件 I/O 函数

命令行外壳添加了两个应用定义的 SQL 函数， 分别用于将文件内容读入表列，以及将列内容写入文件。

readfile(X) SQL 函数读取名为 X 的文件的全部内容，并将其作为 BLOB 返回。 这可用于将内容加载到表中。例如： 

sqlite> CREATE TABLE images(name TEXT, type TEXT, img BLOB);
sqlite> INSERT INTO images(name,type,img)
   ...>   VALUES('icon','jpeg',readfile('icon.jpg'));

writefile(X,Y) SQL 函数将 blob Y 写入名为 X 的文件，并返回写入的字节数。 可用此函数将单列内容提取到文件中。例如： 

sqlite> SELECT writefile('icon.jpg',img) FROM images WHERE name='icon';

请注意 readfile(X) 与 writefile(X,Y) 是扩展函数，不内置于核心 SQLite 库。 这些函数可作为可加载扩展使用， 源文件在ext/misc/fileio.c 的 SQLite 源代码仓库中。

7.4. edit() SQL 函数

CLI 有另一个内置 SQL 函数 edit()。edit() 接受一个或两个参数。 第一个参数是一个值，通常是需要编辑的大段多行文本。 第二个参数是文本编辑器的调用（可包含选项）。若省略第二个参数， 则使用 VISUAL 环境变量。edit() 会将第一个参数写入临时文件， 调用编辑器编辑该文件，编辑完成后重新读取内容并返回。

edit() 可用于编辑大文本值。例如： 

sqlite> UPDATE docs SET body=edit(body) WHERE name='report-15';

在示例中，会将 docs.name 为 “report-15” 的条目的 docs.body 字段内容发送给编辑器。 编辑器返回后，结果会写回 docs.body 字段。

edit() 默认调用文本编辑器。但通过在第二个参数中使用其他编辑程序， 也可以让它编辑图像或其他非文本资源。例如，如果你想修改存储在表字段中的 JPEG 图像，可以运行： 

sqlite> UPDATE pics SET img=edit(img,'gimp') WHERE id='pic-1542';

edit 程序也可用作查看工具，只要忽略返回值即可。例如，要查看上述图像，可以运行： 

sqlite> SELECT length(edit(img,'gimp')) WHERE id='pic-1542';

7.5. 以 CSV 或其他格式导入文件

使用 ".import" 命令可将 CSV（逗号分隔）或类似格式的数据导入 SQLite 表。 “.import” 接收两个参数，分别是数据源和目标表名。 源参数是要读取的文件名，或以 “|” 开头的命令，该命令会生成输入数据。

注意，在运行 “.import” 前设置好 “mode” 很重要， 可防止命令行外壳试图把文件文本解释为与实际结构不符的格式。 若使用 --csv 或 --ascii 选项，它们会控制导入输入的定界符。 否则，使用当前输出模式下设置的定界符。

若要导入位于非 “main” 模式下的表，可使用 --schema 选项指定表所属的模式。 这在导入 ATTACH 的数据库或临时表时很有用。

“.import” 对首行的处理取决于目标表是否已存在。 若不存在，则会自动创建表，第一行数据用于设置所有列名， 此时表数据来自第二行及以后。如果目标表已存在， 则从第一行开始的所有行都视为实际数据。 若输入文件包含列标签行，可使用 “--skip 1” 选项跳过首行。

例如，下面将已有临时表从含列名的 CSV 文件加载： 

sqlite> .import --csv --skip 1 --schema temp C:/work/somedata.csv tab1

除非使用 'ascii' 模式，否则 “.import” 会按照 RFC 4180 的规范将输入解析为记录和字段，唯一的例外是： 字段和记录分隔符由当前模式或 ".separator" 命令决定。 字段总会进行引号移除，以逆转 RFC 4180 的引用。 在 ascii 模式下不会去除引号。

若要使用任意定界符且不进行引用，请先设置 ascii 模式（“.mode ascii”）， 然后使用 ".separator" 设置字段和记录分隔符。 这样在 ".import" 时就不会进行反引用，数据会依据你指定的分隔符拆分字段和记录。

7.6. 导出为 CSV

要将 SQLite 表（或部分）导出为 CSV，只需将 “mode” 设为 “csv”，然后运行查询获取所需行。 输出会根据 RFC 4180 格式化为 CSV。 

sqlite> .headers on
sqlite> .mode csv
sqlite> .once c:/work/dataout.csv
sqlite> SELECT * FROM tab1;
sqlite> .system c:/work/dataout.csv

上述示例中，“.headers on” 会使输出在第一行打印列标签。 因此生成的 CSV 文件的第一行是列标签。若不需要列标签，则设为 “.headers off”。 （“.headers off” 是默认设置，如果之前没有打开 headers 可省略。）

“.once FILENAME” 命令会让所有查询输出写入指定文件，而不是显示在控制台。 在示例中，这会将 CSV 内容写入名为 “C:/work/dataout.csv” 的文件。

示例中的最后一行（“.system c:/work/dataout.csv”）与在 Windows 中双击该文件的效果相同。 通常会打开电子表格程序来显示 CSV 文件。

该命令在 Windows 上才能照常运行。 在 Mac 上等价命令是： 

sqlite> .system open dataout.csv

在 Linux 等类 Unix 系统中，需要输入类似这样的命令： 

sqlite> .system xdg-open dataout.csv

7.6.1. 导出为 Excel

为了简化导出到电子表格，CLI 提供了 “.excel” 命令，它会捕获单次查询的输出并将其发送给主机上的默认电子表格程序。 用法如下： 

sqlite> .excel
sqlite> SELECT * FROM tab;

上述命令会将查询结果以 CSV 写入临时文件， 调用默认的 CSV 处理程序（通常是 Excel 或 LibreOffice）打开该文件， 然后删除临时文件。这实际上是前面描述的 “.csv”、“.once” 与 “.system” 命令序列的简写。

“.excel” 命令本质上是 “.once -x” 的别名。 “.once” 的 -x 选项会将结果以 CSV 写入带 “.csv” 后缀的临时文件， 然后调用系统默认的 CSV 处理程序。

还有一个 “.once -e” 命令，作用类似，只不过临时文件后缀是 “.txt”， 这样就会调用系统默认的文本编辑器而非电子表格。

7.6.2. 导出为 TSV（制表符分隔值）

要导出纯 TSV（无字段引用），只需在运行查询前执行 ".mode tabs"。 然而若输出含双引号字符，".import" 以 tabs 模式读取时可能无法正确理解。 若希望生成符合 RFC 4180 的 TSV（可用 ".import" 在 tabs 模式下读入）， 请先输入 ".mode csv"，然后输入 '.separator "\t"' 再执行查询。

8. 将 ZIP 归档作为数据库文件访问

除了读取和写入 SQLite 数据库文件外， sqlite3 程序还可读写 ZIP 归档。 只需在初始命令行或 “.open” 命令中将 SQLite 数据库文件名替换为 ZIP 归档文件名， sqlite3 会自动检测该文件是 ZIP 归档并以此方式打开。 无论文件后缀如何，该机制都有效。 因此你可以打开 JAR、DOCX、ODP 及任何其他实际上是 ZIP 归档的文件，SQLite 都会帮你读取。

ZIP 归档看起来像是包含如下架构的单表数据库： 

CREATE TABLE zip(
  name,     -- Name of the file
  mode,     -- Unix-style file permissions
  mtime,    -- Timestamp, seconds since 1970
  sz,       -- File size after decompression
  rawdata,  -- Raw compressed file data
  data,     -- Uncompressed file content
  method    -- ZIP compression method code
);

例如，若想查看 ZIP 归档中所有文件的压缩效率（压缩后大小占原始未压缩大小的百分比）并按压缩比从高到低排序，可以运行： 

sqlite> SELECT name, (100.0*length(rawdata))/sz FROM zip ORDER BY 2;

或者使用文件 I/O 函数，可以从 ZIP 归档中提取内容： 

sqlite> SELECT writefile(name,content) FROM zip
   ...> WHERE name LIKE 'docProps/%';

8.1. ZIP 归档访问的实现方式

命令行外壳使用Zipfile 虚拟表来访问 ZIP 归档。你可以在打开 ZIP 归档时运行 “.schema” 命令来查看： 

sqlite> .schema
CREATE VIRTUAL TABLE zip USING zipfile('document.docx')
/* zip(name,mode,mtime,sz,rawdata,data,method) */;

当打开一个文件，如果命令行客户端发现该文件是 ZIP 归档而非 SQLite 数据库， 它实际上会先打开一个内存数据库， 然后在这个内存数据库中创建一个与 ZIP 归档关联的Zipfile 虚拟表 实例。

ZIP 归档的特殊处理是命令行外壳的手段，而非核心 SQLite 库。若要在你的应用中将 ZIP 归档当作数据库打开， 需要启用Zipfile 虚拟表模块，然后执行相应的 CREATE VIRTUAL TABLE 语句。

9. 将整个数据库转换为文本文件

使用 “.dump” 命令可将整个数据库内容转换为单个 UTF-8 文本文件。 此文件可通过管道重定向回 sqlite3 以还原为数据库。

制作数据库归档备份的好方法如下：

$ sqlite3 ex1 .dump | gzip -c >ex1.dump.gz

该命令会生成名为 ex1.dump.gz 的文件，包含完整的数据库重建所需内容，可在以后或其它机器上恢复。 若要恢复数据库，只需输入：

$ zcat ex1.dump.gz | sqlite3 ex2

由于文本格式是纯 SQL，你也可以使用 .dump 命令将 SQLite 数据库导出到其他流行的 SQL 数据库引擎。例如：

$ createdb ex2
$ sqlite3 ex1 .dump | psql ex2

10. 从损坏的数据库恢复数据

与 “.dump” 命令类似，“.recover” 也尝试将整个数据库文件转换为文本。 但不同之处在于，它不是通过正常的 SQL 接口读取数据， 而是从尽可能多的数据库页直接提取数据并试图重建数据库。 若数据库损坏，“.recover” 通常能恢复所有未损坏部分的数据，而 “.dump” 会在首次遇到损坏时停止。

若 “.recover” 得到某些无法归属到任何数据库表的行， 输出脚本会创建一个 “lost_and_found” 表来保存这些孤立行。 lost_and_found 表的模式如下： 

CREATE TABLE lost_and_found(
    rootpgno INTEGER,             -- root page of tree pgno is a part of
    pgno INTEGER,                 -- page number row was found on
    nfield INTEGER,               -- number of fields in row
    id INTEGER,                   -- value of rowid field, or NULL
    c0, c1, c2, c3...             -- columns for fields of row
);

“lost_and_found” 表为每个恢复出的孤立行创建一条记录。 此外，每个无法归属到任何 SQL 索引的恢复索引条目也会产生一条记录。 这是因为在 SQLite 数据库中，SQL 索引条目与 WITHOUT ROWID 表条目使用相同格式存储。

列内容
rootpgno 虽然可能无法将该行归属到具体的数据库表， 但它可能属于数据库文件中的某个树结构。 在这种情况下，会将该树结构的根页面号存储在该列中。 或者如果该行所在页面并非属于树结构，则该列保存页面号本身（即 “pgno” 列的值）。 在许多（但非全部）情况下，lost_and_found 表中 rootpgno 相同的行属于同一个表。
pgno 指示发现该行所在页面的页码。
nfield 该行包含的字段数量。
id 如果该行来自 WITHOUT ROWID 表，则此列为 NULL； 否则，该列包含该行的 64 位整数 rowid 值。
c0, c1, c2... 行中各列的值存储在这些列中。 “.recover” 命令会根据最长的孤立行创建足够多的列。

如果恢复后的数据库模式中已有名为 “lost_and_found” 的表， “.recover” 会使用名称 “lost_and_found0”。若 “lost_and_found0” 也已存在， 则依次尝试 “lost_and_found1” 等。默认名称可以通过 “--lost-and-found” 选项覆盖。例如，若希望输出脚本将表称为 “orphaned_rows”，可执行： 

sqlite> .recover --lost-and-found orphaned_rows

11. 加载扩展

你可以使用 “.load” 命令在运行时向命令行外壳添加新的自定义 应用定义 SQL 函数、 排序序列、 虚拟表 和 VFS。首先将扩展编译为 DLL 或共享库（详见 运行时可加载扩展 文档），然后输入： 

sqlite> .load /path/to/my_extension

请注意 SQLite 会根据平台自动添加适当的扩展后缀 （Windows 为 “.dll”，Mac 为 “.dylib”，大多数其他 Unix 为 “.so”）。 通常建议指定扩展的完整路径。

SQLite 会根据扩展文件名计算入口点。若要覆盖此默认选择，只需向 “.load” 命令提供第二个参数，指定入口点名称。

若要查找几个有用的扩展源代码，请参阅 SQLite 源码树中的 ext/misc 子目录。 你可以直接使用这些扩展，也可以基于它们创建自己的扩展以满足特定需求。

12. 数据库内容的加密哈希

“.sha3sum” 点命令会计算数据库内容的 SHA3 哈希。 需明确：哈希是针对数据库内容计算的，而不是对磁盘上的表示。 这意味着，例如执行 VACUUM 或类似的保持数据不变的重排，不会更改哈希。

“.sha3sum” 命令支持 “--sha3-224”、“--sha3-256”、“--sha3-384” 与 “--sha3-512” 选项， 用于指定哈希的 SHA3 版本。默认是 SHA3-256。

数据库模式（即 sqlite_schema 表）通常不包含在哈希中， 但可以通过 “--schema” 选项添加。

“.sha3sum” 命令接受一个可选的 LIKE 模式参数。 若该选项存在，仅对名称匹配该模式的表计算哈希。

“.sha3sum” 命令的实现依赖于 扩展函数 “sha3_query()”, 该扩展随命令行外壳一起提供。

13. 数据库内容自检

“.selftest” 命令试图验证数据库是否完整且未损坏。 该命令会查找名为 “selftest” 的模式表，其定义如下： 

CREATE TABLE selftest(
  tno INTEGER PRIMARY KEY,  -- Test number
  op TEXT,                  -- 'run' or 'memo'
  cmd TEXT,                 -- SQL command to run, or text of "memo"
  ans TEXT                  -- Expected result of the SQL command
);

“.selftest” 会按 selftest.tno 的顺序读取表中行。 对于每个 ‘memo’ 行，它会将 ‘cmd’ 中的文本输出。 对于每个 ‘run’ 行，它会将 ‘cmd’ 作为 SQL 执行，并将结果与 ‘ans’ 比较； 若结果不同则显示错误信息。

若不存在 selftest 表，则 “.selftest” 会运行 PRAGMA integrity_check。

“.selftest --init” 命令会在 selftest 表不存在时创建该表， 然后追加检查所有表内容 SHA3 哈希的条目。 后续的 “.selftest” 会验证数据库是否未改变。 若只需验证某个子集表不变，可先运行 “.selftest --init”， 然后删除与非恒定表相关的 selftest 行。

14. SQLite 归档支持

“.archive” 点命令与 “-A” 命令行选项提供了对 SQLite 归档格式 的内置支持。 该接口与 Unix 系统上的 “tar” 命令类似。每次调用 “.ar” 命令必须指定一个命令选项。 以下是可用于 “.archive” 的命令：

选项长选项用途
-c--create创建包含指定文件的新归档。
-x--extract从归档中提取指定文件。
-i--insert将文件添加到已存在的归档。
-r--remove从归档中移除文件。
-t--list列出归档中的文件。
-u--update仅当文件已变更时才将其添加到归档。

除命令选项外，每次调用 “.ar” 还可以指定一个或多个修饰选项。有些需要参数，有些不需要。可用的修饰选项如下：

选项长选项用途
-v--verbose在处理每个文件时列出文件名。
-f FILE--file FILE若指定，则使用 FILE 作为归档。 否则，假定当前 “main” 数据库就是要操作的归档。
-a FILE--append FILE与 --file 类似，使用 FILE 作为归档， 但通过apndvfs VFS 打开文件， 以便如果 FILE 已存在则将归档追加到末尾。
-C DIR--directory DIR若指定，则将所有相对路径解释为相对于 DIR，而不是当前工作目录。
-g--glob使用 glob(Y,X) 对归档中的名称进行模式匹配。
-n--dryrun显示执行归档操作所需的 SQL，但不真正修改任何内容。
----后续的命令行单词都视为命令参数，而非选项。

命令行用法时，将短格式命令选项紧跟在 “-A” 之后，无需空格。之后的所有参数都视为 .archive 命令的部分。 例如，下列命令等价： 

sqlite3 new_archive.db -Acv file1 file2 file3
sqlite3 new_archive.db ".ar -cv file1 file2 file3"

长格式和短格式选项可混用。例如如下两组命令等价： 

-- Two ways to create a new archive named "new_archive.db" containing
-- files "file1", "file2" and "file3".
.ar -c --file new_archive.db file1 file2 file3
.ar -f new_archive.db --create file1 file2 file3

另外，".ar" 后的第一个参数也可以是所有必须选项的短格式串联（不含 “-”）。 这时需要参数的选项会从后续命令行中读取，剩余的单词视为命令参数。例如： 

-- Create a new archive "new_archive.db" containing files "file1" and
-- "file2" from directory "dir1".
.ar cCf dir1 new_archive.db file1 file2 file3

14.1. SQLite 归档创建命令

创建新归档，会覆盖现有归档（无论是在当前 “main” 数据库还是由 --file 指定的文件）。 命令选项后的每个参数都是要添加到归档的文件。目录会递归导入。上文已有示例。

14.2. SQLite 归档提取命令

从归档中提取文件（到当前工作目录或由 --directory 指定的目录）。 其名称匹配命令参数（经 --glob 处理）的文件或目录会被提取。 如果命令选项之后没有参数，则提取所有文件和目录。 指定的目录会递归提取。若无法找到匹配的名称或模式，会报错。 

-- Extract all files from the archive in the current "main" db to the
-- current working directory. List files as they are extracted. 
.ar --extract --verbose

-- Extract file "file1" from archive "ar.db" to directory "dir1".
.ar fCx ar.db dir1 file1

-- Extract files with ".h" extension to directory "headers".
.ar -gCx headers *.h

14.3. SQLite 归档列表命令

列出归档内容。若未指定参数，则列出全部文件。 否则仅列出与参数（经 --glob 处理）匹配的文件。 当前 --verbose 选项不会影响该命令行为，但未来可能会变化。 

-- List contents of archive in current "main" db..
.ar --list

14.4. SQLite 归档插入和更新命令

--update 与 --insert 命令与 --create 类似，但不会在操作前删除当前归档。 若有同名文件，新的版本会悄然替换旧的，其它内容保持不变。

对于 --insert 会将列出的所有文件插入归档。 对于 --update，只有当文件此前不存在或其 “mtime” 或 “mode” 与归档中不同才会插入。

兼容性说明：在 SQLite 3.28.0（2019-04-16）之前，仅支持 --update 选项，但该选项行为等同于 --insert，会重新插入所有文件，不论是否变更。

14.5. SQLite 归档删除命令

--remove 命令会删除与提供的参数（经 --glob 处理）匹配的文件或目录。 提供了无法匹配任何归档项的参数时会报错。

14.6. ZIP 归档操作

若 FILE 是 ZIP 归档而非 SQLite 归档， “.archive” 命令与 “-A” 命令行选项同样有效。 这是通过zipfile 扩展实现的。 因此以下命令大致等价，仅输出格式不同：

传统命令等效的 sqlite3.exe 命令
unzip archive.zipsqlite3 -Axf archive.zip
unzip -l archive.zipsqlite3 -Atvf archive.zip
zip -r archive2.zip dirsqlite3 -Acf archive2.zip dir

14.7. 实现 SQLite 归档操作的 SQL

各种 SQLite 归档命令都通过 SQL 语句实现。 应用开发者可以通过运行适当的 SQL 轻松为自己的项目添加 SQLite 归档读写支持。

若要查看实现某个 SQLite 归档操作所用的 SQL，可以添加 --dryrun 或 -n 选项。 这会显示 SQL 但禁止其实际执行。

用于实现 SQLite 归档操作的 SQL 语句依赖于多个可加载扩展。 这些扩展都在 SQLite 源代码树的 ext/misc/ 子文件夹中可用。 完整支持 SQLite 归档所需的扩展包括：

  1. fileio.c — 该扩展提供 readfile() 与 writefile() SQL 函数，用于从磁盘读取和写入内容。 fileio.c 扩展还包括 fsdir() 表值函数，用于列出目录内容，以及 lsmode() 函数，用来将 stat() 系统调用返回的 st_mode 数值转换为类似 “ls -l” 的可读字符串。

  2. sqlar.c — 该扩展添加了 sqlar_compress() 与 sqlar_uncompress() 函数，用于在插入或提取 SQLite 归档内容时进行压缩与解压。

  3. zipfile.c — 该扩展实现了 “zipfile(FILE)” 表值函数，用于读取 ZIP 归档。 在读取 ZIP 归档而非 SQLite 归档时仅需此扩展。

  4. appendvfs.c — 该扩展实现了一个新的VFS，允许将 SQLite 数据库追加到其他文件（如可执行文件）末尾。 仅在使用 .archive 的 --append 选项时需要此扩展。

15. SQL 参数

SQLite 允许在 SQL 语句的任何字面值位置使用绑定参数。 这些参数的值通过sqlite3_bind_...() 系列 API 设置。

参数可以是命名的或未命名的。未命名参数是单个问号（"?"）。 命名参数是在 "?" 后立即跟数字（如 "?15" 或 "?123"）， 或是 "$"、":" 或 "@" 后跟字母数字名称（例如 "$var1"、":xyz"、"@bingo"）。

此命令行外壳会将未命名参数保持未绑定，意味着它们的值为 SQL NULL， 但命名参数可能会被赋值。 若存在一个 TEMP 表名为 "sqlite_parameters"，其模式如下： 

CREATE TEMP TABLE sqlite_parameters(
  key TEXT PRIMARY KEY,
  value
) WITHOUT ROWID;

若该表存在一条 key 列精确匹配参数名（包括前导的 "?"、"$"、":" 或 "@" 字符）的记录， 则该参数会被赋予 value 列的值。若不存在该记录，则参数默认为 NULL。

“.parameter” 命令用于简化该表的管理。 “.parameter init” 命令（常缩写为 ".param init"）会在 temp.sqlite_parameters 表尚不存在时创建它。 “.param list” 显示该表的所有记录。 “.param clear” 删除 temp.sqlite_parameters 表。 “.param set KEY VALUE” 与 “.param unset KEY” 则分别在该表中创建或删除记录。

“.param set KEY VALUE” 接收的 VALUE 可以是 SQL 字面量或任何能被求值的 SQL 表达式或查询。 这允许设置不同类型的值。 若求值失败，则会将该 VALUE 转义为文本并插入。 由于这种初始求值可能成功也可能失败，获取文本值的可靠方式是将其用单引号括起，以保护该值免受上述命令尾解析的影响。例如，（若不希望值为 -1365）：

.parameter init
.parameter set @phoneNumber "'202-456-1111'"

请注意双引号用于保护单引号，确保整段文本被解析为一个参数。

temp.sqlite_parameters 表仅为命令行外壳中的参数提供值。 对于直接通过 SQLite C API 运行的查询，temp.sqlite_parameters 表不起作用。 各个应用应实现自己的参数绑定逻辑。 你可以在命令行外壳的源代码中搜索 “sqlite_parameters”，查看 CLI 如何实现参数绑定，作为你实现的参考。

16. 索引推荐（SQLite Expert）

注意：此命令仍属实验性质。未来某个时点可能会被移除或以不兼容方式修改接口。

对于大多数非平凡的 SQL 数据库，性能的关键在于创建合适的 SQL 索引。 “合适的 SQL 索引”是指那些能让应用需要优化的查询运行更快的索引。 “.expert” 命令可以通过提出可能有助于特定查询的索引建议来协助你。

“.expert” 命令先执行，然后在下一行输入要分析的 SQL 查询。例如，考虑如下会话： 

sqlite> CREATE TABLE x1(a, b, c);                  -- Create table in database 
sqlite> .expert
sqlite> SELECT * FROM x1 WHERE a=? AND b>?;        -- Analyze this SELECT 
CREATE INDEX x1_idx_000123a7 ON x1(a, b);

0|0|0|SEARCH TABLE x1 USING INDEX x1_idx_000123a7 (a=? AND b>?)

sqlite> CREATE INDEX x1ab ON x1(a, b);             -- Create the recommended index 
sqlite> .expert
sqlite> SELECT * FROM x1 WHERE a=? AND b>?;        -- Re-analyze the same SELECT 
(no new indexes)

0|0|0|SEARCH TABLE x1 USING INDEX x1ab (a=? AND b>?)

上述示例中，用户创建了数据库模式（单表 x1）， 并使用 “.expert” 命令分析查询 “SELECT * FROM x1 WHERE a=? AND b>?”。 shell 工具建议用户创建新索引（x1_idx_000123a7），并输出查询在 EXPLAIN QUERY PLAN 格式下的执行计划。 用户随后创建了等效模式的索引，并再次分析相同查询。 这次工具不再推荐新索引，并显示 SQLite 针对现有索引使用的计划。

“.expert” 命令支持以下选项：

选项 用途
‑‑verbose 若提供，则对每个分析的查询输出更详尽的报告。
‑‑sample PERCENT 该参数默认值为 0，使 “.expert” 仅根据查询与数据库模式推荐索引， 类似于 SQLite 查询规划器 在未对数据库运行 ANALYZE 命令生成分布统计时的行为。
若该选项传入非零参数，“.expert” 会基于各数据库表当前存储的 PERCENT 百分比行生成类似的数据分布统计。 对于分布异常的数据库，这可能会得到更好的索引建议，尤其是在计划运行 ANALYZE 的情况下。
对于小型数据库和现代 CPU，通常没有理由不传入 “--sample 100”。 但对于大型表，收集分布统计可能很耗时。若操作过慢，请尝试减小 --sample 的值。

本节描述的功能可以通过 SQLite expert 扩展 代码集成到其他应用或工具中。

若数据库模式通过扩展加载机制引入了 SQL 自定义函数，在使用 .expert 功能时可能需要额外处理。 该功能会使用额外的连接，因此必须在这些连接中提供这些自定义函数。 可以通过以下选项描述的方法实现：在 自动加载静态链接扩展 以及 持久可加载扩展 中描述的扩展加载与使用选项。

17. 管理多个数据库连接

自 3.37.0（2021-11-27）起，CLI 可同时保持多个数据库连接打开。 一次最多只有一个连接处于活动状态。非活动连接依然保持打开但处于空闲。

使用 “.connection” 点命令（通常缩写为 “.conn”）可以查看数据库连接列表，并指出当前处于活动状态的连接。 每个数据库连接由 0 到 9 之间的整数组成（最多可同时打开 10 个连接）。 输入 “.conn” 加上编号，以创建该连接（若尚不存在）并切换过去。 输入 “.conn close N” 可关闭编号为 N 的连接。

虽然底层的 SQLite 连接彼此独立，但许多 CLI 设置（例如输出格式）在所有连接之间共享。 因此在一个连接中更改输出模式会影响所有连接。 另一方面，一些点命令（如 .open）只作用于当前连接。

18. 其他扩展功能

CLI 是在几个不包含在 SQLite 库中的扩展基础上构建的。 其中一些扩展提供以下未在前文描述的功能：

  • UINT 排序序列，可以根据嵌入文本中的无符号整数值以及其它文本进行排序；
  • 由十进制扩展提供的十进制算术；
  • generate_series() 表值函数；
  • base64() 与 base85() 函数，可将 blob 编码为 base64 或 base85 文本，或将其解码为 blob；
  • 对 POSIX 扩展正则表达式的支持，绑定至 REGEXP 操作符。

19. 其他点命令

命令行外壳还提供许多其他点命令。 关于某一特定版本和构建，可以运行 “.help” 命令查看完整列表。

20. 在 shell 脚本中使用 sqlite3

一种常用方式是在 shell 脚本中使用 “echo” 或 “cat” 生成命令文件， 然后启动 sqlite3 并重定向输入至该文件。 这种方法在很多场景下都适用。 作为额外的便利，sqlite3 允许在数据库名之后直接在命令行传入单个 SQL 命令。 当 sqlite3 以两个参数启动时，会将第二个参数传入 SQLite 库处理，查询结果以 list 模式打印到标准输出，然后退出。 该机制旨在便于与 “awk” 等程序配合。例如：

$ sqlite3 ex1 'select * from tbl1' \
>  | awk '{printf "<tr><td>%s<td>%s\n",$1,$2 }'
<tr><td>hello<td>10
<tr><td>goodbye<td>20
$

21. 标记 SQL 语句结束

SQLite 命令通常以分号终止。在 CLI 中，你也可以使用单独一行的 “GO”（不区分大小写）或 “/” 来结束命令。 这些用法分别借鉴于 SQL Server 与 Oracle，SQLite CLI 提供支持以保持兼容性。 由于 CLI 会在将输入传递给 SQLite 核心前将它们转换为分号， 因此这些方式在 sqlite3_exec() 中不起作用。

22. 关于启动命令行的更多细节

如前文所述（参见上文），CLI 通常通过键入 “sqlite3” 并跟上数据库文件名启动。 然而 “sqlite3” 程序还接受许多其他参数，而不仅仅是数据库文件名。

22.1. 额外命令行参数

数据库文件名之后出现的额外命令行参数会被视为输入文本行。 每个额外参数可以是一个 SQL 语句或点命令，按照从左到右顺序求值。 由于 SQL 语句与点命令中通常包含空格，你可能需要将每个语句或命令用单引号或双引号括起（视操作系统而定）。例如： 

$ sqlite3 test.db   ".mode box"   "SELECT * FROM users;"

以此方式提供额外参数时，标准输入不会被读取，CLI 会在处理完这些参数后退出。

CLI 对命令行选项的格式比较灵活。允许连续一个或两个 “-”。 因此 “-box” 与 “--box” 等效。 命令行选项按照从左到右的顺序处理。 因此一个 “--box” 会覆盖之前的 “--quote”。

大多数命令行选项含义显而易见，但下面几个值得补充说明。

22.3. --safe 命令行选项

--safe 命令行选项尝试禁用 CLI 中可能对主机计算机造成任何非指定数据库文件以外修改的所有功能。 其目的是让你在面对来自未知或不受信任来源的大型 SQL 脚本时，可以使用 --safe 选项运行脚本并观察其行为，而无需担心被利用。 --safe 选项会禁用（除了其他之外）以下功能：

  • .open 命令，除非使用 --hexdb 选项或文件名为 ":memory:"。 这样可防止脚本读取或写入原始命令行之外的数据库文件。
  • ATTACH SQL 命令。
  • 具有潜在副作用的 SQL 函数，如 edit()、fts3_tokenizer()、load_extension()、readfile() 与 writefile()。
  • .archive 命令。
  • .backup 与 .save 命令。
  • .import 命令。
  • .load 命令。
  • .log 命令。
  • .shell 与 .system 命令。
  • .excel、.once 与 .output 命令。
  • 其他可能带来有害副作用的命令。

基本上，任何读取或写入磁盘上非主数据库文件的 CLI 功能都会被禁用。

22.3.1. 为特定命令绕过 --safe 限制

如果在命令行同时包含 “--nonce NONCE” 选项（其中 NONCE 是任意一个较长字符串）， 那么随后执行的 ".nonce NONCE" 命令（传入相同的 nonce 字符串）将允许下一条 SQL 语句或点命令绕过 --safe 限制。

假设你要运行一个可疑脚本，而该脚本需要使用一些 --safe 通常禁用的功能。 例如，脚本需要附加额外的数据库，或需要加载特定扩展。 可以在需要执行的 ATTACH 语句或 “.load” 命令前先执行相应的 ".nonce" 命令，并在命令行中提供相同的 nonce 值。 这样这些特定命令就能正常执行，而其他不安全命令仍然受限。

使用 ".nonce" 具有危险性，因为一旦操作失误可能导致恶意脚本破坏系统。 因此，请谨慎、尽量少用，仅在无法通过其他方式在 --safe 模式下运行脚本时才作为最后手段使用。

22.4. --unsafe-testing 命令行选项

--unsafe-testing 命令行选项启用 CLI 中仅用于内部测试的功能。 --unsafe-testing 会禁用 SQLite 中的某些防御机制，例如 SQLITE_DBCONFIG_DEFENSIVE 与 SQLITE_DBCONFIG_TRUSTED_SCHEMA。 该选项还会启用可能在误用时导致数据库损坏、内存错误或 CLI/SQLite 库出现问题的功能。 例如，".testctrl assert false" 命令会故意触发断言错误，以验证断言机制是否有效。

需要使用 --unsafe-testing 的异常行为通常不会被当作 bug。

22.5. --no-utf8 与 --utf8 命令行选项

在 Windows 平台上，当使用控制台进行输入或输出时， 需要在控制台可用的编码与 CLI 内部的 UTF-8 表示之间转换。 过去的 CLI 版本引入了这些选项，以允许或禁用依赖于 Windows 控制台特性（在现代操作系统中可使其产生或接受 UTF-8）的转换。

当前版本的 CLI（3.44.1 及以后）通过 Windows 控制台 API 读取或写入 UTF-16 进行控制台 I/O。 由于该方法即便在 Windows 2000 等旧版本系统上也能正确工作， 因此这些选项已经没有必要了。尽管仍然接受，但不会带来任何效果。

在所有情况下，非控制台文本 I/O 均为 UTF-8 编码。

在非 Windows 平台上，这些选项同样被忽略。

23. 从源码编译 sqlite3 程序

在 Unix 系统或使用 MinGW 的 Windows 上编译命令行外壳，可使用常规的 configure-make 命令： 

sh configure; make

无论是从源码树还是从整合包构建，configure-make 都适用。 依赖项较少。若从源码树构建，则需要运行 tclsh。 若使用 amalgamation bundle，则预处理已完成，只需常规构建工具即可。

要让.archive 命令正常工作， 需要一个可用的 zlib 压缩库。

在 Windows 上使用 MSVC，可用 Makefile.msc 与 nmake： 

nmake /f Makefile.msc

为了正确使用.archive 命令， 请将zlib 源代码复制到源码树的 compat/zlib 子目录， 然后执行： 

nmake /f Makefile.msc USE_ZLIB=1

23.1. 自行构建

sqlite3 命令行接口的源代码集中在一个名为 “shell.c” 的文件中。 shell.c 是从其他源生成的，但大部分代码位于 src/shell.c.in 中。 （从源码树中运行 “make shell.c” 可重新生成 shell.c。） 将 shell.c（与 sqlite3 库源码）一起编译，生成可执行文件。例如：

gcc -o sqlite3 shell.c sqlite3.c -ldl -lpthread -lz -lm

建议添加以下编译时选项，以构建全功能的命令行外壳：

  • -DSQLITE_THREADSAFE=0
  • -DSQLITE_ENABLE_EXPLAIN_COMMENTS
  • -DSQLITE_HAVE_ZLIB
  • -DSQLITE_INTROSPECTION_PRAGMAS
  • -DSQLITE_ENABLE_UNKNOWN_SQL_FUNCTION
  • -DSQLITE_ENABLE_STMTVTAB
  • -DSQLITE_ENABLE_DBPAGE_VTAB
  • -DSQLITE_ENABLE_DBSTAT_VTAB
  • -DSQLITE_ENABLE_OFFSET_SQL_FUNC
  • -DSQLITE_ENABLE_JSON1
  • -DSQLITE_ENABLE_RTREE
  • -DSQLITE_ENABLE_FTS4
  • -DSQLITE_ENABLE_FTS5
