1. 执行摘要

  • 关于 SQLite 的 CVEs 可能不适用于你对 SQLite 的使用。

  • 历史上报告的所有 SQLite 漏洞至少需要满足以下先决条件之一：

    1. 攻击者可以提交并执行任意 SQL 语句。

    2. 攻击者可以向应用程序提交恶意构造的数据库文件，随后该应用程序会打开并查询它。

  • 很少有真实世界的应用程序满足这些先决条件，因此即使它们使用的是较旧且未打补丁的 SQLite 版本，也很少会受到漏洞影响。

  • SQLite 开发团队会迅速修复漏洞，通常在发现几小时内完成。如果该漏洞看起来可能影响真实应用程序，就会发布新的 SQLite 版本。

  • 灰帽黑客根据他们撰写的 CVE 数量和严重程度获得奖励。这导致大量影响轻微或根本没有影响的 CVE 出现，它们往往夸大了影响力。

  • 关于 SQLite 编写的 CVE 很少是真正的漏洞，因为它们并不会给攻击者带来新的能力。请考虑：

    1. 几乎所有针对 SQLite 编写的 CVE 都要求能够注入并执行任意 SQL。

    2. 大多数 CVE 宣称的后果是“拒绝服务”，通常是通过 NULL 指针取消引用、除以零或类似操作导致崩溃。

    3. 但是如果攻击者已经可以执行任意 SQL，他们根本不需要漏洞就能造成拒绝服务。存在大量完全合法且有效的 SQL 语句，可以无限消耗 CPU、内存和磁盘 I/O，从而在不依赖漏洞的情况下造成拒绝服务。

    4. 因此，攻击者能够注入并运行任意 SQL 本身就算是一种拒绝服务攻击。任意 SQL 可能触发 SQLite 的某个漏洞并导致崩溃，并不代表这是一个新漏洞。

  • SQLite 开发者不会撰写 CVE。任何你在 SQLite 上看到的 CVE 都是由第三方生成的，通常在核心开发者毫不知情的情况下。常见的情形是某人报告 SQLite 的一个漏洞，这会迅速被修复，而数周后该漏洞才出现一个 CVE。

  • 你不应当假设某个关于 SQLite 的 CVE 含有权威信息。CVE 往往包含不准确之处。SQLite 开发者已经尝试针对关于 SQLite 的 CVE 添加澄清和更正。

2. 关于 CVEs

CVEs（“常见漏洞和披露”）是可能允许系统被攻破的软件漏洞报告，其理念是合理的。它们提供了一种通用命名方案，使得可能危及信息安全的软件漏洞可以被轻松跟踪。

虽然 CVEs 的初衷是合理的，但目前创建和管理 CVE 的流程并不完善。有无数灰帽黑客对各种开源软件（包括 SQLite 和其他许多项目）运行 fuzz 测试，并针对发现的任何问题撰写 CVE。灰帽黑客依赖他们撰写的 CVE 数量和严重程度获得回报，可能是声望，也可能是金钱。这样的激励导致大量 CVE 出现，它们通常缺乏充分审查，并可能夸大影响。CVE 的质量控制程序无法应对如此大量的输入，导致夸大的、误导的、遗漏的或不准确的描述难以及时纠正。

这并不是说 CVE 没有用。CVE 仍然（大多数情况下）报告实际存在的漏洞。但在大多数案例中，这些漏洞并不是真正的安全漏洞，因为它们本身并不会直接导致数据泄露或系统被攻破。 报告并修复问题是件好事。但并非每个漏洞都能被每个应用程序利用。就 SQLite 而言，大多数 CVE 报告的漏洞在大多数应用中无法触及。升级到最新版本总是值得做的，但仅仅因为某个匿名的灰帽在网络上写了一个 CVE，就认为这是紧急情况并不必要。

2.1. 通常需要一个独立的 SQL 注入漏洞

处理复杂结构化输入的其他 C 语言库经常需要处理来自不受信任来源的未经过滤的输入。像 libjpeg、libzip 或 OpenSSL 这样的库直接接收可能来自敌对方的输入流。

但 SQLite 之类的数据库引擎通常不是这样。传入 SQLite 的 SQL 脚本来自（可信赖的）应用程序本身，而不是攻击者。有时应用包含漏洞，攻击者可以诱使该应用将其设计的 SQL 发送到数据库引擎。这是应用程序中的一个独立漏洞，称为SQL 注入漏洞。由于 SQL 文本是可执行代码，SQL 注入漏洞实际上是远程代码执行 (RCE) 漏洞的一个特例。相比于 Python、Shell 脚本或原始机器码，SQL 虽然强大，但在编写利用代码方面不如它们方便。因此，SQL 注入仍然是一个严重问题。

大多数针对 SQLite 编写的 CVE 都假设攻击者能够在 SQLite 中运行任意 SQL 脚本。在大多数应用中，这意味着首先必须存在一个 SQL 注入漏洞，攻击者才能注入恶意 SQL。

部分应用确实允许来自潜在敌对方的不受信任 SQL 脚本直接在 SQLite 中运行。最典型的例子是 Chrome 和 Safari 浏览器，它们通过 Javascript 的 WebSQL 功能允许匿名网页运行 SQL。此操作在沙盒内执行，并在资源使用方面受到严格限制，以防 SQL 脚本尝试占满所有内存或 CPU 资源从而造成拒绝服务。Chrome 和 Safari 拥有相应基础设施，能够让敌对方运行不会对其余系统造成损害或妥协的代码。它们不得不这么做，因为它们也运行 Javascript，若不严格控制，甚至可能造成比不受限制的 SQL 更大的破坏。除了 Chrome 和 Safari，目前 SQLite 开发者所知没有其他应用故意允许匿名远程代理运行任意 SQL 文本。

然而，大多数针对 SQLite 的 CVE 都轻率地假设攻击者可以在数据库引擎中运行任何任意 SQL。因此可以相当肯定地说，大多数 SQLite CVE 实际上只适用于 Chrome 和 Safari 中使用的 SQLite。换句话说，除非你是 Chrome 或 Safari 的开发者，否则大多数 SQLite CVE 不适用于你。

2.2. 防御灰色技术

大多数应用在使用 SQLite 时不必担心来自晦涩 SQL 输入的漏洞。如果应用控制着 SQL，并且应用并未故意破坏 SQLite，那么一切都会正常运行。 在这种情况下，没有必要使用最新且打了补丁的 SQLite 版本。任何旧版本都应该能够正常工作。

但有些情况下，应用确实需要安全地运行不受信任的 SQL。SQLite 开发者致力于使 SQLite 在这方面尽可能安全，尽管偶尔仍会出现疏漏。在这种情况下，及时跟进最新补丁是有益的。单独的《防御灰色技术》文档包含更多建议，可以帮助在 SQLite 接收直接来自不受信任来源的输入时预防零日攻击。

2.3. SQLite 开发者关于 CVEs 的政策

SQLite 开发者在漏洞报告后会立即修复所有错误，通常在数小时内完成。修复会立即在SQLite 公共源码树中更新。如果某个漏洞看起来可能对现有应用造成影响，就会发布新的补丁版本。

然而，SQLite 开发者不追踪 CVE。原因如下：

  1. 开发者通常在漏洞被修复很久之后才得知该 CVE。你可以从很多 CVE 中看到它们在初始报告中就引用了修复补丁。

  2. CVE 是关于可能影响大多数应用的 SQLite 错误的低质量信息来源。

  3. 几乎所有 CVE 报告的错误不过是普通错误，并不是真正的漏洞。将它们称为漏洞是在扩大“漏洞”一词的含义，而 SQLite 开发者不愿参与这种误导。

  4. 开发者无法对 CVE 的内容施加编辑影响，也不愿被那些他们没有发言权的群体控制。

3. 近期 SQLite CVEs 状态
