合并版本与典型源码

1. 我应该使用哪个代码库来构建 SQLite？

SQLite 是用 C 编写的。要编译 SQLite，需要使用 C 编译器将 C 语言源代码转换为二进制机器代码。

但事情并不完全如此简单。SQLite 项目在“典型源码”和“合并版本”之间做了区分。以下是 SQLite 开发者对这些术语的解释：

1.1. 典型源码 与 合并版本

  • 典型源码 或 典型源码 → 典型源码是 SQLite 的原始文本。 这些是由 SQLite 开发者录入并维护的源代码文件，并保存在源代码管理系统中。如果您从 Fossil 仓库 (https://sqlite.org/src) 或 GitHub 镜像 (https://github.com/sqlite/sqlite) 拉取 SQLite 源代码副本，就能获取 典型源码。

  • 合并版本 → 作为 SQLite 构建流程的一部分，典型源码文件会被转换并串联在一起，形成几个大的 C 语言代码文件：

    • sqlite3.c
    • sqlite3.h
    • shell.c

    这些“合并版本”源文件也是 C 代码。将多个较小的代码文件合并成几个大型 C 文件 可让 SQLite 更容易编译，并帮助 C 编译器优化代码，从而提高运行速度。

如果您希望自己编译 SQLite 并在自己的项目中使用，应该从典型源码还是合并版本开始？

2. 背景

下图展示了完整的 SQLite 构建过程：

完整的构建过程从图中左侧的典型源码文件开始。根据 SQLite 版本的不同，典型源码文件数量在 100 到 200 个之间。 其中一些典型源码文件会被预处理程序转换，以生成“中间源码文件”。Lemon 解析器生成器程序就是一个预处理器示例。典型源码文件之一是 parse.y，它是 SQLite 所支持 SQL 语法的 LALR(1) 文法。Lemon 将该文法转换为 C 代码， 实现一个下推自动机，用于将 SQL 文本解析为 抽象语法树。 SQLite 中还有许多其他程序和脚本，将开发者编辑的代码转换为 C 编译器可以理解的形式。并非所有典型源码文件都会经过预处理，但相当一部分会。

预处理完成后，会有一些脚本将大量中间源码文件以及仍剩下的典型源码文件汇总并串联起来，生成“合并版本”。当我们提到“合并版本”时，通常指的是包含 SQLite 大部分源码的 "sqlite3.c" 文件。但对应的头文件 "sqlite3.h" 也是由其他文件构建而成。SQLite 的命令行工具的源码也是包含在 "shell.c" 中的合并版本。

合并版本文件随后可以由任意普通 C 编译器处理，生成图中右侧所示的最终构建产物。（图中展示的构建产物文件名为 Windows 系统下的名称。请根据实际系统替换为相应名称。）

3. 如何从典型源码编译

要从典型源码编译 SQLite，您需要下载典型源码树的 tar 包或 ZIP 压缩包，解压后在 unix 上运行 "./configure && make"，在 Windows 上运行 "nmake /f Makefile.msc"。 更多详细信息如下：

  • 如何在 Windows 上编译

4. 如何从合并版本编译

随着每个 SQLite 版本发布，开发者会提供预构建的合并版本源文件副本，可在 下载页面获取。查找名为 "sqlite-amalgamation-NNNNNNN.zip" 和 "sqlite-autoconf-NNNNNNN.tar.gz" 的文件。第一个只包含合并版本源文件，第二个则额外包含 "configure" 脚本和 Makefile。

5. 从合并版本构建的优势

  1. 更少的文件需要处理

    您可以将所需的合并版本文件嵌入自己的源代码树，并使用自己的 Makefile 对其编译。并不一定需要单独的 "configure" 脚本或 Makefile。 只需将 "sqlite3.c" 和 "sqlite3.h" 文件添加到您的源码树中，它们就会永久存在，无需新增依赖项。需要升级时只需下载最新版本并替换源代码树中的文件即可。

  2. 下载量更小

    合并版本压缩包的体积不到完整 SQLite 源代码树的四分之一，因此下载更省流量。

  3. 无论如何最终都会使用合并版本

    如果您从典型源码构建，Makefile 会先构建合并版本，然后再将合并版本编译为二进制机器码。为何不直接跳过第一步，直接编译合并版本呢？

  4. 您使用的合并版本已由 SQLite 开发者测试并审核

    如果您直接下载预编译的合并版本文件，就可以确信这些源文件已经通过 SQLite 开发者的严格测试流程。 无需担心在将典型源码编译为合并版本时自己意外破坏了某些东西。

6. 从典型源码构建的优势

  1. 获得更多编译时选项

    SQLite 支持众多编译时选项，其中一些适用于合并版本，而另一些则不适用。根据您希望使用的编译时选项，可能需要从典型源码构建。

  2. 访问历史版本和未发布版本

    合并版本源码通常仅对最新版 SQLite 可用。如果您想获取历史版本，或想尝试包含新功能的未发布主干版本，或者希望通过参与预发布测试回馈社区，那么必须从典型源码编译。

  3. 更容易自定义代码

    如果您正在对 SQLite 进行自己定制的增强或修改，建议使用典型源码。合并版本旨在对 C 编译器友好，而非人类程序员。理论上您可以直接修改合并版本源码，但使用典型源码会让整个过程更轻松。

7. 总结：您应该选择哪个

是从合并版本还是从典型源码构建 SQLite，取决于您的具体情况。请回顾上述段落中对两种方法的优劣分析，然后自行决定。
