SQLite 中的多个小查询依然高效

1. 执行摘要

  • 对于像 MySQL、PostgreSQL 或 SQL Server 这样的客户端/服务器数据库引擎来说，每个网页 200 条 SQL 语句是过多的。

  • 但使用 SQLite，每个网页 200 条或更多 SQL 语句并非问题。

  • SQLite 也可以像客户端/服务器数据库一样高效地执行大型复杂查询，同时也能高效地处理多个较小的查询。应用开发者可以根据当前任务所需，灵活选择最适合的方法。

2. 表面上的问题

SQLite 的适当用途 页面指出，SQLite 网站上的动态页面通常每页执行大约 200 条 SQL 语句。该做法引来了读者的批评。例子如下：

  • “每页 200 条 SQL 语句是一个荒谬的高数字”

  • “对于大多数网站来说，200 条查询远远超出需求。”

  • “[这]是糟糕的设计”

对于传统的客户端/服务器数据库引擎，如 MySQL、PostgreSQL 或 SQL Server，这种批评确实有根据。在客户端/服务器数据库中，每一条 SQL 语句都需要应用程序与数据库服务器之间来回传递消息。顺序执行超过 200 次这样的往返消息，会严重拖慢性能。这种情况有时被称为“N+1 查询问题”或“N+1 选择问题”，属于一种反模式。

3. 在 SQLite 中 N+1 查询不是问题

然而，SQLite 并非客户端/服务器架构。SQLite 数据库与应用程序运行在同一进程地址空间内。查询并不涉及消息往返，只需一次函数调用。SQLite 单条 SQL 查询的延迟远低于客户端/服务器数据库。因此，在 SQLite 中使用大量查询并不会造成问题。

4. 每个网页超过 200 条 SQL 语句的必要性

SQLite 网站上的动态网页大多由 Fossil 版本控制系统 生成。典型的动态页面是诸如 https://sqlite.org/src/timeline 这样的时间线。下面展示了生成该时间线所使用的全部 SQL 的日志。

日志中的第一组查询是从 Fossil 数据库的“config”和“global_config”表中提取显示选项。随后有一个复杂的查询，用于提取将在时间线上展示的所有元素的列表。这个“时间线”查询表明，SQLite 能轻松处理涉及多张表、子查询和复杂 WHERE 子句约束的复杂关系型数据库查询，并能有效利用索引，以最小的磁盘 I/O 解决查询。

在这个大型“时间线”查询之后，接着是为每个时间线元素分别执行的查询。Fossil 采用的是“N+1 查询”的模式，而不是试图用尽可能少的查询获取全部信息。但这并无妨碍，因为并不存在不必要的 IPC 开销。在每个时间线页面的底部，Fossil 会展示生成该页面所花费的大致时间。对于包含 50 条记录的时间线，延迟通常少于 25 毫秒。分析表明，花在数据库引擎内部的时间仅占极少数毫秒。

在 Fossil 中使用 N+1 查询模式不会损害应用性能，并且该模式还具有优势。首先，构建时间线查询的那一部分代码可以与准备用于展示的每个时间线条目的部分完全分离。这种职责分离有助于保持代码简洁且易于维护。其次，展示所需的信息以及用于提取这些信息的查询依赖于将要显示的对象类型。检入记录需要一套查询，问题单需要另一套查询，Wiki 页面又需要不同的查询，依此类推。通过按需在处理不同实体的代码段中实现这些查询，就进一步实现了职责分离并简化了整体代码库。

因此，SQLite 可以执行一两个大型复杂的查询，也可以执行多个较小更简单的查询，两者都高效。应用可以根据实际情况使用其中一种或两种技术。

INSERT OR IGNORE INTO timeline SELECT
  blob.rid AS blobRid,
  uuid AS uuid,
  datetime(event.mtime,toLocal()) AS timestamp,
  coalesce(ecomment, comment) AS comment,
  coalesce(euser, user) AS user,
  blob.rid IN leaf AS leaf,
  bgcolor AS bgColor,
  event.type AS eventType,
  (SELECT group_concat(substr(tagname,5), ', ') FROM tag, tagxref
    WHERE tagname GLOB 'sym-*' AND tag.tagid=tagxref.tagid
      AND tagxref.rid=blob.rid AND tagxref.tagtype>0) AS tags,
  tagid AS tagid,
  brief AS brief,
  event.mtime AS mtime
 FROM event CROSS JOIN blob
WHERE blob.rid=event.objid

 ORDER BY event.mtime DESC LIMIT 50;
