编辑说明： 本文档于2004年编写，作为从 SQLite2 过渡到 SQLite3 的程序员指南。 它被保留为 SQLite 历史记录的一部分。 现代程序员应参考本网站其他地方提供的更新版 SQLite 文档。

SQLite 第3版概述

SQLite 3.0 引入了重要变化，包括：

  • 数据库文件格式更加紧凑。
  • 明确类型系统和 BLOB 支持。
  • 支持 UTF-8 和 UTF-16 文本。
  • 用户定义的文本排序序列。
  • 64 位 ROWID。
  • 改进的并发能力。

本文档为已经熟悉 SQLite 2.8 的用户快速介绍 SQLite 3.0 的变化。

命名更改

SQLite 2.8 版本在可预见的未来将持续获得错误修复。 为了让 SQLite 2.8 和 SQLite 3.0 和平共存，SQLite 3.0 的关键文件和 API 的名称已更改为包含字符 "3"。 例如，用于 C 程序的头文件已从 "sqlite.h" 变为 "sqlite3.h"。 用于与数据库交互的 shell 程序名称已从 "sqlite.exe" 改为 "sqlite3.exe"。 通过这些更改，可以在同一系统上同时安装 SQLite 2.8 和 SQLite 3.0，并且相同的 C 程序也可以同时链接这两个版本并同时使用它们。

新文件格式

SQLite 数据库文件使用的格式已被彻底修订。 旧的 2.1 版本格式与新的 3.0 格式彼此不兼容。 SQLite 2.8 版不能读取 3.0 版本的数据库文件，SQLite 3.0 也不能读取 2.8 版本的数据库文件。

要将 SQLite 2.8 数据库转换为 SQLite 3.0，请同时准备好 2.8 和 3.0 的命令行 Shell。然后输入如下命令： 

sqlite OLD.DB .dump | sqlite3 NEW.DB

新数据库文件格式为表使用 B+ 树。在 B+ 树中，所有数据都存储在叶子结点，而不是叶子和中间分支结点都存储。 表使用 B+ 树可以实现更好的可扩展性，并且在无需使用溢出页的情况下存储更大的数据字段。传统的 B 树仍然用于索引。

新文件格式还支持 512 到 65536 字节之间的可变页大小。页的大小记录在文件头中，因此理论上同一个库可以读取具有不同页大小的数据库，尽管这个特性尚未在实际中实现。

新文件格式在磁盘映像中省略未使用的字段。例如，索引只使用 B 树记录的键部分而不使用数据。因此，对于索引，记录数据长度的字段被省略。 键和数据的长度等整数值使用变长编码存储，这样常见情况只需要一到两字节，就算需要也可以编码高达 64 位的信息。 整数和浮点数据在磁盘上以二进制形式存储，而不会像 SQLite 2.8 那样被转换为 ASCII。 综合这些变化，数据库文件通常比 SQLite 2.8 中的同等文件小 25% 至 35%。

有关 SQLite 3.0 所使用的低级 B 树格式的详细信息，可在 btreeInt.h 源文件的头部注释中以及 文件格式 文档中找到。

明确类型系统与 BLOB 支持

SQLite 2.8 在内部会以各种格式处理数据，但在写入磁盘或通过 API 交互时，总是将数据转换为 ASCII 文本。相比之下，SQLite 3.0 向用户暴露其内部数据表示，并在合适的时候以二进制形式存储到磁盘。添加非 ASCII 表示是为了支持 BLOB。

SQLite 2.8 的特点是，无论列的声明类型为何，各种类型的数据都可以存储在该列中。这个特性在 3.0 中继续保留，只是形式略有修改。每个表列仍可存储任何类型的数据，但列会对其声明的数据类型有一个偏好（affinity）。当数据插入某列时，该列会尝试将数据格式转换为其声明的类型。所有 SQL 数据库引擎都会这么做。不同之处在于，即便格式转换不可行，SQLite 3.0 仍会保存该数据。

例如，如果某个表列声明类型为 "INTEGER"，而你尝试插入一个字符串，该列会查看该文本是否像数字。如果看起来像数字，就将其转换为数字，若该数字没有小数部分则存为整数。即便该字符串不是一个格式良好的数字，也仍作为字符串存储。类型为 "TEXT" 的列在存储前会尝试将数字转换为 ASCII 文本表示。但 BLOB 储存在 TEXT 列时仍以 BLOB 形式存储，因为通常无法将 BLOB 转换为文本。

在大多数其他 SQL 数据库引擎中，数据类型与存放数据的表列（即数据容器）相关联。而在 SQLite 3.0 中，数据类型与数据本身相关，而不是与容器相关。Paul Graham 在其著作 ANSI Common Lisp 中称这一特性为 "Manifest Typing"。其他作者对 "manifest typing" 一词有不同解释，因此注意区分。不过无论名称如何，这就是 SQLite 3.0 支持的数据类型模型。

有关 SQLite 3.0 中数据类型的更多信息，可另行参见。

支持 UTF-8 与 UTF-16

SQLite 3.0 的新 API 提供了既接受 UTF-8 又接受主机机器字节顺序的 UTF-16 文本的例程。每个数据库文件以 UTF-8、UTF-16BE（大端）或 UTF-16LE（小端）形式管理文本。内部和磁盘文件中到处使用相同的文本表示。如果文件头指定的文本表示与接口例程所需的表示不一致，则会动态转换。

不断地在不同表示之间转换文本会消耗计算资源，因此建议程序员选择一个表示并在整个应用中保持一致。

在当前的 SQLite 实现中，SQL 解析器仅能处理 UTF-8 文本。因此，如果供应 UTF-16 文本，它会将其转换。这只是实现上的问题，并不妨碍未来版本的 SQLite 原生解析 UTF-16 编码的 SQL。

当创建新的用户定义 SQL 函数和排序序列时，每个函数或排序序列都可以指定它是否处理 UTF-8、UTF-16be 或 UTF-16le。可以为每种编码注册不同的实现。如果某个 SQL 函数或排序序列必需但当前文本编码版本不可用，则会自动转换文本。如前所述，这种转换会消耗计算时间，因此建议程序员选择一个编码并坚持使用，以尽量减少不必要的格式转换。

SQLite 对接收到的文本并不挑剔，并且可以处理未正规化甚至不是格式良好的 UTF-8 或 UTF-16 文本。因此，希望存储 ISO8859 数据的程序员可以使用 UTF-8 接口。只要不尝试使用 UTF-16 排序序列或 SQL 函数，文本的字节序列就不会以任何方式被修改。

用户定义的排序序列

排序序列只是文本的一个定义顺序。当 SQLite 3.0 排序（或使用像 "<" 或 ">=" 这样的比较运算符）时，排序顺序首先由数据类型决定。

  • NULL 值首先排序
  • 数值按数字顺序排序
  • 文本值排在数值之后
  • BLOB 最后排序

排序序列用于比较两个文本字符串。排序序列不会改变 NULL、数字或 BLOB 的顺序，只影响文本。

排序序列通过一个函数实现，该函数接受两个被比较的字符串作为输入，并根据第一个字符串小于、等于或大于第二个字符串返回负值、零或正值。SQLite 3.0 内置了一个名为 "BINARY" 的排序序列，它使用标准 C 库中的 memcmp() 例程实现。BINARY 排序序列对英文文本效果良好。对于其他语言或区域，可能更喜欢其他排序序列。

选择使用哪个排序序列是由 SQL 中的 COLLATE 子句控制的。COLLATE 子句可以出现在表定义中，用于为表列定义默认排序序列，也可以出现在索引字段中，或出现在 SELECT 语句的 ORDER BY 子句中。SQLite 的计划增强功能包括标准的 CAST() 语法，以允许为表达式定义排序序列。

64 位 ROWID

表的每一行都有一个唯一的 rowid。如果表定义了类型为 "INTEGER PRIMARY KEY" 的列，那么该列就成为 rowid 的别名。但是无论是否存在 INTEGER PRIMARY KEY 列，每一行仍然有 rowid。

在 SQLite 3.0 中，rowid 是一个 64 位有符号整数。这是对 SQLite 2.8 的扩展，后者只允许 32 位的 rowid。

为尽量减少存储空间，64 位 rowid 以变长整数形式存储。rowid 在 0 到 127 之间只使用一个字节；0 到 16383 之间只使用两个字节；最多 2097152 使用三个字节，依此类推。允许使用负 rowid，但它们总是使用九个字节，因此不建议使用。当 rowid 由 SQLite 自动生成时，它们总是非负的。

改进的并发能力

SQLite 2.8 允许多个读者同时读取或一个写者写入，但不能同时进行。SQLite 3.0 允许一个进程开始写数据库时，其他进程仍继续读取。写进程仍需在提交更改时在数据库上获取一次独占锁，但不再需要在整个写操作期间保持独占锁。有关 SQLite 3.0 锁行为的更详细报告，请参见 lockingv3.html。

SQLite 现在也提供有限形式的表级锁定。如果每个表存储在一个单独的数据库文件中，这些独立文件可以通过 ATTACH 命令附加到主数据库，并且组合后的数据库将作为一个整体工作。但锁只会在需要时对单个文件加锁。因此，如果你将 "数据库" 重新定义为两个或更多数据库文件，那么两个进程完全可能在同一时间写入同一个数据库。为了进一步支持这种能力，涉及两个或更多 ATTACH 的数据库的事务提交现在是原子的。
