INDEXED BY 子句

1. INDEXED BY 的工作方式

INDEXED BY 短语强制SQLite 查询规划器在DELETE、SELECT或UPDATE语句上使用特定的命名索引。INDEXED BY 短语是 SQLite 的扩展，在其他 SQL 数据库引擎中不可移植。

qualified-table-name:

“INDEXED BY index-name” 短语指定必须使用所命名的索引来查找前一个表中的值。如果 index-name 不存在或不能用于该查询，则 SQL 语句的准备阶段会失败。“NOT INDEXED” 子句指定在访问前一个表时不能使用任何索引，包括由 UNIQUE 和 PRIMARY KEY 约束隐含创建的索引。不过，即使指定了“NOT INDEXED”，rowid 仍然可以用于查找条目。

某些 SQL 数据库引擎提供非标准的“提示”机制，可以用于向查询优化器提供应在特定语句中使用哪些索引的线索。SQLite 的 INDEXED BY 子句不是一种提示机制，也不应被当作提示机制使用。INDEXED BY 子句不会向优化器提供应使用哪个索引的提示；它向优化器提出必须使用哪个索引的要求。如果查询优化器无法使用 INDEXED BY 子句指定的索引，则查询将因错误而失败。

INDEXED BY 子句并非用于调整查询性能。INDEXED BY 子句的目的是在删除或创建索引等架构更改导致针对时间敏感查询的查询计划发生变化时引发运行时错误。INDEXED BY 子句旨在帮助在回归测试期间检测不理想的查询计划变更。强烈建议应用程序开发人员在应用程序设计、实现、测试和调优期间省略所有 INDEXED BY 的使用。如果确实要使用 INDEXED BY，应该在开发过程的末期即“锁定”设计时插入该子句。

2. 参见

  1. 查询规划器检查清单 描述了应用程序开发人员在解决查询规划器问题时应遵循的步骤。请注意，INDEXED BY 的使用应作为最后手段，仅在所有其他措施都失败时才使用。

  2. 一元“+”运算符 可用于使 WHERE 子句中的某些项不能被索引使用。谨慎使用一元 + 有时可以在不限制优化器必须使用具体某个索引的情况下，防止查询规划器选择糟糕的索引。合理地安排一元 + 运算符位置是控制查询使用哪些索引的更好方法。

  3. sqlite3_stmt_status() C/C++ 接口结合SQLITE_STMTSTATUS_FULLSCAN_STEP和SQLITE_STMTSTATUS_SORT 动词，可以在运行时检测 SQL 语句是否未有效利用索引。许多应用可能更愿意使用sqlite3_stmt_status() 接口来检测索引误用，而不是使用此处描述的 INDEXED BY 短语。
