SQLite 第2版 中的数据类型

1.0   无类型性

SQLite 是“无类型”的。这意味着你可以在任何表的任意列中存储任何你想要的数据，而不管该列声明的数据类型为何。 （详见下面 2.0 节中的唯一例外。） 这种行为是一个特性，而非错误。数据库本应储存并检索数据，数据是以何种格式存储对数据库来说并无差别。 其他大多数 SQL 引擎所采用并在 SQL 语言规范中规定的强类型系统，是一种设计缺陷——它让实现细节显露在接口之上。SQLite 试图克服这种缺陷，通过允许你将任意类型的数据存储到任意类型的列中，并允许在数据类型定义上保持灵活性。

对于 SQLite 来说，数据类型可以是任意个名称构成的序列，后面可选跟着一对有符号整数的括号列表。特别要注意的是，数据类型可以是零个或多个名称。这意味着空字符串也被 SQLite 视作有效的数据类型。因此你可以声明一些列在 CREATE TABLE 语句中完全不指定数据类型，例如： 

CREATE TABLE ex1(a,b,c);

尽管 SQLite 允许省略数据类型，但在 CREATE TABLE 语句中包含它仍然是一个好习惯，因为数据类型通常可以向其他程序员提示你打算在该列中存储什么。如果你将来把代码移植到其他数据库引擎，那些引擎很可能会要求必须指定某种数据类型。SQLite 接受所有通常的数据类型。例如： 

CREATE TABLE ex2(
  a VARCHAR(10),
  b NVARCHAR(15),
  c TEXT,
  d INTEGER,
  e FLOAT,
  f BOOLEAN,
  g CLOB,
  h BLOB,
  i TIMESTAMP,
  j NUMERIC(10,5)
  k VARYING CHARACTER (24),
  l NATIONAL VARYING CHARACTER(16)
);

依此类推。基本上只要是任意个名称序列，可选地后面跟一个或两个括号中的有符号整数，就可以作为数据类型。

2.0   INTEGER PRIMARY KEY

SQLite 无类型特性的唯一例外是类型为 INTEGER PRIMARY KEY 的列。（必须使用 “INTEGER”，不能用 “INT”。类型为 INT PRIMARY KEY 的列同其它任意列一样是无类型的。）INTEGER PRIMARY KEY 列必须包含一个 32 位有符号整数。任何试图插入非整数数据的行为都会导致错误。

INTEGER PRIMARY KEY 列可以用来实现类似 AUTOINCREMENT 的功能。如果你尝试向 INTEGER PRIMARY KEY 列插入 NULL，该列实际上会填入一个比目前表中最大键值大一的整数。若最大键值为 2147483647，则该列会被填入一个随机整数。不论哪种情况，INTEGER PRIMARY KEY 列都会被分配一个唯一整数。可以通过 API 函数 sqlite_last_insert_rowid() 或随后 SELECT 语句中的 SQL 函数 last_insert_rowid() 检索该整数。

3.0   比较与排序顺序

SQLite 在决定某列允许存储何种数据时是无类型的。但在排序或比较数据时，类型的概念会发挥作用。为这些目的，某个列或表达式可以是两种类型之一：数字型 或 文本型。排序或比较的结果可能会因为所比较的数据类型不同而异。

若数据为 文本型，则比较由标准 C 数据比较函数 memcmp() 或 strcmp() 决定。比较时逐字节查看两个输入并返回第一个非零差异。字符串以 '\000' 结尾，因此短字符串会在长字符串之前排序，这与预期一致。

对于数字数据，情况则更为复杂。如果两个输入都看起来像格式良好的数字，就会使用 atof() 将它们转换为浮点值并按数值比较。如果一个输入不是格式良好的数字而另一个是，则数字被认为小于非数字。如果两个输入都不是格式良好的数字，则使用 strcmp() 进行比较。

不要因为某列具有“numeric”数据类型而困惑。这并不意味着该列只能包含数字。它仅仅意味着如果该列中确实有数字，那么它们将在数值顺序中排序。

对于文本和值，NULL 均在其他任何值之前排序。使用诸如 "<" 或 ">=" 之类的操作符将任意值与 NULL 比较，结果总是 false。

4.0   SQLite 如何确定数据类型

在 SQLite 2.6.3 及之前版本中，所有值都使用 numeric 数据类型。text 数据类型从 2.7.0 及以后版本才出现。下文默认你使用的是 SQLite 2.7.0 或更新版本。

对于表达式，结果的数据类型通常由最外层操作符决定。例如，算术操作符（"+", "*", "%"）总是返回 numeric 类型结果。字符串连接操作符 ("||") 返回 text 类型结果。诸如此类。如果你对表达式的数据类型存疑，可以使用特殊的 SQL 函数 typeof() 来确定。例如： 

sqlite> SELECT typeof('abc'+123);
numeric
sqlite> SELECT typeof('abc'||123);
text

对于表列，数据类型由 CREATE TABLE 语句中的类型声明决定。只有当类型声明中包含下列一个或多个字符串时，列的数据类型才为文本：

BLOB
CHAR
CLOB
TEXT

当然，对这些字符串的搜索是不区分大小写的。如果类型声明中任意位置出现上述任意字符串，则该列的数据类型为文本。注意 “VARCHAR” 包含 “CHAR” 作为子串，因此也被视为文本。

如果类型声明中没有出现上述任意字符串，则数据类型为数字。特别要注意的是，对于类型声明为空的列，其数据类型为数字。

5.0   示例

考虑以下两组命令序列： 

CREATE TABLE t1(a INTEGER UNIQUE);        CREATE TABLE t2(b TEXT UNIQUE);
INSERT INTO t1 VALUES('0');               INSERT INTO t2 VALUES(0);
INSERT INTO t1 VALUES('0.0');             INSERT INTO t2 VALUES(0.0);

在左侧的序列中，第二条插入会失败。因为字符串 '0' 和 '0.0' 被当作数字处理，因它们被插入到数字列中，而 0==0.0，这违反了唯一性约束。然而右侧序列中的第二条插入则能成功。在这种情况下，常量 0 和 0.0 被当作字符串处理，因此它们是不同的。

SQLite 在比较时总是将数字转换为双精度（64 位）浮点数。这意味着如果两个数字仅在不重要的位上不同，在数字列中它们会被认为相等，但在文本列中它们会被认为不等。我们有：

INSERT INTO t1                            INSERT INTO t2
   VALUES('12345678901234567890');           VALUES(12345678901234567890);
INSERT INTO t1                            INSERT INTO t2
   VALUES('12345678901234567891');           VALUES(12345678901234567891);

如前所述，左侧第二条插入会失败，因为比较时会先把两个字符串都转换成浮点数，而它们在第 20 位的不同超过了 64 位浮点数的精度限制。相比之下，右侧第二条插入能成功，因为此时被插入的是字符串，比较时使用的是 memcmp()。

数字型与文本型对 DISTINCT 关键字也有影响： 

CREATE TABLE t3(a INTEGER);               CREATE TABLE t4(b TEXT);
INSERT INTO t3 VALUES('0');               INSERT INTO t4 VALUES(0);
INSERT INTO t3 VALUES('0.0');             INSERT INTO t4 VALUES(0.0);
SELECT DISTINCT * FROM t3;                SELECT DISTINCT * FROM t4;

左侧的 SELECT 语句返回一行，因为 '0' 与 '0.0' 被当作数字处理，因此不区分。但右侧的 SELECT 会返回两行，因为 0 与 0.0 被当作字符串处理，从而不同。
