SQLite 与 8+3 文件名

SQLite 的默认配置假定底层文件系统支持长文件名。

SQLite 对数据库文件没有任何命名要求。SQLite 完全可以处理带有任何文件扩展名或根本没有扩展名的数据库文件。当需要一个回滚日志、一个预写日志或其他某种临时磁盘文件时，辅助文件的名称通常通过在数据库文件名末尾追加后缀来构造。例如，如果原始数据库称为“app.db”，那么回滚日志将命名为“app.db-journal”，预写日志将命名为“app.db-wal”。这种辅助文件命名方式在支持长文件名的系统上运行良好。但在强制定 8+3 文件名约束的系统中，尽管原始数据库文件符合 8+3 格式，辅助文件却无法满足。

更换文件系统

为了解决这个问题，建议选择不同的文件系统。如今有大量高性能、可靠且无专利束缚且支持长文件名的文件系统可供选择。尽可能地，建议嵌入式设备使用这些其他文件系统之一。这将避免兼容性问题以及由于不一致使用 8+3 文件名导致的数据库损坏的风险。

调整 SQLite 以使用 8+3 文件名

有些设备出于向后兼容性或其他非技术因素的原因不得不使用旧的带有 8+3 文件名限制的文件系统。在这种情况下，可以通过以下方式强制 SQLite 使用符合 8+3 规则的辅助文件：

  1. 在编译 SQLite 库时使用编译时选项 SQLITE_ENABLE_8_3_NAMES=1 或 SQLITE_ENABLE_8_3_NAMES=2。 SQLite 默认并不包含对 8+3 文件名的支持，因为这会带来一些额外开销。虽然开销非常小，但我们仍不希望给那些不需要 8+3 文件名支持的数十亿 SQLite 应用增加负担。

  2. 如果使用 SQLITE_ENABLE_8_3_NAMES=1 选项，那么 SQLite 虽然具备使用 8+3 文件名的能力，但这一能力默认是禁用的，必须在每次打开数据库或ATTACH数据库文件时通过使用URI 文件名并在 URI 中包含“8_3_names=1”查询参数来单独启用。如果 SQLite 使用 SQLITE_ENABLE_8_3_NAMES=2 编译，则 8+3 文件名默认启用，此步骤可以省略。

  3. 确保数据库文件名遵循 8+3 文件名格式，并且名称和扩展名都不为空。换句话说，数据库文件名的主名称必须包含 1 到 8 个字符，扩展名必须包含 1 到 3 个字符。不允许出现空扩展名。

按上述步骤操作时，SQLite 会通过仅保留扩展名的最后 3 个字符来缩短扩展名。例如，一个通常命名为“app.db-journal”的文件会被缩短为“app.nal”。类似地，“app.db-wal”将变为“app.wal”，而“app.db-shm”将变为“app.shm”。

请注意，数据库文件名必须有某种形式的扩展名。如果没有扩展名，SQLite 会通过在文件的主名称后追加内容来创建辅助文件名。因此，一个名为“db01”的数据库会有一个名为“db01-journal”的回滚日志。由于该文件名没有扩展名可供缩短为 3 个字符，它会照原样使用，从而违反 8+3 命名规则。

数据库损坏警告

如果访问数据库文件时使用的是 8+3 命名而不是默认的长文件名，那么每一次打开数据库的每个数据库连接都必须一致地采用 8+3 命名，否则存在数据库损坏的风险。辅助的回滚日志和预写日志文件对于 SQLite 在崩溃后恢复至关重要。如果某个应用使用 8+3 名称并且崩溃，那么安全恢复所需的信息就保存在具有“.nal”或“.wal”扩展名的文件中。如果下一个打开该数据库的应用没有指定“8_3_names=1”URI 参数，SQLite 会使用长文件名去尝试定位回滚日志或预写日志文件。由于这些文件是由崩溃的应用用 8+3 名称保存的，因此 SQLite 找不到它们，数据库将不能被正确恢复，很可能会损坏。

在某些情况下对某个数据库文件使用 8+3 文件名，而在其他情况下使用长文件名，相当于删除热日志。
