1. 概述

这个 spellfix1 虚拟表可以用于在大型词汇表中搜索近似匹配项。例如，spellfix1 可用于建议拼写错误的单词的更正。或者，它可以与 FTS4 结合使用，以便使用可能拼错的单词执行全文搜索。

spellfix1 虚拟表的实现位于 SQLite 源代码树的 miscellaneous extensions 文件夹中，具体在文件 ext/misc/spellfix1.c 内。 spellfix1 虚拟表未包含在 SQLite 的 合并包 中，也不是任何标准 SQLite 构建的一部分。它是一个 可加载扩展。

一旦加载了 spellfix1 扩展，就可以像这样创建一个 spellfix1 虚拟表的实例： 

CREATE VIRTUAL TABLE demo USING spellfix1;

“spellfix1” 是 spellfix 模块的名称，必须照此输入。“demo” 是你将创建的虚拟表的名称，可以根据应用的需要更改。该虚拟表最初是空的。为了使虚拟表发挥作用，你需要用词汇填充它。假设你在名为 “big_vocabulary” 的表中有一个单词列表，那么可以这样做： 

INSERT INTO demo(word) SELECT word FROM big_vocabulary;

如果你打算将这个虚拟表与一个 FTS4 表联合使用（用于纠正搜索词的拼写），那么你可能会使用 fts4aux 表来提取词汇： 

INSERT INTO demo(word) SELECT term FROM search_aux WHERE col='*';

你还可以为每个单词提供一个“rank”。“rank” 是对该单词常见程度的估计。数值越大表示该单词越常见。如果在填充表时省略了 rank，则默认假定 rank 为 1。但如果你有 rank 信息，就可以提供它，虚拟表会稍微倾向于选择使用频率更高的词。要从 fts4aux 表 “search_aux” 填充 rank，可按如下操作： 

INSERT INTO demo(word,rank)
   SELECT term, documents FROM search_aux WHERE col='*';

要查询虚拟表，请在 WHERE 子句中包含 MATCH 操作符。例如： 

SELECT word FROM demo WHERE word MATCH 'kennasaw';

使用一个基于美国地名的数据集（来源于 http://geonames.usgs.gov/domestic/download_data.htm），上述查询会返回 20 个结果，开头如下： 

kennesaw
kenosha
kenesaw
kenaga
keanak

如果你在模式末尾附加字符 ‘*’，则会执行前缀搜索。例如： 

SELECT word FROM demo WHERE word MATCH 'kennes*';

将得到以以下开头的 20 条结果： 

kennesaw
kennestone
kenneson
kenneys
keanes
keenes

2. 搜索改进

默认情况下，spellfix1 表最多返回 20 个结果。（如果好的匹配少于 20 个，也可能返回更少。）你可以通过在查询的 WHERE 子句中添加形如 “top=N” 的项来更改返回行数的上限，其中 N 是新的最大值。例如，要查看最匹配的 5 个结果： 

SELECT word FROM demo WHERE word MATCH 'kennes*' AND top=5;

spellfix1 虚拟表中的每一条记录都与一个特定的语言相关联，该语言由整数 “langid” 列标识。默认 langid 为 0，如果没有其他操作，整个词汇都属于语言 0。但如果你的应用需要支持多种语言，那么可以在填充表时通过指定 langid 字段为每种语言提供不同的词汇。例如： 

INSERT INTO demo(word,langid) SELECT word, 0 FROM en_vocabulary;
INSERT INTO demo(word,langid) SELECT word, 1 FROM de_vocabulary;
INSERT INTO demo(word,langid) SELECT word, 2 FROM fr_vocabulary;
INSERT INTO demo(word,langid) SELECT word, 3 FROM ru_vocabulary;
INSERT INTO demo(word,langid) SELECT word, 4 FROM cn_vocabulary;

在将多个语言的词汇项填充进虚拟表后，可以在查询的 WHERE 子句中使用 “langid=N” 来指定感兴趣的语言： 

SELECT word FROM demo WHERE word MATCH 'hildes*' AND langid=1;

注意，如果在 WHERE 子句中不包含 “langid=N” 项，搜索将针对语言 0（即上面示例中的英语）。所有 spellfix1 搜索都针对单一语言 id。无法一次搜索所有语言。

3. 虚拟表详细信息

spellfix1 虚拟表中的每一行都有一个唯一的 rowid，共有七列再加上五列隐藏列。 各列如下所示：

rowid

与表中的每个词汇项相关联的唯一整数编号。可以作为数据库中其他表的外键。

word

与模式匹配的单词文本。word 和 pattern 都可以包含 Unicode 字符，并且可以混合大小写。

rank

这是该单词的 rank，按照原始 INSERT 语句中指定的值。

distance

这是从模式到单词的编辑距离或 Levenshtein 距离。

langid

这是该单词所属的语言 id。所有的查询都针对单一的语言 id，默认值为 0。对于任何给定查询，该值在所有行中都是相同的。

score

分数是 rank 与 distance 的组合。其思想是较低的分数更好。虚拟表尝试找到分数最低的单词，并默认（除非被 ORDER BY 重写）按分数递增顺序返回结果。

matchlen

在前缀搜索中，matchlen 是与前缀匹配的字符数。对于非前缀搜索，则等于 length(word)。

phonehash

此列显示用于限制搜索的发音哈希前缀。对于任何给定查询，该列在每一行中都应该相同。此信息用于诊断，在实际应用中通常被视为无用。

top

（隐藏）对于任何查询，该值在所有行中都相同。它是输出行数的最大值。实际输出的行数可能小于该值，但绝不大于。top 的默认值为 20，但可以在每个查询的 WHERE 子句中包含 “top=N” 的项来更改。

scope

（隐藏）对于任何查询，该值在所有行中都相同。scope 是虚拟表查找匹配单词的范围度量。scope 值越小，搜索越广。scope 通常自动选择，最大值为 4。应用可以在 WHERE 子句中包含 “scope=N” 项来更改 scope。增加 scope 会使查询更快，但会减少可能的纠正项。

srchcnt

（隐藏）对于任何查询，该值在所有行中都相同。该整数表示用于找到最终显示的最佳匹配项时，编辑距离算法检查了多少个单词。此值仅用于诊断。

soundslike

（隐藏）插入词汇项时，此字段可以设置为与该词发音相符的拼写。有关详情请参阅下文“处理不常见及困难拼写”部分。

command

（隐藏）“command” 列的值总是 NULL。然而，应用可以向“command”列插入特定字符串，以触发 spellfix1 虚拟表的某些行为。例如，在“command”列插入字符串 'reset' 会使虚拟表重新读取其编辑距离权重（如果有的话）。

4. 算法

spellfix1 虚拟表会创建一个名为 “%_vocab” 的单个影子表（其中 % 会替换为虚拟表名；例如：“demo_vocab” 对应 “demo” 虚拟表）。 影子表包含以下列：

id

唯一的 id（INTEGER PRIMARY KEY）

rank

单词的 rank。

langid

该条目的语言 id。

word

词汇单词的原始 UTF8 文本

k1

将单词转写为小写 ASCII。存在一个标准映射表，用于将非 ASCII 字符映射到 ASCII。例如：“æ”->“ae”，“þ”->“th”，“ß”->“ss”，“á”->“a”等。辅助函数 spellfix1_translit(X) 会执行非 ASCII 到 ASCII 的映射。内置的 lower(X) 函数会转换为小写。于是： k1 = lower(spellfix1_translit(word))。 如果单词已经全部为小写 ASCII，则 k1 列会包含 NULL。这减少了 %_vocab 表的存储需求，并有助于 spellfix 更快地运行。因此，尽可能使用小写 ASCII 词汇填充 spellfix 表是有利的。

k2

此字段保存从 coalesce(k1,word) 推导出的发音代码。发音相近的字母映射到相同符号。例如，所有元音及元音组合都成为单一符号 “A”。字母 “p”、“b”、“f”、“v” 都归为 “B”。所有鼻音表示为 “N”，等等。映射基于 Soundex、Metaphone 以及其他长期使用的语音匹配系统的思想。该键可以通过函数 spellfix1_phonehash(X) 生成。因此：k2 = spellfix1_phonehash(coalesce(k1,word))

还有一个函数用于计算模式和单词之间的 Wagner 编辑距离或 Levenshtein 距离。该函数以 spellfix1_editdist(X,Y) 的形式暴露。编辑距离函数返回将 X 转换为 Y 的“代价”。某些变换的代价比其他的高。例如，将一个元音变为另一元音、双写一个辅音、或删除双辅音的第二个字符，这些操作的代价较低。其他变换则更昂贵。其思想是编辑距离函数为相似的单词返回较低的代价，为相差较大的单词返回较高的代价。在此实现中，任意单字符编辑（删除、插入或替换）的最大代价为 100，一些编辑（如元音变换）具有更低的代价。

比较的“分数”是模式与单词之间的编辑距离，再减去单词 rank 的以 2 为底的对数。例如，距离为 100 但 rank 为 1000 的匹配，分数为 122（= 100 - log2(1000) + 32），而距离为 100 且 rank 为 1 的匹配，分数为 131（100 - log2(1) + 32）。（注：为了防止编辑距离为零时分数变为负数，每个分数都加上常数 32。）如此一来，常用词会得到略低的代价，使它们更倾向于出现在备选拼写列表的顶部。

一个直接的拼写纠正器实现方式是将搜索词与词汇表中的每个单词进行比较，并选择分数最低的 20 个。然而，词汇表通常包含数十万甚至数百万个单词，因此这种方法太慢。

设要纠正的单词为 X。为了缩小搜索空间，先将 X 转换为类似 k2 的键，等价于： 

   key = spellfix1_phonehash(lower(spellfix1_translit(X)))

然后将该键截断为 “scope” 个字符。默认 scope 值为 4，但可以在 WHERE 子句中通过 “scope=N” 项指定其他作用域。键被截断后，编辑距离会针对词汇表中 k2 值以该缩写键开头的每个项执行。

例如，如果输入单词是 “Paskagula”，其发音键为 “BACACALA”，截断为 4 个字符 “BACA”。编辑距离随后在 k2 值以 BACA 开头的 4,980 个词汇项（总计 272,597 个项中的一部分）上运行，得出 “Pascagoula” 作为最佳匹配。

只会搜索 langid 匹配的词汇项。因此，同一张表可以包含多种语言的条目，并且只会使用所请求的语言。默认 langid 为 0。

5. 可配置编辑距离

内置的 Wagner 编辑距离函数具有固定权重，但可以通过在创建虚拟表时向 spellfix1 模块指定 “edit_cost_table=TABLENAME” 参数，用 editdist3() 编辑距离函数替代，该函数允许应用指定权重并支持 Unicode。例如： 

CREATE VIRTUAL TABLE demo2 USING spellfix1(edit_cost_table=APPCOST);

还可以通过向虚拟表的 “command” 列插入适当的字符串，在运行时选用或取消选用 editdist3() 编辑距离函数：

INSERT INTO demo2(command) VALUES('edit_cost_table=APPCOST');

在上述示例中，将查询 APPCOST 表以获取编辑距离系数。正是 spellfix1 模块名称中包含 “edit_cost_table=” 参数，才会使用 editdist3() 替代内置编辑距离函数。如果 APPCOST 是空字符串，则使用内置的 Wagner 编辑距离函数。

编辑距离系数通常只从 APPCOST 表读取一次，并存储在内存中。因此，运行时对 APPCOST 表所做的更改通常不会影响编辑距离结果。然而，向虚拟表的 “command” 列插入特殊字符串 'reset' 会使编辑距离系数重新读取 APPCOST 表。因此，当 APPCOST 表发生更改时，应用应执行类似如下的 SQL 语句：

INSERT INTO demo2(command) VALUES("reset");

6. 处理不常见及困难拼写

上述算法在大多数情况下运行良好，但也存在例外。这些例外可以通过在虚拟表中使用 “soundslike” 列添加额外条目来处理。

例如，许多希腊语来源的单词以 “ps” 开头，其中 “p” 不发音。例如：psalm、pseudonym、psoriasis、psyche。在另一个例子中，许多苏格兰姓氏可以写作 “Mac” 或 “Mc”。因此，“MacKay” 与 “McKay” 发音相同。

对于那些拼写与发音不一致的单词，可以在虚拟表中为同一个词添加额外条目，并在 “soundslike” 列中提供替代拼写。例如，“psalm”的规范条目如下： 

  INSERT INTO demo(word) VALUES('psalm');

为了增强将 “salm” 纠正为 “psalm” 的能力，可以添加一条如下所示的记录： 

  INSERT INTO demo(word,soundslike) VALUES('psalm','salm');

为同一个单词创建多个条目是可以的，只要每条记录的 soundslike 值不同。注意，如果未指定 soundslike 值，则默认与单词本身相同。

下面列出了一些可能值得添加额外 soundslike 条目的情况。具体条目将取决于应用程序和目标语言。

  • 以 “ps” 开头时 “p” 不发音：psalm、psyche
  • 以 “pn” 开头时 “p” 不发音：pneumonia、pneumatic
  • 以 “pt” 开头时 “p” 不发音：pterodactyl、ptolemaic
  • 以 “dj” 开头时 “d” 不发音：djinn、Djikarta
  • 以 “kn” 开头时 “k” 不发音：knight、Knuthson
  • 以 “gn” 开头时 “g” 不发音：gnarly、gnome、gnat
  • 苏格兰姓氏的 “Mac” 与 “Mc”
  • 斯拉夫语中的 “Tch” 发音：Tchaikovsky vs. Chaykovsky
  • 西班牙语中字母 “j” 发音近似 “h”：LaJolla
  • 以 “wr” 开头与以 “r” 开头的单词对比：write vs. rite
  • 各种问题词，例如 “debt”、“tsetse”、“Nguyen”、“Van Nuyes”。

7. 辅助函数

实现 spellfix1 虚拟表的源代码模块还实现了若干 SQL 函数，可能对使用 spellfix1 的应用或在开发时进行测试/诊断工作时有用。提供以下辅助函数：

editdist3(P,W)
editdist3(P,W,L)
editdist3(T)

这些例程提供对支持应用定义编辑操作权重的 Wagner 编辑距离函数的直接访问。该函数的前两种形式比较模式 P 与单词 W 并返回编辑距离。第一种形式假定 langid 为 0，第二种形式则由参数 L 指定 langid。第三种形式会从名称为 T 的表重新加载编辑距离系数。

spellfix1_editdist(P,W)

此例程提供对使用默认固定代价的内置 Wagner 编辑距离函数的访问。返回值是将 W 转换为 P 所需的编辑距离。

spellfix1_phonehash(X)

此例程构造纯 ASCII 输入单词 X 的语音哈希并返回该哈希。spellfix1 在内部使用它将影子表的 K1 列转换为 K2 列。

spellfix1_scriptcode(X)

给定输入字符串 X，该例程尝试判定其主导的文字脚本，并返回该脚本的 ISO-15924 数值代码。当前实现理解以下脚本：
  • 215 - 拉丁文
  • 220 - 西里尔文
  • 200 - 希腊文
未来版本可能会加入其他语言代码。

spellfix1_translit(X)

此例程将 Unicode 文本音译为纯 ASCII，返回输入文本 X 的纯 ASCII 表示。这是用于将词汇单词转换为影子表 K1 列的内部函数。

8. editdist3 函数

editdist3 算法是一个函数，用于计算两个输入字符串之间的最小编辑距离（即 Levenshtein 距离）。editdist3 算法是 spellfix1 默认编辑距离函数的可配置替代方案。 editdist3 的特性包括：

  • 它支持 Unicode（UTF8）文本。

  • 应用可以提供插入、删除和替换的权重表。

  • 在代价表中可以枚举多字符的插入、删除和替换规则。

9. editdist3 COST 表

要为 editdist3 编程代价，请创建如下表： 

CREATE TABLE editcost(
  iLang INT,   -- 语言 ID
  cFrom TEXT,  -- 从此文本转换
  cTo   TEXT,  -- 转换为此文本
  iCost INT    -- 执行转换的代价
);

代价表可以取任意名称，并不一定要叫 “editcost”。表可以包含额外列。唯一要求是表必须包含上面四个列，且名称必须完全一致。

iLang 列是非负整数，用于标识某种语言对应的一组代价。editdist3 函数在任何给定的编辑距离计算中只使用单个 iLang 值。默认值为 0。建议仅需单一语言的应用始终对所有条目使用 iLang==0。

iCost 列是将 cFrom 转换为 cTo 的数值代价。该值应为非负整数，并且最好小于 100。默认的单字符插入和删除代价为 100，默认的单字符对单字符替换代价为 150。代价为 10000 或更高视为“无限”，会导致规则被忽略。

cFrom 与 cTo 列显示编辑转换字符串。任一列可以包含多个字符，也可以（但不能同时）为空字符串。当 cFrom 为空时，表示插入 cTo 的代价。当 cTo 为空时，表示删除 cFrom 的代价。

在 spellfix1 算法中，cFrom 是用户输入的文本，cTo 是数据库中的正确拼写文本。editdist3 算法的目标是判断用户输入的文本与词典文本的接近程度。

代价表中有三条特例记录：

cFromcTo含义
'''?'默认的插入代价
'?'''默认的删除代价
'?''?'默认的替换代价

如果省略上述特例记录中的任意一条，则插入与删除使用 100，替换使用 150。要禁用默认插入、删除或替换，可将对应代价设为 10000 或更高。

代价表中的其他条目描述特定字符的转换。 具体转换的代价应低于默认代价，否则默认代价将生效，使得这些特定转换永远不会被使用。

一些示例代价表记录： 

INSERT INTO editcost(iLang, cFrom, cTo, iCost)
VALUES(0, 'a', 'ä', 5);

上述规则表示用户输入的字母 “a” 可以以代价 5 匹配词典中的 “ä”。 

INSERT INTO editcost(iLang, cFrom, cTo, iCost)
VALUES(0, 'ss', 'ß', 8);

cFrom 与 cTo 中的字符数量不必相同。上述规则表示用户输入的 “ss” 可以以代价 8 匹配 “ß”。

10. 尝试 editcost3() 函数

spellfix1 虚拟表 在创建时指定 “edit_cost_table=TABLE” 参数时会使用 editdist3。 但也可以直接使用内置的 “editdist3()” SQL 函数测试 editdist3()。editdist3() SQL 函数有三种形式：

  1. editdist3('TABLENAME');
  2. editdist3('string1', 'string2');
  3. editdist3('string1', 'string2', langid);

第一种形式从名为 'TABLENAME' 的表加载编辑距离系数。先前的系数会被丢弃。因此，在尝试权重并修改权重表后，只需重新执行单参数形式的 editdist3() 即可重新加载更新后的系数。请注意，editdist3() SQL 函数使用的编辑距离权重与 spellfix1 虚拟表使用的权重是独立的。

第二种与第三种形式返回字符串 'string1' 与 "string2" 之间计算出的编辑距离。第二种形式使用语言 id 为 0。第三种形式可指定语言 id。
