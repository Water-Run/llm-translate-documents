SQLite的独特特性

本页突出介绍了 SQLite 中一些不同寻常的特点，这些特点使得 SQLite 与许多其他 SQL 数据库引擎显著不同。

零配置

SQLite 在使用前无需“安装”。 没有“设置”流程。也不存在必须启动、停止或配置的服务器进程。 不需要管理员创建新的数据库实例或为用户分配访问权限。 SQLite 不依赖任何配置文件。 无需额外操作来告知系统 SQLite 正在运行。 系统崩溃或断电后无需任何恢复工作。 无需排查问题。

SQLite 就是这样直接可用。

其他更为常见的数据库引擎在正常运行后都表现良好。 但完成最初的安装与配置往往复杂得令人望而生畏。

无服务器

（另请参阅 无服务器 文档页面。）

大多数 SQL 数据库引擎都实现为单独的服务器进程。 想要访问数据库的程序通过某种进程间通信 （通常是 TCP/IP）向服务器发送请求并接收结果。 SQLite 并非如此。使用 SQLite 时，想访问数据库的进程直接 从磁盘上的数据库文件中读写数据。 没有中间服务器进程。

无服务器模式有利有弊。 主要优势在于无需安装、设置、配置、初始化、管理和排查的独立服务器进程。 这也是为什么 SQLite 是“零配置”数据库引擎的原因之一。 使用 SQLite 的程序在运行前无需数据库引擎的管理支持。 任何能够访问磁盘的程序都能够使用 SQLite 数据库。

另一方面，使用服务器的数据库引擎可以更好地防护客户端程序中的错误—— 客户端中的野指针无法破坏服务器内存。且由于服务器是持续运行的单一进程， 它可以更精细地控制数据库访问，允许更细粒度的锁定和更好的并发。

大多数 SQL 数据库引擎采用客户端/服务器架构。 在那些无服务器的引擎中，据作者所知，SQLite 是唯一允许多个应用程序 同时访问同一数据库的引擎。

单一数据库文件

SQLite 数据库是一个普通的单个磁盘文件，可以位于目录层级的任何位置。 只要 SQLite 能读取磁盘文件，就能读取其中的数据库内容。 若磁盘文件及其目录可写，则 SQLite 可以更改数据库中的任何内容。 数据库文件可以轻松复制到 USB 记忆棒或通过电子邮件共享。

其他 SQL 数据库引擎通常将数据存储为大量文件。 通常这些文件位于只有数据库引擎本身能够访问的标准位置。 这虽提高了安全性，却也增加了访问难度。 某些 SQL 数据库引擎提供直接写入磁盘的选项，完全绕过文件系统。 这种方式带来更高性能，却付出相当复杂的设置与维护成本。

跨平台稳定的数据库文件

SQLite 文件格式是跨平台的。 在一台机器上写入的数据库文件可以复制到架构不同的另一台机器并使用。 无论是大端还是小端、32 位还是 64 位，都没有影响。 所有机器都使用相同的文件格式。 而且开发者承诺保持文件格式稳定并向后兼容， 使较新版本的 SQLite 可以读取和写入旧的数据库文件。

大多数其他 SQL 数据库引擎在跨平台迁移或升级软件版本时需要 导出并恢复数据库。

紧凑

在为体积优化的情况下，启用所有功能的整个 SQLite 库小于 1MiB （在 ix86 上使用 GNU 编译器套件的 “size” 工具测量）。 不需要的功能可以在编译时禁用，从而进一步减小库体积。

其他大多数 SQL 数据库引擎比这大得多。 IBM 宣称其新发布的 CloudScape 数据库引擎 “仅”为 2MiB 的 jar 文件——即使压缩后也比 SQLite 大一个数量级！ Firebird 说其客户端库只有 350KiB。 这与 SQLite 大小相等，却甚至不包含数据库引擎。 Oracle 的 Berkeley DB 库为 450KiB，且省略了 SQL 支持，只为程序员提供简单的键/值对。

显式类型

大多数 SQL 数据库引擎使用静态类型。 每个表列都与一种数据类型相关联， 只有该数据类型的值才允许存储在该列中。 SQLite 通过使用显式类型放宽了这一限制。 在显式类型中，数据类型是值本身的属性，而不是存储该值的列的属性。 因此 SQLite 允许用户在任何列中存储任意数据类型的值，而不考虑该列声明的类型。 （对该规则有一些例外：INTEGER PRIMARY KEY 列只能存储整数。 并且 SQLite 会在可能的情况下尝试将值强制转换为列声明的数据类型。）

据我们观察，SQL 语言规范允许使用显式类型。 然而，大多数其他 SQL 数据库引擎是静态类型的， 所以有些人认为 SQLite 使用显式类型是一个缺陷。 但 SQLite 的作者坚信这是一个特性。 SQLite 中显式类型的使用是经过深思熟虑的设计决策， 实践证明它使 SQLite 更可靠、更易使用， 尤其是在与 Tcl、Python 等动态类型语言结合使用时。

变长记录

其他大多数 SQL 数据库引擎会为每个表中的每行分配固定的磁盘空间。 它们在处理可变长度的 BLOB 和 CLOB 时会采用特殊技巧。 但对于大多数表而言，如果你声明某列为 VARCHAR(100)， 数据库引擎仍会为其分配 100 字节的磁盘空间， 不管你实际存储了多少信息。

而 SQLite 只使用存储行信息所需的实际磁盘空间。 如果你在 VARCHAR(100) 列中存储一个字符，则只消耗一个字节的磁盘空间。 （实际上是两个字节——每列开头会有一些开销，用于记录其数据类型和长度。）

SQLite 使用变长记录带来了多项优势。 显然，这会生成更小的数据库文件。 这也让数据库运行更快，因为磁盘 I/O 更少。 同时，变长记录的使用使得 SQLite 能够采用显式类型而非静态类型。

可阅读的源代码

SQLite 的源代码设计得便于普通程序员阅读和理解。 所有过程、数据结构以及许多自动变量都配有有用的注释， 说明它们的功能。省略了枯燥的模板式注释。

SQL 语句编译为虚拟机代码

每个 SQL 数据库引擎都会将每条 SQL 语句编译为某种内部数据结构， 然后使用该结构执行语句操作。 但在大多数 SQL 引擎中，内部数据结构是复杂的互联结构与对象网络。 在 SQLite 中，编译后的语句形式是类似机器语言表示的短程序。 数据库用户可以通过在查询前加上 EXPLAIN 关键字， 查看这门 虚拟机语言。

SQLite 使用虚拟机对于库的开发极为有利。 虚拟机为 SQLite 的前端（解析 SQL 语句并生成虚拟机代码的部分） 与后端（执行虚拟机代码并计算结果的部分）之间提供了清晰、定义明确的接口。 虚拟机使开发者能够以易读的形式清楚地看到 SQLite 为每条编译语句所执行的操作， 这对于调试极为有助。 根据编译方式，SQLite 还能够追踪虚拟机的执行过程， 打印出每条虚拟机指令及其执行结果。

公共领域

SQLite 的源代码属于公共领域。 核心源代码的任何部分都未声明版权。（文档与测试代码另当别论—— 某些文档部分与测试逻辑受开源许可证约束。） 所有 SQLite 核心软件的贡献者都签署了声明， 明确放弃对代码的任何版权主张。 这意味着任何人都可以合法地对 SQLite 源代码为所欲为。

还有其他 SQL 数据库引擎采用宽松许可证， 它们允许代码被广泛自由地使用。 但那些引擎仍受版权法约束。 SQLite 不同之处在于版权法根本不适用于它。

其他 SQL 数据库引擎的源代码文件通常以注释开头， 描述您查看和复制该文件的法律权利。 SQLite 的源代码没有许可证，因为它不受版权法管辖。 取而代之的是一段祝福：

SQL 语言扩展

SQLite 为 SQL 语言提供了一些其他数据库引擎通常不具备的增强功能。 前文已提及 EXPLAIN 关键字与显式类型。 SQLite 还提供像 REPLACE 以及 ON CONFLICT 子句这样的语句， 以便对约束冲突的处理提供更多控制。 SQLite 支持 ATTACH 和 DETACH 命令， 允许在同一查询中使用多个独立数据库。 SQLite 还定义了 API，允许用户添加新的 SQL 函数 以及 排序序列。
