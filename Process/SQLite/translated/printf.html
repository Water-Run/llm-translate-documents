1. 概览

SQLite 包含一个自己的字符串格式化例程 "printf()" 实现， 可以通过以下接口访问：

  • format() → 一个返回格式化字符串的 SQL 函数
  • sqlite3_mprintf() → 将格式化字符串存储在通过 sqlite3_malloc64() 获取的内存中。
  • sqlite3_snprintf() → 将格式化字符串存储在静态缓冲区中
  • sqlite3_str_appendf() → 将格式化文本附加到动态字符串
  • sqlite3_vmprintf() → sqlite3_mprintf() 的变参版本
  • sqlite3_vsnprintf() → sqlite3_snprintf() 的变参版本
  • sqlite3_str_vappendf() → sqlite3_str_appendf() 的变参版本

相同的核心字符串格式化器也被 SQLite 内部使用。

1.1. 优势

为什么 SQLite 要有自己的内置 printf() 实现？ 为什么不使用标准 C 库中的 printf() 实现？ 有几个原因：

  1. 通过使用自己的内置实现，SQLite 可以保证输出在所有平台和所有地区设置中都是一致的。 这对于一致性和测试非常重要。如果一台机器输出 "5.25e+08" 而另一台输出 "5.250e+008"，这会带来问题。 两个答案都正确，但 SQLite 始终给出相同答案要好得多。

  2. 我们不知道有什么方法可以使用标准库 printf() 的 C 接口来实现 format() SQL 函数 功能。然而，内置的 printf() 实现很容易适配这个任务。

  3. SQLite 中的 printf() 支持新的非标准替换类型（%q、%Q、%w 和 %z）以及增强的替换行为（%s 与 %z），这对 SQLite 内部以及使用 SQLite 的应用都很有用。 标准库 printf() 通常无法扩展到这种程度。

  4. 通过 sqlite3_mprintf() 和 sqlite3_vmprintf() 接口， 内置 printf() 实现支持将任意长度的字符串渲染到通过 sqlite3_malloc64() 获取的内存缓冲区中。 这比尝试预先计算结果字符串的最大长度、分配合适大小的缓冲区并调用 snprintf() 更安全、更不易出错。

  5. SQLite 特有的 printf() 支持一个名为“alternate-form-2”的新旗标（!）。 alternate-form-2 旗标以微妙方式改变浮点转换的处理，使输出始终是 SQL 兼容的浮点数文本表示——这是标准库 printf() 无法做到的。 对于字符串替换，alternate-form-2 旗标使宽度和精度以字符而非字节为单位，这简化了包含多字节 UTF8 字符的字符串的处理。

  6. 内置的 SQLite printf() 具有编译时选项，如 SQLITE_PRINTF_PRECISION_LIMIT，能为将 printf() 功能暴露给不受信任用户的应用提供抗拒绝服务攻击的防御。

  7. 使用内置 printf() 实现意味着 SQLite 少依赖主机环境，使其更具可移植性。

1.2. 劣势

公平地说，拥有内置 printf() 实现也有一些劣势。主要包括：

  1. 内置 printf() 实现使用额外的代码空间 （在 GCC 5.4 的 -Os 下约为 7800 字节）。

  2. 内置 printf() 的浮点到文本转换子函数在精度上限制为 16 位有效数字， 如果使用 "!" alternate-form-2 旗标则限制为 26 位有效数字。 每个 IEEE-754 双精度数都可以 精确地 表示为十进制值，但对于许多双精度数，精确的十进制表示需要超过 16 或 26 位有效数字。 SQLite printf() 函数只渲染前 16 或 26 位有效数字，因为这样效率更高，并且 16 位十进制数字足以区分每一个可能的双精度值。 对于那些需要精确十进制等价的罕见情况，请使用 十进制扩展。

  3. 内置 snprintf() 实现的缓冲区指针和缓冲区大小参数的顺序与标准库实现相反。

  4. 内置 printf() 实现不处理 posix 定位引用修饰符，这些修饰符允许 printf() 的参数顺序与 %-替换顺序不同。 在内置 printf() 中，参数的顺序必须严格与 %-替换的顺序相符。

尽管存在劣势，开发者认为在 SQLite 内部拥有内置 printf() 实现总体上是积极的。

2. 格式化细节

printf() 的格式字符串是生成字符串的模板。 每当格式字符串中出现 "%" 字符时就进行一次替换。 "%" 后面跟一个或多个描述替换的字符。 每个替换的格式如下：

%[旗标][宽度][.精度][长度]类型

所有替换都以一个 "%" 开始，并以一个类型字符结束。 替换的其他元素是可选的。

要在输出中包含单个 "%" 字符，请在模板中放置两个连在一起的 "%" 字符。

2.1. 替换类型

下表显示 SQLite 支持的替换类型：

替换类型含义
% 连续两个 "%" 字符在输出中翻译为一个 "%"， 不替换任何值。
d, i 参数是一个有符号整数，以十进制显示。
u 参数是一个无符号整数，以十进制显示。
f 参数是一个双精度数，以十进制显示。
e, E 参数是一个双精度数，以指数表示法显示。 指数字符是 'e' 或 'E'，取决于类型。
g, G 参数是一个双精度数，如果指数不接近零则以指数表示法显示， 否则以普通十进制表示。
x, X 参数是一个整数，以十六进制显示。 小写十六进制用于 %x，大写用于 %X
o 参数是一个整数，以八进制显示。
s, z 参数要么是一个以零终止的字符串并显示出来， 要么是一个被视为空字符串的空指针。 对于 C 语言接口中的 %z 类型，在字符串复制到输出后会调用 sqlite3_free()。 对于 SQL printf() 函数，%s 和 %z 替换是相同的，NULL 参数视为空字符串。

%s 替换在所有 printf 函数中通用，但 %z 替换和对空指针的安全处理是 SQLite 的增强功能， 在其他 printf() 实现中找不到。 注意：GNU printf 的新版本使用 %z 来渲染 size_t 整数值。这与 SQLite 对 %z 的使用不兼容。
c 对于 C 语言接口，参数是一个被解释为字符的整数。 对于 format() SQL 函数，参数是一个字符串，提取其中的第一个字符并显示。
p 参数是一个以十六进制地址显示的指针。 由于 SQL 语言没有指针概念，format() SQL 函数 中的 %p 替换表现得像 %x。
n 参数是指向整数的指针。此替换类型不在输出中显示任何内容。 相反，参数指向的整数会被覆写为生成字符串中位于 %n 左侧的所有格式符号所产生字符的数量。
q, Q 参数是一个以零终止的字符串。字符串中的所有单引号 (') 字符会被加倍，以便安全地出现在 SQL 字符串字面值中。 %Q 替换类型还会在替换后的字符串两端加上单引号。

如果 %Q 的参数是空指针，则输出为未加引号的 "NULL"。 换句话说，空指针产生 SQL NULL，非空指针产生有效的 SQL 字符串字面值。 如果 %q 的参数是空指针，则不生成输出。因此 %q 的空指针等价于空字符串。

如果对 %Q 添加了 alternate-form-1 旗标（例如 "%#Q"）， 那么 Unicode 范围 U+0001 到 U+001f 的控制码会使用 JSON 风格的反斜杠转义，整个输出被包裹在 "unistr('...')" 中。 因此输出对于解释 ANSI 转义序列 的设备来说是安全的。 "%#q" 转换也会对控制字符进行转义，但不会添加 unistr() 包装器，因为 %q 旨在使用在更大的字符串字面值中，并且不控制边界标记。

对于这些替换，精度表示从参数中取出的字节或字符数量，而不是写入输出的数量。

%q 和 %Q 替换是 SQLite 的增强功能，其他大多数 printf() 实现没有。
w 这个替换与 %q 类似，但它对所有双引号 (") 字符加倍，使结果适用于 SQL 语句中的双引号标识符。

%w 替换是 SQLite 的增强功能，其他大多数 printf() 实现中没有。

2.2. 可选的长度字段

参数值的长度可以通过出现在替换类型字母之前的一个或多个字母来指定。 在 SQLite 中，长度只对整数类型有效。对于始终使用 64 位值的 format() SQL 函数，长度会被忽略。 下表显示 SQLite 允许的长度说明符：

长度说明符 含义
(默认) "int" 或 "unsigned int"。在所有现代系统上都是 32 位。
l "long int" 或 "long unsigned int"。在所有现代系统上也是 32 位。
ll "long long int"、"long long unsigned" 或 "sqlite3_int64"、"sqlite3_uint64" 值。 在所有现代系统上都是 64 位整数。

只有 "ll" 长度修饰符会对 SQLite 产生影响，并且仅在使用 C 语言接口时才会生效。

2.3. 可选的宽度字段

宽度字段指定输出中替换值的最小宽度。 如果写入输出的字符串或数字比宽度短，则会填充。默认情况下填充发生在左侧（值右对齐）。 如果使用 "-" 旗标，则填充发生在右侧，值左对齐。

宽度默认以字节为单位进行衡量。但是如果存在 "!" 旗标，则以字符为单位。 这只对多字节 utf-8 字符产生影响，这些只在字符串替换中出现。

如果宽度是单个 "*" 字符而不是数字，则实际宽度值从参数列表中读取一个整数。 如果读取的值为负数，则宽度取绝对值，并将值视为左对齐，就像 "-" 旗标存在一样。

如果被替换的值大于宽度，则完整值会写入输出。 换句话说，宽度是输出中渲染值的最小宽度。

2.4. 可选的精度字段

如果存在精度字段，它必须在宽度之后、由单个 "." 字符隔开。 如果没有宽度，则引入精度的 "." 会紧跟在旗标（如果有）或初始 "%" 之后。

对于字符串替换 %s、%z、%q、%Q 或 %w，精度是从参数中使用的字节或字符数量。 默认按字节计数，但如果存在 "!" 旗标，则按字符计数。 如果没有精度，则替换整个字符串。 例如： "%.3s" 替换参数字符串的前 3 个字节。 "%!.3s" 替换参数字符串的前三个字符。

对于整数替换 %d、%i、%x、%X、%o 和 %p，精度指定显示的最小位数。 如果需要，将添加前导零以扩展输出至最小位数。

对于浮点替换 %e、%E 和 %f，精度指定小数点右侧显示的位数。 对于 %g 和 %G，精度是总的有效数字数量，如果指定精度为 0，则四舍五入至 1。

对于字符替换 %c，当精度 N 大于 1 时，会重复该字符 N 次。 这是 SQLite 独有的非标准扩展。

如果精度是单个 "*" 字符而不是数字，则实际精度值从参数列表中读取一个整数。

2.5. 选项旗标字段

旗标由紧跟在引入替换的 "%" 后面的零个或多个字符组成。各旗标及其含义如下：

旗标 含义
- 在输出中左对齐值。默认是右对齐。 如果宽度为零或小于被替换值的长度，则不进行填充，"-" 旗标不起作用。
+ 对于带符号的数字替换，在正数前加上 "+"。 无论旗标设置如何，负数前总是显示 "-"。
(空格) 对于带符号的数字替换，在正数前添加一个空格。
0 （零填充选项） 在数字替换前填充尽可能多的 "0" 字符，以扩展值至指定宽度。如果省略宽度字段，则该旗标无效。 无限和 NaN（非数）浮点值通常呈现为 "Inf" 和 "NaN"，但启用零填充选项后，它们会呈现为 "9.0e+999" 和 "null"。 换句话说，在零填充选项下，浮点的无限和 NaN 会呈现为有效的 SQL 和 JSON 字面值。
# 这是 "alternate-form-1" 旗标。其含义取决于具体转换。
  • 对于 %#g 和 %#G 替换，此旗标会移除尾随的零。
  • 对于 %#f 替换，如果省略 '+' 旗标且所有显示的数字都是零，则负号会被抑制。 例如，'%#.2f' 会将 -0.004 渲染为 "0.00" 而不是 "-0.00"。
  • 此旗标会强制所有浮点替换显示小数点。
  • 对于 %#o、%#x 和 %#X 替换，alternate-form-1 旗标会在值前分别添加 "0"、"0x" 或 "0X"。
  • 对于 %#Q，alternate-form-1 旗标会将包含 C0 控制码（ASCII 字符 0x01 到 0x1f）的字符串渲染为 调用 unistr() SQL 函数 并传入包含 JSON 风格反斜杠转义来编码那些控制码以及 "\" 字符本身的字符串字面值。
  • 对于 %#q，生成的文本会对任意控制码和 '\' 字符添加反斜杠转义，但不会添加 unistr('...') 包装器， 因为 %q 旨在在更大的字符串字面值中使用，并不控制边界标记。
, 逗号选项会在数字替换（%d、%f 等）输出的小数点左侧每隔三位添加逗号分隔符。 小数点右侧的数字不添加逗号。这有助于人类更容易辨别大整数值的数量级。 例如，值 2147483647 用 "%d" 呈现为 "2147483647"，但用 "%,d" 会呈现为 "2,147,483,647"。 此旗标是非标准扩展。
! 这是 "alternate-form-2" 旗标。 对于字符串替换，此旗标使宽度和精度以字符而非字节为单位理解。 对于浮点替换，alternate-form-2 旗标将最大有效数字数量从 16 提高到 26， 强制显示小数点并确保小数点后至少有一个数字出现。

据我们所知，alternate-form-2 旗标是一个非标准扩展，其他 printf() 实现中没有出现。

3. 实现与历史

核心字符串格式化例程是位于 printf.c 源文件中的 sqlite3VXPrintf() 函数。 所有各种接口（有时通过间接方式）都会调用这个核心函数。 sqlite3VXPrintf() 函数最初是 SQLite 的第一作者（Hipp）在 1980 年代末在杜克大学读研究生时编写的代码。 Hipp 将这个 printf() 实现保留在他的个人工具箱中，直到 2000 年开始开发 SQLite。 该代码在 2000-10-08 被并入 SQLite 源代码树，属于 SQLite 1.0.9 版本。

Fossil 版本控制系统 使用的 printf() 实现派生自 SQLite printf() 实现的早期版本，但这两个实现后来已经分道扬镳。

sqlite3_snprintf() 函数的缓冲区指针和缓冲区大小参数的顺序与标准 C 库 snprintf() 例程中的顺序相反。 这是因为 Hipp 最初实现他的版本时，标准 C 库中还没有 snprintf() 例程，他选择了与标准 C 库设计者不同的顺序。
