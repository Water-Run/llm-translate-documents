调试提示

以下是SQLite开发者用来追踪、检查并理解核心SQLite库行为的各种技巧的随意组合。

这些技巧旨在帮助理解核心SQLite库自身，而不是仅仅使用SQLite的应用程序。

  1. 在命令行 shell上使用 ".eqp full" 选项

    当你在调试或试图理解一段SQL脚本时，通常有必要在命令行 shell中以 ".eqp full" 设置运行它。".eqp"设置为FULL时，shell会在真正执行每个命令之前自动显示该命令的EXPLAIN与EXPLAIN QUERY PLAN输出。

    为提高可读性，还可以设置 ".echo on"，使输出包含原始SQL文本。

    较新的 ".eqp trace" 命令完成 ".eqp full" 做的全部工作，同时还开启VDBE 跟踪。

  2. 使用编译时选项启用调试功能。

    建议的编译时选项包括：

    • -DSQLITE_DEBUG
    • -DSQLITE_ENABLE_EXPLAIN_COMMENTS
    • -DSQLITE_ENABLE_TREETRACE
    • -DSQLITE_ENABLE_WHERETRACE

    3. 由于不属于官方支持，编译时选项文档中未记录SQLITE_ENABLE_TREETRACE与SQLITE_ENABLE_WHERETRACE选项。它们的作用是在命令行 shell 中激活 ".treetrace" 与 ".wheretrace" 点命令，分别为生成SELECT与DML语句以及WHERE子句的逻辑提供低级跟踪输出。

  3. 从调试器中调用 sqlite3ShowExpr() 等函数。

    当使用SQLITE_DEBUG编译时，SQLite会包含一些例程，可以将各种内部抽象语法树结构以ASCII艺术图的形式打印出来。这些例程在调试期间可用于了解SQLite所处理的变量。下列例程可用：

    • void sqlite3ShowExpr(const Expr*);
    • void sqlite3ShowExprList(const ExprList*);
    • void sqlite3ShowIdList(const IdList*);
    • void sqlite3ShowSrcList(const SrcList*);
    • void sqlite3ShowSelect(const Select*);
    • void sqlite3ShowWith(const With*);
    • void sqlite3ShowUpsert(const Upsert*);
    • void sqlite3ShowTrigger(const Trigger*);
    • void sqlite3ShowTriggerList(const Trigger*);
    • void sqlite3ShowTriggerStep(const TriggerStep*);
    • void sqlite3ShowTriggerStepList(const TriggerStep*);
    • void sqlite3ShowWindow(const Window*);
    • void sqlite3ShowWinFunc(const Window*);

    这些例程不是API，可能随时变化，仅供交互式调试使用。

  4. 在 test_addoptrace 上设置断点

    在调试字节码生成器时，了解某个操作码是在哪里生成的常常很有用。为便于查找，请在调试器中运行脚本。在 "test_addoptrace" 例程上设置断点，然后运行 "PRAGMA vdbe_addoptrace=ON;" 以及出问题的SQL语句。每个操作码在附加到VDBE程序时都会被显示，紧接着断点就会触发。继续逐步执行直至到达该操作码，然后向后查看调用栈以确定其生成位置与方式。

    仅在使用SQLITE_DEBUG编译时有效。

  5. 使用 ".treetrace" 和 ".wheretrace" shell 命令

    当命令行 shell 与核心SQLite库都使用SQLITE_DEBUG以及SQLITE_ENABLE_TREETRACE与SQLITE_ENABLE_WHERETRACE编译后，shell会提供两个命令，用于为处理SELECT语句和WHERE子句的最复杂的代码生成部分开启调试功能。

    ".treetrace" 与 ".wheretrace" 命令都接受一个可以用十六进制表示的数字参数。每一位都会开启不同的调试部分。通常使用 "0xfff" 与 "0xff" 这样的值。使用参数 "0" 可以关闭所有跟踪输出。

  6. 使用 ".breakpoint" shell 命令

    CLI 中的 ".breakpoint" 命令只是调用名为 "test_breakpoint()" 的过程，该过程本身无任何操作。

    如果有一段脚本并希望在其中某处开始调试，只需在 gdb（或所用的其他调试器）中为 test_breakpoint() 函数设置断点，并在想要停止的位置添加 ".breakpoint" 命令。当到达那个第一个断点时，再设置所需的其他断点或变量跟踪。

  7. 禁用旁路内存分配器

    在查找内存分配问题（内存泄漏、释放后使用错误、缓冲区溢出等）时，有时通过禁用旁路内存分配器然后在 valgrind、MSAN 或其他堆内存调试工具下运行测试会很有帮助。

    启动时可以通过SQLITE_CONFIG_LOOKASIDE 接口禁用旁路内存分配器。命令行 shell如果以 "--lookaside 0 0" 命令行选项启动，也会使用该接口禁用旁路。
