DBSTAT 虚拟表

1. 概述

DBSTAT 虚拟表是一个只读的同名虚拟表，用于返回有关用于存储 SQLite 数据库内容的磁盘空间量的信息。DBSTAT 虚拟表的示例用途包括 sqlite3_analyzer.exe 实用程序以及 表大小饼图，该饼图位于 Fossil 实现的 SQLite 版本控制系统中。

当 SQLite 使用 SQLITE_ENABLE_DBSTAT_VTAB 编译时，所有 数据库连接 都可以使用 DBSTAT 虚拟表。

DBSTAT 虚拟表是一个 同名虚拟表，这意味着在使用它之前不必运行 CREATE VIRTUAL TABLE 来创建 dbstat 虚拟表的实例。可以像使用表名一样直接使用 “dbstat” 模块名称来查询 dbstat 虚拟表。例如： 

SELECT * FROM dbstat;

如果希望使用 dbstat 模块的命名虚拟表，推荐的创建方式如下： 

CREATE VIRTUAL TABLE temp.stat USING dbstat(main);

注意虚拟表名称（“stat”）前面的 “temp.” 限定符。该限定符会使虚拟表成为临时表 —— 仅在当前数据库连接期间存在。这是推荐做法。

传递给 dbstat 的 “main” 参数指定要提供信息的默认模式。默认是 “main”，所以上述示例中的 “main” 是多余的。对于某个特定查询，可以在查询的 FROM 子句中将替代模式作为函数参数传递给虚拟表名称，以改变模式。（有关详细信息，请参见 FROM 子句中的表值函数。）

DBSTAT 虚拟表的模式如下： 

CREATE TABLE dbstat(
  name       TEXT,        -- Name of table or index
  path       TEXT,        -- Path to page from root
  pageno     INTEGER,     -- Page number, or page count
  pagetype   TEXT,        -- 'internal', 'leaf', 'overflow', or NULL
  ncell      INTEGER,     -- Cells on page (0 for overflow pages)
  payload    INTEGER,     -- Bytes of payload on this page or btree
  unused     INTEGER,     -- Bytes of unused space on this page or btree
  mx_payload INTEGER,     -- Largest payload size of all cells on this row
  pgoffset   INTEGER,     -- Byte offset of the page in the database file
  pgsize     INTEGER,     -- Size of the page, in bytes
  schema     TEXT HIDDEN, -- Database schema being analyzed
  aggregate  BOOL HIDDEN  -- True to enable aggregate mode
);

DBSTAT 表只报告数据库文件中 btree 的内容。Freelist 页面、指针映射页面和锁页面不纳入分析。

默认情况下，DBSTAT 表中对于数据库文件的每个 btree 页面都有一行。每行提供有关该页面空间利用率的信息。但是，如果隐藏列 “aggregate” 为 TRUE，则结果会被汇总，DBSTAT 表中每个 btree 只有一行，提供整棵 btree 的空间利用信息。

2. dbstat 虚拟表的 “path” 列

“path” 列描述了从 btree 结构的根节点到每个页面的路径。根节点本身的 “path” 为 '/'。当 “aggregate” 为 TRUE 时，“path”为 NULL。

btree 页面根节点最左侧的子页面的 “path” 是 '/000/'。（Btree 以从左到右的顺序存储内容，因此左边的页面拥有比右边页面更小的键。）根页面的次左子页面是 '/001/'，依此类推，每个兄弟页面由一个三位十六进制值标识。第 451 个最左侧兄弟的子页面的路径类似 '/1c2/000/'、'/1c2/001/' 等。

通过在路径后附加 '+' 字符和六位十六进制值，可以指定溢出页面，该值表示它们链接的单元格。例如，根页面第 450 个子页面的最左侧单元格链接的三个溢出页面的路径为： 

'/1c2/000+000000'         // First page in overflow chain
'/1c2/000+000001'         // Second page in overflow chain
'/1c2/000+000002'         // Third page in overflow chain

如果使用 BINARY 校对序对路径进行排序，则与某个单元格关联的溢出页面会出现在其子页面之前： 

'/1c2/000/'               // Left-most child of 451st child of root

3. 聚合数据

从 SQLite 3.31.0（2020-01-22）开始，DBSTAT 表新增了一个名为 “aggregate” 的 隐藏列，如果约束为 TRUE，将使 DBSTAT 为数据库中的每个 btree 生成一行，而不是每个页面一行。在聚合模式下，“path”、“pagetype”和“pgoffset” 列始终为 NULL，“pageno” 列保存整棵 btree 的页面数，而不是对应行的页面编号。

下表展示了 DBSTAT 的（非隐藏）列在普通模式和聚合模式下的含义：

列 普通含义 聚合模式含义
name 当前行所对应的 btree 所实现的表或索引的名称
path 见上文说明 始终为 NULL
pageno 当前行所对应数据库页面的页面编号 当前行所对应 btree 的页面总数
pagetype 'leaf' 或 'interior' 始终为 NULL
ncell 当前页面或 btree 上的单元格数量
payload 当前页面或 btree 上的有效负载字节数
unused 当前页面或 btree 上未使用的字节数
mx_payload 当前页面或 btree 中出现的最大负载大小
pgoffset 页面起始位置的字节偏移 始终为 NULL
pgsize 当前页面或 btree 使用的总存储空间

4. dbstat 虚拟表的示例用途

要查找模式 “aux1” 中表 “xyz” 使用的页面总数，可运行以下两个查询（第一个是传统方式，第二个展示聚合特性的用法）： 

SELECT count(*) FROM dbstat('aux1') WHERE name='xyz';
SELECT pageno FROM dbstat('aux1',1) WHERE name='xyz';

要了解表的内容在磁盘上的存储效率，可计算实际内容所占空间除以占用的磁盘空间总量。这个数字越接近 100%，说明填充越高效。（在本例中，假设 “xyz” 表位于 “main” 模式中。下面分别给出了在没有和启用聚合特性时使用 DBSTAT 的两种写法。） 

SELECT sum(pgsize-unused)*100.0/sum(pgsize) FROM dbstat WHERE name='xyz';
SELECT (pgsize-unused)*100.0/pgsize FROM dbstat
 WHERE name='xyz' AND aggregate=TRUE;

要查找某个表的平均扇出，请运行： 

SELECT avg(ncell) FROM dbstat WHERE name='xyz' AND pagetype='internal';

现代文件系统在磁盘访问顺序时运行更快。因此，如果数据库文件的内容分布在顺序页面上，SQLite 会更快。要找出数据库中有多少页面是顺序排列的（从而获得一个可能用于判断何时执行 VACUUM 的度量），可运行如下查询： 

CREATE TEMP TABLE s(rowid INTEGER PRIMARY KEY, pageno INT);
INSERT INTO s(pageno) SELECT pageno FROM dbstat ORDER BY path;
SELECT sum(s1.pageno+1==s2.pageno)*1.0/count(*)
  FROM s AS s1, s AS s2
 WHERE s1.rowid+1=s2.rowid;
DROP TABLE s;
