1. 从损坏的 SQLite 数据库恢复（部分）数据

SQLite 数据库异常稳健。应用程序故障和电源中断通常不会破坏数据库的内容。尽管如此，仍然有可能损坏 SQLite 数据库。例如，硬件故障可能会损坏数据库文件，或者某个异常进程可能打开数据库并覆盖其中的部分内容。

面对损坏的数据库文件时，有时希望尽可能多地从中挽救数据。恢复 API 的设计就是为了便于完成这项工作。

1.1. 限制

有时可以完美恢复已经损坏的数据库，但那只是例外。通常恢复后的数据库在多个方面都会存在瑕疵：

  • 部分内容可能永久丢失且无法恢复。比如某个异常进程覆盖了数据库文件的一部分时，就会发生这种情况。

  • 之前已删除的内容可能会重新出现。通常当 SQLite 执行 DELETE 操作时，它并不会真正覆盖旧内容，而是记住该空间可以在下一次 INSERT 时重用。如果在尝试恢复时此类已删除内容仍存在于文件中，就可能被提取出来并“复活”。

  • 恢复得到的内容可能被修改。例如，某行中存储的值可能从 48 变成 49，或者从整数变成字符串或 BLOB。原本为 NULL 的值可能变成整数，字符串值可能变成 BLOB，以此类推。

  • 恢复之后约束可能不再有效。CHECK 约束、FOREIGN KEY 约束、UNIQUE 约束、STRICT 表上的类型约束——恢复后的数据库可能违反其中任意一项。

  • 内容可能会从一个表移动到另一个表。

恢复 API 会尽可能恢复数据库，但结果始终值得怀疑。有时（例如损坏仅限于索引）恢复可以完美地还原数据库内容。但在其他情况下，恢复可能并不完美。这种不完美的影响取决于应用场景。保存书签列表的数据库在恢复之后仍旧是书签列表。恢复后可能会缺失、添加或修改少数书签，但该列表本来就“模糊”且不完美，因此再增加一点不确定性通常不会对应用造成致命影响。但是如果会计数据库损坏后恢复，账目可能会不平衡。

最好把恢复 API 看作一次打捞行动。恢复会从旧数据库的残骸中提取尽可能多的可用数据，但有些部分可能已经损坏得无法修复。在将恢复后的数据库投入使用前，应对其进行一些修复和测试。

2. 在 CLI 中使用“.recover”命令恢复

手动恢复损坏数据库最简单的方法是使用 命令行界面（CLI）。SQLite 的 CLI 程序名为“sqlite3”。使用类似下面的命令从损坏的数据库文件中恢复数据： 

sqlite3 corrupt.db .recover >data.sql

这将会在名为“data.sql”的文件中生成 SQL 文本，可用于重新构建原始数据库： 

sqlite3 recovered.db <data.sql

“.recover”选项实际上是一条发送给 CLI 的命令。该命令可以接受参数。例如，运行： 

sqlite3 corruptdb ".recover --ignore-freelist" >data.sql

请注意，“.recover”命令及其参数必须包含在引号中。支持以下选项：

--ignore-freelist

忽略看起来属于空闲列表（freelist）的数据库页。通常会扫描空闲列表，如果其中包含看起来有内容的页，就会输出这些内容。但如果该页确实在空闲列表中，这可能意味着之前被删除的信息会重新写回数据库。

--lost-and-found TABLE

如果在恢复过程中发现的内容无法关联到特定表，就会被放入“lost_and_found”表。使用此选项可将“lost_and_found”表的名称更改为“TABLE”。

--no-rowids

如果提供此选项，则不会从损坏的数据库中提取那些既不是 INTEGER PRIMARY KEY 的 rowid 值。

3. 将恢复 API 集成到应用程序

3.1. 源代码文件

如果希望将恢复 API 集成到应用程序中，则除了常规的 “sqlite3.c” 和 “sqlite3.h” 源文件之外，还需要将一些源文件添加到构建中。需要的文件有：

sqlite3recover.c 这是实现恢复 API 的主源文件。
sqlite3recover.h 这是与 sqlite3recover.h 搭配使用的头文件。
dbdata.c 该文件实现了名为 “sqlite_dbdata” 和 “sqlite_dbptr” 的两个虚拟表，sqlite3recover.c 需要它们。

上述两个 C 源文件需要以与 “sqlite3.c” 相同的方式链接到应用程序中。头文件在编译这些 C 文件时也必须能够被编译器访问。

另外，应用程序，或更准确地说链接进去的 sqlite3.c，必须使用以下选项进行编译： 

        -DSQLITE_ENABLE_DBPAGE_VTAB

3.2. 如何实现恢复

从损坏的数据库中恢复内容的基本步骤如下：

  1. 通过调用 sqlite3_recover_init() 或 sqlite3_recover_init_sql() 创建一个 sqlite3_recover 句柄。使用 sqlite3_recover_init() 将恢复的内容存放到一个单独的数据库中，使用 sqlite3_recover_init_sql() 生成可重建数据库的 SQL 文本。

  2. 对新的 sqlite3_recovery 句柄进行零次或多次 sqlite3_recover_config() 调用，以设置选项。

  3. 重复调用 sqlite3_recover_step()，直到其返回不是 SQLITE_OK 的值。如果它返回 SQLITE_DONE，则恢复操作已无错误地完成。如果返回其他非 SQLITE_OK 值，则说明发生了错误。sqlite3_recover_run() 接口也可用，作为一个便利的包装，它会重复调用 sqlite3_recover_step()，直到返回不是 SQLITE_DONE 的值。

  4. 使用 sqlite3_recover_errcode() 和 sqlite3_recover_errmsg() 接口分别检索任何错误代码和英文错误消息。

  5. 调用 sqlite3_recover_finish() 来销毁 sqlite3_recover 对象。

接口的详细信息记录在 sqlite3_recover.h 头文件 的注释中。
