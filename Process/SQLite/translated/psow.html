Powersafe 覆盖写入

“Powersafe overwrite” 是 SQLite 团队用来描述某些文件系统和磁盘控制器在断电期间与数据保护相关行为的术语。Powersafe overwrite 是一个布尔属性：存储系统要么具备它，要么不具备。

我们说某个系统具有 powersafe overwrite 属性，当下述陈述为真：

当一个应用程序在文件中写入一段字节时，即使写入恰好发生在崩溃或断电之前，该范围之外的字节也不会改变。

powersafe overwrite 属性并不说明被写入的字节的状态。那些字节可能保留旧值、使用新值、包含随机值，或其中的某种组合。powersafe overwrite 属性仅表示写入不会改变写入范围之外的字节。

换句话说，powersafe overwrite 表示在写入过程中发生断电时不会造成“连带损坏”。只有正在写入的那些字节可能会受到损害。

从实际角度看，powersafe write 属性意味着当磁盘控制器检测到即将断电时，它会在停靠磁头之前将正在处理的扇区写完。这意味着一旦单个扇区的写入开始，即使发生断电，该扇区也会完成写入。

考虑如果磁盘扇区写入被断电打断会发生什么。如果一个应用程序在某个文件的中间写入两三个字节，操作系统会先读取包含这些字节的整个扇区，在内存中修改该扇区，然后再将整个扇区写回磁盘。如果在写回过程中断电并且扇区未完全写入，那么在重启后的下一次读取时，扇区内的纠错码很可能检测到无法修复的损坏，磁盘控制器会将该扇区以全 0 或全 1 的形式读取出来。因此，在应用层写入的两三个字节范围之外的值也会改变——这就违反了 powersafe overwrite 属性。

SQLite 关于 Powersafe 覆写的假设

直到（包括）版本 3.7.9（2011-11-01），所有 SQLite 版本都假设文件系统不提供 powersafe overwrite。SQLite 传统上认为，只要文件中的任何一个字节发生变化，该字节所在扇区的所有其他字节在断电时都有可能损坏。在写入时，SQLite 确保对任何修改所在扇区内的所有字节进行日志记录，并将日志文件填充到下一个扇区边界，以便随后对该日志的追加不会损坏先前的记录。SQLite 理解的扇区大小是由 VFS 中的 xSectorSize 方法返回的值。SQLite 团队经常将 xSectorSize 返回的值称为写入的“爆炸半径”，因为它表示在写入过程中断电时可能遭到损坏的字节范围。直到并包括版本 3.7.9，Unix 和 Windows 的默认 VFSes 一直返回 512 作为扇区大小（或爆炸半径）。

不过，较新的磁盘驱动器已经开始使用 4096 字节的扇区。从 SQLite 版本 3.7.10（2012-01-16）开始，开发团队试验性地修改 xSectorSize，以报告 4096 字节作为爆炸半径。这导致许多数据库的写入开销增加。对于具有 PRAGMA page_size 为 1024（一个非常常见的选择）的数据库来说，对数据库中的单页进行修改时，如今需要 SQLite 将另外三个相邻页面备份到回滚日志，而过去只需备份正在修改的那一页。在 WAL 模式 下，每个事务必须填充到 WAL 文件中的下一个 4096 字节边界，而不是 512 字节边界，导致每个事务多写入数千个字节。

额外的写入开销促使我们重新审视关于 powersafe overwrite 的假设。随着现代磁盘容量变得越来越大、数据密度越来越高，单个扇区变得非常小，写入单个扇区所需时间也非常短。我们知道磁盘驱动器可以检测到即将到来的断电，并能够利用剩余能量在磁头停止旋转之前将磁头停靠。因此，如果磁盘控制器可以检测到即将断电，那么当首次检测到断电临近时，控制器很可能会在停靠磁头之前把当前正在写入的扇区写完，只要这样做不会花费过多时间——对于小而密集的扇区而言，通常不会。鉴于此，假设现代磁盘具备 powersafe overwrite 属性似乎是合理的。事实上，据说 BerkeleyDB 多年来一直这么做。不过仍需谨慎。正如 Roger Binns 在 SQLite 开发者邮件列表中指出的：“对驱动固件的主要假设应该是‘写得很糟糕’。”

撕裂页面

当数据库页面大于磁盘扇区、数据库页面已写入磁盘，但断电发生在页面的所有扇区写入完成之前时，就会出现撕裂页面。恢复时，页面的某些部分会保留旧内容，而其他部分则拥有新内容。有些数据库引擎假设页面写入是原子的，因此认为撕裂页面是不可恢复的错误。

SQLite 从不假设数据库页面写入是原子的，无论 PSOW 设置如何。(1)因此 SQLite 总能自动从因崩溃引起的撕裂页面中恢复。启用 PSOW 并不会降低 SQLite 从撕裂页面中恢复的能力。

SQLite 3.7.10 版本的变更

SQLite 的 VFS 在版本 3.7.10（2012-01-16）中新增了一个名为 SQLITE_IOCAP_POWERSAFE_OVERWRITE 的设备特性。报告该特性的数据库文件被假设托管在具有 powersafe overwrite 属性的存储系统上。默认的 Unix 和 Windows VFSes 在 SQLite 使用 -DSQLITE_POWERSAFE_OVERWRITE=1 编译时会报告 SQLITE_IOCAP_POWERSAFE_OVERWRITE，若使用 -DSQLITE_POWERSAFE_OVERWRITE=0 编译，则会沿用传统假设：存储系统不具备 powersafe overwrite 属性。目前默认启用了 powersafe overwrite，尽管未来我们可能重新评估并将其默认关闭。

可以在打开单个数据库时通过 URI 文件名的 “psow” 查询参数来指定 powersafe overwrite 属性。例如，为了始终假设某个文件具有 powersafe overwrite（也许是为了确保最大的写入性能），可以这样打开：

file:somefile.db?psow=1

或者为了对某个数据库更加谨慎，并强制 SQLite 假设该数据库不具备 powersafe overwrite，可以这样打开：

file:somefile.db?psow=0

还有一个新的 SQLITE_FCNTL_POWERSAFE_OVERWRITE 操作码，可用于 sqlite3_file_control()，允许应用程序查询数据库文件的 powersafe overwrite 属性。

附注：

  1. 在其默认配置中，SQLite 从不假设页面写入是原子的。但一个自定义的 VFS 可以在 xDeviceCharacteristic() 方法的结果中设置某一个 SQLITE_IOCAP_ATOMIC 位，然后 SQLite 会假设页面写入是原子的。不过，应用程序必须提供一个自定义的 VFS 来实现这一点，因为标准 VFSes 中没有任何一个会在 xDeviceCharacteristics() 向量中设置任何一个原子位。
