本文档介绍了 WAL 模式 在 Unix 与 Windows 上的实现的低层细节。

单独的 文件格式 描述提供了数据库文件以及 WAL 模式下写前日志文件结构的详细信息。但锁协议以及 WAL 索引的格式细节被故意省略，因为这些细节留给各个 VFS 实现自行决定。本文档为 Unix 和 Windows 的 VFS 补充了那些缺失的细节。

为了完整性，当与 WAL 模式处理相关时，一些出现在 文件格式 文档及其他地方的更高层格式信息也会在此重复说明。

1. 磁盘上的文件

在积极使用中，WAL 模式数据库的状态由三个独立的文件描述：

  1. 主数据库文件，名称不定，记作 “X”。
  2. 写前日志文件，通常命名为 “X-wal”。
  3. wal 索引文件，通常命名为 “X-shm”。

1.1. 主数据库文件

主数据库文件的格式如 文件格式 文档所述。主数据库中偏移 18 与 19 处的 文件格式版本号 必须都为 2，以表示数据库处于 WAL 模式。主数据库可以拥有底层文件系统允许的任意名称。不需要特殊后缀，尽管 ".db"、".sqlite" 和 ".sqlite3" 是常见选择。

1.2. 写前日志或“–wal”文件

写前日志或 “wal” 文件是一个向前滚动的日志，用于记录已经提交但尚未应用到主数据库的事务。wal 文件的格式细节在主 文件格式 文档中的 WAL 格式 小节中描述。wal 文件的名称是在主数据库文件名后追加四个字符 “-wal”。除非在 8+3 文件系统上，在这种情况下该名称不被允许，此时该文件后缀将改为 ".WAL"。但随着 8+3 文件系统越来越少见，通常可以忽略这一特殊情况。

1.3. Wal-Index 或 “-shm” 文件

wal 索引文件或 “shm” 文件并不实际用作普通文件。相反，各数据库客户端将 shm 文件 mmap 映射为共享内存，用于协调数据库访问并作为快速定位 wal 文件中帧的缓存。shm 文件的名称是在主数据库文件名后追加四个字符 “-shm”。在 8+3 文件系统上，shm 文件名称则是将主数据库文件的后缀改为 ".SHM"。

shm 文件不包含任何数据库内容，因此在崩溃后恢复数据库时并不需要保留它。因此，连接到空闲数据库的第一个客户端通常会在 shm 文件存在时截断该文件。由于 shm 文件的内容不需要在崩溃后保留，所以它从不执行 fsync() 到磁盘。事实上，如果 SQLite 可以通过某种机制告诉操作系统永远不将 shm 文件持久化到磁盘而始终保留在缓存中，SQLite 会利用该机制以避免与 shm 文件相关的任何不必要磁盘 I/O。然而，在标准 POSIX 中并不存在这样的机制。

由于 shm 仅用于在并发客户端之间协调访问，因此在设置了 独占锁定模式 时，会省略 shm 文件以作为优化。设置了 独占锁定模式 后，SQLite 使用堆内存代替 mmap 的 shm 文件。

1.4. 文件生命周期

当 WAL 模式数据库处于积极使用中时，上述三个文件通常都会存在。唯一例外是当设置了 独占锁定模式 时，wal 索引文件会被省略。

如果最后一个使用数据库的客户端通过调用 sqlite3_close() 并干净地关闭，那么会自动运行一个 检查点，以便将 wal 文件中的所有信息传输回主数据库，并删除 shm 与 wal 文件。因此，当数据库没有任何客户端使用时，通常磁盘上只剩下主数据库文件。然而，如果最后一个客户端在关闭前未调用 sqlite3_close()，或者最后一个断开连接的客户端是只读客户端，那么最终的清理操作不会发生，即使数据库没有被使用，shm 与 wal 文件也可能仍然存在于磁盘上。

1.5. 变体

当设置 PRAGMA locking_mode=EXCLUSIVE（独占锁定模式）时，只允许单个客户端在某一时间打开数据库。由于只有一个客户端可以使用数据库，因此 shm 文件会被省略。该客户端使用堆内存缓冲区来替代 mmap 的 shm 文件。

如果一个读/写客户端在关闭前调用了 sqlite3_file_control(SQLITE_FCNTL_PERSIST_WAL)，那么在关闭时仍会运行一次检查点，但 shp 与 wal 文件不会被删除。这允许后续的只读客户端连接并读取数据库。

2. WAL 指数文件格式

WAL 索引或 “shm” 文件用于协调多个客户端对数据库的访问，并作为缓存帮助客户端快速定位 wal 文件中的帧。

由于 shm 文件不参与恢复，因此它不需要与机器字节序无关。因此，shm 文件中的数值以主机本地字节序写入，而不是像主数据库文件和 wal 文件那样转换为特定的跨平台字节序。

shm 文件由一个或多个哈希表组成，每个哈希表大小为 32768 字节。不过，第一张哈希表的开头预留了 136 字节的头部，因此第一张哈希表只有 32632 字节。shm 文件的总大小始终是 32768 的倍数。在大多数情况下，shm 文件的总大小恰好为 32768 字节。shm 文件只有在 wal 文件变得非常大（超过 4079 帧）时才需要超过一个哈希表。由于默认的 自动检查点阈值 是 1000，WAL 文件很少达到使 shm 文件增长所需的 4079 帧。

2.1. WAL 指数头

shm 文件的前 136 字节是一个头部。shm 头部可以划分为三个主要区域如下：

2.1.1. mxFrame 字段

偏移量 16（并在偏移 64 重复）的 32 位无符号整数表示 WAL 中的有效帧数。由于 WAL 帧从 1 开始编号，mxFrame 也是 WAL 中最后一个有效提交帧的索引。提交帧是指其帧头字节 4 到 7 中的“数据库大小”值非零的帧，并表示事务的结束。

当 mxFrame 字段为零时，表示 WAL 为空，所有内容应直接从数据库文件获取。

当 mxFrame 等于 nBackfill 时，表示 WAL 中的所有内容都已经回写到数据库。在这种情况下，所有内容都可以直接从数据库读取。此外，前提是没有其他连接持有 WAL_READ_LOCK(N)（N>0），则下一个写者可以自由地 重置 WAL。

mxFrame 的值总是大于或等于 nBackfill 和 nBackfillAttempted。

2.1.2. nBackfill 字段

WAL 索引头偏移 128 处的 32 位无符号整数称为 “nBackfill”。该字段记录已从 WAL 文件复制回主数据库的帧数。

nBackfill 数值永远不会大于 mxFrame。当 nBackfill 等于 mxFrame 时，表示 WAL 内容已经完全回写到数据库，此时在没有任何 WAL_READ_LOCK(N)（N>0）上的锁时就可以安全地 重置 WAL。

只有在持有 WAL_CKPT_LOCK 的情况下才可以增加 nBackfill。但是，在执行 WAL 重置 时，nBackfill 会被置为零，而该操作是在持有 WAL_WRITE_LOCK 的情况下完成的。

2.1.3. WAL 锁

在头部预留了 8 字节空间用于支持通过 sqlite3_io_methods 对象中的 xShmLock() 方法进行文件锁操作。由于某些 VFS（例如 Windows）可能使用强制文件锁实现锁机制，这 8 字节从未被 SQLite 读取或写入。

2.2. WAL 索引哈希表

shm 文件中的哈希表旨在快速回答以下问题：

FindFrame(P,M)：给定页号 P 与最大 WAL 帧索引 M，返回不大于 M 且页号为 P 的最大 WAL 帧索引；如果没有满足条件的帧，则返回 NULL。

令数据类型 “u8”、“u16” 与 “u32” 分别表示长度为 8、16 与 32 位的无符号整数。那么 shm 文件的第一个 32768 字节单元组织如下： 

u8 aWalIndexHeader[136];
u32 aPgno[4062];
u16 aHash[8192];

shm 文件的第二个及之后的所有 32768 字节单元如下： 

u32 aPgno[4096];
u16 aHash[8192];

aPgno 条目统一记录 WAL 文件中所有帧存储的数据库页号。第一张哈希表中的 aPgno[0] 记录 WAL 文件第一帧对应的数据库页号。第一张哈希表中的 aPgno[i] 记录 WAL 文件第 i 帧的数据库页号。第二张哈希表中的 aPgno[k] 记录 WAL 文件第 (k+4062) 帧的数据库页号。第 n 个 32768 字节哈希表（n>1）中的 aPgno[k] 记录 WAL 文件第 (k+4062+4096*(n-2)) 帧的数据库页号。

另一种描述 aPgno 值的方法是：如果将所有 aPgno 值视为一个连续数组，那么 WAL 文件的第 i 帧的数据库页号就存储在 aPgno[i] 中。当然，aPgno 并非真正的连续数组。前 4062 个条目位于 shm 文件的第一个 32768 字节单元，其后条目以每 4096 个为一个块分布在后续单元中。

一种计算 FindFrame(P,M) 的方法是从 aPgno 数组的第 M 项开始向前扫描，直到找到 aPgno[J]==P 的位置 J。这样的算法可以运作，并且比在整个 WAL 文件中搜索具有页号 P 的最新帧要快。但通过使用 aHash 结构，搜索速度可以大大提升。

数据库页号 P 使用如下哈希函数映射为哈希值：

h = (P * 383)%8192

该函数将每个页号映射为 0 到 8191（含）之间的整数。shm 文件每个 32768 字节单元的 aHash 字段将 P 值映射到同一单元的 aPgno 字段索引，具体如下：

  1. 计算哈希值：h = P * 383
  2. 令 X 为最大的一组连续整数 {h, h+1, h+2, ..., h+N}，使得对于 X 中的每个 j，aPgno[j%8192]!=0。若 aPgno[h%8192]==0，则 X 为空。X 集合可通过从 h%8192 开始，将 h%8192 加入 X 并递增 h，直到遇到首个 aPgno[h%8192] 为 0 的条目为止，从而轻松计算得出。
  3. 集合 X 包含当前 32768 字节单元中可能是 FindFrame(P,M) 解的所有 aPgno 索引。每个条目都必须单独检查，确保其 aPgno 值等于 P 且帧编号不超过 M。满足这两个条件的最大帧编号即为答案。

aPgno 数组中的每个条目都对应着 aHash 数组中的一个条目。aHash 的可用槽比 aPgno 多。aHash 中未使用的槽会以零填充。由于 aHash 中有保证未被使用的槽，因此计算 X 的循环必定会终止。X 的期望大小小于 2。最坏情况下，X 的大小等于 aPgno 条目数，此时算法的运行速度与线性扫描 aPgno 相当。但这种最坏情况极为罕见。通常 X 的大小会比较小，使用 aHash 数组可以显著加快 FindFrame(P,M) 的计算。

另一种描述哈希查找算法的方法是：从 h = (P * 383)%8192 开始，查看 aHash[h] 及后续条目，并在 h 达到 8192 后环回到 0，直到找到 aHash[h]==0 的条目。所有页号为 P 的 aPgno 条目，其索引都在上述所计算的 aHash[h] 值中。但并非所有计算出的 aHash[h] 值都会匹配，因此必须逐一验证。性能优势在于，这组 h 值通常非常小。

注意，每个 32768 字节单元都有自己的 aHash 与 aPgno 数组。单个单元的 aHash 数组仅能帮助查找该单元内的 aPgno 条目。整体的 FindFrame(P,M) 函数需要从最新的单元开始进行哈希查找，并向前推进到最旧的单元，直到找到答案。

2.3. 锁矩阵

在 WAL 模式下，访问协调既依赖于 sqlite3_io_methods 对象的 xLock 与 xUnlock 方法控制的传统 DELETE 模式锁，也依赖于同一对象的 xShmLock 方法控制的 WAL 锁。

概念上，DELETE 模式只有一个锁。单个数据库连接的 DELETE 模式锁可以处于以下五种状态之一：

  1. SQLITE_LOCK_NONE（未加锁）
  2. SQLITE_LOCK_SHARED（共享读取）
  3. SQLITE_LOCK_RESERVED（读取时等待写入）
  4. SQLITE_LOCK_PENDING（阻止新读取者，等待写入）
  5. SQLITE_LOCK_EXCLUSIVE（独占写入）

DELETE 模式锁存储在主数据库文件的 锁字节页 上。对于 WAL 模式数据库，仅有 SQLITE_LOCK_SHARED 和 SQLITE_LOCK_EXCLUSIVE 起作用。其他锁状态用于回滚模式，而非 WAL 模式。

上述 WAL 模式锁 已在前文描述。

2.3.1. 各种锁的使用方式

下面的规则展示了每个锁的使用方式。

  • SQLITE_LOCK_SHARED

    所有连接在附加到 WAL 模式数据库期间持续持有 SQLITE_LOCK_SHARED。这适用于读写连接和只读连接。即便连接未处于事务中，也会持有 SQLITE_LOCK_SHARED。这与回滚模式不同，后者在每次事务结束时才会释放 SQLITE_LOCK_SHARED。

  • SQLITE_LOCK_EXCLUSIVE

    当连接在 WAL 模式与各类回滚模式之间切换时，会持有独占锁。在断开 WAL 模式时，连接也可能尝试获取 EXCLUSIVE 锁。如果连接成功获取 EXCLUSIVE 锁，说明它是唯一连接，因此可以尝试执行检查点并删除 WAL 索引与 WAL 文件。

    当一个连接持有主数据库的 SHARED 锁时，会阻止其他连接获取 EXCLUSIVE 锁，这反过来又防止 WAL 索引与 WAL 文件在其他用户使用时被删除，并且防止在其他用户仍以 WAL 模式访问数据库时转出 WAL 模式。

  • WAL_WRITE_LOCK

    WAL_WRITE_LOCK 仅以独占方式加锁，从未持有共享锁。

    任何向 WAL 尾部追加内容的连接都会持有 EXCLUSIVE WAL_WRITE_LOCK，因此在任意时刻只有单个进程能向 WAL 追加内容。如果由于写入而导致 WAL 重置 发生，则 WAL 索引头的 nBackfill 字段会在持有此锁的情况下被重置为零。

    当连接在共享 WAL 索引上运行 恢复 时，也会独占持有 WAL_WRITE_LOCK 及其他多个锁字节。

  • WAL_CKPT_LOCK

    WAL_CKPT_LOCK 仅以独占方式加锁，从未持有共享锁。

    任何运行 检查点 的连接都会持有 EXCLUSIVE WAL_CKPT_LOCK。在持有此独占锁时，WAL 索引头的 nBackfill 字段可以增加，但不能减少。

    当连接在共享 WAL 索引上运行 恢复 时，也会独占持有 WAL_CKPT_LOCK 及其他多个锁字节。

  • WAL_RECOVER_LOCK

    WAL_RECOVER_LOCK 仅以独占方式加锁，从未持有共享锁。

    任何运行 恢复 来重建共享 WAL 索引的连接都会持有 EXCLUSIVE WAL_RECOVER_LOCK。

    只读连接在重建其私有堆内存 WAL 索引时不会持有此锁（因为只读连接不能持有任何独占锁）。该锁仅在重建内存映射 SHM 文件中的全局共享 WAL 索引时才会被持有。

    在锁定该字节的同时，运行 恢复 的连接还会在除 WAL_READ_LOCK(0) 外的所有其他 WAL 锁上获取独占锁。

  • WAL_READ_LOCK(N)

    共有五个独立的读锁，编号为 0 到 4。读锁可以是共享也可以是独占。连接在事务期间会在某个读锁字节上获取共享锁。在更新对应读取标记值的短暂时间内，还会逐个对读锁获取独占锁。运行 恢复 时，WAL_READ_LOCK(1) 到 WAL_READ_LOCK(4) 会被独占持有。

    每个读锁字节对应 WAL 索引头中第 100 到 119 字节的五个 32 位读取标记，映射如下：

    锁名称锁偏移读取标记名称读取标记偏移
    WAL_READ_LOCK(0)123read-mark[0]100..103
    WAL_READ_LOCK(1)124read-mark[1]104..107
    WAL_READ_LOCK(2)125read-mark[2]108..111
    WAL_READ_LOCK(3)126read-mark[3]112..115
    WAL_READ_LOCK(4)127read-mark[4]116..119

    当连接持有 WAL_READ_LOCK(N) 的共享锁时，表示该连接会使用 WAL 文件而非数据库文件来处理 WAL 中前 read-mark[N] 条目所修改的数据库页。read-mark[0] 始终为零。如果连接持有 WAL_READ_LOCK(0) 的共享锁，说明它期望直接从主数据库读取所需内容，而忽略 WAL。如果 N>0，则该连接可以在需要时继续使用 WAL 中的内容，最多到第一个 mxFrame 帧。但当连接持有 WAL_READ_LOCK(0) 的共享锁时，即表示它保证不会从 WAL 中读取内容，而全部直接从主数据库获取。

    当检查点运行时，如果检测到有 WAL_READ_LOCK(N) 的锁存在，则不能将 WAL 中超过 read-mark[N] 的帧移动到主数据库中。否则就会覆盖持有该锁的进程所期望从主数据库中读取的内容。因此，如果 WAL 文件中包含超过 read-mark[N] 的帧（即对某个被其他进程持有 WAL_READ_LOCK(N) 的读取标记而言，mxFrame>read-mark[N]），则检查点无法完成。

    当写者想要 重置 WAL 时，必须确保 WAL_READ_LOCK(N)（N>0）上没有任何锁，因为这些锁意味着其他连接仍在使用当前 WAL 文件，而 WAL 重置 会删除这些连接还能访问的内容。如果其他连接仅持有 WAL_READ_LOCK(0)，则可以放心地重置 WAL，因为它们承诺不使用 WAL 中的任何内容。

2.3.2. 需要锁的操作及所用锁

  • 在 WAL 模式之间切换

    希望切换进入或退出 WAL 模式的连接必须持有 SQLITE_LOCK_EXCLUSIVE。切换到 WAL 模式因此就像任何其他写事务一样，因为回滚模式中的写事务都需要 SQLITE_LOCK_EXCLUSIVE。若数据库文件已经处于 WAL 模式（即希望切换回回滚模式）且共有两个或更多连接，则每个连接都会持有 SQLITE_LOCK_SHARED。此时无法获取 SQLITE_LOCK_EXCLUSIVE，因而不允许退出 WAL 模式。这防止了某个连接在其他连接仍在使用 WAL 模式时删除 WAL 模式。同时，这也意味着要将数据库从 WAL 模式切换回回滚模式，必须关闭除一个连接以外的全部连接。

  • 关闭 WAL 模式数据库连接

    当数据库连接关闭（通过 sqlite3_close() 或 sqlite3_close_v2()）时，会尝试获取 SQLITE_LOCK_EXCLUSIVE。如果成功，则说明正在关闭的连接是最后一个连接。此时希望清理 WAL 与 WAL 索引文件，因此关闭的连接会在持有 SQLITE_LOCK_EXCLUSIVE 的情况下运行一个 检查点，该操作会删除 WAL 以及 WAL 索引文件。在 WAL 与 WAL 索引文件删除完成之前，SQLITE_LOCK_EXCLUSIVE 不会释放。

    如果应用在关闭之前对数据库连接调用了 sqlite3_file_control(SQLITE_FCNTL_PERSIST_WAL)，则仍会运行最终检查点，但 WAL 与 WAL 索引文件不会像通常那样被删除。这会使数据库处于允许其他没有写权限（对数据库、WAL 或 WAL 索引文件）的进程以只读方式打开的状态。如果 WAL 与 WAL 索引文件缺失，则没有权限创建和初始化这些文件的进程将无法打开数据库，除非该数据库通过 immutable 查询参数被标记为不可变。

  • 在 恢复 期间重建全局共享 WAL 索引

    在 恢复 过程中重建全局共享 WAL 索引时，所有 WAL 索引锁（除 WAL_READ_LOCK(0)）均以独占方式持有。

3. 恢复

恢复是重建 WAL 索引以使其与 WAL 同步的过程。

恢复由第一个连接到 WAL 模式数据库的线程运行。恢复会将 WAL 索引还原为准确描述 WAL 文件的状态。如果第一个线程连接时没有 WAL 文件，则无需恢复内容，但恢复过程仍会运行以初始化 WAL 索引。

如果 WAL 索引以一个内存映射文件实现且该文件对第一个连接线程是只读的，那么该线程会创建一个私有的堆内存 WAL 索引，并运行恢复程序以填充该私有 WAL 索引。尽管存储位置不同，但最终得到的数据与公用共享内存区域一致。

恢复通过对 WAL 从头到尾进行一次扫描来实现。扫描过程中，会在读取每帧时验证校验和。扫描会在文件末尾或遇到第一个校验和错误时停止。mxFrame 字段被设置为 WAL 中最后一个有效提交帧的索引。由于 WAL 帧编号从 1 开始，mxFrame 同样等于 WAL 中的有效帧数。所谓“提交帧”是指帧头第 4 到 7 字节为非零值的帧。由于恢复程序无法得知此前有多少 WAL 帧可能已被复制回数据库，因此它会将 nBackfill 初始化为零。

在恢复全局共享内存 WAL 索引期间，会在 WAL_WRITE_LOCK、WAL_CKPT_LOCK、WAL_RECOVER_LOCK 以及 WAL_READ_LOCK(1) 到 WAL_READ_LOCK(4) 上持有独占锁。换言之，除 WAL_READ_LOCK(0) 外，与 WAL 索引相关的所有锁都会被独占地持有。这样可以防止其它线程写数据库或读取仍然保留在 WAL 中的事务，直到恢复完成。
