SQLite 第二版的 C 语言接口

SQLite 库的设计非常便于 C 或 C++ 程序使用。本文件概述了 C/C++ 编程接口。

1.0 核心 API

SQLite 库的接口包括三个核心函数、一个不透明数据结构以及一些作为返回值使用的常量。 核心接口如下所示：

typedef struct sqlite sqlite;
#define SQLITE_OK           0   /* Successful result */

sqlite *sqlite_open(const char *dbname, int mode, char **errmsg);

void sqlite_close(sqlite *db);

int sqlite_exec(
  sqlite *db,
  char *sql,
  int (*xCallback)(void*,int,char**,char**),
  void *pArg,
  char **errmsg
);

以上就是在 C 或 C++ 程序中使用 SQLite 所需了解的全部内容。当然还存在其他接口函数（将在下文描述），但我们将从上面显示的核心函数开始介绍。

1.1 打开数据库

使用 sqlite_open 函数可以打开一个已有的 SQLite 数据库，也可以创建一个新的 SQLite 数据库。第一个参数是数据库名称。第二个参数本意是用来指示是以读写还是只读方式使用数据库。但在当前实现中，传入 sqlite_open 的第二个参数会被忽略。 第三个参数是一个指向字符串指针的指针。如果第三个参数不为 NULL 且在尝试打开数据库的过程中发生错误，那么将会申请一段来自 malloc() 的内存并写入错误信息，同时 *errmsg 将指向这段错误信息。调用函数有责任在结束使用后释放这段内存。

SQLite 数据库的名称即是包含该数据库内容的文件名称。如果文件不存在，SQLite 会尝试创建并初始化它。如果该文件是只读的（由于权限设置或位于像 CD-ROM 这样的只读介质上），那么 SQLite 将以只读方式打开该数据库。整个 SQL 数据库存储在磁盘上的单个文件中。但在执行 SQL 命令时，可能会为了存储事务回滚日志或中间查询结果而创建额外的临时文件。

sqlite_open 函数的返回值是一个指向不透明 sqlite 结构的指针。所有后续与该数据库相关的 SQLite 函数调用都应将该指针作为第一个参数。如果因任何原因打开失败，则返回 NULL。

1.2 关闭数据库

要关闭一个 SQLite 数据库，调用 sqlite_close 函数，并传入先前从 sqlite_open 获得的 sqlite 结构指针。 如果在关闭数据库时有一个事务正在进行中，该事务将会被回滚。

1.3 执行 SQL 语句

sqlite_exec 函数用于处理 SQL 语句和查询。此函数需要 5 个参数，如下所示：

  1. 一个指向通过先前调用 sqlite_open 获取的 sqlite 结构的指针。

  2. 一个以零结尾的字符串，包含一个或多个要处理的 SQL 语句和/或查询。

  3. 一个指向回调函数的指针，该函数在查询结果中的每一行上都会被调用一次。该参数可以为 NULL，此时不会调用回调函数。

  4. 一个指针，该指针会被转发为回调函数的第一个参数。

  5. 一个指向错误字符串的指针。错误信息被写入 malloc() 获取的空间，并使该参数指向这段空间。调用函数在使用完之后需释放这段空间。 该参数可以为 NULL，此时不会将错误信息报告回调用函数。

回调函数用于接收查询的结果。回调函数的原型如下：

int Callback(void *pArg, int argc, char **argv, char **columnNames){
  return 0;
}

回调函数的第一个参数只是 sqlite_exec 的第四个参数的副本。该参数可用来将任意信息从客户端代码传入回调函数。 第二个参数是查询结果中的列数。 第三个参数是指向字符串指针数组的指针，对于每一行，其中的每个指针都指向该行对应列的值。注意数据库中的 NULL 值在回调函数中表示为 NULL 指针，这与空字符串完全不同。如果第 i 个参数是空字符串，我们会得到：

argv[i][0] == 0

但如果第 i 个参数是 NULL，则得到：

argv[i] == 0

列名包含在第四个参数的前 argc 项中。 如果 SHOW_DATATYPES PRAGMA 打开（默认关闭），则第四个参数的前 argc 项中的第二部分是对应列的数据类型。

如果 EMPTY_RESULT_CALLBACKS PRAGMA 设置为 ON 且查询结果为空集，那么回调函数会被调用一次，此时第三个参数（argv）为 0。换句话说：

argv == 0

第二个参数（argc）和第四个参数（columnNames）仍然有效，可用于判断若有结果时的列数和列名。 默认行为是在结果集为空时根本不调用回调函数。

回调函数通常应返回 0。如果回调函数返回非零，查询会立即中止，sqlite_exec 会返回 SQLITE_ABORT。

1.4 错误代码

sqlite_exec 函数通常返回 SQLITE_OK。但如果出现错误，可以返回不同的值以表示错误类型。下面是完整的返回代码列表：

#define SQLITE_OK           0   /* Successful result */
#define SQLITE_ERROR        1   /* SQL error or missing database */
#define SQLITE_INTERNAL     2   /* An internal logic error in SQLite */
#define SQLITE_PERM         3   /* Access permission denied */
#define SQLITE_ABORT        4   /* Callback routine requested an abort */
#define SQLITE_BUSY         5   /* The database file is locked */
#define SQLITE_LOCKED       6   /* A table in the database is locked */
#define SQLITE_NOMEM        7   /* A malloc() failed */
#define SQLITE_READONLY     8   /* Attempt to write a readonly database */
#define SQLITE_INTERRUPT    9   /* Operation terminated by sqlite_interrupt() */
#define SQLITE_IOERR       10   /* Some kind of disk I/O error occurred */
#define SQLITE_CORRUPT     11   /* The database disk image is malformed */
#define SQLITE_NOTFOUND    12   /* (Internal Only) Table or record not found */
#define SQLITE_FULL        13   /* Insertion failed because database is full */
#define SQLITE_CANTOPEN    14   /* Unable to open the database file */
#define SQLITE_PROTOCOL    15   /* Database lock protocol error */
#define SQLITE_EMPTY       16   /* (Internal Only) Database table is empty */
#define SQLITE_SCHEMA      17   /* The database schema changed */
#define SQLITE_TOOBIG      18   /* Too much data for one row of a table */
#define SQLITE_CONSTRAINT  19   /* Abort due to constraint violation */
#define SQLITE_MISMATCH    20   /* Data type mismatch */
#define SQLITE_MISUSE      21   /* Library used incorrectly */
#define SQLITE_NOLFS       22   /* Uses OS features not supported on host */
#define SQLITE_AUTH        23   /* Authorization denied */
#define SQLITE_ROW         100  /* sqlite_step() has another row ready */
#define SQLITE_DONE        101  /* sqlite_step() has finished executing */

这些返回值的含义如下：

SQLITE_OK

如果一切正常且没有发生错误，则返回该值。

SQLITE_INTERNAL

该值表示 SQLite 库内部的一致性检查失败。只有当 SQLite 库内存在 bug 时才会发生。如果你在调用 sqlite_exec 时得到 SQLITE_INTERNAL，请务必在 SQLite 邮件列表中报告问题。

SQLITE_ERROR

该返回值表示传入 sqlite_exec 的 SQL 存在错误。

SQLITE_PERM

该返回值表示数据库文件的访问权限不足，因而无法打开该文件。

SQLITE_ABORT

如果回调函数返回非零值，则返回此值。

SQLITE_BUSY

该返回码表示数据库文件正被另一个程序或线程锁定。SQLite 允许多个线程同时读取数据库，但同一时刻只能有一个线程以写模式打开数据库。SQLite 的锁定是针对整个数据库文件的。

SQLITE_LOCKED

该返回码与 SQLITE_BUSY 相似，也表示数据库被锁定。但该锁定来自对 sqlite_exec 的递归调用。只有当你尝试在先前 sqlite_exec 的回调例程中再次调用 sqlite_exec 时才可能出现这种返回码。只要不尝试写入同一个表，就允许对 sqlite_exec 的递归调用。

SQLITE_NOMEM

如果调用 malloc 失败，则返回此值。

SQLITE_READONLY

该返回码表示尝试写入一个只读打开的数据库文件。

SQLITE_INTERRUPT

当调用 sqlite_interrupt 中断了正在进行的数据库操作时，会返回该值。

SQLITE_IOERR

当操作系统告知 SQLite 无法执行某项磁盘 I/O 操作时，会返回该值。通常意味着磁盘空间不足。

SQLITE_CORRUPT

如果 SQLite 检测到正在操作的数据库已损坏，则返回该值。损坏可能由恶意进程写入数据库文件引发，也可能由以前未发现的 SQLite 逻辑错误引起。如果磁盘 I/O 错误造成 SQLite 被迫留下损坏的数据库文件，也会返回该值。后一种情况通常由硬件或操作系统故障引起。

SQLITE_FULL

如果插入失败，因为磁盘空间已满或数据库已经无法容纳更多数据，则返回该值。后一种情况通常仅在数据库大于 2GB 时才会发生。

SQLITE_CANTOPEN

该值表示由于某种原因无法打开数据库文件。

SQLITE_PROTOCOL

如果另一个进程干扰了文件锁定并违反了 SQLite 在回滚日志文件上使用的锁定协议，则返回该值。

SQLITE_SCHEMA

当数据库首次打开时，SQLite 会读取数据库模式到内存，并使用该模式解析新的 SQL 语句。如果另一个进程修改了模式，当前正在执行的命令会因虚拟机代码假设了旧模式而中止。对于这种情况，会返回该错误码。通常重试该命令即可解决问题。

SQLITE_TOOBIG

SQLite 不会在同一表的单行中存储超过约 1MB 的数据。如果你尝试存储超过 1MB 的数据，会返回此值。

SQLITE_CONSTRAINT

如果 SQL 语句违反了数据库约束，则返回该常量。

SQLITE_MISMATCH

当尝试向标记为 INTEGER PRIMARY KEY 的列插入非整数数据时会出现此错误。对于大多数列，SQLite 会忽略数据类型并允许任意类型的数据。但 INTEGER PRIMARY KEY 列只能存储整数数据。

SQLITE_MISUSE

如果 SQLite API 的一个或多个例程使用不当，可能会出现此错误。错误用法的例子包括在调用 sqlite_close 关闭数据库后再调用 sqlite_exec，或在两个独立线程中同时对同一个数据库指针调用 sqlite_exec。

SQLITE_NOLFS

该错误表示你在一个缺乏大文件支持的旧 Unix 机器上试图创建或访问超过 2GB 的数据库文件。

SQLITE_AUTH

该错误表示授权回调拒绝你尝试执行的 SQL。

SQLITE_ROW

这是非回调 API 的 sqlite_step 例程的返回码之一。它表示还有另一行结果数据可用。

SQLITE_DONE

这是非回调 API 的 sqlite_step 例程的返回码之一。它表示 SQL 语句已完全执行完毕，且可以调用 sqlite_finalize。

2.0 在不使用回调函数的情况下访问数据

sqlite_exec 例程曾经是从 SQLite 数据库检索数据的唯一方式。但许多程序员觉得使用回调函数来获取结果非常不便。因此自 SQLite 2.7.7 开始，提供了一个不使用回调函数的第二种访问接口。

新的接口使用三个独立的函数替代单个 sqlite_exec 函数。

typedef struct sqlite_vm sqlite_vm;

int sqlite_compile(
  sqlite *db,              /* The open database */
  const char *zSql,        /* SQL statement to be compiled */
  const char **pzTail,     /* OUT: uncompiled tail of zSql */
  sqlite_vm **ppVm,        /* OUT: the virtual machine to execute zSql */
  char **pzErrmsg          /* OUT: Error message. */
);

int sqlite_step(
  sqlite_vm *pVm,          /* The virtual machine to execute */
  int *pN,                 /* OUT: Number of columns in result */
  const char ***pazValue,  /* OUT: Column data */
  const char ***pazColName /* OUT: Column names and datatypes */
);

int sqlite_finalize(
  sqlite_vm *pVm,          /* The virtual machine to be finalized */
  char **pzErrMsg          /* OUT: Error message */
);

策略是先使用 sqlite_compile 编译一个 SQL 语句，然后多次调用 sqlite_step（每行输出一次），最后调用 sqlite_finalize 清理执行完成后的资源。

2.1 将 SQL 语句编译到虚拟机

sqlite_compile 会“编译”第二个参数指定的单条 SQL 语句，并生成一个可以执行该语句的虚拟机。 与大多数接口例程一样，第一个参数必须是先前通过 sqlite_open 获取的 sqlite 结构指针。

通过第四个参数传入一个指针，用于接收虚拟机的地址。 用于存放虚拟机的空间通过动态分配获取。为避免内存泄漏，调用函数在使用完虚拟机后必须调用 sqlite_finalize。 如果在编译过程中发生错误，可将第四个参数设为 NULL。

如果在编译过程中遇到错误，错误信息会被写入来自 malloc 的内存中，并使第五个参数指向这段空间。如果第五个参数为 NULL，则不生成错误信息。否则调用函数应该在使用完错误信息后通过 sqlite_freemem 释放它。

如果第二个参数实际上包含两个或更多的 SQL 语句，则仅编译第一条。（这与 sqlite_exec 的行为不同，后者会执行输入字符串中的所有 SQL 语句。）sqlite_compile 的第三个参数会被设置为指向输入中第一条语句结束后的第一个字符。如果第二个参数只包含一条 SQL 语句，则第三个参数会指向第二个参数末尾的 '\000' 终止符。

编译成功时，sqlite_compile 返回 SQLITE_OK。否则返回错误码。

2.2 步进执行 SQL 语句

当用 sqlite_compile 生成虚拟机后，可通过一次或多次调用 sqlite_step 来执行该虚拟机。每次调用 sqlite_step（除了最后一次）都会返回一行结果。 结果中的列数存储在第二个参数指向的整数中。 第三个参数指向一个列值指针数组。 第四个参数指向一个包含列名和数据类型的指针数组。 与使用 sqlite_exec 接口的回调例程中的第二至第四个参数相同，sqlite_step 的第二至第四个参数也提供了相同的信息。不过使用 sqlite_step 时，第 4 个参数始终包含列的数据类型信息，无论 SHOW_DATATYPES PRAGMA 是否打开。

每次调用 sqlite_step 都会返回一个整数码，表示该步的执行结果，可能为 SQLITE_BUSY、SQLITE_ROW、SQLITE_DONE、SQLITE_ERROR 或 SQLITE_MISUSE。

如果虚拟机因数据库文件被另一个线程或进程锁定而无法打开文件，sqlite_step 会返回 SQLITE_BUSY。调用函数应稍作其他操作或短暂休眠，给锁腾出时间，然后再次调用 sqlite_step。上述过程可以反复尝试。

每当另一行结果数据可用时，sqlite_step 会返回 SQLITE_ROW。该行数据存储在字符串指针数组中，并使第二个参数指向该数组。

当所有处理完成时，sqlite_step 会返回 SQLITE_DONE 或 SQLITE_ERROR。SQLITE_DONE 表示语句执行成功，SQLITE_ERROR 表示发生运行时错误。（错误详情可从 sqlite_finalize 获取。）在 sqlite_step 返回 SQLITE_DONE 或 SQLITE_ERROR 之后再次调用它是一种滥用。

当 sqlite_step 返回 SQLITE_DONE 或 SQLITE_ERROR 时，*pN 和 *pazColName 会被设置为结果集中的列数及列名（与 SQLITE_ROW 返回时一致）。这使调用代码即使在结果集为空时也能获取列数以及列名和数据类型。*pazValue 参数在返回 SQLITE_DONE 或 SQLITE_ERROR 时始终设为 NULL。如果正在执行的 SQL 是不返回结果的语句（例如 INSERT 或 UPDATE），则 *pN 会设为 0，*pazColName 会设为 NULL。

如果你滥用该库，使 sqlite_step 以不当方式被调用，它会尝试返回 SQLITE_MISUSE。这种情况可能发生在以下几种场景：在多个线程中同时对同一个虚拟机调用 sqlite_step()；在 sqlite_step() 返回 SQLITE_DONE 或 SQLITE_ERROR 后再次调用；或者传入无效的虚拟机指针。你不应依赖 SQLITE_MISUSE 来指示错误，因为接口滥用可能未被检测到而导致程序崩溃。SQLITE_MISUSE 仅用于调试，帮助你在发生严重问题前发现错误用法。其滥用检测逻辑也不能保证在所有情况下都能正常工作。

2.3 删除虚拟机

每个由 sqlite_compile 创建的虚拟机最终都应该交给 sqlite_finalize。sqlite_finalize() 会释放虚拟机所使用的内存和其他资源。不调用 sqlite_finalize() 会导致程序中的资源泄漏。

sqlite_finalize 例程还会返回一个结果码，以表示该虚拟机执行的 SQL 操作是否成功。 sqlite_finalize() 返回的值与若直接通过 sqlite_exec 执行同一 SQL 语句时的返回值相同。其返回的错误信息也一样。

在 sqlite_step 返回 SQLITE_DONE 之前调用 sqlite_finalize 是可以接受的。这会中断正在进行的操作。部分完成的更改会被回滚，数据库会恢复到初始状态（除非通过正在执行的 SQL 中的 ON CONFLICT 子句选择了其他恢复算法）。其效果等同于 sqlite_exec 的回调函数返回非零值的情况。

也可以在虚拟机从未被传入 sqlite_step 的情况下调用 sqlite_finalize。

3.1 最近一次插入的 ROWID

SQLite 表中的每一行都有一个唯一的整数键。如果表包含标记为 INTEGER PRIMARY KEY 的列，则该列作为键。如果没有 INTEGER PRIMARY KEY 列，则键就是一个唯一整数。可以在 SELECT 语句中或在 WHERE/ORDER BY 子句中使用任意名称 "ROWID"、"OID" 或 "_ROWID_" 来访问该键。

当你插入一个没有 INTEGER PRIMARY KEY 的表，或虽然表有 INTEGER PRIMARY KEY 但 INSERT 语句的 VALUES 子句中未指定该列的值时，键会自动生成。可以使用 sqlite_last_insert_rowid API 函数获取最近一次 INSERT 语句生成的键。

3.2 更改的行数

sqlite_changes API 函数返回自数据库上次空闲以来被插入、删除或修改的行数。所谓“空闲”数据库是指没有未完成的 sqlite_exec 调用，也没有尚未由 sqlite_finalize 完成的通过 sqlite_compile 创建的虚拟机。在常见用法中，sqlite_changes 返回最近一次 sqlite_exec 调用或最近一次 sqlite_compile 调用所影响的行数。但如果你嵌套调用 sqlite_exec（即某个 sqlite_exec 的回调例程又调用了另一个 sqlite_exec）或在已有虚拟机存在时调用 sqlite_compile 创建新虚拟机，那么 sqlite_changes 返回的值含义会更复杂。 返回值包含后来被 ROLLBACK 或 ABORT 撤销的更改。但因执行 DROP TABLE 而删除的行不予计入。

SQLite 通过删除表后重新创建该表来实现 "DELETE FROM table"（无 WHERE 子句）的命令。这比逐条删除表中的元素要快得多。但这意味着 sqlite_changes 返回的值始终为零，无论表中原本有多少元素。如果需要准确统计删除的元素数量，请改用 "DELETE FROM table WHERE 1"。

3.3 将查询结果写入 malloc() 获取的内存

sqlite_get_table 函数是对 sqlite_exec 的封装，它收集连续回调中的所有信息，并写入 malloc() 分配的内存。这是一个方便函数，允许应用程序通过一次函数调用获取整个查询结果。

sqlite_get_table 的主要结果是一个指向字符串指针的数组。数组中每个元素对应结果集中的每一列和每一行。NULL 结果以 NULL 指针表示。除了常规数据外，数组开头还有一行包含结果集中每列名称。

举个例子，考虑以下查询：

SELECT employee_name, login, host FROM users WHERE login LIKE 'd%';

该查询将返回所有登录名以字母 "d" 开头的员工的姓名、登录名和主机名。如果将该查询提交给 sqlite_get_table，结果可能如下：

nrow = 2
ncolumn = 3
result[0] = "employee_name"
result[1] = "login"
result[2] = "host"
result[3] = "dummy"
result[4] = "No such user"
result[5] = 0
result[6] = "D. Richard Hipp"
result[7] = "drh"
result[8] = "zadok"

注意 “dummy” 记录的 “host” 值为 NULL，因此 result[] 数组在该位置包含 NULL 指针。

如果查询结果集为空，则默认情况下 sqlite_get_table 会将 nrow 设为 0 并让 result 参数为 NULL。但如果 EMPTY_RESULT_CALLBACKS PRAGMA 为 ON，则 result 参数会初始化为仅包含列名。比如，考虑如下查询，它的结果集为空：

SELECT employee_name, login, host FROM users WHERE employee_name IS NULL;

默认行为会得到以下结果：

nrow = 0
ncolumn = 0
result = 0

但如果 EMPTY_RESULT_CALLBACKS PRAGMA 为 ON，则返回如下内容：

nrow = 0
ncolumn = 3
result[0] = "employee_name"
result[1] = "login"
result[2] = "host"

sqlite_get_table 返回的信息存储在 malloc() 分配的内存中。但调用函数不应该直接释放这段内存。应在不再需要该表时将其完整地传给 sqlite_free_table。 即使传入 NULL（如查询结果为空时）也可以安全地调用 sqlite_free_table。

sqlite_get_table 例程返回的整数结果码与 sqlite_exec 相同。

3.4 中断 SQLite 操作

可以从不同线程或信号处理程序中调用 sqlite_interrupt 函数，从而使当前数据库操作在下一个机会点退出。此时启动数据库操作的 sqlite_exec 例程（或其等价函数）会返回 SQLITE_INTERRUPT。

3.5 测试完整 SQL 语句

SQLite 的下一个接口例程是一个便利函数，用于测试某字符串是否构成一条完整的 SQL 语句。如果 sqlite_complete 函数在其输入为某字符串时返回 true，那么该参数构成一条完整 SQL 语句。虽然不能保证语法正确，但至少我们知道语句完整。如果 sqlite_complete 返回 false，则需要更多文本来完成 SQL 语句。

在 sqlite_complete 的判断中，若 SQL 语句以分号结尾即视为完整。

SQLite 命令行工具使用 sqlite_complete 函数来判断何时需要调用 sqlite_exec。每读取一行输入后，sqlite 会在其缓冲区中调用 sqlite_complete。如果返回 true，则调用 sqlite_exec 并重置输入缓冲区。如果返回 false，则提示符变为续行提示符，再读取一行文字并追加到缓冲区。

3.6 库版本字符串

SQLite 库导出名为 sqlite_version 的字符串常量，包含该库的版本号。头文件中还包含与之对应的宏 SQLITE_VERSION。需要时，程序可以将 SQLITE_VERSION 宏与 sqlite_version 字符串进行比较，以确认头文件与库的版本号一致。

3.7 库字符编码

默认情况下，SQLite 假定所有数据使用固定大小的 8 位字符集（iso8859）。但如果在配置脚本中启用 --enable-utf8 选项，则库会假定使用 UTF-8 可变长度字符。这对 LIKE 和 GLOB 运算符以及 LENGTH() 和 SUBSTR() 函数等会有影响。静态字符串 sqlite_encoding 会被设置为 "UTF-8" 或 "iso8859"，以指示库的编译方式。此外，sqlite.h 头文件会根据情况定义宏 SQLITE_UTF8 或 SQLITE_ISO8859。

请注意，SQLite 使用的字符编码机制无法在运行时更改。这是一个编译时选项。sqlite_encoding 字符串仅用于告知库的编译方式。

3.8 更改库对被锁文件的响应

sqlite_busy_handler 过程可用于在已打开的 SQLite 数据库上注册一个忙等待回调。当 SQLite 试图访问锁定的数据库时，该回调会被调用。回调通常会执行其他有用的工作，或睡眠一段时间，以给锁腾出时间。如果回调返回非零，SQLite 会再次尝试访问数据库，如此循环。如果回调返回零，则 SQLite 会终止当前操作并返回 SQLITE_BUSY。

sqlite_busy_handler 的参数包括从 sqlite_open 返回的不透明结构、一个指向忙等待回调函数的指针，以及一个通用指针，该指针会作为第一个参数传给忙等待回调。当 SQLite 调用回调时，会传入三个参数：传递给 sqlite_busy_handler 的第三个参数、库试图访问的数据库表或索引名称，以及库尝试访问该表或索引的次数。

在常见场景中，我们希望忙等待回调进行睡眠。为此，SQLite 提供了便利例程 sqlite_busy_timeout。其第一个参数是一个打开的 SQLite 数据库指针，第二个参数是毫秒数。 执行 sqlite_busy_timeout 后，SQLite 将在返回 SQLITE_BUSY 之前等待至少指定的毫秒数，以便锁定有足够时间被释放。指定零毫秒会恢复默认行为。

3.9 使用 _printf() 封装函数

以下四个工具函数：

  • sqlite_exec_printf()
  • sqlite_exec_vprintf()
  • sqlite_get_table_printf()
  • sqlite_get_table_vprintf()

实现的查询功能与 sqlite_exec 和 sqlite_get_table 相同。但这四个 _printf 例程并不是将完整 SQL 语句作为第二个参数，而是接受 printf 风格的格式字符串。要执行的 SQL 语句由该格式字符串以及附加的参数共同生成。

使用 SQLite printf 函数而不是 sprintf 有两个优点。首先，使用 SQLite printf 例程不会有溢出静态缓冲区的风险，而 sprintf 则有此风险。SQLite printf 例程会自动分配（并在稍后释放）足够大的内存，以容纳生成的 SQL 语句。

SQLite printf 相对于 sprintf 的第二个优势是新增的两个格式选项，专门用于支持 SQL 中的字符串字面量。在格式字符串中，%q 格式选项类似于 %s，从参数列表中读取一个以 NULL 结尾的字符串并插入结果。但 %q 会对插入字符串中的每个单引号 (') 进行重复，从而转义字符串字面量中的结尾标志。%Q 格式选项的处理方式类似：它除了对单引号进行改写外，还将在结果字符串前后加上单引号。如果 %Q 的参数是 NULL 指针，则生成的字符串为 NULL（不加引号）。

举例来说，假设需要将来自用户输入的字符串值插入数据库表。字符串存储在 zString 变量中。插入代码可能如下所示：

sqlite_exec_printf(db,
  "INSERT INTO table1 VALUES('%s')",
  0, 0, 0, zString);

如果 zString 变量的内容是 "Hello"，则该语句可以正常执行。但如果用户输入了 "Hi y'all!"，生成的 SQL 会是：

INSERT INTO table1 VALUES('Hi y'all')

由于 "y'all" 中的撇号，该语句并非合法 SQL。但如果使用 %q 代替 %s：

sqlite_exec_printf(db,
  "INSERT INTO table1 VALUES('%q')",
  0, 0, 0, zString);

则生成的 SQL 如下：

INSERT INTO table1 VALUES('Hi y''all')

此时撇号已被转义，SQL 语句合法。在从可能包含单引号 (') 的数据中动态生成 SQL 时，建议始终使用 SQLite printf 例程和 %q 格式选项，而不是 sprintf。

如果使用 %Q 替换 %q，如下所示：

sqlite_exec_printf(db,
  "INSERT INTO table1 VALUES(%Q)",
  0, 0, 0, zString);

则生成的 SQL 如下：

INSERT INTO table1 VALUES('Hi y''all')

如果 zString 的值为 NULL，则生成的 SQL 为：

INSERT INTO table1 VALUES(NULL)

上述所有 _printf() 例程都建立在以下两个函数之上：

char *sqlite_mprintf(const char *zFormat, ...);
char *sqlite_vmprintf(const char *zFormat, va_list);

sqlite_mprintf() 的行为类似于标准库的 sprintf()，但它将结果写入 malloc() 分配的内存，并返回指向该内存的指针。sqlite_mprintf() 也支持上述的 %q 和 %Q 扩展。sqlite_vmprintf() 是上述例程的 varargs 版本。这些例程返回的字符串指针应通过 sqlite_freemem() 释放。

3.10 在大型查询期间执行后台任务

sqlite_progress_handler() 例程可用于在 SQLite 数据库上注册一个回调，该回调会在调用 sqlite_exec()、sqlite_step() 及其各类封装函数运行时间较长时定期执行。

该回调会在每执行 N 次虚拟机操作后调用一次，其中 N 由传入 sqlite_progress_handler() 的第二个参数指定。第三个与第四个参数分别是要调用的例程的指针以及作为第一个参数传给它的 void 指针。

每次虚拟机操作所需的时间取决于多种因素。对于 1GHz 的 PC，典型值在每秒 50 万到 300 万次之间，但根据具体查询可能更高或更低。因此很难仅依据虚拟机操作次数来安排后台任务。建议将回调频繁调度（比如每 1000 条指令一次），并配合外部计时器函数决定是否需要运行后台任务。

4.0 添加新的 SQL 函数

自 2.4.0 版本起，SQLite 允许通过 C 代码实现新的函数来扩展 SQL 语言。使用以下接口：

typedef struct sqlite_func sqlite_func;

int sqlite_create_function(
  sqlite *db,
  const char *zName,
  int nArg,
  void (*xFunc)(sqlite_func*,int,const char**),
  void *pUserData
);
int sqlite_create_aggregate(
  sqlite *db,
  const char *zName,
  int nArg,
  void (*xStep)(sqlite_func*,int,const char**),
  void (*xFinalize)(sqlite_func*),
  void *pUserData
);

char *sqlite_set_result_string(sqlite_func*,const char*,int);
void sqlite_set_result_int(sqlite_func*,int);
void sqlite_set_result_double(sqlite_func*,double);
void sqlite_set_result_error(sqlite_func*,const char*,int);

void *sqlite_user_data(sqlite_func*);
void *sqlite_aggregate_context(sqlite_func*, int nBytes);
int sqlite_aggregate_count(sqlite_func*);

sqlite_create_function() 接口用于创建普通函数，而 sqlite_create_aggregate() 用于创建新的聚合函数。在这两种情况下，db 参数是应注册函数的已打开 SQLite 数据库，zName 是新函数名称，nArg 是参数个数，pUserData 是一个指针，该指针会原样传入函数的 C 实现。两者成功时返回 0，若发生错误则返回非零值。

函数名称的长度不得超过 255 个字符。任何尝试创建名称长度超过 255 字符的函数都会导致错误。

对于普通函数，xFunc 回调会在每次函数调用时被执行。xFunc 的实现应调用其中一个 sqlite_set_result_... 接口来返回结果。可以使用 sqlite_user_data() 来获取注册函数时传入的 pUserData 指针。

对于聚合函数，xStep 回调用于处理结果集中每一行，之后会调用 xFinalize 得到最终结果。xStep 例程可使用 sqlite_aggregate_context() 接口申请一段在该函数实例中独有的内存。在调用 xFinalize 后，该内存会自动释放。sqlite_aggregate_count() 可用于获取传入聚合的行数。xFinalize 回调应调用某个 sqlite_set_result_... 接口来设置聚合的最终结果。

现在 SQLite 所有内置函数都使用该接口实现。若需了解如何创建新的 SQL 函数的更多信息和示例，请查看源代码树中 func.c 文件。

5.0 多线程与 SQLite

如果 SQLite 的编译将 THREADSAFE 预处理宏设置为 1，则可以安全地在同一进程的两个或多个线程中同时使用 SQLite。但每个线程应拥有自己的由 sqlite_open 返回的 sqlite* 指针。多个线程同时访问同一个 sqlite* 指针是不安全的。

在本站提供的预编译 SQLite 库中，Unix 版本默认关闭 THREADSAFE，而 Windows 版本默认打开。如果你需要不同设置，则必须重新编译。

在 Unix 系统上，不应将 sqlite* 指针随 fork() 系统调用传递给子进程。子进程应在 fork() 后自行打开数据库副本。
