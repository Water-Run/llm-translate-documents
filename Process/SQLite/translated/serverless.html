SQLite 是无服务器的

1. SQLite 是无服务器的

大多数 SQL 数据库引擎都是作为独立的服务器进程实现的。 想要访问数据库的程序通过某种进程间通信（通常是 TCP/IP）与服务器通信， 向服务器发送请求并接收返回结果。 SQLite 并非如此。 使用 SQLite 时，想要访问数据库的进程直接从磁盘上的数据库文件读取和写入。 没有中间的服务器进程。

无服务器架构有其优点和缺点。 主要优点是没有单独的服务器进程需要安装、设置、配置、初始化、管理和排错。 这也是 SQLite 被称为 "零配置" 数据库引擎的原因之一。 使用 SQLite 的程序在运行前无需管理员设置数据库引擎。 任何能够访问磁盘的程序都可以使用 SQLite 数据库。

另一方面，使用服务器的数据库引擎可以更好地防止客户端应用中的错误 —— 客户端中的异常指针无法破坏服务器上的内存。 由于服务器是单个持久进程，它能够更精确地控制数据库访问，允许更细粒度的锁以及更好的并发性。

大多数 SQL 数据库引擎都是基于客户端/服务器架构。 在那些无服务器的引擎中，据作者所知，只有 SQLite 允许多个应用同时访问同一个数据库。

2. 经典无服务器 与 新型无服务器

（本节添加于 2018-04-02）

最近，人们开始将“无服务器”一词用于与本文档中预设含义略有不同的语境。 以下是“无服务器”的两种可能定义：

  1. 经典无服务器： 数据库引擎与应用程序运行在同一进程、线程和地址空间中。 没有消息传递或网络活动。

  2. 新型无服务器： 数据库引擎运行在与应用程序不同的命名空间中，可能位于不同的机器上，但数据库由托管提供商作为即插即用服务提供， 无需应用拥有者进行管理或维护，并且使用非常简单，使得开发者即使知道其背后实际上使用了服务器，也能认为该数据库是无服务器的。

SQLite 是经典无服务器数据库引擎的一个例子。 对于 SQLite，没有其他进程、线程、机器或其他机制（主机操作系统和文件系统除外）来提供数据库服务或实现支持。 实际上确实不存在服务器。

Microsoft Azure Cosmos DB 和 Amazon S3 是新型无服务器数据库的例子。 这些数据库由在云端独立运行的服务器进程实现。 但服务器由 ISP 维护和管理，而不是由应用开发者管理。 应用开发者只需要使用该服务，无需配置、部署或管理数据库服务器实例，因为这些工作都由服务提供商自动处理。 数据库服务器确实存在，只是对开发者隐藏起来了。

理解这两种“无服务器”的不同定义非常重要。 当某个数据库声称自己是“无服务器”时， 务必要辨别他们所指的是“经典无服务器”还是“新型无服务器”。
