1. 介绍

SQLite 的独特特性之一是数据库由单个磁盘文件构成。 这简化了 SQLite 的使用，因为移动或备份数据库就像复制一个文件一样简单。 这也使得 SQLite 适合作为 应用程序文件格式。 不过，尽管完整的数据库保存在一个磁盘文件中， SQLite 在处理数据库的过程中确实使用了许多临时文件。

本文描述了 SQLite 创建和使用的各种临时文件。 它介绍了这些文件何时创建、何时删除、它们的用途、为什么重要， 以及如何在创建临时文件昂贵的系统上避免它们。

SQLite 使用临时文件的方式并不被视为 SQLite 与应用程序之间的合同的一部分。 本文档中的信息是编写或最后更新本文时 SQLite 的正确行为描述。 但并不能保证将来的 SQLite 版本仍会以相同方式使用临时文件。 可能会使用新的临时文件类型，某些当前的临时文件用途也可能在将来版本中被取消。

2. 九种临时文件

SQLite 当前使用九种不同类型的临时文件：

  1. 回滚日志
  2. 超级日志
  3. 预写式日志 (WAL) 文件
  4. 共享内存文件
  5. 语句日志
  6. TEMP 数据库
  7. 视图和子查询的物化
  8. 临时索引
  9. VACUUM 使用的临时数据库

关于每种临时文件类型的附加信息将在后文介绍。

2.1. 回滚日志

回滚日志是一个临时文件，用于在 SQLite 中实现原子提交和回滚功能。 （关于其工作原理的详细讨论，请参见单独的文档 SQLite 中的原子提交。） 回滚日志总是位于与数据库文件相同的目录，并且文件名与数据库文件相同， 仅在末尾附加 8 个字符 "-journal"。 回滚日志通常在事务启动时创建，并在事务提交或回滚时删除。 回滚日志文件对于实现 SQLite 的原子提交和回滚功能是不可或缺的。 如果没有回滚日志，SQLite 将无法回滚未完成的事务， 如果在事务进行中发生崩溃或断电，整个数据库很可能因缺少回滚日志而损坏。

回滚日志通常在事务起始被创建，在事务结束被销毁。 但该规则存在例外。

如果在事务执行过程中发生崩溃或断电，回滚日志文件将留在磁盘上。 下一次其他应用程序尝试打开该数据库文件时，会发现未处理的回滚日志 （在这种情况下称为“热日志”），并使用日志中的信息 将数据库恢复到未完成事务开始前的状态。 这就是 SQLite 实现原子提交的方式。

如果应用程序通过以下 pragma 将 SQLite 置于 独占锁定模式： 

PRAGMA locking_mode=EXCLUSIVE;

SQLite 会在独占锁定模式会话中第一次事务开始时创建新的回滚日志。 但在事务结束时，它不会删除回滚日志。 回滚日志可能会被截断，或其头部被置零（取决于所使用的 SQLite 版本），但不会被删除。 回滚日志会一直保留，直到退出独占访问模式为止。

回滚日志的创建与删除也会受到 journal_mode pragma 的影响。 默认的日志模式是 DELETE，如上所述，在每次事务结束时删除回滚日志文件。 PERSIST 日志模式放弃删除日志文件，而是将回滚日志头部重写为零， 这防止了其他进程回滚该日志，并在效果上相当于删除日志文件， 但避免了实际从磁盘上移除文件的开销。 换句话说，PERSIST 日志模式表现出与 EXCLUSIVE 锁定模式相同的行为。 OFF 日志模式会完全省略回滚日志。 也就是说，当 journal_mode 设置为 OFF 时，永远不会写入回滚日志。 OFF 日志模式会禁用 SQLite 的原子提交和回滚功能。 当设置为 OFF 日志模式时，ROLLBACK 命令不可用。 如果在使用 OFF 日志模式的事务中发生崩溃或断电，将无法恢复，数据库文件很可能损坏。 MEMORY 日志模式会将回滚日志存储在内存中而不是磁盘上。 当 journal_mode 为 MEMORY 时，ROLLBACK 命令仍然有效， 但由于磁盘上不存在可用于恢复的文件，因此在事务执行过程中发生崩溃或断电时， 数据库很可能损坏。

2.2. 预写式日志 (WAL) 文件

当 SQLite 以 WAL 模式 运行时，预写式日志（WAL）文件取代回滚日志使用。 与回滚日志一样，WAL 文件的目的是实现原子提交和回滚。 WAL 文件总是位于与数据库文件相同的目录，文件名与数据库文件相同， 仅在末尾附加 4 个字符 "-wal"。 WAL 文件在第一次连接数据库时创建，并在最后一个数据库连接关闭时通常删除。 但是，如果最后一个连接未正常关闭，WAL 文件会继续留在文件系统中， 并在下次打开数据库时自动清理。

2.3. 共享内存文件

在 WAL 模式 下，所有与同一数据库文件关联的 SQLite 数据库连接都需要共享一块内存， 该内存用作 WAL 文件的索引。 在大多数实现中，这块共享内存是通过对为此目的创建的文件调用 mmap() 来实现的： 即共享内存文件。 共有内存文件（如果存在）位于与数据库文件相同的目录， 文件名与数据库相同，只是在末尾附加 4 个字符 "-shm"。 共享内存文件仅在 WAL 模式运行期间存在。

共享内存文件不包含持久内容。 共享内存文件的唯一目的是为多个同时访问同一数据库的进程提供一块共享内存。 如果 VFS 能提供替代的共享内存访问方式， 则可能使用该替代方法而不是共享内存文件。 例如，如果 PRAGMA locking_mode 设置为 EXCLUSIVE（意味着只有一个进程可以访问数据库文件）， 那么共享内存将从堆中分配，而不是来自共享内存文件，因此不会创建共享内存文件。

共享内存文件与其关联的 WAL 文件具有相同的生命周期。 在创建 WAL 文件时会创建共享内存文件，在删除 WAL 文件时会删除共享内存文件。 在 WAL 文件恢复过程中，共享内存文件会根据正在恢复的 WAL 文件内容从头重新创建。

2.4. 超级日志文件

超级日志文件在一个事务对多个通过 ATTACH 语句添加到单个 数据库连接 的数据库进行更改时，用作原子提交过程的一部分。 超级日志文件总是位于主数据库文件所在的目录 （主数据库文件是在创建该数据库连接时所调用的最初的 sqlite3_open()、sqlite3_open16() 或 sqlite3_open_v2() 所识别的数据库） 中，并带有随机后缀。 超级日志文件包含在事务期间更改的所有附加辅助数据库的名称。 当超级日志文件被删除时，多数据库事务才算提交。 详见题为 SQLite 中的原子提交 的文档。

没有超级日志，多数据库事务的提交只会在每个数据库内部具有原子性， 但不会在所有数据库之间具有原子性。 换句话说，如果在提交过程中发生崩溃或断电， 那么某一个数据库的更改可能完成，而另一个数据库的更改可能回滚。 超级日志确保所有数据库中的更改要么一起回滚，要么一起提交。

超级日志文件只为涉及多个数据库文件且至少两个数据库满足以下所有条件的 COMMIT 操作创建：

  1. 该数据库在事务中被修改
  2. PRAGMA synchronous 设置不是 OFF
  3. PRAGMA journal_mode 不是 OFF、MEMORY 或 WAL

这意味着当数据库文件将 synchronous 设为关闭，或使用 OFF、MEMORY 或 WAL 日志模式时， SQLite 事务在多数据库文件上的电源损失情况下并不具备原子性。 对于 synchronous OFF 以及 journal_mode OFF 和 MEMORY 的情况， 如果事务提交期间发生断电，数据库通常会损坏。 对于 WAL 模式， 各个数据库文件会在断电情况下单独具有原子更新， 但在多文件事务中，某些文件可能会回滚而其他文件在恢复后回滚向前。

2.5. 语句日志文件

语句日志文件用于回滚大型事务中单个语句的部分结果。 例如，假设一条 UPDATE 语句尝试修改数据库的 100 行。 但在修改前 50 行后，UPDATE 触发了约束冲突，应阻止整个语句执行。 语句日志用于撤销前 50 行的更改，使数据库恢复到语句开始时的状态。

只有在可能修改多个数据库行，且可能在触发器内遇到约束或 RAISE 异常 并因此需要撤销部分结果的 UPDATE 或 INSERT 语句，才会创建语句日志。 如果 UPDATE 或 INSERT 不包含在 BEGIN...COMMIT 中，并且同一数据库连接上没有其他活动语句， 则不会创建语句日志，因为可以使用普通回滚日志。 如果使用了替代的 冲突解决算法，语句日志也会被省略。 例如： 

UPDATE OR FAIL ...
UPDATE OR IGNORE ...
UPDATE OR REPLACE ...
UPDATE OR ROLLBACK ...
INSERT OR FAIL ...
INSERT OR IGNORE ...
INSERT OR REPLACE ...
INSERT OR ROLLBACK ...
REPLACE INTO ....

语句日志被赋予随机名称，不一定与主数据库在同一目录中，并在事务结束时自动删除。 语句日志的大小与引发语句日志的 UPDATE 或 INSERT 语句所实施更改的大小成正比。

2.6. TEMP 数据库

使用 "CREATE TEMP TABLE" 语法创建的表仅对最初评估该语句的 数据库连接 可见。 这些 TEMP 表以及任何相关的索引、触发器和视图， 一起存储在一个单独的临时数据库文件中，该文件在首次遇到 "CREATE TEMP TABLE" 语句时创建。 这个单独的临时数据库文件也有一个关联的回滚日志。 用于存储 TEMP 表的临时数据库文件在使用 sqlite3_close() 关闭数据库连接时自动删除。

TEMP 数据库文件与使用 ATTACH 语句添加的辅助数据库文件非常相似，尽管具有一些特殊属性。 TEMP 数据库在关闭数据库连接时总是自动删除。 TEMP 数据库始终使用 synchronous=OFF 和 journal_mode=PERSIST PRAGMA 设置。 并且，TEMP 数据库不能与 DETACH 一起使用，也不能被其他进程通过 ATTACH 来附加。

只有当应用程序使用 "CREATE TEMP TABLE" 语句时，才会创建与 TEMP 数据库及其回滚日志相关的临时文件。

2.7. 视图和子查询的物化

包含子查询的查询有时必须单独评估子查询并将结果存储在临时表中， 然后使用临时表的内容评估外层查询。 我们称这类操作为对子查询的“物化”。 SQLite 的查询优化器试图避免物化， 但有时难以避免。 通过物化创建的临时表分别存储在它们各自的临时文件中，并在查询结束时自动删除。 这些临时表的大小当然取决于子查询物化后的数据量。

IN 运算符右侧的子查询经常需要物化。 例如： 

SELECT * FROM ex1 WHERE ex1.a IN (SELECT b FROM ex2);

在上述查询中，子查询 "SELECT b FROM ex2" 会被评估， 其结果存储在一个临时表（实际上是临时索引）中， 该表允许通过简单的二分查找来确定某值 ex2.b 是否存在。 一旦该表构建完成，外层查询就开始运行， 对于每个候选结果行检查 ex1.a 是否包含在临时表中。 只有检查为真时才输出该行。

为了避免创建临时表，可以将查询重写如下： 

SELECT * FROM ex1 WHERE EXISTS(SELECT 1 FROM ex2 WHERE ex2.b=ex1.a);

如果存在 ex2.b 上的索引，SQLite 的较新版本（ 3.5.4 版 2007-12-14 及以后） 会自动进行这种重写。

如果 IN 运算符右侧是如下值列表： 

SELECT * FROM ex1 WHERE a IN (1,2,3);

那么它会被视为需要物化的子查询。 换句话说，上述语句等价于： 

SELECT * FROM ex1 WHERE a IN (SELECT 1 UNION ALL
                              SELECT 2 UNION ALL
                              SELECT 3);

当 IN 运算符右侧是值列表时，总是使用临时索引来保存这些值。

当子查询出现在 SELECT 语句的 FROM 子句中时，也可能需要进行物化。例如： 

SELECT * FROM ex1 JOIN (SELECT b FROM ex2) AS t ON t.b=ex1.a;

根据查询的不同，SQLite 可能需要将 "(SELECT b FROM ex2)" 子查询物化到临时表中， 然后在 ex1 与临时表之间执行连接。 查询优化器通过“扁平化”查询来尝试避免这种情况。 在前面的例子中，查询可以被扁平化，SQLite 会自动将查询转换为： 

SELECT ex1.*, ex2.b FROM ex1 JOIN ex2 ON ex2.b=ex1.a;

更复杂的查询可能无法通过查询扁平化来避免创建临时表。 查询是否可以扁平化取决于子查询或外层查询是否包含聚合函数、 ORDER BY 或 GROUP BY 子句、LIMIT 子句等因素。 关于何时可以或不能扁平化查询的规则非常复杂，超出了本文档的讨论范围。

2.8. 临时索引

SQLite 可能会使用临时索引来实现以下 SQL 语言特性：

  • ORDER BY 或 GROUP BY 子句
  • 聚合查询中的 DISTINCT 关键字
  • 通过 UNION、EXCEPT 或 INTERSECT 连接的复合 SELECT 语句

每个临时索引都存储在其单独的临时文件中。 为临时索引创建的临时文件会在使用它的语句结束时自动删除。

SQLite 力图使用预先存在的索引来实现 ORDER BY 子句。 如果已经存在合适的索引，SQLite 会遍历该索引而不是底层表来获取所需数据， 从而使结果按期望顺序输出。 但如果 SQLite 找不到合适的索引， 它会评估查询并将每一行存储到一个临时索引中， 该索引的数据为行数据，其键为 ORDER BY 项。 在查询评估完成后，SQLite 再次遍历临时索引以所需顺序输出行。

SQLite 通过按照 GROUP BY 项建议的顺序排序输出行来实现 GROUP BY。 每一输出行都与前一行比较，以判断是否开启新的“组”。 按照 GROUP BY 项排序的方法与按照 ORDER BY 项排序完全相同。 如果可能，会使用预先存在的索引； 如果没有合适的索引，则会创建一个临时索引。

在聚合查询中使用 DISTINCT 关键字是通过在临时文件中创建临时索引实现的， 并将每个结果行存入该索引。 当计算出新行时，会检查其是否已经存在于临时索引中， 如果存在则丢弃该新结果行。

复合查询中的 UNION 操作是通过创建临时索引并将左右子查询的结果存入其中， 同时去除重复项来实现。 在两个子查询都评估完成后，从头到尾遍历该临时索引以生成最终输出。

复合查询中的 EXCEPT 操作是通过创建一个临时索引、 将左侧子查询结果存入该索引、 再从临时索引中移除右侧子查询的结果， 最后遍历该索引以获得最终输出实现的。

复合查询中的 INTERSECT 操作会创建两个独立的临时索引，分别存于单独的临时文件中。 左右子查询分别评估到各自的临时索引中。 然后并行遍历两个索引，输出同时出现在两个索引中的条目。

注意，复合查询中的 UNION ALL 操作自身并不使用临时索引 （但 UNION ALL 的左右子查询可能根据其构造方式使用临时索引）。

2.9. VACUUM 使用的临时数据库

VACUUM 命令通过创建一个临时文件并将整个数据库重建到该临时文件中来工作。 然后将临时文件的内容复制回原数据库文件，并删除临时文件。

VACUUM 命令创建的临时文件只在命令执行期间存在。 该临时文件的大小不会超过原数据库。

3. SQLITE_TEMP_STORE 编译时参数和 PRAGMA

与事务控制相关的临时文件，即回滚日志、超级日志、预写式日志 (WAL) 文件和共享内存文件，总是写入磁盘。 但其他类型的临时文件可能仅存储在内存中，从未写入磁盘。 是否将回滚、超级和语句日志以外的临时文件写入磁盘取决于 SQLITE_TEMP_STORE 编译时参数、 temp_store pragma 以及临时文件的大小。

SQLITE_TEMP_STORE 编译时参数是一个 #define，其值为 0 到 3 之间的整数。 该参数的含义如下：

  1. 临时文件始终存储在磁盘上，不受 temp_store pragma 设置影响。
  2. 临时文件默认存储在磁盘上，但可由 temp_store pragma 覆盖。
  3. 临时文件默认存储在内存中，但可由 temp_store pragma 覆盖。
  4. 临时文件始终存储在内存中，不受 temp_store pragma 设置影响。

SQLITE_TEMP_STORE 编译时参数的默认值为 1， 意味着默认将临时文件存储在磁盘上，但可通过 temp_store pragma 覆盖该行为。

temp_store pragma 具有一个整数值，也会影响临时文件的存储位置决策。 该 pragma 的值含义如下：

  1. 使用磁盘或内存存储临时文件，取决于 SQLITE_TEMP_STORE 编译时参数。
  2. 如果 SQLITE_TEMP_STORE 指定临时文件应存储在内存中，则覆盖该决策并改为使用磁盘存储。 否则按照 SQLITE_TEMP_STORE 的建议。
  3. 如果 SQLITE_TEMP_STORE 指定临时文件应存储在磁盘上，则覆盖该决策并改为使用内存存储。 否则按照 SQLITE_TEMP_STORE 的建议。

temp_store pragma 的默认设置为 0， 意味着遵循 SQLITE_TEMP_STORE 编译时参数的建议。

重申：SQLITE_TEMP_STORE 编译时参数和 temp_store pragma 仅影响回滚日志和超级日志之外的临时文件。 回滚日志和超级日志会始终写入磁盘，不受这两个设置影响。

4. 其他临时文件优化

SQLite 使用一个页面缓存来缓存近期读写的数据库页面。 该页面缓存不仅用于主数据库文件，也用于存储在临时文件中的临时索引和表。 如果 SQLite 需要使用临时索引或表，且 SQLITE_TEMP_STORE 编译时参数与 temp_store pragma 设置为将临时表和索引存储在磁盘上， 则相关信息仍会先存储在页面缓存中。 在页面缓存满之前，临时文件不会打开，信息也不会真正写入磁盘。

这意味着，在许多常见情况下，当临时表和索引足够小（能够完全置入页面缓存）时， 不会创建临时文件，也不会发生磁盘 I/O。 只有当临时数据过大而无法放入内存时，信息才会溢出至磁盘。

每个临时表和索引都有自己的页面缓存，该缓存最多可存储由 SQLITE_DEFAULT_TEMP_CACHE_SIZE 编译时参数确定数量的数据库页面。 （默认值为 500 页。） 每个临时表和索引的页面缓存的最大页面数相同。 该值无法在运行时或按表/索引单独更改。 每个临时文件都有其私有页面缓存，其页面限额由相应的 SQLITE_DEFAULT_TEMP_CACHE_SIZE 决定。

5. 临时文件存储位置

创建临时文件的目录由特定操作系统的 VFS 决定。

在类 Unix 系统中，会按以下顺序搜索目录：

  1. 由 PRAGMA temp_store_directory 或 sqlite3_temp_directory 全局变量设定的目录
  2. SQLITE_TMPDIR 环境变量指定的目录
  3. TMPDIR 环境变量指定的目录
  4. /var/tmp
  5. /usr/tmp
  6. /tmp
  7. 当前工作目录（"."）
上述目录中第一个已存在且具有写入和执行权限的目录会被使用。 最终的 "." 回退路径对一些在无标准临时文件位置的 chroot 监狱中使用 SQLite 的应用至关重要。

在 Windows 系统上，会按以下顺序搜索文件夹：

  1. 由 PRAGMA temp_store_directory 或 sqlite3_temp_directory 全局变量设定的文件夹
  2. GetTempPath() 系统接口返回的文件夹
在这种情况下，SQLite 本身不会关注环境变量，尽管 GetTempPath() 系统调用可能会。 CYGWIN 构建的搜索算法不同，详情请查阅源代码。
