The SQLITE_DBPAGE Virtual Table

1. 概述

SQLITE_DBPAGE 扩展实现了一个仅同名虚拟表，通过与页管理器交互，提供对底层数据库文件的直接访问。 SQLITE_DBPAGE 能够读取和写入数据库的任何页面。由于交互通过页管理层进行，因此所有更改都是事务性的。

警告：向 SQLITE_DBPAGE 虚拟表写入非常容易导致不可恢复的数据库损坏。不要让不受信任的组件访问 SQLITE_DBPAGE 表。使用 SQLITE_DBPAGE 表时务必小心。实验 SQLITE_DBPAGE 表前请备份重要数据。在设置了SQLITE_DBCONFIG_DEFENSIVE 标志时，写入 SQLITE_DBPAGE 虚拟表会被禁用。

SQLITE_DBPAGE 扩展包含在合并源中，但默认未启用。使用SQLITE_ENABLE_DBPAGE_VTAB 编译时选项来启用 SQLITE_DBPAGE 扩展。SQLITE_DBPAGE 扩展使用未公布的内部接口，无法在运行时加载。唯一在应用程序中添加 SQLITE_DBPAGE 的方式是通过SQLITE_ENABLE_DBPAGE_VTAB 编译时选项将其编译进来。

默认构建的命令行外壳已启用 SQLITE_DBPAGE 扩展，但默认仅可读。使用 CLI 的 --unsafe-testing 命令行选项可启用对 sqlite_dbpage 虚拟表的写入。

2. 用法

SQLITE_DBPAGE 虚拟表是一个可读写表，它按页面提供对底层磁盘文件的直接访问。该虚拟表的模式如下： 

CREATE TABLE sqlite_dbpage(
  pgno INTEGER PRIMARY KEY,
  data BLOB
);

SQLite 数据库文件被划分为多个页面。第一个页面编号为 1，第二个页面为 2，依此类推。没有编号为 0 的页面。所有页面大小相同，每个页面大小为 512 到 65536 之间的 2 的幂。有关更多详细信息，请参见文件格式文档。

SQLITE_DBPAGE 表允许应用查看或替换数据库文件中每一页的原始二进制内容，内容不会被解释。返回的内容以磁盘上的字节顺序逐字节输出。

SQLITE_DBPAGE 表为数据库文件中的每个页面提供一行。SQLITE_DBPAGE 允许使用SELECT 语句读取页面，或使用UPDATE 或INSERT 语句覆盖页面。在对 sqlite_dbpage 执行 INSERT 时，其行为类似REPLACE。换句话说，向已存在的页面 INSERT 会覆盖该页面。要增大数据库大小，可向数据库末尾之外的页面执行 INSERT。

2.1. 在 ATTACH 的数据库上使用 SQLITE_DBPAGE

上面展示的 SQLITE_DBPAGE 表模式并不完整。还存在第三个隐藏列名为 schema，它决定应读取或写入哪一个ATTACH 的数据库。由于 schema 列是隐藏的，它可以在将 SQLITE_DBPAGE 作为表值函数调用时用作参数。

例如，假设使用如下语句向数据库连接附加了附加数据库： 

ATTACH 'auxdata1.db' AS aux1;

然后只需运行以下语句即可读取该数据库文件的第一页： 

SELECT data FROM sqlite_dbpage('aux1') WHERE pgno=1;

如果省略了 schema，则默认使用主数据库（通常名为 'main'，除非通过SQLITE_DBCONFIG_MAINDBNAME 重命名）。因此，以下两个查询通常等价： 

SELECT data FROM sqlite_dbpage('main') WHERE pgno=1;
SELECT data FROM sqlite_dbpage WHERE pgno=1;

SQLITE_DBPAGE 表可以像其它表一样参与连接。因此，若要查看所有连接数据库文件的第一页内容，可以运行如下语句： 

SELECT dbpage.data, dblist.name
  FROM pragma_database_list AS dblist
  JOIN sqlite_dbpage(dblist.name) AS dbpage
 WHERE dbpage.pgno=1;

2.2. 截断数据库

通常 UPDATE 和 INSERT 的数据必须是与数据库页面大小完全相同的 BLOB。然而，作为特例，在提交前的最后一次操作中向页面 2 或更高页面 INSERT NULL，会导致插入的页面及其之后的所有页面被删除，从而截断数据库。例如，执行“INSERT INTO sqlite_dbpage(pgno,data) VALUES(52,NULL)”会将数据库截断为 51 页。页面编号从 1 开始，所以剩余页面总数与最后一个页面编号一致，此例为 51 页。

向 sqlite_dbpage 执行内容为 NULL 的 INSERT 称为“null-INSERT”。null-INSERT 是特例，不一定遵循常规规则。以下是 null-INSERT 的一些具体限制：

  1. sqlite_dbpage 表不允许对页面 1 执行 null-INSERT。因此，无法使用 sqlite_dbpage 将数据库截断为零字节或零页面，因为那样会丢失数据库头，而文件将不再是 SQLite 数据库。

  2. 只有当 null-INSERT 是事务提交前的最后一次写操作时，它才会截断数据库。如果 null-INSERT 发生在多语句事务中，并且在同一事务中随后执行了一个或多个 UPDATE 或 INSERT 操作，那么 null-INSERT 会变成无操作，数据库不会被截断。

  3. 每个事务只能截断一个数据库文件。sqlite_dbpage 虚拟表可以在同一事务中写入多个 ATTACH 的数据库，但由于 null-INSERT 必须是在提交前的最后一次写语句，sqlite_dbpage 只能截断其中一个 ATTACH 的数据库。如果需要截断多个 ATTACH 的数据库，必须在单独的事务中分别进行截断。

  4. 任何ROLLBACK TO都会取消 null-INSERT，即使该 ROLLBACK TO 本身与 null-INSERT 无关也是如此。只有在 null-INSERT 作为单独事务或紧接着在提交前执行时，其特殊行为才有保障。

3. 兼容性

在 SQLite 3.47.0（2024-10-21）版本中新增了向 sqlite_dbpage 执行 INSERT 以及通过 INSERT 增减数据库大小的能力。旧版本不允许对 sqlite_dbpage 执行 INSERT。
