Rowid 表

1.0 定义

“rowid 表”是 SQLite 架构中满足以下条件的任何表

  • 不是 虚拟表，且
  • 不是 WITHOUT ROWID 表。
在典型的 SQLite 数据库架构中，大多数表都是 rowid 表。

rowid 表的特点在于它们都拥有唯一、非 NULL、带符号的 64 位整数 rowid，该值被用作底层 B 树 存储引擎中数据的访问键。

2.0 特性

  • rowid 表的 PRIMARY KEY（如果存在）通常并不是真正的主键，亦即它并不是底层 B 树 存储引擎使用的唯一键。唯一下的例外是 rowid 表声明了 INTEGER PRIMARY KEY。在这种例外情况下，INTEGER PRIMARY KEY 成为 rowid 的别名。

  • rowid 表的真正主键（即用于在底层 B 树 存储引擎中查找行的键）就是 rowid。

  • rowid 表的 PRIMARY KEY 约束（只要它不是真正的主键或 INTEGER PRIMARY KEY）其实就等同于一个 UNIQUE 约束。由于它并不是真正的主键，PRIMARY KEY 的列是允许为 NULL 的，这违背了所有 SQL 标准。

  • 可以通过读写任意一个名为 “rowid”、“oid” 或 “_rowid_” 的列来访问（或更改）rowid 表的 rowid。不过，如果表中声明了使用这些特殊名称的列，那么这些名称就指向声明的列，而不是底层的 rowid。

  • 通过 rowid 访问记录的效率极高，非常快速。

  • 如果 rowid 没有被 INTEGER PRIMARY KEY 别名化，那么它并不具备持久性，可能会发生变化。尤其是 VACUUM 命令会修改那些未声明 INTEGER PRIMARY KEY 的表的 rowid。因此，应用程序通常不应该直接访问 rowid，而应改用 INTEGER PRIMARY KEY。

  • 在底层的 文件格式 中，每个 rowid 都以 可变长度整数 的形式存储。这意味着较小的非负 rowid 值占用的磁盘空间比大的或负的 rowid 值少。

  • 上述所有复杂性（以及此处未提及的其他复杂性）都源自需要为数以百亿计的 SQLite 数据库文件保持向后兼容。在一个完美的世界里，不会存在“rowid”，所有表都会遵循以 WITHOUT ROWID 表形式实现的标准语义，只是省略额外的 “WITHOUT ROWID” 关键字。不幸的是，现实很混乱。SQLite 的设计者对此混乱表示诚挚的歉意。
