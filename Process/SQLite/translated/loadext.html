1. 概述

SQLite 具有在运行时加载扩展的能力（包括新的应用程序定义的 SQL 函数、排序序列、虚拟表以及虚拟文件系统）。此功能允许扩展代码与应用程序分开开发和测试，然后按需加载。

扩展也可以静态链接到应用程序。下面的代码模板在充当静态链接扩展或运行时可加载扩展时同样有效，只不过如果应用程序包含两个或更多扩展，请为入口点函数（"sqlite3_extension_init"）起不同名称，以避免发生命名冲突。

2. 加载扩展

SQLite 扩展是一个共享库或 DLL。要加载它，你需要向 SQLite 提供包含共享库或 DLL 的文件名以及用于初始化扩展的入口点。在 C 代码中，通过sqlite3_load_extension() API 提供这些信息。有关该函数的详细说明，请参阅其文档。

请注意，不同操作系统对共享库使用不同的文件名后缀。Windows 使用 ".dll"，macOS 使用 ".dylib"，大多数除 mac 之外的类 unix 系统使用 ".so"。如果希望代码具有可移植性，可以在共享库文件名中省略后缀，sqlite3_load_extension() 接口会自动添加适当的后缀。

也可以使用一个 SQL 函数来加载扩展：load_extension(X,Y)。它的工作方式与sqlite3_load_extension() C 接口完全相同。

两种加载扩展的方法都允许指定扩展的入口点名称。你也可以留空该参数——在sqlite3_load_extension() C 接口中传入 NULL 指针，或在load_extension() SQL 接口中省略第二个参数——扩展加载逻辑会尝试自行找出入口点。它首先尝试通用扩展名称 "sqlite3_extension_init"。如果无效，它会根据文件名构造入口点，模板为 "sqlite3_X_init"，其中 X 是从最后一个 "/" 之后、下一个 "." 之前的文件名部分中的每个 ASCII 字符转换为小写，并在它们以 "lib" 开头时去掉前三个字符。例如，如果文件名是 "/usr/lib/libmathfunc-4.8.so"，入口点名称就是 "sqlite3_mathfunc_init"。又例如，如果文件名是 "./SpellFixExt.dll"，入口点名就是 "sqlite3_spellfixext_init"。

出于安全考虑，默认情况下扩展加载被关闭。要使用 C 或 SQL 的扩展加载函数，必须先在应用程序中通过sqlite3_db_config(db,SQLITE_DBCONFIG_ENABLE_LOAD_EXTENSION,1,NULL) C 接口启用扩展加载。

在命令行 shell中，可以使用 ".load" 点命令加载扩展。例如：

.load ./YourCode

注意，命令行 shell 程序已经为您启用了扩展加载（通过在启动时调用sqlite3_enable_load_extension() 接口），因此上面的命令可以在无需额外开关、配置或其他操作的情况下运行。

带一个参数的 ".load" 命令会调用 sqlite3_load_extension()，将 zProc 参数设为 NULL，使 SQLite 首先查找名为 "sqlite3_extension_init" 的入口点，然后查找从文件名派生的 "sqlite3_X_init"。如果扩展的入口点使用了不同的名称，只需在第二个参数中提供该名称。例如：

.load ./YourCode nonstandard_entry_point

3. 编译可加载扩展

可加载扩展是 C 代码。要在大多数类 unix 操作系统上编译，通常使用如下命令：

gcc -g -fPIC -shared YourCode.c -o YourCode.so

Mac 也是类 unix，但它不遵循普通的共享库约定。在 Mac 上编译共享库，请使用如下命令：

gcc -g -fPIC -dynamiclib YourCode.c -o YourCode.dylib

如果尝试加载库时出现 "mach-o, but wrong architecture" 错误，那么可能需要根据应用程序的构建方式向 gcc 添加 "-arch i386" 或 "-arch x86_64" 等命令行选项。

要在 Windows 上使用 MSVC 编译，通常可以使用如下命令：

cl YourCode.c -link -dll -out:YourCode.dll

要在 Windows 上使用 MinGW 编译，命令行与 unix 类似，只是输出文件后缀改为 ".dll"，并去掉 -fPIC 参数：

gcc -g -shared YourCode.c -o YourCode.dll

4. 编写可加载扩展

模板可加载扩展包含以下三部分：

  1. 在源码文件最前面使用 "#include <sqlite3ext.h>"，而不要使用 "#include <sqlite3.h>"。

  2. 在紧接着 "#include <sqlite3ext.h>" 的下一行单独写入宏 "SQLITE_EXTENSION_INIT1"。

  3. 添加一个类似如下的扩展加载入口点例程： 

    #ifdef _WIN32
__declspec(dllexport)
#endif
int sqlite3_extension_init( /* <== Change this name, maybe */
  sqlite3 *db, 
  char **pzErrMsg, 
  const sqlite3_api_routines *pApi
){
  int rc = SQLITE_OK;
  SQLITE_EXTENSION_INIT2(pApi);
  /* insert code to initialize your extension here */
  return rc;
}

    你最好根据即将生成的共享库名称来定制入口点名称，而不是使用通用的 "sqlite3_extension_init" 名称。为扩展设置自定义入口点名称，将允许你在以后决定使用静态链接而不是运行时链接时将两个或多个扩展静态链接到同一个程序而不会发生链接器冲突。如果共享库最终名为 "YourCode.so"、"YourCode.dll" 或 "YourCode.dylib"（见上文编译示例），那么正确的入口点名称就是 "sqlite3_yourcode_init"。

以下是一个可以复制粘贴来开始的完整模板扩展：

/* Add your header comment here */
#include <sqlite3ext.h> /* Do not use <sqlite3.h>! */
SQLITE_EXTENSION_INIT1

/* Insert your extension code here */

#ifdef _WIN32
__declspec(dllexport)
#endif
/* TODO: Change the entry point name so that "extension" is replaced by
** text derived from the shared library filename as follows:  Copy every
** ASCII alphabetic character from the filename after the last "/" through
** the next following ".", converting each character to lowercase, and
** discarding the first three characters if they are "lib".
*/
int sqlite3_extension_init(
  sqlite3 *db,
  char **pzErrMsg,
  const sqlite3_api_routines *pApi
){
  int rc = SQLITE_OK;
  SQLITE_EXTENSION_INIT2(pApi);
  /* Insert here calls to
  **     sqlite3_create_function_v2(),
  **     sqlite3_create_collation_v2(),
  **     sqlite3_create_module_v2(), and/or
  **     sqlite3_vfs_register()
  ** to register the new features that your extension adds.
  */
  return rc;
}

4.1. 示例扩展

在 SQLite 源代码树中的ext/misc 子目录中可以找到许多完整且可工作的可加载扩展示例。该目录中的每个文件都是一个独立的扩展。文件顶部的注释提供了文档说明。以下是 ext/misc 子目录中几个扩展的简要说明：

  • carray.c — carray 表值函数 的实现。

  • compress.c — 实现了 应用程序定义的 SQL 函数 compress() 和 uncompress()，对文本或 blob 内容进行 zLib 压缩。

  • rot13.c — 实现了 rot13() SQL 函数。 这是一个非常简单的扩展函数示例，可作为创建新扩展的模板。

  • series.c — 实现了 generate_series 虚拟表 和 表值函数。 这是一个相对简单的虚拟表实现示例，可作为编写新虚拟表的模板。

其他更多复杂的扩展可在 ext/ 目录下除了 ext/misc 之外的子文件夹中找到。

5. 持久化可加载扩展

默认情况下，可加载扩展在最初调用 sqlite3_load_extension() 的数据库连接关闭时会从进程内存中卸载。（换句话说，当数据库连接关闭时，会为所有扩展调用 sqlite3_vfs 对象的 xDlClose 方法。）然而，如果初始化过程返回 SQLITE_OK_LOAD_PERMANENTLY 而不是 SQLITE_OK，那么扩展就不会被卸载（xDlClose 不会被调用），扩展将永久驻留在进程内存中。SQLITE_OK_LOAD_PERMANENTLY 的返回值对想要注册新 VFS 的扩展非常有用。

为澄清：初始化函数返回 SQLITE_OK_LOAD_PERMANENTLY 的扩展在数据库连接关闭后仍然存在于内存中。然而，该扩展不会自动向后续数据库连接注册。这使得加载实现新 VFS 的扩展成为可能。若要持久加载并注册实现新 SQL 函数、排序序列和/或虚拟表的扩展，使添加的功能对所有后续数据库连接可用，则初始化例程还应在一个子函数上调用 sqlite3_auto_extension()，该子函数将负责注册这些服务。

vfsstat.c 扩展展示了一个持久注册新 VFS 和新虚拟表的可加载扩展示例。该扩展中的 sqlite3_vfsstat_init() 初始化例程仅在扩展首次加载时调用一次。它仅在那一次注册新的 "vfslog" VFS，并返回 SQLITE_OK_LOAD_PERMANENTLY，以便实现 "vfslog" VFS 的代码会一直留在内存中。初始化例程还在指向 "vstatRegister()" 函数的指针上调用 sqlite3_auto_extension()，使得所有后续数据库连接在启动时都会调用 "vstatRegister()" 函数，从而注册 "vfsstat" 虚拟表。

6. 静态链接运行时可加载扩展

完全相同的源代码既可以用作运行时的共享库或 DLL，也可用作与应用程序静态链接的模块。这提供了灵活性，并允许你以不同方式重用相同的代码。

要静态链接扩展，只需添加编译时选项 -DSQLITE_CORE。SQLITE_CORE 宏会使 SQLITE_EXTENSION_INIT1 和 SQLITE_EXTENSION_INIT2 宏变为空操作。然后修改应用程序，在直接调用入口点时将第三个 "pApi" 参数设为 NULL。

如果你要静态链接两个或多个扩展，尤其要确保使用基于扩展文件名的入口点名称，而不是通用的 "sqlite3_extension_init"。如果使用通用名称，就会出现多个相同符号的定义，链接会失败。

如果应用程序会打开多个数据库连接，而不是为每个数据库连接分别调用扩展入口点，可以考虑使用 sqlite3_auto_extension() 接口来注册扩展，并在每个数据库连接打开时自动启动它们。你只需在 main() 函数的开头附近注册每个扩展一次。使用 sqlite3_auto_extension() 接口注册扩展后，它们就像内建于 SQLite 核心一样——每当打开新的数据库连接时，它们会自动存在，无需初始化。只需确保在调用 sqlite3_auto_extension() 注册扩展之前，已经使用 sqlite3_config() 完成了所需的所有配置，因为 sqlite3_auto_extension() 接口会隐式调用 sqlite3_initialize()。

7. 实现细节

SQLite 使用 sqlite3_vfs 对象的 xDlOpen(), xDlError(), xDlSym() 和 xDlClose() 方法来实现运行时扩展加载。这些方法在 unix 系统上通过 dlopen() 库实现（这解释了为什么 SQLite 通常需要在 unix 系统上链接 "-ldl" 库），在 Windows 上则使用 LoadLibrary() API。在为特殊系统编写的自定义 VFS 中，可以省略这些方法，此时运行时扩展加载机制将无法工作（尽管假设入口点具有唯一名称，仍可以静态链接扩展代码）。SQLite 可以通过 SQLITE_OMIT_LOAD_EXTENSION 进行编译，以从构建中省略扩展加载代码。
