SQLite 在网络上，
注意事项与考虑

介绍

SQLite 库的用户，特别是应用程序开发者，当他们希望从通过网络连接的不同系统访问 SQLite 数据库时，常常倾向于仅仅通过指定一个引用位于网络文件系统某处数据库文件的文件名来打开数据库连接。（此处称为“远程数据库”）然后通过允许模拟本地文件的操作系统 API 来访问该“文件”。这种幻象看似完美，但在一些重要方面存在缺陷。

这种简单的“远程数据库”方式通常不是从多个系统使用单个 SQLite 数据库的最佳方法，（即便它看起来“能够运行”），因为它常常会导致各种麻烦与困扰。由于这些问题在某些使用中是不可避免的，但又不常见或难以重现，因此应用程序开发者不应仅凭早期测试的成功就断定他们的远程数据库用法会按预期正常工作。

通道数据量

“API 调用”通道携带的信息比“文件 I/O”通道少得多。用于提交查询或指定数据修改的 API 调用通常需要传输的位数远少于为了存储或查找数据而在数据库文件之间来回传送的数据。检索查询结果通常需要远比 API 流量更多的文件流量，因为很少能在不读取未请求的数据的情况下就找到要返回的数据。

通道带宽

API 调用通道以处理器主存速度（每秒十亿字）运行，数据常常通过引用传递（因此不复制）。相比之下，即便是最快的文件 I/O 通道也较慢。它们要求将数据复制出来，通常需要经过一个需要比特序列化的媒介。对于旋转磁介质，传输需要等待盘片旋转和磁头移动，然后受旋转速度的限制。

当文件 I/O 通道包括网络连接（远端除了真正的文件 I/O 之外）时，还会引入额外的延迟。即便原始传输速率并未限制带宽，流量仍需在两端打包与缓冲。额外的 I/O 处理层还会增加调度延迟。然而，在网络文件系统中，传输速度变慢是最不重要的问题。

通道可靠性

“API 调用”通道高度可靠，其错误率未被说明且可忽略不计。只有系统断电时（陨石等例外）该通道才会失败。

当“文件 I/O”通道直接访问本地存储设备时，也同样高度可靠。（旋转存储的 MTBF 超过一百万小时，NVRAM 的寿命更长。）本地设备还具有一个特性，它对数据库管理软件设计以确保ACID 行为至关重要：当所有进程对该设备的写入完成后（即 POSIX fsync() 或 Windows FlushFileBuffers() 调用返回时），文件系统要么已经保存了“已写入”的数据，要么会在保存任何后续写入数据之前完成保存。

当网络文件系统的硬件设备和软件层位于文件系统客户端与实际存储设备之间时，就会引入大量失效与异常行为的来源。虽然网络数据传输具备良好的错误校验机制，但传输的数据包一旦发送并非都能可靠地到达目的地。有些数据包会被其他数据包干扰而必须重发。在数据包干扰的情况下，反复重试可能导致的延迟超过类似数据到达本地存储所需的时间。客户端写入的某些部分可能会与其它部分相对地以错误的时间顺序被存储。

由于网络文件系统写入存在乱序和直接的数据丢失，因此必须能够准确确认一组文件写入已经完成，才能开始下一组写入。要获得这种保证，需要使用设计稳健且实现正确的 fsync()（或等效）操作系统函数。对某些应用来说，网络文件系统的同步操作可能不如本地文件系统同步那般稳健。在网络数据包传输错误的情况下实现稳健的同步非常困难，因此有时为了性能会放宽防护措施。

网络文件系统中的文件锁也存在类似风险。SQLite 在写操作中依赖排他锁，而已知在某些网络文件系统中这些锁无法正确工作。这曾导致数据库损坏。随着这些系统的设计者为满足更普遍的用例而改变实现，这种情况可能会再次发生。

归根结底，网络文件系统的同步与锁定可靠性在不同实现与部署中的表现不尽相同。其所依赖的设计假设在应用测试时可能成立，但在使用时未必。依赖它需自行承担风险（及你的客户的风险）。请参阅如何破坏你的数据库文件。

性能与可靠性问题

从上图与以上讨论可以看出，在两条通道之一插入网络连接会降低性能（即“速度”）。考虑 API 调用通道与文件 I/O 通道之间的相对流量，就会发现这种插入在 API 调用通道上造成的性能影响更小。

对可靠性影响的考虑更容易判断，且结论更明确：在 API 调用通道中插入网络链路也可能导致调用出现失败。但如果客户端应用程序已经正确使用 SQL/SQLite 事务，那么这些失败只会导致事务失败并回滚，而不会损害数据的完整性。相比之下，如果网络链路插入在文件 I/O 通道中，事务虽然也可能失败（如同在 API 调用中那样），但会额外导致远程数据库损坏。

通过在回滚模式下使用 SQLite，可以完全或在可接受程度上缓解这些网络不可靠性问题。然而，SQLite 库并未在跨越网络的场景中测试，也无法合理地进行这样的测试。因此，使用远程数据库是在用户的风险下进行的。

建议

通常来说，如果你的数据与应用之间隔着网络，就应当使用客户端/服务器数据库。这是因为数据库引擎对数据库流量起到了降低带宽的过滤作用。

如果你的数据与应用之间隔着网络，你希望网络上传输的链路是低流量的，而不是高流量的。这意味着数据库引擎需要与数据库本身运行在同一台机器上。像 PostgreSQL 这样的客户端/服务器数据库就是这种情况。SQLite 则不同，数据库引擎与应用运行在同一台机器上，这迫使远程数据库场景中的高流量链路跨越网络，从而通常导致性能下降。

网络文件系统不支持在保持数据库一致性的同时进行读写。因此，如果你有多个位于不同机器上的客户端需要同时读写数据库，可以考虑以下选择：

3. 在回滚模式下使用 SQLite。这意味着你可以拥有多个并发的读取者或一个写入者，但不能同时存在读取者与写入者。

应用程序员应意识到，如果用户可以选择使用远程数据库，他们可能就会这么做。除非已经实现上述某种方案，或已使用一次性、排他的访问，否则程序员应当考虑阻止这种做法，除非可靠性并不重要。

总结

选择对你和你的客户而言合适的技术。如果你的数据与应用位于不同机器，那么你应考虑使用客户端/服务器数据库。SQLite 设计用于数据与应用共存于同一台机器的场景。SQLite 仍可以在许多远程数据库场景中使用，但在此种情况下客户端/服务器方案通常更为理想。
