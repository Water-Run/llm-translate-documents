1. 概述

"swarmvtab" 虚拟表允许用户像查询单个数据库表一样查询大量具有相似模式但行 ID 范围不同的表（以下简称“组件”表）。这些表可以（通常确实）位于不同的数据库中。Swarmvtab 表是只读的。

组件表不得声明为 WITHOUT ROWID，并且必须拥有相同的模式，但在各自数据库中的名称可以不同。在此上下文中，“相同的模式”意味着：

  • 所有组件表必须拥有相同的列集合，顺序一致。
  • 每个列所附加的类型和默认排序序列必须在所有组件表中一致。
  • 所有组件表必须拥有相同的 PRIMARY KEY 声明（如有）。

swarmvtab 表具有与其每个组件表相同的模式。

swarmvtab 虚拟表的创建方式如下： 

CREATE VIRTUAL TABLE temp.<name> USING swarmvtab(<sql-statement>);

Swarmvtab 虚拟表必须在 temp 模式中创建。在主数据库或附加数据库中创建 swarmvtab 会导致错误。

作为 CREATE VIRTUAL TABLE 语句参数提供的 SQL 语句在创建表时执行。它必须返回四列或五列。每行描述一个组件表。前四列按顺序解释为：

  • 数据库 URI。可用于打开包含组件表的数据库的文件名或 URI。
  • 表名。组件表在其数据库中的名称。
  • 最小 rowid。组件表可能包含的最小 rowid 值。
  • 最大 rowid。组件表可能包含的最大 rowid 值。

若存在最后一列，其含义在此处描述。

例如，当执行 SQL 语句返回以下数据时：

数据库 URI表名最小 rowid最大 rowid
test.db1 t1 0 10
test.db2 t2 11 20
test.db3 t1 21 30
test.db4 t1 31 40

如果用户查询 rowid 值为 25 的 swarmvtab 表，该表会打开数据库文件 "test.db3" 并从表 "t1" 中读取数据（因为 25 落在 "test.db3" 中分配给表 "t1" 的 rowid 范围内）。

Swarmvtab 仅高效地处理 rowid（或其他 INTEGER PRIMARY KEY）字段上的范围和等值约束。如果查询不包含此类约束，swarmvtab 会依次打开每个数据库并线性扫描组件表以查找结果。这会生成正确结果，但通常较慢。

SQL 语句返回的行中必须没有行 ID 范围重叠，否则会导致错误。

Swarmvtab 实现可在任何时间打开或关闭数据库。默认情况下，它尝试将同时打开的数据库文件数量限制为九个。这不是硬性限制——可以构造出使 swarmvtab 超出该数量的场景。

2. 编译和使用 Swarmvtab

Swarmvtab 虚拟表的代码位于主 SQLite 源树的 ext/misc/unionvtab.c 文件中。可以使用如下命令将其编译为 SQLite 的可加载扩展： 

gcc -g -fPIC -shared unionvtab.c -o unionvtab.so

或者，unionvtab.c 文件可直接编译入应用程序。在这种情况下，应调用以下函数以在每个新的数据库连接上注册扩展： 

int sqlite3_unionvtab_init(sqlite3 *db, void*, void*);

首个参数应为用于注册扩展的数据库句柄。第二个和第三个参数都应传入 0。

源文件和入口点使用 "unionvtab" 命名，而非 "swarmvtab"。Unionvtab 是一个与 swarmvtab 捆绑的单独记录的虚拟表。

3. 高级用法

大多数 swarmvtab 用户仅会使用上述功能。本节描述为更加特殊的用例设计的功能。这些功能都涉及在 CREATE VIRTUAL TABLE 命令的 SQL 语句之后指定额外的可选参数。可选参数通过其名称、后随 "=" 字符、再后随可选的引用值来指定。名称、"=" 字符和值之间可以有空格。例如： 

CREATE VIRTUAL TABLE temp.sv USING swarmvtab (
  'SELECT ...',                -- SELECT 语句
  maxopen = 20,                -- 一个可选参数
  missing='missing_udf'        -- 另一个可选参数
);

以下章节描述受支持的参数。指定未识别的参数名会导致错误。

3.1. SQL 参数

如果参数名以 ":" 开头，则将其视为在执行 SQL 语句前需要绑定的值。该值始终作为文本绑定。如果指定的 SQL 参数不存在，将导致错误。例如： 

CREATE VIRTUAL TABLE temp.x1 USING swarmvtab (
  "SELECT :dir || local_filename, tbl, min, max FROM components",
  :dir = '/home/user/app/databases/'
);

当执行上述 CREATE VIRTUAL TABLE 语句时，swarmvtab 会在执行 SQL 语句前将文本值 "/home/user/app/databases/" 绑定到 :dir 参数。

一个 CREATE VIRTUAL TABLE 语句中可以包含任意数量的 SQL 参数。

3.2. "maxopen" 参数

默认情况下，swarmvtab 会尝试将同时打开的数据库数量限制为九个。此参数允许更改该限制。例如，要创建一个最多可同时打开 30 个数据库的 swarmvtab 表： 

CREATE VIRTUAL TABLE temp.x1 USING swarmvtab (
  "SELECT ...",
  maxopen=30
);

增加打开数据库的数量可能会在某些场景下提升性能。

3.3. "openclose" 回调

"openclose" 参数允许用户指定一个应用程序定义的 SQL 函数名称，在 swarmvtab 打开数据库之前以及关闭数据库之后调用该函数。传递给 openclose 函数的第一个参数是要打开或刚关闭的数据库的文件名或 URI（与 CREATE VIRTUAL TABLE 命令 SQL 语句首列返回的值一致）。第二个参数在函数在打开数据库之前调用时为 0，在关闭后调用时为 1。例如，如果： 

CREATE VIRTUAL TABLE temp.x1 USING swarmvtab (
  "SELECT ...",
  openclose = 'openclose_udf'
);

那么在打开包含组件表的每个数据库之前，swarmvtab 实际上会执行： 

SELECT openclose_udf(<database-name>, 0);

在数据库关闭之后，swarmvtab 执行相当于： 

SELECT openclose_udf(<database-name>, 1);

openclose 函数返回的任何值都会被忽略。如果在打开数据库前调用时返回错误，则不会打开数据库文件，并将错误返回给用户。这是 swarmvtab 唯一可能在发出 “open” 调用后没有最终发出对应 “close” 调用的场景。如果仍有数据库打开，则可能在最终通过 sqlite3_close() 关闭应用程序数据库（删除包含 swarmvtab 表的 temp 模式）时发出 “close” 调用。

“close” 调用返回的错误始终被忽略。

3.4. "missing" 回调

"missing" 参数允许用户指定一个应用程序定义的 SQL 函数名称，当 swarmvtab 发现所需的数据库文件不存在于磁盘上时，在打开数据库之前调用该函数。这为应用程序提供了在 swarmvtab 尝试打开数据库前从远程源获取所需数据库的机会。传递给 "missing" 函数的唯一参数是标识正在打开数据库的名称或 URI。假设： 

CREATE VIRTUAL TABLE temp.x1 USING swarmvtab (
  "SELECT ...",
  openclose = 'openclose_udf',
  missing='missing_udf'
);

那么将以如下方式调用 missing 函数： 

SELECT missing_udf(<database-name>);

如果 missing 函数返回错误，则不会打开数据库，并将错误返回给用户。如果配置了 openclose 函数，则此时会发出一次 "close" 调用以匹配先前的 "open"。以下伪代码展示了在配置了 missing 和 openclose 函数时，swarmvtab 在打开组件数据库时的处理流程。 

SELECT openclose_udf(<database-name>, 0);
if( error ) return error;
if( db does not exist ){
  SELECT missing_udf(<database-name>);
  if( error ){
    SELECT openclose_udf(<database-name>, 1);
    return error;
  }
}
sqlite3_open_v2(<database-name>);
if( error ){
  SELECT openclose_udf(<database-name>, 1);
  return error;
}
// db successfully opened!

3.5. 组件表 "context" 值

如果作为 CREATE VIRTUAL TABLE 命令一部分指定的 SELECT 语句返回五列，则最后一列仅用于应用上下文。Swarmvtab 并不使用此值，只是在调用 openclose 和 missing 函数时（如已指定）以 <database-name> 之后的参数传入。换言之，如果存在 "context" 列，swarmvtab 会改为调用： 

SELECT missing_udf(<database-name>, <context>);
SELECT openclose_udf(<database-name>, <context>, 0);
SELECT openclose_udf(<database-name>, <context>, 1);

视需要而定。
