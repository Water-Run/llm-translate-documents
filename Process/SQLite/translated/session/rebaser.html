会话模块 C 接口

变更集重写基底

typedef struct sqlite3_rebaser sqlite3_rebaser;

重要提示：此接口为实验性，可能随时更改，恕不另行通知。

假设某站点托管的数据库处于状态 S0。然后对其进行修改，使数据库转为状态 S1，并记录一个变更集（“本地”变更集）。接着，从另一站点接收一个基于 S0 的变更集（“远程”变更集）并将其应用到数据库。数据库此时处于状态 (S1+"remote")，其精确状态取决于应用“远程”时所做的冲突解决决定（OMIT 或 REPLACE）。重写基底本地变更集即是更新它以反映这些冲突解决决定，从而使网络其他地方不必再次解决相同冲突。

例如，如果本地和远程变更集都包含在 "CREATE TABLE t1(a PRIMARY KEY, b)" 上插入相同键的 INSERT：

本地: INSERT INTO t1 VALUES(1, 'v1'); 远程: INSERT INTO t1 VALUES(1, 'v2');

且冲突解决为 REPLACE，则该 INSERT 更改将从本地变更集中删除（它被覆盖）。或者，如果冲突解决为 "OMIT"，则本地变更集改为包含：

UPDATE t1 SET b = 'v2' WHERE a=1;

本地变更集中的更改将按如下方式重写基底：

本地 INSERT
这只可能与远程 INSERT 冲突。如果冲突解决为 OMIT，则在重写基底后的变更集中添加一个 UPDATE 更改。或者，如果冲突解决为 REPLACE，则在重写后的变更集中不添加任何内容。

本地 DELETE
这可能与远程 UPDATE 或 DELETE 冲突。在两种情况下，唯一可能的解决方案都是 OMIT。如果远程操作是 DELETE，则在重写后的变更集中不添加任何更改。如果远程操作是 UPDATE，则 change 的 old.* 字段更新为该 UPDATE 中的 new.* 值。

本地 UPDATE
这可能与远程 UPDATE 或 DELETE 冲突。如果与 DELETE 冲突，且冲突解决为 OMIT，则该 UPDATE 更改会变为 INSERT。UPDATE 更改中的 new.* 记录中任何未定义的值都用冲突 DELETE 的 old.* 值填充。或者，如果冲突解决为 REPLACE，则该 UPDATE 更改会从重写后的变更集中直接省略。

若与远程 UPDATE 冲突且解决方案为 OMIT，则 old.* 值以远程更改中的 new.* 值重写。或者，如果解决方案为 REPLACE，则在重写后的变更集中复制该更改，但移除冲突远程 UPDATE 也已更新的列。如果这意味着不会更新任何列，则该更改被省略。

本地更改可能同时针对多个远程更改进行重写基底。如果单个键被多个远程变更集修改，则在对本地变更集进行重写基底之前，先按以下方式合并它们：

  • 如果对某键已有一次或多次 REPLACE 解决，则按 REPLACE 重写基底该键。

  • 如果某键没有任何 REPLACE 解决，则本地变更集按照最近一次 OMIT 解决重写基底。

注意，来自多个远程变更集的冲突解决是按字段逐个合并的，而不是按行合并。这意味着在多个远程 UPDATE 操作的情况下，单个本地更改的某些字段可能按照 REPLACE 重写，而其他字段则按照 OMIT 重写。

为了重写本地变更集，必须先使用 sqlite3changeset_apply_v2() 将远程变更集应用到本地数据库，并记录重写基底信息的缓冲区。然后：

  1. 通过调用 sqlite3rebaser_create() 创建一个 sqlite3_rebaser 对象。
  2. 通过调用 sqlite3rebaser_configure() 使用从 sqlite3changeset_apply_v2() 获取的重写基底缓冲区配置新对象。如果要针对多个远程变更集重写本地变更集，则 sqlite3rebaser_configure() 应该按照执行多个 sqlite3changeset_apply_v2() 调用的顺序多次调用。
  3. 通过调用 sqlite3rebaser_rebase() 重写每个本地变更集。
  4. 通过调用 sqlite3rebaser_delete() 删除 sqlite3_rebaser 对象。
