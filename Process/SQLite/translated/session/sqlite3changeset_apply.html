将更改集应用到数据库

  sqlite3 *db,                    /* Apply change to "main" db of this handle */
  int nChangeset,                 /* Size of changeset in bytes */
  void *pChangeset,               /* Changeset blob */

    void *pCtx,                   /* Copy of sixth arg to _apply() */
    const char *zTab              /* Table name */
  ),
  int(*xConflict)(
    void *pCtx,                   /* Copy of sixth arg to _apply() */
    int eConflict,                /* DATA, MISSING, CONFLICT, CONSTRAINT */
    sqlite3_changeset_iter *p     /* Handle describing change and conflict */
  ),
  void *pCtx                      /* First argument passed to xConflict */
);

将更改集或补丁集应用到数据库。这些函数尝试使用通过第二个和第三个参数传入的更改集更新与处理器 db 关联的“main”数据库。

这些函数所做的所有更改都封装在保存点事务中。如果发生任何其他错误（除了尝试写入目标数据库时的约束失败），则保存点事务会回滚，将目标数据库恢复到原始状态，并返回一个 SQLite 错误代码。此外，从 3.51.0 版本开始，可以通过sqlite3_errcode() 和 sqlite3_errmsg() API 访问的错误代码和错误消息会保留在数据库句柄中。

传递给这些函数的第四个参数（xFilter）是“筛选回调”。该参数可以为 NULL，这种情况下会将更改集中所有更改应用到数据库。对于 sqlite3changeset_apply() 和 sqlite3_changeset_apply_v2()，如果它不为 NULL，则会对更改集中至少影响一个更改的每个表调用一次。在这种情况下，表名作为第二个参数传入，并且作为第一个参数传入 apply() 或 apply_v2() 第六个参数的上下文指针副本。如果“筛选回调”返回 0，则不会尝试对该表应用任何更改。否则，只要返回值非零，就会尝试应用与该表相关的所有更改。

对于 sqlite3_changeset_apply_v3()，xFilter 回调会对每个更改调用一次。在这种情况下第二个参数是一个 sqlite3_changeset_iter，可以使用常规 API 查询当前更改的详细信息。如果“筛选回调”返回 0，则不会尝试应用当前更改。如果返回非零，则应用该更改。

对于每个未被筛选回调排除的表，此函数会检查目标数据库是否包含兼容表。如果满足以下所有条件，则认为该表是兼容的：

  • 该表的名称与更改集中记录的名称相同，
  • 该表的列数不少于更改集中记录的列数，
  • 该表的主键列与更改集中记录的位置一致。

如果没有兼容表，则这不是错误，但与该表相关的更改都不会应用。会通过 sqlite3_log() 机制发出错误代码 SQLITE_SCHEMA 的警告。每个更改集中的表最多发出一次此类警告。

对于每个存在兼容表的更改，会尝试根据未被筛选回调排除的每个 UPDATE、INSERT 或 DELETE 更改修改表内容。如果某个更改无法干净地应用，则可能会调用作为 sqlite3changeset_apply() 第五个参数传入的冲突处理函数。下面详细描述了每种更改类型在何种情况下调用冲突处理函数。

与 xFilter 参数不同，xConflict 不可为 NULL。将除有效函数指针之外的任何内容作为 xConflict 参数传递的结果是未定义的。

每次调用冲突处理函数时，它必须返回SQLITE_CHANGESET_OMIT、SQLITE_CHANGESET_ABORT 或 SQLITE_CHANGESET_REPLACE 之一。只有当传递给冲突处理函数的第二个参数为 SQLITE_CHANGESET_DATA 或 SQLITE_CHANGESET_CONFLICT 时，才能返回 SQLITE_CHANGESET_REPLACE。如果冲突处理程序返回非法值，则会回滚已经完成的任何更改，并且 sqlite3changeset_apply() 调用返回 SQLITE_MISUSE。sqlite3changeset_apply() 会根据冲突处理函数每次调用返回的值采取不同操作。有关三种可用返回值的详细信息，请参阅文档。

DELETE 更改
对于每个 DELETE 更改，函数会检查目标数据库是否包含与更改集中存储的原始行值具有相同主键值（或值）的行。如果有，并且所有非主键列中的值也与更改集中存储的值匹配，则从目标数据库中删除该行。

如果找到具有匹配主键值的行，但一个或多个非主键字段的值与更改集中存储的原始行值不同，则以 SQLITE_CHANGESET_DATA 作为第二个参数调用冲突处理函数。如果数据库表具有比更改集中记录更多的列，则仅比较这些非主键字段的值与当前数据库内容——任何额外的数据库列都会被忽略。

如果在数据库中未找到匹配主键值的行，则以 SQLITE_CHANGESET_NOTFOUND 作为第二个参数调用冲突处理函数。

如果尝试 DELETE 操作时 SQLite 返回 SQLITE_CONSTRAINT（这只有在违反外键约束时才会发生），则以 SQLITE_CHANGESET_CONSTRAINT 作为第二个参数调用冲突处理函数。这也包括由于先前冲突处理函数返回 SQLITE_CHANGESET_REPLACE 而再次尝试 DELETE 操作的情况。

INSERT 更改
对于每个 INSERT 更改，会尝试将新行插入数据库。如果更改集行的字段少于数据库表，则将尾部字段填充为其默认值。

如果插入行失败，因为数据库中已经存在具有相同主键值的行，则以 SQLITE_CHANGESET_CONFLICT 作为冲突处理函数的第二个参数调用该函数。

如果插入行失败是由于其他约束违规（例如 NOT NULL 或 UNIQUE），则以 SQLITE_CHANGESET_CONSTRAINT 作为第二个参数调用冲突处理函数。这也包括由于先前冲突处理函数返回 SQLITE_CHANGESET_REPLACE 而再次尝试 INSERT 操作的情况。

UPDATE 更改
对于每个 UPDATE 更改，函数会检查目标数据库是否包含与更改集中存储的原始行值具有相同主键值（或值）的行。如果有，并且所有被修改的非主键列中的值也与更改集中存储的值匹配，则在目标数据库中更新该行。

如果找到具有匹配主键值的行，但一个或多个被修改的非主键字段的值与更改集中存储的原始值不同，则以 SQLITE_CHANGESET_DATA 作为第二个参数调用冲突处理函数。由于 UPDATE 更改仅包含要修改的非主键字段的值，因此只有这些字段需要与原始值匹配才能避免触发 SQLITE_CHANGESET_DATA 的回调。

如果在数据库中未找到匹配主键值的行，则以 SQLITE_CHANGESET_NOTFOUND 作为第二个参数调用冲突处理函数。

如果尝试 UPDATE 操作时 SQLite 返回 SQLITE_CONSTRAINT，则以 SQLITE_CHANGESET_CONSTRAINT 作为第二个参数调用冲突处理函数。这也包括由于先前冲突处理函数返回 SQLITE_CHANGESET_REPLACE 而再次尝试 UPDATE 操作的情况。

在 xConflict 回调中执行 SQL 语句是安全的，包括那些写入与回调相关的表的语句。这可用于进一步自定义应用程序的冲突解决策略。

如果输出参数（ppRebase 和 pnRebase）均非 NULL 且输入是更改集（而非补丁集），则 sqlite3changeset_apply_v2() 可能在返回前将 (*ppRebase) 设置为一个可与 sqlite3_rebaser API 一起使用的“重定位”缓冲区。在这种情况下，(*pnRebase) 被设置为缓冲区的字节大小。调用方有责任最终使用 sqlite3_free() 释放这样的缓冲区。仅在应用补丁集时遇到一个或多个冲突时才会分配和填充缓冲区。有关详细信息，请参阅 sqlite3_rebaser API 的相关注释。

通过将支持的标志组合作为第 9 个参数传入，可以修改 sqlite3changeset_apply_v2() 及其流式等价版本的行为。

请注意 sqlite3changeset_apply_v2() API 仍然是实验性的，因此可能发生更改。
