会话模块 C 接口

将表附加到会话对象

int sqlite3session_attach(
  sqlite3_session *pSession,      /* 会话对象 */
  const char *zTab                /* 表名 */
);

如果参数 zTab 不为 NULL，那么它就是要附加到作为第一个参数传递的会话对象的表名。会话对象启用后对该表所做的所有后续更改都会被记录。有关详细信息，请参阅 sqlite3session_changeset() 的文档。

或者，如果参数 zTab 为 NULL，那么数据库中的所有表的更改都会被记录。如果在调用之后向数据库添加了其他表（通过执行“CREATE TABLE”语句），这些新表的更改也会被记录。

只有在 CREATE TABLE 语句中显式定义了 PRIMARY KEY 的表才能记录更改。无论 PRIMARY KEY 是否为“INTEGER PRIMARY KEY”（rowid 别名）都无关紧要。PRIMARY KEY 可以由单个列组成，也可以是复合键。

如果指定的表在数据库中不存在，也不会报错。同样，如果指定的表没有 PRIMARY KEY，也不会报错。然而在这两种情况下都不会记录任何更改。

对其一个或多个 PRIMARY KEY 列中存储了 NULL 值的单独行不会记录更改。

如果调用完成且无错误，则返回 SQLITE_OK。否则，会返回一个 SQLite 错误码（例如 SQLITE_NOMEM）。

sqlite_stat1 的特殊处理

从 SQLite 3.22.0 版本起，“sqlite_stat1”表是上述某些规则的例外。在 SQLite 中，sqlite_stat1 的模式为： 

       CREATE TABLE sqlite_stat1(tbl,idx,stat)

尽管 sqlite_stat1 没有 PRIMARY KEY，仍然会像 PRIMARY KEY 是 (tbl,idx) 一样记录其更改。此外，对于 (idx IS NULL) 为真的行，也会记录更改。但是，对于这样的行，在更改集或补丁集中存储的是零长度 blob（SQL 值 X''），而不是 NULL。这样可以使旧版 sqlite3changeset_invert()、concat() 及类似函数能够处理这些更改集。

sqlite3changeset_apply() 函数在更新 sqlite_stat1 表时会自动将零长度 blob 转换回 NULL。不过，如果应用程序直接在更改集迭代器上调用 sqlite3changeset_new()、sqlite3changeset_old() 或 sqlite3changeset_conflict（包括传递给冲突处理回调的更改集迭代器），则会返回 X'' 值。如果需要，应用程序必须自行将 X'' 翻译为 NULL。

旧版（早于 3.22.0）会话模块无法捕获对 sqlite_stat1 表所做的更改。旧版 sqlite3changeset_apply() 函数会悄无声息地忽略更改集或补丁集中针对 sqlite_stat1 表的任何修改。
