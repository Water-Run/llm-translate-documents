会话模块 C 接口

从会话对象生成变更集

int sqlite3session_changeset(
  sqlite3_session *pSession,      /* Session object */
  int *pnChangeset,               /* OUT: Size of buffer at *ppChangeset */
  void **ppChangeset              /* OUT: Buffer containing changeset */
);

获取包含传入会话对象所附表更改的变更集。如果成功，请在返回 SQLITE_OK 之前将 *ppChangeset 指向包含变更集的缓冲区，并将 *pnChangeset 设置为变更集的字节长度。如果发生错误，将 *ppChangeset 和 *pnChangeset 都设为零，并返回 SQLite 错误代码。

变更集由零个或多个 INSERT、UPDATE 和/或 DELETE 更改组成，每个更改表示附加表中单行的变更。INSERT 更改包含新数据库行每个字段的值。DELETE 更改包含被删除数据库行每个字段的原始值。UPDATE 更改包含被更新数据库行每个字段的原始值，以及每个已更新的非主键列的更新值。UPDATE 更改不能表示修改主键列值的变更。如果发生此类更改，在变更集中它将被表示为 DELETE 后跟 INSERT。

对于在一个或多个主键列中保存了 NULL 值的行，不会记录变更。如果这样的行被插入或删除，在本函数返回的变更集中不会有相应的更改。如果具有一个或多个 NULL 主键列的现有行被更新，使得所有主键列都变为非 NULL，则变更集中只出现 INSERT。同样地，如果原本主键值非 NULL 的现有行被更新，使得一个或多个主键列被设为 NULL，则结果变更集中只包含 DELETE 更改。

可以使用 sqlite3changeset_start() API 创建迭代器遍历变更集。可以使用 sqlite3changeset_apply() API 将变更集应用于具有兼容模式的数据库。

在此函数生成的变更集中，属于同一张表的所有更改会集中在一起。换言之，在遍历变更集或将其应用到数据库时，与同一张表相关的所有更改会在继续处理下一张表之前先处理完。表的顺序与它们附加（或自动附加）到 sqlite3_session 对象的顺序相同。属于同一张表的更改的存储顺序未定义。

在成功调用本函数后，调用者需要最终使用 sqlite3_free() 释放 *ppChangeset 指向的缓冲区。

变更集生成

一旦某张表被附加到会话对象，会话对象会记录插入到该表的新行的主键值。它还会记录被删除或更新行的原始主键及其他列的值。对于每个唯一的主键值，数据只记录一次 —— 即在会话生命周期中首次插入、更新或删除具有该主键的行时记录。

上段描述有一个例外：当某行被插入、更新或删除时，如果其一个或多个主键列包含 NULL 值，则不会记录该变更。

因此，会话对象累积两类记录 —— 仅包含主键值的记录（在用户插入新记录时创建）以及包含主键值和其他列原始值的记录（在用户删除或更新记录时创建）。

调用本函数时，请求的变更集会结合累积记录和数据库文件的当前内容生成。具体而言：

  • 对于每个由插入生成的记录，数据库会查询是否存在具有匹配主键的行。如果找到，则在变更集中添加一个 INSERT 更改。如果未找到，则不添加任何更改。

  • 对于每个由更新或删除生成的记录，数据库会查询是否存在具有匹配主键的行。如果找到这样的行，且一个或多个非主键字段已从原始值发生变化，则在变更集中添加 UPDATE 更改。否则，如果在表中找不到这样的行，则添加 DELETE 更改。如果数据库中存在匹配主键的行，但所有字段仍保留原始值，则不会向变更集添加任何更改。

这意味着，例如在会话对象处于活动状态时插入一行后又删除该行，则变更集中既不会包含插入也不会包含删除。或者，在会话对象活动期间删除一行后又插入具有相同主键值的行，则最终的变更集中会包含一个 UPDATE 更改，而不是 DELETE 和 INSERT。

当会话对象被禁用（参见 sqlite3session_enable() API）时，在插入、更新或删除行时不会积累记录。如果在一个会话内某行被写入多次，可能会出现一些反直觉的现象。例如，如果会话对象处于启用状态时插入一个行，然后在同一会话对象被禁用时又删除该行，则变更集中不会出现 INSERT 记录，即使在禁用期间执行了删除。又例如，如果在启用状态下更新行的一个字段，而在禁用状态下更新同一行的另一字段，则生成的变更集中将包含一个 UPDATE 更改，该更改同时更新两个字段。
