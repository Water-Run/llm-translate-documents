1. 概览

以下命令将 REPLICA 复制为 ORIGIN： 

$ sqlite3_rsync ORIGIN REPLICA ?OPTIONS?

使用 --help 或 -? 标志可以查看完整选项列表。选项标志可以出现在 ORIGIN 和 REPLICA 参数的前面、后面或中间。

添加 -v 选项可以看到更多输出，格式类似于 “rsync”。

2. 功能

  1. ORIGIN 或 REPLICA 其中之一可以采用 “USER@HOST:PATH” 形式，另一个则只是普通路径。此工具会将 REPLICA 变为 ORIGIN 的副本。

    1. 如果 REPLICA 尚不存在，会创建它。

    2. ssh 用于通信，因此 “USER@HOST” 可以是 SSH 别名。

    3. 不要求 ORIGIN 或 REPLICA 中有一方是远程的。如果 ORIGIN 和 REPLICA 都在本地，sqlite3_rsync 也能正常工作。

  2. 在此工具运行期间，两个数据库都可以是“活跃”状态。其他程序可以在运行期间连接两个数据库。其他程序可以写入 ORIGIN，也可以在 sqlite3_rsync 运行时读取 REPLICA。

    REPLICA 会成为 sqlite3_rsync 命令启动时 ORIGIN 快照的副本。如果其他进程在此命令运行期间更改了 ORIGIN 的内容，这些更改会应用到 ORIGIN，但不会转移到 REPLICA。 因此，REPLICA 最终会成为某一时刻 ORIGIN 的一致快照。

  3. 同步使用一种带宽高效的协议，类似于 rsync（其名称即源于此）。

3. 限制

  1. 数据库文件必须都处于 WAL 模式，且页面大小必须相同。 ← 这些限制在版本 3.50.0（2025-05-29）中已移除。

  2. 在 sqlite3_rsync 运行期间，REPLICA 是只读的。在此工具运行期间可以在 REPLICA 上执行查询，但不能执行写事务。

  3. 每次调用此工具只能同步一个数据库。尚未像标准 “rsync” 那样支持使用通配符同步多个不同的数据库。

  4. ORIGIN 或 REPLICA 至少有一个必须在本地机器上。它们不能都位于其他机器上的数据库。

  5. 在远程系统上，此工具必须安装在 SSH 默认 $PATH 中的某个目录。/usr/local/bin 目录通常是一个不错的选择。 或者，可以使用 --exe NAME 标志来指定二进制文件的远程位置，例如 --exe /opt/bin/sqlite3_rsync。

  6. 副本将非常接近原始，但不是完全相同。副本中的所有表（和索引）内容在字节级别与原始完全一致。然而，数据库头部可能会有一些细微变化。特别地，副本与原始相比会有如下不同：

    1. 数据库头部第 24 到 27 字节的 修改计数器 在副本中可能会递增。

    2. 数据库头部第 96 到 99 字节的 版本有效数字 将是生成副本的 sqlite3_rsync 程序所使用的 SQLite 版本号，而不是对原始数据库进行最后写入的程序的版本号。

  7. 在 Windows 上，没有 USER@ 前缀的单字母 HOST 会被解释为 Windows 驱动器字母，而不是主机名。

4. 如何安装

只需将 sqlite3_rsync 可执行文件放到 $PATH 中的某个位置即可安装。如果要与远程系统同步，sqlite3_rsync 可执行文件必须在本地和远程系统上都安装好。安装到远程系统时，确保 SSH 使用的 $PATH 中可以找到它。将 sqlite3_rsync 放在 /usr/local/bin 目录通常是个不错的选择。

不幸的是，在 MacOS 上，ssh 的默认 PATH 是 “/usr/bin:/bin:/usr/sbin:/sbin”，而 MacOS 不允许你在这些目录中添加新程序。作为应对措施，sqlite3_rsync 会尝试按如下方式扩展 PATH：

PATH=$HOME/bin:/usr/local/bin:/opt/homebrew/bin:$PATH

因此，如果你打算与远程 Mac 同步，只要将 sqlite3_rsync 二进制文件安装到这三个新 PATH 位置中的任意一个即可：

  • $HOME/bin
  • /usr/local/bin
  • /opt/homebrew/bin

如果需要在远程机器的其他（非标准）位置安装 sqlite3_rsync，只需在命令行中使用 --exe 选项指定其精确位置。例如：

sqlite3_rsync sample.db mac:sample.db --exe /some/weird/place/sqlite3_rsync

本文作者从未成功让 SSHD 在 Windows 上运行。也许他将来能解决这个问题，并在未来的版本中提供将数据库同步到或从远程 Windows 机器的说明。

4.1. 向后兼容性问题

sqlite3_rsync 程序在启动时会协商同步算法的细节，并使用双方都支持的最先进算法。然而，由于该算法协商逻辑中的一个 bug（缺少 fflush() 调用），如果本地连接使用 3.50.0 或更高版本而远程使用 3.49.1 或更早版本，那么应用可能会挂起。最好的解决办法是在两端都安装最新版本的 sqlite3_rsync。但如果无法做到这一点，可以通过在 sqlite3_rsync 命令行中添加 “--protocol 1” 选项来规避该问题。

5. 网络带宽

该协议的核心思路是，副本将页面或页面组的加密哈希发送到原始端。原始端会返回不同的页面内容，或者如果多页哈希不匹配，则请求更细粒度的哈希。如果原始端和副本初始差异很大，由于交换哈希的开销，整个网络带宽可能超过整个数据库的大小，但额外带宽并不大——最多只有几个百分点。另一方面，如果原始端和副本初始非常相似（通常情况），那么总带宽通常不到数据库大小的 0.01%。在测试中，一个 500MB 的数据库通常只需大约 20KB 的网络流量即可同步。

在 3.50.0（2025-05-29）之前，该协议只会传送单页的哈希，而不会传送页面组的哈希。这意味着即使双方初始完全一致，带宽需求也通常至少是数据库大小的 0.5%。3.50.0 及更高版本在原始端和副本差异较少的情况下更节省带宽。然而，连接双方都必须安装 3.50.0 或更高版本的 sqlite3_rsync，否则协议会回退到旧的、带宽效率较低的算法。

6. 为什么不能直接使用普通的 rsync？

普通的 rsync 无法理解 SQLite 事务。rsync 会将 ORIGIN 复制到 REPLICA，但复制可能并不一致。复制中的某些部分可能来自一个事务，而其他部分可能来自另一个事务。数据库副本可能会损坏。

如果在 rsync 运行的整个过程中没有其他进程连接数据库，并且数据库没有 hot journal，那么 rsync 会生成一个一致的数据库副本。但如果你无法保证这两个条件都满足，那么 rsync 可能会生成一个损坏的副本。而 sqlite3_rsync 则始终生成一致的副本。
