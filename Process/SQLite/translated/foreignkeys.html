概述

本文档描述了SQLite 版本 3.6.19（2009-10-14）中引入的 SQL 外键约束支持。

第一节通过示例介绍了 SQL 外键的概念并定义了文档其余部分使用的术语。第2节描述了应用程序为了在 SQLite 中启用外键约束（默认情况下禁用）必须采取的步骤。第3节描述了用户必须创建以使用外键约束的索引，以及为了让外键约束高效运行应该创建的索引。第4节介绍了 SQLite 支持的高级外键相关特性，第5节描述了 ALTER 和 DROP TABLE 命令如何增强以支持外键约束。最后，第6节列举了当前实现中缺失的特性与限制。

本文档并未完整描述 SQLite 中创建外键约束所用的语法。可在 CREATE TABLE 语句文档中找到该部分内容。

1. 外键约束简介

SQL 外键约束用于强制各表之间的“存在”关系。例如，考虑使用以下 SQL 命令创建的数据库模式： 

CREATE TABLE artist(
  artistid    INTEGER PRIMARY KEY, 
  artistname  TEXT
);
CREATE TABLE track(
  trackid     INTEGER,
  trackname   TEXT, 
  trackartist INTEGER     -- 必须映射到 artist.artistid！
);

使用该数据库的应用程序有理由假设 track 表中的每一行都对应于 artist 表中的某一行。毕竟声明中的注释如是所示。不幸的是，如果用户使用外部工具编辑数据库，或者应用程序存在缺陷，可能会插入 track 表中并不对应任何 artist 表行的记录。或者可能从 artist 表中删除行，导致 track 表中出现不对应任何剩余 artist 行的孤儿数据。这可能导致应用程序随后出现异常，或者至少使得编写应用程序更为困难。

一种解决方案是向数据库模式添加 SQL 外键约束以强制 artist 和 track 表之间的关系。为此，可通过将 track 表的声明修改为如下形式来添加外键定义： 

CREATE TABLE track(
  trackid     INTEGER, 
  trackname   TEXT, 
  trackartist INTEGER,
  FOREIGN KEY(trackartist) REFERENCES artist(artistid)
);

这样，约束由 SQLite 强制执行。试图插入 track 表中不对应任何 artist 表行的记录会失败，同样的，试图删除 artist 表中存在依赖行时也会失败。有一个例外：如果 track 表中的外键列为 NULL，则无需对应的 artist 表条目。用 SQL 表达，即对 track 表中的每一行，以下表达式的结果为真： 

trackartist IS NULL OR EXISTS(SELECT 1 FROM artist WHERE artistid=trackartist)

提示：如果应用需要 artist 与 track 之间更严格的关系，即不允许 trackartist 列出现 NULL，只需在模式中为该列添加相应的 “NOT NULL” 约束。

还有多种其他方式可以在 CREATE TABLE 语句中添加等价的外键声明。有关详细信息请参阅 CREATE TABLE 文档。

下面的 SQLite 命令行演示了为 track 表添加外键约束后的效果： 

sqlite> SELECT * FROM artist;
artistid  artistname       
--------  -----------------
1         Dean Martin      
2         Frank Sinatra    

sqlite> SELECT * FROM track;
trackid  trackname          trackartist
-------  -----------------  -----------
11       That's Amore       1  
12       Christmas Blues    1  
13       My Way             2  

sqlite> -- 由于插入到 trackartist 列的值 (3)
sqlite> -- 与 artist 表中任何行都不对应，该操作失败。
sqlite> INSERT INTO track VALUES(14, 'Mr. Bojangles', 3);
SQL error: foreign key constraint failed

sqlite> -- 此操作成功，因为插入的 trackartist 为 NULL
sqlite> -- 不需要对应的 artist 表行。
sqlite> INSERT INTO track VALUES(14, 'Mr. Bojangles', NULL);

sqlite> -- 尝试在记录插入后修改 trackartist 字段仍然无效，
sqlite> -- 因为新的 trackartist 值 (3)
sqlite> -- 仍与 artist 表中的任何行不匹配。
sqlite> UPDATE track SET trackartist = 3 WHERE trackname = 'Mr. Bojangles';
SQL error: foreign key constraint failed

sqlite> -- 向 artist 表插入所需行。此后可将已插入行更新为trackartist=3
sqlite> --（因为此时在 artist 表中已有对应行）。
sqlite> INSERT INTO artist VALUES(3, 'Sammy Davis Jr.');
sqlite> UPDATE track SET trackartist = 3 WHERE trackname = 'Mr. Bojangles';

sqlite> -- 现在 “Sammy Davis Jr.”（artistid = 3）已加入数据库，
sqlite> -- 可使用该歌手插入新曲目而不违反外键约束：
sqlite> INSERT INTO track VALUES(15, 'Boogie Woogie', 3);

正如预期，通过删除或更新 artist 表中的行也无法将数据库变为违反外键约束的状态： 

sqlite> -- 尝试删除 “Frank Sinatra” 的 artist 记录失败，
sqlite> -- 因为 track 表中存在引用该条目的行。
sqlite> DELETE FROM artist WHERE artistname = 'Frank Sinatra';
SQL error: foreign key constraint failed

sqlite> -- 先删除 track 表中引用 “Frank Sinatra” 的所有记录。
sqlite> -- 此后才能删除该 artist。
sqlite> DELETE FROM track WHERE trackname = 'My Way';
sqlite> DELETE FROM artist WHERE artistname = 'Frank Sinatra';

sqlite> -- 尝试更新 artist 表中某行的 artistid，
sqlite> -- 而同时存在引用该行的 track 表记录。
sqlite> UPDATE artist SET artistid=4 WHERE artistname = 'Dean Martin';
SQL error: foreign key constraint failed

sqlite> -- 删除所有引用 artist 表某行的记录后，
sqlite> -- 可修改该行的 artistid。
sqlite> DELETE FROM track WHERE trackname IN('That''s Amore', 'Christmas Blues');
sqlite> UPDATE artist SET artistid=4 WHERE artistname = 'Dean Martin';

SQLite 使用以下术语：

  • 父表 是外键约束所引用的表。本节示例中的父表是 artist 表。有些书籍和文章将其称为 被引用表，这在一定程度上更准确，但往往会引起混淆。

  • 子表 是应用外键约束的表，即包含 REFERENCES 子句的表。本节示例中的子表是 track 表。其他书籍和文章将其称为 引用表。

  • 父键 是父表中外键约束所引用的列或列集合。它通常（但不总是）是父表的主键。父键必须是父表中具有名称的列或列集合，而不是 rowid。

  • 子键 是子表中受外键约束限制且包含 REFERENCES 子句的列或列集合。

如果子表中的每一行要么有一个或多个子键列为 NULL，要么在父表中存在每一父键列都与其对应子键列值相等的行，则外键约束得以满足。

在上段中，“相等”一词是指使用 此处指定 的规则进行比较后的相等。以下补充说明适用：

  • 比较文本值时，总是使用父键列关联的 排序序列。

  • 比较值时，如果父键列具有 亲和性，则该亲和性会先应用到子键值，再执行比较。

2. 启用外键支持

要在 SQLite 中使用外键约束，必须在编译时未定义 SQLITE_OMIT_FOREIGN_KEY 和 SQLITE_OMIT_TRIGGER。如果定义了 SQLITE_OMIT_TRIGGER 但未定义 SQLITE_OMIT_FOREIGN_KEY，则 SQLite 的行为与 版本 3.6.19（2009-10-14）之前一致——外键定义仍会被解析并可以通过 PRAGMA foreign_key_list 查询，但外键约束不会被强制执行。在此配置下，PRAGMA foreign_keys 命令是无效的。如果定义了 OMIT_FOREIGN_KEY，则外键定义甚至无法被解析（尝试指定外键定义会产生语法错误）。

假设库在编译时启用了外键约束，运行时依然需要应用程序通过 PRAGMA foreign_keys 命令启用它们。例如： 

sqlite> PRAGMA foreign_keys = ON;

外键约束默认被禁用（为了向后兼容），因此必须针对每个 数据库连接 单独启用。 （但请注意，SQLite 的未来版本可能会更改默认行为，使外键约束默认启用。谨慎的开发者不会假设外键默认开启，而是根据需要自行启用或禁用。） 应用程序也可以使用 PRAGMA foreign_keys 语句来确定当前是否启用了外键。以下命令行演示了该操作： 

sqlite> PRAGMA foreign_keys;
0
sqlite> PRAGMA foreign_keys = ON;
sqlite> PRAGMA foreign_keys;
1
sqlite> PRAGMA foreign_keys = OFF;
sqlite> PRAGMA foreign_keys;
0

提示：如果命令 “PRAGMA foreign_keys” 返回的是空结果而不是包含 “0” 或 “1” 的单行，则表示当前使用的 SQLite 版本不支持外键（要么因为版本早于 3.6.19，要么因为编译时定义了 SQLITE_OMIT_FOREIGN_KEY 或 SQLITE_OMIT_TRIGGER）。

无法在 多语句事务（当 SQLite 不处于 自动提交模式）的执行过程中启用或禁用外键约束。尝试这样做不会返回错误；只是没有任何效果。

3. 所需和建议的数据库索引

通常，外键约束的父键是父表的主键。如果它们不是主键，则父键列必须共同受到 UNIQUE 约束或具有 UNIQUE 索引。 如果父键列拥有 UNIQUE 索引，则该索引必须使用 CREATE TABLE 为父表指定的排序序列。 例如： 

CREATE TABLE parent(a PRIMARY KEY, b UNIQUE, c, d, e, f);
CREATE UNIQUE INDEX i1 ON parent(c, d);
CREATE INDEX i2 ON parent(e);
CREATE UNIQUE INDEX i3 ON parent(f COLLATE nocase);

CREATE TABLE child1(f, g REFERENCES parent(a));                        -- 合法
CREATE TABLE child2(h, i REFERENCES parent(b));                        -- 合法
CREATE TABLE child3(j, k, FOREIGN KEY(j, k) REFERENCES parent(c, d));  -- 合法
CREATE TABLE child4(l, m REFERENCES parent(e));                        -- 错误！
CREATE TABLE child5(n, o REFERENCES parent(f));                        -- 错误！
CREATE TABLE child6(p, q, FOREIGN KEY(p, q) REFERENCES parent(b, c));  -- 错误！
CREATE TABLE child7(r REFERENCES parent(c));                           -- 错误！

child1、child2 和 child3 表创建的外键约束都是合法的。child4 表中声明的外键是错误的，因为虽然父键列已建立索引，但该索引不是 UNIQUE。 child5 表的外键是错误的，因为父键列虽然有唯一索引，但该索引使用了不同的排序序列。 child6 和 child7 表不正确，因为它们虽然在父键上都有 UNIQUE 索引，但这些键并不完全匹配某个单独 UNIQUE 索引中的列。

如果数据库模式包含需要查看多个表定义才能发现的外键错误，则在创建这些表时不会检测到它们。相反，这些错误会阻止应用程序准备修改子表或父表内容的 SQL 语句，从而使用外键。内容修改时报出的错误称为 “DML 错误”，而模式更改时报出的错误称为 “DDL 错误”。 换句话说，需要查看子表和父表定义才能发现的错误属于 DML 错误。 外键 DML 错误的英文提示通常是 “foreign key mismatch”，但如果父表不存在，也可能是 “no such table”。 如果满足以下条件之一，就会报告外键 DML 错误：

  • 父表不存在，或者
  • 外键约束中命名的父键列不存在，或者
  • 外键约束中命名的父键列既不是父表的主键，也未受到 CREATE TABLE 中指定排序序列的唯一约束，或者
  • 子表引用父表的主键却未指定主键列，并且父表中主键列的数量与子键列的数量不匹配。

上述最后一条通过以下示例说明： 

CREATE TABLE parent2(a, b, PRIMARY KEY(a,b));

CREATE TABLE child8(x, y, FOREIGN KEY(x,y) REFERENCES parent2);        -- 合法
CREATE TABLE child9(x REFERENCES parent2);                             -- 错误！
CREATE TABLE child10(x,y,z, FOREIGN KEY(x,y,z) REFERENCES parent2);    -- 错误！

相反，如果外键错误仅凭子表定义即可识别，而无需查询父表定义，则子表的 CREATE TABLE 语句会失败。由于错误发生在模式更改期间，这属于 DDL 错误。 无论在创建表时外键约束是否启用，只要出现这种情况就会报告外键 DDL 错误。

子键列并不要求建立索引，但通常极为有益。回到 第1节 的示例，每当应用删除 artist（父表）中的一行，就会执行以下等价的 SELECT 语句，以在 track（子表）中搜索引用该行的记录。 

SELECT rowid FROM track WHERE trackartist = ?

上述语句中的 ? 会被替换为从 artist 表中删除的行的 artistid 列的值（请记住，trackartist 是子键，artistid 是父键）。更一般地： 

SELECT rowid FROM <child-table> WHERE <child-key> = :parent_key_value

如果该 SELECT 返回了任意一行，则 SQLite 会认为从父表删除该行会违反外键约束并返回错误。 如果父键内容被修改或向父表插入新行，也可能执行类似的查询。 如果这些查询无法利用索引，就必须对整个子表执行线性扫描。在复杂数据库中，这可能非常昂贵。

因此，在大多数实际系统中，应为每个外键约束的子键列创建索引。子键索引不需要是（通常也不会是）UNIQUE 索引。 再次回到第1节的例子，可实现外键约束所需的高效数据库模式如下： 

CREATE TABLE artist(
  artistid    INTEGER PRIMARY KEY, 
  artistname  TEXT
);
CREATE TABLE track(
  trackid     INTEGER,
  trackname   TEXT, 
  trackartist INTEGER REFERENCES artist
);
CREATE INDEX trackindex ON track(trackartist);

上面的代码块使用了简写形式来创建外键约束。 在列定义中附加 “REFERENCES <parent-table>” 子句会创建一个外键约束，将该列映射到 <parent-table> 的主键。有关详细信息，请参阅 CREATE TABLE 文档。

4. 高级外键约束特性

4.1. 组合外键约束

组合外键约束指子键与父键都是组合键的情况。例如，考虑以下数据库模式： 

CREATE TABLE album(
  albumartist TEXT,
  albumname TEXT,
  albumcover BINARY,
  PRIMARY KEY(albumartist, albumname)
);

CREATE TABLE song(
  songid     INTEGER,
  songartist TEXT,
  songalbum TEXT,
  songname   TEXT,
  FOREIGN KEY(songartist, songalbum) REFERENCES album(albumartist, albumname)
);

在该系统中，song 表中的每个条目都必须映射到 album 表中具有相同艺术家和专辑组合的条目。

父键和子键必须具有相同的基数。 在 SQLite 中，如果子键列（此处为 songartist 和 songalbum）中的任意一个为 NULL，则无需在父表中存在对应的行。

4.2. 延迟外键约束

SQLite 中的每个外键约束都被归类为立即或延迟。默认情况下外键约束为立即执行。 到目前为止提供的所有示例都是立即外键约束。

如果某个语句修改了数据库内容，使得在该语句结束时立即外键约束被违反，则会抛出异常并撤销该语句的影响。相比之下，如果某条语句修改数据库内容，使得延迟外键约束被违反，则违反不会立即报告。延迟外键约束会在事务尝试 提交 时进行检查。 只要用户保持开启事务，数据库可以处于违反任意数量延迟外键约束的状态。 然而，只要仍存在外键约束违规，提交 就会失败。

如果当前语句未位于显式事务中（即没有 BEGIN/COMMIT/ROLLBACK 块），则在语句执行完毕后会自动提交一个隐式事务。在这种情况下，延迟约束行为与立即约束相同。

要将外键约束标记为延迟，必须在其声明中包含以下子句： 

DEFERRABLE INITIALLY DEFERRED                -- 延迟外键约束

指定外键约束的完整语法可在 CREATE TABLE 文档中找到。将上述短语替换为以下任一项即可创建立即外键约束。 

NOT DEFERRABLE INITIALLY DEFERRED            -- 立即外键约束
NOT DEFERRABLE INITIALLY IMMEDIATE           -- 立即外键约束
NOT DEFERRABLE                               -- 立即外键约束
DEFERRABLE INITIALLY IMMEDIATE               -- 立即外键约束
DEFERRABLE                                   -- 立即外键约束

defer_foreign_keys PRAGMA 可用于暂时将所有外键约束置为延迟状态，而不管它们如何声明。

下例展示了使用延迟外键约束的效果： 

-- 数据库模式。两个表开始时均为空。
CREATE TABLE artist(
  artistid    INTEGER PRIMARY KEY, 
  artistname  TEXT
);
CREATE TABLE track(
  trackid     INTEGER,
  trackname   TEXT, 
  trackartist INTEGER REFERENCES artist(artistid) DEFERRABLE INITIALLY DEFERRED
);

sqlite3> -- 如果外键约束为立即执行，此插入会报错，
sqlite3> -- 因为表 artist 中不存在 artistid=5 的行。
sqlite3> -- 但由于约束是延迟的，并且存在未提交事务，
sqlite3> -- 因此不会报错。
sqlite3> BEGIN;
sqlite3>   INSERT INTO track VALUES(1, 'White Christmas', 5);

sqlite3> -- 以下 COMMIT 会失败，因为数据库当前状态未满足
sqlite3> -- 延迟外键约束。事务仍保持打开。
sqlite3> COMMIT;
SQL error: foreign key constraint failed

sqlite3> -- 先向 artist 表插入 artistid=5 的行后，
sqlite3> -- 延迟外键约束得以满足，可安全提交事务。
sqlite3>   INSERT INTO artist VALUES(5, 'Bing Crosby');
sqlite3> COMMIT;

当数据库处于不满足延迟外键约束的状态时，可以 RELEASE 一个 嵌套保存点 事务。另一方面，事务保存点（在没有打开事务时创建的非嵌套保存点）受到与 COMMIT 相同的限制——尝试在此类状态下 RELEASE 保存点会失败。

如果 COMMIT 语句（或事务 SAVEPOINT 的 RELEASE）因数据库当前状态违反延迟外键约束而失败，且存在 嵌套保存点，则该嵌套保存点仍然保持打开状态。

4.3. ON DELETE 和 ON UPDATE 操作

外键的 ON DELETE 和 ON UPDATE 子句用于配置在删除父表行（ON DELETE）或修改现有行的父键值（ON UPDATE）时发生的动作。单个外键约束可以为 ON DELETE 和 ON UPDATE 配置不同的动作。外键动作在许多方面类似触发器。

SQLite 中每个外键对应的 ON DELETE 和 ON UPDATE 动作之一为 “NO ACTION”、“RESTRICT”、“SET NULL”、“SET DEFAULT” 或 “CASCADE”。如果未显式指定动作，则默认是 “NO ACTION”。

  • NO ACTION：配置 “NO ACTION” 即意味着：当父键被修改或从数据库中删除时，不会执行任何特殊操作。

  • RESTRICT：“RESTRICT” 动作意味着当存在一个或多个子键映射到某父键时，应用程序被禁止删除（ON DELETE RESTRICT）或修改（ON UPDATE RESTRICT）该父键。与普通外键约束不同之处在于，RESTRICT 动作在字段更新时立即处理——不是像立即约束那样在当前语句末尾，或像延迟约束那样在当前事务末尾。 即便所附的外键约束为延迟，配置 RESTRICT 动作仍会在删除或修改具有依赖子键的父键时立即返回错误。

  • SET NULL：如果配置为 “SET NULL”，则在父键被删除（ON DELETE SET NULL）或修改（ON UPDATE SET NULL）时，所有映射到该父键的子表行的子键列会被设置为 SQL NULL。

  • SET DEFAULT：“SET DEFAULT” 动作类似于 “SET NULL”， 不同之处在于会将每个子键列设置为该列的默认值而非 NULL。有关默认值如何分配给表列的详细信息，请参阅 CREATE TABLE 文档。

  • CASCADE：“CAS CADE” 动作会将父键上的删除或更新操作传播到所有依赖的子键。对于 “ON DELETE CASCADE” 动作，这意味着与删除的父行关联的子表中每一行也会被删除。对于 “ON UPDATE CASCADE” 动作，这意味着每个依赖子键中的值都会被修改为匹配新的父键值。

例如，在第1节的示例中向外键添加一个 “ON UPDATE CASCADE” 子句，使用户可以在不破坏参照完整性的情况下更新外键约束的父键（artistid）列： 

-- 数据库模式
CREATE TABLE artist(
  artistid    INTEGER PRIMARY KEY, 
  artistname  TEXT
);
CREATE TABLE track(
  trackid     INTEGER,
  trackname   TEXT, 
  trackartist INTEGER REFERENCES artist(artistid) ON UPDATE CASCADE
);

sqlite> SELECT * FROM artist;
artistid  artistname       
--------  -----------------
1         Dean Martin      
2         Frank Sinatra    

sqlite> SELECT * FROM track;
trackid  trackname          trackartist
-------  -----------------  -----------
11       That's Amore       1
12       Christmas Blues    1
13       My Way             2  

sqlite> -- 更新 “Dean Martin” 的 artistid 列。
sqlite> -- 正常情况下这会触发约束，因为会孤立 track 表中
sqlite> -- 的两条依赖记录。但在外键定义中附加的
sqlite> -- ON UPDATE CASCADE 子句会让更新“级联”至子表，
sqlite> -- 避免外键约束违规。
sqlite> UPDATE artist SET artistid = 100 WHERE artistname = 'Dean Martin';

sqlite> SELECT * FROM artist;
artistid  artistname       
--------  -----------------
2         Frank Sinatra    
100       Dean Martin      

sqlite> SELECT * FROM track;
trackid  trackname          trackartist
-------  -----------------  -----------
11       That's Amore       100
12       Christmas Blues    100  
13       My Way             2

配置 ON UPDATE 或 ON DELETE 动作并不意味着可以不满足外键约束。例如，如果配置了 “ON DELETE SET DEFAULT” 动作，但子键列的默认值在父表中没有对应行，则在存在依赖子键的情况下删除父键仍会导致外键违规。如下例所示： 

-- 数据库模式
CREATE TABLE artist(
  artistid    INTEGER PRIMARY KEY, 
  artistname  TEXT
);
CREATE TABLE track(
  trackid     INTEGER,
  trackname   TEXT, 
  trackartist INTEGER DEFAULT 0 REFERENCES artist(artistid) ON DELETE SET DEFAULT
);

sqlite> SELECT * FROM artist;
artistid  artistname       
--------  -----------------
3         Sammy Davis Jr.

sqlite> SELECT * FROM track;
trackid  trackname          trackartist
-------  -----------------  -----------
14       Mr. Bojangles      3

sqlite> -- 删除父表中的行会使依赖行的子键值被设置为整数 0。
sqlite> -- 但该值在父表中没有对应的行，因此
sqlite> -- 外键约束被违反并抛出异常。
sqlite> DELETE FROM artist WHERE artistname = 'Sammy Davis Jr.';
SQL error: foreign key constraint failed

sqlite> -- 此时值 0 对应于父表中的某行，因此
sqlite> -- DELETE 语句不会违反外键约束，
sqlite> -- 且不会抛出异常。
sqlite> INSERT INTO artist VALUES(0, 'Unknown Artist');
sqlite> DELETE FROM artist WHERE artistname = 'Sammy Davis Jr.';

sqlite> SELECT * FROM artist;
artistid  artistname       
--------  -----------------
0         Unknown Artist

sqlite> SELECT * FROM track;
trackid  trackname          trackartist
-------  -----------------  -----------
14       Mr. Bojangles      0

熟悉 SQLite 触发器 的读者会发现，上述示例中的 “ON DELETE SET DEFAULT” 动作与以下 AFTER DELETE 触发器在效果上是类似的： 

CREATE TRIGGER on_delete_set_default AFTER DELETE ON artist BEGIN
  UPDATE child SET trackartist = 0 WHERE trackartist = old.artistid;
END;

每当外键约束的父表中的一行被删除，或父键列的值被修改时，事件的逻辑顺序为：

  1. 执行适用的 BEFORE 触发器程序，
  2. 检查本地（非外键）约束，
  3. 更新或删除父表中的该行，
  4. 执行所需的外键动作，
  5. 执行适用的 AFTER 触发器程序。

ON UPDATE 外键动作与 SQL 触发器之间存在一个重要差异。仅当父键值被修改为与旧值不相等的新值时，才会执行 ON UPDATE 动作。例如： 

-- 数据库模式
CREATE TABLE parent(x PRIMARY KEY);
CREATE TABLE child(y REFERENCES parent ON UPDATE SET NULL);

sqlite> SELECT * FROM parent;
x
----
key

sqlite> SELECT * FROM child;
y
----
key

sqlite> -- 由于以下 UPDATE 并未实际更改父键值，
sqlite> -- 因此 ON UPDATE 动作不会触发，子键也不会被置为 NULL。
sqlite> UPDATE parent SET x = 'key';
sqlite> SELECT IFNULL(y, 'null') FROM child;
y
----
key

sqlite> -- 这次 UPDATE 修改了父键值，
sqlite> -- ON UPDATE 动作被执行，子键被设置为 NULL。
sqlite> UPDATE parent SET x = 'key2';
sqlite> SELECT IFNULL(y, 'null') FROM child;
y
----
null

5. CREATE、ALTER 与 DROP TABLE 命令

本节介绍 CREATE TABLE、ALTER TABLE 和 DROP TABLE 命令如何与 SQLite 的外键交互。

无论是否启用 外键约束，CREATE TABLE 命令的行为相同。外键约束的父键定义在创建表时不会被检查。用户可以在父表不存在、父键列不存在或未被 PRIMARY KEY 或 UNIQUE 约束共同约束的情况下创建指向它们的外键定义。

在启用外键约束时，ALTER TABLE 命令在两个方面表现不同：

  • 无法使用 “ALTER TABLE ... ADD COLUMN” 语法添加包含 REFERENCES 子句的列，除非新列的默认值为 NULL。尝试这样做会返回错误。

  • 如果使用 “ALTER TABLE ... RENAME TO” 命令重命名作为一个或多个外键约束父表的表，则外键约束定义会被修改以引用新的父表名称。存储在 sqlite_schema 表 中的子 CREATE TABLE 语句文本也会被更新以反映新的父表名称。

如果在准备 DROP TABLE 命令时外键约束已启用，则 DROP TABLE 会执行一个隐式的 DELETE，以在删除表之前移除表中所有行。隐式 DELETE 不会触发任何 SQL 触发器，但可能触发外键动作或约束违规。如果立即外键约束被违反，则 DROP TABLE 语句失败且表不会被删除。如果延迟外键约束被违反，则在用户尝试提交事务且违规仍然存在时会报告错误。隐式 DELETE 报告的 “foreign key mismatch” 错误会被忽略。

对 ALTER TABLE 和 DROP TABLE 命令所做的这些增强旨在确保它们在外键约束启用期间无法用来创建包含外键违规的数据库。但这里只有一个例外。如果父键没有在父表定义中受到 PRIMARY KEY 或 UNIQUE 约束的约束，而是通过使用 CREATE INDEX 命令创建的索引受唯一约束控制，则子表可能在不产生 “foreign key mismatch” 错误的情况下被填充。如果随后从数据库模式中删除该 UNIQUE 索引，再删除父表时将不会报告错误。然而数据库可能会因此处于一种状态，即外键约束的子表包含不对应任何父表行的记录。若要避免这种情况，应确保数据库模式中所有父键都由父表定义中的 PRIMARY KEY 或 UNIQUE 约束约束，而非外部的 UNIQUE 索引。

上述 DROP TABLE 和 ALTER TABLE 命令属性仅在外键启用时适用。如果用户认为这些行为不合适，则可以在执行 DROP 或 ALTER TABLE 命令之前使用 PRAGMA foreign_keys 禁用外键约束。当然，在外键约束被禁用时，用户可以不受限制地违反外键约束，从而创建内部不一致的数据库。

6. 限制与不支持的功能

本节列出了一些在其他地方未提及的限制和省略功能。

  1. 不支持 MATCH 子句。 根据 SQL92，可以在组合外键定义中附加 MATCH 子句，以修改子键中 NULL 值的处理方式。如果指定 “MATCH SIMPLE”，则当一个或多个子键值为 NULL 时，子键无需对应父表中的任何行。如果指定 “MATCH FULL”，则只要任一子键值为 NULL，即无需对应父表行，但所有子键值必须为 NULL。最后，如果外键约束声明为 “MATCH PARTIAL” 且有一个子键值为 NULL，则必须至少存在一行父表行，使得非 NULL 的子键值与父键值匹配。

    SQLite 解析 MATCH 子句（即如果指定 MATCH 不会报语法错误），但不会强制执行。SQLite 中所有外键约束都按 “MATCH SIMPLE” 处理。

  2. 不支持在延迟模式与立即模式之间切换约束。 许多系统允许用户在运行时切换单个外键约束的延迟或立即模式（例如使用 Oracle 的 “SET CONSTRAINT” 命令）。SQLite 不支持该操作。在 SQLite 中，外键约束在创建时就永久标记为延迟或立即。

  3. 外键动作的递归深度限制。 SQLITE_MAX_TRIGGER_DEPTH 与 SQLITE_LIMIT_TRIGGER_DEPTH 设置决定触发器程序递归的最大深度。就这些限制而言，外键动作 被视为触发器程序。PRAGMA recursive_triggers 设置不会影响外键动作的行为。无法禁止外键动作的递归。

  4. 外键不得跨越模式边界。 也就是说，在 REFERENCES (X.Y) 中，表 X 仅在包含该 REFERENCES 子句的模式中解析。
