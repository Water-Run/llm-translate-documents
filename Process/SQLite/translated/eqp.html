1. EXPLAIN QUERY PLAN 命令

警告：EXPLAIN QUERY PLAN 命令返回的数据仅供交互式调试使用。输出格式可能会在 SQLite 的不同版本之间发生变化。应用程序不应依赖 EXPLAIN QUERY PLAN 命令的输出格式。

提示：如上所述，EXPLAIN QUERY PLAN 输出格式在 3.24.0 版本（2018-06-04）时发生了重大变化。版本 3.36.0（2021-06-18）中又发生了一些次要变化。后续版本中仍有可能继续更改。

EXPLAIN QUERY PLAN SQL 命令用于获取 SQLite 用以执行特定 SQL 查询的策略或计划的高层描述。最重要的是，EXPLAIN QUERY PLAN 会报告查询如何使用数据库索引。本文档旨在帮助理解与解释 EXPLAIN QUERY PLAN 的输出。背景信息可在以下文档中单独查阅：

  • 一份关于 SQLite 工作原理 的基础介绍。
  • 关于 查询优化器 的说明。
  • 索引 的工作机制说明。
  • 下一代查询规划器。

查询计划以树形结构表示。以原始形式（由 sqlite3_step() 返回）来看，树的每个节点包含四个字段：一个整数节点 ID、一个整数父节点 ID、一个目前未使用的辅助整数字段，以及对该节点的描述。因此整棵树就是包含四列和零行或多行的表。命令行 shell 通常会截取该表，并以 ASCII 图的形式呈现出来，便于查看。要禁止 shell 自动绘图并以表格格式显示 EXPLAIN QUERY PLAN 的输出，请执行 ".explain off" 命令将 “EXPLAIN 格式化模式” 设为关闭。要恢复自动绘图，请执行 ".explain auto"。可通过 ".show" 命令查看当前的 “EXPLAIN 格式化模式” 设置。

也可以使用 ".eqp on" 命令将 CLI 设置为自动 EXPLAIN QUERY PLAN 模式： 

sqlite> .eqp on

在自动 EXPLAIN QUERY PLAN 模式下，shell 会为你输入的每个语句自动运行一条单独的 EXPLAIN QUERY PLAN 查询，并在真正执行查询之前显示结果。使用 ".eqp off" 命令可关闭该模式。

EXPLAIN QUERY PLAN 最有价值的对象是 SELECT 语句，但也可能与读取数据库表数据的其他语句一起出现（例如 UPDATE、DELETE、INSERT INTO ... SELECT）。

1.1. 表扫描与索引扫描

在处理 SELECT（或其他）语句时，SQLite 可能通过多种方式从数据库表中检索数据。它可能扫描表中的所有记录（全表扫描）、基于 rowid 索引扫描表中一个连续的记录子集、扫描某个数据库 索引 的连续条目，或在一次扫描中组合上述多种策略。SQLite 可能从表或索引检索数据的各种方式在 此处 有详细描述。

对于查询读取的每个表，EXPLAIN QUERY PLAN 的输出都会记录 detail 列值以 "SCAN" 或 "SEARCH" 开头的一条记录。"SCAN" 用于全表扫描，包括 SQLite 按索引定义的顺序遍历表中所有记录的情况。"SEARCH" 表示只访问表的某部分行。每条 SCAN 或 SEARCH 记录都包含以下信息：

  • 读取数据的表、视图或子查询的名称。
  • 是否使用了索引或 自动索引。
  • 是否适用了 覆盖索引 优化。
  • WHERE 子句中用于索引的条件项。

例如，以下 EXPLAIN QUERY PLAN 命令针对的 SELECT 语句通过对表 t1 执行全表扫描： 

sqlite> EXPLAIN QUERY PLAN SELECT a, b FROM t1 WHERE a=1;
QUERY PLAN
`--SCAN t1

如果查询能够使用索引，那么 SCAN/SEARCH 记录会包含索引名称，以及对于 SEARCH 记录指出如何定位要访问的行子集。例如： 

sqlite> CREATE INDEX i1 ON t1(a);
sqlite> EXPLAIN QUERY PLAN SELECT a, b FROM t1 WHERE a=1;
QUERY PLAN
`--SEARCH t1 USING INDEX i1 (a=?)

在前述例子中，SQLite 使用索引 "i1" 优化形式为 (a=?) 的 WHERE 子句条件，此处具体为 "a=1"。前一个例子无法使用 覆盖索引，但下例可以，且结果体现出这一点： 

sqlite> CREATE INDEX i2 ON t1(a, b);
sqlite> EXPLAIN QUERY PLAN SELECT a, b FROM t1 WHERE a=1; 
QUERY PLAN
`--SEARCH t1 USING COVERING INDEX i2 (a=?)

SQLite 中的所有连接都通过 嵌套扫描 实现。当对包含连接的 SELECT 查询使用 EXPLAIN QUERY PLAN 分析时，每个嵌套循环都会输出一条 SCAN 或 SEARCH 记录。例如： 

sqlite> EXPLAIN QUERY PLAN SELECT t1.*, t2.* FROM t1, t2 WHERE t1.a=1 AND t1.b>2;
QUERY PLAN
|--SEARCH t1 USING INDEX i2 (a=? AND b>?)
`--SCAN t2

条目的顺序表示嵌套顺序。在本例中，使用索引 i2 扫描表 t1 的那一项是外层循环（因为它排在前面），而对表 t2 的全表扫描是内层循环（因为它排在最后）。在下例中，SELECT 语句的 FROM 子句中 t1 与 t2 的位置对调，查询策略保持不变。EXPLAIN QUERY PLAN 的输出反映了查询实际上是如何执行的，而不是 SQL 语句中如何书写。 

sqlite> EXPLAIN QUERY PLAN SELECT t1.*, t2.* FROM t2, t1 WHERE t1.a=1 AND t1.b>2;
QUERY PLAN
|--SEARCH t1 USING INDEX i2 (a=? AND b>?)
`--SCAN t2

如果 WHERE 子句包含 OR 表达式，SQLite 可能会采用 “按并集的 OR” 策略（也称为 OR 优化）。此时会有一个顶层的搜索记录，并包含两个子记录，每个索引一个： 

sqlite> CREATE INDEX i3 ON t1(b);
sqlite> EXPLAIN QUERY PLAN SELECT * FROM t1 WHERE a=1 OR b=2;
QUERY PLAN
`--MULTI-INDEX OR
   |--SEARCH t1 USING COVERING INDEX i2 (a=?)
   `--SEARCH t1 USING INDEX i3 (b=?)

1.2. 临时排序B树

如果 SELECT 查询包含 ORDER BY、GROUP BY 或 DISTINCT 子句，SQLite 可能需要使用临时 B 树结构对输出行排序。或者，它也可能 使用索引。使用索引几乎总是比执行一次排序更加高效。如果必须使用临时 B 树，则会在 EXPLAIN QUERY PLAN 输出中添加一条记录，其 "detail" 字段的值为 "USE TEMP B-TREE FOR xxx"，其中 xxx 为 "ORDER BY"、"GROUP BY" 或 "DISTINCT" 之一。例如： 

sqlite> EXPLAIN QUERY PLAN SELECT c, d FROM t2 ORDER BY c;
QUERY PLAN
|--SCAN t2
`--USE TEMP B-TREE FOR ORDER BY

在此情况下，可通过在 t2(c) 上创建索引来避免使用临时 B 树，如下所示： 

sqlite> CREATE INDEX i4 ON t2(c);
sqlite> EXPLAIN QUERY PLAN SELECT c, d FROM t2 ORDER BY c; 
QUERY PLAN
`--SCAN t2 USING INDEX i4

1.3. 子查询

在上述所有示例中，都只有一个 SELECT 语句。如果查询包含子 SELECT，则这些子查询会作为外层 SELECT 的子节点显示出来。例如： 

sqlite> EXPLAIN QUERY PLAN SELECT (SELECT b FROM t1 WHERE a=0), (SELECT a FROM t1 WHERE b=t2.c) FROM t2;
|--SCAN TABLE t2 USING COVERING INDEX i4
|--SCALAR SUBQUERY
|  `--SEARCH t1 USING COVERING INDEX i2 (a=?)
`--CORRELATED SCALAR SUBQUERY
   `--SEARCH t1 USING INDEX i3 (b=?)

上例包含两个 “SCALAR” 子查询。之所以称其为 SCALAR，是因为它们返回单个数值——一个一行一列的表。如果实际查询返回超过这个结果，则只会使用第一行第一列的值。

上述第一个子查询相对于外层查询是常量。第一个子查询的值只需计算一次，然后可在外层 SELECT 的每一行中重复使用。第二个子查询则是 “相关的”，其值会根据外层查询当前行的内容而变化。因此该子查询必须为外层 SELECT 的每一行重复执行一次。

除非应用了 展开优化，若子查询出现在 SELECT 的 FROM 子句中，SQLite 可以选择运行子查询并将结果存入临时表，或者将子查询作为协程运行。以下示例即为后者：子查询由协程运行。外层查询在需要从子查询获取新行时会阻塞，控制权转移到协程，协程生成所需输出行，随后控制权再转回主例程继续处理。 

sqlite> EXPLAIN QUERY PLAN SELECT count(*)
      > FROM (SELECT max(b) AS x FROM t1 GROUP BY a) AS qqq
      > GROUP BY x;
QUERY PLAN
|--CO-ROUTINE qqq
|  `--SCAN t1 USING COVERING INDEX i2
|--SCAN qqqq
`--USE TEMP B-TREE FOR GROUP BY

如果在 SELECT 的 FROM 子句中对子查询应用了 展开优化，则相当于将子查询合并进外层查询。EXPLAIN QUERY PLAN 的输出会反映这一点，如下例所示： 

sqlite> EXPLAIN QUERY PLAN SELECT * FROM (SELECT * FROM t2 WHERE c=1) AS t3, t1;
QUERY PLAN
|--SEARCH t2 USING INDEX i4 (c=?)
`--SCAN t1

如果某个子查询的内容可能需要被访问多次，则使用协程并不理想，因为那样协程必须重复计算数据。如果该子查询无法展开，则必须将其显式计算出来并写入瞬时表。 

sqlite> SELECT * FROM
      >   (SELECT * FROM t1 WHERE a=1 ORDER BY b LIMIT 2) AS x,
      >   (SELECT * FROM t2 WHERE c=1 ORDER BY d LIMIT 2) AS y;
QUERY PLAN
|--MATERIALIZE x
|  `--SEARCH t1 USING COVERING INDEX i2 (a=?)
|--MATERIALIZE y
|  |--SEARCH t2 USING INDEX i4 (c=?)
|  `--USE TEMP B-TREE FOR ORDER BY
|--SCAN x
`--SCAN y

1.4. 复合查询

每个 复合查询（UNION、UNION ALL、EXCEPT 或 INTERSECT）中的组成查询会分别计算，并在 EXPLAIN QUERY PLAN 输出中占据独立一行。 

sqlite> EXPLAIN QUERY PLAN SELECT a FROM t1 UNION SELECT c FROM t2;
QUERY PLAN
`--COMPOUND QUERY
   |--LEFT-MOST SUBQUERY
   |  `--SCAN t1 USING COVERING INDEX i1
   `--UNION USING TEMP B-TREE
      `--SCAN t2 USING COVERING INDEX i4

上述输出中的 “USING TEMP B-TREE” 表示用于实现两个子 SELECT 结果并集的临时 B 树结构。另一种实现复合查询的方法是将每个子查询作为协程运行，使它们的输出呈排序状态，然后将结果合并。当查询规划器采用这种方式时，EXPLAIN QUERY PLAN 的输出如下所示： 

sqlite> EXPLAIN QUERY PLAN SELECT a FROM t1 EXCEPT SELECT d FROM t2 ORDER BY 1;
QUERY PLAN
`--MERGE (EXCEPT)
   |--LEFT
   |  `--SCAN t1 USING COVERING INDEX i1
   `--RIGHT
      |--SCAN t2
      `--USE TEMP B-TREE FOR ORDER BY
