1. 介绍

默认情况下，SQLite 中的每一行都有一个特殊列，通常称为“rowid”，该列在表内唯一标识该行。然而，如果在 CREATE TABLE 语句末尾添加 “WITHOUT ROWID” 片语，则会省略这个特殊的 “rowid” 列。省略 rowid 有时可以带来空间和性能优势。

WITHOUT ROWID 表是使用聚集索引作为主键的表。

1.1. 语法

要创建 WITHOUT ROWID 表，只需在 CREATE TABLE 语句末尾添加关键字 “WITHOUT ROWID”。例如：

CREATE TABLE IF NOT EXISTS wordcount(
  word TEXT PRIMARY KEY,
  cnt INTEGER
) WITHOUT ROWID;

与所有 SQL 语法一样，关键字的大小写无关。可以写成 “WITHOUT rowid” 或 “without rowid” 或 “WiThOuT rOwId”，它们代表相同含义。

每个 WITHOUT ROWID 表必须有一个 PRIMARY KEY。如果带有 WITHOUT ROWID 子句的 CREATE TABLE 语句缺少 PRIMARY KEY，就会引发错误。

在大多数情况下，普通表的特殊 “rowid” 列也可以称为 “oid” 或 “_rowid_”。不过，在 CREATE TABLE 语句中只有 “rowid” 可以作为关键字。

1.2. 兼容性

使用 WITHOUT ROWID 表需要 SQLite 3.8.2 版（2013-12-06）或更高版本。如果用较旧版本的 SQLite 打开包含一个或多个 WITHOUT ROWID 表的数据库，就会出现 “malformed database schema” 错误。

1.3. 特性

据我们所知，WITHOUT ROWID 只存在于 SQLite 中，无法与其他任何 SQL 数据库引擎兼容。一个优雅的系统中，所有表即使没有 WITHOUT ROWID 关键字也都应表现为 WITHOUT ROWID 表。然而在 SQLite 最初设计时，仅使用整数 rowid 作为行键，以简化实现。这个方法在多年里运行良好。但随着 SQLite 的应用需求增长，更需要让表的 PRIMARY KEY 真正对应底层行键。为了满足这一需求同时不破坏当时（约 2013 年）已存在的数十亿 SQLite 数据库的向后兼容性，引入了 WITHOUT ROWID 概念。

2. 与普通 rowid 表的差异

WITHOUT ROWID 语法是一种优化，并没有提供新的功能。使用 WITHOUT ROWID 表可以做到的事情，也可以在普通 rowid 表中使用完全相同的方法和语法完成。WITHOUT ROWID 表的唯一优势是，在某些情况下它可以比普通 rowid 表节省磁盘空间和/或稍微快一些。

在大多数情况下，普通 rowid 表和 WITHOUT ROWID 表是可以互换的。但 WITHOUT ROWID 表有一些附加限制，而普通 rowid 表没有这些限制：

  1. 每个 WITHOUT ROWID 表必须有 PRIMARY KEY。 尝试创建没有 PRIMARY KEY 的 WITHOUT ROWID 表会导致错误。

  2. 与 “INTEGER PRIMARY KEY” 相关的特殊行为不适用于 WITHOUT ROWID 表。 在普通表中，"INTEGER PRIMARY KEY" 表示该列是 rowid 的别名。但在 WITHOUT ROWID 表中不存在 rowid，因此该特殊含义不再适用。WITHOUT ROWID 表中的 "INTEGER PRIMARY KEY" 列与普通表中的 "INT PRIMARY KEY" 列表现相同：它是具有整数亲和性的主键。

  3. AUTOINCREMENT 不适用于 WITHOUT ROWID 表。 AUTOINCREMENT 机制假定存在 rowid，因此在 WITHOUT ROWID 表上无法工作。如果在 WITHOUT ROWID 表的 CREATE TABLE 语句中使用 “AUTOINCREMENT” 关键字，就会引发错误。

  4. 在 WITHOUT ROWID 表中，PRIMARY KEY 的每一列都必须强制 NOT NULL。 这是遵循 SQL 标准的要求。主键的每一列都应当单独施加 NOT NULL 约束。然而早期版本的 SQLite 因为一个 bug 并未在 PRIMARY KEY 列上强制 NOT NULL。等到这个 bug 被发现时，已经有大量 SQLite 数据库流通，出于兼容性考虑决定不修复该 bug。因此 SQLite 中的普通 rowid 表违反了 SQL 标准，允许在 PRIMARY KEY 字段中插入 NULL。但 WITHOUT ROWID 表遵循标准，任何试图将 NULL 插入 PRIMARY KEY 列的操作都会导致错误。

  5. sqlite3_last_insert_rowid() 函数不适用于 WITHOUT ROWID 表。 向 WITHOUT ROWID 表插入数据不会改变 sqlite3_last_insert_rowid() 返回的值。SQL 函数 last_insert_rowid() 也不会受到影响，因为它只是对 sqlite3_last_insert_rowid() 的封装。

  6. 递增 BLOB I/O 机制不适用于 WITHOUT ROWID 表。 增量 BLOB I/O 使用 rowid 来创建用于直接 I/O 的 sqlite3_blob 对象。然而 WITHOUT ROWID 表没有 rowid，因此无法为 WITHOUT ROWID 表创建 sqlite3_blob 对象。

  7. sqlite3_update_hook() 接口不会因 WITHOUT ROWID 表的变化而触发回调。 sqlite3_update_hook() 的回调的一部分是发生变化的表行的 rowid。但 WITHOUT ROWID 表没有 rowid。因此，当 WITHOUT ROWID 表发生变化时，更新钩子不会被调用。

3. WITHOUT ROWID 表的优势

WITHOUT ROWID 表是一种可以降低存储和处理需求的优化。

在普通 SQLite 表中，PRIMARY KEY 实际上只是一个 UNIQUE 索引。用于在磁盘上查找记录的关键字是 rowid。普通 SQLite 表中的特殊 “INTEGER PRIMARY KEY” 列类型会导致该列成为 rowid 的别名，因此 INTEGER PRIMARY KEY 是一个真正的主键。但其他任何类型的 PRIMARY KEY，包括 “INT PRIMARY KEY”，在普通 rowid 表中仅仅是唯一索引。

考虑如下表，用来存储单词词汇及其在某段语料中出现的次数：

CREATE TABLE IF NOT EXISTS wordcount(
  word TEXT PRIMARY KEY,
  cnt INTEGER
);

作为普通 SQLite 表，“wordcount” 由两个独立的 B 树实现。主表使用隐藏的 rowid 值作为键，并将 “word” 和 “cnt” 列作为数据存储。“TEXT PRIMARY KEY” 片语会在 “word” 列上创建一个 唯一索引。该索引是一个独立的 B 树，使用 “word” 和 “rowid” 作为键，不存储任何数据。注意，每个 “word” 的完整文本被存储了两次：一次在主表中，一次在索引中。

考虑如下查询，用来查找单词 “xsync” 的出现次数：

SELECT cnt FROM wordcount WHERE word='xsync';

该查询首先在索引 B 树中查找与 “word” 匹配的条目。当在索引中找到条目时，会提取 rowid，并用它去搜索主表。然后从主表读取 “cnt” 值并返回。因此，需要进行两次独立的二叉搜索才能完成请求。

WITHOUT ROWID 表对等表采用不同的数据设计。

CREATE TABLE IF NOT EXISTS wordcount(
  word TEXT PRIMARY KEY,
  cnt INTEGER
) WITHOUT ROWID;

在后者表中，只有一个 B 树，它使用 “word” 列作为键，将 “cnt” 列作为数据。（技术上：底层实现实际上在 B 树的 “键” 区域中同时存储 “word” 和 “cnt”。但除非你查看数据库文件的底层字节编码，否则该事实并不重要。）由于只有单个 B 树，“word” 列的文本只在数据库中存储一次。此外，查询某个具体 “word” 的 “cnt” 值只需对主 B 树进行一次二叉搜索，因为第一次搜索找到的记录就可以直接读取 “cnt” 值，而无需再对 rowid 执行第二次搜索。

因此，在某些情况下，WITHOUT ROWID 表可以使用大约一半的磁盘空间，并且运行速度几乎是普通表的两倍。当然，在真实的数据库模式中，通常会有辅助索引和/或 UNIQUE 约束，情况会更加复杂。但即便如此，对于具有非整数或复合 PRIMARY KEY 的表，使用 WITHOUT ROWID 也往往能带来空间和性能上的优势。

4. 何时使用 WITHOUT ROWID

WITHOUT ROWID 优化通常对那些拥有非整数或复合（多列）PRIMARY KEY，并且不存储大字符串或 BLOB 的表特别有帮助。

对于具有单个 INTEGER PRIMARY KEY 的表，WITHOUT ROWID 表也能正确工作（也就是说，它们会返回正确结果）。然而，在这种情况下，普通 rowid 表的运行速度会更快。因此，从设计角度讲，最好避免为单列 INTEGER PRIMARY KEY 创建 WITHOUT ROWID 表。

当每行的数据不是很大时，WITHOUT ROWID 表的表现最佳。一个实用的经验是，WITHOUT ROWID 表中单行的平均大小应该小于数据库页大小的 1/20。也就是说，对于 1KiB 页面，行大小不应超过约 50 字节；对于 4KiB 页面，行大小不应超过约 200 字节。WITHOUT ROWID 表仍然可以处理任意大的行（最多 2GB），但传统 rowid 表在行较大时往往更快。原因在于 rowid 表实现为B* 树，所有内容都存储在树叶上，而 WITHOUT ROWID 表使用普通 B 树，内容分别存储在叶子节点和中间节点中。将内容存储在中间节点会使每个中间节点条目在页面上占据更多空间，从而降低扇出，增加搜索成本。

“sqlite3_analyzer.exe” 工具程序（可以在 SQLite 源代码树中以源码形式获取，也可在SQLite 下载页面中找到预编译的二进制文件）可用于测量现有 SQLite 数据库中表行的平均大小。

请注意，除了上文详细介绍的一些特殊情况外，WITHOUT ROWID 表和 rowid 表的行为是相同的。它们在相同 SQL 语句下会产生相同答案。因此，在产品开发后期，对应用进行实验来测试是否使用 WITHOUT ROWID 表是很简单的。一个良好的策略是：在开发末期再考虑 WITHOUT ROWID，当需要时再回头进行测试，看看在具有非整数 PRIMARY KEY 的表上添加 WITHOUT ROWID 是否有益；仅在它带来优势的情况下保留 WITHOUT ROWID。

5. 判断已有表是否为 WITHOUT ROWID

WITHOUT ROWID 表对 PRAGMA table_info 和 PRAGMA table_xinfo 返回与普通表相同的内容。但与普通表不同的是，WITHOUT ROWID 表还会响应 PRAGMA index_info 命令。对 WITHOUT ROWID 表执行 PRAGMA index_info 会返回有关该表 PRIMARY KEY 的信息。通过这种方式，PRAGMA index_info 命令可以明确判断某个表是 WITHOUT ROWID 表还是普通表——普通表总是返回空结果，而 WITHOUT ROWID 表总是返回一行或多行。
