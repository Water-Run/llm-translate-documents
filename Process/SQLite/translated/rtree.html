1. 概览

R-Tree 是一种专为执行范围查询而设计的特殊索引。R-Tree 最常见于地理空间系统，其中每条记录都是具有最小和最大 X、Y 坐标的矩形。给定查询的矩形，R-Tree 能快速找到所有被包含在查询矩形内或与查询矩形重叠的记录。这一思路很容易扩展到三维，用于 CAD 系统。R-Tree 也应用于时间域范围查找。例如，假设数据库记录了大量事件的开始和结束时间。R-Tree 能迅速找到在某个时间区间内任意时间处于活动状态的所有事件，或者在特定时间区间内开始的所有事件，或者在给定时间区间内既开始又结束的所有事件，诸如此类。

R-Tree 概念始于 Toni Guttman： R-Trees: A Dynamic Index Structure for Spatial Searching, Proc. 1984 ACM SIGMOD International Conference on Management of Data, 第47-57页。 SQLite 中的实现是 Guttman 原始想法的改进，通常称为“R*Tree”，由 Norbert Beckmann、Hans-Peter Kriegel、Ralf Schneider、Bernhard Seeger 描述： The R*-Tree: An Efficient and Robust Access Method for Points and Rectangles. SIGMOD 1990: 322-331。

2. 编译 R*Tree 模块

SQLite R*Tree 模块的源代码包含在合并源中。然而，取决于配置选项和所用的 SQLite 版本，它可能默认未启用。为确保启用 R*Tree 模块，只需在编译时定义 SQLITE_ENABLE_RTREE C 预处理宏。对于许多编译器，这可以通过在编译命令行中添加选项 "-DSQLITE_ENABLE_RTREE=1" 来实现。

3. 使用 R*Tree 模块

SQLite R*Tree 模块实现为一个虚表。每个 R*Tree 索引都是一个具有 3 到 11 列的虚表，列数为奇数。第一列始终是 64 位有符号整数主键。其余列按维度成对出现，每对依次包含该维度的最小值和最大值。因此一维 R*Tree 有 3 列，二维 R*Tree 有 5 列，三维 R*Tree 有 7 列，四维 R*Tree 有 9 列，五维 R*Tree 有 11 列。SQLite 的 R*Tree 实现不支持超过五维的 R*Tree。

SQLite R*Tree 的第一列类似于普通 SQLite 表的整数主键列。它只能存储 64 位有符号整数值。向该列插入 NULL 会导致 SQLite 自动生成新的唯一主键值。如果尝试插入任何其他非整数值，R*Tree 模块会在写入数据库前悄无声息地将其转换为整数。

在“rtree”虚表中，最小/最大值列存储为 32 位浮点数；在“rtree_i32”虚表中，则存储为 32 位有符号整数。与可存储多种数据类型和格式的普通 SQLite 表不同，R*Tree 严格执行这些存储类型。如果向此类列插入其他类型的值，R*Tree 模块会悄无声息地将其转换为所需类型，再写入新记录。

3.1. 创建 R*Tree 索引

新 R*Tree 索引的创建方式如下： 

CREATE VIRTUAL TABLE <name> USING rtree(<column-names>);

<name> 是应用为 R*Tree 索引选择的名称，<column-names> 是由 3 到 11 个列名组成的逗号分隔列表。虚拟表 <name> 会创建三个影子表来实际存储其内容。这些影子表的名称为： 

<name>_node

<name>_rowid

<name>_parent

影子表是普通的 SQLite 数据表。你可以直接查询它们，但这通常不会提供特别有用的信息。 你也可以UPDATE、DELETE、INSERT，甚至DROP 影子表，尽管这么做会破坏你的 R*Tree 索引。因此最好干脆忽略影子表，认识到它们存储着你的 R*Tree 索引信息，然后放手不管。

例如，考虑为空间查询创建一个二维的 R*Tree 索引： 

CREATE VIRTUAL TABLE demo_index USING rtree(
   id,              -- 整数主键
   minX, maxX,      -- X 坐标的最小值和最大值
   minY, maxY       -- Y 坐标的最小值和最大值
);

3.1.1. 列命名详解

在 CREATE VIRTUAL TABLE 语句中 rtree 的参数里，每个参数的第一个标记被视为列名。参数中的后续标记会被悄无声息地忽略。这意味着，例如，如果你试图为某列指定类型亲和，或者在列上添加 UNIQUE、NOT NULL、DEFAULT 等约束，这些额外的标记会被接受为合法但不会改变 R*Tree 的行为。 在 RTREE 虚表中，第一列的类型亲和始终是 INTEGER，其余数据列的类型亲和为 REAL。在 RTREE_I32 虚表中，所有列的类型亲和都是 INTEGER。

推荐的做法是在 rtree 的规范中省略任何额外的标记。让 rtree 的每个参数只由表示相应列名的普通标签构成，并从参数列表中省略所有其他标记。

3.2. 填充 R*Tree 索引

常用的INSERT、UPDATE 和DELETE 命令可像对普通表一样操作 R*Tree 索引。因此，要向示例 R*Tree 索引中插入数据，可以执行如下操作： 

INSERT INTO demo_index VALUES

  (28269, -80.851471, -80.735718, 35.272560, 35.407925),

3.3. 查询 R*Tree 索引

任何有效查询都可以作用于 R*Tree 索引。R*Tree 实现只是让某些类型的查询格外高效。针对主键的查询非常高效： 

SELECT * FROM demo_index WHERE id=28269;

当然，普通 SQLite 表也能高效地按照整型主键查询，所以上述例子并不特殊。使用 R*Tree 的主要原因是你可以高效地针对坐标范围执行范围查询。例如，SQLite 项目的主办公地点位于 35.37785，-80.77470。要找出可能为该办公地点提供服务的邮政编码，可以写： 

SELECT id FROM demo_index
 WHERE minX<=-80.77470 AND maxX>=-80.77470
   AND minY<=35.37785  AND maxY>=35.37785;

上述查询会快速定位包含 SQLite 主办公室的所有邮政编码边界框，即便 R*Tree 中有大量条目。前面是一个“完全包含”查询的示例。R*Tree 也支持“重叠”查询。例如，要查找所有与 28269 邮政编码重叠的边界框： 

SELECT A.id FROM demo_index AS A, demo_index AS B
 WHERE A.maxX>=B.minX AND A.minX<=B.maxX
   AND A.maxY>=B.minY AND A.minY<=B.maxY
   AND B.id=28269;

第二个查询会找到 28269 的条目（因为任何边界框都会与自身重叠），同时也会找到所有与 28269 的边界框存在重叠的邻近邮政编码。

请注意，R*Tree 索引中并非所有坐标都必须被约束，索引查询仍然可以高效。例如，你可能想要查询所有与第 35 个纬线重叠的对象： 

SELECT id FROM demo_index
 WHERE maxY>=35.0  AND minY<=35.0;

但总体而言，R*Tree 模块可利用的约束越多，边界框越小，返回结果就越快。

3.4. 舍入误差

默认情况下，R*Tree 中的坐标使用 32 位浮点数存储。当某个坐标无法被 32 位浮点数精确表示时，低界坐标会向下舍入，高界坐标会向上舍入。因此，边界框可能略大于指定值，但绝不会更小。这正是进行更常见的“重叠”查询（应用希望找到与查询边界框重叠的所有条目）所期待的行为。将条目的边界框外扩可能会导致在重叠查询中出现多出几条条目，尤其当条目边界正好与查询边界重合时。但重叠查询永远不会漏掉合法表项。

然而，对于“完全包含”样式的查询，将边界框向外扩可能导致某些条目被排除在结果之外，尤其当条目边界的某一边恰好与查询边界重合时。为防范此种情况，应用应略微扩大其完全包含查询的框（放宽 0.000012%），即在每个维度上分别向下舍入下界坐标并向上舍入上界坐标。

3.5. 同时读取和写入

Guttman 的 R-Tree 算法的特性决定了任何写操作都可能彻底重构树结构，并因此改变节点的扫描顺序。因此，一般无法在 R-Tree 查询进行中修改 R-Tree。尝试这样做会因 SQLITE_LOCKED “数据库表已锁定” 错误而失败。

例如，假设应用对 R-Tree 运行如下查询： 

SELECT id FROM demo_index
 WHERE maxY>=35.0  AND minY<=35.0;

然后对于返回的每个 “id” 值，应用创建如下 UPDATE 语句并将返回的 “id” 绑定到 “?1” 参数： 

UPDATE demo_index SET maxY=maxY+0.5 WHERE id=?1;

此时 UPDATE 可能会因 SQLITE_LOCKED 失败。原因是初始查询尚未完成执行，它在 R-Tree 的扫描过程中记住了当前位置。因此，R-Tree 不能在此期间接受更新，否则会打断扫描。

这是 R-Tree 扩展独有的限制。SQLite 中的普通表可以同时读取和写入。其他虚表也可能（也可能不）具备这样的能力。而在某些情况下，R-Tree 也能在某个时间点同时读取和写入，只要它能在启动更新前可靠地完成查询。但你不能指望每次都如此。一般来说，最好避免同时对同一个 R-Tree 执行查询和更新。

如果确实需要针对同一个 R-Tree 基于复杂查询执行更新，最佳做法是先运行复杂查询并将结果保存到临时表，然后再基于临时表中的值更新 R-Tree。

4. 高效使用 R*Tree

对于 3.24.0（2018-06-04）之前的 SQLite 版本，R*Tree 索引仅存储一个对象的整数 ID 及其边界框。附加信息需存储在单独表中，并通过主键与 R*Tree 索引关联。以之前的示例为例，可以创建如下辅助表： 

CREATE TABLE demo_data(
  id INTEGER PRIMARY KEY,  -- 主键
  objname TEXT,            -- 对象名称
  objtype TEXT,            -- 对象类型
  boundary BLOB            -- 对象的详细边界
);

在此示例中，demo_data.boundary 字段存储对象精确边界的某种二进制描述。R*Tree 索引仅保存对象的轴对齐矩形边界，代表对象真实边界的近似。因此，通常的做法是先用 R*Tree 索引将搜索范围缩小到候选对象列表，然后对每个候选进行更详细且开销更大的计算，判断它是否真正满足搜索条件。

关键点： R*Tree 索引通常不会给出确切答案，而只是将潜在答案的数量从数百万缩减到几十个。

假设 demo_data.boundary 字段保存某邮政编码复杂二维边界的专有数据描述，且应用已通过sqlite3_create_function() 接口创建了一个名为 "contained_in(boundary,lat,long)" 的自定义函数，该函数接受 demo_data.boundary 对象与一个纬度和经度，并返回布尔值表示该经纬是否包含在边界内。 可以假设 "contained_in()" 是一个相对较慢的函数，我们不希望频繁调用它。那么，为了查找主 SQLite 办公室对应的具体邮政编码，可以运行如下查询： 

SELECT objname FROM demo_data, demo_index
 WHERE demo_data.id=demo_index.id
   AND contained_in(demo_data.boundary, 35.37785, -80.77470)
   AND minX<=-80.77470 AND maxX>=-80.77470
   AND minY<=35.37785  AND maxY>=35.37785;

请注意上述查询的工作原理：R*Tree 索引在外层循环中运行，以找到边界框包含 SQLite 主办公室的条目。每找到一行，SQLite 就查找 demo_data 表中对应的记录，并将 demo_data 表的 boundary 字段作为参数传递给 contained_in() 函数。如果该函数返回 true，那么我们就知道所查找的坐标落在该邮政编码的边界内。

如果不使用 R*Tree 索引，可以运行下面这个更简单但效率更低的查询：

SELECT objname FROM demo_data
 WHERE contained_in(demo_data.boundary, 35.37785, -80.77470);

后者的问题在于它必须对 demo_data 表中的所有记录调用 contained_in() 函数。上一条查询中使用 R*Tree 的目的是将 contained_in() 的调用次数限制在整个表的一小部分。R*Tree 索引本身并未直接找出准确答案，它只是缩小了搜索空间。

4.1. 辅助列

从 SQLite 3.24.0（2018-06-04）起，R*Tree 表可以包含用于存储任意数据的辅助列。辅助列可以替代以前的“demo_data”这类辅助表。

辅助列通过在列名前加“+”符号来标记。辅助列必须放在所有坐标边界列之后。R*Tree 表的总列数不能超过 100，即整数主键列、坐标边界列和所有辅助列的总和必须不超过 100。下面的示例展示了一个包含辅助列的 R*Tree 表，其等价于之前的 demo_index 与 demo_data 两个表： 

CREATE VIRTUAL TABLE demo_index2 USING rtree(
   id,              -- 整数主键
   minX, maxX,      -- X 坐标的最小值和最大值
   minY, maxY,      -- Y 坐标的最小值和最大值
   +objname TEXT,   -- 对象名称
   +objtype TEXT,   -- 对象类型
   +boundary BLOB   -- 对象的详细边界
);

通过将位置信息和关联信息合并到同一张表中，辅助列可以提供更清晰的模型并减少连接的需求。 例如，此前 demo_index 与 demo_data 的连接现在可以用一个简单查询替代，如下所示： 

SELECT objname FROM demo_index2
 WHERE contained_in(boundary, 35.37785, -80.77470)
   AND minX<=-80.77470 AND maxX>=-80.77470
   AND minY<=35.37785  AND maxY>=35.37785;

4.1.1. 限制

对于辅助列，只有列名本身会被考虑。类型亲和会被忽略。NOT NULL、UNIQUE、REFERENCES 或 CHECK 等约束也会被忽略。但是，未来版本的 SQLite 可能会开始在意类型亲和与约束，因此建议辅助列的使用者保持这两者为空，以避免未来的兼容性问题。

5. 整数值 R-Tree

默认的虚表（“rtree”）将坐标存储为单精度（4 字节）浮点数。如果需要整数坐标，请改用“rtree_i32”创建表： 

CREATE VIRTUAL TABLE intrtree USING rtree_i32(id,x0,x1,y0,y1,z0,z1);

rtree_i32 将坐标存储为 32 位有符号整数。即便它使用整数存储值，rtree_i32 虚表在树算法内部仍然使用浮点运算。

6. 自定义 R-Tree 查询

通过在 SELECT 查询的 WHERE 子句中使用标准 SQL 表达式，程序员可以查询与某个边界框相交或包含在其中的所有 R*Tree 条目。自定义 R*Tree 查询通过在 WHERE 子句中使用 MATCH 操作符，允许程序员查询与任意区域或形状相交的 R*Tree 条目集合，而不仅限于矩形。这一能力在许多场景中非常有用，例如在以三维空间位置放置的相机视野中计算可见对象的子集。

自定义 R*Tree 查询的区域由应用实现并通过以下两个 API 之一注册到 SQLite 的 R*Tree 几何回调函数定义： 

int sqlite3_rtree_query_callback(
  sqlite3 *db,
  const char *zQueryFunc,
  int (*xQueryFunc)(sqlite3_rtree_query_info*),
  void *pContext,
  void (*xDestructor)(void*)
);
int sqlite3_rtree_geometry_callback(
  sqlite3 *db,
  const char *zGeom,
  int (*xGeom)(sqlite3_rtree_geometry *, int nCoord, double *aCoord, int *pRes),
  void *pContext
);

sqlite3_rtree_query_callback() 于 SQLite 3.8.5 版（2014-06-04）引入，是首选接口。 sqlite3_rtree_geometry_callback() 是较早且灵活性较差的接口，出于向后兼容仍予支持。

调用上述 API 之一会创建一个以第二个参数（zQueryFunc 或 zGeom）命名的新 SQL 函数。当此 SQL 函数出现在 MATCH 操作符的右侧，且 MATCH 操作符的左侧是 R*Tree 虚表中的任意列时，就会调用第三个参数（xQueryFunc 或 xGeom）定义的回调，判断某个对象或子树是否与所需区域重叠。

例如，可以用如下查询找到所有与以 (45.3, 22.9) 为圆心、半径为 5.0 圆相交的 R*Tree 条目： 

SELECT id FROM demo_index WHERE id MATCH circle(45.3, 22.9, 5.0)

无论使用 sqlite3_rtree_geometry_callback() 还是 sqlite3_rtree_query_callback() 注册 SQL 函数，自定义查询的 SQL 语法保持一致。但较新的查询式回调给予应用程序更大的查询控制能力。

6.1. 传统 xGeom 回调

传统的 xGeom 回调接收四个参数。第一个参数是指向 sqlite3_rtree_geometry 结构的指针，提供有关 SQL 函数如何被调用的信息。第二个参数是每个 R*Tree 条目中的坐标数量，对于同一个 R*Tree 始终相同。1 维 R*Tree 是 2，2 维是 4，3 维是 6，依此类推。第三个参数 aCoord[] 是包含 nCoord 个坐标的数组，用来定义待测试的边界框。最后一个参数是一个指针，用来写入回调结果。若 aCoord[] 定义的边界框完全在 xGeom 回调定义的区域之外，结果为 0；若边界框位于区域内部或与区域重叠，结果为非零。xGeom 回调通常应该返回 SQLITE_OK。如果返回任何非 SQLITE_OK 值，R-Tree 查询会以错误终止。

xGeom 回调的第一个参数所指向的 sqlite3_rtree_geometry 结构如下所示。对于同一个 MATCH 操作符在同一个查询中的每次回调，都会使用完全相同的 sqlite3_rtree_geometry 结构。SQLite 会初始化 sqlite3_rtree_geometry 结构的内容，但之后不会再修改。回调可以根据需要修改结构中的 pUser 和 xDelUser 成员。 

typedef struct sqlite3_rtree_geometry sqlite3_rtree_geometry;
struct sqlite3_rtree_geometry {
  void *pContext;                 /* sqlite3_rtree_geometry_callback() 中传入的 pContext 的副本 */
  int nParam;                     /* 数组 aParam 的大小 */
  double *aParam;                 /* 传递给 SQL 几何函数的参数 */
  void *pUser;                    /* 回调实现的用户数据 */
  void (*xDelUser)(void *);       /* SQLite 在清理 pUser 时调用 */
};

sqlite3_rtree_geometry 结构的 pContext 成员始终设置为注册回调时传给 sqlite3_rtree_geometry_callback() 的 pContext 参数的副本。aParam[]（大小为 nParam）包含传给 MATCH 操作符右侧 SQL 函数的参数值。在上例中的 “circle” 查询中，nParam 会是 3，aParam[] 数组包含 45.3、22.9 和 5.0。

sqlite3_rtree_geometry 结构的 pUser 与 xDelUser 成员初始为 NULL。回调实现可以将 pUser 设为任意有助于同一查询后续调用的值（例如指向用于测试区域交叉的复杂数据结构）。如果 xDelUser 设为非 NULL，那么当查询执行完成后 SQLite 会自动调用它，并将 pUser 值作为唯一参数。换言之，xDelUser 可以设置为用于销毁 pUser 的析构函数。

xGeom 回调总是以深度优先的方式遍历 R*Tree。

6.2. 新的 xQueryFunc 回调

新的 xQueryFunc 回调在每次调用时会接收来自 R-Tree 查询引擎的更多信息，并在返回前将更多信息发送回引擎。为保持接口的可控性，xQueryFunc 回调通过 sqlite3_rtree_query_info 结构的字段向查询引擎收发信息： 

struct sqlite3_rtree_query_info {
  void *pContext;                   /* 注册时传入的 pContext */
  int nParam;                       /* 函数参数数量 */
  sqlite3_rtree_dbl *aParam;        /* 函数参数的值 */
  void *pUser;                      /* 回调可选使用 */
  void (*xDelUser)(void*);          /* 释放 pUser 的函数 */
  sqlite3_rtree_dbl *aCoord;        /* 待检查的节点或条目的坐标 */
  unsigned int *anQueue;            /* 队列中待处理条目数 */
  int nCoord;                       /* 坐标数量 */
  int iLevel;                       /* 当前节点或条目的层级 */
  int mxLevel;                      /* 树中最大的 iLevel 值 */
  sqlite3_int64 iRowid;             /* 当前条目的 rowid */
  sqlite3_rtree_dbl rParentScore;   /* 父节点评分 */
  int eParentWithin;                /* 父节点的可见性 */
  int eWithin;                      /* 输出：当前可见性 */
  sqlite3_rtree_dbl rScore;         /* 输出：在此填写评分 */
  /* 以下字段仅在 3.8.11 及以后版本可用 */
  sqlite3_value **apSqlParam;       /* 参数的原始 SQL 值 */
};

sqlite3_rtree_query_info 结构的前五个字段与 sqlite3_rtree_geometry 结构一致，意义完全相同。它还包含 nCoord 与 aCoord 字段，其含义与 xGeom 回调中的同名参数一致。

xQueryFunc 必须根据 aCoord[] 定义的边界框是否完全在区域之外、与区域部分重叠或完全包含，于 sqlite3_rtree_query_info 的 eWithin 字段中填入 NOT_WITHIN、PARTLY_WITHIN 或 FULLY_WITHIN 之一。此外，xQueryFunc 还必须为 rScore 字段设置非负值，以指示子树与条目应该按什么顺序被分析和返回。分数越小越先处理。

顾名思义，R*Tree 按树状结构组织。树中每个节点都是一个边界框。根节点是覆盖树中所有元素的边界框。根之下有若干子树（通常 20 个以上），每个子树都有自己的较小边界框，并包含 R*Tree 条目的某个子集。子树可以有子子树，依此类推，直到叶子节点，这些叶子是真正的 R*Tree 条目。

一个 R*Tree 查询通过将根节点作为优先队列中唯一的条目并按 rScore 排序来初始化。查询的执行顺序是：从优先队列中取出评分最低的条目。如果该条目是叶子（也就是实际的 R*Tree 条目，而非子树），则将此条目作为查询结果的一行返回。如果取出的优先队列条目是一个节点（即子树），则将该节点的下一个子元素传给 xQueryFunc 回调。如果该节点还有更多子元素，则将其重新放回优先队列，否则丢弃。xQueryFunc 回调将 eWithin 设为 PARTLY_WITHIN 或 FULLY_WITHIN 的子元素会根据回调提供的分数加入优先队列。返回 NOT_WITHIN 的子元素会被丢弃。查询会一直进行，直到优先队列为空。

R*Tree 中的每个叶子条目与节点（子树）都有一个整数“层级”。叶子层级为 0；包含叶子的第一级子树的层级为 1；根节点拥有最大的层级值。sqlite3_rtree_query_info 结构中的 mxLevel 表示 R*Tree 根节点的层级值，而 iLevel 表示当前正在处理的元素的层级。

大多数 R*Tree 查询都使用深度优先搜索。可以通过将 rScore 设为 iLevel 实现。深度优先搜索通常更优，因为它能最小化优先队列中元素的数量，从而降低内存需求并提升处理速度。然而，一些应用可能偏好广度优先搜索，可以通过将 rScore 设为 mxLevel-iLevel 来实现。通过为 rScore 构造更复杂的公式，应用可以精细控制子树被搜索和叶子条目被返回的顺序。例如，在拥有数百万条 R*Tree 条目的应用中，可以让 rScore 的公式优先返回最大或最重要的条目，使应用能够迅速显示最关键的信息，并在后续逐步补全更小、更不重要的细节。

sqlite3_rtree_query_info 结构的其他字段也可供 xQueryFunc 回调使用。iRowid 字段表示当前正在考虑的条目（R*Tree 中 3 到 11 列中的第一个）对应的 rowid。iRowid 仅对叶子有效。eParentWithin 与 rParentScore 值分别是包含当前行的子树中的 eWithin 与 rScore 的副本。anQueue 是一个大小为 mxLevel+1 的无符号整数数组，表示每个层级当前优先队列中的元素数量。

6.3. 自定义查询的其他注意事项

自定义 R*Tree 查询函数的 MATCH 操作符必须是 WHERE 子句中由 AND 连接的顶层子句，否则 R*Tree 查询优化器无法使用它，查询也无法执行。 如果 MATCH 操作符通过 OR 与 WHERE 子句的其他项连接，例如，查询会因错误而失败。

同一个 WHERE 子句中可以出现两个或多个 MATCH 操作符，只要它们通过 AND 相连即可。然而，R*Tree 查询引擎仅维护一个优先队列。搜索中每个节点的优先级将是任意 MATCH 操作符返回的最低优先级。

7. 实现细节

以下各节描述 R*Tree 实现中的一些底层细节，有助于排查问题或进行性能分析。

7.1. 影子表

R*Tree 索引的内容实际存储在三个普通的 SQLite 表中，其名称由 R*Tree 的名字派生而来。这三张表称为“影子表”。它们的模式如下： 

CREATE TABLE %_node(nodeno INTEGER PRIMARY KEY, data)
CREATE TABLE %_parent(nodeno INTEGER PRIMARY KEY, parentnode)
CREATE TABLE %_rowid(rowid INTEGER PRIMARY KEY, nodeno)

每个影子表名称中的“%”会被 R*Tree 虚表的名称替换。因此，如果 R*Tree 表名为 “xyz”，那么三张影子表分别为 “xyz_node”、“xyz_parent” 和 “xyz_rowid”。

%_node 表中有一条记录对应每个 R*Tree 节点。R*Tree 节点由彼此接近的一个或多个条目组成。R*Tree 的节点格式像树。除根节点之外的所有节点在 %_parent 影子表中都有一条记录，指向其父节点。R*Tree 中的每个条目都有一个 rowid。%_rowid 影子表将条目 rowid 映射到包含该条目的节点。

附加到 %_rowid 表的额外列存放辅助列的内容。这些额外的 %_rowid 列的名称可能与实际的辅助列名称不同。

7.2. 使用 rtreecheck() SQL 函数进行完整性检查

标量 SQL 函数 rtreecheck(R) 或 rtreecheck(S,R) 会对数据库 S 中名为 R 的 rtree 表执行完整性检查。该函数会返回对发现问题的自然语言描述，或者在一切正常时返回字符串 'ok'。在 R*Tree 虚表上运行 rtreecheck() 与在数据库上运行PRAGMA integrity_check 类似。

示例：若要验证名为 “demo_index” 的 R*Tree 是否结构良好且内部一致，请执行： 

SELECT rtreecheck('demo_index');

rtreecheck() 函数执行以下检查：

  1. 对于 R*Tree 结构（%_node 表）中的每个单元：

    1. 对于每个维度，要求 coord1 <= coord2。

    2. 除了根节点之外，要求该单元受到父节点所在父单元的约束。

    3. 对于叶子节点，要求 %_rowid 表中有一条记录与该单元的 rowid 值对应，并指向正确的节点。

    4. 对于非叶子节点，要求 %_parent 表中有一条记录，将该单元的子节点映射到其所在的节点。

  2. %_rowid 表中的记录数应与 R*Tree 结构中的叶子单元数相同，并且 %_rowid 表中的每条记录都必须对应一个叶子单元。

  3. %_parent 表中的记录数应与 R*Tree 结构中的非叶子单元数相同，并且 %_parent 表中的每条记录都必须对应一个非叶子单元。
