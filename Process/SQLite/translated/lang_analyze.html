1. 概述

analyze-stmt:

ANALYZE 命令会收集有关表和索引的统计信息，并将这些信息存储在数据库的内部表中，查询优化器可以访问这些信息并利用它们帮助做出更好的查询规划选择。如果不提供任何参数，则会分析主数据库及所有附加数据库。如果参数是模式名，那么仅分析该数据库中的所有表和索引。如果参数是表名，则仅分析该表及其关联的索引。如果参数是索引名，则仅分析该索引。

2. 推荐的使用模式

ANALYZE 的使用从来不是必需的。然而，如果一个应用执行的查询非常复杂且存在许多可能的查询计划，当 ANALYZE 已运行时，查询规划器会更有能力挑选出最佳计划。这对某些查询可以带来显著的性能提升。

下面的子节按优先级顺序介绍了两种推荐的 ANALYZE 运行方式。

2.1. 定期运行 “PRAGMA optimize”

PRAGMA optimize 命令会在需要时自动运行 ANALYZE。建议用法：

  1. 与数据库连接寿命较短的应用应在关闭每个数据库连接之前一次性运行 “PRAGMA optimize;”。

  2. 使用长连接的应用应在连接首次打开时运行 “PRAGMA optimize=0x10002;”，之后定期运行 “PRAGMA optimize;”，例如每天运行一次，若数据库变化迅速则更频繁。

  3. 所有应用在架构发生更改后，特别是在执行一个或多个 CREATE INDEX 语句之后，都应运行 “PRAGMA optimize;”。

PRAGMA optimize 命令通常是空操作，但偶尔会在某些表上运行一个或多个 ANALYZE 子命令，以便对查询规划器有益。自 SQLite 3.46.0（2024-05-23）起，“PRAGMA optimize” 命令会自动限制 ANALYZE 子命令的范围，使整体命令即便在庞大数据库上也能迅速完成。无需使用 PRAGMA analysis_limit。这是今后运行 ANALYZE 的推荐方式。

PRAGMA optimize 命令通常只会考虑在同一数据库连接已查询过或在 sqlite_stat1 表中尚无条目的表上运行 ANALYZE。但如果在参数中添加 0x10000 位，PRAGMA optimize 就会检查所有表，看它们是否能从 ANALYZE 中获益，而不仅限于最近查询过的表。数据库连接首次打开时没有查询历史，因此在对新的连接运行 PRAGMA optimize 时建议添加 0x10000 位。

更多信息请参见下文的 自动运行 ANALYZE 和 大型数据库的近似 ANALYZE 章节。

2.2. ANALYZE 的固定结果

运行 ANALYZE 会导致 SQLite 为后续查询选择不同的查询计划。这几乎总是件好事，因为 ANALYZE 之后选择的查询计划在几乎所有情况下都比 ANALYZE 之前的查询计划更好。这正是 ANALYZE 的意义所在。但不能证明运行 ANALYZE 总是有益。可以构造出病态的例子，使运行 ANALYZE 反而让某些后续查询变慢。

一些开发者希望在应用设计固定后，SQLite 始终选择开发测试期间选定的相同查询计划。这样，如果向客户发布数百万份应用拷贝，开发者就能确信无论各自数据库中插入了什么数据，这些拷贝都运行相同的查询计划。这有助于重现来自现场的性能投诉。

要实现这一目标，不要在应用中运行完整 ANALYZE 或 “PRAGMA optimize”。相反，只在开发期间，手动使用 命令行界面 或类似方式，在与生产环境大小和内容相似的测试数据库上运行 ANALYZE。然后使用如下脚本捕获这一次 ANALYZE 的结果：

.mode list
SELECT 
  'ANALYZE sqlite_schema;' ||
  'DELETE FROM sqlite_stat1;' ||
  'INSERT INTO sqlite_stat1(tbl,idx,stat)VALUES' ||
  (SELECT group_concat(format('(%Q,%Q,%Q)',tbl,idx,stat),',')
    FROM sqlite_stat1) ||
  ';ANALYZE sqlite_schema;';

在部署版本创建新的数据库实例时，或对于长时间运行的应用每次启动时，运行上述脚本生成的命令。这样可以将 sqlite_stat1 表填充为开发测试时期的状态，并确保现场选取的查询计划与实验室测试中选取的一致。也许将脚本生成的字符串复制粘贴到名为 “zStat1Init” 的静态字符串常量中，然后调用：

sqlite3_exec(db, zStat1Init, 0, 0, 0);

根据脚本运行的上下文，可能还应在字符串常量开头加上 “BEGIN;”，末尾加上 “COMMIT;”。

有关更多信息，请参见 查询规划器稳定性保证。

3. 详细信息

默认实现将所有统计信息存储在名为 "sqlite_stat1" 的单个表中。如果 SQLite 在编译时启用了 SQLITE_ENABLE_STAT4 选项，则还会收集额外的直方图数据并存储在 sqlite_stat4 中。旧版本的 SQLite 在启用 SQLITE_ENABLE_STAT2 或 SQLITE_ENABLE_STAT3 编译时会使用 sqlite_stat2 或 sqlite_stat3，但所有近期版本都忽略 sqlite_stat2 和 sqlite_stat3。未来的增强可能会创建名称模式相同但末尾数字大于 “4” 的其他内部表。所有这些表统称为“统计表”。

统计表的内容可以通过 SELECT 查询，并可使用 DELETE、INSERT 和 UPDATE 命令进行更改。从 SQLite 3.7.9（2011-11-01）开始，DROP TABLE 命令亦可用于统计表。ALTER TABLE 命令不能作用于统计表。修改统计表内容时应格外小心，因为无效内容可能导致 SQLite 选择低效的查询计划。一般来说，应只通过运行 ANALYZE 命令来修改统计表内容。更多信息请参见 “使用 SQLITE_STAT 表手动控制查询计划”。

ANALYZE 收集的统计信息不会随着数据库内容的变化而自动更新。如果数据库内容发生显著变化，或数据库模式发生更改，应考虑重新运行 ANALYZE 以更新统计信息。

查询规划器在读取模式时会将统计表的内容加载到内存中。因此当应用直接修改统计表时，SQLite 不会立即注意到更改。应用可以通过运行 ANALYZE sqlite_schema 强制查询规划器重新读取统计表。

4. 自动运行 ANALYZE

PRAGMA optimize 命令会根据需要自动对单个表运行 ANALYZE。推荐做法是应用在关闭每个数据库连接之前执行 PRAGMA optimize。或者，如果应用保持单个数据库连接长时间打开，那么在首次打开时运行 “PRAGMA optimize=0x10002”，之后再定期运行 “PRAGMA optimize;”，例如每天或甚至每小时运行一次。

每个 SQLite 数据库连接 会记录查询规划器什么时候会因拥有准确的 ANALYZE 结果而受益。这些记录保存在内存中，并在数据库连接生命周期内累积。PRAGMA optimize 命令会检查这些记录，仅对那些新 ANALYZE 数据可能有用的表运行 ANALYZE。在大多数情况下，PRAGMA optimize 不会运行 ANALYZE，但偶尔会对从未分析过的表或自上次分析以来规模显著增长的表运行 ANALYZE。

由于 PRAGMA optimize 的行为在某种程度上受同一数据库连接上之前执行的查询影响，建议将 PRAGMA optimize 延迟到数据库连接关闭时再运行，这样连接就有机会收集尽可能多的使用信息。对于长时间保持打开的连接，也可以设定定时器每隔几小时或几天运行一次 PRAGMA optimize。在连接刚打开时立即运行 PRAGMA optimize 时，可以在位掩码参数中添加 0x10000 位（即运行 “PRAGMA optimize=0x10002”），此时所有表都会被检查，甚至包括本次连接期间尚未查询过的表。

PRAGMA optimize 命令最初在 SQLite 3.18.0（2017-03-28）中引入，并且在之前的版本中没有任何作用。PRAGMA optimize 命令在 SQLite 3.46.0（2024-05-23）中得到了显著增强，本节的建议以这些增强为基础。使用较早版本 SQLite 的应用应参考对应版本的文档，以便获得关于如何更好地使用 PRAGMA optimize 的建议。

5. 大型数据库的近似 ANALYZE

默认情况下，ANALYZE 会对每个索引执行完整扫描。这对大型数据库来说可能很慢。因此自 SQLite 3.32.0（2020-05-22）起，可以使用 PRAGMA analysis_limit 命令限制 ANALYZE 所执行的扫描量，从而让 ANALYZE 在非常大的数据库文件上也更快地运行。我们将这种方式称为“近似 ANALYZE”。

对于 analysis_limit 预处理指令，推荐如下用法：

PRAGMA analysis_limit=1000;

该预处理指令告诉 ANALYZE 命令像往常一样开始对索引执行完整扫描。但当访问的行数达到 1000（或其他由该预处理指令指定的限制）时，ANALYZE 就会开始采取措施停止扫描。如果在前 1000 步中最左列至少改变过一次，则分析立即停止。但如果最左列始终相同，ANALYZE 会跳过至第一个最左列不同的条目，并在终止前再读取额外的 1000 行。

前述分析限制的具体影响在未来的 SQLite 版本中可能会有所变化。但核心思想会保持一致。分析限制 N 会力求将每个索引访问的行数限制在大致 N 左右。

推荐将 N 设置在 100 到 1000 之间。或者将分析限制设置为 0，以使 ANALYZE 对每个索引执行完整扫描，从而禁用该限制。该预处理指令的默认值为 0，以保持向后兼容。

近似 ANALYZE 在 sqlite_stat1 表中写入的值与无约束分析计算的值不完全相同。但通常足够接近。无论如何，sqlite_stat1 表中的索引统计信息本身就是近似值，因此近似 ANALYZE 与传统全扫描 ANALYZE 结果略有不同在实际中影响不大。确实可以构造出病态场景，使近似 ANALYZE 明显不如全扫描 ANALYZE，但在现实问题中这种情况很少见。

一个实用的经验是在运行任何 ANALYZE 之前始终先设置 “PRAGMA analysis_limit=N”，其中 N 介于 100 到 1000 之间。过去也建议在运行 “PRAGMA optimize” 之前这么做，但自 3.46.0（2024-05-23）起，这一步会自动完成。使用 PRAGMA analysis_limit 时，结果虽然没有那么精确，但已经足够准确，且由于计算速度更快，开发者更有可能去执行它。近似 ANALYZE 总比完全不运行 ANALYZE 更好。

5.1. 近似 ANALYZE 的限制

sqlite_stat4 表中的内容必须通过全表扫描才能计算。因此如果指定了非零的分析限制，则不会计算 sqlite_stat4 表。
