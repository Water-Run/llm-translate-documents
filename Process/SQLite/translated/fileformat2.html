本文档描述并定义了自 SQLite 3.0.0（2004-06-18）以来所有版本所使用的磁盘数据库文件格式。

1. 数据库文件

SQLite 数据库的完整状态通常包含在称为“主数据库文件”的单个磁盘文件中。

在事务期间，SQLite 会将附加信息存储在名为“回滚日志”的第二个文件中；如果 SQLite 处于WAL 模式，则使用预写日志文件。

1.1. 热日志

如果在事务完成前应用程序或主机计算机崩溃，那么回滚日志或预写日志中包含恢复主数据库文件一致性状态所需的信息。当回滚日志或预写日志中携带恢复数据库状态所需信息时，它们被称为“热日志”或“热 WAL 文件”。热日志和 WAL 文件仅在错误恢复场景中才会出现，因此并不常见，但它们构成了 SQLite 数据库状态的一部分，不能被忽视。本文定义了回滚日志和预写日志的格式，但重点仍放在主数据库文件上。

1.2. 页面

主数据库文件由一个或多个页面组成。页面的大小为 512 到 65536 之间的二的幂。相同数据库内的所有页面大小相同。数据库文件的页面大小由从数据库文件开头偏移 16 字节处的 2 字节整数确定。

页面编号从 1 开始。最大页面编号为 4294967294（\(2^{32} - 2\)）。最小的 SQLite 数据库为单个 512 字节页面。最大数据库大小为 4294967294 个页面，每页 65536 字节，共计 281,474,976,579,584 字节（约 281 TB）。通常 SQLite 会在达到自身内部限制之前就受到底层文件系统或磁盘硬件最大文件大小的限制。

在实际使用中，SQLite 数据库大小通常在几 KB 到几 GB 之间，尽管在生产环境中也存在 TB 级别的 SQLite 数据库。

在任意时间点，主数据库中的每个页面仅具有以下用途之一：

  • 一个 b 树页面
    • 一个表 b 树内部页面
    • 一个表 b 树叶页面
    • 一个索引 b 树内部页面
    • 一个索引 b 树叶页面
  • 一个空闲列表页面
    • 一个空闲列表主干页面
    • 一个空闲列表叶页面
  • 一个载荷溢出页面
  • 一个指针映射页面
  • 锁字节页面

对主数据库文件的所有读写都以页面边界开始，所有写操作都是整数个页面大小。读取通常也是整数个页面，唯一的例外是初次打开数据库时，数据库文件的前 100 字节（即数据库文件头）作为子页面单位被读取。

1.3.1. 魔术头字符串

每个有效的 SQLite 数据库文件都以以下 16 字节开头（十六进制）：53 51 4c 69 74 65 20 66 6f 72 6d 61 74 20 33 00。此字节序列对应包括末尾空字符的 UTF-8 字符串“SQLite format 3”。

1.3.2. 页面大小

从偏移 16 开始的两字节值决定数据库页面大小。SQLite 3.7.0.1（2010-08-04）及之前版本将其视为大端整数，必须为 512 到 32768 之间的二的幂。自 SQLite 3.7.1 版（2010-08-23）起，支持 65536 字节页面。由于 65536 无法存入两字节整数，因此要表示 65536 字节页面大小，偏移 16 处的值为 0x00 0x01。该值可解读为大端的 1，被视为表示 65536 页面大小的魔数。也可以将两字节字段视为小端数，表示页面大小除以 256。这两种解读等价。

1.3.3. 文件格式版本号

偏移 18 和 19 处的文件格式写版本与读版本旨在允许未来版本对文件格式进行增强。目前版本中，针对回滚日志模式这两个值都是 1，而对于WAL模式则都是 2。如果符合当前文件格式规范的 SQLite 遇到读版本为 1 或 2 且写版本大于 2 的数据库文件，则该文件必须以只读方式处理。如果遇到读版本大于 2 的数据库文件，则无法读取或写入。

1.3.4. 每页保留字节

SQLite 能够在每页末尾预留少量额外字节供扩展使用。例如，SQLite 加密扩展会使用这些额外字节存储与每页相关联的随机数和/或校验和。在头部偏移 20 处的 1 字节整数表示“保留空间”的大小，即每页末尾为扩展保留的字节数。通常该值为 0，可以是奇数。

数据库页面的“可用大小”是头部偏移 16 处的两字节整数所指定的页面大小减去头部偏移 20 处的“保留”空间大小。可用大小可能是奇数，但不得小于 480。换句话说，如果页面大小为 512，则保留空间不能超过 32。

1.3.5. 载荷分数

最大/最小嵌入载荷分数以及叶载荷分数必须分别为 64、32 和 32。这些值原本设计为可调参数，用于修改 b 树算法的存储格式。然而，该功能未被启用，也没有计划未来支持。因此这三个字节被固定为上述值。

1.3.6. 文件更改计数器

偏移 24 处的四字节大端整数表示文件更改计数器，每次数据库文件在修改后解锁时递增。当两个或多个进程同时读取同一数据库文件时，每个进程可通过监视更改计数器来检测其他进程的更改。一旦其他进程修改了数据库，本进程通常希望刷新页面缓存以避免缓存失效。文件更改计数器实现了这一功能。

在 WAL 模式中，数据库更改通过 wal-index 检测，因此无需使用更改计数器。因此，在 WAL 模式下更改计数器可能不会在每个事务都递增。

1.3.7. 头内数据库大小

头部偏移 28 处的四字节大端整数存储数据库文件的页面数。如果该头内数据库大小无效（见下一段），则数据库大小通过实际文件大小推算。旧版本的 SQLite 忽略头内数据库大小，仅使用实际文件大小。新版本会优先使用有效的头内数据库大小，若无效则退回到实际文件大小。

头内数据库大小只有当非零且位于偏移 24 的四字节更改计数器与偏移 92 的四字节版本有效号完全匹配时才被视为有效。在仅由 SQLite 3.7.0（2010-07-21）及更高版本修改数据库时，该值始终有效。如果旧版本 SQLite 写入数据库，它不会更新头内数据库大小，因此该大小可能不准确。但旧版本也不会修改偏移 92 处的版本有效号，使其与更改计数器不匹配。因此，通过观察更改计数器与版本有效号不一致即可检测并忽略无效的头内数据库大小。

1.3.8. 空闲页列表

数据库文件中的未使用页面存储在空闲列表上。头部偏移 32 处的四字节大端整数存储空闲列表首个页面的页码；如果空闲列表为空则为零。头部偏移 36 处的四字节大端整数存储空闲列表页面的总数。

1.3.9. 模式 Cookie

偏移 40 处的四字节大端整数表示模式 Cookie，每当数据库模式发生更改时递增。预编译语句针对特定模式版本编译，若模式变化则必须重新准备。执行语句时，它会首先检查模式 Cookie 是否与编译时一致；若已改变，语句要么自动重新准备并重新执行，要么以SQLITE_SCHEMA错误中止。

1.3.10. 模式格式编号

偏移 44 处的四字节大端整数表示模式格式编号。它类似于偏移 18 与 19 的文件格式读/写版本号，但作用于高层 SQL 格式。当前定义了四个模式格式编号：

  1. 格式 1 可被回溯至3.0.0 版（2004-06-18）的所有 SQLite 版本识别。
  2. 格式 2 允许同一表中行具有不同列数，以支持ALTER TABLE ... ADD COLUMN。SQLite 在3.1.3 版（2005-02-20）新增对格式 2 的读写支持。
  3. 格式 3 为ALTER TABLE ... ADD COLUMN添加的新增列提供非 NULL 默认值支持。该能力在3.1.4 版（2005-03-11）被引入。
  4. 格式 4 使 SQLite 在索引声明中尊重DESC 关键字。（在格式 1、2、3 中忽略 DESC。）格式 4 还增加两个新的布尔记录类型值（序列类型 8 与 9）。SQLite 3.3.0（2006-01-10）增加了对格式 4 的支持。

SQLite 创建的新数据库文件默认使用格式 4。通过sqlite3_db_config()接口的SQLITE_DBCONFIG_LEGACY_FILE_FORMAT选项可使 SQLite 创建格式 1。编译时将SQLITE_DEFAULT_FILE_FORMAT=1可使默认格式编号为 1 而非 4。

如果数据库完全为空且没有模式，则模式格式编号可为零。

1.3.11. 推荐缓存大小

偏移 48 处的四字节大端有符号整数表示推荐的页面缓存大小（单位：页）。该值仅为建议，SQLite 无义务遵循。整数的绝对值作为建议缓存大小。可通过default_cache_size pragma设置推荐缓存大小。

1.3.12. 增量清理设置

偏移 52 与 64 处的两个四字节大端整数用于管理auto_vacuum与incremental_vacuum模式。若偏移 52 的整数为零，则数据库文件中省略指针映射（ptrmap）页面，并且不支持 auto_vacuum 与 incremental_vacuum。若该整数非零，则它表示数据库中最大的根页面页码，数据库文件将包含 ptrmap 页面，模式必须为 auto_vacuum 或 incremental_vacuum。在后一种情况下，偏移 64 的整数为增量清理时为真，为自动清理时为假。若偏移 52 为零，则偏移 64 也必须为零。

1.3.13. 文本编码

偏移 56 处的四字节大端整数决定数据库中所有文本字符串的编码。

值为 1 表示 UTF-8。

值为 2 表示 UTF-16le。

值为 3 表示 UTF-16be。

不允许其他值。

sqlite3.h 头文件定义了 C 预处理宏 SQLITE_UTF8 = 1、SQLITE_UTF16LE = 2 和 SQLITE_UTF16BE = 3，可用于替代文本编码的数字代码。

1.3.14. 用户版本号

偏移 60 处的四字节大端整数为通过user_version pragma设置与查询的用户版本。用户版本不被 SQLite 使用。

1.3.15. 应用程序 ID

偏移 68 处的四字节大端整数是可通过PRAGMA application_id命令设置的“应用程序 ID”，用于识别数据库属于或关联的特定应用。应用程序 ID 旨在供应用程序文件格式使用。诸如file(1)等工具可根据该 ID 判断具体文件类型，而不只是报“SQLite3 Database”。分配的应用程序 ID 可在 SQLite 源仓库的magic.txt文件中查阅。

1.3.16. 写库版本号与版本有效号

偏移 96 处的四字节大端整数存储最近修改该数据库文件的 SQLite 库的SQLITE_VERSION_NUMBER值。偏移 92 处的四字节大端整数为存储版本号时的更改计数器值。偏移 92 的整数表示该版本号适用于的事务，通常称为“版本有效号”。

1.3.17. 头部扩展保留空间

数据库文件头的其他字节保留以供未来扩展，必须设置为零。

1.4. 锁字节页面

锁字节页面是数据库文件中的单页，包含偏移 1073741824 到 1073742335（含）的字节。若数据库文件大小小于等于 1073741824 字节则不包含锁字节页面；若大于 1073741824，则正好包含一个锁字节页面。

锁字节页面保留供操作系统特定的VFS实现用于数据库文件锁原语。SQLite 本身不使用锁字节页面。SQLite 核心永远不会读取或写入该页面，但操作系统特定的VFS实现可根据底层系统的需求选择读取或写入。随 SQLite 搭载的 unix 与 win32VFS实现不写入锁字节页面，但其他操作系统的第三方 VFS 实现可能会。

锁字节页面的设计源于需要支持当时主流的 Win95，其只支持强制文件锁。我们所知的现代操作系统皆支持建议式文件锁，因此锁字节页面已非必须，但为向后兼容仍保留。

1.5. 空闲列表

数据库文件可能包含一个或多个暂不使用的页面。例如删除数据后会产生未使用页面。空闲页面存储在空闲列表中，当需要新页面时则复用。

空闲列表组织为一系列空闲列表主干页的链表，每个主干页包含零个或多个空闲列表叶页的页码。

空闲列表主干页由一组四字节大端整数构成，其数量以能填满页面可用空间为准。最小可用空间为 480 字节，因此数组长度至少为 120 项。主干页中的第一个整数为下一个主干页的页码，若为零则表示该页是最后一页。第二个整数表示后续叶页指针的个数。设第二个整数为 L，若 L 大于零，则索引 2 到 L+1 的整数为空闲列表叶页的页码。

空闲列表叶页不包含信息。SQLite 避免读取或写入空闲列表叶页以减少磁盘 I/O。

SQLite 3.6.0（2008-07-16）之前的版本存在 Bug：若主干页数组的最后 6 项中有非零值，则会报数据库损坏。新版本无此问题。但新版本仍避免使用主干页数组的最后 6 项，以便由新版本创建的数据库能够被旧版本读取。

空闲列表页面数存储在文件头偏移 36 处的四字节大端整数中，空闲列表主干页的页码存储在偏移 32 处的四字节大端整数中。

1.6. B 树页面

B 树算法提供基于页面存储设备的键/数据存储，支持唯一且有序的键。关于 B 树的背景可参考 Knuth 的《计算机程序设计艺术》第三卷《排序与检索》，第 471-479 页。SQLite 使用两类 B 树：“表 B 树”使用 64 位有符号整数键，数据存储在叶子；“索引 B 树”使用任意键，不存储数据。

B 树页面要么是内部页面，要么是叶页面。叶页面包含键，对于表 B 树，每个键还关联数据。内部页面包含 K 个键及 K+1 个指向子页面的指针。内部页面中的“指针”即子页面的 32 位无符号整数页码。

内部 B 树页面上的键数 K 通常至少为 2，且一般远大于 2。唯一例外是第 1 页为内部页面的情况。由于文件头占用前 100 字节，第 1 页可用空间少 100 字节，因此极少数情况下若第 1 页为内部页面，可能只持有一个键。其他情况均为 K≥2。K 的上限为能容纳在页面上的最大键数。索引 B 树中的大键会被拆分成溢出页面，确保单个键占用不超过页面可用空间的四分之一，从而每个内部页面至少可存储四个键。表 B 树的整数键从不需要溢出，因此只有索引 B 树存在键溢出。

定义叶 B 树的深度为 1，任何内部 B 树的深度为其所有子树最大深度再加 1。在结构良好的数据库中，内部页面的所有子节点深度相同。

在内部 B 树页面中，指针与键逻辑上交替排列，两端各有一个指针（此描述为概念性，实际键与指针的布局更复杂，下文将予描述）。同一页面中的所有键唯一且按逻辑从左向右升序排列（该顺序逻辑性存在，非物理排列）。对于任意键 X，位于 X 左侧的指针指向所有键小于等于 X 的页面；位于右侧的指针指向所有键大于 X 的页面。

在内部 B 树页面中，每个键与其左侧的指针组合成一个称为“单元”的结构。最右侧的指针单独存储。叶页面没有指针，但仍使用单元结构存储索引 B 树的键或表 B 树的键与内容，同时数据包含在单元中。

每个 B 树页面最多有一个父页面。未拥有父页面的 B 树页面称为根页面。根页面与其所有子树闭包共同构成完整的 B 树。单页即可构成既为叶又为根的完整 B 树是常见情况。由于父页面含有指向子页面的指针，只要知道根页面即可确定完整 B 树的所有页面，因此 B 树通过其根页面页码识别。

B 树页面要么是表 B 树页面，要么是索引 B 树页面。每棵完整 B 树内所有页面类型一致：要么是表，要么是索引。数据库文件中每个 rowid 表（含系统表如sqlite_schema）对应一棵表 B 树。每个模式中的索引（含由唯一性约束隐式创建的索引）对应一棵索引 B 树。虚表没有 B 树。某些虚表实现可能使用影子表用于存储，但其在数据库模式中有独立条目。WITHOUT ROWID 表使用索引 B 树而非表 B 树，因此每个 WITHOUT ROWID 表对应一棵索引 B 树。sqlite_schema 表对应的 B 树始终是表 B 树且根页面为 1。sqlite_schema 表包含数据库文件中每个其他表与索引的根页面页码。

表 B 树中的每个条目由 64 位有符号整数键及最多 2147483647 字节的任意数据构成。（表 B 树键对应于该 SQL 表的rowid。）内部表 B 树仅保存键与子页面指针，所有数据都存储在叶页面中。

索引 B 树中的每个条目由最多 2147483647 字节的任意键组成，不包含数据。

定义单元的“载荷”为其任意长度部分。对于索引 B 树，键始终为任意长度，故载荷即键。表 B 树内部页面的单元没有任意长度元素，因此没有载荷。表 B 树叶页面的单元含任意长度内容，因此载荷即内容。

当单元的载荷大小超过某个阈值（稍后定义）时，仅将前若干字节存储在 B 树页面上，其余部分存入一系列内容溢出页面。

B 树页面按以下区域顺序划分：

  1. 100 字节数据库文件头（仅出现在第 1 页）
  2. 8 或 12 字节的 B 树页面头
  3. 单元指针数组
  4. 未分配空间
  5. 单元内容区
  6. 保留区域

100 字节的数据库文件头仅存在于第 1 页，该页始终为表 B 树页面。数据库文件中的其他 B 树页面均省略此 100 字节头。

保留区域为每页末尾（锁页面除外）的未使用空间，供扩展存放每页信息。保留区域的大小由数据库文件头偏移 20 处的 1 字节无符号整数决定，通常为零。

叶页面的 B 树页面头大小为 8 字节，内部页面为 12 字节。页面头中的多字节值均为大端。页面头字段如下：

单元指针数组紧随页面头。设 K 为页面上的单元数。单元指针数组由 K 个两字节整数偏移组成，指向单元内容。指针按键顺序排列，最左侧（最小键）优先，最右侧（最大键）最后。

单元内容存储在页面的单元内容区域。SQLite 努力将单元安排在页面末端，以便为将来扩展指针数组留出空间。最后一个指针与第一个单元之间的区域为未分配区域。

若页面没有单元（仅在表中无行的根页面可能发生），则单元内容区域的偏移等于页面大小减去保留空间字节数。若数据库使用 65536 字节页面且保留空间为零（常见情况），则空页面的单元内容偏移应为 65536。但该值无法存入两字节无符号整数，因此以 0 代替。

空闲块结构用于识别页面中的未分配空间。空闲块以链表方式组织。空闲块前两字节为大端整数，指向链中下一个空闲块的偏移，若为零则表示链尾。第三与第四字节为大端整数，表示空闲块大小（包含 4 字节头）。空闲块按偏移递增顺序连接。页面头第二字段为首个空闲块的偏移，若无则为零。在结构良好的页面中，首个空闲块之前总至少有一个单元。

空闲块至少占用 4 字节。若单元内容区中存在孤立的 1、2 或 3 个未使用字节，则构成碎片。这些碎片的总字节数存储在页面头第五字段中。在结构良好的页面中，碎片字节总数不得超过 60。

页面上的自由空间总量为：未分配区域大小 + 所有空闲块大小 + 碎片字节数。SQLite 可能会重新组织页面，使之无空闲块或碎片，所有未使用字节归入未分配区域，所有单元紧凑地排列在页面尾部，称为“碎片整理”。

可变长度整数（varint）是一种针对 64 位二补数整数的静态霍夫曼编码，对较小正数使用更少空间。varint 长度在 1 到 9 字节之间。它由零个或多个高位为 1 的字节，后接一个高位为 0 的字节组成，或最多 9 个字节。前八个字节的低七位与第九字节的全部 8 位用于重建 64 位二补数整数。varint 按大端顺序：来自更早字节的位比来自更晚字节的位更重要。

1.7. 单元载荷溢出页面

当 B 树单元的载荷太大而无法完全存放在页面上时，多余部分会溢出到溢出页面。溢出页面组成单向链表。每个溢出页面的前四个字节为大端整数，表示链中下一个页面的页码，若为零则为链尾。第 5 字节到最后一个可用字节用于存储溢出内容。

1.8. 指针映射或 Ptrmap 页面

指针映射或 ptrmap 页面是插入数据库文件以提高auto_vacuum与incremental_vacuum模式效率的额外页面。数据库中的其他页面通常含有从父指向子页面的指针。例如，内部 B 树页面指向其子页面，溢出链中的链接按先后顺序指向。ptrmap 页面则包含相反方向的链接信息，从子指向父。

若数据库文件头中的偏移 52 所示最大的根 B 树页面值非零，则数据库必须包含 ptrmap 页面。若该值为零，则数据库不得包含 ptrmap 页面。

在使用 ptrmap 页面时，第一张 ptrmap 页面为页面 2。ptrmap 页面由一系列 5 字节条目组成。设 J 为能填入页面可用空间的 5 字节条目数（即 \(J = U/5\)）。首个 ptrmap 页面包含页面 3 到 \(J+2\) 的回指信息。第二张 ptrmap 页面位于页码 \(J+3\)，提供页面 \(J+4\) 到 \(2J+3\) 的回指信息。以此类推，覆盖整个数据库文件。

若数据库使用 ptrmap 页面，则前述计算中确定的所有页必须是 ptrmap 页面，其他页不得为 ptrmap 页面。唯一例外是当字节锁页面恰好落在某个 ptrmap 页面页码上时，该 ptrmap 页面会移动到下一页。

ptrmap 页面上的每个 5 字节条目提供紧随指针映射之后某页的回链信息。若页面 B 为 ptrmap 页面，则其第一个条目提供 B+1 页的信息，第二个条目提供 B+2 页，以此类推。

每个 5 字节 ptrmap 条目由 1 字节“页面类型”信息及 4 字节大端页码组成。共识别五种页面类型：

  1. B 树根页面。页码应为零。
  2. 空闲列表页面。页码应为零。
  3. 单元载荷溢出链的首页。页码为包含发生溢出的单元的 B 树页面。
  4. 溢出链中除首页以外的页面。页码为溢出链的前一页。
  5. 非根 B 树页面。页码为父页面。

在包含 ptrmap 页面的数据库文件中，所有 B 树根页面必须排在任意非根 B 树页面、单元溢出页面或空闲列表页面之前。此限制确保根页面在 auto_vacuum 或 incremental_vacuum 期间不会移动。自动清理逻辑无法更新 sqlite_schema 表的 root_page 字段，因此必须阻止根页面在自动清理时移动，以维护 sqlite_schema 表的完整性。CREATE TABLE、CREATE INDEX、DROP TABLE 和 DROP INDEX 操作会将根页面移动到文件开头。

2. 模式层

此前文本描述了 SQLite 文件格式的低层细节。B 树机制提供了一种强大高效的方式访问大型数据集。本节将描述低层 B 树层如何被用来实现高层 SQL 功能。

2.1. 记录格式

先前已将表 B 树叶页面的数据与索引 B 树页面的键描述为任意字节序列。前文提及一个键小于另一个键，但未定义“较小”的含义。本节将补充这一点。

表 B 树数据或索引 B 树键总是采用“记录格式”。记录格式定义了一系列值，对应表或索引的列。它指定列数、每列的数据类型以及每列的内容。

记录格式大量使用前文定义的可变长度整数（varint）表示 64 位有符号整数。

记录包含头部与主体，按此顺序。头部以单个 varint 开始，用以表示头部的总字节数。该 varint 值包含自身所占字节数。紧随其后的是一个或多个 varint，每列一个。这些附加 varint 称为“序列类型”编号，决定每列数据类型，具体如下表：

2.2. 记录排序顺序

索引 B 树中键的顺序由其所代表记录的排序决定。比较记录时逐列进行，从左到右查看列值。第一组不相等的列决定两个记录的相对顺序。各列的排序如下：

  1. NULL 值（序列类型 0）最先排序。
  2. 数字值（序列类型 1 到 9）在 NULL 之后，按数值顺序排序。
  3. 文本值（奇数序列类型 ≥13）在数字之后，按该列的排序函数确定的顺序排序。
  4. BLOB 值（偶数序列类型 ≥12）最后排序，按 memcmp() 的结果排序。

每列需要一个排序函数以便比较文本字段。SQLite 定义了三种内置排序函数：

BINARY内置 BINARY 排序按字节逐个比较字符串，使用标准 C 库的 memcmp()。
NOCASENOCASE 类似 BINARY，但在比较前将大写 ASCII 字符（'A' 至 'Z'）折叠为小写。仅折叠 ASCII 字符，NOCASE 不实现通用 Unicode 无大小写比较。
RTRIMRTRIM 类似 BINARY，但忽略末尾多余空格。换言之，只要两个字符串仅在末尾空格数量上不同，则被视为相等。

可通过sqlite3_create_collation()接口添加应用程序自定义排序函数。

所有字符串默认排序函数为 BINARY。在CREATE TABLE语句的列定义中可使用 COLLATE 子句指定替代排序函数。若列被建立索引，则默认使用在 CREATE TABLE 中指定的排序函数；也可通过CREATE INDEX语句中的 COLLATE 子句覆盖。

2.3. SQL 表的表示

数据库模式中的每个普通 SQL 表都由一个表 B 树在磁盘上表示。表 B 树中的每个条目对应 SQL 表的一行。SQL 表的rowid即表 B 树每个条目的 64 位有符号整数键。

每行内容通过将各列的值按顺序组合成记录格式的字节数组，再将该字节数组作为载荷存储在表 B 树的一个条目中。记录中值的顺序与 SQL 表定义中的列顺序一致。若 SQL 表包含INTEGER PRIMARY KEY列（即 rowid 的别名），则该列在记录中以 NULL 值出现。SQLite 引用 INTEGER PRIMARY KEY 时始终使用表 B 树键而非 NULL 值。

若某列的倾向为 REAL，并且该列包含可以无损转换为整数的值（即无小数且不超过整数范围），则该值在记录中可作为整数存储。SQLite 在从记录中提取时会将其转换回浮点数。

2.4. 无 ROWID 表的表示

若 SQL 表在 CREATE TABLE 语句末尾使用 "WITHOUT ROWID" 子句，则该表为WITHOUT ROWID 表，并采用不同的磁盘表示。WITHOUT ROWID 表使用索引 B 树而非表 B 树进行存储。每个 WITHOUT ROWID B 树条目的键由 PRIMARY KEY 的列值构成，后接所有剩余列。PRIMARY KEY 列按照声明顺序出现，其他列按 CREATE TABLE 中的顺序出现。

因此，WITHOUT ROWID 表的内容编码与普通 rowid 表相同，但列顺序调整为 PRIMARY KEY 列优先，且内容用作索引 B 树键而非表 B 树数据。 REAL 类型列的特殊编码规则在 WITHOUT ROWID 表中与 rowid 表一致。

2.4.1. WITHOUT ROWID 表主键中冗余列的抑制

若 WITHOUT ROWID 表的 PRIMARY KEY 中某列与其先前出现的列、排序顺序均相同，则后续重复的该列将被忽略。例如，以下 CREATE TABLE 语句都定义了相同的表，其磁盘表示完全一致：

CREATE TABLE t1(a,b,c,d,PRIMARY KEY(a,c)) WITHOUT ROWID;
CREATE TABLE t1(a,b,c,d,PRIMARY KEY(a,c,a,c)) WITHOUT ROWID;
CREATE TABLE t1(a,b,c,d,PRIMARY KEY(a,A,a,C)) WITHOUT ROWID;
CREATE TABLE t1(a,b,c,d,PRIMARY KEY(a,a,a,a,c)) WITHOUT ROWID;

上述第一个示例为首选定义。所有示例都会创建一个 WITHOUT ROWID 表，其主键列为 "a" 与 "c"（按此顺序），然后是数据列 "b" 与 "d"（亦按此顺序）。

2.5. SQL 索引的表示

每个 SQL 索引（无论是通过CREATE INDEX 显式声明，还是由 UNIQUE 或 PRIMARY KEY 约束隐式创建）对应数据库文件中的一棵索引 B 树。索引 B 树中的每个条目对应相关 SQL 表的一行。索引 B 树的键由被索引列的值组成，后接相关表行的键。对于普通表，行键为rowid；对于WITHOUT ROWID 表，行键为 PRIMARY KEY。由于每行键唯一，索引中的键亦唯一。

在普通索引中，表中的每行与该表关联的每个索引之间存在一对一映射。然而在部分索引中，仅包含满足 CREATE INDEX 语句 WHERE 条件的表行。表与索引 B 树中对应的行拥有相同的 rowid 或主键值，并且在所有索引列上含有相同值。

2.5.1. WITHOUT ROWID 次级索引中冗余列的抑制

在 WITHOUT ROWID 表的索引中，若 PRIMARY KEY 的某列也出现在索引中且排序顺序一致，则该索引列不会在索引记录的表键后缀中重复。例如，考虑以下 SQL：

CREATE TABLE ex25(a,b,c,d,e,PRIMARY KEY(d,c,a)) WITHOUT rowid;
CREATE INDEX ex25ce ON ex25(c,e);
CREATE INDEX ex25acde ON ex25(a,c,d,e);
CREATE INDEX ex25ae ON ex25(a COLLATE nocase,e);

ex25ce 索引中的每行记录包含列：c、e、d、a。前两列为被索引列 c 与 e，后续列为对应表行的主键。通常主键为 d、c、a，但由于 c 已在索引中出现，因此在键后缀中省略。

在索引列覆盖 PRIMARY KEY 所有列的极端情况中，索引仅包含被索引列。ex25acde 示例即说明：其索引条目仅包含 a、c、d、e（按此顺序）。

ex25ae 的每行包含五列：a、e、d、c、a。之所以重复 "a"，是因为首个 "a" 使用 nocase 排序，而第二个使用 binary。如果省略重复的 "a"，且表中存在两个或以上的行在 "e" 上相同但 "a" 仅大小写不同，则这些表行将对应到索引中的同一条目，从而破坏表与索引之间的逐一对应关系。

索引记录后缀中冗余列的抑制仅在 WITHOUT ROWID 表中发生。在普通 rowid 表中，即使INTEGER PRIMARY KEY列也被索引，索引记录仍始终以 rowid 结尾。

2.6. SQL 数据库模式的存储

数据库文件的第 1 页是一个表 B 树的根页面，保存一个名为“sqlite_schema”的特殊表。此 B 树称为“模式表”，因为它存储完整的数据库模式。sqlite_schema 表的结构如同通过以下 SQL 创建：

CREATE TABLE sqlite_schema(
  type text,
  name text,
  tbl_name text,
  rootpage integer,
  sql text
);

sqlite_schema 表为数据库模式中的每个表、索引、视图与触发器（统称“对象”）包含一行，但 sqlite_schema 表自身没有条目。除了应用与开发人员定义的对象外，sqlite_schema 表还包含内部模式对象的条目。

sqlite_schema.type 列的值为以下字符串之一：'table'、'index'、'view' 或 'trigger'，对应对象类型。“table”用于普通表与虚表。

sqlite_schema.name 列存放对象名称。表上的UNIQUE与PRIMARY KEY 约束会使 SQLite 创建名为 "sqlite_autoindex_TABLE_N" 的内部索引，其中 TABLE 为包含该约束的表名，N 为以 1 开始、每遇到一个约束递增的整数。在WITHOUT ROWID 表中，PRIMARY KEY 没有 sqlite_schema 条目，但 "sqlite_autoindex_TABLE_N" 名称仍为 PRIMARY KEY 预留，以便不会影响后续 UNIQUE 约束编号。无论 rowid 表还是 WITHOUT ROWID 表，"sqlite_autoindex_TABLE_N" 名称均不会分配给INTEGER PRIMARY KEY。

sqlite_schema.tbl_name 列存放对象所属的表或视图名称。对于表或视图，tbl_name 是 name 的副本。对于索引，tbl_name 为所索引的表名。对于触发器，tbl_name 记录触发器作用的表或视图名称。

sqlite_schema.rootpage 列存储表与索引的根 B 树页面的页码。对于定义视图、触发器与虚表的行，rootpage 为 0 或 NULL。

sqlite_schema.sql 列保存描述对象的 SQL 文本。该文本为CREATE TABLE、CREATE VIRTUAL TABLE、CREATE INDEX、CREATE VIEW或CREATE TRIGGER语句，若在数据库文件为主数据库且在数据库连接中执行，则会重新创建该对象。通常该文本为原始创建语句，在应用了以下规范化规则后的一份副本：

  • 语句开头的 CREATE、TABLE、VIEW、TRIGGER 与 INDEX 关键字转换为大写。
  • 若 TEMP 或 TEMPORARY 出现在初始 CREATE 关键字之后，则移除。
  • 对象名称前的任何数据库名称限定符会被移除。
  • 前导空格被移除。
  • 前两个关键字之后的所有空格统一为单个空格。

sqlite_schema.sql 列的文本为创建对象的原始 CREATE 语句的副本，但已按上述规则规范化且可能经ALTER TABLE语句修改。由UNIQUE或PRIMARY KEY 约束自动创建的内部索引的 sqlite_schema.sql 值为 NULL。

2.6.1. 模式表的替代名称

“sqlite_schema”名称并未出现在文件格式中，仅为数据库实现所使用的约定。由于历史与操作方面的考虑，该表有时也可通过以下别名引用：

  1. sqlite_master
  2. sqlite_temp_schema
  3. sqlite_temp_master

由于文件格式中不包含模式表的名称，因此若应用程序选择使用上述别名之一引用模式表，数据库文件含义不会改变。

2.6.2. 内部模式对象

除由应用或开发人员通过 CREATE 语句创建的表、索引、视图与触发器外，sqlite_schema 表还可能包含零个或多个由 SQLite 自身为内部用途创建的内部模式对象。内部模式对象的名称总以 "sqlite_" 开头，凡名称以 "sqlite_" 开头的表、索引、视图或触发器均属内部模式对象。SQLite 禁止应用创建名称以 "sqlite_" 开头的对象。

SQLite 使用的内部模式对象可能包括：

  • 名称形如 "sqlite_autoindex_TABLE_N" 的索引用于实现普通表上的UNIQUE与PRIMARY KEY 约束。

  • 名为 "sqlite_sequence" 的表用于跟踪使用AUTOINCREMENT 的表的最大历史INTEGER PRIMARY KEY。

  • 名称形如 "sqlite_statN"（N 为整数）的表存储由ANALYZE 命令收集并供查询优化器使用的数据库统计信息。

未来版本的 SQLite 可能在文件格式中引入新的以 "sqlite_" 开头的内部模式对象。

2.6.3. sqlite_sequence 表

sqlite_sequence 是一个内部表，用于帮助实现AUTOINCREMENT。每当创建包含 AUTOINCREMENT 整数主键的普通表时，系统会自动创建 sqlite_sequence 表。创建后，该表将永远存在于 sqlite_schema 表中，无法删除。 sqlite_sequence 表的模式为：

CREATE TABLE sqlite_sequence(name,seq);

sqlite_sequence 表为每个使用 AUTOINCREMENT 的普通表保留一行。sqlite_sequence.name 字段存储表名（与 sqlite_schema.name 相同），sqlite_sequence.seq 字段保存该表插入过的最大INTEGER PRIMARY KEY值。AUTOINCREMENT 表会保证新自动生成的整数主键大于 sqlite_sequence.seq 字段。当 sqlite_sequence.seq 已达到最大整数值（9223372036854775807）时，再尝试插入自动生成的主键将导致SQLITE_FULL错误。若需要，插入 AUTOINCREMENT 表时会自动更新 sqlite_sequence.seq。若删除 AUTOINCREMENT 表，则对应的 sqlite_sequence 行会自动删除。若 AUTOINCREMENT 表在更新时找不到对应的 sqlite_sequence 行，则会新建一条。若手动将 sqlite_sequence.seq 设置为非整数且随后尝试插入或更新 AUTOINCREMENT 表，则行为未定义。

应用程序代码可对 sqlite_sequence 表进行修改、插入、删除现有行。但应用无法在该表不存在时创建它。应用可以删除所有 sqlite_sequence 条目，但不能 DROP 该表。

2.6.4. sqlite_stat1 表

sqlite_stat1 是由ANALYZE 命令创建的内部表，用于保存查询优化器可用的有关表与索引的补充信息。应用可更新、删除、插入或 DROP sqlite_stat1 表，但不得创建或更改其模式。sqlite_stat1 表的模式如下：

CREATE TABLE sqlite_stat1(tbl,idx,stat);

通常每个索引对应一行，sqlite_stat1.idx 列即索引名称。sqlite_stat1.tbl 列为该索引所属的表名。stat 列为一个字符串，由若干整数与可选参数构成。第一整数为索引中行数的近似值。（索引行数等于表行数，除非是部分索引。）第二整数为索引中在第一列上具有相同值的行数近似值。第三整数为在前两列上具有相同值的行数。第 N 个整数（N>1）为索引中前 N-1 列相同的行数的估计平均值。若索引有 K 列，则 stat 列共有 K+1 个整数。若索引唯一，则最后一个整数为 1。

stat 列的整数列表可以后接参数，每个参数为不含空格的字符序列，且前置一个空格。未识别的参数将被静默忽略。

若存在 “unordered” 参数，查询优化器会认为该索引无序，从而不会用于范围查询或排序。

“sz=NNN” 参数（NNN 为一个或多个数字）表示该表或索引的平均行大小为 NNN 字节。SQLite 查询优化器可能会使用此信息选取更小的表或索引以减少磁盘 I/O。

索引的 sqlite_stat1.stat 字段若包含 “noskipscan” 标记，则该索引无法用于skip-scan 优化。

未来版本可能在 stat 列末尾添加新的文本标记。为兼容起见，末尾的未识别标记会被静默忽略。

若 sqlite_stat1.idx 列为 NULL，则 sqlite_stat1.stat 列包含一个整数，表示由 sqlite_stat1.tbl 标识的表的近似行数。若 sqlite_stat1.idx 与 sqlite_stat1.tbl 相同，则该表为WITHOUT ROWID 表，sqlite_stat1.stat 字段包含实现该 WITHOUT ROWID 表的索引 B 树信息。

2.6.5. sqlite_stat2 表

sqlite_stat2 表仅在 SQLite 使用 SQLITE_ENABLE_STAT2 编译并且版本号介于 3.6.18（2009-09-11）与 3.7.8（2011-09-19）之间时创建与使用。早于 3.6.18 或晚于 3.7.8 的 SQLite 不读取或写入 sqlite_stat2 表。该表提供有关索引键分布的额外信息。sqlite_stat2 表的模式为：

CREATE TABLE sqlite_stat2(tbl,idx,sampleno,sample);

sqlite_stat2 表的每行记录表示由该行描述的索引，其中 sqlite_stat2.idx 与 sqlite_stat2.tbl 指明索引。在 sqlite_stat2 表中，每个索引通常有 10 行。

对于 sqlite_stat2.sampleno 在 0 到 9（含）之间的条目，sample 字段保存索引最左侧键在索引空间均匀划分 10 个区段中间的样本值。设索引行数为 C，则样本行号为：row number = (i*C*2 + C)/20，其中 i 从 0 到 9 变动。概念上，索引空间被划分为 10 个等大桶，样本为各桶的中间行。

sqlite_stat2 的格式仅为历史参考。新版本 SQLite 不再支持 sqlite_stat2，若存在 sqlite_stat2 表则会被忽略。

2.6.6. sqlite_stat3 表

sqlite_stat3 表仅在 SQLite 使用SQLITE_ENABLE_STAT3或SQLITE_ENABLE_STAT4编译，且版本号为 3.7.9（2011-11-01）或以上时才使用。低于 3.7.9 的版本不会读写该表。若使用SQLITE_ENABLE_STAT4且版本号为 3.8.1（2013-10-17）或更高，则 sqlite_stat3 可能会读取但不会写入。该表提供有关索引键分布的额外信息，供查询优化器制定更优查询策略。sqlite_stat3 表的模式如下：

CREATE TABLE sqlite_stat3(tbl,idx,nEq,nLt,nDLt,sample);

每个索引通常在 sqlite_stat3 表中有多个条目。sqlite_stat3.sample 列存储与 sqlite_stat3.idx 及 sqlite_stat3.tbl 指定索引相关的最左列的值。sqlite_stat3.nEq 列保存该索引中与样本的最左列完全相等的条目近似数量。sqlite_stat3.nLt 表示索引中最左列小于样本的条目数量近似值。sqlite_stat3.nDLt 表示索引中小于样本的不同最左列值的近似数量。

每个索引可以有任意数量的 sqlite_stat3 条目。ANALYZE 命令通常生成包含 10 到 40 个样本、分布在键空间并具有较大 nEq 值的 sqlite_stat3 表。

在结构良好的 sqlite_stat3 表中，任一索引的样本顺序必须与其在索引中的顺序一致。换言之，若样本 S1 在索引 B 树中出现在 S2 之前，则 sqlite_stat3 表中 S1 的 rowid 必须小于 S2。

2.6.7. sqlite_stat4 表

sqlite_stat4 表仅在 SQLite 使用SQLITE_ENABLE_STAT4编译且版本号为 3.8.1（2013-10-17）或以上时创建与使用。低于 3.8.1 的 SQLite 不读写 sqlite_stat4 表。该表提供有关索引键或 WITHOUT ROWID 表主键键分布的附加信息，可协助查询优化器制定更优查询策略。sqlite_stat4 表的模式如下：

CREATE TABLE sqlite_stat4(tbl,idx,nEq,nLt,nDLt,sample);

每个索引的 sqlite_stat4 表中通常包含 10 到 40 条目，但没有硬性限制。各列含义如下：

tbl: sqlite_stat4.tbl 列保存拥有该索引的表名。
idx: sqlite_stat4.idx 列保存该条目描述的索引名，若为 WITHOUT ROWID 表，则为表名。
sample: sqlite_stat4.sample 列存储以记录格式编码的 BLOB，描述索引列及表行键（rowid 表为 rowid，WITHOUT ROWID 表为主键列）。sqlite_stat4.sample BLOB 对于 WITHOUT ROWID 表自身仅包含主键列。设该 BLOB 编码的列数为 N。对于普通 rowid 表的索引，N 为被索引列数加 1。对于 WITHOUT ROWID 表的索引，N 为被索引列数加主键列数。对于 WITHOUT ROWID 表自身，N 为主键列数。
nEq: sqlite_stat4.nEq 列保存 N 个整数，第 K 个整数为索引中与样本的前 K 列完全相等的条目近似数量。
nLt: sqlite_stat4.nLt 列保存 N 个整数，第 K 个整数为索引中前 K 列整体小于样本前 K 列的条目近似数量。
nDLt: sqlite_stat4.nDLt 列保存 N 个整数，第 K 个整数为索引中前 K 列在唯一性上小于样本前 K 列的条目近似数量。

sqlite_stat4 是 sqlite_stat3 的泛化。sqlite_stat3 仅提供有关索引最左列的信息，而 sqlite_stat4 提供关于索引所有列的信息。

每个索引的 sqlite_stat4 条目数量不限。ANALYZE 命令通常生成包含 10 到 40 个样本、分布于键空间并具有较大 nEq 值的 sqlite_stat4 表。

在结构良好的 sqlite_stat4 表中，任一索引的样本顺序必须与其在索引中的顺序一致。也就是说，若索引中 S1 位于 S2 之前，则 sqlite_stat4 表中 S1 的 rowid 必须小于 S2。

3. 回滚日志

回滚日志是与每个 SQLite 数据库文件关联的文件，用于在事务期间恢复数据库文件的初始状态。回滚日志始终位于与数据库文件相同的目录下，文件名为数据库文件名后追加 "-journal"。每个数据库只能关联一个回滚日志，从而一个数据库同时只能打开一个写事务。

在修改任何包含信息的数据库页面之前，该页面的原始未修改内容会被写入回滚日志。如果事务中断并需要回滚，回滚日志即可用于恢复数据库至原始状态。空闲列表叶页面不包含需要回滚恢复的信息，因此在修改前不会写入日志，以减少磁盘 I/O。

若事务因应用崩溃、操作系统崩溃、硬件断电等原因中止，则主数据库文件可能处于不一致状态。下一次 SQLite 尝试打开数据库时会检测到回滚日志的存在，并自动回放日志，以恢复数据库至未完成事务开始时的状态。

回滚日志仅在存在并具有有效头部时才认为有效。因此，一个事务可以通过以下三种方式提交：

  1. 删除回滚日志文件。
  2. 将回滚日志截断为零长度。
  3. 用无效头文本（例如全零）覆盖回滚日志头。

这三种提交方式分别对应journal_mode pragma的 DELETE、TRUNCATE 与 PERSIST 设置。

有效的回滚日志以如下格式的头部开始：

回滚日志头格式
偏移量大小说明
08头字符串：0xd9, 0xd5, 0x05, 0xf9, 0x20, 0xa1, 0x63, 0xd7
84“页面计数”——接下来段落中页面的数量，或 -1 表示一直到文件末尾
124校验和所用的随机 nonce
164数据库初始页面数
204写日志进程所假定的磁盘扇区大小
244此日志中的页面大小

回滚日志头会填充零以扩展到单个扇区大小（由偏移 20 处的扇区大小整数定义）。头部位于独立扇区中，以便在写入该扇区时若断电，可（希望）使头部之后的信息保持完好。

头部及零填充之后是零个或多个页面记录。每个页面记录保存事务开始前某数据库页面的内容。单个回滚日志中不能重复记录同一页面。要回滚未完成事务，只需从头到尾读取回滚日志并将其中的页面内容写回数据库相应位置即可。

设数据库页面大小（即日志头偏移 24 处的整数值）为 N，则页面记录格式如下：

回滚日志页面记录格式
偏移量大小说明
04数据库文件中的页面编号
4N事务开始前该页面的原始内容
N+44校验和

校验和为无符号 32 位整数，计算方法如下：

  1. 将校验和初始化为头部偏移 12 处的 nonce。
  2. 将索引 X 初始化为 \(N-200\)（N 为数据库页面大小）。
  3. 将页面中偏移 X 处的字节解释为 8 位无符号整数，并加到校验和上。
  4. X 减去 200。
  5. 若 X ≥ 0，返回步骤 3。

校验和用于防止电源故障导致的页面记录写入不完整问题。每次事务开始时都会使用不同的随机 nonce，以最大限度降低未写入扇区恰好包含先前日志中相同页面内容的风险。通过每次使用不同 nonce，即使磁盘上存在陈旧数据，也会因校验和不同而被检测出来。校验和仅对数据记录中的部分 32 位字进行采样，以提升性能——SQLite 3.0.0 规划阶段的设计研究显示完整校验的性能开销显著。

设日志头偏移 8 处的页面计数为 M。若 M 大于零，则在记录 M 个页面之后，日志可能用零填充至扇区大小的倍数，并插入另一个日志头。同一日志中的所有头必须具有相同的数据库页面大小与扇区大小。

若初始日志头的 M 为 -1，则剩余空间中可以容纳多少页面记录就写入多少，以填满可用空间。

4. 预写日志

自3.7.0 版（2010-07-21）起，SQLite 引入了一种称为“预写日志”或“WAL”的新事务控制机制。处于 WAL 模式的数据库要求所有连接使用 WAL。一个数据库在任意时刻只能使用回滚日志或 WAL，而不会同时使用两者。WAL 始终与数据库文件位于同一目录，文件名为数据库文件名后追加 "-wal"。

4.1. WAL 文件格式

WAL 文件由头部以及零个或多个“帧”组成。每帧记录数据库文件单个页面的修改内容。所有数据库更改通过写入 WAL 帧记录。当帧包含提交标记时，事务提交。单个 WAL 通常记录多个事务。周期性地，会将 WAL 内容回写数据库，称为“检查点”。

单个 WAL 文件可重复使用。这意味着 WAL 可先填满，然后执行检查点再由新的帧覆盖旧帧。WAL 总是从开始位置向文件末尾增长。附加在每帧上的校验和与计数用于判定哪些帧有效、哪些为先前检查点残留。

WAL 头部大小为 32 字节，由以下八个大端 32 位无符号整数组成：

WAL 头格式
偏移量大小说明
04魔数：0x377f0682 或 0x377f0683
44文件格式版本。当前为 3007000。
84数据库页面大小。例如：1024

紧随 WAL 头部的是零个或多个帧。每帧包含一个 24 字节帧头及随后一个等于页面大小的页面数据。帧头由六个大端 32 位无符号整数组成：

偏移量大小说明

帧被认定有效须满足以下条件：

  1. 帧头中的 salt-1 与 salt-2 与 WAL 头中的值一致。

  2. 帧头最后 8 字节中的校验和值与依次计算 WAL 头前 24 字节及包括当前帧在内所有帧的前 8 字节与内容所得校验和完全匹配。

4.2. 校验和算法

校验和通过将输入视为偶数个无符号 32 位整数 \(x(0)\) 到 \(x(N)\) 计算得出。若 WAL 头前 4 字节魔数为 0x377f0683，则整数为大端；若为 0x377f0682，则为小端。校验和在帧头中始终以大端格式存储，无论计算时采用何种字节序。

该算法仅适用于长度为 8 字节倍数的内容。也就是说，若输入为 \(x(0)\) 到 \(x(N)\)，则 N 必须为奇数。算法如下：

 
s0 = s1 = 0
for i from 0 to n-1 step 2:
   s0 += x(i) + s1;
   s1 += x(i+1) + s0;
endfor
# 结果存于 s0 与 s1

输出 s0 与 s1 均为加权校验和，使用反向的斐波那契权重。（权重最大者对应序列首项。）s1 跨越所有 32 位整数项，而 s0 则省略最后一项。

4.3. 检查点算法

在检查点期间，首先通过VFS的 xSync 方法将 WAL 刷新到持久存储。然后将 WAL 中的有效内容转移回数据库文件。最后再次通过 xSync 将数据库刷新到持久存储。xSync 操作作为写屏障——在 xSync 发起前提交的写入必须在接下来任何写入开始前完成。

检查点不一定要完成。可能仍有读取器使用包含在数据库文件中的旧事务数据。在这种情况下，将 WAL 中的新事务内容转移至数据库可能会破坏仍在使用旧事务的读取器。为避免此类情况，只有当所有读取器正在使用 WAL 中的最后一个事务时，检查点才会完整执行。

4.4. WAL 重置

完成检查点后，若没有其他连接正在使用 WAL，则后续写事务可以从头开始覆盖 WAL 文件，称为“重置 WAL”。在第一个新写事务开始时，WAL 头部的 salt-1 值会递增，salt-2 值会重新随机化。这些盐的变化使得已被检查点处理但尚未被覆盖的旧帧失效，并阻止它们再次被检查点纳入。

可选择在重置时截断 WAL 文件，但非必须。若不截断，性能通常略好，因为文件系统通常覆盖现有文件比扩展文件更快。

4.5. 读取器算法

要读取数据库中的某页（记作页码 P），读取器会先检查 WAL 是否包含 P。若包含，则最后一个有效且为提交帧或后面有提交帧的 P 页内容即为所读。如果 WAL 中没有满足条件的 P 页帧，则从数据库文件读取 P 页。

为开始读取事务，读取器记录 WAL 中包含的值帧数量作为 “mxFrame”。(更多细节) 读取器在后续所有读取操作中均使用此 mxFrame 值。虽然可以在 WAL 末尾追加新事务，但只要读取器坚持使用原始 mxFrame 并忽略之后追加的内容，就能看到来自同一时间点的数据库一致快照。这一技术允许多个并发读取器同时查看数据库不同版本的内容。

上述读取算法虽然正确，但由于针对页 P 的帧可能分布在 WAL 的任意位置，读取器需扫描整个 WAL 才能寻找 P 页帧。若 WAL 较大（常见为数 MB），此扫描会拖慢读取性能。为解决此问题，创建了名为 wal-index 的辅助数据结构来加速查找某页的帧。

4.6. WAL 索引格式

wal-index 本质上是共享内存，不过当前 VFS 实现使用内存映射文件以便跨操作系统移植。该映射文件与数据库位于同一目录，文件名为数据库名称后追加 "-shm"。由于 wal-index 属于共享内存，SQLite 不支持在多台机器上运行客户端的网络文件系统上启用journal_mode=WAL，因为所有客户端必须共享同一段内存。

wal-index 的目的是快速回答以下问题：

给定页码 P 与最大 WAL 帧索引 M，返回不超过 M 的页 P 的最大 WAL 帧索引；若不存在则返回 NULL。

上述 M 即在第 4.4 节中定义的 mxFrame 值，读取事务开始时会读取该值，并定义读取器将使用 WAL 中的最大帧。

wal-index 是瞬态结构。崩溃后会根据原始 WAL 文件重建。VFS 必须在最后一个连接关闭时截断或清零 wal-index 头。由于 wal-index 是瞬态的，它可以采用与架构相关的格式，无需跨平台兼容。因此，与数据库与 WAL 文件格式始终以大端存储不同，wal-index 中的多字节值以主机本机字节序存储。

本文关注的是数据库文件的持久状态，且由于 wal-index 是瞬态结构，因此不再详细说明其格式。有关 wal-index 详细格式，请参阅单独的WAL-index 文件格式文档。
