1. RBU 扩展

RBU 扩展是 SQLite 的一个附加组件，专为在网络边缘低功耗设备上使用大型 SQLite 数据库文件而设计。RBU 可用于两个独立的任务：

  • RBU 更新操作。一个 RBU 更新 是对一个或多个表进行大量插入、更新和删除操作的批量更新。
  • RBU 真空操作。一个 RBU 真空 会优化并重建整个数据库文件，其结果类似于 SQLite 原生的 VACUUM 命令。

RBU 的缩写代表“可恢复的批量更新”。

这两个 RBU 功能也可以通过 SQLite 内置的 SQL 命令来完成——通过一系列 INSERT、DELETE 和 UPDATE 命令在一个事务中执行 RBU 更新，而 RBU 真空则通过一个 VACUUM 命令完成。RBU 模块在这些更简单的方法之上提供了以下优势：

  1. RBU 可能更高效

    最有效的方式是按键序对 B 树（SQLite 用于在磁盘上存储每个表和索引的数据结构）进行修改。但如果一个 SQL 表具备一个或多个索引，每个索引的键序可能与主表以及其他辅助索引不同。因此，在执行一系列 INSERT、UPDATE 和 DELETE 语句时，通常无法按键序更新所有 B 树。RBU 更新流程通过先对主表进行一轮变更，然后再对每个索引进行独立的轮次，从而确保每个 B 树都以最优顺序更新。对于一个较大的数据库文件（无法完全驻留在操作系统磁盘缓存中的那种），这一过程能带来两个数量级的速度提升。

    RBU 真空操作所需的临时磁盘空间比 SQLite 的 VACUUM 更少，写入磁盘的数据也更少。一个 SQLite VACUUM 大约需要等于最终数据库文件大小两倍的临时磁盘空间。写入的数据总量约为最终数据库文件大小的三倍。相比之下，RBU 真空只需要大约等于最终数据库文件大小的临时磁盘空间，并且写入的数据总量为该大小的两倍。

    另一方面，RBU 真空使用的 CPU 会比普通 SQLite VACUUM 多——在一次测试中多达五倍。因此，在相同条件下，RBU 真空通常明显比 SQLite VACUUM 慢。

  2. RBU 可在后台运行

    正在进行的 RBU 操作（无论是更新还是真空）不会干扰对数据库文件的读取访问。

  3. RBU 以增量方式运行

    RBU 操作可以被暂停，然后在可能发生断电或系统重置之后恢复。对于 RBU 更新，在整个更新应用完之前，原始数据库内容对所有读取者仍然可见——即使更新曾被暂停然后恢复。

RBU 扩展默认未启用。要启用它，需要在编译 合并源文件 时使用 SQLITE_ENABLE_RBU 编译时选项。

2. RBU 更新

2.1. RBU 更新限制

RBU 更新受限于以下条件：

  • 更改必须仅包含 INSERT、UPDATE 和 DELETE 操作。不支持 CREATE 和 DROP 操作。

  • INSERT 语句不得使用默认值。

  • UPDATE 和 DELETE 语句必须通过 rowid 或非空的 PRIMARY KEY 值指定目标行。

  • UPDATE 语句不得修改 PRIMARY KEY 或 rowid 值。

  • RBU 更新不能应用于包含名为 "rbu_control" 列的任何表。

  • RBU 更新不会触发任何触发器。

  • RBU 更新不会检测或避免外键或 CHECK 约束违规。

  • 所有 RBU 更新都使用 "OR ROLLBACK" 约束处理机制。

  • 目标数据库不得处于 WAL 模式。

  • 目标数据库不得包含 表达式上的索引。 从 SQLite 3.30.0（2019-10-04）开始支持表达式索引。

  • 在应用 RBU 更新期间，不允许对目标数据库进行其他写操作。为防止这种情况，会对目标数据库保持一个读锁。

2.2. 准备 RBU 更新文件

所有将由 RBU 应用的更改都存储在一个名为 “RBU 数据库” 的独立 SQLite 数据库中。要被修改的数据库称为 “目标数据库”。

针对目标数据库中每个将被更新的表，在 RBU 数据库中都要创建一个对应的表。RBU 数据库表的模式不同于目标数据库，但按照 下面所述 从其推导。

RBU 数据库表为目标数据库中每个插入、更新或删除的行包含单个记录。如何填充 RBU 数据库表在 下一节 中描述。

2.2.1. RBU 数据库模式

针对目标数据库中的每个表，RBU 数据库中应该包含一个名为 "data<integer>_<target-table-name>" 的表，其中 <target-table-name> 是目标数据库中该表的名称，而 <integer> 可以是任意长度的数字序列（0-9）。RBU 数据库内的表按名称顺序（根据 BINARY 校对序列从小到大）被处理，因此定义 data_% 表名中 <integer> 部分的值会影响目标表的更新顺序。虽然在使用 RBU 更新某些类型的虚拟表时可以利用这一点，但通常没有必要在 <integer> 位置使用非空字符串，使用空字符串即可。

data_% 表必须拥有与目标表相同的所有列，并额外包含一个名为 "rbu_control" 的列。data_% 表不应拥有 PRIMARY KEY 或 UNIQUE 约束，但每一列都应与目标数据库中对应列的类型相同。rbu_control 列不应指定任何类型。例如，如果目标数据库包含：

CREATE TABLE t1(a INTEGER PRIMARY KEY, b TEXT, c UNIQUE);

那么 RBU 数据库应该包含：

CREATE TABLE data_t1(a INTEGER, b TEXT, c, rbu_control);

data_% 表中列的顺序无关紧要。

如果目标数据库表是虚拟表或没有 PRIMARY KEY 声明的表，则 data_% 表还必须包含一个名为 "rbu_rowid" 的列。rbu_rowid 列映射到表的 ROWID。例如，如果目标数据库包含以下任意一种情况：

CREATE VIRTUAL TABLE x1 USING fts3(a, b);
CREATE TABLE x1(a, b);

那么 RBU 数据库应该包含：

CREATE TABLE data_x1(a, b, rbu_rowid, rbu_control);

若虚拟表的 "rowid" 列不充当主键值，则无法使用 RBU 更新该虚拟表。

目标表的所有非隐藏列（即通过 "SELECT *" 匹配的所有列）都必须出现在输入表中。对于虚拟表，隐藏列是可选的——如果存在于输入表中，则 RBU 也会更新它们，否则不更新。例如，要写入具有隐藏 languageid 列的 fts4 表：

CREATE VIRTUAL TABLE ft1 USING fts4(a, b, languageid='langid');

下面任一输入表模式都可以使用：

CREATE TABLE data_ft1(a, b, langid, rbu_rowid, rbu_control);
CREATE TABLE data_ft1(a, b, rbu_rowid, rbu_control);

2.2.2. RBU 数据库内容

对于 RBU 更新中需要插入到目标数据库的每一行，对应的 data_% 表应该包含一条记录，其 "rbu_control" 列设置为整数值 0，其他列设置为表示要插入的新记录的值。

"rbu_control" 列也可以设置为整数值 2 来表示 INSERT。在这种情况下，新行会默默地替换具有相同主键值的现有行。这相当于一个 DELETE 然后用相同主键值的 INSERT。它不同于 SQL REPLACE 命令，因为 REPLACE 可能替换任何冲突的行（例如因 UNIQUE 约束或索引冲突的行），而这里仅替换主键冲突的行。

如果目标数据库表具有 INTEGER PRIMARY KEY，就无法在 IPK 列中插入 NULL 值。尝试这么做会导致 SQLITE_MISMATCH 错误。

对于 RBU 更新中要从目标数据库删除的每一行，对应的 data_% 表应该包含一条记录，其 "rbu_control" 列设置为整数值 1。要删除行的真实主键值应该存储在 data_% 表的相应列中。其他列中的值不会被使用。

对于 RBU 更新中要更新的每一行，对应的 data_% 表应包含一条记录，其 "rbu_control" 列设置为文本类型的值。该行中用于标识要更新行的实际主键值应存储在相应列中，并且所有要更新的列的新值也存储在相应列中。"rbu_control" 列中的文本值必须与目标数据库表的列数相同，并且只能包含 'x' 与 '.' 字符（或在某些特殊情况下为 'd' ——详见下文）。每个要更新的列对应的字符应设为 'x'；保持原样的列对应字符应设为 '.'。例如，在前述表格中，语句：

UPDATE t1 SET c = 'usa' WHERE a = 4;

由以下 data_t1 行表示：

INSERT INTO data_t1(a, b, c, rbu_control) VALUES(4, NULL, 'usa', '..x');

如果 RBU 被用来更新目标数据库中的大型 BLOB 值，比起在 RBU 数据库中存储一个完整的新值，更高效的方式可能是保存一个可以修改现有 BLOB 的补丁或增量。RBU 允许通过两种方式指定增量：

  • 采用 “fossil delta” 格式——这是 Fossil 源代码管理系统 用于 BLOB 增量的格式，或者
  • 采用 RBU 应用程序自定义的格式。

“fossil delta” 格式只能用于更新 BLOB 值。与其在 data_% 表中存储新的 BLOB，不如存储 fossil delta。并且在 rbu_control 字符串中对应要更新列的字符应设为 'f'。处理 'f' 更新时，RBU 会从磁盘加载原始 BLOB 数据，将 fossil delta 应用到该数据上，并将结果写回数据库文件。由 sqldiff --rbu 生成的 RBU 数据库会在节省空间的情况下尽可能使用 fossil delta。

要使用自定义增量格式，RBU 应用程序必须在开始处理更新前注册一个名为 "rbu_delta" 的用户定义 SQL 函数。rbu_delta() 将接收两个参数——目标表列中存储的原始值和 RBU 更新中提供的增量值。它应返回将增量应用到原始值之后的结果。要使用自定义增量函数，rbu_control 值中对应目标列的字符应设为 'd' 而不是 'x'。然后 RBU 不会直接使用 data_% 表中对应列存储的值更新目标表，而是调用用户定义的 SQL 函数 "rbu_delta()"，并将其结果存储到目标表列中。

例如，这一行：

INSERT INTO data_t1(a, b, c, rbu_control) VALUES(4, NULL, 'usa', '..d');

会导致 RBU 以类似于以下方式更新目标数据库表：

UPDATE t1 SET c = rbu_delta(c, 'usa') WHERE a = 4;

如果目标数据库表是虚拟表或没有 PRIMARY KEY 的表，rbu_control 值中不应该包含与 rbu_rowid 值对应的字符。例如，这一行：

INSERT INTO data_ft1(a, b, rbu_rowid, rbu_control) 
  VALUES(NULL, 'usa', 12, '.x');

会产生类似于：

UPDATE ft1 SET b = 'usa' WHERE rowid = 12;

data_% 表本身不应声明 PRIMARY KEY。然而，如果从每个 data_% 表中按 “rowid” 顺序读取行的大致顺序与按对应目标数据库表的 PRIMARY KEY 排序一致，RBU 的效率会更高。换句话说，在插入 data_% 表之前，应根据目标表的 PRIMARY KEY 字段对行进行排序。

2.2.3. 在 FTS3/4 表上使用 RBU

通常，FTS3 或 FTS4 表是虚拟表的一个示例，其 rowid 表现得像 PRIMARY KEY。因此，对于以下 FTS4 表：

CREATE VIRTUAL TABLE ft1 USING fts4(addr, text);
CREATE VIRTUAL TABLE ft2 USING fts4;             -- implicit "content" column

可以创建 data_% 表如下：

CREATE TABLE data_ft1 USING fts4(addr, text, rbu_rowid, rbu_control);
CREATE TABLE data_ft2 USING fts4(content, rbu_rowid, rbu_control);

并像对普通 SQLite 表那样填充，将其视为没有显式 PRIMARY KEY 列的表。

无内容 FTS4 表 也是类似处理，只是当尝试更新或删除行时，在应用更新时会报错。

外部内容 FTS4 表 也可以使用 RBU 更新。在这种情况下，用户需要将 RBU 数据库配置为对 FTS4 索引应用与底层内容表相同的 UPDATE、DELETE 和 INSERT 操作。对于所有外部内容 FTS4 表的更新，用户还需要确保先对 FTS4 索引应用 UPDATE 或 DELETE 操作，再对底层内容表应用（详见 FTS4 文档）。在 RBU 中，这是通过确保用于写入 FTS4 表的 data_% 表的名称在 BINARY 校对序列中排在用于更新底层内容表的 data_% 表之前来实现，以避免在 RBU 数据库中重复数据，可以用一个 SQL 视图代替其中一个 data_% 表。例如，对于目标数据库模式：

CREATE TABLE ccc(addr, text);
CREATE VIRTUAL TABLE ccc_fts USING fts4(addr, text, content=ccc);

下面的 RBU 数据库模式可以使用：

CREATE TABLE data_ccc(addr, text, rbu_rowid, rbu_control);
CREATE VIEW data0_ccc_fts AS SELECT * FROM data_ccc;

然后可以像往常一样填充 data_ccc 表，以写入目标数据库表 ccc 的更新。RBU 会从 data0_ccc_fts 视图中读取相同的更新，并将其应用到 FTS 表 ccc_fts。由于 “data0_ccc_fts” 的名称比 “data_ccc” 更小，FTS 表会先被更新，这是所需的顺序。

当底层内容表具有显式 INTEGER PRIMARY KEY 列时，情况会稍微复杂一些，因为存储于 rbu_control 列的文本值对于 FTS 索引和其底层内容表略有不同。对于底层内容表，rbu_control 文本值中必须包含对应显式 IPK 的字符，而对于具有隐式 rowid 的 FTS 表本身，则不应该包含。这虽然不太方便，但可以通过更复杂的视图解决，如下所示：

-- Target database schema
CREATE TABLE ddd(i INTEGER PRIMARY KEY, k TEXT);
CREATE VIRTUAL TABLE ddd_fts USING fts4(k, content=ddd);

-- RBU database schema
CREATE TABLE data_ccc(i, k, rbu_control);
CREATE VIEW data0_ccc_fts AS SELECT i AS rbu_rowid, k, CASE 
  WHEN rbu_control IN (0,1) THEN rbu_control ELSE substr(rbu_control, 2) END
FROM data_ccc;

上述 SQL 视图中的 substr() 函数会返回 rbu_control 参数的文本，其第一字符（对应列 "i" 的字符，对于 FTS 表并不需要）被移除。

2.2.4. 使用 sqldiff 自动生成 RBU 更新

从 SQLite 版本 3.9.0（2015-10-14）开始，sqldiff 工具能够生成表示两个具有相同架构的数据库之间差异的 RBU 数据库。例如，以下命令：

sqldiff --rbu t1.db t2.db

输出一个 SQL 脚本，用于创建一个 RBU 数据库，如果用它来更新数据库 t1.db，就可以将其内容修补为与数据库 t2.db 相同。

默认情况下，sqldiff 会尝试处理提供给它的两个数据库中的所有非虚拟表。如果某个表只出现在一个数据库中，或者两个数据库中某个表的模式略有不同，就会报错。若遇到问题，可以使用 "--table" 选项。

默认情况下，sqldiff 会忽略虚拟表。但可以显式地为一个具有像主键一样的 rowid 的虚拟表创建一个 RBU data_% 表，使用如下命令：

sqldiff --rbu --table <virtual-table-name> t1.db t2.db

不幸的是，尽管虚拟表默认被忽略，但它们为了在数据库中存储数据而创建的 底层数据库表 并不会被忽略，sqldiff 会将这些表也添加进任何 RBU 数据库。因此，尝试用 sqldiff 为包含一个或多个虚拟表的目标数据库生成 RBU 更新的用户，可能需要针对目标数据库中的每个表分别运行带 --table 选项的 sqldiff。

2.3. RBU 更新 C/C++ 编程

RBU 扩展接口允许应用程序将存储在 RBU 数据库中的 RBU 更新应用到现有的目标数据库。其过程如下：

  1. 使用 sqlite3rbu_open(T,A,S) 函数打开 RBU 句柄。

    T 参数是目标数据库文件的名称。A 参数是 RBU 数据库文件的名称。S 参数是用于保存在中断后恢复更新所需状态信息的 “状态数据库” 的名称。如果 S 参数为 NULL，则状态信息会存储在 RBU 数据库的多个表中，表名都以 "rbu_" 开头。

    sqlite3rbu_open(T,A,S) 函数会返回一个指向 “sqlite3rbu” 对象的指针，之后的接口会将其作为参数传入。

  2. 使用 sqlite3rbu_db(X)（其中参数 X 是 sqlite3rbu_open() 返回的 sqlite3rbu 指针）返回的数据库句柄注册任何需要的虚拟表模块。另外，如有必要，通过 sqlite3_create_function_v2() 注册 rbu_delta() SQL 函数。

  3. 对 sqlite3rbu 对象指针 X 调用 sqlite3rbu_step(X) 函数一次或多次。每次调用 sqlite3rbu_step() 都会执行一次 B 树操作，因此要完成整个更新可能需要进行数千次调用。当更新完全应用后，sqlite3rbu_step() 接口将返回 SQLITE_DONE。

  4. 调用 sqlite3rbu_close(X) 销毁 sqlite3rbu 对象指针。如果 sqlite3rbu_step(X) 已经调用足够多次以将更新完全应用到目标数据库，则 RBU 数据库被标记为已完全应用。否则，RBU 更新应用的状态将保存到状态数据库（如果 sqlite3rbu_open() 中状态数据库文件名为 NULL，则保存到 RBU 数据库）以便后续恢复。

如果在调用 sqlite3rbu_close() 时更新只部分应用到了目标数据库，状态信息会保存到状态数据库（如存在），否则保存到 RBU 数据库。这样后续的进程就能自动从中断处恢复 RBU 更新。如果状态信息存储在 RBU 数据库中，可以通过删除所有以 "rbu_" 开头的表来清除。

更多细节参见 头文件 sqlite3rbu.h 中的注释。

3. RBU 真空

3.1. RBU 真空限制

与 SQLite 的内置 VACUUM 命令相比，RBU 真空具有以下限制：

  • 它不能用于包含 表达式索引 的数据库。

  • 正在进行真空的数据库不得处于 WAL 模式。

3.2. RBU 真空 C/C++ 编程

本节概述并展示了将 RBU 真空集成到应用程序中的示例代码。更多细节参见 头文件 sqlite3rbu.h 中的注释。

RBU 真空应用程序都实现下述某种变体：

  1. 通过调用 sqlite3rbu_vacuum(T, S) 创建 RBU 句柄。

    参数 T 是要进行真空的数据库文件名。参数 S 是在真空操作被中断时 RBU 模块会保存状态的数据库名称。

    如果 sqlite3rbu_vacuum() 被调用时状态数据库 S 不存在，该数据库会被自动创建并填充用于保存 RBU 真空状态的单表 “rbu_state”。如果正在进行的 RBU 真空被中断，该表会填充状态数据。下次再用相同的 S 参数调用 sqlite3rbu_vacuum() 时，会检测到这些数据并尝试恢复被中断的真空操作。当 RBU 真空完成或遇到错误时，RBU 会自动删除 rbu_state 表的内容。在这种情况下，下一次调用 sqlite3rbu_vacuum() 会从头开始一个全新的真空操作。

    根据目标数据库名称确定 RBU 真空状态数据库名称是个好习惯。下面的示例代码使用 “<target>-vacuum”，其中 <target> 是正在真空的数据库名称。

  2. 将正在真空的数据库中索引所用的任何自定义排序序列注册到 sqlite3rbu_db() 返回的两个数据库句柄中。

  3. 对 RBU 句柄调用 sqlite3rbu_step()，直到 RBU 真空完成、发生错误或应用程序希望暂停 RBU 真空为止。

    每次调用 sqlite3rbu_step() 都会完成部分真空操作。根据数据库大小，单次真空可能需要调用 sqlite3rbu_step() 数千次。如果真空操作已完成，sqlite3rbu_step() 返回 SQLITE_DONE；如果还未完成但没有出现错误，则返回 SQLITE_OK；如果发生错误，则返回相应的 SQLite 错误码。如果发生错误，之后所有对 sqlite3rbu_step() 的调用都会立即返回相同的错误码。

  4. 最后，调用 sqlite3rbu_close() 关闭 RBU 句柄。如果应用程序在真空完成或发生错误前停止调用 sqlite3rbu_step()，真空的状态会保存到状态数据库，以便以后恢复。

    与 sqlite3rbu_step() 类似，如果真空操作已完成，sqlite3rbu_close() 返回 SQLITE_DONE；如果尚未完成但未出现错误，则返回 SQLITE_OK；如果发生错误，则返回相应的 SQLite 错误码。如果错误发生在 sqlite3rbu_step() 的某次调用中，sqlite3rbu_close() 会返回相同的错误码。

下面的示例代码展示了上述技巧。

** database zTarget. Return when either an error occurs, the RBU 
** vacuum is finished or when the application signals an interrupt
** (code not shown).
**
** If the RBU vacuum is completed successfully, return SQLITE_DONE.
** If an error occurs, return SQLite error code. Or, if the application
** signals an interrupt, suspend the RBU vacuum operation so that it
** may be resumed by a subsequent call to this function and return
** SQLITE_OK.
**
** This function uses the database named "<zTarget>-vacuum" for
** the state database, where <zTarget> is the name of the database 
** being vacuumed.
*/

  sqlite3rbu *pRbu;               /* RBU vacuum handle */

  zState = sqlite3_mprintf("%s-vacuum", zTarget);
  if( zState==0 ) return SQLITE_NOMEM;
  pRbu = sqlite3rbu_vacuum(zTarget, zState);
  sqlite3_free(zState);

  if( pRbu ){
    sqlite3 *dbTarget = sqlite3rbu_db(pRbu, 0);
    sqlite3 *dbState = sqlite3rbu_db(pRbu, 1);

    /* Any custom collation sequences used by the target database must
    ** be registered with both database handles here.  */

      if( <application has signaled interrupt> ) break;
    }
  }
  rc = sqlite3rbu_close(pRbu);

}
