1. 摘要

以下两种对象和八个方法构成 SQLite 接口的基本要素：

  • sqlite3 → 代表数据库连接对象。由 sqlite3_open() 创建， 并由 sqlite3_close() 销毁。

  • sqlite3_stmt → 代表预处理语句对象。由 sqlite3_prepare() 创建， 并由 sqlite3_finalize() 销毁。

  • sqlite3_open() → 打开一个新的或已有的 SQLite 数据库连接。 这是 sqlite3 对象的构造函数。

  • sqlite3_prepare() → 将 SQL 文本编译为字节码，该字节码用于查询或更新数据库。 这是 sqlite3_stmt 对象的构造函数。

  • sqlite3_bind() → 将应用程序数据存储到原始 SQL 的参数中。

  • sqlite3_step() → 将 sqlite3_stmt 推进到下一行结果或执行完成。

  • sqlite3_column() → 获取当前结果行中 sqlite3_stmt 的列值。

  • sqlite3_finalize() → 销毁 sqlite3_stmt。

  • sqlite3_close() → 销毁 sqlite3。

  • sqlite3_exec() → 一个包装函数，依次执行 sqlite3_prepare()、sqlite3_step()、 sqlite3_column() 和 sqlite3_finalize()， 用于一串一个或多个 SQL 语句。

2. 介绍

SQLite 拥有 225 多个 API。 然而，其中大多数 API 是可选的、极其专业的， 初学者可以忽略。 核心 API 小巧、简单、易学。 本文总结了核心 API。

另一份文档：SQLite C/C++ 接口， 提供了所有 SQLite C/C++ API 的详细规范。一旦读者 理解了 SQLite 的基本运行原理， 该文档 应当作为参考手册使用。 本文仅作介绍，并非 SQLite API 的完整或权威参考。

3. 核心对象与接口

SQL 数据库引擎的主要任务是评估 SQL 语句。 为完成这一任务，开发者需要两个对象：

  • 数据库连接 对象：sqlite3
  • 预处理语句 对象：sqlite3_stmt

严格来说，预处理语句 对象并非必需，因为 便捷封装接口 sqlite3_exec 或 sqlite3_get_table 可用，这些包装将 预处理语句 对象封装隐藏起来。 然而，理解 预处理语句 对象对于充分利用 SQLite 是必要的。

数据库连接 和 预处理语句 对象由下面列出的一小组 C/C++ 接口例程控制。

  • sqlite3_open()
  • sqlite3_prepare()
  • sqlite3_step()
  • sqlite3_column()
  • sqlite3_finalize()
  • sqlite3_close()

上述例程列表是概念性的，而非实际列表。 这些例程中很多存在多个版本。 例如，上面列出的是名为 sqlite3_open() 的单个例程， 但实际上有三个彼此略有不同的例程可完成相同任务： sqlite3_open()、sqlite3_open16() 和 sqlite3_open_v2()。 列表中提到的 sqlite3_column() 实际上并不存在。 列表中出现的“sqlite3_column()”是一个占位符， 代表一整个家族的例程，用于以多种数据类型提取列数据。

以下是核心接口功能的摘要：

  • sqlite3_open()

    此例程 打开一个 SQLite 数据库文件的连接，并返回一个 数据库连接 对象。这通常是应用第一次调用的 SQLite API， 且是大多数其他 SQLite API 的前置条件。许多 SQLite 接口都需要将 数据库连接 对象的指针作为首个参数，因此可视作 数据库连接 对象上的方法。 此例程是 数据库连接 对象的构造函数。

  • sqlite3_prepare()

    此例程 将 SQL 文本转换为 预处理语句 对象，并返回该对象的指针。 此接口需要此前调用 sqlite3_open() 所创建的 数据库连接 指针，以及包含要准备的 SQL 语句的文本字符串。 该 API 并不实际执行 SQL 语句，而只是为其求值做准备。

    把每条 SQL 语句看作一个小型计算机程序。 sqlite3_prepare() 的目的是将该程序编译为目标代码。 预处理语句 就是目标代码。 sqlite3_step() 接口随后运行目标代码以获取结果。

    新开发的应用应总是调用 sqlite3_prepare_v2()， 而非旧的 sqlite3_prepare()。 旧版的 sqlite3_prepare() 为了向后兼容保留了下来。 但是 sqlite3_prepare_v2() 提供了更好的接口。

  • sqlite3_step()

    此例程用于评估由 sqlite3_prepare() 所创建的 预处理语句。该语句会被执行至首行结果可用。 若要推进到第二行结果，再次调用 sqlite3_step()。 持续调用 sqlite3_step() 直至语句执行完成。 不返回结果的语句（例如 INSERT、UPDATE 或 DELETE）在一次 sqlite3_step() 调用中即可执行完成。

  • sqlite3_column()

    此例程从 sqlite3_step() 正在评估的 预处理语句 所产生的当前结果行中返回单列。 每当 sqlite3_step() 停在新的一行结果时， 此例程都可被多次调用以获取该行所有列的值。

    如上所述，SQLite API 中并没有真正的“sqlite3_column()”函数。 此处所谓“sqlite3_column()”是一个占位符， 代表返回不同数据类型的结果集值的整个函数家族。 该家族中还有返回结果大小（若是字符串或 BLOB）和返回结果集列数的例程。

    • sqlite3_column_blob()
    • sqlite3_column_bytes()
    • sqlite3_column_bytes16()
    • sqlite3_column_count()
    • sqlite3_column_double()
    • sqlite3_column_int()
    • sqlite3_column_int64()
    • sqlite3_column_text()
    • sqlite3_column_text16()
    • sqlite3_column_type()
    • sqlite3_column_value()

  • sqlite3_finalize()

    此例程销毁先前由 sqlite3_prepare() 调用创建的 预处理语句。每个预处理语句都必须通过 本例程销毁以避免内存泄漏。

  • sqlite3_close()

    此例程关闭先前由 sqlite3_open() 打开的 数据库连接。 所有关联的 预处理语句 在关闭连接前应先经 sqlite3_finalize() 处理。

4. 核心例程与对象的典型用法

应用通常在初始化阶段调用 sqlite3_open() 创建单个 数据库连接。 注意 sqlite3_open() 可用于打开现有数据库文件或创建并打开新数据库文件。 虽然许多应用只使用单个 数据库连接， 但应用完全可以多次调用 sqlite3_open()， 以打开多个 数据库连接——无论是对同一数据库还是不同数据库。 有时多线程应用会为每个线程创建独立的 数据库连接。 需要注意的是，单个 数据库连接 可以通过 ATTACH 命令访问多个数据库， 因此不必为每个数据库文件创建单独的连接。

许多应用在关闭时通过调用 sqlite3_close() 销毁它们的 数据库连接。 又比如，使用 SQLite 作为应用文件格式的应用可能 在响应“文件/打开”菜单操作时打开 数据库连接，并在响应“文件/关闭”菜单时销毁对应连接。

要执行一条 SQL 语句，应用按如下步骤操作：

  1. 使用 sqlite3_prepare() 创建 预处理语句。
  2. 调用 sqlite3_step() 一次或多次以评估该 预处理语句。
  3. 对于查询，在两次 sqlite3_step() 调用之间调用 sqlite3_column() 提取结果。
  4. 使用 sqlite3_finalize() 销毁 预处理语句。

如上所述，掌握这些即可有效使用 SQLite。 其余部分都是优化与细节。

5. 核心例程的便捷封装

sqlite3_exec() 接口是一个便捷包装函数， 一次函数调用即可完成上述四个步骤。 传入的回调函数用于处理结果集的每一行。 sqlite3_get_table() 也是一个便捷包装， 同样完成上述四个步骤。 与 sqlite3_exec() 不同的是，它将查询结果存储在堆内存中，而非调用回调。

需要意识到的是，sqlite3_exec() 或 sqlite3_get_table() 都不会执行核心例程无法完成的操作。 实际上，这些包装完全使用核心例程实现。

6. 参数绑定与重复使用预处理语句

在前述讨论中，默认每条 SQL 语句准备一次、评估一次、然后销毁。 然而 SQLite 允许对同一个 预处理语句 进行多次评估。实现方式如下：

  • sqlite3_reset()
  • sqlite3_bind()

在对 预处理语句 调用一或多次 sqlite3_step() 评估后， 可以调用 sqlite3_reset() 重置它以再次评估。 可以将 sqlite3_reset() 看作将 预处理语句 程序倒带回起点。 在已有的 预处理语句 上使用 sqlite3_reset()， 可避免重新创建新语句从而减少对 sqlite3_prepare() 的调用。 对于许多 SQL 语句来说，执行 sqlite3_prepare() 所需时间 等于或超过 sqlite3_step() 的执行时间。 因此，避免调用 sqlite3_prepare() 可显著提升性能。

通常不会真的多次评估完全相同的 SQL 语句。 更多时候是需评估相似语句。 例如，可能希望使用不同的值多次执行 INSERT 语句； 或者希望对同一查询使用不同 WHERE 子句中的键重复执行。 为此 SQLite 允许 SQL 语句包含 参数，这些参数在求值前会“绑定”到某个值。 这些值可在之后更改，同样的 预处理语句 可在新值下再次执行。

SQLite 允许在查询或数据修改语句（DQL 或 DML）中， 在字符串字面量、BLOB 字面量、数字常量或 NULL 所允许的位置使用 参数。 （参数不能用于列名或表名，也不能作为约束或默认值的值。即 DDL 中不可用） 参数可采用以下形式之一：

  • ?
  • ?NNN
  • :AAA
  • $AAA
  • @AAA

上述示例中，NNN 表示整数值， AAA 表示标识符。 参数初始值为 NULL。 在第一次调用 sqlite3_step() 之前， 或在 sqlite3_reset() 之后立即， 应用可以调用 sqlite3_bind() 接口将值绑定到参数。 每次调用 sqlite3_bind() 都会覆盖同一参数之前的绑定。

应用可以预先准备多个 SQL 语句，并在需要时评估它们。 未完成的 预处理语句 数量没有任意限制。 有些应用在启动时多次调用 sqlite3_prepare()， 以创建所有可能需要的 预处理语句。 其他应用则维护一个最近使用的 预处理语句 缓存，并在有可用语句时重用。 还有一种方法是只在循环内部重用 预处理语句。

7. SQLite 配置

SQLite 的默认配置对大多数应用非常适合。 但有时开发者希望微调设置，以争取更高性能或利用某些特殊功能。

sqlite3_config() 接口用于对 SQLite 执行全局、进程范围的配置修改。 调用该接口前不得创建任何 数据库连接。 sqlite3_config() 接口允许程序员执行如下操作：

  • 调整 SQLite 的内存分配方式，包括设置适用于安全关键实时嵌入式系统的替代内存分配器以及应用自定义的分配器。
  • 设置进程范围的错误日志。
  • 指定应用自定义的页面缓存。
  • 调整互斥锁的使用方式，使其适配不同的线程模型，或替换为应用自定义的互斥系统。

进程范围的配置完成并创建 数据库连接 后， 可通过调用 sqlite3_limit() 和 sqlite3_db_config() 对单个连接进行配置。

8. 扩展 SQLite

SQLite 提供可用于扩展其功能的接口。 这样的例程包括：

  • sqlite3_create_collation()
  • sqlite3_create_function()
  • sqlite3_create_module()
  • sqlite3_vfs_register()

sqlite3_create_collation() 接口用于创建新的 文本排序序列。 sqlite3_create_module() 接口用于注册新的 虚拟表 实现。 sqlite3_vfs_register() 接口用于创建新的 虚拟文件系统。

sqlite3_create_function() 接口用于创建新的 SQL 函数—— 可以是标量函数或聚合函数。新函数的实现通常会使用以下额外接口：

  • sqlite3_aggregate_context()
  • sqlite3_result()
  • sqlite3_user_data()
  • sqlite3_value()

所有 SQLite 内置的 SQL 函数均使用完全相同的接口创建。 参考 SQLite 源码，特别是 date.c 与 func.c 源文件了解示例。

共享库或 DLL 可以作为可加载扩展用于 SQLite。

9. 其他接口

本文仅提及最重要与最常用的 SQLite 接口。 SQLite 库包含许多其他实现实用功能的 API， 它们未在此处详述。 可在 完整函数列表 中找到构成 SQLite 应用程序接口的所有函数， 该列表位于 C/C++ 接口规范 中。 请参阅该文档以获取有关所有 SQLite 接口的完整且权威的信息。
