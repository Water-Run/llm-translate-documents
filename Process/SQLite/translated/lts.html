长期支持

开发者的意图是支持 SQLite 直到 2050 年。

在撰写本文时，2050 年仍然还有 25 年的距离。无人知道那段时间会发生什么，我们也不能绝对保证 SQLite 在那时仍然可用或有用。但我们可以承诺：我们会按照支持 SQLite 至 2050 年的目标来进行规划。这样的长期视角在重要方面影响我们的决策。

  • 跨平台代码 → SQLite 可以在任何拥有 8 位字节、二进制补码的 32 位和 64 位整数以及 C 编译器的平台上运行。它在所有当前流行的 CPU 和操作系统上都经过积极测试。 SQLite 代码和文件格式的极端可移植性将帮助其在未来平台上保持可用。

  • 稳定的跨平台数据库文件 → SQLite 数据库文件 在 32 位、64 位、大端和小端平台上逐位相同。您可以将 SQLite 数据库文件从一个系统复制到另一个系统，而无需转换数据库。此外，该文件格式文档完备且稳定。 今天创建的数据库文件，在未来几十年发布的 SQLite 版本中也能被读取和写入。

  • 航空级测试 → 每个机器码分支指令都在两个方向上进行测试。多次。在多个平台上并配合多个编译器。这有助于使代码在未来的迁移中保持健壮。高强度的测试也意味着新的开发者可以对 SQLite 做实验性增强，并且在所有旧有测试通过的前提下，可以合理确信这些增强不会破坏旧有功能。

  • 详尽的文档 → SQLite 拥有坦诚、面向开发者并且开源的文档。文档由程序员为程序员撰写。 （几个示例： [1] [2] [3] [4] [5] [6] [7] [8]） 详尽的文档让新开发者能非常迅速地掌握 SQLite。

  • 大量注释的源代码 → SQLite 源代码中超过 35% 是注释。不是模板式注释，而是解释变量和对象含义以及方法和过程意图的有用注释。 代码的设计旨在让新程序员容易理解，并能在数十年间保持可维护。

  • 灾难恢复规划 → SQLite 的每一个源代码历史字节都经过加密保护，并会自动复制到多个地理位置分散的服务器上，这些数据中心由不同公司拥有。 全球还有数千个额外的私有服务器克隆。 SQLite 的主要开发者分布在世界不同地区。 SQLite 可以在大陆级的灾难中幸存。

  • 老派作风 → 没有人能完全免疫潮流和时尚，但 SQLite 的开发者努力避免陷入最新的编程风格。我们的目标是编写出永不过时的代码，让尚未出生的程序员也能读取、理解并维护。

除了“支持” SQLite 到 2050 年之外，开发者还承诺会保持 SQLite 的 C 语言 API 和 磁盘格式 完全向后兼容。这意味着如今编写的应用程序应该可以链接并使用未来几十年发布的 SQLite 版本。

我们的目标是让您今天在 SQLite 中保存的内容，对您的孙辈而言，跟您一样易于访问。

2018-05-31 更新： 我们长期支持 SQLite 的目标显然已经引起了 美国国会图书馆 中保存史学家的注意，他们已将 SQLite 认定为 推荐的数字内容保存格式。
