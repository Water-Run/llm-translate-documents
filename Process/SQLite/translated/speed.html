数据库速度比较

注意：本文档非常非常老旧。它描述的是SQLite、MySQL和PostgreSQL这些过时版本之间的速度比较。

这里的数字已经失去了意义。此页面仅作为历史文献保留。

执行摘要

运行了一系列测试来衡量SQLite 2.7.6、PostgreSQL 7.1.3和MySQL 3.23.41之间的相对性能。以下是这些实验得出的总体结论：

  • SQLite 2.7.6在大多数常见操作上明显比RedHat 7.2上默认安装的PostgreSQL 7.1.3快很多（有时快10倍或20倍）。

  • SQLite 2.7.6在大多数常见操作上也常常比MySQL 3.23.41快（有时快两倍以上）。

  • SQLite在执行CREATE INDEX或DROP TABLE时没有其他数据库快。但这并不被视为问题，因为这些操作发生得不频繁。

  • 如果将多个操作组合到单个事务中，SQLite表现最佳。

这里展示的结果附带以下限制：

  • 这些测试并未尝试衡量多用户性能或涉及多个连接与子查询的复杂查询优化。

  • 这些测试是在一个相对较小（约14兆字节）的数据库上进行的。它们未衡量这些数据库引擎在更大问题上的扩展能力。

测试环境

用于这些测试的平台是一台1.6GHz的Athlon，配备1GB内存和一个IDE磁盘驱动器。操作系统为带有默认内核的RedHat Linux 7.2。

使用的PostgreSQL和MySQL服务器均为RedHat 7.2默认交付的版本。（PostgreSQL版本7.1.3，MySQL版本3.23.41。）并未对这些引擎进行调优。特别指出的是RedHat 7.2上默认的MySQL配置不支持事务。不需要支持事务为MySQL提供了很大的速度优势，但SQLite在大多数测试中仍能与之匹敌。

有人告诉我RedHat 7.3中的PostgreSQL默认配置过于保守（它设计用于只有8MB内存的机器），只要进行一些有经验的配置调优，PostgreSQL可以运行得快得多。Matt Sergeant报告他已经调优了自己的PostgreSQL安装并重新运行了下面展示的测试。他的结果显示PostgreSQL和MySQL运行速度相差不大。关于Matt的结果，请访问

过时的网址: http://www.sergeant.org/sqlite_vs_pgsync.html

SQLite的测试是在网站上所展示的相同配置中进行的，它使用了-O6优化并加上了-DNDEBUG=1开关以禁用SQLite代码中的许多“assert()”语句。-DNDEBUG=1编译选项大致使SQLite的速度翻倍。

所有测试均在其他方面处于静默状态的机器上进行。使用一个简单的Tcl脚本来生成并运行所有测试。该Tcl脚本的副本可以在SQLite源代码树中的tools/speedtest.tcl文件中找到。

报告的所有测试时间均表示墙钟时间（单位为秒）。SQLite的两组时间分别报告。第一个值是SQLite在默认配置下启用完全磁盘同步时的时间。开启同步时，SQLite在关键点调用fsync()系统调用（或等效操作），以确保关键数据确实已写入磁盘表面。如果操作系统在数据库更新过程中崩溃或计算机意外断电，同步对于保证数据库完整性是必要的。第二个时间是SQLite在关闭同步时的时间。关闭同步后，SQLite有时会快得多，但存在操作系统崩溃或意外断电可能损坏数据库的风险。一般而言，同步SQLite的时间用于与PostgreSQL（也是同步的）进行比较，而异步SQLite的时间则用于与异步的MySQL引擎进行比较。

测试1：1000个INSERT

CREATE TABLE t1(a INTEGER, b INTEGER, c VARCHAR(100));
INSERT INTO t1 VALUES(1,13153,'thirteen thousand one hundred fifty three');
INSERT INTO t1 VALUES(2,75560,'seventy five thousand five hundred sixty');
... 995 lines omitted
INSERT INTO t1 VALUES(998,66289,'sixty six thousand two hundred eighty nine');
INSERT INTO t1 VALUES(999,24322,'twenty four thousand three hundred twenty two');
INSERT INTO t1 VALUES(1000,94142,'ninety four thousand one hundred forty two');
PostgreSQL：   4.373
MySQL：   0.114
SQLite 2.7.6：   13.061
SQLite 2.7.6（无同步）：   0.223

由于没有中央服务器来协调访问，SQLite必须在每次事务之后关闭并重新打开数据库文件，从而使其缓存失效。在本测试中，每条SQL语句都是一个独立事务，因此数据库文件必须被打开和关闭1000次，并且缓存必须刷新。尽管如此，异步版本的SQLite仍然几乎和MySQL一样快。不过可以看到同步版本慢了很多。SQLite在每个同步事务之后调用fsync()，以确保所有数据在继续之前都安全地写入磁盘表面。在同步测试的大部分13秒中，SQLite都在等待磁盘I/O完成，处于空闲状态。

测试2：在一个事务中插入25000条记录

BEGIN;
CREATE TABLE t2(a INTEGER, b INTEGER, c VARCHAR(100));
INSERT INTO t2 VALUES(1,59672,'fifty nine thousand six hundred seventy two');
... 24997 lines omitted
INSERT INTO t2 VALUES(24999,89569,'eighty nine thousand five hundred sixty nine');
INSERT INTO t2 VALUES(25000,94666,'ninety four thousand six hundred sixty six');
COMMIT;
PostgreSQL：   4.900
MySQL：   2.184
SQLite 2.7.6：   0.914
SQLite 2.7.6（无同步）：   0.757

当所有INSERT都放在一个事务中时，SQLite不再需要在每条语句之间关闭和重新打开数据库或使缓存失效。它也直到最后才需要执行任何fsync()。在这种解放下，SQLite比PostgreSQL和MySQL都快得多。

测试3：向带索引的表插入25000条记录

BEGIN;
CREATE TABLE t3(a INTEGER, b INTEGER, c VARCHAR(100));
CREATE INDEX i3 ON t3(c);
... 24998 lines omitted
INSERT INTO t3 VALUES(24999,88509,'eighty eight thousand five hundred nine');
INSERT INTO t3 VALUES(25000,84791,'eighty four thousand seven hundred ninety one');
COMMIT;
PostgreSQL：   8.175
MySQL：   3.197
SQLite 2.7.6：   1.555
SQLite 2.7.6（无同步）：   1.402

曾有报道说SQLite在索引表上的表现不佳。这个测试最近被加入以驳斥那些传言。确实，SQLite在创建新索引项方面不如其他引擎快（见下面的测试6），但其整体速度依然更好。

测试4：在无索引的情况下执行100次SELECT

BEGIN;
SELECT count(*), avg(b) FROM t2 WHERE b>=0 AND b<1000;
SELECT count(*), avg(b) FROM t2 WHERE b>=100 AND b<1100;
... 96 lines omitted
SELECT count(*), avg(b) FROM t2 WHERE b>=9800 AND b<10800;
SELECT count(*), avg(b) FROM t2 WHERE b>=9900 AND b<10900;
COMMIT;
PostgreSQL：   3.629
MySQL：   2.760
SQLite 2.7.6：   2.494
SQLite 2.7.6（无同步）：   2.526

此测试在一个25000条记录的表上执行100次查询，没有索引，因此需要全表扫描。早期版本的SQLite在此测试中比PostgreSQL和MySQL慢，但最近的性能提升使它现在成为最快的一个。

测试5：基于字符串比较的100次SELECT

BEGIN;
SELECT count(*), avg(b) FROM t2 WHERE c LIKE '%one%';
SELECT count(*), avg(b) FROM t2 WHERE c LIKE '%two%';
... 96 lines omitted
SELECT count(*), avg(b) FROM t2 WHERE c LIKE '%ninety nine%';
SELECT count(*), avg(b) FROM t2 WHERE c LIKE '%one hundred%';
COMMIT;
PostgreSQL：   13.409
MySQL：   4.640
SQLite 2.7.6：   3.362
SQLite 2.7.6（无同步）：   3.372

这个测试依然是100次全表扫描，但采用字符串比较而非数值比较。SQLite在这里比PostgreSQL快了三倍以上，比MySQL快约30%。

测试6：创建索引

CREATE INDEX i2a ON t2(a);
CREATE INDEX i2b ON t2(b);
PostgreSQL：   0.381
MySQL：   0.318
SQLite 2.7.6：   0.777
SQLite 2.7.6（无同步）：   0.659

SQLite在创建新索引时较慢。虽然这不是大问题（因为新索引创建并不频繁），但确实是一个正在改进的方向。希望未来版本的SQLite在这方面表现更好。

测试7：有索引的5000次SELECT

SELECT count(*), avg(b) FROM t2 WHERE b>=0 AND b<100;
SELECT count(*), avg(b) FROM t2 WHERE b>=100 AND b<200;
SELECT count(*), avg(b) FROM t2 WHERE b>=200 AND b<300;
... 4994 lines omitted
SELECT count(*), avg(b) FROM t2 WHERE b>=499700 AND b<499800;
SELECT count(*), avg(b) FROM t2 WHERE b>=499800 AND b<499900;
SELECT count(*), avg(b) FROM t2 WHERE b>=499900 AND b<500000;
PostgreSQL：   4.614
MySQL：   1.270
SQLite 2.7.6：   1.121
SQLite 2.7.6（无同步）：   1.162

当拥有可以利用的索引时，三种数据库引擎都运行得更快。但SQLite仍然是最快的。

测试8：无索引的1000次UPDATE

BEGIN;
UPDATE t1 SET b=b*2 WHERE a>=0 AND a<10;
UPDATE t1 SET b=b*2 WHERE a>=10 AND a<20;
... 996 lines omitted
UPDATE t1 SET b=b*2 WHERE a>=9980 AND a<9990;
UPDATE t1 SET b=b*2 WHERE a>=9990 AND a<10000;
COMMIT;
PostgreSQL：   1.739
MySQL：   8.410
SQLite 2.7.6：   0.637
SQLite 2.7.6（无同步）：   0.638

在这个UPDATE测试中，MySQL始终比PostgreSQL和SQLite慢五倍甚至十倍。我也不清楚原因。MySQL通常是一个非常快的引擎。也许这个问题在后续版本中已被解决。

测试9：有索引的25000次UPDATE

BEGIN;
UPDATE t2 SET b=468026 WHERE a=1;
UPDATE t2 SET b=121928 WHERE a=2;
... 24996 lines omitted
UPDATE t2 SET b=35065 WHERE a=24999;
UPDATE t2 SET b=347393 WHERE a=25000;
COMMIT;
PostgreSQL：   18.797
MySQL：   8.134
SQLite 2.7.6：   3.520
SQLite 2.7.6（无同步）：   3.104

直到2.7.0版本，SQLite在这个测试中的速度与MySQL差不多。但最近对SQLite的优化使UPDATE速度翻倍。

测试10：带索引的25000条文本UPDATE

BEGIN;
UPDATE t2 SET c='one hundred forty eight thousand three hundred eighty two' WHERE a=1;
UPDATE t2 SET c='three hundred sixty six thousand five hundred two' WHERE a=2;
... 24996 lines omitted
UPDATE t2 SET c='three hundred eighty three thousand ninety nine' WHERE a=24999;
UPDATE t2 SET c='two hundred fifty six thousand eight hundred thirty' WHERE a=25000;
COMMIT;
PostgreSQL：   48.133
MySQL：   6.982
SQLite 2.7.6：   2.408
SQLite 2.7.6（无同步）：   1.725

再次强调，SQLite 2.7.0版本在这个测试中的速度与MySQL相近。但现在2.7.6版本比MySQL快两倍多，比PostgreSQL快二十倍以上。

为了公平起见，PostgreSQL在这个测试中开始出现频繁交换。一个有经验的管理员或许可以通过微调服务器使PostgreSQL在这里运行得快很多。

测试11：来自SELECT的INSERT

BEGIN;
INSERT INTO t1 SELECT b,a,c FROM t2;
INSERT INTO t2 SELECT b,a,c FROM t1;
COMMIT;
PostgreSQL：   61.364
MySQL：   1.537
SQLite 2.7.6：   2.787
SQLite 2.7.6（无同步）：   1.599

异步SQLite在这个测试中仅比MySQL稍微慢一点。（MySQL似乎在INSERT...SELECT语句方面特别擅长。）PostgreSQL引擎仍在频繁交换——它花费的大部分61秒都在等待磁盘I/O。

测试12：无索引的DELETE

DELETE FROM t2 WHERE c LIKE '%fifty%';
PostgreSQL：   1.509
MySQL：   0.975
SQLite 2.7.6：   4.004
SQLite 2.7.6（无同步）：   0.560

在这个测试中，同步版本的SQLite是最慢的，但异步版本是最快的。区别在于执行fsync()所需的额外时间。

测试13：带索引的DELETE

DELETE FROM t2 WHERE a>10 AND a<20000;
PostgreSQL：   1.316
MySQL：   2.262
SQLite 2.7.6：   2.068
SQLite 2.7.6（无同步）：   0.752

这个测试很重要，因为它是少数几个PostgreSQL比MySQL更快的测试之一。然而，异步SQLite却比其他两个都快。

测试14：大DELETE之后的大INSERT

INSERT INTO t2 SELECT * FROM t1;
PostgreSQL：   13.168
MySQL：   1.815
SQLite 2.7.6：   3.210
SQLite 2.7.6（无同步）：   1.485

某些旧版本的SQLite（2.4.0之前）在一系列DELETE之后再INSERT新的数据时会表现出性能下降。正如这个测试所示，这个问题现在已被解决。

测试15：大DELETE之后的许多小INSERT

BEGIN;
DELETE FROM t1;
INSERT INTO t1 VALUES(1,10719,'ten thousand seven hundred nineteen');
... 11997 lines omitted
INSERT INTO t1 VALUES(11999,72836,'seventy two thousand eight hundred thirty six');
INSERT INTO t1 VALUES(12000,64231,'sixty four thousand two hundred thirty one');
COMMIT;
PostgreSQL：   4.556
MySQL：   1.704
SQLite 2.7.6：   0.618
SQLite 2.7.6（无同步）：   0.406

SQLite在事务中执行INSERT非常擅长，这大概解释了为什么它在这个测试中比其他数据库快那么多。

测试16：DROP TABLE

DROP TABLE t1;
DROP TABLE t2;
DROP TABLE t3;
PostgreSQL：   0.135
MySQL：   0.015
SQLite 2.7.6：   0.939
SQLite 2.7.6（无同步）：   0.254

SQLite在删除表方面比其他数据库慢。这很可能是因为当SQLite删除一个表时，它必须遍历并清除数据库文件中与该表相关的记录。MySQL和PostgreSQL则使用单独的文件来表示每张表，因此可以通过删除文件来快速删除表。

另一方面，删除表并不是一个很常见的操作，因此如果SQLite稍微慢一些，也不被认为是大问题。
