1. 概述

SQLite 实现原子提交与回滚的默认方式是回滚日志。从3.7.0 版本（2010-07-21）起，新增了一个“预写式日志”选项（下文简称“WAL”）。

相比使用回滚日志，WAL 有优点也有缺点。优点包括：

  1. 在大多数情况下，WAL 的速度明显更快。
  2. WAL 提供更高的并发性，因为读操作不会阻塞写入，写入也不会阻塞读取。读写可以同时进行。
  3. 使用 WAL 时磁盘 I/O 操作趋向于更加顺序化。
  4. WAL 使用的 fsync() 操作更少，因此在 fsync() 系统调用有缺陷的系统中不易出问题。

但也存在缺点：

  1. 所有使用同一数据库的进程必须运行在同一台宿主机上；WAL 无法在网络文件系统上工作。这是因为 WAL 需要所有进程共享少量内存，而不同主机的进程显然无法互相共享内存。
  2. 包含多个ATTACH数据库的事务对每个数据库而言是原子的，但并非对所有数据库作为一个整体原子。
  3. 进入 WAL 模式后，无法更改page_size，无论是在空数据库中、通过VACUUM，还是通过备份 API恢复。必须处于回滚日志模式下才能更改页大小。
  4. 无法打开只读 WAL 数据库。打开进程必须对数据库关联的"-shm"wal-index共享内存文件（如果存在）具有写权限；如果"-shm"文件不存在，则需要对包含数据库文件的目录具有写权限。 从3.22.0 版本（2018-01-22）开始，如果"-shm"和"-wal"文件已存在或可以创建，或者数据库是不可变的，就可以打开只读的 WAL 模式数据库文件。
  5. 在主要以读取为主、很少写入的应用中，WAL 的性能可能稍微慢一些（大约慢 1% 到 2%）。
  6. 每个数据库都会有额外的准持久"-wal"文件和"-shm"共享内存文件，使 SQLite 作为应用文件格式时不那么吸引人。
  7. 存在额外的检查点操作，虽然默认自动触发，但应用开发者仍需注意。
  8. WAL 最适合较小的事务。WAL 对非常大的事务支持不佳。对于超过约 100 MB 的事务，传统的回滚日志模式可能更快。对于超过 1 GB 的事务，WAL 模式可能会因 I/O 或磁盘满而失败。建议对超过几十 MB 的事务使用回滚日志模式。 从3.11.0 版本（2016-02-15）起，WAL 模式对大事务的效率与回滚模式相当。

2. WAL 工作原理

传统的回滚日志通过将原始未修改的数据库内容写入单独的回滚日志文件，然后将更改直接写入数据库文件来工作。当发生崩溃或ROLLBACK时，回滚日志中原始内容会被重新写回数据库文件，从而将数据库恢复到原始状态。COMMIT发生在回滚日志被删除时。

WAL 的方式正好相反。数据库文件中保留原始内容，更改则追加到单独的 WAL 文件。COMMIT发生在一个表示提交的特殊记录被追加到 WAL 中时。因此，提交可以在无需写入原始数据库的情况下完成，这使得读取者可以继续从未更改的数据库中读取，而写入则同时提交到 WAL。多个事务可以追加到同一个 WAL 文件的末尾。

2.1. 检查点

当然，最终希望将 WAL 文件中追加的所有事务转移回原始数据库。将 WAL 中的事务回写到数据库称为“检查点”。

另一种理解回滚日志与预写式日志区别的方式是：在回滚日志方法中，有两个基本操作，读取与写入；而在预写式日志中，基本操作增加为三个：读取、写入以及检查点。

默认情况下，当 WAL 文件达到 1000 页时，SQLite 会自动执行检查点。（可以通过SQLITE_DEFAULT_WAL_AUTOCHECKPOINT编译时选项指定一个不同默认值。）使用 WAL 的应用无需做任何事情即可触发这些检查点。但如果需要，应用可以调整自动检查点的阈值，也可以关闭自动检查点，并在空闲时刻或单独线程/进程中执行检查点。

2.2. 并发

当对 WAL 模式数据库开始一次读取操作时，SQLite 会先记录 WAL 中最后一个有效提交记录的位置。称该点为“终点标记”。由于 WAL 可能在多个读取者连接数据库时不断增长并添加新的提交记录，每个读取者理论上都可以有自己的终点标记。但对于任何特定读取者而言，终点标记在事务期间保持不变，确保单个读取事务只能看到一个时间点的数据库内容。

当读取者需要某个页时，它首先检查 WAL，看该页是否出现在其中；若出现，则取其在终点标记之前的最新副本。如果在终点标记之前 WAL 中没有该页的副本，则从原始数据库文件读取。读取者可能存在于不同进程，为避免每个读取者都必须扫描整个 WAL（WAL 文件可能增长至数 MB，取决于检查点频率），维护了一个称为“wal-index”的数据结构在共享内存中，帮助读取者快速定位 WAL 中的数据并减少 I/O。wal-index 极大提升了读取性能，但由于使用共享内存，因此所有读取者必须在同一台机器上，这也是 WAL 无法在网络文件系统上运行的原因。

写入者仅将新内容追加到 WAL 文件末尾。由于写入者不会干扰读取者的行为，读写可以同时进行。但由于只有一个 WAL 文件，因此同一时间只能有一个写入者。

检查点操作将 WAL 中的内容转移回原始数据库文件。检查点可以与读取操作并发运行，但当它遇到 WAL 中某一页、该页已经超过任何当前读取者的终点标记时，必须停止。检查点必须在该点停止，否则可能覆盖读取者正在使用的数据库部分。检查点会在 wal-index 中记住进度，并在下一次调用时从离开的地方继续将内容从 WAL 转移到数据库。

因此，长时间运行的读取事务可能会阻止检查点的进度。但每个读取事务终究会结束，检查点最终会继续进行。

每当发生写入操作时，写入者会检查检查点的进度；如果整个 WAL 已经转移到数据库并完成同步，且没有读取者在使用 WAL，则写入者会将 WAL 倒回到起始位置，并从头开始写入新事务。该机制防止 WAL 文件无限增长。

2.3. 性能考虑

写事务非常快，因为只需写一次内容（相比回滚日志事务要写两次），并且写入是顺序的。此外，只要应用愿意在断电或硬重启后牺牲持久性，就无需将内容同步到磁盘。（如果设置了PRAGMA synchronous 为 FULL，则每次事务提交时会同步 WAL；如果设置为 NORMAL，则会略过此同步。）

另一方面，随着 WAL 文件增大，读取性能会下降，因为每个读取者必须检查 WAL 文件，而检查时间与 WAL 文件大小成正比。wal-index 虽然提升了查找速度，但随着 WAL 文件增长，性能仍会下降。因此，为保持良好的读取性能，需要通过定期运行检查点来控制 WAL 文件大小。

为了避免在断电或硬重启后发生数据库损坏，检查点确实需要执行同步操作。必须在从 WAL 将内容移动到数据库之前同步 WAL，并在重置 WAL 之前同步数据库文件。检查点还需要更多的寻址操作。检查点尽量以顺序方式向数据库写入页面（页面按升序从 WAL 转移到数据库），但仍常常会在页面写入之间夹杂许多寻址。这些因素使得检查点比写事务慢。

默认策略是允许连续写事务一直增长 WAL 直到约 1000 页大小，然后对每次随后的 COMMIT 运行一次检查点，直到 WAL 被重置为小于 1000 页。默认情况下，当某个 COMMIT 使 WAL 超过设定大小限制时，执行该 COMMIT 的线程会自动运行检查点。这使得大多数 COMMIT 非常快，但偶尔（即触发检查点的 COMMIT）会比较慢。如果这种表现不理想，应用可以禁用自动检查点，并在独立线程或进程中运行周期性检查点。（完成该操作的命令与接口见下文所示。）

请注意，当将PRAGMA synchronous 设置为 NORMAL 时，只有检查点操作会发出 I/O 屏障或同步操作（在 Unix 上是 fsync()，在 Windows 上是 FlushFileBuffers()）。因此，如果应用在另一线程或进程中运行检查点，执行查询和更新的主线程或主进程就永远不会在同步操作上阻塞。这有助于防止在繁忙磁盘上运行的应用中出现“卡顿”。该配置的缺点是事务不再持久，可能在断电或硬重启后回滚。

还要注意，平均读取性能与平均写入性能之间存在权衡。为了最大化读取性能，需要保持 WAL 尽可能小，频繁运行检查点，甚至在每次 COMMIT 时都运行。为了最大化写入性能，需要将每次检查点的成本摊薄到尽可能多的写入操作，即减少检查点频率、让 WAL 在每次检查点前尽可能大。因此，检查点运行的频率会根据不同应用的读写性能要求而异。默认策略是在 WAL 达到 1000 页时运行检查点，这个策略在工作站的测试应用中效果不错，但其他平台或工作负载可能更适合其他策略。

3. 启用与配置 WAL 模式

SQLite 数据库连接默认使用journal_mode=DELETE。要切换到 WAL 模式，请使用以下 pragma：

PRAGMA journal_mode=WAL;

journal_mode pragma 返回一个字符串，表示新的日志模式。成功时，该 pragma 会返回字符串"wal"。如果无法完成切换（例如VFS不支持所需的共享内存原语），则日志模式不会改变，返回的字符串将是先前的日志模式（例如"delete"）。

3.1. 自动检查点

默认情况下，当一个COMMIT使 WAL 文件达到 1000 页或更多时，或当数据库文件的最后一个连接关闭时，SQLite 会自动执行检查点。默认配置旨在适用于大多数应用。但需要更多控制的程序可以使用wal_checkpoint pragma或调用sqlite3_wal_checkpoint() C 接口强制执行检查点。可以使用wal_autocheckpoint pragma或调用sqlite3_wal_autocheckpoint() C 接口更改自动检查点阈值或完全禁用自动检查点。程序也可以使用sqlite3_wal_hook()注册回调，在任何事务提交到 WAL 时调用。该回调可根据需要调用sqlite3_wal_checkpoint()或sqlite3_wal_checkpoint_v2()。（自动检查点机制是围绕sqlite3_wal_hook()的简单封装。）

3.2. 应用发起的检查点

应用可以通过任何可写数据库连接调用sqlite3_wal_checkpoint()或sqlite3_wal_checkpoint_v2()来启动检查点。检查点有三个子类型，按激进程度依次为 PASSIVE、FULL 和 RESTART。默认检查点样式为 PASSIVE，它会在不干扰其他数据库连接的前提下尽最大努力完成，如果有并发读取或写入，它可能无法运行完毕。通过sqlite3_wal_checkpoint()和自动检查点机制启动的所有检查点均为 PASSIVE。FULL 和 RESTART 检查点则更积极地尝试执行完毕，仅能通过调用sqlite3_wal_checkpoint_v2()启动。关于 FULL 和 RESET 检查点的更多信息，请参阅sqlite3_wal_checkpoint_v2()文档。

3.3. WAL 模式的持久性

与其他日志模式不同，PRAGMA journal_mode=WAL是持久的。如果一个进程设置了 WAL 模式，然后关闭并重新打开数据库，数据库仍然会保持 WAL 模式。相比之下，如果进程设置了（例如）PRAGMA journal_mode=TRUNCATE，然后关闭并重新打开数据库，数据库将恢复到默认的 DELETE 回滚模式，而非此前的 TRUNCATE 设置。

WAL 模式的持久性意味着可以在不修改应用的前提下将其转换为使用 WAL 模式。只需对数据库文件运行"PRAGMA journal_mode=WAL;"，无论是使用命令行 shell还是其他工具，然后重新启动应用即可。

如果某一连接设置了 WAL 日志模式，同一数据库文件上的所有连接都会切换到 WAL 模式。

4. WAL 文件

只要某个数据库连接以 WAL 模式打开数据库，SQLite 就会维护一个名为“预写式日志”或“WAL 文件”的额外日志文件。该文件在磁盘上的名称通常是数据库文件名再加上"-wal"后缀；如果 SQLite 编译时启用了SQLITE_ENABLE_8_3_NAMES，命名规则可能会有所不同。

只要有任一数据库连接保持打开，WAL 文件就存在。通常，在最后一个连接关闭后，WAL 文件会自动删除。然而，如果最后一个打开数据库的进程在未正常关闭连接的情况下退出，或使用了SQLITE_FCNTL_PERSIST_WAL文件控制，那么在关闭所有连接后 WAL 文件可能仍然保留在磁盘上。WAL 文件是数据库的持久状态的一部分，在复制或移动数据库时应一并保留。如果数据库文件与其 WAL 文件分离，那么以前提交到数据库的事务可能会丢失，甚至可能导致数据库文件损坏。删除 WAL 文件的唯一安全方式是使用某个sqlite3_open()接口打开数据库文件，然后立即调用sqlite3_close()关闭数据库。

WAL 文件格式有精确定义，且跨平台一致。

5. 只读数据库

早期版本的 SQLite 无法读取只读的 WAL 模式数据库。换言之，读取 WAL 模式数据库时需要写权限。该限制从3.22.0 版本（2018-01-22）起放宽。

在较新版本的 SQLite 中，只要满足下列任一条件，位于只读介质上的 WAL 模式数据库或不具备写权限的 WAL 模式数据库仍可被读取：

  1. 已经存在且可读的"-shm"和"-wal"文件。
  2. 包含数据库的目录具有写权限，能够创建"-shm"和"-wal"文件。
  3. 使用immutable 查询参数打开数据库连接。

即便可以打开只读的 WAL 模式数据库，仍建议在将 SQLite 数据库刻录到只读介质前，将其转换为PRAGMA journal_mode=DELETE。

6. 避免过大的 WAL 文件

在正常情况下，新内容会追加到 WAL 文件，直到 WAL 累积约 1000 页（约 4MB），此时会自动运行检查点并回收 WAL 文件。检查点通常不会截断 WAL 文件（除非设置了journal_size_limit pragma）。相反，SQLite 会从头开始覆盖 WAL 文件。这么做是因为覆盖已有文件通常比追加更快。当最后一个连接关闭数据库时，该连接会执行最后一次检查点，并删除 WAL 及其关联的共享内存文件，以清理磁盘。

因此，在绝大多数情况下，应用无需担心 WAL 文件；SQLite 会自动处理。但仍有可能使 SQLite 进入 WAL 文件无限增长的状态，导致磁盘空间浪费和查询变慢。以下列举了一些可能出现的问题及其避免方法。

  • 禁用自动检查点机制。 在默认配置下，只要 WAL 文件超过 1000 页，任何事务完成时都会执行检查点。然而，存在编译时和运行时选项可以禁用或延迟此自动检查点。如果应用禁用了自动检查点，就没有机制阻止 WAL 文件不断增长。

  • 检查点饥饿。 检查点只有在没有其他数据库连接正在使用 WAL 文件的情况下才能执行完毕并重置 WAL。如果另一个连接打开了读事务，那么检查点无法重置 WAL，因为这可能会在读取者使用期间删除数据。检查点会尽可能多地工作而不干扰读取者，但无法完成。下一次写事务发生时，检查点会从停止的位置继续。此过程会重复，直到某个检查点能够完成。

    然而，如果数据库有大量并发的阅读者且始终至少有一个活跃的读取者，那么就没有检查点能够完成，WAL 文件将无限增长。

    可以通过确保存在“读取者间隙”（即某些时间段没有任何进程在读取数据库）并在这些时间段尝试检查点来避免这一情况。在同时有大量读取者的应用中，也可以考虑使用SQLITE_CHECKPOINT_RESTART或SQLITE_CHECKPOINT_TRUNCATE手动运行检查点，以确保检查点在返回前完成。使用这两个选项的缺点是，在检查点运行期间可能会阻塞读取者。

  • 非常大的写事务。 只有在没有其他事务运行时，才可能完成检查点，这意味着在写事务进行中无法重置 WAL。因此，对大型数据库的重大修改可能会导致 WAL 文件非常大。写事务完成后（假设没有其他读取者阻塞），WAL 文件会被检查点处理；但在此期间，文件可能会迅速增长。

    从3.11.0 版本（2016-02-15）起，单个事务生成的 WAL 文件大小应与事务本身成正比。事务中修改的页面应该只进入 WAL 一次。然而，在旧版本的 SQLite 中，如果事务大于页面缓存，同一页面可能会被多次写入 WAL。

7. WAL 索引的共享内存实现

wal-index是通过一个普通文件实现并通过 mmap 提供鲁棒性的。早期（预发布）WAL 模式实现将 wal-index 存储在易失性共享内存中，例如 Linux 的 /dev/shm 或其他 Unix 系统的 /tmp。该方法的问题在于，如果某些进程通过chroot 更改了根目录，它们会看到不同的文件，因此使用不同的共享内存区域，最终导致数据库损坏。其他方式创建无名共享内存块在各种 Unix 发行版之间也不具备可移植性，且我们无法在 Windows 上找到创建无名共享内存的方法。我们发现保证访问同一数据库文件的所有进程都使用相同共享内存的唯一方法，是通过 mmap 在与数据库位于同一目录下创建一个文件作为共享内存。

使用普通磁盘文件提供共享内存的缺点是可能会通过将共享内存写入磁盘而进行不必要的磁盘 I/O。不过，开发者认为这并非重大问题，因为 wal-index 通常不超过 32 KiB 且从不执行同步。此外，当最后一个数据库连接断开时，wal-index 的 backing 文件就会被删除，这通常阻止了任何真正的磁盘 I/O。

对于默认共享内存实现无法接受的特定应用，可以通过自定义VFS设计替代方案。例如，如果某个数据库仅在单一进程内的线程之间访问，那么 wal-index 可以使用堆内存而非真正的共享内存实现。

8. 在无共享内存情况下使用 WAL

从 SQLite 3.7.4 版本（2010-12-07）起，只要在首次访问前将locking_mode 设置为 EXCLUSIVE，WAL 数据库就可以在不具备共享内存的情况下被创建、读取和写入。换言之，如果某个进程能够确保其是唯一访问数据库的进程，那么它就可以在不使用共享内存的情况下和 WAL 数据库交互。这个特性使得某些缺乏“版本 2”共享内存方法 xShmMap、xShmLock、xShmBarrier 与 xShmUnmap 的遗留VFS也能创建、读取和写入 WAL 数据库。

如果在第一次 WAL 模式数据库访问前设置了EXCLUSIVE 锁模式，那么 SQLite 就永远不会调用任何共享内存方法，因此也不会创建共享内存 wal-index。在这种情况下，只要日志模式是 WAL，数据库连接就会保持在 EXCLUSIVE 模式；尝试通过"PRAGMA locking_mode=NORMAL;"更改锁模式将无效。要退出 EXCLUSIVE 锁模式，必须先退出 WAL 日志模式。

若在第一次 WAL 模式数据库访问时使用 NORMAL 锁模式，则会创建共享内存 wal-index。这意味着底层 VFS 必须支持“版本 2”的共享内存。如果 VFS 不支持这些方法，则尝试打开已在 WAL 模式下的数据库或将数据库转换为 WAL 模式时将失败。只要恰好有一个连接使用共享内存 wal-index，锁模式都可以在 NORMAL 与 EXCLUSIVE 之间自由切换。只有在省略共享内存 wal-index（即首次 WAL 模式访问前锁模式为 EXCLUSIVE）时，锁模式才会锁定在 EXCLUSIVE。

9. 在 WAL 模式下有时查询返回 SQLITE_BUSY

WAL 模式的第二个优点是写入者不阻塞读取者，读取者也不阻塞写入者。这在大多数情况下成立。但也有一些偏僻的场景，WAL 模式数据库的查询可能返回SQLITE_BUSY，应用需对此有所准备。

可能返回SQLITE_BUSY的情况包括：

  • 如果另一个数据库连接以独占锁模式打开数据库，那么对该数据库的所有查询都会返回SQLITE_BUSY。例如，Chrome 与 Firefox 会以独占锁模式打开其数据库文件，因此在那些应用运行时尝试读取其数据库时就会遇到该问题。

  • 当某个数据库的最后一个连接正在关闭，该连接会在清理 WAL 和共享内存文件时暂时获取独占锁。如果此时另一个进程尝试打开并查询数据库，则可能会遇到SQLITE_BUSY错误。

  • 如果数据库的最后一个连接崩溃，那么第一个新连接在打开数据库时会启动恢复过程。在恢复期间会保留独占锁。因此，如果第三个数据库连接在第二个连接执行恢复时尝试查询，第三个连接将收到SQLITE_BUSY错误。

10. 向后兼容性

WAL 模式不会改变数据库文件格式。然而，WAL 文件和wal-index是新概念，因此旧版本的 SQLite 无法在崩溃时恢复当时处于 WAL 模式的数据库。为了防止旧版本的 SQLite（3.7.0 版本前，2010-07-22）尝试恢复 WAL 模式数据库（从而使问题更糟），在 WAL 模式下会将数据库文件头中版本号（数据库头部的第 18 和 19 字节）从 1 提高到 2。因此，如果旧版本的 SQLite 尝试连接处于 WAL 模式的数据库，它会报错，如“文件已加密或不是数据库”。

可以显式通过如下 pragma 退出 WAL 模式：

PRAGMA journal_mode=DELETE;

故意退出 WAL 模式会将数据库文件格式版本号恢复为 1，以便旧版本的 SQLite 再次访问该数据库文件。
