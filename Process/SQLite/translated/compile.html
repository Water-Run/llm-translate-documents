1. 概述

对于大多数用途而言，使用默认的编译选项即可成功构建 SQLite。然而，如果需要，可以使用下文记录的编译时选项来 省略 SQLite 功能（从而产生一个更小的编译库体积）或更改一些参数的默认值。

已经尽最大努力确保各种编译选项组合能够协调工作并生成可用的库。然而，仍然强烈建议在使用非标准编译选项构建 SQLite 库之前，执行 SQLite 测试套件以检查是否有错误。

可以使用compile_options pragma来确定在构建某个 SQLite 副本时采用了下文列出的哪些选项。

2. 推荐的编译时选项

对于能够使用以下选项的应用程序，推荐这些编译时选项，以尽量减少 SQLite 使用的 CPU 周期和内存字节数。 并非所有这些编译时选项对每个应用程序都适用。例如，SQLITE_THREADSAFE=0 选项仅适用于从未让 SQLite 被多个线程同时访问的应用程序。 SQLITE_OMIT_PROGRESS_CALLBACK 选项仅适用于不使用sqlite3_progress_handler() 接口的应用程序。诸如此类的限制依然存在。

无法测试 SQLite 所有可能的编译选项组合。但下面这组编译选项是一个始终经过充分测试的配置。

  1. SQLITE_DQS=0。 此设置禁用了双引号字符串字面值的缺陷特性。

  2. SQLITE_THREADSAFE=0。 设置 -DSQLITE_THREADSAFE=0 会使 SQLite 中所有的互斥和线程安全逻辑被省略。这是唯一一个可使 SQLite 运行速度提升约 2% 的编译时选项，同时还会让库的大小缩小约 2%。但其缺点是启用该选项意味着 SQLite 再也不能被多个线程同时使用，即便每个线程拥有自己的数据库连接。

  3. SQLITE_DEFAULT_MEMSTATUS=0。 此设置会禁用用于跟踪内存使用情况的sqlite3_status() 接口。这有助于sqlite3_malloc() 运行得更快，而且由于 SQLite 内部使用了sqlite3_malloc()，这会让整个库的性能提升。

  4. SQLITE_DEFAULT_WAL_SYNCHRONOUS=1。 为了在断电后实现最大的数据库安全性，推荐设置PRAGMA synchronous=FULL。然而，在WAL 模式下，使用PRAGMA synchronous=NORMAL即可保证数据库完整性。采用PRAGMA synchronous=NORMAL的 WAL 模式下，电源断电可能会回滚近期更改，但不会破坏数据库。此外，在 WAL 模式中使用 synchronous=NORMAL 的事务提交速度远快于默认的 synchronous=FULL。因此，建议在切换到 WAL 模式时将 synchronous 设置从 FULL 改为 NORMAL。这个编译时选项可以实现该修改。

  5. SQLITE_LIKE_DOESNT_MATCH_BLOBS。 历史上 SQLite 允许 BLOB 操作数用于LIKE 和GLOB 运算符。但当 BLOB 用作LIKE 或GLOB 的操作数时，会使LIKE 优化变得复杂且缓慢。启用此选项后，只要任一操作数为 BLOB，LIKE 和 GLOB 运算符就始终返回 FALSE。这简化了LIKE 优化的实现，并使使用该优化的查询运行更快。

  6. SQLITE_MAX_EXPR_DEPTH=0。 将表达式解析树的最大深度设置为零可以禁用所有有关表达式解析树深度的检查，从而简化代码、提升执行速度，并且减少解析树占用的内存。

  7. SQLITE_OMIT_DECLTYPE。 省略从查询结果集中返回列声明类型的（很少需要的）功能后，准备语句可以占用更少的内存。

  8. SQLITE_OMIT_DEPRECATED。 省略已弃用的接口和功能不会让 SQLite 运行更快，但会减小库的体积。而且这是正确的做法。

  9. SQLITE_OMIT_PROGRESS_CALLBACK。 进度处理回调计数必须在字节码引擎的内部循环中被检查。省略该接口可以从字节码引擎的内部循环中去除一个条件，从而让 SQL 语句运行略快一些。

  10. SQLITE_OMIT_SHARED_CACHE。 省略使用共享缓存的可能性可以在性能关键代码中消除许多条件判断，从而带来明显的性能提升。

  11. SQLITE_USE_ALLOCA。 在支持 alloca() 的系统上，使用 alloca() 来在单个函数内部动态分配临时栈空间。若不使用该选项，则临时空间将从堆中分配。

  12. SQLITE_OMIT_AUTOINIT。 在使用某些接口前，SQLite 库需要通过调用sqlite3_initialize()来初始化。通常首次使用时会自动完成这一步骤。但启用 SQLITE_OMIT_AUTOINIT 后，自动初始化将省略。这帮助许多 API 调用运行得更快（因为它们无需检查初始化是否已经发生再执行初始化），但同时意味着应用程序必须手动调用sqlite3_initialize()。如果 SQLite 使用 -DSQLITE_OMIT_AUTOINIT 编译，而在调用sqlite3_malloc()、sqlite3_vfs_find()或sqlite3_open()等例程之前未先调用sqlite3_initialize()，很可能会导致段错误。

  13. SQLITE_STRICT_SUBTYPE=1。 如果定义的应用程序函数未设置SQLITE_RESULT_SUBTYPE属性却调用了sqlite3_result_subtype()接口，则此选项会引发错误。若函数未注册 SQLITE_RESULT_SUBTYPE 属性，该接口无法可靠运行。此编译时选项旨在尽早提醒开发者注意这个问题。

当上述所有推荐的编译时选项都被使用时，SQLite 库的体积大约会减少 3%，并节省约 5% 的 CPU 周期。因此这些选项并不会带来巨大的差异，但在某些设计环境中，哪怕是微小的提升也至关重要。

库级配置选项（如上所列）也可以选择在客户端头文件中定义。定义 SQLITE_CUSTOM_INCLUDE=myconfig.h（无引号）会使 sqlite3.c 在编译早期阶段包含 myconfig.h，从而允许客户端自定义标志而无需显式将其全部传递给编译器。

3. 平台配置

_HAVE_SQLITE_CONFIG_H

如果定义了 _HAVE_SQLITE_CONFIG_H 宏，那么 SQLite 源代码会尝试 #include 名为 "sqlite_cfg.h" 的文件。 "sqlite_cfg.h" 文件通常包含其他配置选项，尤其是由 configure 过程生成的 "HAVE_接口" 类型选项。请注意，这个头文件只用于平台级配置，而非库级配置。如果要在自定义头文件中设置 SQLite 级配置标志，请按照上一节所述定义 SQLITE_CUSTOM_INCLUDE=myconfig.h。

HAVE_FDATASYNC

如果 HAVE_FDATASYNC 编译时选项为真，则 Unix 系统的默认VFS会在适当情况下尝试使用 fdatasync() 代替 fsync()。如果该标志缺失或为假，则总是使用 fsync()。

HAVE_GMTIME_R

如果 HAVE_GMTIME_R 为真且SQLITE_OMIT_DATETIME_FUNCS 为真，则 CURRENT_TIME、CURRENT_DATE 和 CURRENT_TIMESTAMP 关键字将使用线程安全的 "gmtime_r()" 接口而非 "gmtime()"。在通常情况下，当SQLITE_OMIT_DATETIME_FUNCS 未定义或为假时，则使用内建的日期与时间函数来实现 CURRENT_TIME、CURRENT_DATE 和 CURRENT_TIMESTAMP，并且 gmtime_r() 与 gmtime() 都不会被调用。

HAVE_ISNAN

如果 HAVE_ISNAN 为真，则 SQLite 会调用系统库中的 isnan() 函数来判断双精度浮点值是否为 NaN。如果 HAVE_ISNAN 未定义或为假，则 SQLite 会使用自有实现的 isnan()。

HAVE_LOCALTIME_R

如果 HAVE_LOCALTIME_R 为真，则 SQLite 会使用线程安全的 localtime_r() 库例程而非 localtime()，以帮助实现内建localtime 修饰符。

HAVE_LOCALTIME_S

如果 HAVE_LOCALTIME_S 为真，则 SQLite 会使用线程安全的 localtime_s() 库例程而非 localtime()，以帮助实现内建localtime 修饰符。

HAVE_MALLOC_USABLE_SIZE

如果 HAVE_MALLOC_USABLE_SIZE 为真，则 SQLite 会使用 malloc_usable_size() 接口来查找由标准库的 malloc() 或 realloc() 获取的内存分配的大小。此选项仅在使用标准库 malloc() 时适用。在 Apple 系统上使用的是 "zone malloc"，因此该选项不适用。当然，如果应用程序通过SQLITE_CONFIG_MALLOC 提供了自己的 malloc 实现，则此选项也无效。

如果省略 HAVE_MALLOC_USABLE_SIZE 或将其设为假，则 SQLite 会在系统 malloc() 和 realloc() 周围使用一个包装器，将每次分配增加 8 个字节，并在前 8 个字节中写入分配大小，然后 SQLite 也实现自己版本的 malloc_usable_size()，通过查阅这 8 字节前缀来获取分配大小。这种方式可行但并非最优。鼓励应用程序在可能的情况下启用 HAVE_MALLOC_USABLE_SIZE。

HAVE_STRCHRNUL

如果 HAVE_STRCHRNUL 为真，则 SQLite 会使用 strchrnul() 库函数。如果该选项缺失或为假，则 SQLite 会使用自有实现的 strchrnul()。

HAVE_UTIME

如果 HAVE_UTIME 为真，则内建但非标准的 "unix-dotfile" VFS 会使用 utime() 系统调用（而非 utimes()）来设置锁文件的最后访问时间。

SQLITE_BYTEORDER=(0|1234|4321)

SQLite 需要知道目标 CPU 的本机字节序是大端还是小端。SQLITE_BYTEORDER 预处理器通常会自动在所有常见平台和编译器中设为 4321（大端）或 1234（小端），也可以设成 0 表示在运行时确定字节序。当为不常见目标编译 SQLite 时，手动设置 SQLite_BYTEORDER 可能会更合适。如果在编译时无法确定目标字节序，SQLite 会退回到运行时检查，虽然会有些微的性能损失，但总能正常工作。

4. 设置默认参数值的选项

SQLITE_DEFAULT_AUTOMATIC_INDEX=<0 或 1>

此宏决定新打开的数据库连接的PRAGMA automatic_index的初始设置。在 SQLite 3.7.17 及之前的版本中，如果省略此编译时选项，则新数据库连接默认启用自动索引。然而，将来版本可能会改变这一行为。

另请参见：SQLITE_OMIT_AUTOMATIC_INDEX

SQLITE_DEFAULT_AUTOVACUUM=<0 或 1 或 2>

此宏决定 SQLite 默认创建数据库时是否设置auto_vacuum 标志为关闭（0）、完全（1）或增量（2）。默认值为 0，表示创建的数据库默认关闭自动清理。 无论如何，该编译时默认值可以通过PRAGMA auto_vacuum 命令在运行时覆盖。

SQLITE_DEFAULT_CACHE_SIZE=<N>

此宏设置为每个附加数据库的页面缓存默认最大大小。正值表示限制为 N 页；若 N 为负数，则表示将缓存限制为 -N * 1024 字节。 建议的最大缓存大小可通过PRAGMA cache_size 命令覆盖。默认值为 -2000，相当于每个缓存最多 2048000 字节。

SQLITE_DEFAULT_FILE_FORMAT=<1 或 4>

此宏设置 SQLite 创建新数据库文件时使用的默认模式格式编号。模式格式很相似。格式 1 与 4 的主要区别在于格式 4 支持降序索引并且对布尔值采用更紧凑的编码。

自 SQLite 3.3.0（2006-01-10）以来的所有版本都能读取和写入介于 1 和 4 之间的任何模式格式。但更旧版本可能无法读取大于 1 的格式。因此，为保证旧版本 SQLite 能够读取和写入由新版本创建的数据库文件，直到 SQLite 3.7.9（2011-11-01）为止默认使用格式 1。从3.7.10 版本（2012-01-16）起，默认模式格式为 4。

可以通过PRAGMA legacy_file_format 命令在运行时设置新数据库的模式格式编号。

SQLITE_DEFAULT_FILE_PERMISSIONS=N

Unix 系统中新建数据库文件的默认数值文件权限。如果未指定，默认权限是 0644，表示文件可由所有用户读取，但仅能由创建者写入。

SQLITE_DEFAULT_FOREIGN_KEYS=<0 或 1>

此宏决定新的数据库连接是否默认启用外键约束的强制执行。每个数据库连接始终可以在运行时使用foreign_keys pragma来开启或关闭外键约束的强制执行。默认情况下外键约束通常是关闭的，但如果将此编译时参数设为 1，则默认启用外键约束。

SQLITE_DEFAULT_MMAP_SIZE=N

此宏设置每个打开的数据库文件用于内存映射 I/O 的默认内存限制。如果N为零，则默认禁用内存映射 I/O。此编译时限制以及SQLITE_MAX_MMAP_SIZE可以在启动时通过sqlite3_config(SQLITE_CONFIG_MMAP_SIZE)调用，或在运行时通过mmap_size pragma修改。

SQLITE_DEFAULT_JOURNAL_SIZE_LIMIT=<bytes>

此选项设置在回滚日志文件（在永久日志模式和独占锁定模式下）以及WAL 模式的预写日志文件中的大小上限。如果省略此编译时选项，则对回滚日志或预写日志的大小没有上限。可以通过journal_size_limit pragma在运行时更改日志文件大小上限。

SQLITE_DEFAULT_LOCKING_MODE=<1 或 0>

若设为 1，则默认的locking_mode 被设为 EXCLUSIVE。若省略或设为 0，则默认的locking_mode 为 NORMAL。

SQLITE_DEFAULT_LOOKASIDE=SZ,N

将lookaside 内存分配器内存池的默认大小设置为 N 个条目，每个条目 SZ 字节。此设置可以在启动时通过sqlite3_config(SQLITE_CONFIG_LOOKASIDE) 修改，或在每个数据库连接打开时通过sqlite3_db_config(db, SQLITE_DBCONFIG_LOOKASIDE) 修改。

SQLITE_DEFAULT_MEMSTATUS=<1 或 0>

该宏用于决定默认是否启用通过sqlite3_config()的 SQLITE_CONFIG_MEMSTATUS 参数开启或关闭的功能。默认值为 1（启用与SQLITE_CONFIG_MEMSTATUS 相关的功能）。

当内存使用跟踪禁用时，sqlite3_memory_used() 与 sqlite3_memory_highwater() 接口、sqlite3_status64(SQLITE_STATUS_MEMORY_USED) 接口以及SQLITE_MAX_MEMORY 编译时选项都无法工作。

SQLITE_DEFAULT_PCACHE_INITSZ=N

此宏决定当SQLITE_CONFIG_PAGECACHE 配置未使用且页面缓存内存由sqlite3_malloc() 提供时，页面缓存模块最初分配的页数。该宏设置的页数会在一次性分配中分配，从而减少内存分配器的负担。

SQLITE_DEFAULT_PAGE_SIZE=<bytes>

此宏用于设置创建数据库时默认使用的页面大小。赋值必须为 2 的幂。默认值为 4096。编译时默认值可通过PRAGMA page_size 命令在运行时覆盖。

SQLITE_DEFAULT_SYNCHRONOUS=<0-3>

此宏决定PRAGMA synchronous 设置的默认值。如果未在编译时覆盖，则默认值为 2（FULL）。

SQLITE_DEFAULT_WAL_SYNCHRONOUS=<0-3>

此宏决定以WAL 模式打开的数据库文件的PRAGMA synchronous 设置的默认值。如果未在编译时覆盖，此值与SQLITE_DEFAULT_SYNCHRONOUS相同。

如果 SQLITE_DEFAULT_WAL_SYNCHRONOUS 与 SQLITE_DEFAULT_SYNCHRONOUS 不同，且应用程序未在运行时通过PRAGMA synchronous 语句修改数据库文件的同步设置，则数据库连接首次切换到 WAL 模式时，会将同步设置更改为 SQLITE_DEFAULT_WAL_SYNCHRONOUS 定义的值。 如果 SQLITE_DEFAULT_WAL_SYNCHRONOUS 没有在编译时被覆盖，它将始终与SQLITE_DEFAULT_SYNCHRONOUS 相同，因此不会发生自动的同步设置更改。

SQLITE_DEFAULT_WAL_AUTOCHECKPOINT=<pages>

此宏设置WAL自动检查点功能的默认页数。如果未指定，默认页数为 1000。

SQLITE_DEFAULT_WORKER_THREADS=N

此宏设置一个准备语句默认启动的辅助线程的最大数量。如果未指定，默认最大值为 0。此处设置的值不能超过SQLITE_MAX_WORKER_THREADS。

SQLITE_DQS=N

此宏决定SQLITE_DBCONFIG_DQS_DDL 和 SQLITE_DBCONFIG_DQS_DML 的默认值，进而决定 SQLite 如何处理每个双引号字符串字面值。“DQS”为“Double-Quoted String”的缩写。N 参数应该是整数 0、1、2 或 3。

SQLITE_EXTRA_DURABLE

SQLITE_EXTRA_DURABLE 编译时选项曾使默认的PRAGMA synchronous 设置为 EXTRA，而非 FULL。此选项已不再支持。请改用SQLITE_DEFAULT_SYNCHRONOUS=3。

SQLITE_FTS3_MAX_EXPR_DEPTH=N

此宏设置对应于FTS3或FTS4全文索引中 MATCH 运算符右侧的搜索树的最大深度。全文搜索使用递归算法，因此限制树深度可防止占用过多的栈空间。默认限制为 12，该限制足以支持 MATCH 右侧最多 4095 个搜索项，并将栈空间使用控制在 2000 字节以内。

对于普通的 FTS3/FTS4 查询，搜索树深度大约等于 MATCH 右侧词项数量的二进制对数。但对于短语查询和NEAR 查询，搜索树深度与右侧词项数量成线性关系。因此默认的 12 层限制足以支持最多 4095 个普通词项，但仅能支持 11 或 12 个短语或 NEAR 词项。即便如此，对于大多数应用而言默认值已经足够。

SQLITE_JSON_MAX_DEPTH=N

此宏设置 JSON 对象和数组的最大嵌套深度。默认值为 1000。

JSON SQL 函数使用递归下降解析器。这意味着深度嵌套的 JSON 可能需要大量栈空间来解析。在栈空间有限的系统上，可以编译 SQLite 时将最大 JSON 嵌套深度大幅降低，以避免来自恶意输入的栈溢出。通常 10 或 20 的值就足够用于最复杂的真实世界 JSON。

SQLITE_LIKE_DOESNT_MATCH_BLOBS

此编译时选项会使LIKE 运算符在任一操作数为 BLOB 时始终返回 False。LIKE 的默认行为是将 BLOB 操作数转换为 TEXT 后再比较。

此编译时选项可以让 SQLite 在处理使用LIKE 运算符的查询时运行得更高效，但会破坏向后兼容性。然而，这种破坏可能只是技术细节。长期存在的LIKE 处理逻辑错误（参见 https://sqlite.org/src/info/05f43be8fdda9f）曾导致在 BLOB 操作数上表现异常，但在近 10 年的实际使用中几乎无人注意到。因此，对大多数用户而言，启用此编译时选项并在 LIKE 查询中节省一点 CPU 时间可能是安全的。

此编译时选项仅影响 SQL LIKE 运算符，对sqlite3_strlike() C 语言接口没有影响。

SQLITE_MAX_MEMORY=N

此选项限制 SQLite 向 malloc() 请求的内存总量不超过 N 字节。SQLite 尝试分配任何会使当前所有分配总和超过 N 字节的新内存时，都会返回内存不足错误。 这是一个硬性上限。另请参见sqlite3_soft_heap_limit()接口。

此选项限制的是总内存量。有关单次内存分配大小的限制，请查看SQLITE_MAX_ALLOCATION_SIZE选项。

该限制只有在可以通过sqlite3_memory_used()与sqlite3_status64(SQLITE_STATUS_MEMORY_USED)获取内存使用统计信息时才有效。若没有这些信息，SQLite 无法知道何时接近上限，因此无法阻止超过上限的内存申请。默认情况下启用内存使用跟踪，但可以通过SQLITE_DEFAULT_MEMSTATUS在编译时禁用，或通过sqlite3_config(SQLITE_CONFIG_MEMSTATUS)在启动时禁用。

SQLITE_MAX_MMAP_SIZE=N

此宏为任意单个数据库可使用的内存映射 I/O 地址空间设置硬性上限。将此值设为 0 将完全禁用内存映射 I/O，并使与内存映射 I/O 相关的逻辑在构建中被省略。只要其他设置（由SQLITE_DEFAULT_MMAP_SIZE或 sqlite3_config(SQLITE_CONFIG_MMAP_SIZE)设定的默认地址空间大小，或由 sqlite3_file_control(SQLITE_FCNTL_MMAP_SIZE)或PRAGMA mmap_size设定的运行时地址空间大小）小于此最大值，则此选项不会更改那些设置。

SQLITE_MAX_SCHEMA_RETRY=N

每当数据库模式改变时，准备语句会自动重新准备以适应新模式。这其中存在一个竞态条件：若某线程不断修改模式，另一线程可能会不断重新解析和重新准备准备语句而无从得到实际工作。此参数通过强制旋转线程在尝试重新编译准备语句固定次数后放弃，从而防止无限循环。默认设置为 50，足以满足大多数应用。

SQLITE_MAX_WORKER_THREADS=N

为 sqlite3_limit(db,SQLITE_LIMIT_WORKER_THREADS,N) 设置上限，该设置决定单个准备语句在处理 CPU 密集型计算（主要是排序）时最多可以使用多少辅助线程。另请参见SQLITE_DEFAULT_WORKER_THREADS选项。

SQLITE_MEMDB_DEFAULT_MAXSIZE=N

为使用sqlite3_deserialize()创建的内存数据库设置默认大小限制（单位：字节）。这只是默认值。可以在启动时通过sqlite3_config(SQLITE_CONFIG_MEMDB_MAXSIZE,N) 或针对单个数据库在运行时通过SQLITE_FCNTL_SIZE_LIMIT文件控制命令修改。如果未指定默认值，则使用 1073741824。

SQLITE_MINIMUM_FILE_DESCRIPTOR=N

Unix VFS 将永远不会使用小于 N 的文件描述符。N 的默认值为 3。

避免使用低编号文件描述符是防止意外数据库损坏的手段。例如，如果数据库文件是通过文件描述符 2 打开的，然后某个 assert() 失败并调用 write(2,...)，这可能会通过覆盖数据库文件的一部分（写入断言错误消息）导致数据库损坏。使用较高编号的文件描述符可避免此类问题。可以通过将此编译时选项设为 0 来禁用对低编号文件描述符的保护。

SQLITE_POWERSAFE_OVERWRITE=<0 或 1>

此选项改变了 Unix 和 Windows VFS 底层文件系统对powersafe overwrite 的默认假设。将 SQLITE_POWERSAFE_OVERWRITE 设置为 1 会让 SQLite 假设应用级写入不会在写入范围之外修改字节，即使写入发生在断电前。而将其设为 0 则表示 SQLite 假设同一扇区中与写入字节相邻的其他字节在断电时可能会被更改或损坏。

SQLITE_PRINTF_PRECISION_LIMIT=N

此选项限制printf() SQL 函数以及其他 C 语言字符串格式化函数（如sqlite3_mprintf()和sqlite3_str_appendf()）中替换项的最大宽度与精度。例如可以防止恶意或故障脚本通过执行 "printf('%*s',2147483647,'hi')" 来耗尽内存。通常设置 N 为 100000 左右即可。

printf() SQL 函数受到SQLITE_LIMIT_LENGTH（sqlite3_limit() 的一个限制）约束。因此任何宽度或精度超过 SQLITE_LIMIT_LENGTH 的 printf() 结果都会导致SQLITE_TOOBIG错误。然而，printf() 函数的低级格式化子程序无法访问 SQLITE_LIMIT_LENGTH，因此会在内存分配中进行格式化，这可能会远超 SQLITE_LIMIT_LENGTH，而 SQLITE_LIMIT_LENGTH 检查仅在格式化完成后执行。因此，可能会出现暂时超出 SQLITE_LIMIT_LENGTH 的缓冲区。SQLITE_PRINTF_PRECISION_LIMIT 选项提供了额外的检查，防止低级格式化子程序在 SQLITE_LIMIT_LENGTH 检查之前产生过大的暂存缓冲区。

请注意不要将 SQLITE_PRINTF_PRECISION_LIMIT 设置得太低。SQLite 使用其内建 printf() 功能格式化存储在sqlite_schema 表中的 CREATE 语句文本。因此 SQLITE_PRINTF_PRECISION_LIMIT 应至少与您可能遇到的最大表、索引、视图或触发器定义一样大。

若宽度或精度超过 SQLITE_PRINTF_PRECISION_LIMIT 不会引发错误。大宽度或精度会被静默截断。

SQLITE_PRINTF_PRECISION_LIMIT 的默认值为 2147483647（0x7fffffff）。

SQLITE_QUERY_PLANNER_LIMIT=N

在查询规划过程中，SQLite 会枚举所有可用的索引与 WHERE 子句约束组合。对于某些异常查询，这些索引与约束组合的数量可能非常庞大，从而导致查询规划器性能极慢。SQLITE_QUERY_PLANNER_LIMIT 的值（与相关的SQLITE_QUERY_PLANNER_LIMIT_INCR设置结合使用）限制了查询规划器将考虑的索引与约束组合数量，以避免规划器耗费过多 CPU 时间。该选项的默认值设得足够高，以便现实中很少触及。规划器搜索限制仅适用于故意构造以耗费大量规划时间的查询。

SQLITE_QUERY_PLANNER_LIMIT_INCR=N

SQLITE_QUERY_PLANNER_LIMIT 选项设置了查询规划器将考虑的索引与约束组合数量的初始基线值。在处理连接查询中的每个表之前，该基线限制会先增加 SQLITE_QUERY_PLANNER_LIMIT_INCR，以确保即使先前表已经耗尽了基线限额，当前表也至少能够向优化器提供一些索引与约束组合。此编译时选项和SQLITE_QUERY_PLANNER_LIMIT 选项的默认值都设置得足够高，以便现实中不会达到上限。

SQLITE_REVERSE_UNORDERED_SELECTS

该选项使PRAGMA reverse_unordered_selects 设置默认开启。开启后，缺少 ORDER BY 子句的SELECT 语句将以相反顺序执行。

此选项对于检测应用程序何时（错误地）假设未指定 ORDER BY 的 SELECT 语句总会以相同顺序返回行非常有用。

SQLITE_SORTER_PMASZ=N

如果通过PRAGMA threads 设置启用了多线程处理，则当待排序内容超过由cache_size与由SQLITE_CONFIG_PMASZ启动时选项确定的 PMA 大小的最小值时，排序操作会尝试启动辅助线程。此编译时选项设置SQLITE_CONFIG_PMASZ 启动选项的默认值。默认值为 250。

SQLITE_STMTJRNL_SPILL=N

SQLITE_STMTJRNL_SPILL 编译时选项确定SQLITE_CONFIG_STMTJRNL_SPILL 启动设置的默认值。该设置决定超过多少字节的语句日志会从内存移动到磁盘。

SQLITE_WIN32_MALLOC

此选项启用使用 Windows Heap API 函数进行内存分配，而非标准库的 malloc() 与 free()。

YYSTACKDEPTH=<max_depth>

此宏设置 SQLite 中 SQL 解析器使用的 LALR(1) 栈的最大深度。默认值为 100。典型应用大约使用不到 20 个栈层。若应用包含需要超过 100 个 LALR(1) 栈条目的 SQL 语句，建议对 SQL 进行重构，因为它们可能已经超出了任何人类的理解能力。

5. 设置大小限制的选项

存在一些编译时选项可以对 SQLite 中各种结构的大小设置上限。这些编译时选项通常设置了硬性上限，并可以在运行时通过数据库连接调用sqlite3_limit()接口进行修改。

设置上限的编译选项在另一处文档中另行描述。以下是可用设置列表：

  • SQLITE_MAX_ATTACHED
  • SQLITE_MAX_COLUMN
  • SQLITE_MAX_COMPOUND_SELECT
  • SQLITE_MAX_EXPR_DEPTH
  • SQLITE_MAX_FUNCTION_ARG
  • SQLITE_MAX_LENGTH
  • SQLITE_MAX_LIKE_PATTERN_LENGTH
  • SQLITE_MAX_PAGE_COUNT
  • SQLITE_MAX_SQL_LENGTH
  • SQLITE_MAX_VARIABLE_NUMBER

还有一些大小限制无法通过sqlite3_limit()修改。例如：

  • SQLITE_FTS3_MAX_EXPR_DEPTH
  • SQLITE_JSON_MAX_DEPTH
  • SQLITE_MAX_ALLOCATION_SIZE
  • SQLITE_MAX_MEMORY
  • SQLITE_MAX_MMAP_SIZE
  • SQLITE_PRINTF_PRECISION_LIMIT
  • SQLITE_TRACE_SIZE_LIMIT
  • YYSTACKDEPTH

6. 控制运行特性的选项

SQLITE_4_BYTE_ALIGNED_MALLOC

在大多数系统中，malloc() 系统调用返回对齐到 8 字节边界的缓冲区。但在某些系统（如 Windows）中，malloc() 返回 4 字节对齐的指针。此编译时选项必须用于那些 malloc() 返回 4 字节对齐指针的系统。

SQLITE_CASE_SENSITIVE_LIKE

如果启用此选项，则内建LIKE 运算符将区分大小写。相同效果可以在运行时通过pragma case_sensitive_like 设置实现。

SQLITE_DIRECT_OVERFLOW_READ

启用此选项时，在读取事务期间，数据库文件中的溢出页面内容会直接从磁盘读取，绕过页面缓存。对于大量读取大型 BLOB 或字符串的应用，此选项能提升读取性能。

从 3.45.0 版本（2024-01-15）起，该选项默认启用。要禁用它，请使用 -DSQLITE_DIRECT_OVERFLOW_READ=0。

SQLITE_HAVE_ISNAN

如果定义此选项，SQLite 会使用系统数学库中的 isnan() 函数。这是HAVE_ISNAN 配置选项的别名。

SQLITE_MAX_ALLOCATION_SIZE=N

此编译时选项为使用sqlite3_malloc64()、sqlite3_realloc64()等函数请求的内存分配设置上限。默认值为 2,147,483,391（0x7ffffeff），可被视为上限。大多数应用只需几百万字节的最大分配就够了。

这是针对单次内存分配大小的限制，而非总内存量的限制。有关总内存限制，请参见SQLITE_MAX_MEMORY。

降低单次内存分配的最大值可以额外防御那些试图通过大量大分配耗尽系统内存的拒绝服务攻击。此外，还能减轻使用带符号 32 位整数计算分配大小时可能发生的溢出错误。设置较小的最大分配值能使有缺陷的分配计算更快被触发并导致内存不足，从而在整数溢出前暴露出问题。

SQLITE_OS_OTHER=<0 或 1>

此选项会让 SQLite 省略内置的 Unix、Windows 与 OS/2 操作系统接口。生成的库将没有默认的操作系统接口。应用程序必须在使用 SQLite 之前，通过sqlite3_vfs_register() 注册合适的接口。应用程序还必须提供sqlite3_os_init()与sqlite3_os_end()接口的实现。通常做法是提供的sqlite3_os_init() 在内部调用sqlite3_vfs_register()。SQLite 在初始化时会自动调用sqlite3_os_init()。

该选项通常用于为带有自定义操作系统的嵌入式平台构建 SQLite。

SQLITE_SECURE_DELETE

此编译时选项改变secure_delete pragma的默认设置。如果未启用该选项，secure_delete 默认为关闭。启用后，secure_delete 默认为开启。

secure_delete 设置会将已删除内容覆盖为零。这会带来轻微的性能损失，因为需要额外的 I/O。但另一方面，它可以防止敏感信息在数据库文件中的未使用部分遗留。更多信息参见secure_delete pragma文档。

SQLITE_THREADSAFE=<0 或 1 或 2>

此选项控制是否在 SQLite 中包含能够在多线程环境中安全运行的代码。默认值为 SQLITE_THREADSAFE=1，可安全用于多线程程序。若编译时设置为 SQLITE_THREADSAFE=0，则所有互斥代码都被省略，SQLite 无法在多线程程序中安全使用。若设置为 SQLITE_THREADSAFE=2，只要多个线程不会同时使用同一数据库连接（或从该连接派生的任何准备语句），就可以在多线程程序中使用 SQLite。

换句话说，SQLITE_THREADSAFE=1 会将默认的线程模式设为 Serialize；SQLITE_THREADSAFE=2 将默认线程模式设为 Multi-threaded；而 SQLITE_THREADSAFE=0 则设为 Single-threaded。

可以通过sqlite3_threadsafe() 接口在运行时检测 SQLITE_THREADSAFE 的值。

如果 SQLite 是使用 SQLITE_THREADSAFE=1 或 SQLITE_THREADSAFE=2 编译的，那么可以在运行时通过sqlite3_config()接口以及下列动词之一更改线程模式：

  • SQLITE_CONFIG_SINGLETHREAD
  • SQLITE_CONFIG_MULTITHREAD
  • SQLITE_CONFIG_SERIALIZED

SQLITE_OPEN_NOMUTEX 和 SQLITE_OPEN_FULLMUTEX 标志也可以与sqlite3_open_v2()配合使用， 实现对单个数据库连接（或其派生的准备语句）的线程模式进行运行时调整。

注意：当 SQLite 以 SQLITE_THREADSAFE=0 编译时，用于线程安全的代码会从构建中省略。此时无法在启动时或运行时更改线程模式。

更多关于在多线程环境中使用 SQLite 的信息，请参见线程模式文档。

SQLITE_TEMP_STORE=<0 到 3>

此选项控制临时文件是存储在磁盘还是内存中。各个设置的含义如下：

SQLITE_TEMP_STORE含义
0始终使用临时文件
1默认使用文件，但允许通过PRAGMA temp_store命令覆盖
2默认使用内存，但允许通过PRAGMA temp_store命令覆盖
3始终使用内存

默认设置为 1。 详细信息请参见tempfiles.html。

SQLITE_TRACE_SIZE_LIMIT=N

如果此宏定义为正整数 N，则sqlite3_trace() 输出中参数展开的字符串和 BLOB 的长度会被限制为 N 字节。

SQLITE_TRUSTED_SCHEMA=<0 或 1>

此宏决定SQLITE_DBCONFIG_TRUSTED_SCHEMA 与PRAGMA trusted_schema 设置的默认值。如果未另行指定，trusted-schema 默认开启（值为 1），以兼容旧版本。但为了最佳安全性，执行应用程序定义 SQL 函数和/或虚拟表的系统应考虑将默认值设为关闭。

SQLITE_USE_URI

此选项使URI 文件名处理逻辑默认启用。

7. 启用默认关闭功能的选项

SQLITE_ALLOW_URI_AUTHORITY

URI 文件名正常情况下如果 authority 部分既不是空串也不是 "localhost" 就会报错。但如果 SQLite 在编译时启用了 SQLITE_ALLOW_URI_AUTHORITY，则 URI 会被转换为统一命名规则 (UNC) 文件名，并以此传递给底层操作系统。

SQLite 的未来某些版本可能会改为默认启用此特性。

SQLITE_ALLOW_COVERING_INDEX_SCAN=<0 或 1>

此 C 预处理宏决定SQLITE_CONFIG_COVERING_INDEX_SCAN 配置设置的默认值。其默认值为 1（开启），表示在可能的情况下为全表扫描使用覆盖索引，以减少 I/O 并提升性能。然而，使用覆盖索引进行全表扫描会让结果的顺序与旧版不同，可能导致某些（实现不正确的）旧版应用程序发生错误。因此，对于希望尽量避免暴露旧版应用错误的系统，可以在编译时禁用覆盖索引扫描选项。

SQLITE_ENABLE_8_3_NAMES=<1 或 2>

如果定义此 C 预处理宏，则会包含额外代码，使 SQLite 能在仅支持 8+3 文件名的文件系统上运行。如果该宏的值为 1，则默认继续使用长文件名，只有在通过URI 文件名并带上 "8_3_names=1" 查询参数时才使用 8+3 文件名。如果值为 2，则默认使用 8+3 文件名，但可通过 8_3_names=0 查询参数禁用。

SQLITE_ENABLE_API_ARMOR

定义该 C 预处理宏后，会启用额外代码，尝试检测 SQLite API 的误用，例如将 NULL 指针传入必需参数或在对象销毁后继续使用它们。当检测到非法 API 使用时，接口通常会返回 SQLITE_MISUSE。

SQLITE_ENABLE_API_ARMOR 选项并不保证能检测到所有非法 API 使用。即使启用了该选项，将不正确的值传入 C 语言 API 仍可能导致进程因段错误、空指针解引用或其他原因崩溃。该编译时选项旨在作为应用测试和调试的辅助工具。应用程序不应依赖 SQLITE_ENABLE_API_ARMOR 来保证安全。它适合作为检测应用错误的第二道防线，但不应依赖其为唯一防线。如果任何 SQLite 接口返回 SQLITE_MISUSE，则说明应用程序违反了规范，即应用本身存在缺陷。SQLITE_MISUSE 的返回让应用有机会以优雅方式响应该缺陷，而不是崩溃或调出未定义行为，但没有更多含义。应用程序也不应在常规处理中使用或依赖 SQLITE_MISUSE。

SQLITE_ENABLE_ATOMIC_WRITE

如果定义此 C 预处理宏，并且数据库文件的sqlite3_io_methods 对象的 xDeviceCharacteristics 方法通过其中一个SQLITE_IOCAP_ATOMIC 位报告文件系统支持原子写入，且事务仅改变数据库文件的一页，则事务会以一条单页写入请求提交，同时不会创建或写入回滚日志。在支持原子写入的文件系统上，该优化可显著加速小型更新。但由于只有少数文件系统支持此能力，而检测能力的代码路径会在不支持原子写入的系统上降低写入性能，因此此功能默认禁用。

SQLITE_ENABLE_BATCH_ATOMIC_WRITE

此编译时选项使 SQLite 能利用底层文件系统的批量原子写入能力。截至 SQLite 3.21.0（2017-10-24），该能力仅在F2FS 中受支持。不过，该接口是通用实现的，使用sqlite3_file_control() 搭配 SQLITE_FCNTL_BEGIN_ATOMIC_WRITE 与 SQLITE_FCNTL_COMMIT_ATOMIC_WRITE，因此未来可将其扩展到其他文件系统。启用该选项后，SQLite 会自动检测底层文件系统是否支持批处理原子写入，若支持，则在事务控制期间避免写入回滚日志。这可使事务速度提高至原来的两倍以上，同时减少 SSD 存储设备的磨损。

SQLite 的未来版本可能会默认启用批处理原子写入功能，此时该编译时选项将不再必要。

SQLITE_ENABLE_BYTECODE_VTAB

此选项启用bytecode 与 tables_used 虚拟表。

SQLITE_ENABLE_CARRAY

该选项启用carray() 扩展，自 SQLite 3.51.0（2025-11-04）起已经集成到amalgamation 中。在该版本之前，carray() 扩展位于独立的源码文件中，需要单独编译并作为可加载扩展链接。

SQLITE_ENABLE_COLUMN_METADATA

定义此 C 预处理宏后，SQLite 会包含一些额外 API，方便访问表与查询的元数据。启用该选项的 API 如下：

  • sqlite3_column_database_name()
  • sqlite3_column_database_name16()
  • sqlite3_column_table_name()
  • sqlite3_column_table_name16()
  • sqlite3_column_origin_name()
  • sqlite3_column_origin_name16()

SQLITE_ENABLE_DBPAGE_VTAB

此选项启用SQLITE_DBPAGE 虚拟表。

SQLITE_ENABLE_DBSTAT_VTAB

此选项启用dbstat 虚拟表。

SQLITE_ENABLE_DESERIALIZE

此选项曾用于启用sqlite3_serialize()与sqlite3_deserialize()接口。然而，从 SQLite 3.36.0（2021-06-18）起，这些接口默认启用，并新增编译时选项SQLITE_OMIT_DESERIALIZE以省略它们。

SQLITE_ENABLE_EXPLAIN_COMMENTS

此选项向 SQLite 添加额外逻辑，在EXPLAIN 输出中插入注释文本。这些额外注释会消耗更多内存，因此会让准备语句变大并略微变慢，因此默认和大多数应用中都关闭。但对于那些更注重 EXPLAIN 输出清晰度而非纯性能的应用（如命令行工具），此编译时选项非常实用。若启用了SQLITE_DEBUG，则也会自动启用 SQLITE_ENABLE_EXPLAIN_COMMENTS。

SQLITE_ENABLE_FTS3

当在amalgamation中定义此选项时，会自动将全文搜索引擎的 3 与 4 版本包含进构建。

SQLITE_ENABLE_FTS3_PARENTHESIS

此选项修改 FTS3 中查询模式解析器，以支持 AND 与 NOT 运算符（除了通常的 OR 与 NEAR），并允许查询表达式包含嵌套括号。

SQLITE_ENABLE_FTS3_TOKENIZER

此选项启用两参数版本的fts3_tokenizer()接口。第二个参数应为指向函数的指针（以 BLOB 形式编码），该函数实现应用程序定义的分词器。如果恶意行为者能够以任意第二个参数调用两参数版本的 fts3_tokenizer()，可能会导致进程崩溃或被控制。

出于安全考虑，自3.11.0 版本（2016-02-15）起，除非使用此编译时选项，否则两参数版本的 fts3_tokenizer() 会被禁用。 3.12.0 版本（2016-03-29）新增了sqlite3_db_config(db,SQLITE_DBCONFIG_ENABLE_FTS3_TOKENIZER,1,0)接口，可在运行时为特定数据库连接启用两参数版本的fts3_tokenizer()。

SQLITE_ENABLE_FTS4

当在amalgamation中定义此选项时，会自动将全文搜索引擎的 3 与 4 版本包含。

SQLITE_ENABLE_FTS5

当在amalgamation中定义此选项时，会自动将全文搜索引擎第 5 版（fts5）包含。

SQLITE_ENABLE_GEOPOLY

当在amalgamation中定义此选项时，会包含Geopoly 扩展。

SQLITE_ENABLE_HIDDEN_COLUMNS

当在amalgamation中定义此选项时，会为虚拟表启用隐藏列功能。

SQLITE_ENABLE_ICU

此选项会将国际 Unicode 组件（ICU）扩展添加到构建中。

SQLITE_ENABLE_IOTRACE

如果 SQLite 核心与命令行界面（CLI）都以此选项编译，则 CLI 会提供名为 ".iotrace" 的额外命令，记录底层 I/O 活动日志。此选项处于实验阶段，未来版本可能会移除。

SQLITE_ENABLE_JSON1

该编译时选项现在是空操作。在 SQLite 3.38.0（2022-02-22）之前，必须使用此选项才能在构建中包含JSON SQL 函数。但从 3.38.0 起，这些函数默认包含。若要省略这些函数，请使用-DSQLITE_OMIT_JSON。

SQLITE_ENABLE_LOCKING_STYLE

此选项在 macOS 的操作系统接口层添加额外逻辑，尝试检测底层文件系统类型并选择适合的锁定策略。可用的五种锁定策略如下：

  • POSIX 锁定风格。默认的锁定风格，其他（非 macOS）Unix 使用 fcntl() 系统调用获取与释放锁。

  • AFP 锁定风格。用于采用 AFP（Apple Filing Protocol）协议的网络文件系统。通过调用 _AFPFSSetLock() 获取锁。

  • Flock 锁定风格。用于不支持 POSIX 锁定风格的文件系统。通过 flock() 系统调用获取与释放锁。

  • Dot-file 锁定风格。当文件系统既不支持 flock 也不支持 POSIX 锁定风格时使用。通过在数据库文件附近（即“点文件”）创建条目获取数据库锁，然后通过删除该文件释放锁。

  • 无锁定风格。如果上述任何方式均不可支持，则采用此风格，不使用数据库锁。这种情况下不适合由多个客户端同时访问单个数据库。

此外，还提供了五个额外的VFS 实现与默认实现一同提供。通过在调用sqlite3_open_v2() 时指定其中之一，应用可以跳过文件系统检测逻辑而直接选择上述锁定风格之一。这五个额外的VFS 分别叫做 "unix-posix"、"unix-afp"、"unix-flock"、"unix-dotfile" 与 "unix-none"。

SQLITE_ENABLE_MATH_FUNCTIONS

此宏启用了内建 SQL 数学函数。在 Unix 平台上，除非 configure 脚本使用 --disable-math，否则会自动将此选项加入 Makefile。Windows 平台上的 nmake Makefile.msc 也会包含该选项。

SQLITE_ENABLE_MEMORY_MANAGEMENT

此选项为 SQLite 添加额外逻辑，使其可以按需释放未使用的内存。启用此选项后，sqlite3_release_memory() 接口才能工作。如果未使用此编译时选项，则sqlite3_release_memory() 接口是空操作。

SQLITE_ENABLE_MEMSYS3

此选项使 SQLite 包含实现另一种内存分配器的代码。当通过sqlite3_config() 的SQLITE_CONFIG_HEAP 选项提供一大块内存供所有内存分配使用时，该替代分配器才会启用。MEMSYS3 内存分配器采用类似 dlmalloc() 的混合算法。SQLITE_ENABLE_MEMSYS3 与 SQLITE_ENABLE_MEMSYS5 不能同时启用。

SQLITE_ENABLE_MEMSYS5

此选项使 SQLite 包含另一种内存分配器的实现。当通过sqlite3_config() 的SQLITE_CONFIG_HEAP 选项提供大块内存时，该分配器才会启动。MEMSYS5 模块会将所有分配向上舍入到下一个 2 的幂，并采用先适应、伙伴分配算法，以便在特定操作约束下提供强防碎片与分配失败能力。

SQLITE_ENABLE_NORMALIZE

此选项包含sqlite3_normalized_sql() API。

SQLITE_ENABLE_NULL_TRIM

此选项启用一个优化，会省略行末尾的 NULL 列以节省磁盘空间。

使用此选项生成的数据库无法被 SQLite 3.1.6（2005-03-17）及更早版本读取。此外，它还可能触发e6e962d6b0f06f46 bug，这个 bug 存在于sqlite3_blob_reopen() 接口中。因此，默认情况下禁用了此优化。但未来某些版本可能会默认启用该优化。

SQLITE_ENABLE_OFFSET_SQL_FUNC

此选项启用sqlite_offset(X) SQL 函数。

sqlite_offset(X) 需要对 B-树存储引擎提供新的接口、在运行 SQL 的虚拟机中新增操作码、以及在代码生成器的关键路径中添加条件判断。为了避免在不需要 sqlite_offset(X) 的应用中引入这些开销，该函数默认被禁用。

SQLITE_ENABLE_ORDERED_SET_AGGREGATES

此选项启用了标准 SQL 中的 “WITHIN GROUP ORDER BY” 有序集合聚合语法。例如： 

          SELECT percentile_cont(0.75) WITHIN GROUP (ORDER BY x) FROM tab;

以上是 SQL 标准中计算分布第 75 百分位值的方式。在 SQLite 中更常见的写法如下： 

          SELECT percentile_cont(x,0.75) FROM tab;

虽更简洁、更易读写，但该写法并不符合 SQL 标准。只要编译时使用 -DSQLITE_ENABLE_ORDERED_SET_AGGREGATES，就可以使用标准语法。该语法默认禁用，原因在于（A）它在编译后的 SQLite 二进制中增加约 1200 字节代码，且（B）会使 SQL 代码更难阅读、编写及理解，而并无额外价值。仅在需要编写使用有序集合聚合函数（如percentile_cont()）的跨平台 SQL 时再启用此选项。

启用此编译时选项会将 WITHIN 作为关键字。

只有具有SQLITE_SELFORDER1 属性的聚合函数才能使用有序集合聚合语法。如果未指定 SQLITE_ENABLE_ORDERED_SET_AGGREGATES 编译时选项，则该属性基本无效。

由于此编译时选项会修改 LALR(1) 解析器表，因此仅在从规范源代码构建时有效。无法用于更改amalgamation 的行为。

SQLITE_ENABLE_PERCENTILE

此选项启用percentile 扩展，该扩展自 SQLite 3.51.0（2025-11-04）起已集成于amalgamation。在此版本之前，percentile 扩展位于独立源码文件中，需要单独编译并作为可加载扩展链接。

SQLITE_ENABLE_PREUPDATE_HOOK

此选项启用一组新的 API，在对rowid 表进行任何更改前提供回调。回调可用于记录更改前行的状态。

preupdate 钩子的行为类似于update hook，但回调在更改发生前调用，而且在未启用此编译时选项时 preupdate 接口会被省略。

preupdate 钩子最初是为支持session 扩展而添加的。

SQLITE_ENABLE_QPSG

此选项使查询规划器稳定性保证（QPSG）默认开启。通常 QPSG 默认关闭，必须在运行时通过SQLITE_DBCONFIG_ENABLE_QPSG 选项配合sqlite3_db_config() 接口启用。

SQLITE_ENABLE_RBU

启用实现RBU 扩展的代码。

SQLITE_ENABLE_RTREE

此选项使 SQLite 包含对R*Tree 索引扩展的支持。

SQLITE_ENABLE_SESSION

此选项启用session 扩展。

SQLITE_ENABLE_SNAPSHOT

此选项启用对sqlite3_snapshot 对象及其相关接口的支持：

  • sqlite3_snapshot_get()（构造函数）
  • sqlite3_snapshot_free()（析构函数）
  • sqlite3_snapshot_open()
  • sqlite3_snapshot_cmp()
  • sqlite3_snapshot_recover()

SQLITE_ENABLE_SORTER_REFERENCES

此选项启用一种优化，可以在排序完成后通过增加 B 树查找次数来减少排序器所需的内存。

默认的排序过程会将最终输出所需的所有信息收集到一个“记录”中，然后将完整记录传给排序器。但某些情况下（例如部分输出列为大型 BLOB 值），每条记录会非常大，从而迫使排序器使用更多内存或将更多内容写入临时存储。

启用SQLITE_ENABLE_SORTER_REFERENCES后，传给排序器的记录通常仅包含一个ROWID。这样的记录小得多，排序器处理的“负载”更少，运行更快。排序后，会使用 ROWID 在原始表中查找输出列值，这就需要额外一次查找，可能导致性能下降，也可能因值较大而获益。

即便启用了该编译时选项，排序引用仍默认禁用。要使用它，应用必须在启动时通过sqlite3_config(SQLITE_CONFIG_SORTERREF_SIZE) 接口设置排序引用大小阈值。

由于 SQLite 开发者不确定该选项对性能的影响，因此截至 2018-05-04 仍默认关闭。未来某个版本可能会默认开启。

SQLITE_ENABLE_STMT_SCANSTATUS

此选项启用sqlite3_stmt_scanstatus()与sqlite3_stmt_scanstatus_v2()接口，并激活nexec 与 ncycle列的字节码虚拟表。这些功能通常因会带来运行时性能损耗而被省略或禁用。

由于即便未使用性能统计数据也会产生额外开销，SQLITE_ENABLE_STMT_SCANSTATUS 不推荐用于生产构建。但它对诊断与调试构建非常有用。

SQLITE_ENABLE_STMTVTAB

此编译时选项启用SQLITE_STMT 虚拟表逻辑。

SQLITE_RTREE_INT_ONLY

此编译时选项已弃用且未经测试。

SQLITE_ENABLE_SQLLOG

此选项启用额外代码（特别是SQLITE_CONFIG_SQLLOG），可用于记录应用执行的所有 SQLite 操作。这些日志对应用行为的离线分析、尤其是性能分析十分有用。要使 SQLITE_ENABLE_SQLLOG 选项发挥作用，需要一些额外代码。SQLite 源码树中的test_sqllog.c 文件就是一个示例。在 Unix 与 Windows 系统上，开发者可以将 test_sqllog.c 文件的内容追加到 sqlite3.c amalgamation 的末尾，使用 -DSQLITE_ENABLE_SQLLOG 重新编译应用，然后通过环境变量控制日志记录。更多细节见 test_sqllog.c 源文件头部注释。

SQLITE_ENABLE_STAT2

此选项曾使ANALYZE 命令收集索引直方图数据到 sqlite_stat2 表中。但从 SQLite 3.7.9 版本（2011-11-01）起，该功能被SQLITE_ENABLE_STAT3 所取代。SQLITE_ENABLE_STAT2 现在是空操作。

SQLITE_ENABLE_STAT3

此选项曾使ANALYZE 命令收集索引直方图数据到 sqlite_stat3 表中。但从 SQLite 3.8.1 版本（2013-10-17）起，该功能被SQLITE_ENABLE_STAT4 所取代。SQLITE_ENABLE_STAT3 编译时选项继续支持至3.29.0 版本（2019-07-10），现已成为空操作。

SQLITE_ENABLE_STAT4

此选项为ANALYZE 命令与查询规划器添加额外逻辑，可在某些情况下帮助 SQLite 选择更好的查询计划。ANALYZE 命令会收集每个索引的所有列的直方图数据，并将其存储在sqlite_stat4 表中。查询规划器会使用这些直方图数据做出更好的索引选择。此编译时选项的缺点是它违反了查询规划器稳定性保证，使得在大规模生产环境中难以确保持久的性能。

SQLITE_ENABLE_STAT4 是SQLITE_ENABLE_STAT3的增强版。STAT3 仅记录每个索引最左侧列的直方图数据，而 STAT4 会记录每个索引所有列的直方图数据。 SQLITE_ENABLE_STAT3 编译时选项已成为空操作。

SQLITE_ENABLE_TREE_EXPLAIN

此编译时选项现已不再使用。

SQLITE_ENABLE_UPDATE_DELETE_LIMIT

此选项为UPDATE与DELETE 语句启用可选的 ORDER BY 与 LIMIT 子句。

若定义此选项，则在使用Lemon 解析器生成器生成 parse.c 文件时，也必须定义该选项。基于此原因，该选项仅可在从源码构建库时使用，不能用于amalgamation或网站上为非 Unix 平台提供的预打包 C 文件集合。

SQLITE_ENABLE_UNKNOWN_SQL_FUNCTION

启用此编译时选项后，当执行EXPLAIN或EXPLAIN QUERY PLAN且遇到“未知函数”时，SQLite 会抑制错误。取而代之，它会插入一个名为 "unknown()" 的占位无效函数。该替换仅在EXPLAIN与EXPLAIN QUERY PLAN 中发生，而不会影响普通语句。

在命令行工具中启用该功能后，含有应用程序定义函数的 SQL 文本可以粘贴到 shell 中进行分析与调试，而无需先创建并加载实现这些函数的扩展。

SQLITE_ENABLE_UNLOCK_NOTIFY

此选项启用sqlite3_unlock_notify() 接口及其相关功能。更多信息请参见标题为使用 SQLite 解锁通知特性的文档。

SQLITE_INTROSPECTION_PRAGMAS

此选项已废弃。它曾启用一些附加的 PRAGMA 语句，例如PRAGMA function_list、PRAGMA module_list与PRAGMA pragma_list，但这些 PRAGMA 现在都默认启用。参见SQLITE_OMIT_INTROSPECTION_PRAGMAS。

SQLITE_SOUNDEX

此选项启用soundex() SQL 函数。

SQLITE_STRICT_SUBTYPE=1

此选项会让应用程序定义的 SQL 函数在调用sqlite3_result_subtype()接口但未注册SQLITE_RESULT_SUBTYPE属性时抛出 SQL 错误。此推荐选项有助于在开发早期识别应用程序定义函数实现中的问题。

SQLITE_USE_ALLOCA

启用此选项将使 alloca() 内存分配器在合适的场合被使用，从而生成稍小、稍快的二进制文件。当然，此编译时选项只在支持 alloca() 的系统上有效。

SQLITE_USE_FCNTL_TRACE

此选项会使 SQLite 发送额外的SQLITE_FCNTL_TRACE 文件控制命令，为 VFS 提供补充信息。扩展 "vfslog.c" 便利用此信息提供增强的 VFS 活动日志。

SQLITE_USE_SEH

此选项曾用来切换现在由SQLITE_OMIT_SEH控制的行为。客户端代码不应使用此定义，因为它仅供库内部使用。

SQLITE_HAVE_ZLIB

此选项使某些扩展链接到zlib 压缩库。

此选项对 SQLite 核心无影响，仅为扩展使用。它是支持SQL Archive 中压缩与解压函数（用于命令行工具）所必需的。

使用此选项编译时，通常需要添加链接器选项以包含 zlib 库，通常为 "-lz"，但在不同系统上可能不同。

在 Windows 上使用 MSVC 构建时，可将 zlib 源码放入源树中的 compat/zlib 子目录，然后在 nmake 命令中添加 USE_ZLIB=1，使 Makefile.msc 自动构建并使用合适的 zlib 库。

YYTRACKMAXSTACKDEPTH

此选项会跟踪并通过sqlite3_status()(SQLITE_STATUS_PARSER_STACK,...) 接口报告 LALR(1) 解析器的栈深度。SQLite 的 LALR(1) 解析器具有固定栈深度（通过YYSTACKDEPTH 在编译时确定）。此选项可用于判断应用是否接近超过最大栈深度。

8. 禁用默认开启功能的选项

SQLITE_DISABLE_LFS

定义此 C 预处理宏后，会禁用大文件支持。

SQLITE_DISABLE_DIRSYNC

定义此 C 预处理宏后，会禁用目录同步。SQLite 通常在删除文件后尝试同步其父目录，以确保目录条目立即写入磁盘。

SQLITE_DISABLE_FTS3_UNICODE

定义此 C 预处理宏后，unicode61 分词器在FTS3 中将不包含在构建中，应用无法使用。

SQLITE_DISABLE_FTS4_DEFERRED

定义此宏后，会禁用FTS4 中的“延迟令牌”优化。该优化避免了为出现在集合中大多数文档中的词项加载庞大的帖子列表，而改为在文档源中扫描这些令牌。FTS4无论是否使用该优化，应该都能得到相同的结果。

SQLITE_DISABLE_INTRINSIC

此选项禁用对 GCC 和 Clang 中的 __builtin_bswap32() 与 __builtin_add_overflow()、或 MSVC 中的 _byteswap_ulong() 与 _ReadWriteBarrier() 等编译器专用内建函数的使用。

SQLITE_DISABLE_PAGECACHE_OVERFLOW_STATS

此选项禁用对sqlite3_status() 中SQLITE_STATUS_PAGECACHE_OVERFLOW 与 SQLITE_STATUS_PAGECACHE_SIZE 统计信息的收集。在高并发多线程应用中，设置此选项已被证明能够提高性能。

9. 省略功能的选项

以下选项可以用来通过省略未使用的功能来减小编译库体积。此类做法在空间尤其紧张的嵌入式系统中最有用，因为即便包含所有功能，SQLite 库本身也相对较小。不要忘记让编译器优化二进制大小！（GCC 使用 -Os 选项）。相比使用这些编译时选项，让编译器进行大小优化通常对库体积的影响更大。还请确认调试选项已关闭。

本节中的宏不需要赋值。下列编译开关效果相同：
-DSQLITE_OMIT_ALTERTABLE
-DSQLITE_OMIT_ALTERTABLE=1
-DSQLITE_OMIT_ALTERTABLE=0

如果定义了任何这些选项，则在使用Lemon 解析器生成工具生成 parse.c 文件以及编译生成 keywordhash.h 的 mkkeywordhash 工具时，也必须定义相同的一组 SQLITE_OMIT_* 选项。因此，这些选项仅可在从规范源码构建库时使用，不能用于amalgamation。某些 SQLITE_OMIT_* 选项在与amalgamation搭配使用时可能有效或看似有效。但这并不保证可靠。通常建议始终从规范源代码编译，以便利用 SQLITE_OMIT_* 选项。

重要提示： SQLITE_OMIT_* 选项可能无法与amalgamation一起使用。SQLITE_OMIT_* 编译时选项通常仅在从规范源文件构建 SQLite 时才能正确工作。

可以生成特殊版本的 SQLite amalgamation，使其支持预定的 SQLITE_OMIT_* 选项。为此，请复制规范源码分发中的 Makefile.linux-gcc 模板，并将复制品重命名为“Makefile”。然后编辑“Makefile”以配置合适的编译时选项。最后输入： 

make clean; make sqlite3.c

生成的 "sqlite3.c" amalgamation 源代码文件（以及其关联的头文件 "sqlite3.h"）可移至非 Unix 平台，在该平台上使用本地编译器进行最终编译。

SQLITE_OMIT_* 选项为不受支持。就是说，当前版本中省略代码的 SQLITE_OMIT_* 选项在下一个版本中可能变为空操作。反之亦然：当前版本中是空操作的 SQLITE_OMIT_* 选项在下一版本中可能会导致代码被排除。此外，并非所有 SQLITE_OMIT_* 选项都经过测试。有些选项可能会导致 SQLite 出现故障或返回错误结果。

重要提示： SQLITE_OMIT_* 编译时选项大多不受支持。

以下为可用的 OMIT 选项：

SQLITE_OMIT_ALTERTABLE

定义此选项后，ALTER TABLE 命令不会被包含在库中。执行 ALTER TABLE 语句将导致语法错误。

SQLITE_OMIT_ANALYZE

定义此选项后，ANALYZE 命令会从构建中省略。

SQLITE_OMIT_ATTACH

定义此选项后，ATTACH 和 DETACH 命令会从构建中省略。

SQLITE_OMIT_AUTHORIZATION

定义此选项会使sqlite3_set_authorizer() API 函数及相关逻辑被省略。

SQLITE_OMIT_AUTOINCREMENT

此选项省略AUTOINCREMENT 功能。当定义了此宏时，声明为 "INTEGER PRIMARY KEY AUTOINCREMENT" 的列在插入 NULL 时会表现得与声明为 "INTEGER PRIMARY KEY" 的列相同。sqlite_sequence 系统表既不会创建，也不会在已存在的情况下被尊重。

SQLITE_OMIT_AUTOINIT

为了向缺少sqlite3_initialize() 接口的旧版本 SQLite 保持兼容，某些关键接口（如sqlite3_open()、sqlite3_vfs_register() 与 sqlite3_mprintf()）入口处会自动调用sqlite3_initialize()。通过构建时定义 SQLITE_OMIT_AUTOINIT，可省略此自动初始化。在启用 SQLITE_OMIT_AUTOINIT 的构建中，SQLite 不会自动初始化，应用程序必须在使用 SQLite 库前手动调用sqlite3_initialize()。

SQLITE_OMIT_AUTOMATIC_INDEX

此选项用于省略自动索引功能。 另请参见：SQLITE_DEFAULT_AUTOMATIC_INDEX。

SQLITE_OMIT_AUTORESET

默认情况下，sqlite3_step() 接口会在必要时自动调用sqlite3_reset() 来重置准备语句。此编译时选项会改变该行为，使得如果 sqlite3_step() 在返回非 SQLITE_ROW、SQLITE_BUSY 或 SQLITE_LOCKED 后再次调用，而未进行中间的sqlite3_reset()，那么会返回SQLITE_MISUSE。

在 SQLite 3.6.23.1 版本（2010-03-26）及更早版本中，如果 sqlite3_step() 在返回非 SQLITE_ROW 且未经中间调用 sqlite3_reset() 的情况下再次被调用，会始终返回SQLITE_MISUSE。这导致某些编写不当的智能手机应用程序无法正确处理 SQLITE_LOCKED 与 SQLITE_BUSY 错误返回。为了避免修复大量有缺陷的智能手机应用，SQLite 在 3.6.23.2 中修改行为，使其自动重置准备语句。但该改动又让一些错误实现的应用依赖于 SQLITE_MISUSE 返回以终止查询循环而失效。（当应用从 SQLite 收到 SQLITE_MISUSE 错误时，表示应用在误用 SQLite 接口，存在缺陷。）因此，在 3.7.5 版本（2011-02-01）中添加了 SQLITE_OMIT_AUTORESET 接口，以便使所有（错误的）应用重新工作而无需修复它们。

SQLITE_OMIT_AUTOVACUUM

定义此选项后，库无法创建或写入支持auto_vacuum 的数据库。执行PRAGMA auto_vacuum 语句不会报错（因为未知 PRAGMA 会被静默忽略），但不会返回值或修改数据库文件中的 auto_vacuum 标志。如果打开一个支持 auto_vacuum 的数据库，而库是通过该选项编译的，则数据库将自动以只读模式打开。

SQLITE_OMIT_BETWEEN_OPTIMIZATION

定义此选项后，会禁止 SQLite 使用索引处理 WHERE 子句中包含 BETWEEN 运算符的条件。

SQLITE_OMIT_BLOB_LITERAL

定义此选项后，就无法在 SQL 语句中通过 X'ABCD' 语法指定 BLOB。

SQLITE_OMIT_BTREECOUNT

此选项不再用于任何用途，是空操作。

SQLITE_OMIT_BUILTIN_TEST

此编译时选项已重命名为SQLITE_UNTESTABLE。

SQLITE_OMIT_CASE_SENSITIVE_LIKE_PRAGMA

此编译时选项禁用PRAGMA case_sensitive_like 命令。

SQLITE_OMIT_CAST

此选项会让 SQLite 省略 CAST 运算符的支持。

SQLITE_OMIT_CHECK

此选项会让 SQLite 省略对 CHECK 约束的支持。解析器仍会接受 SQL 中的 CHECK 约束，但不会执行它们。

SQLITE_OMIT_COMPILEOPTION_DIAGS

此选项用于省略 SQLite 中可用的编译时选项诊断功能，包括sqlite3_compileoption_used()与sqlite3_compileoption_get() C/C++ 函数、sqlite_compileoption_used()与sqlite_compileoption_get() SQL 函数，以及compile_options pragma。

SQLITE_OMIT_COMPLETE

此选项会省略sqlite3_complete() 与 sqlite3_complete16() 接口。

SQLITE_OMIT_COMPOUND_SELECT

此选项用于省略复合SELECT 功能。使用 UNION、UNION ALL、INTERSECT 或 EXCEPT 复合 SELECT 操作符的语句会导致语法错误。

带有多个 VALUES 的INSERT 语句实际上是内部作为复合 SELECT 实现的。因此，此选项也会禁用 INSERT INTO ... VALUES ... 语句一次插入多行的能力。

SQLITE_OMIT_CTE

此选项会省略对公共表表达式的支持。

SQLITE_OMIT_DATETIME_FUNCS

定义此选项后，SQLite 内建的日期与时间函数会被省略。具体而言，julianday()、date()、time()、datetime() 与 strftime() 等 SQL 函数将不可用。但默认列值 CURRENT_TIME、CURRENT_DATE 与 CURRENT_TIMESTAMP 仍然可用。

SQLITE_OMIT_DECLTYPE

此选项会让 SQLite 省略sqlite3_column_decltype() 与 sqlite3_column_decltype16() 接口。

SQLITE_OMIT_DEPRECATED

此选项会让 SQLite 省略被标记为弃用的接口。这包括sqlite3_aggregate_count()、sqlite3_expired()、sqlite3_transfer_bindings()、sqlite3_global_recover()、sqlite3_thread_cleanup() 和 sqlite3_memory_alarm() 接口，以及PRAGMA 语句 PRAGMA count_changes、PRAGMA data_store_directory、PRAGMA default_cache_size、PRAGMA empty_result_callbacks、PRAGMA full_column_names、PRAGMA short_column_names 与 PRAGMA temp_store_directory。

SQLITE_OMIT_DESERIALIZE

此选项会让sqlite3_serialize() 与 sqlite3_deserialize() 接口从构建中省略。

SQLITE_OMIT_DISKIO

此选项会省略所有对磁盘写入的支持，并强制数据库仅存在于内存中。该选项未持续维护，可能与较新版 SQLite 不兼容。

SQLITE_OMIT_EXPLAIN

定义此选项会让EXPLAIN 命令从库中省略。尝试执行 EXPLAIN 语句将导致语法错误。

SQLITE_OMIT_FLAG_PRAGMAS

此选项省略用于查询与设置布尔属性的一部分PRAGMA 命令。

SQLITE_OMIT_FLOATING_POINT

此选项用于从 SQLite 库中省略浮点数支持。若启用此选项，在 SQL 语句中将浮点数字面量（如 "1.01"）列为参数会导致语法错误。

未来，此选项也有可能禁用其他浮点功能，例如sqlite3_result_double()、sqlite3_bind_double()、sqlite3_value_double() 以及 sqlite3_column_double() API 函数。

SQLITE_OMIT_FOREIGN_KEY

如果定义此选项，则不识别外键约束语法。

SQLITE_OMIT_GENERATED_COLUMNS

如果定义此选项，则不识别生成列语法。

SQLITE_OMIT_GET_TABLE

此选项会省略sqlite3_get_table() 与 sqlite3_free_table() 的支持。

SQLITE_OMIT_HEX_INTEGER

此选项省略对十六进制整数字面量 的支持。

SQLITE_OMIT_INCRBLOB

此选项会省略对增量 BLOB I/O 的支持。

SQLITE_OMIT_INTEGRITY_CHECK

此选项省略对integrity_check pragma 的支持。

SQLITE_OMIT_INTROSPECTION_PRAGMAS

此选项省略对PRAGMA function_list、PRAGMA module_list 和PRAGMA pragma_list 的支持。

SQLITE_OMIT_JSON

此选项从构建中省略JSON SQL 函数。

SQLITE_OMIT_LIKE_OPTIMIZATION

此选项禁用 SQLite 使用索引协助解析 WHERE 子句中包含LIKE 和GLOB 运算符的能力。

SQLITE_OMIT_LOAD_EXTENSION

此选项从 SQLite 中省略整个扩展加载机制，包括sqlite3_enable_load_extension() 与sqlite3_load_extension() 接口。

SQLITE_OMIT_LOCALTIME

此选项会省略日期与时间函数中的 “localtime” 修饰符。当尝试在不支持本地时间概念的平台上编译日期与时间函数时，该选项有时会非常有用。

SQLITE_OMIT_LOOKASIDE

此选项会省略lookaside 内存分配器。

SQLITE_OMIT_MEMORYDB

定义此选项后，库将不再识别特殊数据库名 ":memory:"（通常用于创建内存数据库）。若将 ":memory:" 传给sqlite3_open()、sqlite3_open16() 或 sqlite3_open_v2()，则会打开或创建名为 ":memory:" 的文件。

SQLITE_OMIT_OR_OPTIMIZATION

此选项禁用 SQLite 将索引与 WHERE 子句中由 OR 运算符连接的条件结合使用的能力。

SQLITE_OMIT_PAGER_PRAGMAS

定义此选项会从构建中省略与分页器子系统相关的 PRAGMA。

SQLITE_OMIT_PRAGMA

此选项用于从库中省略PRAGMA 命令。需注意，还应定义省略具体 PRAGMA 的宏，因为它们可能还会删除其他子系统所依赖的代码。此宏仅删除 PRAGMA 命令本身。

SQLITE_OMIT_PROGRESS_CALLBACK

定义此选项会省略sqlite3_progress_handler() API 及相关逻辑。

SQLITE_OMIT_QUICKBALANCE

此选项省略一种替代的、更快的 B 树平衡例程。启用此选项会让 SQLite 稍小一些，但运行略慢一些。

SQLITE_OMIT_REINDEX

定义此选项后，REINDEX 命令不会包含在库中。尝试执行 REINDEX 语句会导致语法错误。

SQLITE_OMIT_SCHEMA_PRAGMAS

定义此选项会从构建中省略用于查询数据库模式的 PRAGMA。

SQLITE_OMIT_SCHEMA_VERSION_PRAGMAS

定义此选项会从构建中省略用于查询和修改数据库模式版本与用户版本的 PRAGMA。具体而言，会省略schema_version与user_version PRAGMA。

SQLITE_OMIT_SHARED_CACHE

此选项会构建一个不支持共享缓存模式的 SQLite。sqlite3_enable_shared_cache() 会被省略，并且与共享缓存管理相关的 B 树子系统中相当一部分逻辑也会被省略。

由于此编译时选项可以提升性能并减小库体积，因此推荐大多数应用启用。

SQLITE_OMIT_SEH

定义此选项会在 Windows 构建中禁用结构化异常处理（SEH）。SEH 是 Windows 上用于捕获访问内存映射文件时产生的异常的技术。SEH 用于拦截访问shm 文件（WAL 模式处理的一部分）时操作系统抛出的错误。当操作系统在 SQLite 访问 shm 文件时抛出错误，该选项会让这些错误由 SQLite 捕获并处理，而不是终止整个进程。

仅在使用 MSVC 在 Windows 上编译时才有作用。

SQLITE_OMIT_SUBQUERY

定义此选项会省略子查询与 IN() 运算符的支持。

SQLITE_OMIT_TCL_VARIABLE

定义此宏后，用于自动将 SQL 变量绑定到 TCL 变量的 "$" 语法将被省略。

SQLITE_OMIT_TEMPDB

此选项会省略对 TEMP 或 TEMPORARY 表的支持。

SQLITE_OMIT_TRACE

此选项会省略sqlite3_profile()与sqlite3_trace() 接口及其相关逻辑。

SQLITE_OMIT_TRIGGER

定义此选项会省略 TRIGGER 对象的支持。CREATE TRIGGER 与 DROP TRIGGER 命令将不可用，尝试执行会导致语法错误。此外，由于实现触发器的代码也用于执行外键操作，因此启用该选项也会禁用外键约束的执行。

SQLITE_OMIT_TRUNCATE_OPTIMIZATION

此选项会省略truncate 优化。

SQLITE_OMIT_UTF16

此宏用于省略 UTF16 文本编码的支持。启用后，所有以 '16' 结尾的 API 函数（例如 sqlite3_prepare16()、sqlite3_column_text16() 与 sqlite3_bind_text16()）都不可用。

SQLITE_OMIT_VACUUM

定义此选项后，VACUUM 命令不会包含在库中。尝试执行 VACUUM 语句会导致语法错误。

SQLITE_OMIT_VIEW

定义此选项会省略 VIEW 对象的支持。CREATE VIEW 与DROP VIEW 命令都将不可用，执行任一命令都会导致语法错误。

警告：如果定义此宏，则无法打开包含 VIEW 对象的数据库。

SQLITE_OMIT_VIRTUALTABLE

此选项省略对虚拟表机制的支持。

SQLITE_OMIT_WAL

此选项省略 "write-ahead log"（即 "WAL"） 功能。

SQLITE_OMIT_WINDOWFUNC

此选项会将窗口函数从构建中移除。

SQLITE_OMIT_WSD

此选项构建一个不包含可写静态数据（WSD）的 SQLite 库。WSD 指全局变量和/或静态变量。一些平台不支持 WSD，因此必须启用该选项以让 SQLite 在这些平台上运行。

与其他会缩小库体积的 OMIT 选项不同，此选项实际上会增大 SQLite 的体积并让其运行稍慢。仅当构建目标为不支持 WSD 的嵌入式平台时，才应使用。

SQLITE_OMIT_XFER_OPT

此选项省略对帮助加速类似 "INSERT INTO ... SELECT ..." 语句的优化。

SQLITE_UNTESTABLE

标准 SQLite 构建包含与sqlite3_test_control() 相关的少量逻辑，用以验证部分原本难以测试的核心。此编译时选项会省略这些额外的测试逻辑。在 SQLite 3.16.0（2017-01-02）之前，该选项被称为 "SQLITE_OMIT_BUILTIN_TEST"。更名是为了更明确地描述其影响。

启用此选项会使 SQLite 变得无法完全测试。分支测试覆盖率将从 100% 降至约 95%。

SQLite 开发者遵循“先测试所飞行的，后飞行所测试的”原则。如果该选项在交付时启用却在测试时禁用，就违反了此原则。但如果在测试过程中启用此选项，则无法覆盖所有分支。因此，不推荐在交付构建中启用此编译时选项。

SQLITE_ZERO_MALLOC

此选项会从构建中省略默认内存分配器与调试内存分配器，改为使用始终失败的桩内存分配器。SQLite 无法在该桩分配器下运行，因为它无法申请内存。但可以在启动时通过sqlite3_config(SQLITE_CONFIG_MALLOC,...) 或 sqlite3_config(SQLITE_CONFIG_HEAP,...) 来替换此桩分配器。因此，此选项的整体效果是允许 SQLite 链接至不支持 malloc()、free() 或 realloc() 的系统库。

10. 分析和调试选项

SQLITE_DEBUG

SQLite 源代码包含成千上万个 assert() 语句，用于验证内部假设以及子例程的前置与后置条件。这些 assert() 通常被关闭（不生成代码），因为启用它们会让 SQLite 的运行速度下降约三倍。但对于测试与分析工作，启用 assert() 是有价值的。SQLITE_DEBUG 编译时选项即是用于此目的。

SQLITE_DEBUG 还启用其他调试功能，例如用于排查与分析VDBE和代码生成器的特殊PRAGMA 语句。

SQLITE_MEMDEBUG

SQLITE_MEMDEBUG 选项会让 SQLite 默认使用经过仪表化的调试内存分配器。该分配器会检查动态内存的滥用情况，例如释放后继续使用、越界写入、释放非分配得到的内存、或未初始化新分配内存等。

11. Windows 特定选项

SQLITE_WIN32_HEAP_CREATE

启用此选项会强制 Win32 原生内存分配器在启用时创建私有堆，用以保存所有内存分配。

SQLITE_WIN32_MALLOC_VALIDATE

启用此选项会强制 Win32 原生内存分配器在启用时调用 HeapValidate() 函数，前提是 assert() 同时启用。

12. 编译器链接和调用约定控制

以下宏指定某些 SQLite 构建的接口细节。Makefile 通常会自动处理这些宏。应用开发者通常无需关心这些宏。以下有关这些宏的文档仅为完整性而提供。

SQLITE_API

此宏标识 SQLite 的外部可见接口。此宏有时被定义为 "extern"，但定义方式依赖于编译器。

SQLITE_APICALL

此宏标识 SQLite 中接受固定参数数量的公共接口函数所使用的调用约定。通常此宏定义为空，不过在 Windows 上有时会被设为 "__cdecl" 或 "__stdcall"。默认是 "__cdecl"，但当 SQLite 被期望编译为 Windows 系统库时会使用 "__stdcall"。

单个函数声明最多只能包含下列之一：SQLITE_APICALL、SQLITE_CDECL 或 SQLITE_SYSAPI。

SQLITE_CALLBACK

此宏指定在 SQLite 中使用的回调指针调用约定。此宏通常定义为空，但在 Windows 构建中有时会设为 "__cdecl" 或 "__stdcall"。默认是 "__cdecl"，但当 SQLite 被期望编译为 Windows 系统库时会使用 "__stdcall"。

SQLITE_CDECL

此宏指定 SQLite 中变参接口函数所使用的调用约定。此宏通常定义为空，但在 Windows 构建中有时会设为 "__cdecl"。由于 __stdcall 不支持变参函数，因此此宏不能设为 "__stdcall"。

单个函数声明最多只能包含下列之一：SQLITE_APICALL、SQLITE_CDECL 或 SQLITE_SYSAPI。

SQLITE_EXTERN

此宏指定 SQLite 公共接口变量的链接方式。通常允许其默认设为 "extern"。

SQLITE_STDCALL

此宏现已弃用，不再使用。

SQLITE_SYSAPI

此宏标识用于目标平台 SQLite 构建的操作系统接口的调用约定。此宏通常定义为空，在 Windows 构建中有时会设为 "__stdcall"。

单个函数声明最多只能包含下列之一：SQLITE_APICALL、SQLITE_CDECL 或 SQLITE_SYSAPI。

SQLITE_TCLAPI

此宏指定TCL 库接口例程所采用的调用约定。此宏不被 SQLite 核心使用，仅用于TCL 接口与TCL 测试套件。 此宏通常定义为空，但在 Windows 构建中有时会设为 "__cdecl"。该宏用于 TCL 库接口例程，而这些例程总是以 __cdecl 编译，即便在偏好 __stdcall 的平台上也如此，因此除非平台提供支持 __stdcall 的自定义 TCL 库，否则此宏不应设为 __stdcall。

不得将此宏与任何一个SQLITE_APICALL、SQLITE_CALLBACK、SQLITE_CDECL 或 SQLITE_SYSAPI 同时使用。
