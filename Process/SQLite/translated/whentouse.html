SQLite 的适当用途
目录
1. SQLite 适用的情况
2. 更适合使用客户/服务器 RDBMS 的情况
3. 选择合适数据库引擎的核对清单

SQLite 无法与 MySQL、Oracle、PostgreSQL 或 SQL Server 等客户/服务器 SQL 数据库引擎直接相比， 因为 SQLite 试图解决的是一个不同的问题。

客户/服务器 SQL 数据库引擎努力实现企业数据的共享存储库。 它们强调可扩展性、并发性、集中化与控制。 SQLite 力求为单个应用程序和设备提供本地数据存储。 SQLite 强调经济性、效率、可靠性、独立性与简单性。

SQLite 不与客户/服务器数据库竞争。 SQLite 与 fopen() 竞争。

1. SQLite 适用的情况

  • 嵌入式设备与物联网

    因为 SQLite 数据库无需管理， 所以它非常适合必须在没有专家支持的情况下运行的设备。 SQLite 非常适合用于 手机、机顶盒、电视、游戏机、 相机、手表、厨房电器、恒温器、汽车、 机床、飞机、远程传感器、无人机、医疗设备 和机器人等“物联网”设备。

    客户/服务器数据库引擎设计之初就是在网络核心的 精心照料的数据中心内部运行。 SQLite 在那里也可以运行，但它也能在网络边缘蓬勃发展， 在自己照顾自己的同时向那些连接不稳定的应用程序 提供快速且可靠的数据服务。

  • 应用程序文件格式

    SQLite 经常被用作桌面应用程序的磁盘文件格式， 如版本控制系统、财务分析工具、多媒体编目与编辑套件、CAD 软件、记录保存程序等等。传统的“文件/打开”操作调用 sqlite3_open() 以附加到数据库 文件。应用内容更新时自动发生更改，因此“文件/保存”菜单选项就显得多余。 “文件/另存为”菜单选项可通过 备份 API 实现。

    这种方法有很多好处，包括性能提升、成本与复杂度降低，以及 可靠性增强。 有关更多信息，请参见技术说明 “aff_short.html”、 “appfileformat.html” 和 “fasterthanfs.html”。 此用例与下述 数据传输格式 和 数据容器 用例密切相关。

  • 网站

    SQLite 作为大多数中低流量网站的数据库引擎（也就是大多数网站） 运行良好。SQLite 能处理多少网页流量取决于网站使用数据库的程度。 一般而言，访问量低于每日 10 万次的网站应该可以很好地使用 SQLite。 每天 10 万次访问是一个保守估计，而非严格上限。 SQLite 已被实验证明能够应对十倍的流量。

    当然，SQLite 网站（https://sqlite.org/）本身就是使用 SQLite， 截至本文撰写（2015 年）每天处理约 40 万至 50 万个 HTTP 请求， 其中约 15%~20% 是涉及数据库的动态页面。 动态内容每个网页约使用 200 条 SQL 语句。 该配置运行在一台与其他 23 台共享物理服务器的单一虚拟机上， 但大多数时间负载平均值仍保持在 0.1 以下。

    另请参阅：2022-12-13 的 Hacker News 讨论。

  • 数据分析

    懂得 SQL 的人士可以使用 sqlite3 命令行外壳（或各类第三方 SQLite 访问程序）来分析大型数据集。 原始数据可以从 CSV 文件导入，然后对这些数据进行切片和分组以生成大量汇总报告。 更复杂的分析可以使用内置组件 Tcl 或 Python 编写的简单脚本完成， 也可以通过 R 或其他语言使用可用的适配器完成。 可能的用途包括网站日志分析、体育统计分析、编程指标汇总以及实验结果分析。 许多生物信息学研究人员也以此方式使用 SQLite。

    当然，用企业级客户/服务器数据库也能实现同样的事情。 SQLite 的优势在于更容易安装与使用，且生成的数据库是一个单一文件， 可以写入 USB 存储器或发送给同事。

  • 企业数据缓存

    许多应用程序使用 SQLite 作为企业 RDBMS 的相关内容缓存。 这减少了延迟，因为大多数查询都在本地缓存中执行，而无需进行网络往返。 它还降低了网络和中央数据库服务器的负载。 在许多情况下，它意味着客户端应用程序在网络中断期间仍能继续运行。

  • 服务器端数据库

    系统设计师报告称，在数据中心运行的服务器应用中将 SQLite 用作数据存储很成功， 换句话说，将 SQLite 用作特定应用的数据库服务器的底层存储引擎。

    在此模式下，整个系统仍为客户/服务器： 客户端通过网络向服务器发送请求并接收回复。 但客户端发送的请求与服务器返回的响应是高层且特定于应用的， 而非发送通用 SQL 并返回原始表内容。 服务器将请求翻译成多个 SQL 查询，收集结果，进行后处理、过滤与分析， 然后构造只包含必要信息的高层回复。

    开发人员报告称，在这种场景下 SQLite 往往比客户/服务器 SQL 数据库引擎更快。 数据库请求由服务器串行化，因此并发不是问题。 通过“数据库分片”也可改善并发性： 对不同子域使用独立的数据库文件。 例如，服务器可能为每位用户都设置一个单独的 SQLite 数据库， 这样服务器就可以处理数百或数千个并发连接， 但每个 SQLite 数据库仅被单个连接使用。

  • 数据传输格式

    由于 SQLite 数据库是一个遵循 良好定义的跨平台格式 的紧凑单一文件， 它常被用作从一个系统向另一个系统传输内容的容器。 发送方将内容收集到一个 SQLite 数据库文件中，传输该文件给接收方，然后接收方根据需要使用 SQL 提取内容。

    SQLite 数据库支持即使端点具有不同字长与字节序的系统之间的数据传输。 数据可以是大型二进制 BLOB、文本以及小型数值或布尔值的复杂混合体。 通过添加新表和/或列即可轻松扩展数据格式，而不会破坏旧的接收方。 SQL 查询语言意味着接收方无需一次性解析整个传输内容， 而可以根据需要查询接收到的内容。 该数据格式具有“透明性”，可以使用来自多个厂商的各种通用开源工具 轻松对其进行解码以供人工查看。

  • 文件归档与/或数据容器

    SQLite 归档 思路展示了如何使用 SQLite 替代 ZIP 归档或 Tar 包。 存储在 SQLite 中的文件归档仅略大， 在某些情况下甚至比等效的 ZIP 归档更小。 SQLite 归档具备增量与原子更新功能，并能存储更丰富的元数据。

    Fossil 2.5 及以后版本提供 SQLite 归档文件 作为下载格式，此外还有传统的 tar 包和 ZIP 归档。 sqlite3.exe 命令行外壳 3.22.0 及更高版本可使用 .archive 命令 创建、列出或解包 SQL 归档。

    SQLite 非常适合任何需要将不同内容打包进一个自包含、自描述且可通过网络传输的包中的情况。 内容采用 良好定义、跨平台且稳定的文件格式 编码。 该编码高效，接收方可以在不读取并解析整个文件的情况下提取内容的子集。

    SQL 归档非常适合作为向多个客户端广播的软件或内容更新的分发格式。 该理念的变体例如用于向机顶盒传输电视节目指南，或向车辆导航系统发送空中更新。

  • 替代临时性自制磁盘文件

    许多程序使用 fopen()、 fread() 和 fwrite() 创建并管理自定义格式的数据文件。 SQLite 特别适合替代这些临时性自制数据文件。 与直觉相反，SQLite 在将内容读写到磁盘时可能比文件系统更快，详见 这里。

  • 内部或临时数据库

    对于必须以多种方式筛选与排序的大量数据程序而言， 通常将数据加载到内存中的 SQLite 数据库并使用带有 JOIN 与 ORDER BY 子句的查询 以所需的形式和顺序提取数据，比尝试手动编写相同操作要更容易、更快。 以这种方式在内部使用 SQL 数据库还赋予程序更大的灵活性， 因为可以在无需重写每个查询的情况下添加新列与索引。

  • 演示或测试期间替代企业数据库

    客户端应用程序通常使用通用数据库接口， 使其能够连接多种 SQL 数据库引擎。 将 SQLite 纳入支持数据库组合并将其与客户端静态链接是明智之举。 这样客户端程序在测试或演示时可以独立使用 SQLite 数据文件。

  • 教育与培训

    由于 SQLite 安装和使用非常简单（只需将 sqlite3 或 sqlite3.exe 可执行文件复制到目标计算机并运行）， 它非常适合作为教授 SQL 的数据库引擎。 学生可以轻松创建任意多个数据库，并可以将数据库通过电子邮件发送给教师以供评论或评分。 对于那些有兴趣研究关系数据库管理系统实现方式的高级学生， 模块化、注释充分且文档完善的 SQLite 源码可以作为一个很好的基础。

  • 实验性 SQL 语言扩展

    SQLite 的简单模块化设计使其成为试验性数据库语言新特性的良好原型平台。

2. 更适合使用客户/服务器 RDBMS 的情况

  • 客户/服务器应用

    如果有许多客户端程序通过网络向同一数据库发送 SQL，请使用客户/服务器数据库引擎而不是 SQLite。 SQLite 可以在网络文件系统上运行，但由于大多数网络文件系统存在延迟， 性能不会理想。此外，许多网络文件系统的文件锁定逻辑存在问题（无论在 Unix 还是 Windows 上）。 如果文件锁定不能正确工作，多于一个客户端可能试图同时修改 同一数据库的同一部分，导致数据库损坏。 由于这个问题源于底层文件系统实现的缺陷，SQLite 无能为力。

    一个良好的经验法则是，在数据库将被多个计算机同时直接访问（没有经过应用服务器）且通过网络进行的情况下，避免使用 SQLite。

  • 高流量网站

    SQLite 通常可用作网站的数据库后端。 但如果网站写操作频繁或繁忙到需要多个服务器，那么可以考虑使用企业级的客户/服务器数据库引擎，而非 SQLite。

  • 非常大的数据集

    SQLite 数据库大小限制为 281 TB （\(2^{48}\) 字节，即 256 TiB）。 即便它可以处理更大的数据库，SQLite 也会将整个数据库存储在一个磁盘文件中， 而许多文件系统将文件大小限制得低于此值。 因此，如果您考虑的数据量达到此级别，最好选择一个可以将内容分布在多个磁盘文件甚至多个卷上的客户/服务器数据库引擎。

  • 高并发

    SQLite 支持无限数量的同时读取者，但它在任意时刻只允许一个写入者。 在许多场景下，这并非问题。写入者会排队。每个应用程序快速完成数据库工作然后继续， 没有任何锁定会持续超过几十毫秒。但某些应用程序需要更高的并发性，这类应用可能需要寻找其他解决方案。

3. 选择合适数据库引擎的核对清单

  1. 数据是否通过网络与应用分离？→ 选择客户/服务器

    关系数据库引擎充当带宽降低型数据过滤器。 因此最好将数据库引擎与数据放在同一物理设备上， 这样高带宽的引擎到磁盘链路无需穿越网络，只有较低带宽的应用到引擎链路才需跨网络。

    但 SQLite 已集成在应用程序中。因此如果数据存放在与应用程序不同的设备上， 高带宽的引擎到磁盘链路必须跨网络。 这虽然可行，但并不理想。因此当数据存放在与应用程序不同的设备上时， 通常应选择客户/服务器数据库引擎。

    注： 在此规则中，“应用程序”是指发出 SQL 语句的代码。 如果“应用程序”是一个 应用服务器， 并且内容与应用服务器位于同一台物理机器上， 那么即使最终用户还要再跨一个网络跳跃，SQLite 仍可能是适当的选择。

  2. 有许多并发写入？→ 选择客户/服务器

    如果许多线程或进程需要在同一时间写入数据库（且无法排队按顺序写入）， 那么最好选择支持此功能的数据库引擎，也就是说必须使用客户/服务器数据库引擎。

    SQLite 每个数据库文件一次只支持一个写入者。 但在大多数情况下，写事务只需几毫秒，因此多个写入者可以轮流写入。 SQLite 能处理的写并发比很多人想象的多。 尽管如此，由于客户/服务器数据库系统拥有长期运行的服务器进程来协调访问， 它们通常能处理远超 SQLite 的写并发。

  3. 大数据？→ 选择客户/服务器

    如果您的数据将成长到无法舒适地或无法容纳在单个磁盘文件中的规模， 那么您应该选择其他方案而非 SQLite。 SQLite 支持最大 281 TB 的数据库，前提是您能找到支持 281 TB 文件的磁盘驱动器与文件系统。 即便如此，当内容大小看起来可能达到 TB 级别时，考虑使用集中式的客户/服务器数据库会是一个好主意。

  4. 否则 → 选择 SQLite！

    对于设备本地存储、写入者并发较少且内容少于 1 TB 的场景， SQLite 几乎总是更好的解决方案。 SQLite 快速可靠，不需要配置或维护。 它使事情变得简单。SQLite“就是能用”。
