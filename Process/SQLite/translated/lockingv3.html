本文件最初创建于2004年初，当时 SQLite 2 版本仍被广泛使用，编写本文件是为了向已经熟悉 SQLite 2 的读者介绍 SQLite 3 的新概念。但如今阅读本文件的大多数读者可能从未见过 SQLite 2，只熟悉 SQLite 3。然而，本文件仍然是 SQLite 3 中文件锁定工作原理的权威参考。

本文仅描述较旧的回滚模式事务机制的锁定。对较新的预写日志或WAL 模式的锁定另有说明。

1.0 SQLite 第3版中的文件锁定与并发

SQLite3.0.0 版本引入了一种旨在提升并发性并减少写进程饥饿问题的新锁定和日志机制。新机制还允许跨多个数据库文件的事务实现原子提交。本文档描述了这一新锁定机制。目标读者是希望理解或修改分页器（pager）代码的程序员，以及审核 SQLite 3 设计的审查者。

2.0 概述

锁定与并发控制由分页器模块负责。分页器模块负责保证 SQLite 的“ACID”特性（原子性、一致性、隔离性和持久性）。分页器模块确保变更要么全部发生，要么全部不发生；确保两个或更多进程不会以不兼容的方式同时访问数据库；一旦变更写入后便会持久保留，直到明确删除；分页器还为磁盘文件的一部分内容提供内存缓存。

分页器不会关心 B 树、文本编码、索引等细节。从分页器的角度来看，数据库由大小一致的单个文件块组成。每个块称为“页面”，通常为1024字节大小。页面从1开始编号。因此数据库的前1024字节称为“第1页”，第二1024字节称为“第2页”，以此类推。其他编码细节由库的更高层处理。分页器通过多个模块与操作系统通信（示例：os_unix.c、os_win.c），这些模块为操作系统服务提供统一的抽象层。

分页器模块实际上控制来自不同线程或进程的访问。本文档中每当提到“进程”时，可将其替换为“线程”而不影响陈述的真实性。

3.0 锁定

从单个进程的角度来看，数据库文件可以处于五种锁定状态之一：

UNLOCKED 无需任何数据库锁定。数据库既不能读也不能写。任何内部缓存的数据都被视为有疑问，使用前需与数据库文件核对。其他进程可根据其自身的锁定状态读取或写入数据库。这是默认状态。
SHARED 数据库可以被读取但不能写入。任意数量的进程可以同时保持 SHARED 锁，因此可以有多个并发读取者。但在一个或多个 SHARED 锁活动时，其他线程或进程不得写入数据库文件。
RESERVED RESERVED 锁意味着该进程计划在未来某个时间写入数据库文件，但目前仅在读取。任意时刻仅允许一个 RESERVED 锁，但多个 SHARED 锁可以与单个 RESERVED 锁共存。RESERVED 与 PENDING 的区别在于，当存在 RESERVED 锁时，仍可以获取新的 SHARED 锁。
PENDING PENDING 锁意味着持有该锁的进程希望尽快写入数据库，并正在等待所有当前的 SHARED 锁清除以便获得 EXCLUSIVE 锁。如果存在 PENDING 锁，则不允许获取新的 SHARED 锁，但现有的 SHARED 锁可以继续存在。
EXCLUSIVE 写入数据库文件时需要 EXCLUSIVE 锁。文件上只能存在一个 EXCLUSIVE 锁，并且不允许与任何其他类型的锁共存。为了最大化并发性，SQLite 会努力缩短 EXCLUSIVE 锁持有时间。

操作系统接口层理解并跟踪上述五种锁定状态。分页器模块只跟踪其中四种状态。PENDING 锁始终只是走向 EXCLUSIVE 锁的临时中转，因此分页器模块不跟踪 PENDING 锁。

4.0 回滚日志

当一个进程希望修改数据库文件（且当前不处于WAL 模式）时，它首先会在回滚日志中记录原始、未更改的数据库内容。回滚日志是一个普通磁盘文件，始终位于与数据库文件相同的目录或文件夹中，文件名为数据库文件名加上-journal后缀。回滚日志还记录数据库的初始大小，以便在回滚时将数据库截断回原始大小。

如果 SQLite 同时处理多个数据库（使用ATTACH命令），则每个数据库都有自己的回滚日志。但还有一个名为超级日志的聚合日志。超级日志不包含用于回滚的页面数据，而是包含每个已 ATTACH 数据库的各自回滚日志的名称。每个单独的数据库回滚日志也包含超级日志的名称。如果没有 ATTACH 的数据库（或没有任何 ATTACH 的数据库参与当前事务），则不会创建超级日志，普通回滚日志在原本记录超级日志名称的位置写入空字符串。

当某个回滚日志需要被回滚以恢复数据库完整性时，该日志被称为热日志。在进程正在进行数据库更新且程序或操作系统崩溃或断电导致更新未完成时，会生成热日志。热日志属于异常情况，用于在崩溃和断电后恢复。如果一切正常（即没有崩溃或断电），就不会生成热日志。

如果不涉及超级日志，则当日志存在且有非零头部，且对应的数据库文件没有 RESERVED 锁时，该日志即为热日志。如果文件日志中指明了超级日志名称，则当该超级日志存在且对应的数据库文件没有 RESERVED 锁时，该文件日志即为热日志。要理解何时日志为热，从而重复说明如下：

  • 日志为热当且仅当...
    • 它存在，且
    • 其大小超过512字节，且
    • 日志头为非零且格式良好，且
    • 其超级日志存在或超级日志名称为空字符串，且
    • 对应的数据库文件没有 RESERVED 锁。

4.1 处理热日志

在读取数据库文件之前，SQLite 总会检查该数据库文件是否存在热日志。如果存在热日志，则会在读取文件之前先回滚该日志。如此一来，我们可以确保在读取数据库文件前其处于一致的状态。

当进程希望读取数据库文件时，它会执行以下步骤：

  1. 打开数据库文件并获取 SHARED 锁。如果无法获取 SHARED 锁，则立即失败并返回 SQLITE_BUSY。
  2. 检查数据库文件是否有热日志。如果没有热日志，则完成操作并立即返回。如果存在热日志，则必须通过后续步骤回滚该日志。
  3. 获取 PENDING 锁，然后获取 EXCLUSIVE 锁。（注意：不要获取 RESERVED 锁，否则会让其他进程误以为日志已不再热。）如果无法获取这些锁，意味着另一个进程已经在进行回滚。在这种情况下，释放所有锁、关闭数据库，并返回 SQLITE_BUSY。
  4. 读取日志文件并回滚更改。
  5. 等待回滚的更改写入持久存储。这能在再次发生电力故障或崩溃时保护数据库完整性。
  6. 删除日志文件（或者若设置了PRAGMA journal_mode=TRUNCATE，则将日志截断为零字节；若设置了PRAGMA journal_mode=PERSIST，则将日志头置零）。
  7. 如果安全，删除超级日志文件。此步骤可选，仅用于防止陈旧的超级日志堆积占用磁盘空间。详细讨论见下文。
  8. 释放 EXCLUSIVE 和 PENDING 锁，但保留 SHARED 锁。

在上述算法成功完成后，即可安全地读取数据库文件。读取完成后，释放 SHARED 锁。

4.2 删除陈旧超级日志

陈旧超级日志是指不再被任何使用的超级日志。虽然并非必须删除陈旧超级日志，但删除可回收磁盘空间。

超级日志在没有任何单个文件日志指向它时即为陈旧。为判断超级日志是否陈旧，首先读取超级日志获取所有文件日志的名称，然后检查这些文件日志。如果超级日志中列出的任何文件日志存在并指向该超级日志，那么该超级日志即非陈旧。如果所有文件日志要么不存在，要么引用其他超级日志，或者根本不引用超级日志，那么正在测试的超级日志即为陈旧，可以安全删除。

5.0 向数据库文件写入数据

要写入数据库，进程必须首先获取前文所述的 SHARED 锁（如果存在热日志，可能需先回滚该日志）。获取 SHARED 锁后，还必须获取 RESERVED 锁。RESERVED 锁表示该进程未来某个时刻打算写入数据库。一次仅允许一个进程持有 RESERVED 锁，但其他进程仍可在 RESERVED 锁存在时读取数据库。

如果意图写入的进程无法获取 RESERVED 锁，则说明已有其他进程持有该锁。这种情况下，写入尝试失败并返回 SQLITE_BUSY。

在获得 RESERVED 锁后，写进程会创建回滚日志。日志头初始化为数据库文件的原始大小。日志头中还为超级日志名称预留空间，但初始为空。

在修改数据库的任何页面之前，进程会将该页面的原始内容写入回滚日志。页面更改先保存在内存中，暂不写入磁盘。原始数据库文件保持不变，因此其他进程仍可继续读取数据库。

最终，写进程会希望更新数据库文件，可能是因为内存缓存已满，或准备提交更改。在此之前，写进程必须确保没有其他进程正在读取数据库，并确保回滚日志数据已安全写入磁盘，以便在发生断电时可用于回滚未完成的更改。步骤如下：

  1. 确保所有回滚日志数据确实写入磁盘表面（而非仅驻留在操作系统或磁盘控制器缓存中），以便在突然断电后数据仍能恢复。
  2. 为数据库文件获取 PENDING 锁，然后获取 EXCLUSIVE 锁。如果其他进程仍持有 SHARED 锁，写进程可能需等待这些 SHARED 锁释放，才能获得 EXCLUSIVE 锁。
  3. 将当前内存中保存的所有页面修改写入原始数据库磁盘文件。

如果写入数据库的原因是内存缓存已满，则写进程不会立即提交。它可能继续修改其他页面。在将随后的更改写入数据库文件之前，必须再次将回滚日志刷新到磁盘。还要注意，为最初写入数据库而获取的 EXCLUSIVE 锁必须一直保持，直到所有更改提交。也就是说，从内存缓存首次溢写到磁盘到事务提交期间，其他进程无法访问数据库。

当写进程准备提交更改时，它执行如下步骤：

  1. 为数据库文件获取 EXCLUSIVE 锁，并确保使用上述步骤 1-3 的算法已将所有内存更改写入数据库文件。
  2. 刷新所有数据库文件更改到磁盘。等待这些更改真正写入磁盘表面。
  3. 删除日志文件。（或者如果PRAGMA journal_mode为 TRUNCATE 或 PERSIST，则分别截断日志文件或将日志头置零。）此时更改被提交。在删除日志文件之前发生断电或崩溃时，下一个打开数据库的进程会发现热日志并回滚更改。日志文件删除后，不再存在热日志，更改得以持久保存。
  4. 释放数据库文件的 EXCLUSIVE 和 PENDING 锁。

一旦 PENDING 锁从数据库文件释放，其他进程即可重新读取数据库。在当前实现中，RESERVED 锁也会被释放，但这对正确运行并非必需。

如果一个事务涉及多个数据库，则使用更复杂的提交顺序，如下：

  1. 确保所有单独数据库文件均持有 EXCLUSIVE 锁并拥有有效日志。
  2. 创建一个超级日志。超级日志名称可任意。（当前实现会在主数据库文件名后追加随机后缀，直到找到一个尚未存在的名称。）将所有单个日志名称写入超级日志并将其内容刷新到磁盘。
  3. 将超级日志名称写入所有单个日志（在日志头为此预留的空间中），并将这些日志的内容刷新到磁盘，等待这些更改写入磁盘表面。
  4. 刷新所有数据库文件更改到磁盘。等待这些更改真正写入磁盘表面。
  5. 删除超级日志文件。此时更改被提交。在删除超级日志文件之前如果发生断电或崩溃，各个文件日志会被视为热日志，并由下一个尝试读取它们的进程回滚。超级日志被删除后，文件日志不再被视为热日志，更改保持。
  6. 删除所有单独的日志文件。
  7. 释放所有数据库文件的 EXCLUSIVE 和 PENDING 锁。

5.1 写进程饥饿

在 SQLite 2 中，如果许多进程正在读取数据库，可能永远不会出现没有活动读取者的时刻。如果数据库上始终至少有一个读取锁，则任何进程都无法写入数据库，因为无法获取写锁。这种情况称为写进程饥饿。

SQLite 3 通过使用 PENDING 锁来避免写进程饥饿。PENDING 锁允许现有读取者继续读取，但阻止新的读取者连接数据库。因此，当某个进程想写繁忙数据库时，可以先设置 PENDING 锁，这将阻止新的读取者进入。假设现有读取者最终会完成，所有 SHARED 锁将逐渐释放，写进程就能获得机会进行更改。

6.0 如何破坏你的数据库文件

分页器模块非常健壮，但仍有可能被破坏。本节试图识别并解释风险。（另见原子提交文章中可能出错的事项部分。）

显然，如果硬件或操作系统故障在数据库文件或日志中间写入错误数据，就会导致问题。同样，如果某个恶意进程打开数据库文件或日志并向其中写入格式错误的数据，数据库就会损坏。对于此类问题几乎无能为力，因此不会进一步讨论。

SQLite 在 Unix 上使用 POSIX 建议锁来实现锁定，在 Windows 上使用 LockFile()、LockFileEx() 和 UnlockFile() 系统调用。SQLite 假设这些系统调用都按宣传运行。如果事实并非如此，就可能导致数据库损坏。需注意的是，许多 NFS 实现（包括较新版本的 macOS）对 POSIX 建议锁有已知缺陷甚至未实现。而在 Windows 上也有网络文件系统锁定问题的报告。最好的防御措施是不要在网络文件系统上使用 SQLite。

SQLite 在 Unix 下使用 fsync() 将数据刷新到磁盘，在 Windows 下使用 FlushFileBuffers() 实现同样功能。SQLite 同样假设这些操作系统服务都正常运行。但据报道，fsync() 和 FlushFileBuffers() 并不总是可靠，特别是在一些网络文件系统或廉价 IDE 磁盘上。显然某些 IDE 磁盘制造商的控制器芯片会错误地报告数据已到达磁盘表面，实际上数据仍驻留在磁盘驱动电子设备的易失性缓存中。还有报告称 Windows 有时会无缘无故忽略 FlushFileBuffers()。作者无法验证这些报告是否属实。但若确有其事，则在意外断电后数据库可能损坏。这些属于硬件和/或操作系统 bug，SQLite 无法防御。

如果 Linux ext3 文件系统在 /etc/fstab 中未使用 "barrier=1" 选项挂载，且磁盘写缓存被启用，则在断电或操作系统崩溃后可能会发生文件系统损坏。损坏是否发生取决于磁盘控制硬件的细节；廉价消费级磁盘上更容易出现，而具有非易失性写缓存等高级功能的企业级存储设备问题较少。多位 ext3 专家证实了此行为。据说大多数 Linux 发行版没有启用 barrier=1，也未禁用写缓存，所以大多数 Linux 发行版都容易受到此问题影响。请注意，这是操作系统和硬件问题，SQLite 无法绕过此限制。其他数据库引擎也遇到过相同问题。

如果发生崩溃或断电，导致生成热日志但该日志被删除，那么下一个打开数据库的进程将无法知道它包含需要回滚的更改。回滚将不会执行，数据库将处于不一致状态。回滚日志可能因多种原因被删除：

  • 管理员可能在 OS 崩溃或断电后清理，看见日志文件误认为是垃圾并删除。
  • 某人（或某个进程）可能重命名数据库文件但未同步重命名其对应的日志。
  • 如果数据库文件有多个别名（硬链接或软链接），且通过不同的别名打开文件，那么日志将无法找到。为避免此问题，请不要为 SQLite 数据库文件创建链接。
  • 断电后文件系统损坏可能导致日志被重命名或删除。

上述最后一点值得额外说明。当 SQLite 在 Unix 上创建日志文件时，会打开包含该日志的目录并对该目录调用 fsync()，以努力将目录信息写入磁盘。但假设在断电瞬间，另一个进程正在向包含数据库和日志的目录添加或删除无关文件。该进程的看似无关操作可能导致日志文件从目录中被移除并移动到“lost+found”。虽然这种情况不常见，但可能发生。最佳防御措施是使用带日志功能的文件系统，或将数据库和日志单独放在一个目录中。

对于涉及多个数据库并使用超级日志的提交，如果各数据库位于不同磁盘卷上，而提交过程中发生断电，则在机器重新启动后，磁盘可能以不同名称重新挂载，或者某些磁盘根本未被挂载。这种情况下，单个文件日志和超级日志可能无法相互定位。最坏情况是提交不再是原子的——某些数据库日志会被回滚，而另一些则不会。所有数据库都会保持自洽。为防止此问题，建议将所有数据库放在同一磁盘卷上，或者在断电后以完全相同的名称重新挂载磁盘。

7.0 SQL 级别的事务控制

SQLite 3 中锁定与并发控制的变更还对 SQL 语言级别的事务工作方式带来一些微妙变化。默认情况下，SQLite 3 运行在自动提交模式。在自动提交模式下，只要当前数据库连接关联的所有操作完成，所有更改就会立即提交。

SQL 命令“BEGIN TRANSACTION”（TRANSACTION 关键字可选）用于将 SQLite 置于非自动提交模式。注意 BEGIN 命令并不会在数据库上获取任何锁。发出 BEGIN 命令后，当执行第一条 SELECT 语句时会获取 SHARED 锁。当执行第一条 INSERT、UPDATE 或 DELETE 语句时会获取 RESERVED 锁。在内存缓存溢出并必须写入磁盘之前，或在事务提交之前，都不会获取 EXCLUSIVE 锁。通过这种方式，系统会将对文件的阻塞性读取延迟到最后一刻。

SQL 命令“COMMIT”并不会立即将更改提交到磁盘。它只是将自动提交重新开启。然后在命令结束时，常规的自动提交逻辑接手并导致实际的磁盘提交。SQL 命令“ROLLBACK”同样通过重新开启自动提交操作，但它还设置了一个标志，指示自动提交逻辑执行回滚而不是提交。

如果 SQL COMMIT 命令开启自动提交，而自动提交逻辑随后尝试提交更改但因其他进程持有 SHARED 锁而失败，则自动提交会自动关闭。这允许用户在 SHARED 锁有机会清除后稍后重试 COMMIT。
