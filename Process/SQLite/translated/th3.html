1. 概述

SQLite 测试框架 #3（以下称为“TH3”）是用于测试 SQLite 的三种测试框架之一。TH3 满足以下目标：

  • TH3 能够在缺乏工作站支持基础设施的嵌入式平台上运行。

  • TH3 使用仅有公开和记录的接口在部署后的配置中测试 SQLite。 换句话说，TH3 测试已编译的目标代码，而不是源代码，从而验证编译器错误未引入问题。“测试你要运行的，并运行你测试的。”

  • TH3 检查 SQLite 对内存不足、磁盘 I/O 错误以及在事务提交期间电源中断的响应。

  • TH3 以多种运行时配置（UTF8 与 UTF16、不同的数据页大小、各种日志模式等）测试 SQLite。

  • TH3 在 SQLite 核心上实现了 100% 的分支测试覆盖率（以及 100% 的MC/DC）。 （诸如 FTS 与 RTREE 等扩展的测试覆盖率低于 100%。）

TH3 最初只为验证测试而编写，但随后也被用于开发测试与调试，并在这些角色中表现出极大帮助。完整覆盖的测试在工作站上运行不到五分钟，因此可以在 SQLite 代码库日常维护期间作为快速回归测试。

1.1. 历史

TH3 起源于在SymbianOS上测试 SQLite 的工作。TH3 之前，所有 SQLite 测试都通过TCL 脚本语言运行，但 TCL 在 SymbianOS 上难以编译，这使得测试变得艰难。最初尝试的方案是“TH1”（测试框架 #1）脚本语言——一种以更可移植形式重新实现部分 TCL 语言的版本，可以在 SymbianOS 上编译和运行，并足以运行 SQLite 测试。TH1 并未作为 SQLite 的标准测试工具继续存在，但它继续作为一种脚本语言用于定制Fossil 版本控制系统。此外还有一个“测试框架 #2”，它尝试使用操作符前缀表示法创建一个简单的脚本语言来驱动测试。TH3 是第三次尝试。

就在那段时间，一些航空电子设备制造商对 SQLite 表示兴趣，于是 SQLite 开发者设计 TH3 以支持DO-178B的严格测试标准。

TH3 的首段代码写于 2008-09-25。接下来的十个月里持续努力，于 2009-07-25 实现了 100% 的 MC/DC。TH3 代码持续被改进与扩展。

截至 2018-05-19，TH3 源代码树包含 1709 个单独文件，源代码行数远超 500,000 行。

2. 运作

TH3 是一个测试程序生成器。TH3 的输出是一个用 C 语言实现的程序，旨在链接到被测试的 SQLite 库。生成的测试程序在目标平台上编译并运行，以验证 SQLite 在该平台上的正确性。

TH3 的输入是用 C 或 SQL 编写的测试模块，以及确定如何初始化 SQLite 的小型配置文件。TH3 包含 1,444 个测试模块以及超过 47 个配置（截至 2018-05-19）。可以添加新的模块与配置以定制 TH3 以满足特殊应用。每次运行 TH3 时，它会读取一部分可用的测试模块与配置文件，以生成一个在所有指定配置下执行所有指定测试的自定义 C 程序。完整测试 SQLite 通常需要多次运行 TH3，以生成多个涵盖 SQLite 不同操作方面的测试程序，然后将所有测试程序链接到一个共享的 SQLite 库，并在目标平台上逐个运行。

TH3 没有任意限制。可以生成一个包含所有测试模块与所有配置文件的测试程序，然而这样的程序可能过大而无法部署到嵌入式平台。（截至 2018-05-19，完整 TH3 测试生成的 C 代码超过 850,000 行，58MB。）TH3 提供了将测试模块库拆分成更小、易于处理的部分的能力。

每个测试模块可能包含数十、数百甚至数千个独立测试。测试模块可以用 C 编写，也可以是 SQL 脚本，或两者混合。现有测试模块中约三分之二是纯 SQL，其余为纯 C 或 C 与 SQL 混合。

每个测试模块文件包含描述该测试在何种条件下有效的头信息。对于某个特定配置，只运行与该配置兼容的模块。

3. 生成测试程序

TH3 程序生成器是一个名为“mkth3.tcl”的 TCL 脚本。要生成测试程序，只需运行该脚本，并在命令行中提供包含测试模块与配置的文件名。测试模块文件以“.test”为后缀，配置文件以“.cfg”为后缀。一次典型的 mkth3.tcl 调用可能如下所示：

tclsh mkth3.tcl *.test *.cfg >testprog1.c

mkth3.tcl 脚本的输出是一个包含运行测试所需一切的 C 程序——唯一缺少的是 SQLite 库本身。生成的测试程序包含所有测试模块使用的支持接口的实现，以及驱动测试的main()例程。要将测试程序转换为可执行文件，只需用 SQLite 编译：

cc -o testprog1 testprog1.c sqlite3.c

上面展示的编译步骤仅为示例。在实际安装中，通常希望在编译器命令行上指定优化参数与编译时开关。

在嵌入式系统上测试时，mkth3.tcl 脚本与上面所示的编译步骤在普通工作站上使用交叉编译器完成，然后将生成的测试程序传输到设备上运行。

一旦生成测试程序，就直接运行（无需参数）以执行测试。进度信息与错误诊断显示在标准输出上。（对于缺乏标准输出通道的嵌入式设备，可以使用编译时选项进行替代输出安排。）若未检测到错误，程序返回零；若发现问题，则返回非零。

3.1. 测试自动化脚本

TH3 附带额外的 TCL 脚本，帮助在工作站上自动化测试流程。“th3make”脚本自动运行“mkth3.tcl”与“gcc”，然后运行生成的测试程序并检查结果。th3make 的参数包括所有要包含的“*.test”测试模块与“*.cfg”配置。th3make 的附加选项可以使测试程序使用不同编译器（GCC、Clang、MSVC）进行编译、使用不同的输出详细程度、在 valgrind 下运行测试程序、使用 gcov 检查覆盖率等。th3make 脚本还接受“*.rc”文件名作为参数。这些 *.rc 文件只是常为单一目的组合在一起的其他参数。例如，“quick.rc”文件包含一组八个参数，使 th3make 运行一个快速（3 分钟）全覆盖测试。这允许操作员通过输入“./th3make quick.rc”来快捷使用所有所需命令行选项。以下是 40 多个可用 *.rc 文件中的一部分：

  • alignmentN.rc - 这些文件包含传递给编译器的 -D 选项，被多个显著的下游用户使用。
  • cov.rc - 测试覆盖测量选项
  • extensions.rc - 启用FTS4、R-Tree和STAT4的选项。
  • fast.rc - 运行大多数测试，包括非覆盖所需的，跳过浸泡测试，使用发布编译器选项（如 -O3）
  • memdebug.rc - 类似 test.rc，但还启用-DSQLITE_MEMDEBUG。
  • min.rc - 运行实现 100% 覆盖所需的最少测试集。
  • quick.rc - 使用 -Os 与-DSQLITE_DEBUG运行实现 100% 覆盖的所有测试。
  • test.rc - 运行与 fast.rc 相同的测试，但不启用编译器优化，并启用如-DSQLITE_DEBUG与 -DSQLITE_ENABLE_EXPENSIVE_ASSERT 的选项。
  • test-ex.rc - 长时间运行的浸泡测试。

th3make quick.rc -DSQLITE_DISABLE_INTRINSIC -DSQLITE_ENABLE_RTREE

4. 测试覆盖

使用可用 TH3 测试模块的特定子集（“cov1”测试），SQLite 在 Linux x86 与 x86_64 硬件上通过100% 的分支测试覆盖率与100% 的 MC/DC，由gcov 测量。从版本 3.6.17（2009-08-10）开始的所有 SQLite 发布都已根据该标准进行测试。SQLite 开发者致力于在未来所有发行版中维持 100% 的分支覆盖与 MC/DC。

用于实现 100% 分支测试覆盖的 cov1 测试集只是当前用 TH3 实现的测试中的一部分。新的测试模块会定期添加。

5. 变异测试

TH3 源代码树包含名为“mutation-test.tcl”的脚本，用于自动变异测试流程。

mutation-test.tcl 脚本处理运行变异测试的所有细节：

  1. 脚本将在必要时将 TH3 测试框架编译为机器码（“th3.o”）。
  2. 脚本将在必要时将 sqlite3.c 源文件编译为汇编语言（“sqlite3.s”）。
  3. 脚本遍历汇编语言文件中的指令以定位分支操作。
    1. 脚本复制原始 sqlite3.s 文件。
    2. 复制文件被编辑，将分支指令更改为无操作或无条件跳转。
    3. 被修改的 sqlite3.s 被汇编为 sqlite3.o，然后与 th3.o 链接生成“th3”可执行文件。
    4. 运行“th3”二进制，检查输出是否有错误。
  4. 脚本显示前述循环每个周期的进度，并在结束时列出“幸存者”的摘要。“幸存者”是 TH3 未检测出的变异。

变异测试可能比较慢，因为每个测试在快速工作站上最多需要 5 分钟，每个分支指令有两次测试，而分支指令超过 20,000 条。为加快运行速度，会采取一些措施。例如，TH3 编译方式使其在发现首个错误时立即退出，许多变异也容易被检测到，因此很多循环只需几秒钟。此外，mutation-test.tcl 脚本还包括命令行选项，可限制测试的代码行范围，从而只对最近变更的代码块执行变异测试。

6. TH3 许可证

SQLite 本身属于公有领域，可用于任何目的。但 TH3 是专有软件，需要许可证。

尽管开源用户无法直接访问 TH3，所有 SQLite 用户仍间接受益于 TH3，因为每个 SQLite 版本在发布前都会在多个平台（Linux、Windows、WinRT、Mac、OpenBSD）上运行 TH3 进行验证。因此，任何使用官方 SQLite 发行版的用户都可以放心地部署其应用程序，因为它已通过 TH3 测试。只是他们不得不购买 TH3 许可证才能重新运行这些测试。
