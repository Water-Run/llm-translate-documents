Defense Against The Dark Arts

1. SQLite 始终验证其输入

SQLite 不应崩溃、溢出缓冲区、泄露内存或表现出其它任何有害行为，即使在面对恶意格式错误的 SQL 输入或数据库文件时也是如此。SQLite 应该始终检测错误的输入并抛出错误，而不是崩溃或损坏内存。任何由 SQL 输入或数据库文件引起的故障都被视为严重缺陷，一旦 SQLite 开发人员得知会立即处理。SQLite 经过广泛的模糊测试，以帮助确保它能抵御这类错误。

尽管如此，缺陷还是会偶尔发生。如果你正在编写向 SQLite 发送不受信任 SQL 输入或数据库文件的应用程序，可以采取额外措施以减少攻击面并防止由于未检测到的缺陷导致的零日漏洞。

1.1. 不受信任的 SQL 输入

接受不受信任 SQL 输入的应用程序应采取以下预防措施：

  1. 设置 SQLITE_DBCONFIG_DEFENSIVE 标志。这可以防止普通 SQL 语句故意破坏数据库文件。SQLite 应该能够抵御同时包含恶意 SQL 输入和恶意损坏数据库文件的攻击。不过，拒绝仅脚本攻击者访问受损的数据库输入可以提供额外的防线。

  2. 降低 SQLite 对输入施加的 限值。这有助于防止拒绝服务攻击以及因非常大的输入而可能导致的其它问题。可在编译时使用 -DSQLITE_MAX_... 选项或在运行时使用 sqlite3_limit() 接口来完成。大多数应用程序可以在不影响功能的前提下大幅减少这些限值。下表给出一些建议，具体值会根据应用程序而有所不同：

    限制设置默认值高安全值
    LIMIT_LENGTH1,000,000,0001,000,000
    LIMIT_SQL_LENGTH1,000,000,000100,000
    LIMIT_COLUMN2,000100
    LIMIT_EXPR_DEPTH1,00010
    LIMIT_COMPOUND_SELECT5003
    LIMIT_VDBE_OP250,000,00025,000
    LIMIT_FUNCTION_ARG1278
    LIMIT_ATTACH100
    LIMIT_LIKE_PATTERN_LENGTH50,00050
    LIMIT_VARIABLE_NUMBER99910
    LIMIT_TRIGGER_DEPTH1,00010

  3. 考虑使用 sqlite3_set_authorizer() 接口限制将被处理的 SQL 范围。例如，不需要更改数据库模式的应用程序可以添加 sqlite3_set_authorizer() 回调，使任何 CREATE 或 DROP 语句都失败。

  4. SQL 语言非常强大，因此恶意 SQL 输入（或因应用程序缺陷而产生的错误输入）可能会提交运行时间很长的 SQL。为了防止这成为拒绝服务攻击，可以考虑使用 sqlite3_progress_handler() 接口，在每个 SQL 语句运行时定期调用回调，并让该回调返回非零值以在运行时间过长时中止语句。或者，在单独线程中设置定时器，当定时器触发时调用 sqlite3_interrupt()，防止 SQL 语句无限运行。

  5. 限制 SQLite 可以分配的最大内存，使用 sqlite3_hard_heap_limit64() 接口。这有助于防止拒绝服务攻击。为了了解应用程序实际需要多少堆空间，请在典型输入下运行应用程序，然后使用 sqlite3_memory_highwater() 接口测量最大瞬时内存使用量。将硬堆限制设置为观测到的最大瞬时内存使用量加上一些余量。

  6. 考虑将编译时选项 SQLITE_MAX_ALLOCATION_SIZE 设置为比默认值 2147483391 (0x7ffffeff) 更小的值。根据应用程序情况，设置为 100000000（1 亿）甚至更小都是合理的。

  7. 对于嵌入式系统，考虑使用 -DSQLITE_ENABLE_MEMSYS5 选项编译 SQLite，然后通过 sqlite3_config(SQLITE_CONFIG_HEAP) 接口为 SQLite 提供固定大小的堆内存。这可防止恶意 SQL 通过使用过多内存发起拒绝服务攻击。例如，如果为 SQLite 提供了 5 MB 内存，一旦消耗完毕，SQLite 就会开始返回 SQLITE_NOMEM 错误，而不是占用应用程序其它部分所需的内存。这也将 SQLite 的内存沙箱化，以防应用程序其它部分的写后释放错误影响 SQLite，反之亦然。

  8. 为了控制 printf() SQL 函数 的内存使用，请使用 "-DSQLITE_PRINTF_PRECISION_LIMIT=100000" 或类似的合理值进行编译。此 #define 限制 printf() 函数中 %-替代符的宽度和精度，从而防止敌对的 SQL 语句通过类似 "printf('%1000000000s','hi')" 的构造消耗大量 RAM。

    请注意，SQLite 在内部使用其内置 printf() 来帮助格式化 sqlite_schema 表 中的 sql 列。因此，任何表、索引、视图或触发器定义都不能比精度限制大得多。你可以设置小于 100000 的精度限制，但要注意所设置的精度限制至少要与模式中最长的 CREATE 语句一样长。

1.2. 不受信任的 SQLite 数据库文件

读取或写入来源不明的 SQLite 数据库文件的应用程序应当采取下列列出的预防措施。

即使应用程序并非刻意接受来自不受信任来源的数据库文件，也需警惕本地数据库文件被篡改的攻击。为了获得最佳安全性，任何曾经可能被不同安全域的主体写入的数据库文件都应视为可疑。

  1. 如果应用程序包含任何具有副作用或可能泄露特权信息的 自定义 SQL 函数 或 自定义虚拟表，则应用程序应使用下述一种或多种技术，防止恶意构造的数据库模式在暗中运行这些 SQL 函数和/或虚拟表以图不轨：

    1. 一打开每个 数据库连接，立即调用 sqlite3_db_config(db,SQLITE_DBCONFIG_TRUSTED_SCHEMA,0,0)。
    2. 一打开每个数据库连接，立即运行 PRAGMA trusted_schema=OFF。
    3. 使用 -DSQLITE_TRUSTED_SCHEMA=0 编译时选项编译 SQLite。
    4. 通过在所有自定义 SQL 函数上设置 SQLITE_DIRECTONLY 标志，以及在所有自定义虚拟表上设置 SQLITE_VTAB_DIRECTONLY 标志，禁止这些函数和虚拟表被暗中使用。

  2. 如果应用程序不使用触发器或视图，请考虑通过以下方式禁用未使用的功能： 

    sqlite3_db_config(db,SQLITE_DBCONFIG_ENABLE_TRIGGER,0,0);
sqlite3_db_config(db,SQLITE_DBCONFIG_ENABLE_VIEW,0,0);

对于读取风险特别高的数据库文件，例如来自远程机器甚至匿名贡献者的数据库文件，可以采取以下额外防护措施。不过这些额外防护会带来性能成本，因此并非在所有场景中都合适：

  1. 在打开数据库文件并在运行其他 SQL 语句之前，将 PRAGMA integrity_check 或 PRAGMA quick_check 作为第一条 SQL 语句运行。拒绝并拒绝处理任何包含错误的数据库文件。

  2. 启用 PRAGMA cell_size_check=ON 设置。

  3. 不要启用内存映射 I/O。换句话说，确保 PRAGMA mmap_size=0。

2. 总结

上述预防措施并非在面对潜在敌对输入时安全使用 SQLite 的必要条件。但它们确实为零日漏洞提供了额外防御层，建议对将来自不受信任来源的数据传入 SQLite 的应用程序采取这些措施。
