1. 介绍

每个 SQLite 数据库都包含一个单一的“schema 表”，用于存储该数据库的模式。数据库的模式是对数据库中包含的所有其他表、索引、触发器和视图的描述。模式表如下所示： 

CREATE TABLE sqlite_schema(
  type text,
  name text,
  tbl_name text,
  rootpage integer,
  sql text
);

sqlite_schema 表为模式中的每个表、索引、视图和触发器（统称“对象”）包含一行，除了 sqlite_schema 表自身没有条目。有关 SQLite 如何在内部使用 sqlite_schema 表的更多信息，请参阅 文件格式 文档中的 模式存储 小节。

2. 替代名称

模式表始终可以使用名称“sqlite_schema”进行引用，特别是在指定了模式名称（例如“main.sqlite_schema”或“temp.sqlite_schema”）的情况下。但出于历史兼容性，也会识别一些替代名称，包括：

  1. sqlite_master
  2. sqlite_temp_schema
  3. sqlite_temp_master

替代项 (2) 和 (3) 仅适用于与每个数据库连接相关联的 TEMP 数据库，但替代项 (1) 可在任何地方使用。出于历史原因，来自 sqlite3_set_authorizer() 接口的回调始终使用名称 (1) 或 (3) 来引用模式表。

3. 模式表的解释

模式表字段的含义如下：

type

sqlite_schema.type 列将根据所定义对象的类型为以下文本字符串之一：'table'、'index'、'view' 或 'trigger'。'table' 字符串用于普通表和 虚拟表。

name

sqlite_schema.name 列将保存对象的名称。表上的 UNIQUE 和 PRIMARY KEY 约束会导致 SQLite 创建名称形式为“sqlite_autoindex_TABLE_N”的 内部索引，其中 TABLE 被约束所在的表名替换，N 是从 1 开始并随着表定义中看到的每个约束递增的整数。在 WITHOUT ROWID 表中，PRIMARY KEY 没有对应的 sqlite_schema 条目，但“sqlite_autoindex_TABLE_N”名称仍为 PRIMARY KEY 保留，仿佛 sqlite_schema 条目确实存在。这会影响后续 UNIQUE 约束的编号。对于 INTEGER PRIMARY KEY，无论是 rowid 表还是 WITHOUT ROWID 表，都不会分配“sqlite_autoindex_TABLE_N”名称。

tbl_name

sqlite_schema.tbl_name 列保存与对象相关联的表或视图的名称。对于表或视图，tbl_name 列是 name 列的副本。对于索引，tbl_name 是被索引的表的名称。对于触发器，tbl_name 列存储触发器所基于的表或视图的名称。

rootpage

sqlite_schema.rootpage 列存储表和索引的根 B 树页的页码。定义视图、触发器和虚拟表的行，其 rootpage 列为 0 或 NULL。

sql

sqlite_schema.sql 列存储描述对象的 SQL 文本。这段 SQL 文本是一个 CREATE TABLE、CREATE VIRTUAL TABLE、CREATE INDEX、CREATE VIEW 或 CREATE TRIGGER 语句，如果在数据库作为 数据库连接 的主数据库时对其求值，将重新创建该对象。该文本通常是用于创建对象的原始语句的副本，但经过规范化，使其符合以下规则：

  • 语句开头的 CREATE、TABLE、VIEW、TRIGGER 和 INDEX 关键字均转换为大写字母。
  • 如果 TEMP 或 TEMPORARY 关键字出现在初始 CREATE 关键字之后，则将其移除。
  • 在要创建的对象名称之前出现的任何数据库名称限定符都会被移除。
  • 删除前导空格。
  • 首两个关键字后的所有空格都合并为一个空格。

sqlite_schema.sql 列中的文本是创建该对象的原始 CREATE 语句文本的副本，但如上所述进行了规范化，并可能被后续的 ALTER TABLE 语句修改。对于 由 UNIQUE 或 PRIMARY KEY 约束自动创建的内部索引，sqlite_schema.sql 为 NULL。

4. 模式表的创建和修改

SQLite 在数据库创建时创建模式表，并在用户提交 DDL 语句执行时修改其内容。通常情况下，用户无需手动修改它，且如果进行修改，将面临 数据库损坏 的风险。
