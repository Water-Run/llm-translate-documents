1. 引言

每个 SQL 数据库引擎的工作方式大致相同：它首先将输入的 SQL 文本翻译为“预处理语句”。然后它“执行”该预处理语句以生成结果。

预处理语句是一个表示完成输入 SQL 所需步骤的对象。换句话说，预处理语句是 SQL 语句被翻译为更容易被计算机理解的形式。

在 SQLite 中，预处理语句是一个sqlite3_stmt 对象的实例。在其他系统中，预处理语句通常是一个不会直接向应用程序开发者显示的内部数据结构。其他 SQL 数据库引擎的开发人员不一定称这些对象为“预处理语句”。但是无论它们被如何命名，这些对象确实存在。本文将使用“预处理语句”一词。

实现预处理语句的方法数不胜数，本文将关注最常见的两种方法：

  1. 字节码 → 输入的 SQL 被翻译为一种虚拟机语言，然后由虚拟机解释器运行。这正是 SQLite 所采用的技术。

  2. 对象树 → 输入的 SQL 被翻译为表示要执行处理的对象树。SQL 的执行通过遍历这棵树完成。这是 MySQL 和 PostgreSQL 使用的技术。

每种预处理语句表示都有其优缺点。本文旨在阐明这些优点和缺点中的一些。

1.1. 如何提供反馈

本文从 SQLite 原始作者的角度撰写。如果你不同意本文中提出的任何观点，欢迎在SQLite 论坛上提出更正和/或相反的意见。你也可以直接给作者发送邮件。

1.2. “字节码”的定义

SQLite 生成的字节码可能与许多读者对字节码的理解略有不同。例如，Java 虚拟机或WebAssembly使用的字节码几乎完全由低级操作组成，类似于物理 CPU 实现的基本数学运算、比较、条件跳转以及在不同内存位置之间移动内容的指令。SQLite 的字节码也有这些低级指令。但 SQLite 字节码还包含一些特定于数据库引擎需求的高级操作。以下是其中的几个示例：

  • OP_Column → 从当前游标指向的数据库行的第 N 列中提取值。

  • OP_CreateBtree → 在数据库文件中为新的 B 树分配空间。

  • OP_ParseSchema → 重新读取并解析全部或部分sqlite_schema 表，并相应刷新内部符号表。

  • OP_SeekGE → 将特定 B 树上的游标移动到第一个大于或等于给定键的条目。

  • OP_Next → 将特定 B 树上的游标前进到 B 树中的下一个条目，并跳转；如果该 B 树中没有更多条目，则继续执行下一条指令。

换句话说，SQLite 所使用的“字节码”与其说是一组 CPU 指令，不如说是按特定顺序运行的一组数据库原语。

1.3. “抽象语法树”或“AST”的定义

“抽象语法树”或 AST 是一种描述某种形式语言中程序或语句的数据结构。在此上下文中，形式语言是 SQL。AST 通常以对象树的形式实现，每个对象代表整个 SQL 语句的一个小部分。AST 自然地从形式语言的解析器中生成。常见的技术是使用LALR(1) 解析器。使用此类解析器时，每个终结符都携带元数据，该元数据将成为 AST 的叶节点；每个非终结符都携带元数据，该元数据将成为整体 AST 的子分支。随着解析器“归约”语法规则，AST 的新节点被分配并连接到子节点。解析完成后，语法的开始符号保存了 AST 的根。

AST 是一个对象树。但 AST 并不适合作为预处理语句。生成之后，AST 需经过各种转换才能执行。需要解析符号、检查语义规则、应用优化，以将输入 SQL 语句转化为执行更快的不同形式。最后，AST 需要被翻译成更适合执行的替代表示。

有人将 MySQL 和 PostgreSQL 使用的、作为可执行形式的对象树称为 AST。这可能是不恰当的，因为当对象树准备好执行时，它已经被改动得与原始 SQL 文本几乎毫无相似之处。这种混淆部分源于最终的预处理语句对象和原始 AST 都是对象树。通常的做法是，将解析器直接生成的原始 AST 逐步、分多次转换，直到最终完全转化为一棵不再严格是 AST 而可用于求值以生成结果的对象树。在此过程中，并没有明确的界点表明对象树何时不再是 AST 而成为预处理语句。因此，由于 AST 与预处理语句之间没有清晰界限，人们经常将以对象树形式表示的预处理语句称为“AST”，尽管这种描述并不精确。

1.4. 数据流编程

数据流编程是一种编程风格，其中各个节点专注于执行整体计算的一小部分。每个节点从其他节点接收输入，并将输出发送给其他节点。因此，这些节点组成一张有向图，将输入传递至输出。

“数据流程序”或许比“AST”更能准确描述 SQL 数据库引擎作为预处理语句所使用的对象树。

2. 编译为字节码的优势

SQLite 编译为字节码，SQLite 的开发人员对这种方法非常满意。原因如下：

2.1. 字节码更易理解

一组扁平的操作码可以轻松打印出来，准确看出 SQL 语句如何被实现。当你在 SQL 语句前加上“EXPLAIN”关键字时，SQLite 不会真正执行 SQL，而是输出实现该 SQL 所用的字节码。

字节码之所以便于这样做，是因为字节码程序可被轻松表示为表格。在 SQLite 的字节码中，每条指令有一个操作码和五个操作数。因此，预处理语句可以像查询一个六列表一样呈现。

对象树表示更难以用人类可读的形式公布。组成树的对象往往千差万别，因此想要为这些对象设计一个一致且简单的表格表示就比较棘手。即使你想出了这样的表格表示，也几乎肯定会有六列以上，甚至更多。将对象树呈现为表格的问题十分复杂，据我所知没有人这么做。因此，没有任何一个对象树型数据库引擎在其“EXPLAIN”输出中提供与 SQLite 同样详细的内容。

2.2. 字节码更易调试

字节码在前端解析与分析以及后端评估之间提供了明确的分隔。当出现问题（错误的答案和/或性能差）时，开发人员可以检查字节码，迅速判断问题是源于前端分析还是后端数据存储部分。

在 SQLite 的调试构建中，PRAGMA vdbe_trace=ON; 命令会在控制台上显示字节码执行的跟踪。

2.3. 字节码可以逐步运行

以字节码书写的 SQL 语句可以逐步评估。例如，可以运行语句直到只生成第一行输出。语句随后暂停，直到再次执行步进。无需等语句执行完成即可查看第一行输出。

在对象树设计中实现这一点更困难。当预处理语句是对象树时，执行通常通过遍历整棵树来实现。要在计算中途暂停语句，就必须将堆栈回退到调用者，同时保存足够的状态以便在上次停止处继续求值。这并非不可能，但难度很大，至少我从未见过有人真正这么做过。

大多数 SQL 数据库引擎实际上并不需要逐步执行预处理语句，因为大多数 SQL 数据库引擎采用客户端/服务器架构。在客户端/服务器引擎中，单条 SQL 语句被发送到服务器，然后完整的回复一次性通过网络返回。因此，每条语句一次性运行到底。但 SQLite 不是客户端/服务器。SQLite 是一个在与应用程序相同的地址空间并使用相同堆栈运行的库。能够轻松且可靠地对 SQL 语句执行逐步运行对 SQLite 来说非常重要。

2.4. 字节码更紧凑

SQLite 生成的字节码通常比解析器生成的 AST 更小。在 SQL 文本的初始处理中（在调用sqlite3_prepare()等期间），AST 和字节码会同时存在于内存中，因此使用更多内存。但那只是暂时状态。AST 很快被丢弃，其内存被回收，甚至在调用sqlite3_prepare() 返回之前就完成了，因此最终的预处理语句在字节码表示下所占用的内存比在 AST 时更少。这一点很重要，因为对sqlite3_prepare() 的调用是短暂的，但预处理语句通常会被缓存以便重用，并在内存中保持较长时间。

2.5. 字节码更快速

我相信预处理语句的字节码表示运行得更快，因为每一步计算需要做出的决策更少。上句中的“相信”被强调——因为从未有人投入多年时间去生成等价的字节码和对象树表示的预处理语句，以便确切比对哪种表示运行更快，因此要通过实验验证此结论十分困难。我们确实知道SQLite 非常快，但我们没有与其他 SQL 数据库做出色对比的好资料，因为其他数据库花大量时间处理客户端/服务器消息，而很难将消息往返开销与实际处理时间区分开来。

3. 编译为对象树的优势

SQLite 的开发者认为字节码方法是最优选择，至少对于 SQLite 所试图填补的应用场景如此，但对象树处理 SQL 的方法确实比字节码拥有一些优势。权衡总是存在的。

3.1. 查询规划决策可以推迟到运行时

当预处理语句是字节码时，一旦生成字节码，算法就固定了，除非完全重写字节码，否则无法修改。这在对象树预处理语句中并非如此。对象树更容易在运行中动态修改。查询计划是可变的，可以根据查询进展在运行过程中微调。因此，查询可以动态自我调优。

3.2. 数据流程序易于并行化

在数据流程序中，每个处理节点可以分配给不同的线程。需要一种线程安全的队列机制，将中间结果从一个节点传递到下一个节点。但通常不需要在每个节点内部使用同步原语。节点调度非常简单：当节点有可用数据且其输出队列有空间时，该节点即可运行。

这对于为在大型多核服务器上运行大规模分析查询（OLAP）而设计的数据库引擎来说是一个重要考量。SQLite 的主要关注点是物联网环境中的事务处理（OLTP），因此在 SQLite 中将预处理语句表示为数据流程序的需求较小。
