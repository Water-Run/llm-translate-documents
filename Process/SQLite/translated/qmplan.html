1. 概述

这是 SQLite 的质量管理计划。

质量管理文档往往扩展成装订夹里充满无人阅读的难懂术语。本文档力求打破这一模式，做到简明且有用。

本文档的灵感来自DO-178B。在各种质量标准中，DO-178B 似乎具有最高的文书工作效用比。即便如此，完整实施 DO-178B 所需的文档量仍然庞大。SQLite 力求灵活且低礼节，因此省略了许多 DO-178B 所需的文档。我们仅保留那些确实能够提高 SQLite 这种开源软件项目质量的部分。

本文档的目的是概述 SQLite 开发团队的日常运作方式，他们不断增强 SQLite 软件并努力提升其本已很高的可靠性。若一位称职的开发者在阅读本文档后能够迅速融入开发团队，则说明本文档达到了目标。

1.1. 关于本文档

本质量管理计划最初是通过查阅 DO-178B 第 11 节（第 48 至 56 页）中对输出的描述，并记录那些看起来与 SQLite 相关的元素。文本随后会根据 SQLite 质量流程的增强而修订。

2. 软件开发计划

本节融合了 DO-178B 中关于软件认证方面的计划以及软件开发计划部分。

请参阅SQLite 简介以了解 SQLite 软件的概况、功能与特色。

2.1. 软件生命周期

SQLite 使用持续集成流程。软件不断增强与优化。最新的主干提交常被用于内部的关键任务操作。

没有预设的发布周期。发布通常在功能增强和/或错误修复达到临界质量时进行。历史上每年发布约五到六次。SQLite 的用户按需从网站获取新版本。

2.1.1. 维护版本

常规维护版本包含功能增强、性能提升及/或非关键问题的修复。主版本号形式为“3.N.0”的某个整数 N。详见版本编号惯例文档。

即将发布的维护版本会在预计发布前约两周在SQLite 论坛中宣布。发布前大约一周，主开发者会宣布“停止修改”，此后主干仅允许提交错误修复。新的发布清单会被创建并根据需要更新。当清单项被验证后，即勾选并标绿。所有清单项目变为绿色时即进行发布。该过程通常需要约一周时间。

2.1.2. 补丁版本

偶尔会发现严重问题，必须针对常规维护版本发布一个小型“补丁”版本。补丁版本与维护版本的区别在于相对于上一个版本所更改的代码行数较少。我们尽最大努力通过确保维护版本零缺陷来避免补丁版本。

补丁版本是否包含发布清单依具体问题而定。这由项目负责人判断。

2.2. 发布历史

文档系统自动维护了过去发布的年表，以及一个含有变更摘要的完整 SQLite 发布列表。

2.3. 计划

SQLite 有着远景规划。规划基于假设 SQLite 将在至少 2050 年之前得到使用和维护。所有代码均以便于尚未出生的人未来阅读和维护为目标。代码附带了详尽的注释，目的在于帮助未来的开发者更容易理解逻辑及编写原因。

3. 软件开发环境

SQLite 使用可移植的 C 语言编写。开发工作在 Linux、Mac 和 Windows 工作站的混合环境中进行。开发者使用命令行工具，并尽可能避免使用集成开发环境（IDE）。所有开发者都必须熟练掌握 Unix 命令行。

从官方源代码编译和测试 SQLite 的最小环境如下：

  • 具有 32 位或 64 位地址空间的主机。操作系统可以是 Linux、Mac、Windows、*BSD、Solaris 或其他。
  • 支持 C99 的编译器，如 GCC（含 Windows 的 MinGW 版本）、Clang 或 MSVC
  • 用户自选的支持 UTF-8 文本的文本编辑器。
  • Tcl 版本 8.6 或更高。
  • “make” 工具，或者 Windows 上的“nmake”。

Tcl 脚本语言用于帮助将官方源代码转换为集合生成文件并管理测试。SQLite 本身不会直接使用 Tcl（除非通过编译时选项）。SQLite 集合源的最终用户不需要 Tcl。

构建命令行界面（CLI）时，拥有以下第三方库会很有帮助，但不是必须的：

  • zLib
  • readline 或 editline 或 linenoise 用于 命令行编辑。

完整的发布测试还需要额外的软件，

  • valgrind
  • gcov

SQLite 期望在所有现代操作系统、所有现代计算机架构以及所有现代 C 编译器上都以相同方式运行，并使用完全相同的磁盘格式。开发者持续在尽可能多的不同平台上测试 SQLite。

4. 软件验证计划

SQLite 的测试过程在测试文档中描述。测试目标包括：

  • 在交付配置中达到 100% MC/DC
  • 源代码和目标代码的测试
  • 在多平台和多编译器上测试
  • 模糊测试
  • 代码变更检查
  • 代码的动态和静态分析

测试过程由发布测试清单控制。清单简洁地总结了验证 SQLite 全面所需的所有步骤，并记录每个步骤由谁何时完成。

每个发布的清单项集可能会有所更新。每个发布清单的内容与完整历史都会保留以作记录。

5. 软件配置管理

5.1. 版本控制

SQLite 源代码使用Fossil版本控制系统进行管理。Fossil 是专为支持 SQLite 开发而编写的。Fossil 同时提供分布式版本控制和问题追踪。

5.2. 存活能力

所有代码都在三台独立机器上归档： https://sqlite.org、https://www2.sqlite.org、https://www3.sqlite.org。 这些机器位于不同城市（分别是达拉斯、纽瓦克和旧金山），由两家不同的托管公司管理（前两台由Linode托管，第三台由Digital Ocean托管）。这种多样性旨在避免单点失效。

达拉斯的主机https://sqlite.org/是主要服务器，也是大多数人使用的那台。其他两台被视为备份。

除了官方仓库，开发者通常也会在个人电脑上保留所有软件的完整克隆副本。互联网上也存在其他克隆。

5.3. 仓库

SQLite 源代码被拆分到多个仓库中，每个仓库在下文的单独部分中描述。

5.3.1. SQLite 源代码

SQLite 源代码与TCL 测试套件存储在同一个仓库中。要构建 SQLite 源代码，只需这一份仓库。源代码仓库是公开的，任何匿名访客都可以在互联网上读取。

  • 主要位置: https://sqlite.org/src
  • 备份 A: https://www2.sqlite.org/src
  • 备份 B: https://www3.sqlite.org/cgi/src
  • GitHub 镜像: https://github.com/sqlite/sqlite/

5.3.2. SQLite 文档源

文档源包括文档文本和图像，以及构建 SQLite 网站文档所需的脚本与 makefile。本文档包含在文档源中。文档源独立于源代码，储存在另一个仓库。文档源仓库对外公开可读。

用于生成文档的 makefile 和脚本从文档源仓库中的基线文档提取文本。额外文本从 SQLite 源代码中的注释摘录。需求覆盖信息从属于源仓库的TCL 测试套件中的特定注释提取，以及位于另一个私有仓库中的TH3测试套件中的注释提取。

  • 主要位置: https://sqlite.org/docsrc
  • 备份 A: https://www2.sqlite.org/docsrc
  • 备份 B: https://www3.sqlite.org/cgi/docsrc

5.3.3. SQL 逻辑测试

SQL 逻辑测试是一组测试用例，用于证明 SQLite 与其它 SQL 数据库引擎行为一致。这些测试托管在一个独立的公共仓库中。

  • 主要位置: https://sqlite.org/sqllogictest
  • 备份在私有服务器上

5.3.4. 测试平台 #3

测试平台 #3或TH3测试套件是一组用于在交付配置中将 SQLite 测试到 100% MC/DC 的私有测试用例。TH3 源码在与其他 SQLite 仓库相同的服务器上提供，但由于为专有，因此与其他仓库不同。TH3 代码仅对 SQLite 开发者开放。

  • 主要位置: https://sqlite.org/th3
  • 备份 A: https://www3.sqlite.org/cgi/th3
  • 私有服务器上的其他备份

5.3.5. Dbsqlfuzz

dbsqlfuzz 模块是一个基于libFuzzer的 SQLite 模糊测试器。Dbsqlfuzz 同时对 SQL 和数据库文件进行模糊测试，并使用定制的变异器。

Dbsqlfuzz 似乎比其他任何可用模糊测试器更擅长发现问题。因此其代码保持私有。我们不希望黑客获得这项技术。

  • 主要位置: https://sqlite.org/dbsqlfuzz
  • 备份 A: https://www3.sqlite.org/cgi/dbsqlfuzz
  • 私有服务器上的其他备份

5.4. 软件验证结果

发布测试通过清单执行。每个清单的当前状态和完整变更历史存储在单独的 SQLite 数据库文件中。这些文件不受版本控制，但会在私有备份服务器上保留多个副本。

运行清单的软件源代码存储在它自己的 Fossil 仓库中，位于 https://sqlite.org/checklistapp。

6. 软件需求标准与数据

在 SQLite 项目中，“需求”就是项目文档。文档文本中的特殊标记会标识单个需求。需求编号基于归一化需求文本的加密哈希，因此无法更改需求文本而不同时更改编号。

文档文本（也就是需求文本）取自前文所述的 SQLite 文档源仓库以及实现中的注释。用于构建文档的 makefile 也在文档源仓库。

构建文档时，需求会被识别并标记。文档构建过程还会扫描验证每个需求的测试用例，并构建一个矩阵，展示哪些需求已被测试以及具体测试用例。

7. 软件设计与编码标准

SQLite 的客观编码标准很少：

  • 两个空格缩进
  • 每行不超过 80 个字符
  • 不使用制表符

所有其他设计与编码规则都具有主观性。目标是让软件在 2050 年之前依然可读、可维护。为此，我们追求简洁而有用的注释（不使用模板化语句）、精心选择的变量名，以及对每个数据结构含义和每段代码作用的详细解释。

8. 问题报告

所有问题都会迅速修复。SQLite 软件中没有持续存在的问题。

SQLite 使用的Fossil 版本控制系统内建问题单跟踪支持。这个内置的票据系统用于跟踪和记录许多历史问题。

SQLite 社区论坛是任何互联网用户都可以前往提问或报告 SQLite 错误的地方。第三方发现的错误通常首先在论坛报告。论坛报告的错误有时会转为票据，尽管最近的做法是直接在论坛处理这些错误。论坛具备出色的全文搜索功能，被镜像到多台机器，和票据系统一样可搜索也具备存活能力，因此将论坛起源的 bug 报告再复制到票据系统似乎没有必要。论坛的公共位置包括：

  • 主要位置: https://sqlite.org/forum
  • 备份 A: https://www2.sqlite.org/forum
  • 备份 B: https://www3.sqlite.org/cgi/forum

与源仓库一样，论坛也会同步到各个私有机器。请注意，由于 Fossil 的工作方式，“备份”不仅仅是只读备份。它们也可以作为数据输入。所有输入的内容会同步到所有仓库，不管是在哪一个仓库中插入的。
