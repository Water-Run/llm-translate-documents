SQLite 中的隔离

数据库的“隔离”属性决定一个操作对数据库所做的更改何时会对其它并发操作可见。

数据库连接之间的隔离

如果使用两个不同的数据库连接（即分别调用sqlite3_open()后得到的两个独立的sqlite3对象）对同一个数据库进行读取和写入，且这两个数据库连接之间没有共享缓存，那么读取者只能看到写入者已经提交的完整事务。写入者尚未提交的部分更改对于读取者是不可见的。无论这两个数据库连接是在同一线程、同一进程的不同线程，还是在不同进程中运行，这种行为都是成立的。这也是 SQL 数据库系统通常且预期的行为。

只要在共享缓存模式下read_uncommitted pragma保持关闭，上述段落的内容依然成立（也就是说，独立的数据库连接之间还是彼此隔离）。read_uncommitted pragma默认是关闭的，因此如果应用程序没有显式开启它，它就会一直保持关闭。因此，除非使用read_uncommitted pragma来改变默认行为，否则在共享缓存中共享同一缓存的不同数据库连接在写入者提交事务之前彼此之间是不可见的。

如果两个数据库连接共享同一个缓存且读取者启用了read_uncommitted pragma，那么读取者能够在写入者事务提交之前看到写入者所做的更改。共享缓存模式与read_uncommitted pragma的结合是唯一一种让一个数据库连接看到另一个数据库连接的未提交更改的方式。在所有其他情况下，独立的数据库连接之间完全隔离。

除非在共享缓存中启用了PRAGMA read_uncommitted，SQLite 中的所有事务都表现出“可串行化”隔离。SQLite 通过实际将写操作串行化来实现可串行化事务。一次只能有一个写入者对 SQLite 数据库进行写入。虽然同一时间可以打开多个数据库连接，这些连接都可以向数据库文件写入，但它们必须轮流进行。SQLite 自动使用锁来串行化写操作；这是使用 SQLite 的应用程序无需担心的事情。

隔离与并发

SQLite 通过出现在数据库文件所在目录中的临时日志文件来实现隔离、并发控制（以及原子性）。有两种主要的“日志模式”。旧的“回滚模式”对应于向journal_mode pragma传入“DELETE”、“PERSIST”或“TRUNCATE”选项。在回滚模式下，更改直接写入数据库文件，同时还构建一个独立的回滚日志文件，用于在事务回滚时将数据库恢复到原始状态。回滚模式（特别是 DELETE 模式，即每个事务结束时从磁盘删除回滚日志）是当前默认行为。

自 版本 3.7.0（2010-07-21）起，SQLite 也支持“WAL 模式”。在 WAL 模式下，更改不会写入原始数据库文件，而是写入一个独立的“预写日志”或“WAL”文件。在事务提交之后，这些更改会通过称为“检查点”的操作从 WAL 文件移动回原始数据库。可以通过执行“PRAGMA journal_mode=WAL”来启用 WAL 模式。

在回滚模式中，SQLite 通过锁定数据库文件并在每个写事务进行期间阻止其他数据库连接的读取来实现隔离。读取者可以在写入开始阶段活跃，在任何内容被刷新到磁盘之前，在所有更改仍保存在写入者的私有内存中时仍然可以读取。但在对数据库文件执行任何更改之前，必须临时驱逐所有读取者，以便写入者获得对数据库文件的独占访问权。因此，读取者被锁定在数据库之外，无法看到未完成的事务。只有在事务完全写入并同步到磁盘后、提交完成后，读取者才被允许重新进入数据库。因此读取者永远看不到部分写入的更改。

WAL 模式允许读取者和写入者同时存在。之所以能够实现这一点，是因为更改不会覆盖原始数据库文件，而是写入独立的预写日志文件。这意味着读取者可以在写入者向 WAL 文件追加内容的同时继续读取原始数据库文件中旧的、未更改的内容。在WAL 模式下，SQLite 表现为“快照隔离”。当一个读取事务开始时，该读取者会持续看到在读取事务开始时刻原始数据库文件的静态“快照”。在读取事务仍然活跃的情况下提交的任何写事务对该读取事务仍然不可见，因为读取者看到的是来自之前某个时间点的数据库文件快照。

举例：假设有两个数据库连接 X 和 Y。X 使用BEGIN开始一个读取事务，并执行一个或多个SELECT语句。随后 Y 执行一个UPDATE语句来修改数据库。X 之后可以对 Y 修改的记录执行SELECT，但 X 看到的是旧的未修改条目，因为在 X 持有读取事务期间，Y 的更改对 X 是不可见的。如果 X 想看到 Y 所做的更改，就必须结束其读取事务并重新开始一个新事务（即先执行COMMIT，然后再执行一次BEGIN）。

另一个例子：X 使用BEGIN和SELECT开始一个读取事务，随后 Y 使用UPDATE对数据库做出更改。然后 X 试图使用UPDATE对数据库做进一步更改。由于 X 正在查看的数据库快照不再是数据库的最新版本，因此 X 尝试将事务从读取事务升级为写事务会因SQLITE_BUSY_SNAPSHOT错误而失败。如果允许 X 写入，它将分叉数据库文件的历史，这是 SQLite 所不支持的。为了让 X 写入数据库，它必须先释放它的快照（例如执行ROLLBACK），然后再通过新的BEGIN开始一个新的事务。

如果 X 启动一个最初只读取的事务，但 X 知道最终会想要写入，并且不希望为其他连接抢先而导致的可能出现的 SQLITE_BUSY 错误而烦恼，则 X 可以发出BEGIN IMMEDIATE来启动事务，而不是普通的 BEGIN。BEGIN IMMEDIATE命令会直接启动一个写事务，从而阻止所有其他写入者。如果BEGIN IMMEDIATE操作成功执行，那么该事务中的后续操作就永远不会因为SQLITE_BUSY错误而失败。

同一数据库连接上的操作之间没有隔离

SQLite 为独立的数据库连接之间的操作提供隔离。然而，对同一数据库连接内发生的操作之间并不存在隔离。

换句话说，如果 X 使用BEGIN IMMEDIATE开始一个写事务，然后执行一个或多个UPDATE、DELETE和/或INSERT语句，那么这些更改会对在数据库连接 X 中评估的后续SELECT语句可见。不同数据库连接 Y 上的SELECT语句在 X 事务提交之前不会显示任何更改。但 X 中的SELECT语句会在提交之前看到这些更改。

在单个数据库连接 X 中，SELECT语句总是能看到在该 SELECT 语句开始之前完成的所有数据库更改，无论这些更改是否已经提交。而该 SELECT 语句显然不会看到在其完成之后发生的任何更改。但问题是，在 SELECT 语句执行的过程中发生的更改会怎样？假设一个 SELECT 语句启动后，sqlite3_step() 接口已经返回了大约一半的结果，此时应用程序执行了一些UPDATE语句来修改该 SELECT 语句正在读取的表，随后又继续调用sqlite3_step()来完成 SELECT 语句。后续的 sqlite3_step() 会看到 UPDATE 所做的更改吗？答案是这种行为是未定义的。具体来说，SELECT 语句是否会看到并发更改取决于正在运行的 SQLite 版本、数据库文件的模式、是否执行过ANALYZE以及查询的细节。在某些情况下，甚至可能取决于数据库文件的内容。没有一个好的方法来确定 SELECT 语句是否会看到在该 SELECT 启动后由相同数据库连接所做的更改。因此，开发者应当谨慎，避免对这种情况下的行为做出假设。

如果应用程序在单个表上发出类似“SELECT rowid, * FROM table WHERE ...”的 SELECT 语句，并通过sqlite3_step()逐行遍历输出，则该应用可以安全地删除当前行或任何之前的行，使用“DELETE FROM table WHERE rowid=?”。同样地，即使某行预计稍后会出现但尚未出现，应用删除该行也是安全的（即不会破坏数据库）。不过，如果未来某一行被删除，该行可能会在随后的 sqlite3_step() 中再次出现，甚至出现在删除之后；也可能不会。这种行为是未定义的。应用程序还可以在 SELECT 语句运行时向表中插入新行，但这些新行是否会出现在后续的 sqlite3_step() 中是未定义的。应用也可以 UPDATE 当前行或任何之前的行，尽管这样做可能导致该行在随后一次 sqlite3_step() 中再次出现。只要应用程序准备好处理这些不确定性，这些操作本身是安全的，不会损坏数据库文件。

就前两段而言，两个拥有相同共享缓存并开启了PRAGMA read_uncommitted的数据库连接被认为是同一个数据库连接。

总结

  1. SQLite 中的事务是可串行化的。

  2. 一个数据库连接所做的更改在提交之前对所有其他数据库连接都是不可见的。

  3. 一个查询会看到在该查询开始之前在同一数据库连接上完成的所有更改，无论这些更改是否已经提交。

  4. 如果在查询启动后但在查询完成之前，同一数据库连接上发生了更改，则该查询是否能够看到这些更改是未定义的。

  5. 如果在查询启动后但在查询完成之前，同一数据库连接上发生了更改，则该查询可能会返回重复的行，或者返回之前已经被删除的行。

  6. 就前四项而言，使用相同共享缓存并启用了PRAGMA read_uncommitted的两个数据库连接被视为同一个数据库连接，而不是独立的连接。
