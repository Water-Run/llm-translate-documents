1. SQLite 如何存储数字

SQLite 将整数值存储在 64 位 二进制补码 格式¹。 这提供了 -9223372036854775808 到 +9223372036854775807 的存储范围， 包含两端。该范围内的整数是精确的。

所谓“REAL”或浮点值以 IEEE 754 Binary-64 格式¹ 存储。 这提供了正值范围大约在 1.7976931348623157e+308 与 4.9406564584124654e-324 之间，对应 的负值范围也成立。Binary64 也可以是 0.0（以及 -0.0）、正负无穷或“NaN”（非数）。 浮点值是近似的。

请特别注意上一段的最后一句：

浮点值是近似的。

如果你需要的是精确答案，那么不应该在 SQLite 或任何其他产品中使用 Binary64 浮点值。这不是 SQLite 的限制， 而是浮点数设计固有的数学限制。

—
¹ 例外：R 树扩展将信息作为 32 位浮点或整数值存储。

1.1. 浮点精度

SQLite 承诺保留浮点值的 15 位最重要的有效数字。 然而，它并不保证对浮点值的计算精度，因为无法给出这样的保证。 对浮点值执行数学运算会引入误差。 例如，考虑试图减去两个数量级相近的浮点数会发生什么：

1152693165.1106291898
-1152693165.1106280772
0.0000011126

上面显示的结果（0.0000011126）是正确答案。但如果你使用 Binary64 浮点来做这个计算，得到的结果是 0.00000095367431640625——约 14% 的误差。如果你的程序中执行很多类似计算，误差会累积，以至于最终结果可能完全没有意义。

误差产生的原因是每个数字只有大约前 15 位有效数字被准确存储，而被减的两个数字之间的第一个差异出现在第 16 位。

1.2. 浮点数

Binary64 浮点格式每个数字使用 64 位。因此存在 1.845e+19 种不同的浮点值。 另一方面，在 1.7977e+308 到 4.9407e-324 的范围内有无限多的实数。 因此 Binary64 不可能表示该范围内的所有实数，需要使用近似。

一个 IEEE 754 浮点值是一个整数乘以二的幂：

M × 2E

M 是“尾数”，E 是“指数”。两者都是整数。

对于 Binary64，M 是一个 53 位整数，E 是一个带偏移的 11 位整数， 表示的值范围在 -1074 到 +972 之间，含两端。

（附注：IEEE 754 的通常描述更加复杂，如果你真的想理解 IEEE 754 的细节、优点和局限，需要了解这些额外的复杂性。但这里展示的整数描述虽然不完全精确，却更容易理解，足以用于本文的目的。）

1.2.1. 无法表示的数字

并非所有具有少于 16 个有效数字的十进制数字都可以精确表示为 Binary64 数。 事实上，大多数带有小数部分的十进制数字都没有精确的 Binary64 等价。 例如，如果数据库列用于存储按美元和美分计的商品价格，那么唯一可以精确表示的美分值是 0.00、0.25、0.50 和 0.75。 小数点右边的其他数字都会导致近似。 如果你提供一个 “47.49” 的“价格”值，该数字会被表示为：

6683623321994527 × 2-47

该值计算出来是：

47.49000000000000198951966012828052043914794921875

这个数字非常接近 47.49，但并不精确，略微偏大。 如果我们将 M 减 1 变为 6683623321994526，得到下一个更小的 Binary64 值，则是：

47.4899999999999948840923025272786617279052734375

这个第二个数字偏小。 第一个数字更接近期望值 47.49，因此被使用。但它仍然不精确。在 IEEE 754 中，大多数十进制值都是这样。记住我们在上面强调的关键点：

浮点值是近似的。

如果你只记住浮点值的一件事，那就不要忘记这个关键概念。

1.2.2. 是否足够接近？

IEEE 754 Binary64 提供的精度对于大多数计算已经足够。 例如，如果“47.49” 表示一个价格，而年通胀率为 2%，那么该价格每秒大约上涨 0.0000000301 美元。 记录的 47.49 的误差相当于约 66 纳秒的通胀值。 所以如果你输入时 47.49 是精确的，那么通胀的影响会使实际值在不到十分之一百万分之一秒内 恰好等于实际存储的值 （47.4900000000000019895196601282805204391479492187）。 这个精度对于大多数用途来说肯定是足够的吧？

2. 处理浮点数的扩展

2.1. ieee754.c 扩展

ieee754 扩展在其 Binary64 表示和 M×2E 格式之间转换浮点数。 换句话说，在表达式：

F = M × 2E

ieee754 扩展在 F 和 (M,E) 之间相互转换。

ieee754 扩展不是 amalgamation 的一部分，但默认包含在 CLI 中。 如果你想在应用程序中包含 ieee754 扩展，需要单独编译并加载。

2.1.1. ieee754() 函数

ieee754(F) SQL 函数接受一个浮点参数作为输入，并返回如下字符串：

'ieee754(M,E)'

只是将 M 和 E 替换为该浮点数的尾数和指数。例如： 

sqlite> .mode box
sqlite> SELECT ieee754(47.49) AS x;
┌───────────────────────────────┐
│               x               │
├───────────────────────────────┤
│ ieee754(6683623321994527,-47) │
└───────────────────────────────┘

反过来，ieee754() 的两个参数版本接受 M 和 E 值，并将它们转换回对应的 F 值： 

sqlite> select ieee754(6683623321994527,-47) as x;
┌───────┐
│   x   │
├───────┤
│ 47.49 │
└───────┘

2.1.2. ieee754_mantissa() 和 ieee754_exponent() 函数

ieee754() 的单参数形式的文本输出对人类可读性很好，但在更大的表达式中使用起来很笨重。因此 添加了 ieee754_mantissa() 与 ieee754_exponent() 例程，以返回对应其单个参数 F 值的 M 与 E。 例如： 

sqlite> .mode box
sqlite> SELECT ieee754_mantissa(47.49) AS M, ieee754_exponent(47.49) AS E;
┌──────────────────┬─────┐
│        M         │  E  │
├──────────────────┼─────┤
│ 6683623321994527 │ -47 │
└──────────────────┴─────┘

2.1.3. ieee754_from_blob() 和 ieee754_to_blob() 函数

ieee754_to_blob(F) SQL 函数将浮点数 F 转换为 8 字节 BLOB，它是该数的 Big-Endian Binary64 编码。 ieee754_from_blob(B) 函数则反方向，将 8 字节的 blob 转换为 Binary64 编码表示的浮点值。

例如，如果你在 维基百科 上 查到最小正 Binary64 值的编码是 0x0000000000000001，那么可以这样找到对应的浮点值： 

sqlite> .mode box
sqlite> SELECT ieee754_from_blob(x'0000000000000001') AS F;
┌───────────────────────┐
│           F           │
├───────────────────────┤
│ 4.94065645841247e-324 │
└───────────────────────┘

或者反过来： 

sqlite> .mode box
sqlite> SELECT quote(ieee754_to_blob(4.94065645841247e-324)) AS binary64;
┌─────────────────────┐
│      binary64       │
├─────────────────────┤
│ X'0000000000000001' │
└─────────────────────┘

2.2. decimal.c 扩展

decimal 扩展在作为文本的数字上提供任意精度的十进制运算。 由于数字以任意精度形式作为文本存储，因此无需近似，可进行精确计算。

decimal 扩展目前不是 SQLite amalgamation 的一部分。 不过，它包含在 CLI 中。这个扩展的源代码可在 ext/misc/decimal.c 找到。

decimal 扩展支持以下 SQL 函数和排序序列：

2.2.1. decimal_add(A,B)、decimal_sub(A,B) 和 decimal_mul(A,B) 函数

这些函数分别计算两个输入 A 与 B 的和、差与积，并以文本形式返回结果。 输入可以是十进制文本或数字值。数字输入在执行计算前会转换为十进制文本。

没有 “decimal_div(A,B)” 函数，因为除法的十进制结果并不总是有限长度。

2.2.2. decimal_pow2(N) 函数

decimal_pow2(N) 函数计算 2 的 N 次幂的精确十进制表示。N 必须是介于 -20000 与 +20000 之间的整数。

该函数在 N 较大时可能运行缓慢并占用大量内存。

2.2.3. decimal(X) 和 decimal_exp(X) 函数

decimal(X) 与 decimal_exp(X) 为输入 X 生成十进制表示。 输入 X 可以是整数、浮点数或十进制文本。 decimal_exp(X) 返回指数记数法的结果（以 “e+NN” 结尾），而 decimal(X) 返回纯十进制表示（不带 “e+NN”）。

如果输入 X 是浮点值，它会展开为其精确的十进制等价。例如： 

sqlite> .mode qbox
sqlite> select decimal(47.49);
┌──────────────────────────────────────────────────────┐
│                    decimal(47.49)                    │
├──────────────────────────────────────────────────────┤
│ '47.49000000000000198951966012828052043914794921875' │
└──────────────────────────────────────────────────────┘

2.2.4. decimal_cmp(X) 函数

decimal_cmp(A,B) 函数比较两个十进制值 A 与 B。如果 A 小于、等于或大于 B，结果分别为负、零或正。

2.2.5. decimal_sum(X) 聚合函数

decimal_sum(X) 函数是一种聚合函数，类似内置的 sum() 聚合函数，不同之处在于 decimal_sum() 将结果计算到任意精度，因此是精确的。

2.2.6. decimal 排序序列

decimal 扩展提供了 “decimal” 排序序列，用于按数值顺序比较十进制文本字符串。
