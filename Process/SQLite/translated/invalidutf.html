无效 UTF 策略

1. 垃圾进，垃圾出

关于无效 UTF，SQLite 遵循垃圾进、垃圾出 (GIGO) 的策略。如果你向 SQLite 数据库插入无效 UTF，然后尝试查询这些数据，返回的结果可能并不完全等同于你插入的内容。既然你输入的是垃圾，如果得到不同的垃圾也不用抱怨。

在此讨论中，“无效 UTF” 可以指以下任一情形：

  • UTF-16 中的无效代理对。

  • UTF-8 中的无效多字节序列。

  • 使用比表示单个代码点所需更多的 UTF-8 字节。（例如：用两字节序列 0xc1, 0x01 编码 'A'，而不是单个 0x41 字节。）

  • 字符串中嵌入的 NUL 字符（U+0000）。

  • 无效的组合字符序列。

  • UTF-8 或 UTF-16 字节序列所编码的数字不是定义在 Unicode 中的字符。

1.1. 无效 UTF 永远不会引发内存错误

如果你向 SQLite 数据库插入无效 UTF，那么 SQLite 不保证你会得到怎样的文本。但它承诺，在 SQLite 的内置处理过程中，无效 UTF 不会导致内存错误（数组越界、读取或写入未初始化内存等）。换句话说，无效 UTF 不会导致 SQLite 崩溃。

这个承诺只适用于 SQLite 的核心组件，当然不包括应用程序提供的扩展。如果应用程序添加了新的自定义 SQL 函数、虚拟表、排序序列或其他扩展，而数据库中存在无效 UTF，那么这些无效 UTF 可能会传递给这些扩展。如果无效 UTF 导致这些扩展中的某个崩溃，那是扩展的问题，而不是 SQLite 的问题。

2. 不强制文本格式规则

SQLite 不会尝试强制执行 UTF 格式规则。你可以在 TEXT 字段中插入无效 UTF，SQLite 不会因此报错。它会尽其所能存储这些无效的文本。SQLite 认为自己是一个存储引擎，而不是文本格式验证引擎。

3. 尽力保留文本

SQLite 不承诺始终保留无效 UTF，但会尽力而为。一般而言，如果你向 SQLite 插入无效 UTF，只要你不要求 SQLite 对文本进行任何转换，就会拿回完全相同的字节序列。

例如，如果你向一个设置了 PRAGMA encoding=UTF16LE 的数据库表的 TEXT 列插入带有无效代理的 UTF-16LE，当你稍后使用 sqlite3_column_text16() 查询该列时，你很可能会得到完全相同的无效 UTF-16。然而，如果你将相同的无效 UTF-16LE 内容插入一个 PRAGMA encoding=UTF8 的数据库，它在存储时必须转换为 UTF-8，这可能导致内容发生不可逆的更改。或者，如果你把相同的无效 UTF-16LE 内容插入一个 PRAGMA encoding=UTF16LE 的数据库，但随后使用 sqlite3_column_text() 读取，则在读取过程中必须进行 UTF-16 到 UTF-8 的转换，而该转换可能引入不可逆的变化。

又或者，假设你完全采用 UTF-8（最常见的情况）。无效的 UTF-8 通常会在数据库中原样通过其字节序列。然而，如果你尝试用 substr() 或 replace() 等 SQL 函数转换无效 UTF-8，或者尝试用 LIKE 运算符进行字符串匹配，你可能会得到意想不到的结果。

换句话说，SQLite 不会主动破坏你的无效文本。但当你要求 SQLite 对无效 UTF 进行转换时，并不能保证这些转换是可逆的、甚至是合理的。

4. 数据库模式中的无效 UTF

如果数据库模式包含无效 UTF 的名称（表名、列名、索引名等），SQLite 仍会照常运行。在 SQLite 看来，这些名称只是字节序列。SQLite 并不在乎它们是否有效的 UTF。

当生成错误消息（例如使用 sqlite3_errmsg()）时，有时 SQLite 会将数据库模式的部分嵌入错误信息中。如果这些嵌入的模式元素是无效 UTF，那么生成的错误消息也可能是无效 UTF。同样，PRAGMA integrity_check 及类似语句的输出有时也会包含模式元素的名称。如果这些模式元素名称是无效 UTF，那么命令的输出也会是无效 UTF。
