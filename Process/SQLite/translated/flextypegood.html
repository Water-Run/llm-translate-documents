1. 介绍

SQLite 为开发者提供了自由，让他们可以用任意想要的格式来存储内容，而不必受限于列声明的数据类型。 有些人觉得这个特性很棘手。 有些开发者发现居然可以将文本插入到标记为 INTEGER 的列时会震惊。

本文主张支持 SQLite 中的灵活类型规则。

2. 关于灵活类型

有关 SQLite 灵活类型系统的详细信息，请参阅独立的 SQLite 中的数据类型 文档。 这里是一个简要总结：

  • 在列定义上，数据类型名称是可选的。 一个列定义可以只包含列名而不包含其他内容。

  • 提供数据类型名称时，可以是几乎任何文本。 SQLite 会尝试根据信息推断该列的首选数据类型，但该首选数据类型是建议性的，而不是强制性的。 这个首选数据类型被称为“列亲和性（column affinity）”。

  • 会尝试将传入的数据转换为该列的首选数据类型。 （所有 SQL 数据库引擎都会这么做，不仅仅是 SQLite。） 如果转换成功，一切正常。 但如果转换失败，SQLite 不会报错，而是以原始数据类型存储内容。

  • 上述行为可能导致那些支持严格类型的人感到不便：

    列数据类型该列允许的类型
    INTEGERINTEGER、REAL、TEXT、BLOB
    REALREAL、TEXT、BLOB
    TEXTTEXT、BLOB
    BLOBINTEGER、REAL、TEXT、BLOB

  • 注意 INTEGER 或 REAL 值永远不会最终存储在 TEXT 列中，因为 INTEGER 或 REAL 值可以并且总会被转换为其等价的 TEXT 表示形式。 同样，INTEGER 也永远不会存储在 REAL 列中，因为它总会转换为 REAL。 但 TEXT 并不总是看起来像 INTEGER 或 REAL 值，因此不能总是被转换。而 BLOB 无法被转换为任何其他类型，也没有其他类型可以被转换为 BLOB。

3. 灵活类型有用的场景

一些读者在第一次接触 SQLite 的灵活类型时会问自己“这怎么可能有用？” 这里尝试回答这个问题：

3.1. 属性表

许多应用程序，尤其是那些使用 SQLite 作为应用程序文件格式的情况，需要一个地方来存储各种属性，例如缩略图图像（以 BLOB 值）、简短文本（例如用户姓名）、以及数字、日期和 JSON 值。 创建一个单表来处理这类存储非常方便： 

CREATE TABLE attribute(name TEXT PRIMARY KEY, value) WITHOUT ROWID;

如果没有灵活类型，这样的表需要更加复杂，为每种可能的数据类型设置单独的列。 “value” 列的灵活类型让这个表在概念上更简单、更节省空间，也更易于访问和更新。

在 Fossil 版本控制系统 中，每个仓库都有一个 CONFIG 表，用于以各种可能的数据类型存储各种设置。 Fossil 的用户特定配置文件（~/.fossil 文件）是一个独立的 SQLite 数据库，其中包含一个属性表，该表保存跨所有仓库的用户特定状态。

一些应用使用 SQLite 数据库作为纯粹的键值存储。 数据库模式包含一个类似如下的表： 

CREATE TABLE storage(name TEXT PRIMARY KEY, value ANYTHING);

3.2. json_tree 虚拟表输出的 "value" 列

SQLite 内置的 json_tree 和 json_each 表值函数都有一个 “value” 列，该列可以根据对应 JSON 字段的类型存储 INTEGER、REAL 或 TEXT 类型的值。 例如： 

SELECT typeof(value) FROM json_each('{"a":1,"b":2.5,"c":"hello"}');

上述查询返回三行一列，值分别为 “integer”、“real” 和 “text”。

3.3. 脏数据存储

分析人员有时会遇到某些列包含整数、浮点数和文本混合的 CSV 文件。 例如，从 Excel 导出的 CSV 文件通常具有这种特征。在将这类“脏数据”导入 SQL 数据库时，用灵活类型的列进行导入非常方便。

当然，脏数据并不仅限于来自 Excel 的 CSV 文件。 许多数据源中，一个字段可能包含多种类型的混合。例如，一个数据列有时包含自 1970 年起的秒数，有时则是文本形式的日期字符串。清理这些不一致的表示方式是件很有必要的事，但与此同时，在清理期间能把所有不同的表示暂时存储在同一列中也非常方便。

3.4. 动态编程语言

SQLite 最初是一个后来走向大众的 TCL 扩展。TCL 是一种动态语言，程序员无需关心数据类型。 在内部，TCL 会仔细追踪每个值的数据类型，但对 TCL 程序的开发者和用户来说，一切都像字符串一样。灵活类型与像 TCL 这样的动态编程语言天然契合，因为在动态语言中你无法总是提前预测变量会保存何种数据类型。 所以当你需要将该变量的值存储到数据库时，拥有一个支持灵活类型的数据库会让存储更加容易。

3.5. 数据类型名称的跨兼容性

每个 SQL 数据库引擎似乎都有自己独特的支持数据类型名称集：

  • BIGINT
  • UNSIGNED SMALL INT
  • TEXT
  • VARCHAR
  • VARYING CHARACTER
  • NATIONAL VARYING CHARACTER
  • NVARCHAR
  • JSON
  • REAL
  • FLOAT
  • DOUBLE PRECISION
  • ... 等等 ...

SQLite 接受上述任何名称作为有效类型名，并允许你在该列中存储任何类型的内容，这增加了最初为其他 SQL 数据库引擎编写的脚本也能在 SQLite 中运行的概率。

3.6. 重新利用遗留数据库中未使用或废弃的列

由于 SQLite 数据库文件就是磁盘上的单个文件，一些应用将 SQLite 用作应用程序文件格式。 这意味着在其生命周期中，应用的一个实例可能会与数百或数千个独立的数据库通信，每个都在单独的文件中。 当这类应用多年演进时，底层数据库中的某些列的含义可能会被微妙地改变。 或者，可能希望将现有列重新用于两个或更多用途。如果列支持灵活类型，这种操作要容易得多。

4. 对灵活类型的看法缺点（附反驳）

以下对灵活类型的看法缺点来自 Hacker News、Reddit 和类似的开发者论坛的无数帖子。如果你能想到其他为什么灵活类型是坏主意的理由，请联系 SQLite 开发团队或在SQLite 论坛发表帖子，以便将你的观点加入列表。

4.1. 我们以前从未这样做过

许多灵活类型的怀疑者只是表达震惊与难以置信，却没有提供为何认为灵活类型是坏主意的理由。没有支持性的论据，只能假设他们不喜欢灵活类型是因为它与他们惯用的方式不同。

可能，很多对 SQLite 灵活类型感到惊讶的开发者是因为他们此前从未遇到过这种东西。 所有先前接触的数据库，尤其是 SQL 数据库，都涉及严格类型，而他们对 SQL 的认知模型将严格类型视为基础功能。 灵活类型颠覆了他们的世界观。

是的，灵活类型是一种全新的思维方式，用于在 SQL 数据库中处理数据。但新颖并不一定糟糕。 有时，尤其在灵活类型的案例中，创新能带来改进。

4.2. 严格类型强制有助于防止应用程序错误

在许多程序员中已经成为一种教条：防止应用程序错误的最好方式是严格的类型强制。但我没找到支持这一点的证据。

诚然，严格类型强制确实有助于在贴近机器硬件的低级语言（如 C 和 C++）中防止某些类型的错误。但在所有数据都以某种“Value”超类传递的高层抽象语言中，这似乎并不成立。 当一切都是 Value 对象时，具体的数据类型就不再重要。

本技术说明由 SQLite 的原作者撰写。我写了 27 年的 TCL 程序。TCL 完全没有类型强制。TCL 中的 “Value” 类（称为 Tcl_Obj）可以保存多种不同的数据类型，但它对程序和应用用户呈现的内容是字符串。 这些年我在 TCL 程序中遇到许多错误，但我不记得有哪一次是严格类型系统本可以捕捉到的。我也写了大量 C 代码，跨度 35 年，其中就包括 SQLite 本身。我发现 C 的类型系统对发现和防止问题非常有帮助。对于用 C 编写的Fossil 版本控制系统，我甚至实现了额外的静态分析程序，在编译前扫描 Fossil 源代码，以查找编译器遗漏的问题。这对编译型程序非常有效。

SQL 语言模型是比 C/C++ 更高层的抽象。在 SQLite 中，每个数据项在内存中以 “sqlite3_value” 对象存储。该对象有字符串、整数、浮点数、Blob 以及其他表示方式的子类。所有内容在 SQLite 实现的 SQL 语言中都作为 “sqlite3_value” 对象传递，因此底层数据类型并不重要。我从未发现严格类型强制对像 TCL 和 SQLite 这样拥有单一 “Value” 超类用以表示任意数据元素的语言有帮助。Fossil 在实现中大量使用 SQLite。Fossil 的 14 年历史中出现过很多错误，但我想不起有任何一个错误是严格类型强制在 SQLite 中本可以预防的。有些 C 语言错误可能通过更好的类型强制被捕捉（这也是我编写附加源代码扫描器的原因），但没有 SQL 错误。

基于数十年的经验，我拒绝认为严格类型强制有助于防止应用程序错误。我接受并相信一个略作修改的命题：严格类型强制有助于防止在缺乏单一顶层 “Value” 超类的语言中的应用程序错误。但 SQLite 确实拥有单一的 “sqlite3_value” 超类，因此该格言并不适用。

4.3. 严格类型强制防止数据污染

有些人认为，如果你对模式施加严格约束，尤其对列数据类型进行严格执行，这将有助于防止不正确的数据进入数据库。但这并不属实。 类型强制可能有助于防止极其错误的数据进入系统。但类型强制对防止微妙错误无能为力。

例如，严格类型强制可以成功阻止客户姓名（文本）被插入到整数的 Customer.creditScore 列。另一方面，如果该错误发生，很容易发现问题并找出所有受影响的行。但类型强制对防止客户姓与名颠倒的错误无能为力，因为两者都是文本字段。

通过抑制易于检测的错误并放行难以检测的错误，严格类型强制实际上可能使得定位和修复错误更困难。数据错误往往聚集。如果你有 20 个不同的数据源，大多数数据错误通常来自其中的 2 到 3 个。严重错误（例如在整数列中出现文本）是一个方便的早期警告信号，表明问题可能存在。可以迅速追踪问题来源，并对产生严重错误的数据源施加额外审查，从而有望也修复微妙错误。当严重错误被抑制时，你就失去了一个帮助识别并修复微妙错误的重要信号。

数据错误是不可避免的。无论进行多少类型检查，它们都会发生。严格类型强制只能捕捉其中的一小部分——最明显的那些。它对找到并修复更微妙的错误毫无帮助。而且通过抑制哪个数据源存在问题的信号，它有时会让微妙错误更难定位。

4.4. 其他 SQL 数据库引擎不是这样工作的

因为 SQLite 限制较少、允许做更多事情，所以在其他数据库引擎上工作的 SQL 脚本通常也能在 SQLite 上运行，但最初为 SQLite 编写的脚本可能在更严格的数据库引擎上无法运行。 当开发者使用 SQLite 进行原型设计和测试，并在部署时将应用迁移到更严格的 SQL 引擎时，这可能会导致问题。如果应用（无意中）利用了 SQLite 提供的灵活类型，那么迁移后就会失败。

人们用这个问题来主张 SQLite 应该在数据类型方面更严格。但你完全可以把这个论点反过来说——其他数据库引擎应该在数据类型方面更灵活。毕竟，应用在迁移前在 SQLite 上运行正常。如果严格类型强制真的这么有用，为什么它会让一个此前工作的应用崩溃？

5. 如果你坚持严格类型强制...

从 SQLite 3.37.0 版本（2021-11-27）起，SQLite 通过STRICT 表支持这种开发风格。

如果你发现一个现实案例，其中 STRICT 表阻止了或本可以阻止应用中的某个错误，请在SQLite 论坛发布消息，以便我们将你的经历添加到本文档中。

6. 拥抱自由

如果在 SQL 数据库中使用灵活类型对你来说是新概念，我鼓励你尝试一下。 它可能不会给你带来任何问题，反而可能让你的程序更简单、更易编写和维护。 我认为即使你一开始持怀疑态度，只要愿意试一试灵活类型，最终你会意识到这是更好的方法，并会开始鼓励其他数据库厂商至少支持 ANY 数据类型，如果不能完全实现 SQLite 风格的类型灵活性，那也至少向那个方向发展。

大多数时候，灵活类型并不重要，因为某个列存储的是单一明确定义的类型。但有时候你会遇到这样的情况：拥有灵活类型系统让问题的解决方案更加清晰、更容易。
