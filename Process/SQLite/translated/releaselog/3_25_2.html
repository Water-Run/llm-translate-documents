SQLite 发布 3.25.2 于 2018-09-25

版本 3.25.0（2018-09-15）以来的先前更改：

  1. 添加对窗口函数的支持
  2. 增强ALTER TABLE命令：
    1. 添加支持使用 ALTER TABLE table RENAME COLUMN oldname TO newname 重命名表内的列。
    2. 修复表重命名功能，使其在重命名后也会更新触发器和视图中对该表的引用，参见触发器和视图。
  3. 查询优化器改进：
    1. 避免在聚合查询中加载那些不属于聚合函数且不在 GROUP BY 子句中的列。
    2. IN 提前退出优化：在使用多列索引并且 IN 操作符作用于非最左列时，如果第一个 IN 值没有匹配行，则在继续下一个 IN 值之前先检查是否存在匹配右侧列的行。
    3. 利用传递性尝试在 WHERE 子句中传播常量值。例如，将 “a=99 AND b=a” 转换为 “a=99 AND b=99”。
  4. 为 unix VFS 中的每个 inode 使用独立的互斥锁，而不是所有 inode 共享一个互斥锁，以在多线程环境中提供稍好的并发性。
  5. 增强PRAGMA integrity_check命令，以更好地检测页面空闲列表中的问题。
  6. 在命令行 shell的“.dump”命令中将无穷大输出为 1e999。
  7. 添加SQLITE_FCNTL_DATA_VERSION文件控制。
  8. 添加Geopoly 模块

    错误修复：

  9. 在极其罕见的情况下，由于查询优化器中的多个小缺陷，ORDER BY LIMIT 优化可能导致准备语句字节码进入无限循环。对此问题的修复见工单 9936b2fa443fec03ff25
  10. 在 UPSERT 操作中，如果约束检查顺序被重新安排，需确保在任何约束检查之前对插入内容执行亲和性转换。修复见工单 79cad5e4b2e219dd197242e9e。
  11. 避免在“.eqp full”逻辑关闭后继续使用已经关闭的“.stats on”命令的准备语句。修复见工单 7be932dfa60a8a6b3b26bcf76。
  12. LIKE 优化在左操作数具有数值亲和性且右侧模式为 '/%' 或模式以 ESCAPE 字符开头时，会生成错误的字节码并返回错误结果。修复见工单 c94369cae9b561b1f996d0054b

版本 3.25.1（2018-09-18）以来的先前更改：

  1. 3.25.0 版本中添加到 ALTER TABLE 的额外合理性检查在修改的表具有更新虚表的触发器时，有时会产生误报。误报会导致 ALTER TABLE 回滚，从而保持模式不变。相关工单 b41031ea2b537237。
  2. 3.25.0 版本中对 ORDER BY LIMIT 优化导致的字节码无限循环的修复在某些涉及窗口函数的查询中不起作用，需要额外修正。相关工单 510cde277783b5fb

此补丁发行版（版本 3.25.2，2018-09-25）的更改：

  1. 添加PRAGMA legacy_alter_table=ON命令，使“ALTER TABLE RENAME”命令行为恢复到 SQLite 3.24.0 及更早版本：触发器和视图体内对重命名表的引用不会更新。此新 pragma 为期望旧版 ALTER TABLE RENAME 奇怪行为的旧程序提供兼容性解决方案。
  2. 修复新实现的窗口函数在视图中使用复杂表达式时出现的故障。
  3. 修复与晦涩配置相关的其他编译器警告和轻微问题。
