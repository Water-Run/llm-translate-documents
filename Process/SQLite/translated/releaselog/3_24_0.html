SQLite 3.24.0 版本 发布于 2018-06-04

  1. 添加对 PostgreSQL 风格的 UPSERT 的支持。
  2. 添加对 R 树表中辅助列 的支持。
  3. 添加用于发现 SQLite 所使用 SQL 关键字的 C 语言 API：sqlite3_keyword_count()、sqlite3_keyword_name() 和 sqlite3_keyword_check()。
  4. 添加基于 sqlite3_str 对象的动态字符串 C 语言 API。
  5. 增强 ALTER TABLE，使其将 “true” 与 “false” 识别为 DEFAULT 的有效参数。
  6. 将 sorter-reference 优化作为编译时选项添加。 仅在使用 SQLITE_ENABLE_SORTER_REFERENCES 编译时可用。
  7. 改进 EXPLAIN QUERY PLAN 原始输出的格式，使其在查询计划以及计划各组成部分之间的关系方面提供更清晰的信息。
  8. 在 sqlite3_db_config() API 中添加了 SQLITE_DBCONFIG_RESET_DATABASE 选项。

    命令行工具 (CLI) 增强：

  9. 自动截取原始 EXPLAIN QUERY PLAN 输出并将其重新格式化为 ASCII 艺术图形。
  10. 以 “#” 开头且不在 SQL 语句中间的行会被解释为注释。
  11. 向 “.backup” 命令添加了 --append 选项。
  12. 添加了 “.dbconfig” 命令。

    性能改进：

  13. UPDATE 在数据库文件内容实际未改变时避免执行不必要的底层磁盘写入。例如，当 x 列的值已经是 25 时，"UPDATE t1 SET x=25 WHERE y=?" 不会产生额外的磁盘 I/O。同样地，在对跨多个页面的记录执行 UPDATE 时，仅对实际发生变化的页面子集写入磁盘。这是仅针对底层性能的优化，不会影响触发器或其他更高层 SQL 结构的行为。
  14. 使用 ORDER BY 与 LIMIT 的查询现在会尝试避免计算那些不可能在 LIMIT 限制下出现的行。这可以大幅提升 ORDER BY LIMIT 查询的性能，尤其是在 LIMIT 相对于无限制输出行数量较小时。
  15. 即使 OR 表达式已转换为 IN 表达式，OR 优化 仍被允许继续执行。OR 优化的使用现在也在 EXPLAIN QUERY PLAN 输出中显示得更清晰。
  16. 查询规划器在无法创建持久索引的视图与子查询上更积极地使用 自动索引。
  17. 在适当的时候，在 R 树扩展 中利用一次性 UPDATE 与 DELETE 查询计划。
  18. LEMON 生成的解析器性能得到改进。

    错误修复：

  19. 对于 LEFT JOIN 的右侧表，直接计算表达式的值，而不是从 表达式索引 中读取预计算值，因为表达式索引可能不包含正确的值。工单 7fa8049685b50b5aeb0c2
  20. 不再尝试使用 WHERE 子句中的术语来启用 LEFT JOIN 右侧表的索引查找。工单 4ba5abf65c5b0f9a96a7a
  21. 修复在打开失败错误之后可能发生的 CSV 虚拟表 内存泄露问题。
  22. 修复一个长期存在的问题：由 AUTOINCREMENT 使用的 sqlite_sequence 表架构损坏可能导致崩溃的情况。工单 d8dc2b3a58cd5dc2918a1
  23. 修复 json_each() 函数，使其在输入为简单值而非数组或对象时，其 “fullkey” 列也能返回有效结果。
