SQLite 版本 3.18.1（2017-06-16 发布）

来自版本 3.18.0（2017-03-30）的先前更改：

  1. 添加了 PRAGMA optimize 命令
  2. 由 sqlite_source_id() SQL 函数和 sqlite3_sourceid() C API 返回并在 SQLITE_SOURCE_ID 宏中找到的 SQLite 版本标识符现在是 64 位 SHA3-256 哈希，而不是 40 位 SHA1 哈希。
  3. 向 JSON1 扩展 添加了 json_patch() SQL 函数。
  4. 增强了 LIKE 优化，使其在左侧为任意表达式时也能工作，只要右侧的 LIKE 模式不以数字或减号开头。
  5. 添加了 sqlite3_set_last_insert_rowid() 接口，并在 FTS3、FTS4 和 FTS5 扩展中使用该接口，以确保 sqlite3_last_insert_rowid() 接口始终返回合理的值。
  6. 增强了 PRAGMA integrity_check 和 PRAGMA quick_check，使它们也能验证 CHECK 约束。
  7. 增强了联接的查询计划，能够提前检测空表并在无需多余工作的情况下停止。
  8. 增强了 sqlite3_mprintf() 系列接口和 printf SQL 函数，在使用 “，” 格式修饰符（即在 “%” 和 “d” 之间使用 “,”，例如 "%,d"）时在整数的千位处添加逗号分隔符。
  9. 添加了 -DSQLITE_MAX_MEMORY=N 编译时选项。
  10. 向 命令行 shell 添加了 .sha3sum dot-command 和 .selftest dot-command
  11. 开始强制执行 SQLITE_LIMIT_VDBE_OP。例如，在接受来自不受信任用户的 SQL 查询的系统中，可以利用该限制防止准备语句过大。
  12. 各种性能改进。

    错误修复：

  13. 确保带有排序序列的索引表达式被正确处理。修复了工单 eb703ba7b50c1a5。
  14. 修复了 日期和时间函数 的 “start of ...” 修饰符中的一个错误。工单 6097cb92745327a1
  15. 修复了在复杂递归触发器中可能导致段错误的一个潜在问题，该问题由 3.15.0 版本作为性能优化引入的 OP_Once 操作码错误引起。工单 06796225f59c057c
  16. 在 RBU 扩展 中添加额外的同步操作，以避免停电后可能出现的损坏。
  17. sqlite3_trace_v2() 对嵌套 SQL 语句的输出现在应始终以 “--” 注释标记开始。
