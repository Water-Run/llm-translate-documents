2014-10-17 - 发布 3.8.7

SQLite version 3.8.7 是一次定期安排的维护发布。 建议从所有之前的版本升级。

与前一个版本相比，多数更改是旨在帮助 SQLite 运行得稍微快一点的微优化。 每个单独的优化对性能影响微乎其微。但这些改进累加起来。 在 Linux 上使用 cachegrind 并以 gcc 4.8.1 与 -Os 编译的 x64 linux 上对一个经过良好定义的工作负载（SQLite 开发者用其作为典型应用工作负载的代理）进行测量时， 当前版本在相同 CPU 周期内完成的工作量比前一个版本多出 20% 以上。 Cachegrind 并不是真实的 CPU，并且用于测量的工作负载只是一个代理。因此你的性能可能会有所不同。 我们预期在真实应用中只会看到大约一半的测量和报告的提升。10% 比 20% 少，但我们认为仍然相当不错。

本次发布包含一套新的 C 语言接口，其长度参数是无符号 64 位而非有符号 32 位。 这些新 APIs 并未提供任何新功能，但确实使得编写对整数溢出漏洞更有抵抗力的应用程序更容易。

本次发布还包括一个新的排序器，该排序器能够使用多线程来协助大型排序操作。（排序操作有时用于实现 ORDER BY 和/或 GROUP BY 子句，几乎总是在 CREATE INDEX 时用到。） 多线程排序器默认关闭，必须使用 PRAGMA threads SQL 命令启用。 请注意，多线程排序器为大型排序提供更快的实时性能，但它也会使用更多的 CPU 周期和能量。

2014-08-15 - 发布 3.8.6

SQLite version 3.8.6 是一次定期安排的维护发布。 建议从之前所有版本升级。

本次发布包含常见的多种晦涩 bug 修复。 然而，有一个 bug 值得特别关注。 CREATE INDEX 命令从 version 3.8.2（2013-12-06）开始出现一个问题，在某些情况下允许在非唯一列上创建 UNIQUE 索引。 一旦索引创建完成，就无法插入新的非唯一条目，但已有的非唯一条目会保留。详见工单 9a6daf340df99ba93c。 除了修复该 bug 之外，PRAGMA integrity_check 命令已增强， 使其能够检测 UNIQUE 索引中的非唯一性，因此如果此 bug 实际在数据库中引入了问题，就可以轻松发现这些问题。

其他值得注意的更改包括添加对 hexadecimal integers（例如 0x1234）的支持， 以及对 IN operator 的性能增强，据 mailing list reports， 这些增强可让某些查询运行速度提高多达五倍。

根据 valgrind 及 test/speedtest1.c 测试程序， version 3.8.6 比大约一年前的 version 3.8.0 使用的 CPU 周期少 25%。 另一方面，3.8.6 的编译二进制比 3.8.0 大约大 5%。 大小增加部分是因为添加了诸如 WITHOUT ROWID 表和 common table expressions 之类的新特性。

2014-02-11 - 发布 3.8.3.1

SQLite version 3.8.3.1 修复了存在于 3.8.1、3.8.2 和 3.8.3 中的一个错误， 该错误可能导致查询遗漏有效的输出行。 建议从这些版本进行升级。

仅当 SQLite 使用了 SQLITE_ENABLE_STAT3 或 SQLITE_ENABLE_STAT4 编译选项时才会遇到该问题。 在这种情况下，如果查询的 WHERE 子句包含如下表达式：

2013-12-06 - 发布 3.8.2

SQLite version 3.8.2 是一次定期安排的维护发布。 升级自前一个版本是可选的。

version 3.8.2 添加了对 WITHOUT ROWID 表的支持。 这是对 SQLite 的一个重要扩展。包含 WITHOUT ROWID 表的数据库文件无法被先前版本的 SQLite 读取或写入，但不使用 WITHOUT ROWID 表的数据库在向前和向后兼容性上完全没问题。

3.8.2 发布包含一个潜在的不兼容改动。 在所有先前版本的 SQLite 中，将非常大的正浮点数强制转换为整数会得到最小的负整数。 换句话说，CAST(+99.9e99 to INT) 会产生 -9223372036854775808。 这种行为出现是因为 x86/x64 硬件在 C 语言中的等价转换就是这样。 但这种行为十分奇怪。因此从本次发布起已经将其更改为： 从浮点数转换到整数时，返回浮点值和零之间距离最近的整数。因此 CAST(+99.9e99 to INT) 现在返回 +9223372036854775807。 由于像 sqlite3_column_int64() 这样的例程在访问的值实际上是浮点数时会进行隐式转换，因此它们也受到该更改的影响。

除了上述两项更改之外，3.8.2 发布还包括多个性能增强。 skip-scan optimization 现在可用于已经通过 ANALYZE 处理的数据库。 常量 SQL 函数现在在内部循环之外被提取出来，对于包含类似 “date>datetime('now','-2 days')” 的 WHERE 子句的查询，这可以显著加速。 各种运行频繁的内部例程也被重构以减少 CPU 负载。

2013-10-17 - Release 3.8.1

SQLite version 3.8.1 是一次定期安排的维护发布。 升级上一版本是可选的，尽管如果你正在使用 partial indices， 应该升级，因为上一版本中与 partial indices 相关的 bug 可能会导致 count(*) 查询返回错误答案。

在前一个版本中首秀的 next generation query planner 继续表现良好。 当前版本对这个新的查询规划器做了小幅调整，以在某些情况下做出更优决策，但整体保持不变。 新增了两个 SQL 函数，likelihood() 和 unlikely()，帮助开发者向查询规划器提供提示，而不会强制规划器做出特定决策。

Version 3.8.1 是首个在选择查询计划时考虑表和索引行估算大小的 SQLite 版本。 行大小估算基于列的声明数据类型。例如，类型为 VARCHAR(1000) 的列被认为比类型为 INT 的列使用更多空间。 基于数据类型的行大小估算可以通过在表或索引的 sqlite_stat1.stat 记录末尾附加形如 “sz=NNN”（NNN 是平均行大小，单位为字节）的一项来覆盖。 当前行大小仅用于在执行表扫描或 count(*) 操作时帮助查询规划器在一个表或其某个索引之间做选择，不过未来版本很可能在其他场景中也会利用估算的行大小。 新的 PRAGMA stats 语句可以用来查看行大小估算。

Version 3.8.1 添加了 SQLITE_ENABLE_STAT4 编译时选项。 STAT4 与 STAT3 十分相似，都是使用索引的样本来猜测 WHERE 子句约束将满足索引中多少行。 区别在于 STAT4 会采样索引的所有列，而旧的 STAT3 仅采样最左边的列。 STAT3 的使用者被鼓励升级到 STAT4。 应用开发者在使用 STAT3 和 STAT4 时应谨慎，因为这两个选项按照设计会违反 query planner stability guarantee， 从而使在大规模部署并量产的嵌入式应用中维持一致性能更加困难。

2012-12-12 - Release 3.7.15

SQLite version 3.7.15 是一次定期安排的发布。 此版本包含对查询规划器与优化器的数项改进以及一个重要的 bug 修复。 这是首个正式支持 Windows 8 Phone 的发行版。

该重要 bug 修复针对的是在使用 shared cache mode 且 schema 中的 COLLATE operator 位于 CHECK constraint 或 view 中时可能导致段错误的问题。 排序函数与特定数据库连接相关。但一个指向排序函数的指针也被缓存在表达式中。 如果某表达式属于 schema 并包含已缓存的排序函数，它会指向最初解析 schema 的数据库连接中的排序函数。 如果该数据库连接关闭，而使用相同 shared cache 的其它连接继续运行，它们就会尝试使用已经释放的排序函数。 3.7.15 中的修复是不在表达式结构中缓存排序函数指针，而是在每次准备新语句时重新查找它们。

此发布还包含对查询规划器的重要增强，应当（我们希望）让某些查询运行得更快。 增强内容包括：

  1. 在执行全表扫描时，尝试使用索引替代原始表，基于这样一种想法：索引包含的信息更少，因此体积更小，扫描时所需的磁盘 I/O 也更少。

  2. 增强 IN operator，使其能够使用具有 numeric affinity 的索引。

  3. 更好地识别何时可以借助索引来实现 ORDER BY 子句——尤其在 ORDER BY 子句包含来自连接中两个或更多表的项的情况下。

2011-05-19 - Version 3.7.6.3

SQLite version 3.7.6.3 是一个补丁发行版，用于修复与 WAL mode 相关的 single bug。这个错误自 WAL 引入以来就存在，但问题非常隐蔽，因此之前没人注意到。不过，我们仍建议所有用户升级到 version 3.7.6.3 或更高版本。

该错误如下：如果 cache_size 设置得非常小（小于 10）且 SQLite 遭遇内存压力，同时又开始了一个多语句事务，其中在 COMMIT 之前的最后一条语句是 SELECT，而在事务提交之后紧接着发生了 checkpoint，那么事务可能会在未提交的情况下悄悄回滚。

cache_size 的默认设置是 2000。因此在大多数情况下，这个错误永远不会出现。但有时程序员会在内存非常有限的设备上将 cache_size 设置得非常小以节省内存空间。这类应用程序就会受到影响。请注意，此错误并不会导致数据库损坏。某些情况下，它就像执行了 ROLLBACK 而不是 COMMIT。

Bug Details

WAL 模式下的事务通过在 WAL（预写日志）末尾添加一个包含“commit”标记的记录来提交。因此，为了提交事务，SQLite 会将事务期间发生变化的所有页面追加到 WAL，并在最后一页上设置 commit 标记。现在，如果 SQLite 遭遇内存压力，它可能会在提交前通过将更改页面写入 WAL 来释放内存空间。我们称之为“将缓存溢出到 WAL”。将缓存溢出到 WAL 本身并无不妥。但如果内存压力非常严重，那么在执行 COMMIT 时，所有被修改的页面可能早已写入 WAL，而没有剩下任何页面可供写入。没有未写入的页面，就没有地方设置 commit 标记。而没有 commit 标记，事务最终就会被回滚。

为该问题所做的修复是：如果在提交开始时所有已修改页面已经被写入 WAL，那么数据库的第 1 页将再次写入 WAL，以确保总有一个页面可用来设置 commit 标记。

计划仍是通过修复 bug 来继续支持 SQLite 2.8。 但所有新的开发都将在 3.0 中进行。
