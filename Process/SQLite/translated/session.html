变更集的重基

typedef struct sqlite3_rebaser sqlite3_rebaser;

重要提示：此接口是 experimental 的，可能会在无通知的情况下发生变化。

假设有一个站点托管的数据库处于状态 S0。随后对该数据库进行修改，使其变为 S1，并记录一个变更集（“本地”变更集）。 然后，从另一个站点接收到一个基于 S0 的变更集（“远程”变更集）并应用到数据库。 数据库最终处于状态 (S1+"remote")，其确切状态取决于在应用“remote”时所作的冲突解决决策（OMIT 或 REPLACE）。 重基变更集的目的在于将其更新为充分考虑这些冲突解决决策，以便网络中其他位置不必再次解决相同冲突。

例如，如果本地与远程的变更集中都包含针对 “CREATE TABLE t1(a PRIMARY KEY, b)” 的相同键的 INSERT：

local: INSERT INTO t1 VALUES(1, 'v1'); remote: INSERT INTO t1 VALUES(1, 'v2');

若冲突解决为 REPLACE，则 INSERT 更改会从本地变更集中移除（被覆盖）。若冲突解决为“OMIT”， 则本地变更集会改为包含：

UPDATE t1 SET b = 'v2' WHERE a=1;

本地变更集中的更改按如下方式进行重基：

Local INSERT
这可能仅与远程 INSERT 冲突。若冲突解决为 OMIT，则在重基后的变更集中添加一个 UPDATE 更改。 若冲突解决为 REPLACE，则在重基后的变更集中不添加任何内容。

Local DELETE
这可能与远程 UPDATE 或 DELETE 冲突。在这两种情况下，唯一可能的解决是 OMIT。 若远程操作为 DELETE，则重基后的变更集中不添加任何更改。 若远程操作为 UPDATE，则将更改的 old.* 字段更新为 UPDATE 中的 new.* 值。

Local UPDATE
这可能与远程 UPDATE 或 DELETE 冲突。如与 DELETE 冲突且冲突解决为 OMIT，则该 UPDATE 更改改为 INSERT。 更新更改中 new.* 记录的任何未定义值会使用冲突 DELETE 的 old.* 值填充。 若冲突解决为 REPLACE，则直接从重基后的变更集中省略该 UPDATE 更改。

若冲突为远程 UPDATE 且解决为 OMIT，则使用远程更改中的 new.* 值对 old.* 进行重基。 若解决为 REPLACE，则将此更改复制到重基后的变更集中，同时删除与冲突远程 UPDATE 也更新的列。 若因此 no columns would be updated，该更改被省略。

本地更改可同时针对多个远程更改进行重基。 若同一键被多个远程变更集修改，则在重基前将它们按如下方式组合：

  • 若某键上有一次或多次 REPLACE 解决，则按 REPLACE 重基。

  • 若某键上没有 REPLACE 解决，则按最近一次 OMIT 解决重基。

请注意，多远程变更集的冲突解决会逐字段合并，而不是逐行合并。 这意味着在多个远程 UPDATE 的情况下，单个本地更改的某些字段可能为 REPLACE 重基，而其他字段为 OMIT 重基。

为对本地变更集进行重基，必须先使用 sqlite3changeset_apply_v2() 将远程变更集应用到本地数据库，并捕获重基信息缓冲区。之后：

  1. 通过调用 sqlite3rebaser_create() 创建 sqlite3_rebaser 对象。
  2. 通过调用 sqlite3rebaser_configure() 使用从 sqlite3changeset_apply_v2() 获得的重基缓冲区配置新对象。 若要针对多个远程变更集重基本地变更集，则应按执行多个 sqlite3changeset_apply_v2() 调用的相同顺序多次调用 sqlite3rebaser_configure()。
  3. 通过调用 sqlite3rebaser_rebase() 对每个本地变更集进行重基。
  4. 调用 sqlite3rebaser_delete() 删除 sqlite3_rebaser 对象。

向 Changegroup 添加变更集

int sqlite3changegroup_add(sqlite3_changegroup*, int nData, void *pData);

将缓冲区 pData（大小为 nData 字节）中的所有变更（或补丁集）添加到 changegroup。

若缓冲区包含补丁集，则对同一 changegroup 对象的所有先前调用也必须指定补丁集。 或者，若缓冲区包含变更集，则先前的调用也必须是变更集。否则将返回 SQLITE_ERROR，并且不会将任何更改添加到 changegroup。

变更集与 changegroup 中的行通过主键列的值来识别。 若两行具有相同的主键，则认为它们涉及同一行。

尚未出现在 changegroup 中的行的更改将直接复制到其中。 或者，若新变更集与 changegroup 都包含应用于同一行的更改，则 changegroup 的最终内容取决于各个更改类型，如下：

现有更改 新更改 输出更改
INSERT INSERT 忽略新更改。若新变更集是紧随已添加变更集录制的，则不会出现此情况。
INSERT UPDATE INSERT 更改保留在 changegroup 中。INSERT 的值会被修改，就好像先有该 INSERT，然后再应用新的 UPDATE。
INSERT DELETE 移除现有的 INSERT，更改 group 中不添加 DELETE。
UPDATE INSERT 忽略新更改。若新变更集是紧随已添加变更集录制的，则不会出现此情况。
UPDATE UPDATE 现有 UPDATE 保留在 changegroup 中，并进行修订，使其伴随的值等同于先应用现有更改再应用新更改的结果。
UPDATE DELETE changegroup 中用新的 DELETE 替换现有 UPDATE。
DELETE INSERT 若新更改插入的行的任一列值与现有 DELETE 删除的行不同，则用 UPDATE 替换现有 DELETE。 若插入的行与删除的行完全一致，则直接丢弃现有 DELETE。
DELETE UPDATE 忽略新更改。若新变更集是紧随已添加变更集录制的，则不会出现此情况。
DELETE DELETE 忽略新更改。若新变更集是紧随已添加变更集录制的，则不会出现此情况。

若新变更集包含对已在 changegroup 中的表的更改，则该表的列数与主键位置必须一致。 若不一致，则此函数返回 SQLITE_SCHEMA，变更操作失败。 但是，若 changegroup 对象已通过 sqlite3changegroup_schema() API 配置了数据库模式， 则可以将列数不同但其余兼容的变更集合并在同一表中。

若输入变更集似乎已损坏且检测到该损坏，则返回 SQLITE_CORRUPT。 若在处理期间出现内存不足，则返回 SQLITE_NOMEM。

无论何种情况，若发生错误，changegroup 最终内容的状态未定义。 若无错误发生，则返回 SQLITE_OK。

向 Changegroup 添加单个更改

int sqlite3changegroup_add_change(
  sqlite3_changegroup*,

该函数将第二个参数所指示的迭代器当前位置的单个更改添加到 changegroup 对象中。 添加规则与 sqlite3changegroup_add() 所述相同。

创建新的 Changegroup 对象

int sqlite3changegroup_new(sqlite3_changegroup **pp);

sqlite3_changegroup 对象用于将两个或多个变更集（或补丁集）合并为一个变更集（或补丁集）。 单个 changegroup 对象可以合并变更集或补丁集，但不能同时包含两者。输出格式始终与输入一致。

若成功，此函数返回 SQLITE_OK，并在返回前将 (*pp) 设置为指向新的 sqlite3_changegroup 对象的指针。 调用者应通过 sqlite3changegroup_delete() 释放该对象。若发生错误，则返回 SQLite 错误代码（例如 SQLITE_NOMEM），并将 *pp 设为 NULL。

sqlite3_changegroup 对象的常见使用模式如下：

  • 通过调用 sqlite3changegroup_new() 创建对象。

  • 通过 sqlite3changegroup_add() 将零个或多个变更集（或补丁集）添加到对象。

  • 通过 sqlite3changegroup_output() 获取将所有输入变更集合并后的结果。

  • 通过 sqlite3changegroup_delete() 删除该对象。

在 new() 与 delete() 调用之间，可按任意顺序多次调用 add() 与 output()。

除了常规的 sqlite3changegroup_add() 与 sqlite3changegroup_output() 函数外， 还提供了流式版本 sqlite3changegroup_add_strm() 与 sqlite3changegroup_output_strm()。

从 Changegroup 获取复合变更集

int sqlite3changegroup_output(
  sqlite3_changegroup*,
  int *pnData,                    /* OUT: Size of output buffer in bytes */
  void **ppData                   /* OUT: Pointer to output buffer */
);

获取一个包含表示 changegroup 当前内容的变更集（或补丁集）的缓冲区。 若 changegroup 的输入本身是变更集，则输出为变更集。若输入为补丁集，则输出同样为补丁集。

与 sqlite3session_changeset() 与 sqlite3session_patchset() 输出类似， 同一表的所有更改在此函数输出中会分组在一起。 表的顺序与首次添加到 changegroup 的变更集中的顺序相同。 若后续添加的变更集中包含未在第一变更集中出现的表，则这些表按首次出现的顺序附加到输出变更集末尾。

若发生错误，则返回 SQLite 错误代码，且输出变量 (*pnData) 与 (*ppData) 均设为 0。 否则返回 SQLITE_OK，并将输出变量设置为缓冲区大小与指针。 在此情况下，调用者有责任最终调用 sqlite3_free() 释放缓冲区。

为 Changegroup 添加模式

int sqlite3changegroup_schema(sqlite3_changegroup*, sqlite3*, const char *zDb);

此方法可选地强制要求添加到 changegroup 的变更集必须与数据库 zDb（“main”、“temp”或附加数据库的名称）的模式匹配。 若调用 sqlite3changegroup_add() 添加的变更集与配置的模式不兼容，则返回 SQLITE_SCHEMA，且 changegroup 对象将处于未定义状态。

变更集的模式是否与数据库模式兼容，判定方式与 sqlite3changeset_apply() 相同。 具体而言，对于每个变更集中的表，数据库中必须存在一个表，具有：

  • 变更集所标识的名称，且
  • 至少与变更集中记录的列数相同，且
  • 主键列的位置与变更集中记录的位置一致。

changegroup 对象的输出始终具有由此函数指定的数据库模式。 当传入 sqlite3changegroup_add() 的变更集列数少于数据库模式中对应表的列数时，这些列将使用数据库模式中的默认列值填充。 这使得在 changegroup 中可以组合列数不同但其他方面兼容的同一表的变更集。

连接两个 Changeset 对象

int sqlite3changeset_concat(
  int nA,                         /* Number of bytes in buffer pA */
  void *pA,                       /* Pointer to buffer containing changeset A */
  int nB,                         /* Number of bytes in buffer pB */
  void *pB,                       /* Pointer to buffer containing changeset B */
  int *pnOut,                     /* OUT: Number of bytes in output changeset */
  void **ppOut                    /* OUT: Buffer containing output changeset */
);

此函数用于将两个 changeset（A 和 B）连接成一个单独的 changeset。其结果等价于先应用 changeset A，再应用 changeset B。

此函数使用一个 sqlite3_changegroup 对象组合两个输入 changeset。调用它将产生与下面代码片段类似的结果：

  sqlite3_changegroup *pGrp;
  rc = sqlite3_changegroup_new(&pGrp);
  if( rc==SQLITE_OK ) rc = sqlite3changegroup_add(pGrp, nA, pA);
  if( rc==SQLITE_OK ) rc = sqlite3changegroup_add(pGrp, nB, pB);
  if( rc==SQLITE_OK ){
    rc = sqlite3changegroup_output(pGrp, pnOut, ppOut);
  }else{
    *ppOut = 0;
    *pnOut = 0;
  }

有关详细信息，请参阅下方的 sqlite3_changegroup 文档。

从 Changeset 迭代器获取冲突行的值

int sqlite3changeset_conflict(
  sqlite3_changeset_iter *pIter,  /* Changeset iterator */
  int iVal,                       /* Column number */
  sqlite3_value **ppValue         /* OUT: Value from conflicting row */
);

此函数仅应与通过 sqlite3changeset_apply() 传递给冲突处理回调的迭代器对象一起使用，且该调用使用了 SQLITE_CHANGESET_DATA 或 SQLITE_CHANGESET_CONFLICT。如果在任何其他迭代器上调用此函数，将返回 SQLITE_MISUSE，并将 *ppValue 设置为 NULL。

参数 iVal 必须大于等于 0，且小于当前更改所影响表的列数。否则，将返回 SQLITE_RANGE 并将 *ppValue 设置为 NULL。

如果成功，此函数将 *ppValue 设置为指向一个受保护的 sqlite3_value 对象，该对象包含与当前冲突处理回调相关联的“冲突行”的第 iVal 个值，并返回 SQLITE_OK。

如果发生其他错误（例如 OOM），则返回一个 SQLite 错误码，并将 *ppValue 设置为 NULL。

终结 Changeset 迭代器

int sqlite3changeset_finalize(sqlite3_changeset_iter *pIter);

此函数用于终结通过 sqlite3changeset_start() 分配的迭代器。

此函数仅应在使用 sqlite3changeset_start() 创建的迭代器上调用。如果应用程序对通过 sqlite3changeset_apply() 传递给冲突处理程序的迭代器调用此函数，将立即返回 SQLITE_MISUSE，并且调用不会产生任何效果。

如果在调用某个 sqlite3changeset_xxx() 函数时遇到错误（例如在 sqlite3changeset_next() 中发生 SQLITE_CORRUPT，或在 sqlite3changeset_new() 中发生 SQLITE_NOMEM），那么此函数将返回与该错误对应的错误码。否则，将返回 SQLITE_OK。这允许使用如下模式（伪代码）：

  sqlite3changeset_start();
  while( SQLITE_ROW==sqlite3changeset_next() ){
    // Do something with change.
  }
  rc = sqlite3changeset_finalize();
  if( rc!=SQLITE_OK ){
    // An error has occurred 
  }

确定外键约束违规的数量

int sqlite3changeset_fk_conflicts(

反转 Changeset

int sqlite3changeset_invert(
  int nIn, const void *pIn,       /* Input changeset */
  int *pnOut, void **ppOut        /* OUT: Inverse of input */
);

此函数用于“反转”一个 changeset 对象。将反转后的 changeset 应用于数据库会撤销应用原始 changeset 的效果。具体如下：

  • 每个 DELETE 更改变为 INSERT，
  • 每个 INSERT 更改变为 DELETE，
  • 对于每个 UPDATE 更改，old.* 和 new.* 值互换。

此函数不会改变 changeset 中更改出现的顺序。它只是逆转每个独立更改的方向。

如果成功，会将指向包含反转 changeset 的缓冲区的指针存储在 *ppOut 中，将该缓冲区的大小存储在 *pnOut 中，并返回 SQLITE_OK。如果发生错误，则会将 *pnOut 和 *ppOut 清零，并返回一个 SQLite 错误码。

调用者有责任在成功调用此函数后最终对 *ppOut 指针调用 sqlite3_free() 以释放缓冲区分配。

警告/待办：此函数当前假设输入是一个有效的 changeset。如果不是，结果未定义。

从 Changeset 迭代器获取 new.* 值

int sqlite3changeset_new(
  sqlite3_changeset_iter *pIter,  /* Changeset iterator */
  int iVal,                       /* Column number */
  sqlite3_value **ppValue         /* OUT: New value (or NULL pointer) */
);

传递给此函数的 pIter 参数可以是通过 sqlite3changeset_apply() 传递给冲突处理程序的迭代器，也可以是通过 sqlite3changeset_start() 创建的迭代器。在后一种情况下，最近一次对 sqlite3changeset_next() 的调用必须返回 SQLITE_ROW。此外，仅当迭代器当前指向的更改类型是 SQLITE_UPDATE 或 SQLITE_INSERT 时才可调用此函数。否则，该函数返回 SQLITE_MISUSE 并将 *ppValue 设置为 NULL。

推进 Changeset 迭代器

此函数仅可用于由 sqlite3changeset_start() 创建的迭代器。如果在由 sqlite3changeset_apply() 传递给冲突处理回调的迭代器上调用，将返回 SQLITE_MISUSE，并且调用不会产生任何效果。

如果发生错误，将返回一个 SQLite 错误码。可能的错误码包括 SQLITE_CORRUPT（如果 changeset 缓冲区已损坏）或 SQLITE_NOMEM。

sqlite_stat1 的特殊处理

从 SQLite 3.22.0 版本开始，“sqlite_stat1” 表是上述某些规则的例外。在 SQLite 中，sqlite_stat1 的模式为： 

       CREATE TABLE sqlite_stat1(tbl,idx,stat)

尽管 sqlite_stat1 没有 PRIMARY KEY，但其更改仍按主键为 (tbl,idx) 处理。此外，对于满足 (idx IS NULL) 的行也会记录更改。不过，对于此类行，在 changeset 或 patchset 中存储的不是 NULL，而是零长度 blob（SQL 值 X''）。这使得这些 changeset 可被 sqlite3changeset_invert()、concat() 及类似旧实现操作。

sqlite3changeset_apply() 函数在更新 sqlite_stat1 表时会自动将零长度 blob 转换回 NULL。然而，如果应用程序直接在 changeset 迭代器上调用 sqlite3changeset_new()、sqlite3changeset_old() 或 sqlite3changeset_conflict（包括在传递给冲突处理回调的迭代器上），则会返回 X'' 值。应用程序需要在必要时自行将 X'' 转换为 NULL。

旧版（早于 3.22.0）会话模块无法捕获对 sqlite_stat1 表所做的更改。旧版 sqlite3changeset_apply() 函数会默默忽略 changeset 或 patchset 中对 sqlite_stat1 表的任何修改。

从会话对象生成 Changeset

int sqlite3session_changeset(
  sqlite3_session *pSession,      /* Session object */
  int *pnChangeset,               /* OUT: Size of buffer at *ppChangeset */
  void **ppChangeset              /* OUT: Buffer containing changeset */
);

获取一个包含附加到第一个参数所指会话对象的表更改的 changeset。如果成功，将 *ppChangeset 设置为指向包含 changeset 的缓冲区，并将 *pnChangeset 设置为 changeset 的字节大小，然后返回 SQLITE_OK。如果发生错误，则将 *ppChangeset 和 *pnChangeset 都设置为零，并返回一个 SQLite 错误码。

一个 changeset 由零个或多个 INSERT、UPDATE 和/或 DELETE 更改组成，每个更改代表附加表中单行的修改。INSERT 更改包含新数据库行的各字段值。DELETE 更改包含被删除数据库行的原始字段值。UPDATE 更改包含更新前数据库行各字段的原始值，以及每个被更新的非主键列的新的值。UPDATE 更改无法表示修改主键列值的操作。如果发生这种更改，它将在 changeset 中表示为先 DELETE 再 INSERT。

对于在一个或多个 PRIMARY KEY 列中存储了 NULL 值的行，不会记录其更改。如果插入或删除了这样的行，则本函数返回的 changeset 中不会包含对应的更改。如果现有行在一个或多个 PRIMARY KEY 列中存储了 NULL 值，随后被更新为所有 PRIMARY KEY 列都非 NULL，则 changeset 中只会出现一个 INSERT。类似地，如果现有行的非 NULL PRIMARY KEY 值被更新为使其中一个或多个 PRIMARY KEY 列变为 NULL，则结果 changeset 中只包含 DELETE 更改。

可以使用 sqlite3changeset_start() API 创建的迭代器遍历本函数生成的 changeset。也可以使用 sqlite3changeset_apply() API 将 changeset 应用于具有兼容模式的数据库。

在本函数生成的 changeset 中，与同一表相关的所有更改会被分组。换句话说，当遍历 changeset 或将其应用于数据库时，会先处理与单个表相关的所有更改，然后再转到下一个表。表按照附加（或自动附加）到 sqlite3_session 对象的顺序排序。与同一表相关的更改在 changeset 中的存储顺序未定义。

在成功调用本函数后，调用者有责任最终使用 sqlite3_free() 释放 *ppChangeset 指向的缓冲区。

Changeset 的生成

一旦将表附加到会话对象，会话对象会记录插入的新行的主键值。它还会记录删除或更新的行的原始主键和其他列值。对于每个唯一主键值，只会在该行在会话期间首次被插入、更新或删除时记录一次数据。

前段所述的内容有一个例外：当某行被插入、更新或删除时，如果其一个或多个主键列包含 NULL 值，则不会记录该更改。

因此，会话对象会累计两类记录——仅包含主键值的记录（在用户插入新记录时创建）以及既包含主键值又包含其他列原始值的记录（在用户删除或更新记录时创建）。

调用本函数时，请求的 changeset 是使用累计记录和当前数据库文件内容一起创建的。具体如下：

  • 对于每条由插入生成的记录，会查询数据库中是否存在具有匹配主键的行。如果找到，则向 changeset 中添加一个 INSERT 更改。如果未找到，则不添加更改。

  • 对于每条由更新或删除生成的记录，会查询数据库中是否存在具有匹配主键的行。如果找到，并且一个或多个非主键字段的值与原始值不同，则向 changeset 中添加一个 UPDATE 更改。或者，如果表中未找到该主键对应的行，则添加一个 DELETE 更改。如果数据库中存在匹配主键的行，但所有字段都保留原始值，则不添加更改。

这意味着，例如，如果在会话对象启用期间插入了一行，随后在同一会话期间又删除了该行，则 changeset 中不会出现插入和删除这两个更改。再比如，如果在会话对象启用期间删除了一行，随后又插入了一行具有相同主键值的行，则最终的 changeset 将包含一个 UPDATE 更改，而不是 DELETE 和 INSERT。

当会话对象被禁用（参见 sqlite3session_enable() API）时，在插入、更新或删除行时不会累积记录。如果在一个会话期间，多次写入相同行，则可能出现一些反直觉的效果。例如，如果在启用会话对象期间插入一行，之后在相同会话对象禁用期间又删除该行，则 changeset 中不会出现 INSERT 记录，即使删除发生在会话禁用期间。亦或者，如果在会话启用期间更新了一行的某个字段，然后在禁用期间更新了同一行的另一个字段，则最终 changeset 中的 UPDATE 更改会同时更新这两个字段。

将两个表的差异加载到 Session 中

int sqlite3session_diff(
  sqlite3_session *pSession,
  const char *zFromDb,
  const char *zTbl,
  char **pzErrMsg
);

如果 zTbl 表尚未附加到作为第一个参数传入的 session 对象，该函数将以与 sqlite3session_attach() 相同的方式附加该表。如果 zTbl 不存在或没有主键，该函数是 no-op（但不会返回错误）。

参数 zFromDb 必须是附加到与包含与通过本函数附加的表兼容的表的 session 对象相同数据库句柄的数据库名称（如 "main"、"temp" 等）。若要兼容，表必须满足以下条件：

  • 名称相同，
  • 在相同顺序中声明的列集合相同，并且
  • 主键定义相同。

如果表不兼容，则返回 SQLITE_SCHEMA。如果表兼容但没有任何主键列，则不算错误，但不会向 session 对象添加任何更改。与其他 session API 一样，缺少主键的表会被忽略。

此函数向 session 对象添加一组更改，这些更改可用于更新 zFrom 数据库中的表（称为 "源表"），使其内容与附加到 session 对象的表（称为 "目标表"）相同。具体而言：

  • 对于存在于目标表中但不存在于源表中的每一行（主键），向 session 对象添加一条 INSERT 记录。
  • 对于存在于源表中但不存在于目标表中的每一行（主键），向 session 对象添加一条 DELETE 记录。
  • 对于在两个表中都存在但非主键字段值不同的每一行（主键），向 session 对象添加一条 UPDATE 记录。

为澄清起见，如果调用此函数后再使用 sqlite3session_changeset() 构造变化集，然后将该变化集应用于 zFrom 数据库，则两个兼容表的内容将一致。

除非此函数如上文所述是 no-op，否则如果 zFrom 数据库不存在或不包含所需的兼容表，则会发生错误。

如果操作成功，则返回 SQLITE_OK。否则返回 SQLite 错误代码。在此情况下，如果 pzErrMsg 不为 NULL，则 *pzErrMsg 可能设置为指向包含英文错误消息的缓冲区。调用者有责任使用 sqlite3_free() 释放该缓冲区。

启用或禁用 Session 对象

int sqlite3session_enable(sqlite3_session *pSession, int bEnable);

启用或禁用 session 对象记录更改。启用时，session 对象会记录对数据库的更改。禁用时则不会。新创建的 session 对象是启用的。有关启用和禁用 session 对象如何影响最终变更集的更多细节，请参阅 sqlite3session_changeset() 的文档。

传入 0 会禁用 session。传入大于 0 的值会启用 session。传入小于 0 的值则是 no-op，可用于查询当前 session 的状态。

返回值表示 session 对象的最终状态：若禁用返回 0，启用返回 1。

设置或清除间接更改标志

int sqlite3session_indirect(sqlite3_session *pSession, int bIndirect);

session 对象记录的每个更改都标记为直接或间接。如果满足以下任一条件，则将更改标记为间接：

  • 更改发生时 session 对象的“间接”标志已设置，或
  • 更改是由 SQL 触发器或外键操作所致，而不是用户 SQL 语句直接引起。

如果单行在一个 session 中受到多个操作影响，则仅当所有操作都满足上述间接更改条件时，才将其视为间接；否则视为直接。

此函数用于设置、清除或查询 session 对象的间接标志。如果传入的第二个参数为 0，则清除间接标志。如果大于 0，则设置间接标志。传入小于 0 的值不会修改当前间接标志，可用于查询指定 session 对象间接标志的当前状态。

返回值表示间接标志的最终状态：如果清除则返回 0，设置则返回 1。

测试变更集中是否记录了任何更改。

int sqlite3session_isempty(sqlite3_session *pSession);

如果传入的 session 对象未记录任何附加表的更改，则返回非零。否则，如果已记录一个或多个更改，则返回零。

即使此函数返回零，调用 sqlite3session_changeset() 仍可能返回包含零更改的变更集。这种情况发生在附加表的某一行被修改后又恢复为原始值。但是，如果此函数返回非零，则可确保调用 sqlite3session_changeset() 会返回包含零更改的变更集。

查询 session 对象使用的堆内存量。

sqlite3_int64 sqlite3session_memory_used(sqlite3_session *pSession);

此 API 返回传入的 session 对象当前使用的堆内存总字节数。

配置 Session 对象

int sqlite3session_object_config(sqlite3_session*, int op, void *pArg);

此方法用于在创建 Session 对象之后对其进行配置。目前，第二个参数的唯一有效值是 SQLITE_SESSION_OBJCONFIG_SIZE 和 SQLITE_SESSION_OBJCONFIG_ROWID。

补丁集与变更集之间的差异在于：

  • DELETE 记录仅由主键字段组成。其他字段的原始值被省略。
  • UPDATE 记录省略了任何被修改字段的原始值。

补丁集 blob 可与所有最新版本的 sqlite3changeset_xxx API 函数一起使用，除了 sqlite3changeset_invert()，如果传入补丁集则返回 SQLITE_CORRUPT。同样，尝试将补丁集 blob 与旧版本的 sqlite3changeset_xxx API 一起使用也会导致 SQLITE_CORRUPT 错误。

由于省略了非主键的 “old.*” 字段，因此如果将补丁集传递给 sqlite3changeset_apply() API，就无法检测或报告 SQLITE_CHANGESET_DATA 冲突。其他冲突类型的行为与变更集相同。

补丁集中的更改与 sqlite3session_changeset() 函数生成的变更集的顺序相同（即：同一表的所有更改分组在一起，表按附加到 session 对象的顺序出现）。

sqlite3changeset_apply_v2 的标志

#define SQLITE_CHANGESETAPPLY_NOSAVEPOINT   0x0001
#define SQLITE_CHANGESETAPPLY_INVERT        0x0002
#define SQLITE_CHANGESETAPPLY_IGNORENOOP    0x0004
#define SQLITE_CHANGESETAPPLY_FKNOACTION    0x0008

下列标志可以通过第九个参数传递给 sqlite3changeset_apply_v2 和 sqlite3changeset_apply_v2_strm：

SQLITE_CHANGESETAPPLY_NOSAVEPOINT
通常，session 模块会将单次对 apply_v2() 或 apply_v2_strm() 的调用所执行的所有操作封装在一个 SAVEPOINT 中。如果变更集或补丁集成功应用，则提交 SAVEPOINT；如果出错则回滚。指定此标志会导致 session 模块省略该保存点。在这种情况下，如果调用 apply_v2() 时调用者已开启事务或保存点，它可以通过回滚来恢复部分应用的变更集。

SQLITE_CHANGESETAPPLY_INVERT
在应用变更集之前先进行反转。这等同于在应用之前使用 sqlite3changeset_invert() 反转变更集。对于补丁集，指定此标志会导致错误。

SQLITE_CHANGESETAPPLY_IGNORENOOP
不对不会实际修改数据库的变更调用冲突处理程序回调。具体来说，这意味着冲突处理程序不会因以下情况而被调用：
  • 若无法找到待删除的行，则不会调用 DELETE 更改，
  • 若被更新的字段已在冲突行中设置为新值，则不会调用 UPDATE 更改，
  • 若冲突行的所有字段与要插入的行完全匹配，则不会调用 INSERT 更改。

SQLITE_CHANGESETAPPLY_FKNOACTION
如果设置了此标志，则目标数据库中的所有外键约束将行为上等同于使用 “ON UPDATE NO ACTION ON DELETE NO ACTION” 声明，即便其实际为 CASCADE、RESTRICT、SET NULL 或 SET DEFAULT。

冲突处理程序返回的常量

#define SQLITE_CHANGESET_OMIT       0
#define SQLITE_CHANGESET_REPLACE    1
#define SQLITE_CHANGESET_ABORT      2

冲突处理程序回调必须返回以下三者之一。

SQLITE_CHANGESET_OMIT
若冲突处理程序返回此值，不执行任何特殊操作。导致冲突的更改不会应用。session 模块继续处理变更集中的下一个更改。

SQLITE_CHANGESET_REPLACE
仅当传递给冲突处理程序的第二个参数是 SQLITE_CHANGESET_DATA 或 SQLITE_CHANGESET_CONFLICT 时，才可返回此值。如果不是，将回滚已经应用的更改，并且对 sqlite3changeset_apply() 的调用返回 SQLITE_MISUSE。

如果 SQLITE_CHANGESET_DATA 冲突处理程序返回 CHANGESET_REPLACE，则根据更改类型更新或删除冲突行。

如果 SQLITE_CHANGESET_CONFLICT 冲突处理程序返回 CHANGESET_REPLACE，则将冲突行从数据库中移除，并再次尝试应用更改。如果第二次尝试失败，则在继续之前将原始行恢复到数据库中。

SQLITE_CHANGESET_ABORT
如果返回此值，则回滚迄今为止应用的所有更改，并且对 sqlite3changeset_apply() 的调用返回 SQLITE_ABORT。

传递给冲突处理程序的常量

#define SQLITE_CHANGESET_DATA        1
#define SQLITE_CHANGESET_NOTFOUND    2
#define SQLITE_CHANGESET_CONFLICT    3
#define SQLITE_CHANGESET_CONSTRAINT  4
#define SQLITE_CHANGESET_FOREIGN_KEY 5

可能作为冲突处理程序第二个参数传递的值。

SQLITE_CHANGESET_DATA
在处理 DELETE 或 UPDATE 更改时，如果数据库中存在具有所需 PRIMARY KEY 字段的行，但由更新修改的一个或多个其他（非主键）字段的“之前”值不符合预期，则冲突处理程序会以 CHANGESET_DATA 作为第二个参数被调用。

在这种情况下，冲突行是具有匹配主键的数据库行。

SQLITE_CHANGESET_NOTFOUND
在处理 DELETE 或 UPDATE 更改时，如果数据库中找不到具有所需 PRIMARY KEY 字段的行，则冲突处理程序以 CHANGESET_NOTFOUND 作为第二个参数被调用。

此时不存在冲突行。调用 sqlite3changeset_conflict() API 的结果是未定义的。

SQLITE_CHANGESET_CONFLICT
在处理 INSERT 更改时，如果操作将导致重复的主键值，则冲突处理程序会以 CHANGESET_CONFLICT 作为第二个参数被调用。

此时的冲突行为是具有匹配主键的数据库行。

SQLITE_CHANGESET_FOREIGN_KEY
如果启用了外键处理，并且应用变更集后数据库中存在外键违规，则在变更集提交之前，冲突处理程序将被调用一次，第二个参数为 CHANGESET_FOREIGN_KEY。如果处理程序返回 CHANGESET_OMIT，则包括导致外键违规的更改在内的所有更改都会被提交。若返回 CHANGESET_ABORT，则变更集会被回滚。

不提供当前或冲突行信息。在提供的 sqlite3_changeset_iter 句柄上，仅可调用 sqlite3changeset_fk_conflicts()。

SQLITE_CHANGESET_CONSTRAINT
在应用更改期间发生其他任何约束违规（如 UNIQUE、CHECK 或 NOT NULL）时，冲突处理程序会以 CHANGESET_CONSTRAINT 作为第二个参数被调用。

此时不提供冲突行。调用 sqlite3changeset_conflict() API 的结果是未定义的。

sqlite3session_object_config 的选项

#define SQLITE_SESSION_OBJCONFIG_SIZE  1
#define SQLITE_SESSION_OBJCONFIG_ROWID 2

以下值可以作为 sqlite3session_object_config() 的第二个参数传递。

SQLITE_SESSION_OBJCONFIG_SIZE
此选项用于设置、清除或查询启用 sqlite3session_changeset_size() API 的标志。由于它会带来一定的计算开销，默认情况下此 API 是禁用的。pArg 必须指向 int 类型的值。如果初始值为 0，则禁用 sqlite3session_changeset_size() API。如果大于 0，则启用。同样，如果初始值小于 0，则不做更改。在所有情况下，如果当前调用之后该 API 已启用，则 int 变量设置为 1；否则设置为 0。

在向 session 对象附加第一张表之后尝试修改此设置将导致错误（SQLITE_MISUSE）。

SQLITE_SESSION_OBJCONFIG_ROWID
此选项用于设置、清除或查询启用为无明确 PRIMARY KEY 的表收集数据的标志。

通常，session 模块会忽略没有显式 PRIMARY KEY 的表。然而，如果设置了此标志，则它的行为就像在此类表的最左侧插入一列 "_rowid_ INTEGER PRIMARY KEY"。

在向 session 对象附加第一张表之后尝试修改此设置将导致错误（SQLITE_MISUSE）。

接受变更集（或补丁集）作为输入的非流式函数要求整个变更集存储在内存中的单个缓冲区中。类似地，那些返回变更集或补丁集的函数也通过返回 sqlite3_malloc() 分配的单个大缓冲区的指针来实现。通常这很方便。但是，如果应用程序在低内存环境中需要处理非常大的变更集，那么所需的大块连续内存分配就会变得异常困难。

为避免此类问题，流式 API 将输入通过回调函数逐步提供，session 模块在需要时调用该回调函数以递增请求输入数据。所有情况下，类似

       int nChangeset,
       void *pChangeset,

会被替换为：

       int (*xInput)(void *pIn, void *pData, int *pnData),
       void *pIn,

每次 session 模块调用 xInput 回调时，传入的第一个参数都是提供的 pIn 上下文指针的复制。第二个参数 pData 指向大小为 (*pnData) 字节的缓冲区。假设没有错误，xInput 方法应将最多 (*pnData) 字节的数据复制到缓冲区，并在返回 SQLITE_OK 之前将 (*pnData) 设置为实际复制的字节数。如果输入耗尽，则 (*pnData) 应设置为零以表示这一点。或者如果发生错误，则应返回 SQLite 错误代码。在所有情况下，如果 xInput 回调返回错误，则会中止所有处理，并将错误代码返回给调用者。

在 sqlite3changeset_start_strm() 的情况下，session 模块可能在迭代器生命周期内的任何时间调用 xInput 回调。如果这样的 xInput 回调返回错误，则迭代器进入错误状态，之后对迭代器函数的所有调用将立即以相同的错误代码失败。

同样，返回变更集（或补丁集）的流式 API 通过回调函数分块返回数据，而不是通过单个大缓冲区的指针。此时，类似下面的一对参数：

       int *pnChangeset,
       void **ppChangeset,

会被替换为：

       int (*xOutput)(void *pOut, const void *pData, int nData),
       void *pOut

xOutput 回调将被调用零次或多次以返回数据给应用程序。每次调用传入的第一个参数都是应用程序提供的 pOut 指针的复制。第二个参数 pData 指向大小为 nData 字节的缓冲区，该缓冲区包含正在返回的输出数据块。如果 xOutput 回调成功处理了提供的数据，应返回 SQLITE_OK 表示成功。否则，应返回其他 SQLite 错误代码。在这种情况下会立即中止处理，并将 xOutput 的错误代码返回给应用程序。

session 模块永远不会以第三个参数小于或等于零的值调用 xOutput 回调。除此之外，对返回数据块大小不做其他保证。

将变更集或补丁集应用到数据库。这些函数尝试使用通过第二个和第三个参数提供的变更集中的更改来更新附加到句柄 db 的 “main” 数据库。

所有由这些函数进行的更改都包含在保存点事务中。如果发生任何其他错误（除了尝试写入目标数据库时的约束失败以外），则保存点事务会回滚，从而将目标数据库恢复到原始状态，并返回一个 SQLite 错误代码。此外，从 3.51.0 版本开始，可以通过 sqlite3_errcode() 和 sqlite3_errmsg() API 访问留下在数据库句柄中的错误代码和错误消息。

传递给这些函数的第四个参数 xFilter 是“过滤回调”。该参数可以传递 NULL，此时变更集中的所有更改都会应用到数据库。对于 sqlite3changeset_apply() 和 sqlite3changeset_apply_v2()，如果它不为 NULL，则它会针对变更集中至少包含一次更改的每个表调用一次。在这种情况下，表名作为第二个参数传递，作为第一个参数传递一个与 apply() 或 apply_v2() 的第六个参数相同的上下文指针副本。如果“过滤回调”返回零，则不会尝试对该表应用任何更改。否则，如果返回值非零，则会尝试应用与该表相关的所有更改。

对于 sqlite3_changeset_apply_v3()，xFilter 回调会针对每个更改调用一次。在这种情况下，第二个参数是一个 sqlite3_changeset_iter，可通过常用 API 查询当前更改的详细信息。如果“过滤回调”在此情况下返回零，则不会尝试应用当前更改。如果返回非零，则会应用该更改。

对于每个未被过滤回调排除的表，此函数会验证目标数据库是否包含一个兼容的表。如果满足以下所有条件，则该表视为兼容：

  • 表名与变更集中记录的名称相同，
  • 表的列数至少与变更集中记录的列数一样多，
  • 表的主键列位置与变更集中记录的位置相同。

如果没有兼容的表，则这不是错误，但与该表关联的任何更改都不会被应用。通过 sqlite3_log() 机制生成一个带有错误代码 SQLITE_SCHEMA 的警告消息。每个表在变更集中最多发出一次此类警告。

对于每个具有兼容表的更改，会尝试根据不被过滤回调排除的每个 UPDATE、INSERT 或 DELETE 更改修改表内容。如果某个更改无法干净地应用，则可能会调用作为 sqlite3changeset_apply() 第五个参数传递的冲突处理函数。下面描述了针对每种更改类型在何时调用冲突处理函数。

与 xFilter 参数不同，xConflict 不能传递 NULL。除有效函数指针以外的任何内容作为 xConflict 参数传递都会导致未定义行为。

每次调用冲突处理函数时，它必须返回 SQLITE_CHANGESET_OMIT、SQLITE_CHANGESET_ABORT 或 SQLITE_CHANGESET_REPLACE 之一。仅当传递给冲突处理函数的第二个参数为 SQLITE_CHANGESET_DATA 或 SQLITE_CHANGESET_CONFLICT 时，才可以返回 SQLITE_CHANGESET_REPLACE。如果冲突处理函数返回非法值，则已经完成的任何更改都会回滚，并且 sqlite3changeset_apply() 会返回 SQLITE_MISUSE。根据每次调用冲突处理函数返回的值，sqlite3changeset_apply() 会采取不同的操作。有关详细信息，请参阅三个 可用返回值 的文档。

DELETE 更改
对于每个 DELETE 更改，函数会检查目标数据库是否包含与变更集中存储的原始行值具有相同主键值（或值）的行。如果是，并且所有非主键列中存储的值也与变更集中存储的值匹配，则从目标数据库中删除该行。

如果找到具有匹配主键值的行，但一个或多个非主键字段包含与变更集中存储的原始行值不同的值，则会以 SQLITE_CHANGESET_DATA 作为第二个参数调用冲突处理函数。如果数据库表的列数多于变更集中记录的列数，则只比较这些非主键字段的值与当前数据库内容——任何附加的数据库表列都会被忽略。

如果在数据库中找不到具有匹配主键值的行，则会以 SQLITE_CHANGESET_NOTFOUND 作为第二个参数调用冲突处理函数。

如果尝试 DELETE 操作但 SQLite 返回 SQLITE_CONSTRAINT（这仅在违反外键约束时发生），则会以 SQLITE_CHANGESET_CONSTRAINT 作为第二个参数调用冲突处理函数。这包括在冲突处理函数先前返回 SQLITE_CHANGESET_REPLACE 而再次尝试 DELETE 操作的情况。

INSERT 更改
对于每个 INSERT 更改，会尝试将新行插入数据库。如果变更集行包含的字段少于数据库表，则尾随字段会用其默认值填充。

如果插入行的尝试失败，因为数据库已包含具有相同主键值的行，则会将第二个参数设置为 SQLITE_CHANGESET_CONFLICT 调用冲突处理函数。

如果插入行的尝试因为其他某些约束违例（例如 NOT NULL 或 UNIQUE）而失败，则会将第二个参数设置为 SQLITE_CHANGESET_CONSTRAINT 调用冲突处理函数。这包括冲突处理函数先前返回 SQLITE_CHANGESET_REPLACE 而再次尝试 INSERT 操作的情况。

UPDATE 更改
对于每个 UPDATE 更改，函数会检查目标数据库是否包含与变更集中存储的原始行值具有相同主键值（或值）的行。如果是，并且所有被修改的非主键列中存储的值也与变更集中存储的值匹配，则会在目标数据库中更新该行。

如果找到具有匹配主键值的行，但一个或多个被修改的非主键字段包含与变更集中存储的原始值不同的值，则会以 SQLITE_CHANGESET_DATA 作为第二个参数调用冲突处理函数。由于 UPDATE 更改仅包含要修改的非主键字段的值，因此只需这些字段与原始值匹配，以避免触发 SQLITE_CHANGESET_DATA 冲突处理回调。

如果在数据库中找不到具有匹配主键值的行，则会以 SQLITE_CHANGESET_NOTFOUND 作为第二个参数调用冲突处理函数。

如果尝试 UPDATE 操作但 SQLite 返回 SQLITE_CONSTRAINT，则会以 SQLITE_CHANGESET_CONSTRAINT 作为第二个参数调用冲突处理函数。这包括在冲突处理函数先前返回 SQLITE_CHANGESET_REPLACE 而再次尝试 UPDATE 操作的情况。

在 xConflict 回调中执行 SQL 语句（包括对与回调相关的表执行写操作）是安全的。这可以用来进一步定制应用程序的冲突解决策略。

如果输出参数 (ppRebase) 和 (pnRebase) 非空并且输入为变更集（而非补丁集），则 sqlite3changeset_apply_v2() 可能会在返回之前将 (*ppRebase) 设置为指向可与 sqlite3_rebaser APIs 一起使用的“重定位”缓冲区。在这种情况下，(*pnRebase) 设置为缓冲区的字节大小。调用者有责任最终使用 sqlite3_free() 释放任何此类缓冲区。仅在应用补丁集期间遇到一个或多个冲突时才会分配并填充缓冲区。有关更多详情，请参阅 sqlite3_rebaser APIs 周围的注释。

可通过将受支持的标志组合作为第 9 个参数传递，修改 sqlite3changeset_apply_v2() 及其流式等效项的行为。

请注意，sqlite3changeset_apply_v2() API 仍然是实验性的，因此可能会发生变化。

创建用于遍历变更集的迭代器

int sqlite3changeset_start(
  sqlite3_changeset_iter **pp,    /* 输出：新的变更集迭代器句柄 */
  int nChangeset,                 /* 变更集二进制块大小（字节） */
  void *pChangeset                /* 指向包含变更集的二进制块 */
);
int sqlite3changeset_start_v2(
  sqlite3_changeset_iter **pp,    /* 输出：新的变更集迭代器句柄 */
  int nChangeset,                 /* 变更集二进制块大小（字节） */
  void *pChangeset,               /* 指向包含变更集的二进制块 */
  int flags                       /* SESSION_CHANGESETSTART_* flags */
);

创建一个用于遍历变更集内容的迭代器。如果成功，*pp 会被设置为指向迭代器句柄，并返回 SQLITE_OK。否则，如果发生错误，*pp 会被设置为零，并返回 SQLite 错误代码。

可以使用以下函数推进并查询由此函数创建的变更集迭代器：

  • sqlite3changeset_next()
  • sqlite3changeset_op()
  • sqlite3changeset_new()
  • sqlite3changeset_old()

调用者有责任最终通过将迭代器传递给 sqlite3changeset_finalize() 来销毁它。包含变更集的缓冲区 (pChangeset) 必须在迭代器销毁之前保持有效。

假设变更集二进制块是由 sqlite3session_changeset()、sqlite3changeset_concat() 或 sqlite3changeset_invert() 函数之一创建的，则变更集中应用于单个表的所有更改将被分组在一起。这意味着当应用程序使用由此函数创建的迭代器遍历变更集时，与单个表相关的所有更改都会连续访问。迭代器不会先访问一个应用到表 X 的更改，再访问一个表 Y 的更改，然后再访问另一个表 X 的更改。

可通过将受支持的标志组合作为第 4 个参数传递，修改 sqlite3changeset_start_v2() 及其流式等效项的行为。

请注意，sqlite3changeset_start_v2() API 仍然是实验性的，因此可能会发生变化。
