确定性 SQL 函数

1. 概述

SQLite 中的 SQL 函数可以是“确定性”的，也可以是“非确定性”的。

确定性函数在输入相同的情况下总是返回相同的答案。SQLite 中的大多数内置 SQL 函数都是确定性的。例如，只要其输入 X 相同，abs(X) 函数就始终返回相同的结果。

非确定性函数即使在参数始终相同的情况下，每次调用也可能返回不同的答案。以下是一些非确定性函数的示例：

  • random()
  • changes()
  • last_insert_rowid()
  • sqlite3_version()

random() 函数显然是非确定性的，因为每次调用都会返回不同的答案。changes() 和 last_insert_rowid() 的结果取决于先前的 SQL 语句，因此它们也是非确定性的。sqlite3_version() 函数在大多数情况下是恒定的，但在 SQLite 升级时它可能会改变，因此即使在某个特定会话中它始终返回相同结果，由于跨会话答案可能发生变化，它仍然被视为非确定性。

2. 使用非确定性函数的限制

SQLite 中有一些上下文不允许使用非确定性函数：

  • 在 CHECK 约束 的表达式中。
  • 在 部分索引 的 WHERE 子句中。
  • 在用作 表达式索引 的一部分的表达式中。
  • 在 生成列 的表达式中。

在上述情况下，函数返回的值会影响数据库文件中存储的信息。CHECK 约束中的函数值决定了表中哪些条目是有效的，部分索引的 WHERE 子句或表达式索引中的函数会计算存储在索引 B 树中的值。如果其中任何一个函数后来返回了不同的值，那么数据库可能不再是良好结构。因此，为了避免数据库损坏，上述上下文中只能使用确定性函数。

3. 日期/时间函数的特殊处理

SQLite 的内置日期和时间函数是一个特殊情况。这些函数通常被认为是确定性的。然而，如果这些函数使用字符串“now”作为日期，或者使用localtime 修饰符或utc 修饰符，那么它们就被视为非确定性的。因为函数的输入在运行时之前不一定已知，所以如果这些日期/时间函数在仅允许确定性函数的上下文中遇到任何非确定性特性，它们会抛出异常。

在 SQLite 3.20.0（2017-08-01）之前，所有日期/时间函数始终被视为非确定性的。根据其参数，日期/时间函数有时是确定性的、有时是非确定性的这一能力是在 3.20.0 版本中添加的。

3.1. 版本 3.35.2 的错误修复

当 SQLite 3.20.0 引入增强，使得日期/时间函数在不依赖当前时间的情况下被视为确定性时，一个情况被忽略了：许多日期/时间函数可以在完全没有参数的情况下调用。这些无参数的日期/时间函数表现得就像它们有一个单独的“'now'”参数。因此 “datetime()” 和 “datetime('now')” 都会返回当前的日期和时间。然而，只有第二种形式被视为非确定性的。这意味着开发者可以将非确定性的 “datetime()” 形式悄悄地插入到 CHECK 约束、索引表达式、生成列表达式以及其他不允许非确定性函数的地方。该漏洞在 3.35.2（2021-03-17）版本中修复。然而，可能仍有由 SQLite 3.20.0 到 3.35.1 创建的旧数据库，其模式中包含非确定性的日期/时间函数。

4. 应用程序定义的确定性函数

默认情况下，应用程序定义的 SQL 函数被视为非确定性的。然而，如果将 sqlite3_create_function_v2() 的第四个参数与 SQLITE_DETERMINISTIC 进行按位或运算，那么 SQLite 会将该函数视为确定性函数。

请注意，如果一个非确定性函数被标记为 SQLITE_DETERMINISTIC，并且该函数最终用于 部分索引 的 WHERE 子句或 表达式索引 中，那么当该函数开始返回不同答案时，相关索引可能会损坏。如果某个 SQL 函数几乎是确定性的（也就是说，它很少变化，比如 sqlite_version()），并且它被用于导致索引损坏的索引中，则可以通过运行 REINDEX 来修复损坏。

构造一个根据输入有时是确定性、有时是非确定性的函数（如内置的日期/时间函数）所需的接口并未公开。通用的应用程序定义的 SQL 函数必须始终是确定性的或始终是非确定性的。
