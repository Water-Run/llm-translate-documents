2. 保存点

SAVEPOINTs 是创建事务的一种方式，类似于BEGIN 和 COMMIT，不过 SAVEPOINT 和 RELEASE 命令是有名称的并且可以嵌套。

SAVEPOINT 命令以一个名称开始一个新的事务。事务名称不必唯一。SAVEPOINT 可以在 BEGIN...COMMIT 之内或之外启动。当一个 SAVEPOINT 是最外层的保存点并且不在 BEGIN...COMMIT 之内时，其行为与 BEGIN DEFERRED TRANSACTION 相同。

ROLLBACK TO 命令将数据库状态回退到对应 SAVEPOINT 之后的状态。请注意，与不带 TO 关键字的普通 ROLLBACK 命令不同，ROLLBACK TO 命令不会取消事务。ROLLBACK TO 命令不是取消事务，而是重新从起点启动该事务。所有中间的 SAVEPOINT 都会被取消。

RELEASE 命令相当于 SAVEPOINT 的 COMMIT。RELEASE 命令会移除事务堆栈中直到最近匹配名称的那一个保存点及其之前的所有保存点。释放内层事务不会将任何更改写入数据库文件；它只是从事务堆栈中移除保存点，使得无法再回滚至这些保存点。如果 RELEASE 命令释放了最外层的保存点，以致事务堆栈变为空，那么 RELEASE 就相当于 COMMIT。即使事务最初是由 SAVEPOINT 命令而不是 BEGIN 命令启动，也可以使用 COMMIT 命令来释放所有保存点并提交事务。

如果 RELEASE 命令中的 savepoint-name 与事务堆栈中目前的任何保存点都不匹配，则不会释放任何保存点，数据库保持不变，RELEASE 命令返回错误。

请注意，内层事务可能通过 RELEASE 命令提交，但随后可能被外层事务中的 ROLLBACK 撤销。断电、程序崩溃或操作系统崩溃会导致最外层事务回滚，撤销该外层事务期间发生的所有更改，甚至是已经通过 RELEASE 命令“提交”的更改。内容在磁盘上真正提交之前，最外层事务必须提交。

可以从几种方式理解 RELEASE 命令：

  • 有些人认为 RELEASE 相当于 SAVEPOINT 的 COMMIT。这种观点是可以接受的，只要记住内层事务提交的更改可能会后来被外层事务的回滚撤销。

  • 另一种观点是，RELEASE 会将具名事务合并到其父事务中，使得具名事务及其父事务成为同一个事务。RELEASE 之后，具名事务和它的父事务将一起提交或回滚，命运一致。

  • 也可以将保存点看作事务时间线上的“标记”。在这种观点下，SAVEPOINT 命令创建一个新标记，ROLLBACK TO 命令将时间线倒退到紧接该标记之后的位置，而 RELEASE 命令则擦除时间线上的标记而不真正对数据库做出任何更改。

3. 事务嵌套规则

最后启动的事务将是第一个提交或回滚的事务。

BEGIN 命令仅在事务堆栈为空时有效，也就是说没有挂起的事务。如果在调用 BEGIN 命令时事务堆栈不为空，则该命令失败并返回错误。

COMMIT 命令提交所有未完成的事务并清空事务堆栈。

RELEASE 命令从事务堆栈中最近添加的保存点开始，向后释放保存点直到释放一个具有匹配 savepoint-name 的保存点为止。先前的保存点，即便名称匹配，也保持不变。如果 RELEASE 命令导致事务堆栈为空（即 RELEASE 释放了堆栈中的最外层事务），则事务提交。

不带 TO 子句的 ROLLBACK 命令回滚所有事务并清空事务堆栈。

带有 TO 子句的 ROLLBACK 命令沿时间线向后回滚事务，直到遇到最近的具有匹配名称的 SAVEPOINT。匹配名称的 SAVEPOINT 仍保留在事务堆栈上，但在该 SAVEPOINT 创建之后发生的所有数据库更改都会被回滚。如果 ROLLBACK TO 命令中的 savepoint-name 与堆栈上的任何 SAVEPOINT 都不匹配，则 ROLLBACK 命令失败并返回错误，数据库状态保持不变。
