2. 事务

除非在一个事务中，否则不会发生任何读写。任何访问数据库的命令（基本上是任何 SQL 命令，除了少数 PRAGMA 语句）如果尚未有事务生效，将自动启动一个事务。自动启动的事务会在最后一个 SQL 语句完成后提交。

可以使用 BEGIN 命令手动启动事务。这样的事务通常会持续到下一个 COMMIT 或 ROLLBACK 命令。但如果数据库被关闭，或者发生错误并指定了 ROLLBACK 冲突解决算法，事务也会回滚。有关 ROLLBACK 冲突解决算法的更多信息，请参阅 ON CONFLICT 子句的文档。

END TRANSACTION 是 COMMIT 的别名。

通过 BEGIN...COMMIT 创建的事务不支持嵌套。对于嵌套事务，请使用 SAVEPOINT 和 RELEASE 命令。上面语法图中 ROLLBACK 命令的 “TO SAVEPOINT name” 子句仅适用于 SAVEPOINT 事务。无论事务是由 SAVEPOINT 还是先前的 BEGIN 启动的，尝试在事务内再次调用 BEGIN 命令都会以错误失败。COMMIT 命令以及没有 TO 子句的 ROLLBACK 命令对 SAVEPOINT 事务和由 BEGIN 启动的事务具有相同的行为。

2.1. 读事务与写事务

SQLite 支持来自不同数据库连接（可能在不同线程或进程中）的多个并发读事务，但同时只能有一个写事务。

读事务仅用于读取。写事务可以同时读取和写入。读事务由 SELECT 语句启动，写事务由 CREATE、DELETE、DROP、INSERT 或 UPDATE 等语句（统称为“写语句”）启动。如果在读事务仍然活动时出现写语句，则会尝试将读事务升级为写事务。如果另一个数据库连接已经修改了数据库或正在修改数据库，则无法升级为写事务，此时写语句将返回 SQLITE_BUSY 错误。

当读事务活动时，由其他数据库连接实施的数据库更改对启动该读事务的连接是不可见的。如果数据库连接 X 保持着读事务，那么其他数据库连接 Y 可能在 X 的事务仍然打开期间更改数据库内容，但 X 在该事务结束前将无法看到这些更改。在读事务活动期间，X 会持续看到修改前的历史快照。

2.2. DEFERRED、IMMEDIATE 与 EXCLUSIVE 事务

事务可以是 DEFERRED、IMMEDIATE 或 EXCLUSIVE。默认的事务行为是 DEFERRED。

DEFERRED 表示事务在数据库首次访问之前实际上并不会启动。内部而言，BEGIN DEFERRED 语句只是设置数据库连接上的一个标志，以关闭当最后一个语句完成时通常发生的自动提交。这会导致自动启动的事务持续，直到明确执行 COMMIT 或 ROLLBACK，或遭遇错误或带有 ON CONFLICT ROLLBACK 子句时被迫回滚。如果 BEGIN DEFERRED 之后的第一个语句是 SELECT，则启动读事务。随后出现的写语句会在可能的情况下将事务升级为写事务，否则返回 SQLITE_BUSY。如果 BEGIN DEFERRED 之后的第一个语句是写语句，则直接启动写事务。

IMMEDIATE 使数据库连接立即启动一个新的写事务，而无需等待写语句。如果其他数据库连接上已经存在写事务，BEGIN IMMEDIATE 可能会失败并返回 SQLITE_BUSY。

EXCLUSIVE 与 IMMEDIATE 相似，都会立即启动写事务。在 WAL 模式 中，EXCLUSIVE 和 IMMEDIATE 行为相同，但在其他日志记录模式中，EXCLUSIVE 会在事务进行时阻止其他数据库连接读取数据库。

2.3. 隐式事务与显式事务

隐式事务（即自动启动的事务，而非由 BEGIN 启动的事务）会在最后一个活动语句完成后自动提交。语句完成意味着其最后一个游标关闭，这在调用 reset 或 finalize 时是有保证的。有些语句在被重置或终结之前可能就已经“完成”，但对此没有保证。确保语句已经“完成”的唯一方法是调用该语句上的 sqlite3_reset() 或 sqlite3_finalize()。用于增量 BLOB I/O 的打开的 sqlite3_blob 也被视为未完成的语句。该 sqlite3_blob 在被 关闭 时才完成。

显式的 COMMIT 命令会立即执行，即使还有待处理的 SELECT 语句。但如果存在待处理的写操作，则 COMMIT 命令会以 SQLITE_BUSY 错误失败。

如果另一个线程或进程有打开的读连接，尝试执行 COMMIT 也可能返回 SQLITE_BUSY。在这种失败的情况下，事务仍然保持活动状态，等到读者有机会释放后可以重新尝试 COMMIT。

在非常旧的 SQLite 版本（3.7.11 之前，即 2012-03-20 前）中，如果有任何待处理查询，ROLLBACK 会以 SQLITE_BUSY 错误失败。在更新的 SQLite 版本中，ROLLBACK 会继续执行，待处理语句通常会被中止，使它们返回 SQLITE_ABORT 或 SQLITE_ABORT_ROLLBACK 错误。在 SQLite 3.8.8（2015-01-16）及之后的版本中，只要 ROLLBACK 不修改数据库模式，待处理的读取在 ROLLBACK 后仍会继续运行。

如果将 PRAGMA journal_mode 设置为 OFF（从而禁用回滚日志文件），则 ROLLBACK 命令的行为不确定。

3. 事务内错误的响应

如果在事务中发生某些类型的错误，事务可能会或可能不会自动回滚。可能导致自动回滚的错误包括：

  • SQLITE_FULL：数据库或磁盘已满
  • SQLITE_IOERR：磁盘 I/O 错误
  • SQLITE_BUSY：数据库正被其他进程使用
  • SQLITE_NOMEM：内存不足

对于以上所有错误，SQLite 会尝试撤销其正在执行的语句，仅保留该事务中之前语句的更改并继续该事务。然而，根据正在执行的语句和错误发生的时点，SQLite 也可能需要回滚并取消整个事务。应用可以使用 sqlite3_get_autocommit() C 语言接口来判定 SQLite 采取了哪种处理方式。

建议应用在遇到上述错误时显式发出 ROLLBACK 命令。如果事务已经被错误响应自动回滚，则 ROLLBACK 命令会因错误而失败，但不会造成任何问题。

将来的 SQLite 版本可能会扩展可能导致自动事务回滚的错误列表。将来的版本也可能修改错误响应。特别是，我们可能会在未来版本中选择通过强制上述错误进行无条件回滚来简化接口。
