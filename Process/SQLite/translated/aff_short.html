SQLite 作为应用程序文件格式

（注：当前页面简要总结了 SQLite 作为应用程序文件格式的优势。该主题在单独的技术说明中有更详细的讨论。）

SQLite 一直被成功用作桌面应用程序的磁盘文件格式，例如版本控制系统、金融分析工具、媒体目录与编辑套件、CAD 套件、记录管理程序等。传统的“文件/打开”操作通过 sqlite3_open() 附加到数据库文件。随着应用内容的更新，更新操作会自动发生，因此“文件/保存”菜单选项变得多余。“文件/另存为”菜单选项可以使用备份 API实现。

将 SQLite 用作应用程序文件格式具有许多优势，包括：

  1. 更好的性能
    • 从 SQLite 数据库读取和写入通常比从磁盘读取和写入单个文件更快。参见比文件系统快 35%和内部与外部 BLOB。
    • 应用只需加载所需的数据，而不是读取整个文件并在内存中保持完整的解析内容。
    • 小的修改仅覆盖文件中发生更改的部分，从而减少写入时间并降低对 SSD 的磨损。
  2. 降低应用程序成本和复杂性
    • 无需编写和调试应用程序文件 I/O 代码。
    • 内容可以通过简洁的 SQL 查询访问和更新，而不是冗长且易出错的过程式例程。
    • 文件格式可以通过在未来版本中添加新表和/或列来扩展，同时保持向后兼容性。
    • 应用程序可以利用全文搜索和RTREE索引，并使用触发器实现自动撤销/重做栈。
    • 即便在开发周期的后期，性能问题也经常可以通过CREATE INDEX等方式解决，避免昂贵的重设计、重写和重测工作。
  3. 可移植性
    • 应用程序文件可以在所有操作系统、32 位与 64 位以及大端与小端架构之间移植。
    • 有可能由不同编程语言编写的一组程序可以访问同一个应用程序文件而没有兼容性问题。
    • 多个进程可以附加到同一个应用程序文件，并能够并发读写而互不干扰。
    • 本可能以“文件堆”形式存储的多样内容被封装到一个单一的磁盘文件中，便于通过 scp/ftp、USB 盘或电子邮件附件等方式传输。
  4. 可靠性
    • 内容可以持续地进行原子更新，因此在停电或崩溃时几乎不会丢失工作。
    • 在 SQLite 中出现 bug 的概率远低于自定义文件 I/O 代码。
    • SQL 查询远比等价的过程式代码短，而每行代码出现 bug 的概率大致相同，这意味着总体 bug 更少。
  5. 可访问性
    • 可以使用各种第三方工具查看 SQLite 数据库内容。
    • 存储在 SQLite 数据库中的内容更有可能在几十年后被恢复，即使原始应用程序的所有痕迹都已消失。数据寿命长于代码。
    • 美国国会图书馆推荐使用 SQLite 数据库文件作为长期保存数字内容的存储格式。

SQLite 允许数据库文件使用任何所需的文件扩展名，因此应用可以按需为自身选择自定义扩展名。可以使用application_id pragma在数据库文件中设置“应用程序 ID”整数，让像file(1)这样的工具判断该文件是与您的应用程序相关，而不仅仅是通用的 SQL 数据库。
