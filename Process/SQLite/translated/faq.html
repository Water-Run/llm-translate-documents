(1) 如何创建自增长字段？

简短回答：声明为 INTEGER PRIMARY KEY 的列将自动增长。

详细回答： 如果你将表的某一列声明为 INTEGER PRIMARY KEY， 那么每当你向该列插入 NULL 时，NULL 会自动转换为一个整数， 其值比该表其他行中该列的最大值大 1；如果表为空，则取值为 1。 如果现存的最大整数键值 9223372036854775807 已被使用， 那么将随机选择一个未被使用的键值。 例如，假设你有如下的表： 

CREATE TABLE t1(
  a INTEGER PRIMARY KEY,
  b INTEGER
);

在这个表中，以下语句

INSERT INTO t1 VALUES(NULL,123);

在逻辑上等价于：

INSERT INTO t1 VALUES((SELECT max(a) FROM t1)+1,123);

有一个名为 sqlite3_last_insert_rowid() 的函数，它会返回最近一次插入操作的整数键。

请注意，整数键比插入之前表中存在的最大键大 1。 新的键在当前表中是唯一的，但可能与之前被删除的键重叠。 如果想让键在整个表生命周期内都是唯一的，请在 INTEGER PRIMARY KEY 声明后添加 AUTOINCREMENT 关键字。 那样选择的键将比该表中曾经存在的最大键还大 1。 如果该表中曾经存在过可能的最大键，则 INSERT 会以 SQLITE_FULL 错误代码失败。

(2) SQLite 支持哪些数据类型？

SQLite 使用 动态类型。内容可以存储为 INTEGER、REAL、TEXT、BLOB，或 NULL。

(3) SQLite 竟然允许我将字符串插入整数类型的列！

这是一个特性，不是 bug。SQLite 使用 动态类型。 它不会强制执行数据类型约束。任何类型的数据（通常）都可以插入到任意列中。 你可以在整数列中插入任意长度的字符串，在布尔列中插入浮点数， 或在字符列中插入日期。你在 CREATE TABLE 命令中为列指定的 数据类型 并不会限制该列能存储什么数据。每一列都可以保存任意长度的字符串。 （有一个例外：INTEGER PRIMARY KEY 类型的列只能保存 64 位有符号整数。 如果你试图向 INTEGER PRIMARY KEY 列插入其他类型的值，会导致错误。）

但 SQLite 会将列声明的类型视为你偏好的值格式提示。 所以，例如，如果某列是 INTEGER 类型，而你尝试插入字符串，SQLite 会尝试将该字符串转换为整数。 如果转换成功，它就插入整数；否则，它插入字符串。 这个特性称为 类型亲和性。

(4) 为什么 SQLite 不允许我在同一个表的两行中分别使用 '0' 和 '0.0' 作为主键？

当你的主键是数值类型时，会出现这个问题。 将主键的 数据类型 改为 TEXT，它就可以工作。

每一行必须有唯一的主键。对于数值类型的列，SQLite 认为 '0' 和 '0.0' 在数值上是相等的。（参见前一个问题。）因此这些值不是唯一的。

(5) 多个应用或同一应用的多个实例可以同时访问一个数据库文件吗？

多个进程可以同时打开同一个数据库文件。多个进程可以同时执行 SELECT。 但在任意时刻，只允许一个进程对数据库做修改。

SQLite 使用读写锁来控制数据库访问。 （在不支持读写锁的 Win95/98/ME 上，使用一种概率模拟机制。） 但请注意：如果数据库文件存放在 NFS 文件系统上，这种锁机制可能无法正常工作。 这是因为很多 NFS 实现中的 fcntl() 文件锁存在缺陷。 如果多个进程可能同时尝试访问该文件，应避免将 SQLite 数据库放在 NFS。 在 Windows 上，微软的文档指出如果没有运行 Share.exe 守护进程， FAT 文件系统上的锁可能无法正常工作。 拥有丰富 Windows 经验的人告诉我，网络文件锁非常不稳定且不可靠。 如果他们说的是真的，在两台或多台 Windows 机器之间共享 SQLite 数据库可能会导致意外问题。

我们知道没有其他嵌入式 SQL 数据库引擎能像 SQLite 那样支持这么多并发。 SQLite 允许多个进程同时打开数据库文件，并同时读取数据库。 当任一进程想写入时，它必须在更新期间锁住整个数据库文件。 但这通常只需要几毫秒。其他进程只需等待写入完成，然后继续自己的操作。 其他嵌入式 SQL 数据库引擎通常只允许单个进程连接数据库。

然而，客户端/服务器数据库引擎（如 PostgreSQL、MySQL 或 Oracle）通常支持更高的并发级别，并允许多个进程同时写入同一个数据库。 这是因为客户端/服务器数据库始终有一个受控的服务器进程来协调访问。 如果你的应用需要大量并发，应考虑使用客户端/服务器数据库。 但经验表明，绝大多数应用所需的并发远低于设计者的想象。

当 SQLite 尝试访问被其他进程锁住的文件时，默认行为是返回 SQLITE_BUSY。 你可以在 C 代码中通过 sqlite3_busy_handler() 或 sqlite3_busy_timeout() 接口函数调整这一行为。

(6) SQLite 是线程安全的吗？

线程是邪恶的。 尽量避免使用它们。

SQLite 是线程安全的。我们做出这个让步是因为很多用户选择忽略前一段中的建议。 但为了线程安全，SQLite 必须用 SQLITE_THREADSAFE 预处理宏设置为 1 编译。 发行版中的 Windows 和 Linux 预编译二进制文件都是这样编译的。 如果你不确定链接的 SQLite 库是否线程安全，可以调用 sqlite3_threadsafe() 接口来确认。

SQLite 通过互斥锁序列化对公共数据结构的访问，因此是线程安全的。 不过，获取和释放这些互斥锁会略微降低 SQLite 的速度。 因此，如果你不需要线程安全，应该禁用互斥锁以获得最大性能。 详情请参阅 线程模式 文档。

在 Unix 下，不要在 fork() 系统调用中将已打开的 SQLite 数据库传给子进程。

(7) 如何列出 SQLite 数据库中包含的所有表/索引？

如果你正在运行 sqlite3 命令行访问程序， 可以输入 ".tables" 以获取所有表的列表。 或者输入 ".schema" 查看完整的数据库模式，包括所有表和索引。 这些命令都可以接一个 LIKE 模式参数来限制显示的表。

在 C/C++ 程序（或使用 Tcl/Ruby/Perl/Python 绑定的脚本）中， 可以通过对名为 "SQLITE_SCHEMA" 的特殊表执行 SELECT 来获取表和索引的名称。 每个 SQLite 数据库都有一个 SQLITE_SCHEMA 表来定义数据库的模式。 SQLITE_SCHEMA 表如下所示：

CREATE TABLE sqlite_schema (
  type TEXT,
  name TEXT,
  tbl_name TEXT,
  rootpage INTEGER,
  sql TEXT
);

对于表，type 字段始终为 'table'，name 字段即表名。 所以，要获取数据库中所有表的列表，请使用如下 SELECT 命令：

SELECT name FROM sqlite_schema
WHERE type='table'
ORDER BY name;

对于索引，type 等于 'index'，name 是索引名，tbl_name 是该索引所属的表名。 对于表和索引，sql 字段是创建该表或索引时的原始 CREATE TABLE 或 CREATE INDEX 语句文本。 对于自动创建的索引（用于实现 PRIMARY KEY 或 UNIQUE 约束），sql 字段为 NULL。

不能使用 UPDATE、INSERT 或 DELETE 修改 SQLITE_SCHEMA 表（除非在 特殊情况下）。 SQLITE_SCHEMA 表由 CREATE TABLE、CREATE INDEX、DROP TABLE 和 DROP INDEX 等命令自动更新。

临时表不会出现在 SQLITE_SCHEMA 表中。临时表及其索引和触发器位于另一个特殊表 SQLITE_TEMP_SCHEMA 中。 SQLITE_TEMP_SCHEMA 的工作方式与 SQLITE_SCHEMA 相同，只对创建临时表的应用可见。 如果要列出永久表和临时表，可以使用如下命令：

SELECT name FROM 
   (SELECT * FROM sqlite_schema UNION ALL
    SELECT * FROM sqlite_temp_schema)
WHERE type='table'
ORDER BY name

(8) SQLite 数据库有没有已知的大小限制？

有关 SQLite 限制的完整讨论，请参见 limits.html。

(9) SQLite 中 VARCHAR 的最大长度是多少？

SQLite 不会强制执行 VARCHAR 的长度。 你可以声明 VARCHAR(10)，SQLite 也会乐意存储一个 5 亿字符的字符串。 且它会完整保留这 5 亿个字符，不会被截断。 SQLite 将 "VARCHAR(N)" 视为与 "TEXT" 相同的列类型，无论 N 的值是多少。

(10) SQLite 支持 BLOB 类型吗？

SQLite 允许你在任何列中存储 BLOB 数据，即使该列声明为存储其他类型。 BLOB 甚至可以用作 PRIMARY KEY。

(11) 如何在 SQLite 中对现有表添加、删除或重命名列？

SQLite 提供有限的 ALTER TABLE 支持，可用于添加、重命名或删除列，或更改表名， 详情请参见 ALTER TABLE。

如果你想对表或其列的结构或约束进行更复杂的更改，就必须重新创建表。 可以先将现有数据保存到临时表，删除旧表，创建新表，然后再从临时表复制数据回来。 详见 执行其它类型的表模式更改。

(12) 我删除了大量数据但数据库文件并没有变小。这是 bug 吗？

不是。当你从 SQLite 数据库中删除信息时，未使用的磁盘空间会加入内部的“空闲列表”，并在下一次插入数据时重用。 磁盘空间不会丢失，但也不会返回给操作系统。

如果删除了大量数据并想缩小数据库文件，请运行 VACUUM 命令。 VACUUM 将从头重建数据库，从而清空空闲列表并使文件达到最小大小。 但请注意，VACUUM 可能需要一些时间执行，并且在运行期间可能使用比原始文件多一倍的临时磁盘空间。

另一种替代 VACUUM 的方式是使用自动回收模式，可通过 auto_vacuum pragma 启用。

(13) 我可以在商业产品中使用 SQLite 而不支付版权费吗？

可以。SQLite 属于 公共领域。 对代码的任何部分均不主张所有权。你想怎么用都可以。

(14) 如何使用包含嵌入单引号 (') 的字符串字面量？

SQL 标准规定字符串中的单引号通过连续两个单引号来转义。 SQL 在这方面像 Pascal 编程语言。示例： 

    INSERT INTO xyz VALUES('5 O''clock');

(15) 什么是 SQLITE_SCHEMA 错误，我为什么会遇到它？

当准备好的 SQL 语句不再有效且无法执行时，会返回 SQLITE_SCHEMA 错误。 当发生这种情况时，必须使用 sqlite3_prepare() API 从 SQL 重新编译该语句。 只有在使用 sqlite3_prepare() 和 sqlite3_step() 接口执行 SQL 时才可能出现 SQLITE_SCHEMA 错误。 你永远不会从 sqlite3_exec() 收到 SQLITE_SCHEMA 错误。 如果使用 sqlite3_prepare_v2() 而不是 sqlite3_prepare() 准备语句，也不会收到该错误。

sqlite3_prepare_v2() 接口会创建一个 prepared statement，当模式发生变化时会自动重新编译。 处理 SQLITE_SCHEMA 错误最简单的方式就是始终使用 sqlite3_prepare_v2()，而不是 sqlite3_prepare()。

(17) 我在编译 SQLite 时会看到一些编译器警告。这是个问题吗？这是不是说明代码质量不好？

SQLite 的质量保障通过 全覆盖测试 实现，而不是通过编译器警告或其他静态代码分析工具。 换句话说，我们验证的是 SQLite 是否能得出正确答案，而不是仅满足风格约束。 SQLite 的大部分代码量都用于测试。 SQLite 测试套件运行数万条独立测试用例，其中很多是参数化的，因此在每次发布前会执行数亿条 SQL 语句相关的测试并验证正确性。 开发者使用代码覆盖工具来验证所有路径都得到了测试。 每当在 SQLite 中发现 bug，就会编写新的测试用例来展现该 bug，以便它不会在未来再次未被察觉。

在测试期间，SQLite 库会使用特殊的插桩编译，这些测试脚本可以模拟各种故障以验证 SQLite 是否能够正确恢复。 内存分配被仔细跟踪，即使在内存分配失败之后也不会发生泄漏。 使用自定义 VFS 层模拟操作系统崩溃和断电，以确保事务在这些事件中仍然是原子性的。 还引入了刻意注入 I/O 错误的机制，以展示 SQLite 对此类故障的韧性。 （作为实验，试着在其他 SQL 数据库引擎上引发此类错误，看看会发生什么！）

我们还在 Linux 上使用 Valgrind 运行 SQLite，并确认未检测到问题。

有人说应该消除所有警告，因为无害的警告会掩盖未来可能出现的真正警告。确实如此。 但开发者回复说，他们用于 SQLite 开发的构建（各种版本的 GCC、MSVC 和 clang）中的所有警告都已经被修复。 编译器警告通常只会在 SQLite 开发者自己不使用的编译器或编译选项下出现。

(18) Unicode 字符的大小写不敏感匹配无法工作。

SQLite 的默认配置仅支持 ASCII 字符的大小写不敏感比较。 这是因为实现完整的 Unicode 不区分大小写比较和大小写转换需要的表格与逻辑几乎会使 SQLite 库的体积翻倍。 SQLite 开发者认为，任何需要完整 Unicode 大小写支持的应用很可能已经拥有了必要的表格和函数，所以 SQLite 不应占用空间去重复实现这一能力。

SQLite 没有默认提供完整的 Unicode 大小写支持，而是提供了链接外部 Unicode 比较与转换例程的能力。 应用可以重载内建的 NOCASE 排序序列（通过 sqlite3_create_collation()）以及内建的 like()、upper() 和 lower() 函数（通过 sqlite3_create_function()）。 SQLite 源码包含一个 “ICU” 扩展来执行这些重载。或者，开发者也可以基于自己项目中已有的 Unicode 识别比较例程编写自己的重载。

(19) INSERT 非常慢——我每秒只能执行几十次 INSERT

实际上，SQLite 在普通桌面电脑上每秒可以轻松执行 50,000 次或更多的 INSERT 语句。 但它每秒只能完成几十个事务。事务速度受限于磁盘驱动器的转速。 一个事务通常需要磁盘盘片旋转两周，在 7200RPM 硬盘上限制了大约 60 次事务每秒。

事务速度受限于磁盘驱动器，是因为（默认情况下）SQLite 会等待数据真正安全地写入磁盘表面后才完成事务。 这样，即使突然断电或操作系统崩溃，你的数据仍然安全。 有关细节，请阅读 SQLite 中的原子提交。

默认情况下，每个 INSERT 语句都是一个独立事务。 但如果你将多个 INSERT 语句包裹在 BEGIN...COMMIT 中， 那么所有插入都组成一个事务。 提交事务所需的时间会在所有包含的插入语句之间摊销，因此每条插入语句的耗时会大大降低。

另一种选择是执行 PRAGMA synchronous=OFF。 该命令会让 SQLite 不再等待数据写入磁盘表面，这会让写操作看起来更快。 但如果在事务中途断电，数据库文件可能会损坏。

更新 2024-11-19：上述内容是非常古老的回答。 现在 SQLite 的插入速度远远超过 5 万次/秒，而且现在大家都使用 SSD，使性能更加出色。 但上述回答的要旨仍然正确：将多条操作放入单个事务可以显著提升性能，避免每条操作后都进行事务控制的开销。

(20) 我不小心从 SQLite 数据库中删除了一些重要信息。如何恢复？

如果你有数据库文件的备份，请从备份中恢复信息。

如果没有备份，恢复就非常困难。 你可能能够在原始数据库文件的二进制转储中找到部分字符串数据。 在有特殊工具的情况下，恢复数值数据也可能可行，尽管据我们所知尚无此类工具。 SQLite 有时会使用 SQLITE_SECURE_DELETE 选项编译，会将所有已删除内容覆盖为零。 如果是这种情况，恢复显然是不可能的。 如果自删除以来执行过 VACUUM，恢复也不可能。 如果未使用 SQLITE_SECURE_DELETE 且未执行 VACUUM，那么某些已删除内容可能仍留在数据库文件中、等待重用的区域中。 但同样，我们不知道有任何程序或工具能帮助你恢复这些数据。

(21) 什么是 SQLITE_CORRUPT 错误？数据库“格式错误”是什么意思？为什么会出现这个错误？

当 SQLite 检测到数据库文件的结构、格式或其他控制元素存在错误时，会返回 SQLITE_CORRUPT 错误。

在没有外部干扰的情况下，SQLite 不会损坏数据库文件。 如果你的应用在更新过程中崩溃，你的数据是安全的。 即使你的操作系统崩溃或断电，数据库也是安全的。 SQLite 的抗崩溃性经过广泛研究和测试，并有数十亿用户多年的实际经验为证。

尽管如此，外部程序或硬件/操作系统中的 bug 可能导致数据库文件损坏。 有关更多信息，请参见 如何破坏 SQLite 数据库文件。

你可以使用 PRAGMA integrity_check 进行一次彻底但耗时的完整性测试。

你可以使用 PRAGMA quick_check 进行更快但不那么彻底的完整性测试。

根据数据库损坏的程度，你也许可以通过 CLI 将模式和内容导出到文件再重建来恢复部分数据。 不幸的是，一旦“混乱小人”掉下墙来，通常就无法再把它拼回来。

(22) SQLite 支持外键吗？

自 3.6.19 版本（2009-10-14）起， SQLite 支持 外键约束。 但默认情况下为了向后兼容，外键约束的执行是关闭的。 若要启用外键约束执行，请运行 PRAGMA foreign_keys=ON，或在编译时使用 -DSQLITE_DEFAULT_FOREIGN_KEYS=1。

(23) 如果在构建 SQLite 时使用 SQLITE_OMIT_... 编译时选项，我会遇到编译错误。

SQLITE_OMIT_... 编译时选项仅在从规范源文件构建时有效。 从 SQLite 合并源或预处理后的源文件构建时，它们 无效。

可以构建一个特殊的 合并源，使其与预设的一组 SQLITE_OMIT_... 选项配合使用。 相关说明请参阅 SQLITE_OMIT_... 文档。

(24) 我的 WHERE 子句表达式 column1="column1" 不起作用。表中的所有行都返回，而不仅仅是 column1 值为 "column1" 的那些行。

在 SQL 中，请对字符串字面量使用单引号，而不是双引号。 这是 SQL 标准的要求。你的 WHERE 子句应该写成：column1='column1'

SQL 使用双引号来引用包含特殊字符或为关键字的标识符（列名或表名）。 因此双引号是转义标识符名称的方式。 所以当你写 column1="column1" 时，相当于 column1=column1，这显然总是成立。

(25) SQLite 的语法图（即“铁路”图）是如何生成的？

每个图都是使用 Pikchr 图形语言手工编写的。 这些手工编写的规范被转换为 SVG，并作为文档构建过程的一部分内联插入 HTML 文件中。

SQLite 文档的许多历史版本使用另一种流程生成语法图。 该历史流程基于 Tcl/Tk，描述见 http://wiki.tcl-lang.org/21708。 基于 Pikchr 的新语法图于 2020-09-26 首次出现在 trunk。

(26) SQL 标准要求即使约束中的一个或多个列为 NULL 也必须强制执行 UNIQUE 约束，但 SQLite 没有这样做。这不是 bug 吗？

也许你指的是 SQL92 中的如下声明：
当且仅当在唯一列中没有两行拥有相同的非空值时，唯一约束才得以满足。
该声明含糊不清，至少有两种解释：
  1. 唯一约束当且仅当表中的任意两行在唯一列上既值不同又具有非空值时得以满足。
  2. 唯一约束当且仅当表中任意两行在非空唯一列的子集上值不同。
SQLite 采用解释（1），PostgreSQL、MySQL、Oracle 和 Firebird 也是如此。 虽然 Informix 和 Microsoft SQL Server 采用解释（2）， 但我们 SQLite 开发者认为解释（1）是最自然的理解方式， 并且我们也希望最大限度地与其他 SQL 数据库引擎兼容，而大多数其他数据库引擎也选择了解释（1）， 所以 SQLite 也是如此。

(27) SQLite 的出口管制分类号 (ECCN) 是什么？

在仔细审查商务管制清单（CCL）后，我们确信核心公共领域的 SQLite 源代码不属于任何 ECCN， 因此应该将 ECCN 报告为 EAR99。

以上结论适用于核心公共领域的 SQLite。 如果你通过添加新代码扩展 SQLite，或将 SQLite 静态链接到你的应用，情况可能会改变，具体 ECCN 视你的情况而定。

(28) 我的查询没有返回我期望的列名。这是 bug 吗？

如果结果集的列名由 AS 子句指定，SQLite 会保证使用 AS 关键字右边的标识符作为列名。 如果结果集没有使用 AS 子句，SQLite 就可以自由地为该列命名。 详见 sqlite3_column_name() 文档。

(29) 我的数据库去哪里了？（或者：我的数据库怎么变空了？）

除非使用禁止创建的标志，否则 SQLite 会在数据库不存在时自动创建它。 新创建的数据库初始为空。 这可能会让人困惑，当由于文件名拼写错误或在不同当前目录中使用相对路径而在不同上下文中打开不同的数据库文件时，会出现这种情况。
