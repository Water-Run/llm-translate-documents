1. 概述

SQL 语言是一个“标准”。即便如此，没有两个 SQL 数据库引擎的工作方式完全相同。 每个 SQL 实现都有其自身的特殊性和奇怪之处，SQLite 也不例外。

本文档力求突显 SQLite 与其他 SQL 实现之间的主要差异，帮助那些正在将应用迁移到 SQLite 或从 SQLite 迁出的开发者，或者试图构建一个能够跨多个数据库引擎正常运行的系统。

如果你是 SQLite 用户并遇到了本文未提及的某些 SQLite 特性，请通过在 SQLite 论坛上发布简短留言让开发者知道。

2. SQLite 是嵌入式的，而非客户端-服务器

每当将 SQLite 与 SQL Server、PostgreSQL、MySQL 或 Oracle 等其他 SQL 数据库引擎进行比较时，首先需要认识到 SQLite 并不打算取代或与其中任何一个系统竞争。SQLite 是一个无服务器的系统。它没有单独的服务器进程来管理数据库。一个应用通过函数调用与数据库引擎交互，而不是向独立的进程或线程发送消息。

SQLite 是嵌入式的并且无服务器的这一事实是一个特性，而不是漏洞。

像 MySQL、PostgreSQL、SQL Server、Oracle 等客户端/服务器数据库在现代系统中仍然扮演着重要角色。它们解决了一个重要的问题。但 SQLite 解决的是另一个问题。SQLite 与客户端/服务器数据库各有其定位。那些在比较 SQLite 与其他 SQL 数据库引擎的开发者必须清楚地区分这两者。

请参见SQLite 的适用情景文档以获取更多信息。

3. 灵活的数据类型

在数据类型方面，SQLite 非常灵活。数据类型是建议性的，而非强制性的。

一些评论者称 SQLite 为“弱类型”，而其他 SQL 数据库是“强类型”。我们认为这些术语既不准确又带有贬义。我们更愿意说 SQLite 是“灵活类型”，而其他 SQL 数据库引擎则“僵化类型”。

请参见SQLite 中的数据类型文档，深入了解 SQLite 的类型系统。

关键点在于 SQLite 对你插入数据库的数据类型非常宽容。例如，如果某列的数据类型声明为“INTEGER”，而应用向该列插入了文本字符串，SQLite 会像其他 SQL 引擎一样先尝试将文本转换为整数。因此，如果向 INTEGER 列插入 '1234'，该值会被转换为整数 1234 并存储。但如果向 INTEGER 列插入像 'wxyz' 这样的非数字字符串，SQLite 会与其他 SQL 数据库不同，它不会抛出错误，而是将实际的字符串值存入该列。

类似地，SQLite 允许你将一个 2000 字符的字符串存入类型为 VARCHAR(50) 的列。其他 SQL 实现会报错或截断字符串。SQLite 会完整地存储这 2000 字符的字符串，不会丢失信息，也不会抱怨。

问题的根源在于，当开发者使用 SQLite 进行初期编码并让应用运行起来，然后又尝试为部署转向 PostgreSQL 或 SQL Server 等其他对数据类型更严格的数据库时，会出现故障。如果应用最初利用了 SQLite 灵活的数据类型，那么迁移到一个对数据类型更挑剔的数据库时就会失败。

灵活类型是 SQLite 的一个特性，不是一个漏洞。灵活类型关乎自由。然而我们也理解，对于习惯了更严格数据类型规则的其他数据库的开发者来说，这个特性有时会令人困惑。回头看来，也许如果 SQLite 只是实现了一个 ANY 类型，让开发者在需要时显式声明使用灵活类型，而不是默认启用灵活类型，可能会少些迷惑。作为对那些期望僵化类型的开发者的体贴，SQLite 3.37.0（2021-11-27）引入了STRICT 表选项。STRICT 表要么强制执行其他 SQL 数据库引擎中存在的强制数据类型约束，要么允许显式的 ANY 类型来保留 SQLite 的灵活类型。

3.1. 没有单独的 BOOLEAN 数据类型

与大多数其他 SQL 实现不同， SQLite 没有单独的 BOOLEAN 数据类型。 相反，TRUE 和 FALSE 通常表示为整数 1 和 0。 这似乎并不会引起太多问题，我们很少收到相关投诉。 但这点值得注意。

从 SQLite 3.23.0 版本（2018-04-02）开始，SQLite 也将 TRUE 和 FALSE 关键字识别为分别等价于整数 1 和 0。 这提升了与其他 SQL 实现的兼容性。 但为了向后兼容，如果有名为 TRUE 或 FALSE 的列，那么这些关键字会被当作引用那列的标识符，而不是 BOOLEAN 字面量。

3.2. 没有单独的 DATETIME 数据类型

SQLite 没有 DATETIME 数据类型。 相反，日期和时间可以通过以下任意一种方式存储：

  • 以 ISO-8601 格式的 TEXT 字符串。例如：'2018-04-02 12:13:46'。
  • 以自 1970 年以来的秒数表示的 INTEGER（也称为“unix 时间”）。
  • 以 REAL 值表示的分数朱利安日数。

SQLite 内置的日期与时间函数理解以上所有格式，并可以自由地在它们之间转换。 你选择使用哪种格式，完全由应用决定。

3.3. 数据类型是可选的

由于 SQLite 在数据类型方面非常灵活且宽容，可以创建完全未指定数据类型的表列。 例如： 

CREATE TABLE t1(a,b,c,d);

“t1” 表有四个列 “a”、“b”、“c”和“d”，没有赋予它们任何特定的数据类型。 你可以在这些列中存储任何内容。

4. 外键约束默认未启用

SQLite 自诞生以来就能解析外键约束，但真正能够执行这些约束是后来才加的，在3.6.19 版本（2009-10-14）。到外键约束执行被引入时，已经有数以百万计的数据库在使用，这些数据库中有些外键约束并不正确。为了避免破坏那些遗留数据库，SQLite 默认关闭外键约束执行。

应用可以在运行时使用PRAGMA foreign_keys语句启用外键约束执行。或者也可以通过-DSQLITE_DEFAULT_FOREIGN_KEYS=1编译选项在编译时启用。

5. 主键有时可以包含 NULL

SQLite 表中的 PRIMARY KEY 通常只是 UNIQUE 约束。由于历史上的一个疏忽， PRIMARY KEY 的列值被允许为 NULL。 这是一个漏洞，但在问题被发现时，已经有大量数据库依赖于这种行为，因此决定继续支持该“有漏洞”的行为。 你可以通过在 PRIMARY KEY 的每个列上添加 NOT NULL 约束来规避这个问题。

例外情况：

  • INTEGER PRIMARY KEY 列的值必须始终是非 NULL 的整数，因为 INTEGER PRIMARY KEY 是ROWID的别名。 如果尝试向 INTEGER PRIMARY KEY 列插入 NULL，SQLite 会自动将其转换为一个唯一整数。

  • WITHOUT ROWID 和 STRICT 功能是在发现此漏洞之后加入的，因此 WITHOUT ROWID 和 STRICT 表可以正确运行： 它们禁止 PRIMARY KEY 中出现 NULL。

6. 聚合查询可以包含未在 GROUP BY 子句中的非聚合结果列

在大多数 SQL 实现中，聚合查询的输出列只能引用聚合函数或 GROUP BY 子句中列出的列。 在聚合查询中引用普通列毫无意义，因为每个输出行可能由输入表中的两行或更多行组成。

SQLite 并不强制执行这一限制。 聚合查询的输出列可以是包括 GROUP BY 子句中未出现的列在内的任意表达式。 该特性有两个用途：

  1. 在 SQLite（但不是我们所知的任何其他 SQL 实现）中，如果聚合查询只包含一个 min() 或 max() 函数， 那么输出列的值来自实现了该 min() 或 max() 值的行。 如果有两行或更多行拥有相同的 min() 或 max() 值，那么列值将从其中任一行随意选取。

    例如，查找工资最高的员工： 

    SELECT max(salary), first_name, last_name FROM employee;

    在上述查询中，first_name 与 last_name 的值对应于满足 max(salary) 条件的那一行。

  2. 如果一个查询根本不包含任何聚合函数，则可以在 GROUP BY 子句中添加一个子句来代替 DISTINCT ON 子句。换句话说，输出行经过过滤后，每一组 GROUP BY 列的不同取值只显示一行。如果两行或更多行在 GROUP BY 列上原本会具有相同的取值集合，那么会任意选取其中一行。（SQLite 支持 DISTINCT，但不支持 DISTINCT ON，其功能可由 GROUP BY 代替。）

7. SQLite 默认不执行完整的 Unicode 大小写折叠

SQLite 并不识别所有 unicode 字符的大小写区分。像 upper() 与 lower() 这样的 SQL 函数只对 ASCII 字符有效。造成这一现象有两个原因：

  1. 虽然现在已经稳定，但在 SQLite 初期设计阶段，Unicode 大小写折叠的规则仍在不断变化。 这意味着行为可能随着每个新 Unicode 版本而改变，进而扰乱应用并破坏索引。

  2. 完整且正确的 Unicode 大小写折叠所需的表格比整个 SQLite 库还要大。

如果在编译时使用了-DSQLITE_ENABLE_ICU选项并链接了 国际 Unicode 组件库，则 SQLite 支持完整的 Unicode 大小写折叠。

8. 接受双引号字符串字面量

SQL 标准要求用双引号包围标识符，用单引号包围字符串字面量。例如：

  • "this is a legal SQL column name"
  • 'this is an SQL string literal'

SQLite 接受以上两种写法。但是，为了兼容 MySQL 3.x（在 SQLite 初期设计时 MySQL 3.x 是最广泛使用的关系型数据库之一），SQLite 还会在双引号字符串不匹配任何有效标识符时，将其解释为字符串字面量。

这一“错误特性”意味着拼写错误的双引号标识符会被当作字符串字面量，而不是报错。 它也使得刚接触 SQL 语言的开发者养成错误习惯——在实际上应该使用单引号字符串字面量时，反而使用双引号。

事后看来，我们不应该尝试让 SQLite 接受 MySQL 3.x 的语法，也不应该允许双引号字符串字面量。 然而，有无数应用依赖双引号字符串字面量，因此为了不破坏遗留系统，我们继续保留这一能力。

从 SQLite 3.27.0（2019-02-07）开始，使用双引号字符串字面量将向错误日志发送警告信息。

从 SQLite 3.29.0（2019-07-10）开始，可以在运行时通过 SQLITE_DBCONFIG_DQS_DDL 和 SQLITE_DBCONFIG_DQS_DML 操作， 调用 sqlite3_db_config() 来禁用双引号字符串字面量。 默认设置可以在编译时通过-DSQLITE_DQS=N选项修改。 我们鼓励应用开发者在编译时使用 -DSQLITE_DQS=0，以默认禁用双引号字符串字面量这个“错误特性”。 如果无法做到这一点，也可以在 C 代码中对单个数据库连接禁用双引号字符串字面量，如下所示： 

sqlite3_db_config(db, SQLITE_DBCONFIG_DQS_DDL, 0, (void*)0);
sqlite3_db_config(db, SQLITE_DBCONFIG_DQS_DML, 0, (void*)0);

或者，如果默认禁用了双引号字符串字面量，但需要针对某些历史数据库连接选择性启用，可以使用与上面相同的 C 代码，只需将第三个参数从 0 改为 1。

从 SQLite 3.41.0（2023-02-21）开始，命令行工具默认禁用 SQLITE_DBCONFIG_DQS_DDL 与 SQLITE_DBCONFIG_DQS_DML。 如果需要恢复旧行为，请使用“.dbconfig”点命令重新启用。

9. 关键字常可用作标识符

SQL 语言关键字种类繁多。 大多数 SQL 实现不允许在不使用双引号的情况下将关键字作为标识符（表名或列名）。 然而 SQLite 更加灵活。只要上下文明确，许多关键字都可以在不用引号的情况下直接作为标识符使用。

例如，以下语句在 SQLite 中是合法的： 

CREATE TABLE union(true INT, with BOOLEAN);

我们所知的其他任何 SQL 实现都会因为使用关键字 “union”、“true” 和 “with” 作为标识符而报错。

能够将关键字用作标识符有助于向后兼容。新增关键字时，如果旧的 schema 恰好用这些关键字作为表或列名，仍然能正常工作。 不过，将关键字当作标识符也有可能导致意外结果。例如： 

CREATE TRIGGER AFTER INSERT ON tableX BEGIN
  INSERT INTO tableY(b) VALUES(new.a);
END;

上一条语句创建的触发器名为 “AFTER”，但它是一个 “BEFORE” 触发器。因为必须把 “AFTER” 当作标识符来解析，无法把它当作关键字使用，这是唯一的解析方式。 再比如： 

CREATE TABLE tableZ(INTEGER PRIMARY KEY);

tableZ 表有一个名为 “INTEGER” 的列。该列未指定数据类型，但它是 PRIMARY KEY。 该列不是表的INTEGER PRIMARY KEY，因为它没有指定数据类型。 “INTEGER” 词被用作列名标识符，而非数据类型关键字。

10. 可疑的 SQL 不会报错或警告

SQLite 最初的实现遵循Postel 定律，其中一条内容是“对接受的内容要宽容”。 过去这种设计被视为良好：系统可以接受不规范的输入，并尽力在不频繁抱怨的前提下完成处理。 然而近年来，人们更倾向于严格接受的输入，以便更容易发现错误。

如今有数以百万计的应用依赖 SQLite 灵活且宽容的设计选择。 如果我们将 SQLite 改造成遵循当前对于严格行为的偏好，就会破坏那些遗留应用。

11. AUTOINCREMENT 的行为与 MySQL 不同

AUTOINCREMENT 在 SQLite 中的工作方式与 MySQL 不同。 这常常令最初在 MySQL 上学习 SQL 然后开始使用 SQLite 的人迷惑， 他们期望两个系统的行为完全一致。

请参阅SQLite AUTOINCREMENT 文档，了解 AUTOINCREMENT 在 SQLite 中的具体行为和限制。

12. 文本字符串中允许 NUL 字符

NUL 字符（ASCII 代码 0x00 以及 Unicode \u0000）可以出现在 SQLite 字符串的中间。 这可能导致预期之外的行为。 有关更多信息，请参见“字符串中的 NUL 字符”文档。

13. SQLite 区分整数和文本字面量

SQLite 声称以下查询返回假： 

SELECT 1='1';

它这么做的原因是整数并不是字符串。 其他所有主流 SQL 数据库引擎都说这是真，至于原因，SQLite 的创建者也不理解。

14. SQLite 处理逗号连接的优先级错误

SQLite 赋予所有连接运算符相同的优先级，并从左向右处理它们。 但这并不完全正确。逗号连接的优先级应该低于所有其他连接运算符。 换句话说，像这样的 FROM 子句：
