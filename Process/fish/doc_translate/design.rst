Design
======

本文描述了用于设计 fish 的设计原则。fish 的设计有三个高层目标：

1. 在其他 shell 语言中能够完成的所有事情，都应该能够在 fish 中完成，尽管 fish 在实现时可能依赖外部命令。

2. fish 应当对用户友好，但不能以牺牲表达能力为代价。通过精心设计，大多数“功能强大”与“易于使用”之间的权衡都是可以避免的。

3. 在不违背上述目标的前提下，fish 应尽可能遵循 POSIX 标准。

为了实现这些高层目标，fish 的设计依赖于若干更具体的设计原则。下面将逐一介绍这些原则，并附上相应的设计动机与示例。


正交性原则
----------

Shell 语言应当拥有一组小而正交的特性。如果两个特性相关但不完全相同，那么应当移除其中一个，并将另一个设计得足够强大和通用，以覆盖这两个特性的常见使用场景。

动机：

相似的特性会使语言变得更庞大，从而更难学习。同时也会增加源代码规模，使程序更难维护和更新。

示例：

- Here 文档（here documents）与在管道中使用 echo 非常相似。

- 子 shell、命令替换和进程替换之间关系密切。``fish`` 仅支持命令替换，其他功能可以通过代码块或 psub shell 脚本函数实现。

- 同时拥有别名（alias）和函数会让人困惑，尤其是它们各自都有局限和问题。``fish`` 的函数不存在这两种语法的缺点。

- 众多 Posix 引号风格设计得很愚蠢，尤其是 ``$``。


响应性原则
----------

即使在文件系统繁忙或无响应的情况下，shell 也应尽力始终保持对用户的响应。只有在响应用户主动发起的操作（例如运行命令）时，才允许阻塞。

动机：

糟糕的性能会增加面向用户的复杂性，因为它会迫使用户学会识别并绕过那些缓慢的使用场景。这也会带来极大的挫败感。

示例：

- 语法高亮和自动建议等功能必须以异步方式执行所有磁盘 I/O 操作。

- 启动过程应尽量减少 fork 操作和磁盘 I/O，以便即使系统负载较高时也能启动 fish。


可配置性是一切罪恶之源
----------------------

程序中的每一个配置选项，都是程序无法自行判断用户真正需求的表现，应被视为程序以及实现该选项的程序员的失败。

动机：

不同的配置选项在维护上是噩梦，因为特定配置组合可能引发的潜在 bug 数量会迅速增加。配置选项通常隐含对代码的某些假设，而当代码重构时这些假设可能发生变化，从而引发向后兼容问题。但更重要的是，应该尽量避免配置选项的存在——程序应当足够智能，能够自行做出最佳选择，或者至少做出足够接近最佳的判断。

示例：

- Fish 允许用户设置各种语法高亮颜色。这是必要的，因为 fish 不知道终端默认使用的颜色，而默认颜色可能导致某些内容不可读。理想的解决方案应是：由用户为所有程序统一定义文本颜色偏好，并由终端模拟器将这些颜色属性传递给 fish。

- Fish 不允许用户设置历史记录条目数量、不同语言子风格或其他许多常见的 shell 配置选项。

关于可配置性弊端的一个特别说明是：某些 shell 中存在大量非常有用的功能，但默认未启用。zsh 和 bash 都支持针对特定命令的补全功能，但 bash 默认并未附带这些补全，而 zsh 也默认关闭。zsh 默认关闭的其他功能还包括：包含通配符字符串的 Tab 补全、合理的补全分页器以及历史文件。


用户优先原则
------------

在设计程序时，应首先思考如何创建一个直观且强大的程序。实现问题应当在用户界面设计完成之后再考虑。

动机：

这一设计规则与其他规则不同，因为它描述的是应当如何设计新特性，而不是特性本身应具备什么。若过于关注“能做什么”以及“什么容易实现”，就会暴露过多实现细节。这意味着用户必须深入了解底层系统，才能猜测 shell 的工作方式，同时也会使语言变得相当底层化。

示例：

- Shell 只应有一种输入类型：命令列表。循环、条件语句和变量赋值都应通过普通命令完成。

- 内建命令与 shell 脚本函数之间的差异应尽可能小。内建命令和 shell 脚本函数应当与其他命令具有完全相同的参数展开方式，应能出现在管道中的任何位置，并支持任何 I/O 重定向。

- 在进行命令替换时，不通过 fork 来提供伪变量作用域；fish 中的所有命令都在同一进程中执行，并通过真正的作用域机制来支持变量范围。

- 所有代码块都以 ``end`` 内建命令结束。


可发现性原则
------------

程序应当被设计为让用户尽可能容易发现其功能。

动机：

一个功能易于发现的程序，可以在更短时间内将新用户转变为专家，因为用户仅通过使用程序本身就能逐渐掌握它。

图形界面程序相对于命令行程序的主要优势在于可发现性。在图形界面中，用户可以通过观察界面，并猜测不同按钮、菜单和控件的功能，从而发现大多数常用特性。而在命令行程序中，传统的功能发现方式是通过手册页。这不仅要求用户切换到另一个程序，还要求他们在下次使用该程序时记住新学到的信息。

示例：

- 所有内容都应支持 Tab 补全，并且每个补全项都应附带说明。

- 每个语法错误和内建命令错误都应包含描述问题原因的错误信息，并提供相关帮助页面。在可能的情况下，语法高亮应以红色标示错误。

- 帮助文档应易于阅读，可从 shell 中轻松访问，内容完整，并包含大量示例。

- 语言应保持一致性，使得一旦用户理解了命令/参数语法，就等于理解了整门语言，并能够通过 Tab 补全来发现新特性。
